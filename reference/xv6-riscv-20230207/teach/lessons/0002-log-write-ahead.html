🧠 一句类比:银行的"先存草稿、再正式入账"

写前日志 = 转账前先在草稿本上改、确认后再正式入账。
你去银行转账,柜员不会直接改主账本(怕改一半停电账本错乱)。
草稿本(log):在另一块固定磁盘区,把所有要改的块先抄一遍。
两阶段提交:① 把草稿写完(commit)→ ② 把草稿内容搬进主账本(install)→ ③ 擦掉草稿。
如果第 1 步后停电:草稿在,重启后从草稿恢复。如果第 2 步后停电:草稿和主账本都有,重启后检测草稿非空就再装一次(幂等)。

口诀:"先写日志,再写 home,最后擦日志" — 老师最爱让你背的一句。

🔍 源码精读(log.c 关键段)

1. log_write 记 blockno(log.c:170-188)

void
log_write(struct buf *b)
{
  int i;
  acquire(&log.lock);
  if (log.lh.n >= LOGSIZE || log.lh.n >= log.size - 1)
    panic("too big a transaction");
  if (log.outstanding < 1)
    panic("log_write outside of trans");

  for (i = 0; i < log.lh.n; i++) {
    if (log.lh.block[i] == b->blockno)   // log absorption
      break;
  }
  log.lh.block[i] = b->blockno;
  if (i == log.lh.n) {  // Add new block to log?
    bpin(b);
    log.lh.n++;
  }
  release(&log.lock);
}

讲法提示(给老师用):log_write 做 3 件事,顺序很关键: ① 抢 log.lock(spinlock,保护内存中的 logheader) ② 扫 log.lh.block[],如果 blockno 已存在 → 不增加 n,不 bpin(log absorption 关键!) ③ 新 block 才 bpin(b) + log.lh.n++ ④ 放锁

⚠️ 老师追问点 1:为什么不直接调 bwrite?——因为 bwrite 会立即同步写盘,无法实现"事务合并"。log_write 是"记账",commit 才是"真写"。

⚠️ 老师追问点 2:bpin 有什么用?——保护这块 buffer 在 commit 真正发生前不被 LRU 淘汰(回到 Day 1 的 bpin 设计)。

2. commit 三步走(log.c:152-160)

static void
commit()
{
  if (log.lh.n > 0) {
    write_log();     // (1) 拷 cache → log[1..n]
    write_head();    // (2) 写 logheader(真实 commit 点)
    install_trans(0); // (3) 拷 log[1..n] → home
    log.lh.n = 0;
    write_head();    // (4) 写 logheader.n=0(擦日志)
  }
}

讲法提示:第 (2) 步是真正的 commit 点。如果系统在第 (2) 步之后、第 (3) 步之前崩溃:log[1..n] 在,home 还没改,重启时 install_trans(1) 把 log 拷到 home(幂等,安全)。

口诀:"write_head 之后,事务就 commit 了;install_trans 是后台搬运工"。

3. begin_op / end_op 事务边界(log.c:90-130)

void
begin_op(void)
{
  acquire(&log.lock);
  while(1){
    if(log.committing){
      sleep(&log, &log.lock);
    } else if(log.lh.n + (log.outstanding+1)*MAXOPBLOCKS > LOGSIZE){
      // this op might exhaust log space; wait for commit.
      sleep(&log, &log.lock);
    } else {
      log.outstanding += 1;
      release(&log.lock);
      break;
    }
  }
}

void
end_op(void)
{
  int do_commit = 0;
  acquire(&log.lock);
  log.outstanding -= 1;
  if(log.outstanding == 0){
    do_commit = 1;
    log.committing = 1;
  } else {
    wakeup(&log);
  }
  release(&log.lock);

  if(do_commit){
    commit();         // 注意:放掉 log.lock 才调 commit!
    acquire(&log.lock);
    log.committing = 0;
    wakeup(&log);
    release(&log.lock);
  }
}

讲法提示:这是 xv6 日志层"最妙"的一段,讲清 3 个细节: ① outstanding 计数 → 最后一个 end_op 触发 commit ② 临界资源(log.lh.n + outstanding*MAXOPBLOCKS > LOGSIZE)防止日志空间耗尽 ③ commit() 必须在释放 log.lock 之后调 — 因为 commit 内调 bread/bwrite 涉及 sleeplock,持 spinlock 调 sleeplock 会死锁(回到 Day 1 锁顺序)

🧩 5 个设计权衡(老师追问时用)

权衡 1:为什么是 redo log 而不是 undo log?

Undo log:先把 home 的"原值"写日志,改 home,崩溃时回滚到原值。Redo log:先把"新值"写日志,再装到 home,崩溃时重做。

xv6 选 redo 因为:并发事务合并(log absorption)— 多个系统调用改同一块时,undo 需要保留多个版本,redo 只要"最终新值"。

Linux 对照:ext4 默认是 journaling 模式(类似 redo),但 jbd2 还支持 ordered(只 journal 元数据,数据写 home)和 writeback(都不 journal)。xv6 没分模式,因为教学只留一种。

权衡 2:为什么 outstanding 计数就够了,不用事务 ID 区分?

xv6 的"事务"实际上是"若干并发 FS 系统调用" — 它们的修改会被合并到下一次 commit。没有事务 ID、没有回滚、没有隔离性。这是"组提交(group commit)+ 最终一致性"的最简实现。

代价:如果某次系统调用 panic,它之前对 buffer 的修改可能已经被下一次 commit 写盘 — 理论上"半成品事务"会被持久化。xv6 接受这个代价,因为崩溃恢复时 install_trans 是幂等的。

权衡 3:为什么 commit 必须在放掉 log.lock 之后才调?

锁顺序:commit 内部调 bread / bwrite → 内部会 acquiresleep。spinlock 持锁时调 sleeplock 会死锁。所以 end_op 第 33 行的写法是显式先放 spinlock、再 commit、再拿回 spinlock、清 committing 标志位、wakeup。

这是 xv6 锁顺序的最严格示范,也是老师必问点。

权衡 4:崩溃恢复的"幂等性"怎么保证?

install_trans 实现是"无脑覆盖" — 把 log[1..n] 的内容 memmove 到 home 块,然后 bwrite。如果系统崩溃时 install_trans 做了一半(部分 home 块被覆盖、部分没有),重启后再调 install_trans(1) 会把未覆盖的 home 块再覆盖一次。

幂等性来源:每次 install_trans 都从 log[1..n] 拷完整新值到 home,而不是"原值+差量"。所以重复执行结果是确定的。

权衡 5:为什么 LOGSIZE=30、MAXOPBLOCKS=10?

约束公式:log.lh.n + outstanding*MAXOPBLOCKS ≤ LOGSIZE。意思是"日志里已有的块 + 所有进行中调用可能写的块 ≤ 日志总容量"。LOGSIZE=30、MAXOPBLOCKS=10 → 最多 3 个并发调用(3*10=30)。

这是最简单的事务并发控制:不是真"并发",是"批处理 + 排队"。Linux jbd2 用事务链表 + 显式 handle,精细但复杂。

🐧 关键 Linux 对照(老师追问预案)

xv6 做法ext4 / jbd2 做法为什么不同
整块 redo logjournal / ordered / writeback 三种教学简化 vs 性能调优
无事务 ID,outstanding 计数handle + transaction 链表组提交 vs 真事务
崩溃恢复 install_trans 幂等jbd2 recovery 也幂等(replay)同样思路,实现复杂度差 10 倍
LOGSIZE=30 静态jbd2 动态,默认 128 MB教学 vs 生产
单一日志区可以 external journal(独立设备)xv6 无多设备抽象

📝 3 道自测(讲完后做,10 分钟)

  1. (选择) xv6 写前日志中,"真实 commit 点"是哪个步骤?

    A. write_log() 写完 log[1..n]　B. write_head() 第一次把 logheader 写盘　C. install_trans() 把 log 内容拷到 home　D. write_head() 第二次写 logheader.n=0

    答案: B。write_head() 第一次写完后,logheader 持久化,系统即使崩溃也能在重启时检测到"有事务未完成"。C 是后台搬运,不是 commit;D 是"擦日志",此时事务已完成。

  2. (简答) log_write 中,为什么要 bpin 被记入日志的 buffer?

    答案: 防止 LRU 淘汰。buffer 内容在内存里改好了,日志还没 commit(还没真写盘)。如果 LRU 把它淘汰,这块数据就丢了,bget 又得从磁盘重读(读到的是旧值),事务不完整。bpin 让 refcnt++ 永远不会被 LRU 选中。commit 完后 bunpin。

  3. (追问) 假设系统在 install_trans 中途崩溃(已经覆盖了 home[5]、还没覆盖 home[7]),重启后会发生什么?

    答案: 安全。recover_from_log 读到 logheader.n > 0,调 install_trans(1) 重新执行一次 — 它会再次覆盖 home[5](幂等,结果还是新值)并继续覆盖 home[7]。最后 write_head 擦日志。整个过程对 home 是"最终新值",不丢不乱。

❓ 老师追问预案(5 个最可能的)

老师可能问答法要点
"为什么 redo 而不是 undo?为什么 5 个权衡里把 redo 放第一?" 关键差异:redo 一次 install_trans 就能恢复最终状态(幂等);undo 要保留每个事务的原值,且多个并发事务的原值会冲突。xv6 用 outstanding 计数天然适合 redo。
"log absorption 怎么工作?同 block 多次 log_write 会发生什么?" 第二次 log_write 扫到 log.lh.block[i] == b->blockno 直接 break,不 bpin、不增加 n。结果:同事务内多次改同一块,日志只记一次;commit 时只 install 一次(用最新 buffer 内容)。
"begin_op 里 sleep(&log, &log.lock) 是什么?为什么这么写?" 这是 xv6 标准 sleep 模式:"sleep 在 chan 上,释放 lk"。spinlock 持锁时调 sleep 会让所有等 log.lock 的进程一起睡死;所以 sleep 接受的第二个参数就是"释放哪个锁"。这是 Unix 早期 sleep/wakeup 协议。
"ext4 的 ordered 模式怎么做的?为什么比 journaling 快?" ordered:只 journal 元数据(如 inode 修改、bitmap 修改),数据块直接写 home。崩溃时:元数据是原子的(要么新、要么旧),数据最多是旧 — 不会"元数据已改但数据没改"的中间态。比 journaling 少一次 IO。
"log.committing 标志位存在的意义?不用可以吗?" 防止并发 commit:有两个 end_op 同时进来,都看到 outstanding==0,都调 commit() — 会双写日志,可能 panic。committing 标志位让 begin_op 在 commit 期间 sleep(&log) 等待。
