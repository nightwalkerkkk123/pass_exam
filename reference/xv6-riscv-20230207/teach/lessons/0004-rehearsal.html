📝 3 张板书图(白板可徒手画)

板书图 1 · 缓冲层(Day 1 核心)

┌────────────────────────────────────────────────┐ │ struct bcache { │ │ spinlock lock; ← 链表锁(纳秒级) │ │ buf buf[NBUF]; ← 30 个 buffer │ │ buf head; ← LRU 哨兵 │ │ } │ └────────────────────────────────────────────────┘ LRU 双向链表(头是最近用): head ◄──► b[0] ◄──► b[1] ◄──► ... ◄──► b[29] ◄──► head ▲ next = MRU prev = LRU ▲ 每个 b 还有 sleeplock(保护 b->data,可睡) 调用协议: bread → 修改 b->data → bwrite → brelse └─ 移 b 到 head.next 释放 sleeplock + spinlock 关键:spinlock 永不睡,sleeplock 可睡,锁顺序固定

画法:左半部画 bcache 结构(3 行),右半部画 LRU 链(箭头方向清楚即可),底部画协议 1 行。**总耗时 60 秒**。

板书图 2 · 日志层(Day 2 核心)

磁盘布局: ┌──────┬────────┬─────┬─────────┬───────┬──────┐ │ boot │ super │ log │ inodes │bitmap │ data │ └──────┴────────┴─────┴─────────┴───────┴──────┘ ▲ log 区 commit() 三步走(全场最重点): ┌────────────────────────────────────────────────┐ │ 1. write_log() 拷 buf → log[1..n] │ │ 2. write_head() 写 logheader ★ 真实 commit│ │ 3. install_trans() 拷 log[1..n] → home │ │ 4. write_head() 写 logheader.n=0 擦日志 │ └────────────────────────────────────────────────┘ 口诀:先写日志,再写 home,最后擦日志 崩溃恢复:recover_from_log → 幂等 install_trans

画法:顶部画磁盘布局 1 行(简化为 6 段),底部画 commit 四步(用方框),重点圈第 2 步写 ★。**总耗时 45 秒**。

板书图 3 · 文件系统层(Day 3 核心)

dinode 结构(64 字节,定长): ┌──────┬─────┬─────┬─────┬─────┬──────────────┐ │ type │ maj │ min │ nln │ sz │ addrs[13] │ │ 2B │ 2B │ 2B │ 2B │ 4B │ 52B = 12+1 │ └──────┴─────┴─────┴─────┴─────┴──────────────┘ bmap 怎么工作(12 直 + 1 间): 假设文件第 bn 块 (0 ≤ bn < 268): bn < 12: return addrs[bn] "12 块内不用翻译" bn ≥ 12: return 间接块[bn - 12] "12 块外要查表" 关键点: ① MAXFILE = 12 + 256 = 268 块 ≈ 268 KB ② addrs[12] = 间接块指针(不是数据) ③ 没有 ext4 的 extents

画法:左半部画 dinode 6 列(标出 addrs[13] 占 52 字节),右半部画 bmap 两分支。**总耗时 60 秒**。

❓ 5 个老师追问预案(整合 Day 1–3)

序号老师可能问答法要点(30 秒内)对应 Day
1 "xv6 缓冲层和 Linux page cache 比,主要缺点是什么?" vivification 风险:NBUF=30 极小,LRU 单链表,热点 buffer 可能被冷数据挤掉。Linux 用 active/inactive 双链表缓解。 Day 1
2 "write_head 才是真实 commit 点 — 如果第 2 步写完、第 3 步 install_trans 中途崩溃,会怎样?" 安全。重启时 recover_from_log 读到 logheader.n > 0,调 install_trans(1) 重新执行,幂等(覆盖同样的新值),不丢不乱。 Day 2
3 "ext4 的 ordered 模式怎么做的?为什么比 journaling 快?" ordered:只 journal 元数据(inode/bitmap 修改),数据块直接写 home。少一次 IO,但不能完全防止"元数据新、数据旧"的极端不一致(实际上由 ordered 标志位 + write barrier 解决)。 Day 2
4 "为什么 bmap 要走 log_write?bget 之后直接 bwrite 不行吗?" log_write 是事务提交方式 — 把 blockno 记入 log.lh,等所有 FS 系统调用都结束,一次性 commit 写盘。这是 Day 2 学的崩溃一致性机制。 Day 3
5 "xv6 的 12+1 和 ext4 extents 相比,数据规模差多少?" xv6 单文件 268 KB,ext4 16 TB(用 extents 树表达连续区间)。xv6 适合教学,ext4 适合生产 — 规模差 6 个数量级。 Day 3

追问预案 6(备用):"xv6 用什么 RISC-V 特性影响 FS 实现?"

答:xv6 用 virtio MMIO 设备(磁盘)+ PLIC 中断控制器。virtio_disk_rw() 直接读写 MMIO 寄存器,等 PLIC 转发的中断。这是 RISC-V 设备驱动的典型模式。Day 1 缓冲层可以引出"等 IO 用什么机制"。

🗣️ 8 分钟讲稿(逐字稿)

[0:00–0:30] 开场

"今天讲 xv6 文件系统源码,主战场三个文件:bio.c、log.c、fs.c,加起来 450 行。我会按"缓冲 → 日志 → 文件系统"三层讲,每层 2–3 分钟,3 张板书图。3 层各回答一个问题:缓冲层回答"数据怎么读到内存",日志层回答"崩溃了怎么办",文件系统层回答"文件怎么找到数据"。"

[0:30–1:30] 画板书图 1 + 讲 bio

"先画图。bcache 是个全局结构,装 30 个 buf,加一个 LRU 链表的哨兵。LRU 链表 head.next 是最近用的,head.prev 是最久没用的。每个 buf 有两个锁:bcache.lock 是 spinlock,保护链表,纳秒级;buf 自己的 lock 是 sleeplock,保护 buf 内容,持锁可长达毫秒,允许等磁盘 IO。"

"调用方遵守三件套:bread 拿到 buf,bwrite 写盘,brelse 释放。brelse 做两件事:释放 sleeplock,加 refcnt 减 1,只在 refcnt 归零时把 buf 移到 LRU 头部(MRU 端)。"

← 板书图 1 圈点:圈出 bcache.lock / 圈出 b->lock / 圈出 brelse 协议

"金句:缓冲层解决的是"如何在内存里高效地管理磁盘块的缓存",核心是双锁粒度匹配 — 链表必须原子,buf 内容可睡。"

[3:30–4:00] 过渡 + 画板书图 2

"接下来第二层 — 日志层。它回答的问题是:如果系统崩溃在"写到一半"会怎样?"

[4:00–6:00] 讲 log

"xv6 用整块 redo log。磁盘布局上有一段固定区域叫 log,里面有 1 块 header 加若干数据块。FS 系统调用开始调 begin_op,结束调 end_op。end_op 时如果 outstanding 归零,触发 commit。"

"commit 走 4 步:① 拷 buf 到 log → ② 写 logheader ★ 这是真实 commit 点 → ③ 拷 log 到 home → ④ 写 logheader.n=0 擦日志。崩溃恢复时,如果重启读到 logheader.n > 0,就 install_trans 重做 — 幂等。"

← 板书图 2 圈点:圈出第 2 步 write_head(标 ★)/ 圈出口诀"先写日志再写 home"

"金句:日志层解决的是"崩溃一致性",不是"性能"。xv6 用 outstanding 计数实现最简单的组提交,ext4 走得更远但本质没变。"

[6:00–6:30] 过渡 + 画板书图 3

"最后第三层 — 文件系统层。它回答"文件怎么找到数据"。"

[6:30–7:30] 讲 fs

"xv6 磁盘布局分 5 段:boot、super、log、inodes、bitmap、data。superblock 是个总账本,记着每段起点。"

"dinode 是 64 字节的定长结构,关键字段是 addrs[13] — 12 个直接块 + 1 个间接块指针。bmap(ip, bn) 把"文件第 bn 块"翻译成"磁盘第几块":bn<12 直接返回 addrs[bn],bn>=12 要先读间接块(里面是 uint[256] 数组),再返回数组第 bn-12 项。最大文件 268 KB。"

← 板书图 3 圈点:圈出 addrs[13]/ 圈出 bmap 两个分支

"金句:inode 是号码牌,bmap 是翻译官。xv6 的 12+1 是 ext2 减配版,ext4 用 extents 表达连续区间,规模差 6 个数量级。"

[7:30–8:00] 收尾

"总结一下。3 层各解决一个独立问题:bio 管"内存缓存",log 管"崩溃一致",fs 管"名字到数据"。xv6 总共 450 行,讲得清 8 分钟;ext4 是它的 1000 倍,但分层思想没变。"
