🧠 一句类比:酒店档案柜

文件系统层 = 酒店的档案柜系统。
客人(进程)说"我要看 305 房的钱包" — 这个 305 不是物理房间号,而是房号。
superblock:档案柜总账本,记着"几号柜子放什么类别"。
dinode(房卡):每位客人的房卡,记着"我的物品放在 12 个贴身柜 + 1 个仓库柜"。
bmap:前台"翻译官",把"305 房第 3 件物品"翻译成"3 号柜子的 3 抽屉"。

口诀:"inode 是号码牌,bmap 是翻译官" — 老师最爱的金句。

xv6 文件系统层 — 一张图讲完 磁盘布局(从 0 块开始): ┌──────┬────────┬─────┬─────────┬───────┬──────┐ │ boot │ super │ log │ inodes │bitmap │ data │ │ 0 │ 1 │ 2.. │ ... │ ... │ ... │ └──────┴────────┴─────┴─────────┴───────┴──────┘ (由 superblock 各项指明各段起点) dinode 结构(磁盘 inode,64 字节): ┌────────┬─────┬─────┬─────┬─────┬─────────────────────┐ │ type │ maj │ min │ nln │ sz │ addrs[13] = 52 字节 │ │ (2B) │ (2) │ (2) │ (2) │ (4) │ [12 直 + 1 间指] │ └────────┴─────┴─────┴─────┴─────┴─────────────────────┘ bmap 怎么工作(12 + 1 结构): 假设文件第 bn 块 (0 ≤ bn < 268): bn < 12: 直接返回 addrs[bn] ← "12 块以内不用翻译" 若为 0,balloc 分配新块填入 bn ≥ 12: bn' = bn - 12 (0 ≤ bn' < 256) 1) 加载 addrs[12] 指向的"间接块"(1 次 IO) 2) 间接块是 uint[256] 数组 3) 返回 间接块[bn'] ← "12 块以外要查一张表" 若间接块中该位置为 0,balloc 分配 关键点(板书时圈出来): ① MAXFILE = 12 + 256 = 268 块 ≈ 268 KB(BSIZE=1024) ② addrs[] 第 13 个 = 一级间接块指针(不是数据) ③ 没有 ext4 的 extents(连续块)概念 ④ bmap 自动分配 = 写时即分配(eager allocation)

🔍 源码精读(fs.c 关键段)

1. bmap — inode 块映射核心(fs.c:383-417)

static uint
bmap(struct inode *ip, uint bn)
{
  uint addr, *a;
  struct buf *bp;

  if(bn < NDIRECT){
    if((addr = ip->addrs[bn]) == 0){
      addr = balloc(ip->dev);
      if(addr == 0)
        return 0;
      ip->addrs[bn] = addr;
    }
    return addr;
  }
  bn -= NDIRECT;

  if(bn < NINDIRECT){
    // Load indirect block, allocating if necessary.
    if((addr = ip->addrs[NDIRECT]) == 0){
      addr = balloc(ip->dev);
      if(addr == 0)
        return 0;
      ip->addrs[NDIRECT] = addr;
    }
    bp = bread(ip->dev, addr);
    a = (uint*)bp->data;
    if((addr = a[bn]) == 0){
      addr = balloc(ip->dev);
      if(addr){
        a[bn] = addr;
        log_write(bp);
      }
    }
    brelse(bp);
    return addr;
  }

  panic("bmap: out of range");
}

讲法提示(给老师用):

  1. 第 8-16 行(直接块分支):如果 addrs[bn] 已经有 blockno,直接返回;没有就 balloc 分配,再写回 addrs[]。
  2. 第 17 行:bn -= NDIRECT — 把 bn 改写成"间接块数组里的下标"。
  3. 第 19-26 行(间接块分支):先确保 addrs[NDIRECT] 指向一个已分配的间接块(没有就 balloc);再 bread 这个间接块,拿到它的内容指针 a。
  4. 第 27-33 行(分配新数据块):如果 a[bn] 是 0(还没分配数据块),balloc 拿到新 blockno,写回 a[bn]。关键:log_write(bp) — 修改间接块本身要走日志(Day 2 的设计),保证崩溃一致。
  5. 第 35 行:brelse(bp) — 释放间接块的 sleeplock。

⚠️ 老师追问点 1:为什么修改间接块要 log_write 而直接块不需要?——因为直接块的修改写回 dinode 本身就 log_write 了(在 writei() 里),而间接块的修改不直接写在 inode 的 addrs[] 里,必须自己 log_write。

⚠️ 老师追问点 2:bmap 为什么不 lazy 分配(类似 ext4 的 delayed allocation)?——教学简化,加上 xv6 假设"分配即写入",不需要缓冲。

2. dinode 在磁盘上长啥样(fs.h:32-39)

struct dinode {
  short type;           // File type
  short major;          // Major device number (T_DEVICE only)
  short minor;          // Minor device number (T_DEVICE only)
  short nlink;          // Number of links to inode in file system
  uint size;            // Size of file (bytes)
  uint addrs[NDIRECT+1];   // Data block addresses
};

讲法提示:64 字节的定长结构,IPB = 1024/64 = 16 个 dinode 装在一个 1 KB 块里。inode number 就是 dinode 在所有 inode 块中的全局索引(不是块号!)。

关键宏:IBLOCK(i, sb) = i / IPB + sb.inodestart — 给定 inode 号,算出它在哪个磁盘块。

3. 目录是 dirent 数组(fs.h:42-46)

struct dirent {
  ushort inum;
  char name[DIRSIZ];
};

讲法提示:16 字节/项,inum == 0 表示"此槽位空"。目录本身就是一个 inode(type == T_DIR),它的 data 块里装 dirent 数组。"文件名"在 xv6 里硬限 14 字符(DIRSIZ=14,含结尾 NUL)。

🧩 5 个设计权衡(老师追问时用)

权衡 1:为什么 12 个直接块 + 1 个间接块,没有 extents?

12 直接 + 1 间接能表达任意稀疏(每个数据块独立分配),最大 268 KB(BSIZE=1024)。

ext4 extents用"起始块 + 长度"表达连续区间,极大文件(GB 级)只需要几个 extent 项。xv6 不需要,因为教学内核 + 假设"小文件" + 想让学生看清单层块映射而不是"extent 树"。

Linux 对照:ext4 默认是 extents(4 KB 数据块时一个 extent 可表达 128 MB 连续)。早期 ext2/ext3 用 12 直 + 1/2/3 间接,xv6 抄的是 ext2 那套。

权衡 2:为什么 1 级间接,没有 2 级或 3 级?

1 级间接支持 256 块(BSIZE=1024)。最大 268 块对教学够用。2 级间接会让"逻辑块号到物理块号"的代码复杂度翻倍,不划算。

真实场景:ext2/3 用 12 直 + 1 一级 + 1 二级 + 1 三级,最大 16 GB 左右的单文件。xv6 砍掉是为了讲得清。

权衡 3:bmap 自动 balloc 是不是"贪婪"?

是的 — xv6 一旦访问文件的某块,该块就必须分配。缺点:稀疏文件空洞 = 实际占盘 = 浪费。优点:写时简单,不需要 delayed allocation 逻辑。

Linux 对照:ext4 引入 delayed allocation(写到 page cache 不立即分配 block,等写回时再分配并尽量连续),性能高 30%+,但实现复杂。

权衡 4:为什么 addrs[NDIRECT+1] 不用结构体?

用 addrs[13] 数组,第 13 个专门存"间接块指针",靠代码约定区分。替代方案是 struct { uint direct[12]; uint indirect; },但 C 数组访问更快、序列化布局更明确、fs.h 同步被 mkfs 引用 — 兼容性更稳。

权衡 5:为什么 NDIRECT=12 而不是 10 或 15?

sizeof(struct dinode) 决定 IPB,IPB 决定每块放多少 inode,xv6 选 12 让 dinode 恰好 64 字节(IPB=16),1 KB 块对齐。这是个魔法数字,没什么玄学。

📝 3 道自测(讲完后做,10 分钟)

  1. (选择) xv6 中,bmap(ip, 100) 实际经过几次磁盘 IO(假设 inode 已在内存,所有数据块都未分配)?

    A. 0 次　B. 1 次(只读间接块)　C. 2 次(读间接块 + 写新数据块)　D. 3 次(读间接块 + 写新数据块 + 写间接块)

    答案: D(粗略说 3 次,准确说 4 次含 inode 回写)。流程:① balloc 找空闲块要读 bitmap → ② bread 间接块 → ③ bwrite 间接块(log_write 走日志,要写 3 次:home+log+header)→ ④ balloc 完成后调用 writei 把 inode 写回(也要走日志)。老师可能更正"3 次"为"更复杂",关键看追问深度。

  2. (简答) xv6 的目录是 inode,还是普通文件?创建新文件需要更新哪几个数据结构?

    答案: 目录 = inode(type==T_DIR),data 块装 dirent 数组。创建新文件要更新:
    ① 分配新的 inode(ialloc 改 bitmap + 写 dinode 块)
    ② 在父目录的 dirent 数组里加一项(dirlink 改父目录的 data 块)
    ③ 更新父目录 inode 的 size
    全部要走 log_write,Day 2 学的日志层保证这三个修改一起 commit 或一起回滚。

    3. <3>

    (追问) xv6 没有 extents,bmap 是 O(1) 找块映射。为什么 ext4 还要用 extents?

    答案: 元数据大小。xv6 268 块文件用 52 字节 addrs[] 足够(每个指针 4 字节)。ext4 单个文件 16 TB = 4 G 块,需要 4 G 个 4 字节指针 = 16 GB 元数据,完全不可接受。ext4 一个 extent 项 = "起始块 + 长度" = 12 字节,4 G 块只需要 ~340 M 项(还要 B+ 树压缩)。xv6 不需要因为它假设文件小。

❓ 老师追问预案(5 个最可能的)

老师可能问答法要点
"为什么 bmap 直接写 ip->addrs[bn] = addr,没走 log_write?" 因为 addrs[] 在内存 inode里(由 iget 读入并 lock)。整个 inode 的回写由 writei() 在文件 close/写盘时统一调 log_write。直接块的修改跟着 inode 一起 commit。
"bmap 中 balloc 失败返回 0,bmap 也返回 0,调用方怎么处理?" 调用方(通常是 writei)会检查返回值,返回 -ENOSPC 给用户态。xv6 的 -ENOSPC 走标准 errno 路径(在 syscall.c 里查 syscalls[] 表)。
"目录是文件,那 / 的 inode 是几号?" ROOTINO = 1(fs.h:5)。ialloc 顺序分配 inode 号,1 永远是 root。文件系统格式化时 mkfs 写好。
"硬链接怎么实现?为什么 dinode.nlink 字段存在?" 硬链接 = 在某目录加一个 dirent,inum 指向同一个 inode,nlink++。删除文件 = nlink--,归零时调 itrunc 释放所有 block。软链接 = 新建一个 type==T_SYMLINK 的文件,data 块装目标路径字符串。
"xv6 inode 缓存怎么做的?iget 和 bget 什么关系?" inode 缓存 = `struct inode icache[NINODE]`(proc.h 或 fs.c 定义)。iget 查 icache,命中就 refcnt++ 返回;未命中就 readsb + balloc 一个空闲 icache 项 + bread 读 dinode。**和 bget 一样的"双重 LRU + refcnt"模式**,只是粒度从块变 inode。
