🧠 一句类比:酒店的前台 + 行李寄存柜

缓冲层 = 行李寄存系统。
客人(进程)进酒店要先存行李再上楼(把磁盘块先取到内存再读写)。
前台(bcache.lock, spinlock):管"哪些柜子被占用、哪些空着",改这条信息必须瞬时完成。
柜子锁(b->lock, sleeplock):管"这个柜子里装什么",可能要去仓库调货(磁盘 IO),可以慢慢来。
LRU:最久没人取的行李会被踢出去腾地方。

为什么要这个类比:能讲清为什么需要两种锁(spinlock 不让睡、sleeplock 可以睡)、为什么 LRU(空间有限)。

📝 板书图(讲台上要画出来)

xv6 缓冲层 — 一张图讲完 ┌────────────────────────────────────────────────┐ │ struct bcache { │ │ spinlock lock; ← 保护链表结构(纳秒级) │ │ buf buf[NBUF]; │ │ buf head; ← LRU 链表的哨兵节点 │ │ } │ └────────────────────────────────────────────────┘ LRU 双向链表(头是最近用,尾是最久没用): head ◄──► b[0] ◄──► b[1] ◄──► ... ◄──► b[NBUF-1] ◄──► head ▲ ▲ │ │ next 方向 = "最近用过的" ────────────────▶ prev 方向 = "最久没用的" 每个 b 还有自己的 sleeplock(保护 b->data,持锁可睡) 调用协议: ┌──────────┐ │ 进程 P │ acquire(sleeplock on b) ┐ │ bread() ├────────────────────────────► 在这里读/写 b->data(可以睡) │ bwrite()│ release sleeplock │ (可包含磁盘 IO) │ brelse()└──────────────────────────-─┘ └──────────┘ ──► 原子地把 b 移到 head.next(MRU 端) 释放 sleeplock + 释放 spinlock 关键点(板书时圈出来): ① spinlock 持锁时间 < 100ns,绝对不能调 IO ② sleeplock 持锁可长达毫秒级(等磁盘) ③ brelse 必被调,否则 LRU 永不更新 → OOM

🔍 源码精读(bio.c 关键段)

1. bget 找 buffer(bio.c:48-77)

static struct buf*
bget(uint dev, uint blockno)
{
  acquire(&bcache.lock);

  // Is the block already cached?
  for(b = bcache.head.next; b != &bcache.head; b = b->next){
    if(b->dev == dev && b->blockno == blockno){
      b->refcnt++;
      release(&bcache.lock);
      acquiresleep(&b->lock);
      return b;
    }
  }

  // Not cached. Recycle the LRU unused buffer.
  for(b = bcache.head.prev; b != &bcache.head; b = b->prev){
    if(b->refcnt == 0) {
      b->dev = dev;
      b->blockno = blockno;
      b->valid = 0;
      b->refcnt = 1;
      release(&bcache.lock);
      acquiresleep(&b->lock);
      return b;
    }
  }
  panic("bget: no buffers");
}

逐行讲(给老师用,挑重点):

  1. acquire(&bcache.lock) — "先抢链表锁,这一段不能睡"
  2. 第一个 for 循环 — "从头往后扫(MRU→LRU),命中就 refcnt++ 返回"
  3. 第二个 for 循环 — "从尾往前扫(LRU→MRU),找 refcnt==0 的淘汰"
  4. release(&bcache.lock) — "放链表锁之后才 acquiresleep,这个顺序很关键"(详见设计权衡 1)
  5. acquiresleep(&b->lock) — "再睡锁保护 buffer 内容"

⚠️ 老师最可能问的点:为什么两个 for 循环方向相反?第一个"找存在的"从 MRU 端开始效率高(刚用的很可能在前面);第二个"找可淘汰的"从 LRU 端开始(保证真淘汰最久没用的)。

2. brelse 释放 buffer(bio.c:112-125)

void
brelse(struct buf *b)
{
  if(!holdingsleep(&b->lock))
    panic("brelse");
  releasesleep(&b->lock);

  acquire(&bcache.lock);
  b->refcnt--;
  if (b->refcnt == 0) {
    // no one is waiting for it.
    b->next->prev = b->prev;
    b->prev->next = b->next;
    b->next = bcache.head.next;
    b->prev = &bcache.head;
    bcache.head.next->prev = b;
    bcache.head.next = b;
  }

  release(&bcache.lock);
}

讲法提示:这是全场最被低估的函数,要做 3 件事: ① 释放 sleeplock(让别人能用) ② 拿 spinlock、refcnt--、放 spinlock ③ **只在 refcnt==0 时**把 b 从当前位置"摘下",重新挂到 head.next(MRU 端)

⚠️ 老师追问点:为什么释放 sleeplock 在拿 spinlock 之前?答:锁顺序固定 — sleeplock 永远在 spinlock 之外。如果你先拿 spinlock 再调 releasesleep 就会违反自己定的规则。

🧩 5 个设计权衡(老师追问时用)

权衡 1:为什么用两种锁(spinlock + sleeplock)?

单一锁(假设全用 spinlock)会有问题:进程在锁内等磁盘 IO 时仍占着 CPU 旋转,系统卡死;单一 sleeplock 又有问题:链表操作必须原子。**双锁粒度匹配**:spinlock 保护"短操作(纳秒)"、sleeplock 保护"长操作(可含 IO)"。

Linux 对照:Linux 用 address_space->i_pages 的 radix tree + 每页 spinlock;xv6 用 LRU 双链表 + 两种锁。复杂度:xv6 ~120 行能讲完,Linux page cache 几千行。

权衡 2:为什么 LRU 而不是 LFU / ARC?

教学内核原则:O(1) 替换、代码可读。LRU 从链表头扫到链表尾就够了;LFU 要额外维护访问次数;ARC(IBM 发明)用两个 LRU 列表自动调权重。xv6 选 LRU 不是因为它最好,是因为它够用且能讲清。

Linux 对照:Linux page cache 默认是 LRU 改良版 — 分 active/inactive 两个链表,避免一次性把热点挤出去(vivification 问题)。

权衡 3:NBUF=30 这么小,实际能撑多大的 FS?

NBUF=30 表示同时缓存 30 个磁盘块(15 KB 内存)。xv6 FS 默认 1000 块,意味着任何时候至少 97% 的磁盘块不在缓存里。这显然不是性能设计,是为了把"缓存 vs 不缓存"的语义逼到极端 — 让学生看明白"buffer cache"是个抽象层,不是魔法。

权衡 4:为什么 bget 里"释放 spinlock"和"acquiresleep"分两步?为什么不在 spinlock 里面 acquiresleep?

因为 sleeplock 可能睡眠等磁盘 IO(虽然这里 bget 还没真读盘,但acquiresleep 内部会短暂让出 CPU 调度)。锁顺序原则:spinlock 永远在 sleeplock 之外;若反过来,sleeplock 内睡眠时还抱着 spinlock,其他 CPU 永远进不来,死锁。

权衡 5:为什么 bpin / bunpin 单独存在?

正常流程 bread→brelse,refcnt 自然增减。但日志层(Day 2)需要:log_write 时,buffer 内容已经在内存里改好,但还没 commit 写盘 — 这时 LRU 不能把它淘汰,否则丢了未提交的数据。bpin 就是"告诉我,这块别 LRU 走"。

Linux 对照:Linux 用 page->flags & PG_locked 或 try_to_unmap 反向保护,概念等价但更复杂。

📝 3 道自测(讲完后做,10 分钟)

  1. (选择) xv6 缓冲层中,为什么 bget 函数在第二个 for 循环里"释放 spinlock 之后"才 acquiresleep?

    A. 节省一次原子操作　B. 避免 spinlock 持锁时间过长导致其他 CPU 饥饿　C. sleeplock 内部会睡眠,违反"spinlock 内不能睡"原则　D. 没有特殊原因,只是 xv6 代码风格

    答案: C。这是锁顺序原则 — spinlock 永远在外面,sleeplock 在里面。bget 第 4 步释放 spinlock 才 acquiresleep,正是为了让 sleeplock 拿到时 spinlock 已经被释放。干扰项 A/B 听起来合理但不是真正原因;D 错。

  2. (简答) 如果 brelse 被某个路径忘记调用,会发生什么?

    答案: refcnt 永远不归零 → LRU 永不更新 → 30 个 buffer 全被钉住 → bget 找不到可淘汰的 buffer → panic("bget: no buffers")。所以"调用方必须调 brelse"是协议级约束,不是优化建议。内核用 holdingsleep 在 bwrite/brelse 入口检查就是防这个。

    3. <3>

    (追问) xv6 的 LRU 和 Linux 的 active/inactive LRU 相比,主要缺点是什么?

    答案: vivification(失活) — 一个热点 buffer(刚被用过)如果一次性读了大量冷数据,会被挤到 inactive 然后可能被淘汰,即使它本来是热点。Linux 用 active/inactive 两层链表缓解,冷数据先入 inactive 停留一会儿再淘汰。xv6 不解决,因为它假设 NBUF=30 极端情况下 vivification 也不是性能瓶颈。

❓ 老师追问预案(5 个最可能的)

老师可能问答法要点
"spinlock 和 sleeplock 怎么实现的?为什么 sleeplock 可以睡?" sleeplock 持锁时若被另一个 CPU 占用,本 CPU sleep(chan, lk) 释放自己的 spinlock 并阻塞;持锁方释放时 wakeup(chan)。spinlock 持锁时不让睡是因为自旋方没机会让出 CPU,持锁方释放不了。
"bget 第二个 for 循环用 prev 方向遍历,这是 O(N) 最坏情况;有没有 O(1) 的 LRU?" 有 — Linux 用 list_lru + per-node 锁,近似 O(1)。但 xv6 不优化,因为 NBUF=30、CPU 数 ≤ 8,O(N) 完全够。
"virtio 磁盘驱动怎么和缓冲层交互的?用了哪些 RISC-V 寄存器?" virtio_disk_rw(buf, write):把 buf 数据 DMA 到 virtio 设备的描述符环,写 VIRTIO_REG_QUEUE_NOTIFY MMIO 寄存器触发设备,等中断(PLIC 路由到 usertrap)。xv6 用 volatile uint32 * 直接读写 MMIO,这是 RISC-V 设备驱动典型模式。
"bwrite 调用时是同步还是异步?brelse 释放时数据真的落盘了吗?" 同步。virtio_disk_rw(b, 1) 等设备返回(等中断)才返回,brelse 时数据已落盘。Linux 的 submit_bh 是异步的,这是 xv6 简化。
"为什么 bwrite 不在写盘后清掉 valid 位?brelse 不清 dirty 位?" xv6 没 dirty 概念 — 因为写是同步的,brelse 时数据已落盘,不需要标记 dirty。Linux page cache 必须有 dirty 位因为写是异步的。
