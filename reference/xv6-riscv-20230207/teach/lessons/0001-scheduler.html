🧠 一句类比:酒店前台

xv6 调度器 = 只有一个前台的小酒店。
客人(进程)从大门进来到大堂等;前台(CPU) 一次只服务一个客人,办完一个叫下一个;
叫号单(proc[] 数组) 上写了"现在可办理"(RUNNABLE) 的房间号;
前台办完会把客人"交接"给对应的服务员(swtch 切到进程),办完了服务员再把客人送回大堂。

为什么要这个类比:你能用手指着"前台"、"叫号单"、"服务员"讲完整个 scheduler() 循环,不需要背源码。

📝 板书图(讲台上要画出来)

xv6 调度器 — 一张图讲完 ┌──────────────────────────────────────────┐ │ scheduler() — 一个死循环 │ │ │ │ for(;;) { │ │ intr_on(); ← 开中断(防死锁) │ │ │ │ for(p = proc; p < &proc[NPROC]; p++) { │ ← 扫花名册 │ acquire(&p->lock); │ │ if (p->state == RUNNABLE) { │ ← 找可办理 │ p->state = RUNNING; │ ← 改状态 │ c->proc = p; │ ← 记录"现在谁在跑" │ swtch(&c->context, &p->context); │ ← 切过去! │ c->proc = 0; │ ← 它回来后清空 │ } │ │ release(&p->lock); │ │ } │ │ } │ └──────────────────────────────────────────┘ 关键点(板书时圈出来): ① 改 RUNNABLE→RUNNING *在 swtch 之前* ② swtch 是汇编,只切 14 个寄存器 ③ 进程回来时自己改状态,scheduler 不动

🔍 源码精读(proc.c:445–472)

void
scheduler(void)
{
  struct proc *p;
  struct cpu *c = mycpu();

  c->proc = 0;
  for(;;){
    // Avoid deadlock by ensuring that devices can interrupt.
    intr_on();

    for(p = proc; p < &proc[NPROC]; p++) {
      acquire(&p->lock);
      if(p->state == RUNNABLE) {
        // Switch to chosen process.  It is the process's job
        // to release its lock and then reacquire it
        // before jumping back to us.
        p->state = RUNNING;
        c->proc = p;
        swtch(&c->context, &p->context);

        // Process is done running for now.
        // It should have changed its p->state before coming back.
        c->proc = 0;
      }
      release(&p->lock);
    }
  }
}

逐行讲(每行配一句口语)

  1. c->proc = 0 — "先把前台清空,等下要登记新客人"
  2. for(;;) — "调度器永远不死,死循环"
  3. intr_on() — "开中断,不然设备永远叫不醒我们,会死锁"
  4. for(p = proc; p < &proc[NPROC]; p++) — "扫整个花名册,挨个问"
  5. acquire(&p->lock) — "先把这位的房门锁了,别让别人动它的状态"
  6. if(p->state == RUNNABLE) — "只挑'可办理'的;睡眠的、僵尸的不理"
  7. p->state = RUNNING — "决定就要它,先把它的状态改成'正在被办'"
  8. c->proc = p — "在前台登记'现在办的是这位'"
  9. swtch(&c->context, &p->context) — ⭐ 全场重点 — "切过去!"保存前台的 14 个寄存器,加载它的 14 个寄存器
  10. c->proc = 0 — "它办完回来了,前台清空"
  11. release(&p->lock) — "开锁,继续问下一个人"

讲法提示:第 9 行 swtch 一定要停下来,用手指着它说"全场最神奇的一行,汇编代码,切 14 个寄存器,几行就完事"。

🧩 3 个设计意图(被追问时用)

意图 1:为什么用最简单的轮询?为什么不上 priority / 时间片?

xv6 是教学内核,调度器要"一眼能讲完"。Linux 的 CFS 有 200+ 行,vscore 数组、红黑树,5 分钟讲不完。 xv6 选择 O(N) 扫一遍,NPROC=64,64 次循环 < 1 微秒,够用且代码可读。

意图 2:为什么"改 RUNNABLE→RUNNING"要在 swtch 之前?

怕自己骗自己。如果 swtch 之后才改状态,刚切过去那一瞬间,调度器那边还可能"以为这个进程 RUNNABLE",下一次扫到它又切一次,就乱了。 所以改状态 = 表态,要在切之前就让全世界知道。

意图 3:为什么 swtch 后没有"恢复现场"的代码?

对称性。swtch 是双向切换:从 A 切到 B,以后 B 切回 A 时,程序计数器会自动回到 swtch 的下一行继续执行。所以调度器"恢复现场"的代码就是 c->proc = 0 那几行,不需要单独写"还原"逻辑。

📝 3 道自测(讲完后回头看,10 分钟)

  1. (选择) xv6 调度器在 swtch 之前,先做了哪件事?

    A. 保存当前进程的页表　B. 把进程状态改成 RUNNING　C. 关中断　D. 调用 forkret

    答案: B。p->state = RUNNING 必须在 swtch 前,见设计意图 2。干扰项 A 错在 xv6 用 trampoline 切页表,不在 scheduler;C 错在 intr_on 反而是开中断;D 是 fork 子进程第一次跑时的入口,不是 swtch 前调用。

  2. (简答) 为什么 scheduler() 里 intr_on() 在每轮循环开头?

    答案: 防止死锁。如果调度器一直关中断,磁盘/网络设备发中断时没人响应,进程会一直 sleep 等 IO,调度器也等不到任何进程变 RUNNABLE,死锁。每轮开一下,设备有机会唤醒睡眠进程,系统才能前进。

  3. (追问) xv6 调度器和 Linux 0.11 的调度器,本质区别是什么?

    答案: 时间片 vs 协作。Linux 0.11 是"非抢占"(协作式),进程必须主动调用 schedule() 才让出 CPU;xv6 是"抢占式",定时器中断会强制把运行进程切回调度器(走 yield() 路径)。xv6 更现代,但代价是调度器要更小心地处理"中断发生时正在干什么"。

❓ 老师追问预案(3 个最可能的)

老师可能问答法要点
"swtch 切的是哪些寄存器?为什么是这 14 个?" 看 proc.h:1–21:ra/sp/s0–s11 共 14 个。不切 tp/gp(线程局部)、不切 t0–t6(调用者保存,被调用方会自己保存)。口诀:被调用方保存的我们不切,调用方保存的我们切。
"如果 NPROC=64 全是 RUNNABLE,这循环公平吗?" 对当前这一遍公平(都跑一次);对多轮之间,谁先被扫到谁先跑,可能略不公平(扫到头部有偏倚)。xv6 不解决这个,Linux CFS 用红黑树 + 虚拟时间解决。
"yield() 和 sleep() 怎么回到 scheduler?" 都走 sched()(proc.c:482):把自己的状态改好(改 RUNNABLE 或 SLEEPING),然后 swtch(&p->context, &c->context)。scheduler 是从 swtch 下一行继续。
