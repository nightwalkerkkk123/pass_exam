从聊天到 messages：喂给大脑的到底是什么

上一课我们说，循环每转一圈都要"组装一份东西发给大脑"。这一课就把这份东西彻底摊开：它不是一句话，而是一摞贴了"谁在说话"标签的消息。看懂这一摞，你就看懂了 agent 的记忆是怎么装进去的。

上一课留了个悬念：心跳循环里那句"组装一份东西发给大脑"（buildConductorMessages），到底组装的是什么？我们只知道它不是一句话，而是一个"有结构的数组"。可这数组长什么样？里面装了几种东西？为什么每转一圈都要重新拼一遍？这一课全部讲清。

一、先打个比方：这不是聊天，是一份带署名的剧本

ELI5 时间。你可能以为，跟大模型说话就像发微信：你发一句，它回一句。但真相是——大模型没有记忆。它不像微信那样有个聊天记录存在服务器上。每次你问它，它都像第一次醒来、什么都不记得的人。

那它怎么"记得"刚才聊过什么？答案很朴素：你每次都把从头到尾的完整对话再抄一遍递给它。就像你去看一个失忆的老中医，每次进门都要把"上次你说我该怎样、我吃了药之后怎样"从头复述一遍，他才能接着看。

而这份"从头复述"的稿子，不是一段大白话，而是一份带署名的剧本：每一句话前面都标着"这句是谁说的"。这个"谁"，在代码里叫 role（角色）。

message（消息）
剧本里的一句话。它至少包含两样东西：role（谁说的）和内容（说了什么）。一条消息就是数组里的一个元素。
role（角色 / 署名）
标明这句话出自谁之口。它不是装饰——模型会根据 role 用完全不同的态度去读这句话：system 的话它当"最高指令"，user 的话它当"要满足的需求"，tool 的话它当"客观事实"。

二、四种署名（外加一个彩蛋）

经典的 agent 剧本里有四种署名，各司其职。用一场"帮我查今天汇率"的对话来认一遍：

逐个认清它们：

  • system（系统）——整场戏的"导演须知"，写在最前面：你是谁、能干什么、守什么规矩。模型把它当最高优先级。这是下一课的主角。
  • user（用户）——真人（她）提出的需求。
  • assistant（助手）——模型自己上一次的回复。注意：模型说过的话，下一轮要作为剧本的一部分再喂回给它，它才知道"我刚才说过要去查汇率"。
  • tool（工具）——工具跑完把结果写回来的那句话。它让模型"看见"上一步动手的成果——这正是上一课说的"工具结果变成下一轮输入"的闭环。
彩蛋·真实代码里其实有 5 种 role。mino_server 在 common.go 里定义了 assistant / user / system / tool / developer 五个常量。多出来的 developer（开发者）专门装"开发者悄悄塞给模型的背景资料"：比如今天几号、用户是谁、当前项目信息——就是图里第二条那个 <context_info> 块。你可以把它理解成导演在剧本边上贴的便签，观众（user）看不见，但演员（模型）能看到。下一课细讲它怎么来。

二·五、为什么"无状态"逼你每轮重发全部历史

这是最反直觉、也最重要的一点，慢慢来。

无状态（stateless）的意思是：模型服务器不替你保存任何对话记录。它没有一个叫"会话"的东西挂在那里。在 mino_server 里也一样——没有独立的会话对象，轮次（谁先说、谁后说）完全靠 messages 数组的先后顺序来表达。

后果就是：想让模型记得第 1 轮说过什么，第 3 轮请求时你必须把第 1、2 轮原封不动再抄进数组。对话越长，每次递过去的稿子越厚。

那"本轮新产生的消息"是怎么攒起来的？就是上一课见过的两个动作：模型每回一次话，AddMessage 把它存成一条 Role=common.Assistant；每跑完一个工具，AddToolResult 把结果存成一条 Role=common.Tool——都追加进 agent.newMessages。下一圈重拼数组时，它们就自动被读回来当上下文。这就是循环能滚下去的物理原因。

三、真实组装顺序：一摞是分段拼出来的

现在看真代码。buildConductorMessages 拼这摞消息不是随手一塞，而是写死的四段、每步都一样的顺序：

① system人格+规矩
挂 1 小时缓存
→
② 模式 developer 段当前激活模式的
行为约束
→
③ hook developer 段hook 注入的
参考文档
→
④ chatMessages上下文+历史+
本轮（会增长）

顺序来自 buildConductorMessages（agent.go）：system → buildModeDeveloperMessages() → hooks.BuildDeveloperMessages() → chatMessages。前三段稳定、放前面（绿=能命中缓存）；第四段每步增长、放后面（橙）。

为什么是这个死顺序？还记得第 01 课的"前缀缓存"吗——稳定的放前面，易变的放后面。前三段几乎每步都一模一样，正好命中缓存省钱；对话历史每轮变长，只能垫底。所以这不是随意的排版，是被计费规则逼出来的纪律。

那第四段 chatMessages 里面又是什么？它自己也有一套细分顺序（在 newConductorBasicMessages 里，context.go）：

context_infodeveloper 块
→
历史消息含旧轮 user/assistant/tool
→
本轮用户请求user
→
本轮新消息assistant+tool
末尾打缓存点
→
结尾提示还剩几步/上轮报错

（历史消息与本轮用户请求之间，还可能插入 hook 的 <system_hint>，默认为空。）

两个细节值得记住，都是真代码：

  • 那条 context_info（今天几号、用户是谁）由 buildDeveloperContextMessage（context.go）拼成一条 Role=common.Developer 消息，内部块严格按 space_info → user_identity → current_time → main_language → current_plan → user_personalization → user_memory → mobile_app 排列。顺序固定，同样是为了缓存稳定。
  • 连历史里真人说的话也会被 convertOriginalMessages 用 <user_request> 标签重新包一层——目的是让新旧消息格式完全一致，别让格式的小差异击穿缓存。
一个容易踩的误解，顺手澄清：你可能以为"能用哪些工具"也写在 messages 里。不是。在 mino_server 里，工具清单是通过 model.WithTools(toolInfos) 作为 API 的带外参数单独传给模型的，不进 messages 数组。messages 里的 system 只写工具的"使用规则"文字，真正的工具"说明书"走另一条道。这正是 chat 和 agent 的一处关键差异——第 05 课细讲。

四、关键定义

messages 数组
一次请求喂给模型的全部输入，一个按时间排好的消息列表。它同时承载了：规矩（system）、背景（developer）、需求（user）、模型自己的话（assistant）、工具结果（tool）。
无状态（stateless）
模型服务端不保存对话；轮次靠数组顺序表达。代价：每轮都要把完整历史重发一遍，数组只增不"续"。
稳定前缀（stable prefix）
数组里每步都不变的前几段（system + 两段 developer + 上下文 + 历史 + 用户请求）。刻意排在前面以命中前缀缓存；易变的（本轮新消息、步数提醒）垫在后面。
developer role
mino_server 的第 5 种角色，专门承载"开发者注入的背景资料"：上下文块、模式约束、结尾的 <system_hint> 提醒。user 看不见，模型看得见。

五、自测题

Q1：多轮对话里，为什么每次请求都要把之前所有轮次重发一遍？只发最新那句不行吗？

不行。因为模型是无状态的——服务端不替你存对话，mino_server 里也没有独立的"会话对象"。模型只能看到这一次请求 messages 数组里的东西。如果只发最新一句"那日元呢"，模型根本不知道你前面在聊汇率、更不知道刚查过美元和欧元，它会一脸茫然。轮次（谁先说谁后说）完全靠数组顺序表达，所以历史必须原样带上。代价就是数组只会越来越长——这也埋下了第 09 课"上下文爆炸"的雷。

Q2：工具跑完得到的结果，会以什么 role 出现在下一轮的 messages 里？由哪个函数放进去？

Role=common.Tool，由 AddToolResult（message.go）把结果包成一条 tool 消息、append 进 agent.newMessages。这一步是循环闭环的关键：工具结果变成下一轮 buildConductorMessages 的输入，模型才"看得见"自己上一步动手的成果，从而决定下一步做什么。对照一下：模型自己的回复则由 AddMessage 存成 Role=common.Assistant。两者都追加到同一个 newMessages 里。

Q3：为什么 system 必须放在数组最前面、而"还剩几步 / 上轮报错"这种提示要垫在最后面？

因为前缀缓存（第 01 课）：厂商按"从头开始的严格前缀"缓存已算过的部分，改动任何一段，它之后的缓存全部失效。system 几乎每步都一模一样，放最前面能被稳稳缓存、反复命中；而步数提醒、上轮报错这类东西每步都在变，一旦放前面就会把后面全部缓存击穿。所以 mino_server 把它们用 <system_hint> 包好、故意垫在整摞的最末尾（populateRunStepCount / populateLastErr，在 newConductorBasicMessages 收尾处）。一句话：稳定在前，易变在后，是刻进组装顺序里的纪律。

我们已经会拼这摞消息了。但整摞里最靠前、也最要命的那一条——system——我们一直只说它是"人格 + 规矩"，还没打开看过。它到底怎么写？为什么它几乎决定了这个 agent 是聪明还是笨、听话还是乱来？而且它还不是死的：里面塞着"今天几号、用户是谁"这些动态信息，这又是从哪来、怎么保证不击穿缓存的？下一课把 system 掀开。
