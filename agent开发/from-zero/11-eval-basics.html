怎么知道 Agent 没变笨：评估入门

你手里的 agent 已经会说、会做、会记了。可从今天起你还会不停地改它：改一句 prompt、换一个更便宜的模型、加一个新工具。每一次改动，它都可能悄悄变笨——而你不会立刻发现。这一课装上最后两个零件里的第一个：一套能自动回答"它到底有没有变笨"的评估（eval）系统。

上一课你给 agent 装上了长期记忆，它看起来相当完整了。但紧接着的问题很扎人：你今天往 prompt 里改了一句话、给记忆抽取换了个更小的模型，你怎么知道它没因此变笨？

服务端代码有 CI（持续集成，代码一提交就自动跑测试）替你把关每次改动。可 agent 的输出天生带随机性——同一个输入，模型每次说的话都不一样。传统单元测试那句"断言输出等于某个固定值"在这里当场失灵。那 agent 的"CI"到底长什么样？

一、为什么"断言相等"会死在这里

先用生活化的方式想（ELI5）。传统单元测试像判断题：标准答案是 4，你答 4 就对，答 5 就错，一刀切。这套逻辑对普通代码完美好用——add(2,2) 必须永远等于 4。

可 agent 的输出像作文。你问它"我上次说我喜欢喝什么？"，它这次答"你之前提到你喜欢喝美式咖啡"，下次答"根据我的记忆，你偏爱美式"。两句话意思一样、字面完全不同。要是拿"逐字相等"去判分，两次都会判错——但它其实两次都答对了。

所以 agent 的评估必须换一套判分哲学：不是确定性的（对/错二选一），而是统计性的（打一个 0 到 1 之间的分，跑一批用例看整体过没过）。这就是 eval——你可以把它理解成"agent 的新 CI"：改完 prompt / 换完模型，跑一遍 eval，分数没塌，才敢上线。

二、三维判分：从三个角度同时看一次回复

既然不能"逐字相等"，那就换成从三个互补的角度给同一次回复打分。mino_server 的 cmd/memory_eval/eval/scorer.go 里，score 函数正是这么做的，它填出 scoreDetail 的三个分数字段（外加一个评语字段 JudgeReasoning）：

① 关键词命中（ContainsScore）
最"死板"的一维：回复里该出现的词出现了没。用例可以要求 ContainsAll（按命中比例给分（命中数 ÷ 关键词总数），全部命中才拿满分）、ContainsAny（命中一个即可）、NotContains（一旦出现就判 0）。它便宜、确定、不花钱，专抓"事实有没有说到"。
② 工具调对没（ToolCallScore）
专门盯"记忆类"行为：该存的时候有没有真的调用写文件的工具（write_file / edit_file，甚至"更新记忆""追加信息"这类中文任务名，见 isWriteTool）；该拒绝的时候有没有忍住别乱写。它检查的是"做了没"，而不是"说得好不好"。
③ 裁判打质量分（LLMJudgeScore）
最"聪明"也最贵的一维：另请一个大模型当阅卷老师，读完被测回复给一个 0.0–1.0 的质量分，外加一句 reasoning（评语）。它能判前两维判不了的东西——回答自不自然、有没有幻觉、有没有礼貌地承认"我不知道"。

三维互补：关键词管"说没说到点上"，工具管"手脚有没有动对",裁判管"整体答得好不好"。单看任何一维都会漏，合起来才是一次靠谱的评价。

三、按 case 类别加权：不同题型看重不同的东西

可三维不该一视同仁地相加。想想真实场景：

  • 问它"我上次说喜欢喝什么"（recall，回忆题）——重点是答得对不对、好不好，你根本不在乎它调没调工具。
  • 让它"记住我明天要开会"（store，存储题）——重点是它有没有真的把工具调起来把这句话写进硬盘，光嘴上答应"好的记下了"却没写，等于没记。
  • 问它一个记忆里根本没有的事（rejection，拒答题）——重点是它老不老实、会不会瞎编，这几乎全看裁判。

所以 scorer.go 的 weightedScore 按用例的 category 给三维配了不同权重（下面每行权重相加都是 1.0）：

类别 category加权公式这一题最看重
recall 回忆关键词 × 0.3 ＋ 裁判 × 0.7答得准不准（工具无关）
store 存储工具 × 0.5 ＋ 裁判 × 0.5手脚动没动 + 存得好不好，各占一半
update 更新工具 × 0.4 ＋ 裁判 × 0.6改对没 + 改得好不好
rejection 拒答关键词 × 0.2 ＋ 裁判 × 0.8诚不诚实（几乎全看裁判）
default 其它（三维之和）÷ 3没指定类别时三者取平均

拿一道 recall 题走一遍：假设关键词全命中 ContainsScore = 1.0，裁判觉得答得还行给了 0.6。加权总分 ＝ 1.0 × 0.3 ＋ 0.6 × 0.7 ＝ 0.30 ＋ 0.42 ＝ 0.72。runner.go 的 runAll 里，只要总分 不小于 0.6 就判 PASS——0.72 ≥ 0.6，这题过关。

四、裁判为什么要"跨厂商"——考生不能自己改自己的卷

第三维的裁判是个大模型。这里藏着一个很容易忽略、却致命的坑：要是让被测的模型自己给自己打分，会怎样？

ELI14：这就像让考生自己批改自己的作文。人会天然偏袒自己写的东西，模型也一样——研究里这叫自评偏袒（self-preference bias），模型倾向于给"和自己风格相近的输出"打高分。你辛辛苦苦搭 eval 就是为了得到一个诚实的信号，结果裁判是自己人，信号就废了。

mino_server 的解法很干脆：让裁判和考生来自不同厂商。看 cmd/memory_eval/eval/config.go 的 parseFlags 里三个默认值：

  • 被测模型 -model 默认 claude-sonnet-4-5——Anthropic 的模型，经内部的 AnyClaw Gateway 跑。
  • 裁判模型 -judge-model 默认 google/gemini-2.5-flash——Google 的模型。
  • 裁判走的接口 -base-url 默认 https://openrouter.ai/api/v1——OpenRouter（一个能统一调各家模型的中转站）。

被测是 Claude、裁判是 Gemini，一个 Anthropic 一个 Google，厂商不同，自评偏袒无从谈起。

五、裁判本身也会错——所以要留兜底

关键的清醒认识：裁判也是个会出错、会掉线、会胡言乱语的大模型。它可能没配 API key、可能 HTTP 请求超时、可能返回一坨不合法的 JSON 让你解析失败。如果这些情况直接判 0 分，那就是拿裁判的故障去冤枉被测的 agent——明明 agent 答得好好的，只因为阅卷老师那边网断了就记零分，这不公平。

所以 scorer.go 的 llmJudge 分了几档处理：

  • 没配 API key → 返回中间分 0.5（"这题我没法评，先给个不偏不倚的分"）。
  • 裁判请求出错 / HTTP 非 200 / 返回的 JSON 解析不出来 → 同样兜底 0.5。
  • 但如果是被测调用本身就报错了（result.Error 非空，即 agent 那头压根没跑通）→ 这不是裁判的锅，直接 0 分，绝不能兜底。

顺带一个防失控的小细节：裁判请求的 MaxTokens 上限是 1024——它只需要吐一个分数加一句评语，不需要长篇大论，卡住上限既省钱又快。

本课带走的一句话：eval 是 agent 的新 CI，但判分是统计性的——三维（关键词 / 工具 / 裁判）分角度看，按题型加权成一个总分，拿 0.6 卡通过；裁判要跨厂商防自评偏袒，裁判自己出故障还要兜底 0.5，唯独被测自己崩了才判 0。

六、这只是入门

你现在看到的是 eval 最朴素的样子：一个独立的小程序（cmd/memory_eval 是 package main，靠 run.sh 里 go build -o ./output/memory_eval ./cmd/memory_eval/ 单独编译出来跑，不是 go test），扫一批用例、打分、出报告。真正成体系的评估还有更多讲究——数据集怎么建、指标怎么设计、怎么防止 eval 自己也在骗你。那是进阶篇"评估支柱"的内容，这一课只是通往那里的一座小桥。

七、自测题

Q1：一道 store（存储）用例里，agent 嘴上答"好的，我已经帮你记下来了！"，但整个回复没有任何工具调用。裁判被这句漂亮话骗到，给了 0.9 的质量分。这道题最后大概率 PASS 还是 FAIL？为什么？

大概率 FAIL。 store 类别的加权是 ToolCallScore × 0.5 + LLMJudgeScore × 0.5。这题该调写文件工具却一个都没调，ToolCallScore 判 0。就算裁判给到满分：0 × 0.5 + 0.9 × 0.5 = 0.45，低于 0.6 阈值 → FAIL。

这正是"三维互补 + 按类别加权"的价值：store 题把一半权重压在工具到底调没调上，光靠嘴甜骗不过去。裁判会被话术带偏，但工具维不会——它只认"手脚有没有真的动"。

Q2：你半夜跑 eval，OpenRouter 那边网络抖动，裁判请求全部超时。结果一批 recall 用例分数普遍是 0.5 上下。这是不是说明 agent 变笨了？该怎么判断？

几乎可以肯定不是 agent 变笨，而是裁判掉线了。 scorer.go 的 llmJudge 在裁判 HTTP 出错时会兜底返回 0.5。recall 加权是 contains × 0.3 + judge × 0.7，裁判维被钉死在 0.5，总分自然被拖到 0.5 附近——这是裁判故障的特征信号，不是被测的真实水平。

怎么判断：看结果里裁判的 reasoning 字段——兜底时它写的是 "judge HTTP error…" 这类故障说明而不是真评语。正确做法是网络恢复后重跑，别拿这批被污染的分数下结论。这也正是"裁判本身会错，所以要留兜底"的现实意义：兜底分保护 agent 不被裁判的故障冤枉，但你要能识别出这是兜底、而非真实评价。

好了，有了 eval 这道自动关卡，agent 的质量能守住了——改 prompt、换模型、加工具，跑一遍就知道有没有变笨。

可是你有没有算过这笔账？每一轮对话都要把全部历史重新发给模型、把所有工具定义带上、还要维护缓存，现在又要额外跑一整套 eval……这一切都在烧钱、烧时间。token 悄悄成了这台机器的新计费单位。下一课就来算清楚：一次 agent 交互到底花了多少 token，钱都烧在了哪儿。

本课收束：eval ≈ agent 的新 CI，但判分是统计性的——三维（ContainsScore / ToolCallScore / LLMJudgeScore）按题型在 weightedScore 里加权，runAll 拿 0.6 卡 PASS/FAIL；裁判跨厂商（Claude 被测 vs Gemini 裁判经 OpenRouter）防自评偏袒，裁判出故障兜底 0.5，被测自己报错才判 0。下一课：12 Token 经济学：一次交互到底烧多少钱。返回 课程门户。
