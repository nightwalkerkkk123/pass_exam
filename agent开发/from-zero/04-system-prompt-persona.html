System Prompt：给 Agent 一个人格与规矩

上一课你知道了：喂给大脑的是一个 messages 数组，而数组的第一条总是 role 为 system 的消息。这一课只讲清一件事——这第一条消息到底写了什么，凭它就能把同一个模型捏成完全不同的"人"。

上一课收尾时我们留了个钩子：消息最前面永远是 system。它凭什么能让同一个模型时而是严谨助手、时而是段子手？模型的参数一个字节都没变，换的只是这段开头的文字。它到底有什么魔力？这一课我们把它拆开看。

一、先打个比方：system prompt 是演员的"剧本 + 岗前培训手册"

想象一个很厉害的话剧演员（这就是大模型）。他演技一流，但本身没有角色——你给他一份《哈姆雷特》剧本，他就是忧郁王子；给他一份脱口秀稿子，他就是段子手。演员没换人，换的是他开场前读的那份东西。

system prompt 就是这份"开场前必读"。它不参与对话，用户通常也看不见它，但它在每一句回答之前都先被模型读一遍。更贴切一点，它其实是一份岗前培训手册，把四件事一次性交代清楚：

所以"时而严谨助手、时而段子手"根本不是玄学。模型是同一个，换的只是它开场前读的那份手册。把手册第一句从"你是一位一丝不苟的法律顾问"换成"你是一个爱抖包袱的段子手"，同一个模型的输出风格立刻两样。

二、真实代码里，这份手册长什么样

在 mino_server 里，这份"手册"不是硬编码在某个字符串里，而是一批模板文件。它们放在 biz/domain/dmservice/agents/conductorservice/prompts/ 目录下，文件名以 .go.tmpl 结尾，程序启动时用 Go 的 go:embed 把它们打包进二进制。不同的模式用不同的模板——这就是"多副人格"的来源：

go:embed 模板（多副人格）
同一个 Conductor Agent 备了 8 套以上人设模板，按当前 SubType/模式挑一套渲染：system_super_agent（通用超级助手）、system_designer（设计）、system_developer（写代码）、system_html_slides（做幻灯片）、system_code_interpreter、system_anyclaw/_wiki、system_testing ……换模式就是换人格。prompts.go 顶部有一排逐文件的 //go:embed 指令——system_super_agent / system_developer / system_designer … 各占一行，每条 embed 一个 .go.tmpl（真实例子：//go:embed prompts/system_super_agent.go.tmpl），共 8 个 system_*.go.tmpl 模板。

以最常用的 system_super_agent.go.tmpl 为例，它开头第一句就是身份声明（真实原文）：

You are AnyGen, an AI agent created by the AnyGen team.

<core_settings>      <!-- 能力清单、工作语言、Ubuntu 沙箱环境说明 -->
<mcp_integration>    <!-- 外部工具接入规则 -->
<agent_principles>   <!-- 工具使用规则 + 先计划再动手 + 出错怎么办 -->
<user_interaction>   <!-- 怎么回复用户、什么时候找人确认(HITL) -->
<execution_rules>    <!-- 技能库、搜索/浏览器/shell/文件的具体规矩 -->
<deliverable_standards> <!-- 交付物长啥样 -->

注意一个反直觉的点：模板里写的是工具的"使用规矩"（文字描述），而不是工具的技术说明书（参数 schema）。真正告诉模型"这个工具接收哪些字段"的东西是带外传的（下一课细讲）。这里只讲"你应该怎么用它们、什么时候用"。

模板里那些 <agent_principles> 之类的段落，也不是一整块写死的。渲染之前，代码先分别生成一堆子段落——计划规则、回复规则、找人确认(HITL)规则、搜索规则、记忆规则、安全规则等等——再连同一大堆"功能开关"塞进一个 SystemPromptData 结构，最后用模板拼成完整的一段。哪些段落出现、哪些不出现，由功能开关决定。这段拼装逻辑在 prompts.go 的 populateSystemPrompt 里。

关键判断（本课的第一根钉子）：人设不是"随手写的一句咒语"，而是一份可版本化的工程资产——它是磁盘上的模板文件、有目录结构、按模式分文件、能进 git、能 code review、能针对性改某一段。想改 agent 的行为，你改的是这份文件，而不是去动模型。

三、静态人设 + 动态 context：两股水汇进同一条河

但只有一份固定的模板还不够。模板是"死"的——它不知道现在几点、跟它说话的是谁、这个用户之前告诉过它什么。这些是每次请求都在变的信息。于是系统在运行时另外拼一段动态 context，注入进消息数组。

用招待所打个比方：模板是墙上那张永远不变的《住客须知》（静态人设）；而前台每天早上更新的那块小黑板——"今天是 X 月 X 日，接待的客人是老王，他不吃辣"——就是动态 context。两者你都要读，但一个几年不动，一个天天换。

这里冒出一个新角色 developer。上一课我们只见过 system/user/assistant/tool 四种 role——但 mino_server 其实用了五种：

developer 角色
第五种 role（common.go 里的 RoleType 常量共 5 个：assistant/user/system/tool/developer）。它专门承载"开发者在运行时注入"的内容：结构化上下文块、模式约束、步数提醒等。可以理解成：system 是不变的人格，developer 是同样由系统方（而非用户）塞进去、但每次可以不一样的补充说明。

那块动态 context，最终长成一条 developer 消息，外层用 <context_info> 包住，里面按固定顺序写入若干小块。顺序是写死的（这样才能被 prompt cache 稳定命中）：

space_info项目/技能
→
user_identity用户是谁
→
current_time只到日期+时区
→
main_language主语言
→
current_plan当前计划
→
user_personalization昵称/职业
→
user_memory长期记忆
→
mobile_app端信息

这些块的数据从哪来？在 agent.go 的 buildContextPayload 里一项项采集：时间取 agent.startTime、主语言取 agentFeatures.MainLanguage、用户指令取 agentFeatures.ConductorInstruction、记忆和个性化各有对应的 populate* 函数……采集完交给 buildDeveloperContextMessage 拼字符串。如果一样数据都没有（整个 context_info 是空标签），这条消息直接返回 nil，压根不注入——不浪费一个 token。

另外，</context_info> 闭合之后，同一条 developer 消息里还会追加用户自定义指令区（<user_instruction> 和 <anygen_md>，注明这些来自 AnyGen 控制台）以及子代理场景的 <subagent_mode>。也就是说：官方人设在 system 消息里，用户/管理员的定制在 developer 消息里——两层分开，互不污染。

四、为什么这件事重要

把上面三段串起来，你会发现人设这件事被 mino_server 拆成了三层，一层比一层稳定：

层是什么变化频率落在哪
静态人格身份/能力/规则/格式，按模式选模板几乎不变（改代码才变）system 消息 · prompts/*.go.tmpl
用户覆盖用户在控制台自定义的整段 system prompt按用户设置GetSystemPromptByAgent 整段替换
动态 context时间/用户/计划/记忆等运行时事实每次请求都可能变developer 消息 · <context_info>

这种"稳定的在前、易变的在后"的分层，不只是好看——它是省钱的地基。因为 system 消息挂着 1 小时缓存点（NewLLMMessageWithCache1Hour），只要它一个字节不变，下次请求这段就能命中缓存、按输入价的一小部分计费。反过来，如果你手贱把"当前秒级时间"写进 system 开头，每次请求这段都不同，缓存全线击穿，账单直接翻量级。这也正是 current_time 被刻意降到日期粒度、且被放进靠后的 developer 消息而不是 system 里的根本原因。（缓存的钱怎么算，是本课程后面成本那一课的主题。）

五、自测题

Q1：产品同学说"我想让 agent 每次都知道现在精确到秒的时间"。你把 2026-07-15 14:32:07 写进了 system_super_agent.go.tmpl 的第一行。这会出什么问题？mino_server 实际是怎么做的？

问题：system 消息挂着 1 小时缓存点。你把秒级时间写进它的开头，意味着每次请求这条 system 消息都不一样，缓存永远命中不了；而且前缀缓存是严格前缀匹配，system 一变，排在它后面的模式段、context 段的缓存也跟着全部失效。相当于每次都按全价重算整段前缀，成本量级上涨。

mino_server 的做法：时间根本不放 system，而是放进靠后的 developer 消息的 <current_time> 块；而且只保留到"日期 + 时区"粒度（StartTime.Format("2006-01-02")），代码注释写明是"避免秒级精度导致跨 session prompt cache miss"。原则一句话：越易变的东西，越要么降精度、要么往后放。真需要精确时间，让 agent 调一个"取当前时间"的工具就行——这正好引出下一课。

Q2：你想让同一个模型在"写代码"和"做幻灯片"两种场景下表现得完全不同。以下哪种做法符合 mino_server 的设计？(A) 训练两个不同的模型   (B) 在一条 user 消息里写"请你现在扮演程序员"   (C) 按模式选不同的 system_*.go.tmpl 模板渲染出不同的 system prompt   (D) 把两套规则都塞进同一个 system prompt，让模型自己挑

正确答案是 (C)。mino_server 备了 8 套以上模板（system_developer、system_html_slides 等），启动时用 go:embed 打包，运行时按 SubType/模式挑一套渲染。这就是"多副人格"的工程实现。

为什么其它不行：(A) 模型是同一个、也很贵，为换风格重训一个模型是巨大浪费——人格本就该用文字（system prompt）解决，不该动模型。(B) 把人设写进 user 消息，一是会被后续对话冲淡、二是位置靠后注意力弱、三是用户能覆盖它，不稳。(D) 把两套规则混在一起，既让 system 变长（多花 token、稀释注意力），又给模型制造"我到底该听哪套"的歧义。分文件、按需渲染，才是"可版本化的工程资产"的正确姿势。

现在这台 agent 有了稳定的人格，也知道了"现在几点、面对的是谁"。但请注意：到这一步，它会说话，却还是个只会说话的嘴——system prompt 里写满了"你应该怎么用工具"的规矩，可它手上其实一个真工具都没有。它能说"我建议你运行这条命令"，却没法自己去运行。

那么：怎么让这颗大脑真的"请求"调用一个真实的函数、我们再把函数的执行结果还给它，让它接着往下想？这就是下一课要装的零件——给大脑装上手脚。
