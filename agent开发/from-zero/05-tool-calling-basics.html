Tool Calling：给大脑装上手脚

上一课，我们给 agent 装上了人格和规矩。可它还是一个"只会动嘴"的大脑。这一课只做一件事：让它能真的去查天气、发消息、跑代码——但你会看到一个反直觉的真相：模型自己从头到尾没碰过任何工具，真正动手的一直是我们的代码。

上一课 agent 会说话了，也知道自己"是谁、该守什么规矩"。但它被关在一个没有网线、没有手机、没有键盘的房间里：问它"北京今天几度"，它只能凭记忆瞎猜；让它"给张三发条消息"，它一个字也发不出去。

大脑本身不能联网、不能动手——怎么办？

一、先打个比方：隔着玻璃罩的天才

想象一个绝顶聪明的人（这就是大模型），被关在一个封闭的玻璃罩里。他能读你递进去的纸条，也能写纸条递出来。但他够不到罩子外面的任何东西——没有网络、没有电话、没有文件。

你想让他帮你查天气，他没法自己查。他能做的只有一件事：写一张纸条递出来，上面写着"请帮我查一下：北京今天天气（工具：search，参数 query=北京天气）"。然后你——罩子外面的人——照着纸条真的去查，把结果写成另一张纸条递回去。他读到结果，才能接着往下想。

这就是 tool calling（工具调用）的全部秘密：模型只会"请求"调用一个工具，它输出的只是一段文字（一张纸条）；真正去执行那个工具的，永远是我们罩子外面的后端代码。模型碰不到网络、碰不到函数、碰不到文件——这个边界是理解整门课的关键。很多人第一次会误以为"模型自己上网了"，其实没有。

那"纸条"长什么样、怎么递、递给谁、结果又怎么塞回去？这就是一次完整的"一来一回"。

三、落到 mino_server：注册 → 注入 → 执行 → 回注

在真实的 Conductor Agent 里，"给大脑装手脚"被拆成四个清清楚楚的环节。我们顺着走一遍：

① 注册：一张写着所有"手脚"的大表

agent 能用的全部工具，先登记在一张表里。位置是 biz/domain/dmservice/agents/conductorservice/tools.go 的 GetAllToolServices：它返回一个大 map，把每一类工具服务（浏览器、代码执行、搜索、画图、发消息、子代理……）都挂进去，共 54 个在用的工具服务（还有一个 browser_sub_agent 被注释掉了）。

注意用词是"工具服务"，不是"工具"。一个服务可以吐出 0 到多个具体工具——真正决定"这次用户能用哪些工具"的，是两道过滤：先 hooks.GetToolServices 按当前场景/开关从 54 个里挑出要用的那批，再 service.AsTools 把每个服务展开成一个个具体工具。所以工具集是按人、按场景现拼的，不是所有人都看到全部 54 个。

② 转 schema：把"手脚"翻译成模型看得懂的说明书

模型不认识 Go 的函数，它只认 JSON。所以每个工具的参数要先翻译成一份"说明书"，告诉模型"这个工具接收哪些字段、什么类型、哪个必填"。

在 mino_server 里，工具参数用 eino 的 ParamsOneOf 描述，要发给模型时调 ToOpenAPIV3() 序列化成 OpenAPI v3 的 JSON schema（见 tools.go 里的 tool.ToolInfo.ParamsOneOf.ToOpenAPIV3()）。这份 schema 就是玻璃罩里的天才唯一能读到的"这个工具怎么用"的信息——记住这句话，下一课整节都在讲它。

schema（工具参数说明书）
用 JSON 写的"这个工具接收哪些字段"的规格。模型靠它决定该往参数里填什么。mino_server 用 ToOpenAPIV3() 把内部的 ParamsOneOf 转成 OpenAPI v3 格式的 schema。
tool call（工具调用请求）
模型输出的一段结构化文字，形如"调用 search，参数 {query:'北京天气'}"。它只是请求，不是执行；在代码里表现为模型回复消息上的 ToolCalls 字段。

③ 注入：每一轮都把工具清单重新挂到请求上

工具装好了，怎么让模型这一轮"看见"它们？在每次请求模型前，agent.go 的 getCurrentToolInfos() 从 agent.toolMap 里取出当前所有工具，按工具名字典序排好（顺序稳定才利于命中缓存），再用 model.WithTools(toolInfos) 一次性挂进这一轮请求的 options。

这里藏着 chat 和 agent 的一个关键区别：工具定义不写在对话消息（messages）里，而是作为一个"带外"的 API 参数（model.WithTools）单独传给模型。你在上一课看到的 system prompt 里只有工具的"使用规则"文字，真正的工具 schema 走的是 tools 这条独立通道。这也意味着：工具越多，每一轮请求都要把全部工具定义重新带一遍——这是固定的 token 底盘成本。

顺带一提，这一轮请求还显式打开了 WithEnableParallelToolCalls(true)：允许模型一次纸条上写多个工具，让后端并发执行（并发怎么控、上限多少，是第 07 课的事）。挂好工具后，streamCallLLM 里的 agent.llmClient.Stream(ctx, messages, options...) 才真正把请求发出去——这是 Conductor 主循环里真正把请求发给大模型的地方（各子 agent、摘要/压缩等模块另有各自的 LLM 调用点）。

④ 执行：真正动手的一步（也是本课的题眼）

模型回复回来了。runConductorStep 先看它的回复里有没有工具调用：如果 len(ToolCalls) == 0，说明模型不想再调工具了，就把 agent.done = true 收工；如果有，就把这些工具调用分组、过策略，交给 toolcall.go 的 executeToolCallGroups。

顺着 executeToolCallGroups → executeToolCall 走到底，你会看到这一行——工具真正被跑起来的地方：

// toolcall.go: executeToolCall
callSerial, callParallel, callStreaming := toolInfo.CallTool.Unpack()
// ... 在 goroutine 里调用其中之一，真正执行这个工具

请把这一步和第②步对照着看：模型在②里说"我要调 search"，我们在④里做了 search。说和做，分属玻璃罩内外两个世界。模型永远只递纸条，动手的永远是这段后端代码。

⑤ 回注：让结果变成下一轮的输入，循环才能滚下去

工具跑完拿到返回值，message.go 的 AddToolResult 把它包成一条 Role = common.Tool 的消息，append 进 agent.newMessages：

// message.go: AddToolResult
toolMessage := &minomessage.Message{
    Role: common.Tool,
    Contents: []minomessage.MessageContent{ messageContent },
}
agent.newMessages = append(agent.newMessages, toolMessage)

这一步是整个闭环的关键：下一步再拼消息（buildConductorMessages）时，这条 tool 消息就在历史里，模型于是"看见"了上一轮工具返回了什么，据此接着想。工具结果变成了下一轮的输入——这正是第 02 课那个"心跳循环"能一圈圈转下去的燃料。

四、自测题

Q1：有人说"agent 自己上网查了天气"。基于这节课，这句话哪里不准确？谁才真正联了网？

不准确在"自己"。模型（玻璃罩里的天才）从头到尾没碰过网络——它只是输出了一段文字（ToolCalls：请求调用 search）。真正发起网络请求、拿到天气数据的，是我们后端在 executeToolCall 里跑的那段代码（toolInfo.CallTool）。

为什么这个区分重要？因为它决定了"安全边界"和"能力边界"都握在我们手里：模型能做什么，取决于我们在那张大表（GetAllToolServices）里给了它哪些工具、以及我们的执行代码愿意替它做什么。模型请求调一个不存在的工具，也只会被我们判为无效、退回去让它改——它无法绕过我们直接触碰世界。

Q2：工具定义为什么走 model.WithTools 这条"带外"通道，而不是像普通对话那样写进 messages 数组？漏掉这一步会怎样？

因为工具清单在概念上不是"对话内容"，而是"这一轮你可以调用的能力表"，属于请求的元信息。mino_server 每轮用 getCurrentToolInfos() 取出全部工具、排好序，再 model.WithTools(toolInfos) 挂上去（agent.go）。

如果不挂（或忘了挂），模型这一轮就不知道自己有任何工具可用，只能凭记忆瞎答——退化回上一课那个"只会动嘴的大脑"。这也解释了为什么每轮都要重挂一遍：工具定义不像 system prompt 那样一次写死在对话里，它是每次请求独立带上的参数。

Q3（想清楚"为什么"）：为什么 getCurrentToolInfos 要专门把工具"按名字排序"？乱序会有什么坏处？

为了让请求前缀稳定，从而命中前缀缓存。工具定义在每轮请求里位置靠前，一旦它们的排列顺序在不同轮之间抖动，缓存就会从分歧点起失效，成本和延迟都上去。排序保证"同一批工具永远以同一顺序出现"，这段前缀就能被缓存反复命中。

这是"稳定在前、易变在后"这条原则的一个具体落点——它把"成本"和"工具怎么组织"焊在了一起。（前缀缓存的完整机制是后面成本课的内容，这里先埋个种子。）

现在 agent 有手有脚了。但回头看第②步——模型是怎么决定该调哪个工具、往参数里填什么的？它手上只有一份东西：每个工具的文字描述和参数 schema（第②步转出来的那份说明书）。

换句话说，工具描述写得好不好，直接决定模型调得对不对：描述含糊，它就调错工具、传错参数，整个 agent 就"变笨"了。所以工具描述其实是 prompt 的一部分——这就是下一课要拆的东西。

一句话收束：tool calling 是一来一回的纸条游戏——模型只负责"请求"（ToolCalls），执行永远在我们手里（executeToolCall），结果回注（AddToolResult）后循环才滚得下去。下一课《工具描述就是 Prompt》：为什么一句描述没写好，agent 就会调错工具、变笨。读到卡壳，回会话里问你的 AI 老师，可要求 ELI5 或看真实代码。 · ← 返回课程门户
