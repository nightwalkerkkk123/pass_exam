从零到生产：你亲手装完的这台 Agent

前面 12 课，我们从"只会聊天的大脑"开始，一个零件一个零件地把它装成了一台能自己干活、扛得住长对话、还负担得起的生产级 agent。这最后一课不装新零件，只做一件事：退后一步，把整台机器看一遍——每个零件当初是被什么问题逼出来的，串起来就是这门课真正想让你带走的东西。

12 个零件都装上了：会说（messages）、会做（工具）、不失控（编排 + 可靠性）、记得住（压缩 + 记忆）、可信又可负担（评估 + 成本）。可现在把它们摆在一起看，一个更根本的问题浮出来了——

这些零件不是一份功能清单，而是一条问题链：每一个的出现，都是为了填上一个零件留下的坑。如果说不清"这一层是被下一层的什么痛逼出来的"，你就只是背了 12 个名词，而没有真正理解 agent 是怎么长出来的。这一课，我们把这条链彻底看清。

一、先站远一点：这台机器是"长"出来的，不是"设计"出来的

先用最生活化的方式想（ELI5）。假设你要拼一辆自行车：你不会一上来就照着成品图纸把零件全焊死。你会先有两个轮子和一个车架——它能滚，但你得用脚蹬地（"只会聊天的大脑"）。滚起来才发现：得有脚踏板（循环），有了脚踏板发现坐着累，得有座椅（人格与规矩）……每加一个零件，都是因为上一个状态暴露了一个新的痛点。

mino_server 里这个真实的 agent 就是这么长出来的。它的心脏是一个叫 Conductor（指挥官）的东西——一个无状态的大模型，外面套着一圈会不断"再走一步"的循环。这门课的 13 节，本质上就是这圈循环的施工顺序。下面这张图，是我们从第 01 课到第 12 课一路盖起来的塔。请你从最底下一层往上读，每读一层，都在心里补一句"因为下面那层的痛，所以有了这一层"。

二、把塔读成一句话：问题链

塔是空间的看法。把它拉直成时间，就是一条环环相扣的问题链——这才是这门课真正的脊椎。你可以试着不看下面这段，自己先复述一遍；卡住的地方，就是还没打通的关节。

一口气读完的问题链：

大模型只会聊天 → 给它套个外部循环让它能"再走一步"（第02课，streamChat 的 step 循环，上限 maxStepCount=100）→ 可它无状态、记不住，于是每步都得重新组装 messages（第03课，buildConductorMessages）→ 这包消息最前面得先立人格与规矩（第04课，prompts.go）→ 光会说不够，得装上工具的完整往返（第05课，tools.go 约 55 个工具服务）→ 可工具装了模型不会用，因为工具描述本身就是 prompt（第06课，ToOpenAPIV3，还得按字典序稳定注入）→ 会用工具又怕它停不下来 / 太慢，于是要编排：何时停、能否并行（第07课，maxParallelToolCount=5）→ 模型还是个不可靠的调用方，得靠分层超时、坏 JSON 回注自愈、多 provider 兜底（第08课，toolcall.go；通用供应商兜底在 LLM 客户端层的多 provider 重试 AnthropicDirect→GCP；tryModelFallback 则仅是 slides 编辑专用的限流降级 Gemini 3.1 Pro→Gemini 3 Flash）→ 干得久了 context 会爆，得主动压缩（第09课，summary.go）→ 压缩有损会失忆，得把重要的事落盘成长期记忆（第10课，memory_flush.go）→ 改了这么多东西怎么知道没变笨？得靠统计性评估守（第11课，memory_eval）→ 最后，这一切每轮都在花钱，token 就是新的钱，前缀缓存把成本和 context 排列焊死（第12课）。

看清这条链，你就拿到了这门课最贵的那个判断：agent 工程里几乎每一个"零件"，都不是凭空的最佳实践，而是上一步的痛留下的补丁。 当你以后读任何一个陌生的 agent 框架，就该反过来问："它这层是在补什么痛？"——这比记住 API 有用一百倍。

三、经验清单：每条经验，绑一个真实机制

下面这张表，是把整门课压成 8 条"以后能直接用"的经验。左边是经验，中间是为什么，右边钉死在 mino_server 的真实文件与符号上——因为脱离了机制的经验只是鸡汤，绑上机制才是你能复现的手艺。

一句话经验为什么成立绑定的真实机制（文件 · 符号）
无状态大脑 + 外部循环 = agent 大脑本身不会连续做事，"能自己干活"来自外面那圈"再走一步"的循环，不是来自模型更聪明 conductorservice/agent.go · streamChat 的 step for 循环 · defaultMaxStepCount = 100
每轮都重发完整历史与工具定义 → 成本与"缓存 + context 排列"强耦合 模型无状态，每次调用都得把一切带过去；于是内容放哪、顺序如何，直接决定缓存命中与账单 agent.go · getCurrentToolInfos（按工具名字典序，保持稳定顺序）＋ context.go 里的 <current_time>
工具描述就是 prompt 模型只能看到工具的名字 + 描述 + 参数 schema，写不好它就不会用 / 乱用——工具治理其实是文字治理 工具 schema 经 ToOpenAPIV3() 转成模型能读的定义（llms/claude/convert.go 等处调用）
把模型当不可靠的调用方：分层超时 / 并发上限 / 坏 JSON 回注自愈 模型会卡、会超时、会吐出解析不了的 JSON；生产系统必须假设它随时出错并自愈，而不是期待它永远正确 conductorservice/toolcall.go（defaultToolCallTimeout=5.5min、子代理/ACP/文件传输 30min、slides 10min、maxParallelToolCount=5）＋ agent.go · categorizeToolCalls；通用供应商兜底在 LLM 客户端层多 provider 重试 claude/provider.go · getProvider（AnthropicDirect→GCP）；另 tryModelFallback 仅 slides 编辑专用的限流降级（Gemini 3.1 Pro→Gemini 3 Flash，429/overload）
主动压缩，而不是等窗口爆 context 窗口有限且"越长越笨"；等到撑爆才处理就晚了，要在一条红线前主动摘要 conductorservice/summary.go · summaryContextLine = min(窗口×0.8, 200_000)
短期 context 会被压缩，长期 memory 要落盘可检索 压缩是有损的，重要的事必须在被糊掉之前抢救到硬盘，日后还能按语义捞回 conductorservice/memory_flush.go · concurrency.SafeGo + context.WithoutCancel → 写 memory/YYYY-MM-DD.md 再入向量库
质量靠统计性 eval 守，裁判要跨厂商 agent 输出天生带随机性，不能靠"跑一次看着对"；要像 CI 一样批量判分，且用别家模型当裁判避免自评偏袒 cmd/memory_eval · 判分 Score < 0.6 即视为不达标（eval/runner.go）
稳定在前、易变在后，是前缀缓存的铁律 前缀缓存按字节严格前缀匹配，改动越靠前失效越多；把易变的东西降精度、往后放，缓存才留得住 conductorservice/context.go · <current_time> 只用 Format("2006-01-02")（只到日期，一天之内不变）
把这张表钉在墙上。 它不是这门课的"总结页"，而是你下一次真正动手写 agent 时的 checklist：每做一个决定，回来问一句"这条经验管不管这里、对应哪个机制？"

四、你现在站在哪：从"发育学"走向"解剖学"

这门入门课，讲的是发育学——一台 agent 是怎么从一个大脑一层层长出来的。它回答的是"为什么会有这个零件"。但一个成熟系统，光懂它怎么长出来还不够，你还得会按关注点把它解剖开——这就是进阶课要做的事。

两门课不是重复，而是互补的两个视角。举个例子你就懂它们怎么接上：本课经验清单第 (2)、(8) 条讲的"每轮重发历史"和"稳定在前、易变在后"，在进阶课里被提炼成一个更锋利的判断——进阶 0001《context 是主轴》证明：成本、质量、评估、工具这四支柱不是并列的四个问题，而是一个中心（context 组织）投影出的四张脸。你在这里辛苦一层层装出来的直觉，到那边会被收拢成一句可推导的原理。

五、关键定义

问题链（problem chain）
这门课的组织方式：每一节课的"零件"都是为了填补上一节留下的一个真实痛点而出现，环环相扣。理解 agent 的正确姿势，是理解这条链，而不是记住零件清单。
能力叠加塔
把问题链画成空间结构：以"大脑+循环+工具"为地基，自底向上每层补一个痛，塔顶收束成生产级 agent。塔的层序 = 课程的施工顺序。
发育学视角 vs 解剖学视角
发育学（本入门课）回答"这台 agent 怎么长出来、每个零件为什么存在"；解剖学（进阶《四支柱》课，../index.html）回答"成熟系统按 context / 工具 / 评估 / 成本四个关注点怎么切、怎么权衡"。同一个 mino_server，两种互补看法。
前缀缓存契约
由厂商定价规则强加的排列约束：prompt 按字节严格前缀匹配缓存，改动越靠前失效越多。工程上落成"稳定在前、易变在后"，mino_server 的体现是 <current_time> 只精确到日期（Format("2006-01-02")）。

六、自测题

Q1：有人说"agent 就是给大模型加了一堆工具"。用这门课的问题链反驳他——为什么"工具"其实排在很靠后的位置，它前面必须先有什么？

这句话把"手脚"当成了"生命"。按塔的层序，工具（第05课）之前必须先有三层地基：外部循环（第02课，streamChat 的 step 循环）让它能"再走一步"，否则调完一个工具就结束了；每步重组 messages（第03课，buildConductorMessages）让无状态的大脑记得住上一步发生了什么；system 人格与规矩（第04课，prompts.go）让它知道自己是谁、该干嘛。

更关键的是工具后面还有一长串：工具装了不会用要靠"描述即 prompt"（第06课），会用了怕失控要靠编排（第07课，maxParallelToolCount=5），还要假设它不可靠而自愈（第08课）。所以"加一堆工具"只是塔中间的一层——真正让它成为 agent 的是那圈循环和它周围的一整套治理，而不是工具本身。

Q2：为什么"落盘长期记忆"（第10课）必须排在"主动压缩"（第09课）之后，而不能反过来？把这两层调换会发生什么？

因为第10课这个零件，正是被第09课的痛逼出来的——它俩是问题链上紧挨着的一环，顺序反了整条逻辑就断了。

第09课的 summary.go 解决"context 会爆"，代价是有损压缩：旧对话被糊成一段摘要，细节没了。正因为压缩会让 agent"失忆"，才需要第10课在压缩发生之前把重要的事抢救到硬盘（memory_flush.go 里落盘就是刻意安排在压缩前触发的）。如果先有长期记忆、还没有压缩机制，那 context 根本不会爆，你也就没有动机去建长期记忆——这个零件会显得多余。这正说明：零件的价值只在它对应的痛出现之后才成立，这就是问题链的意义。

Q3（收口题）：本课经验清单第 (2) 条说"每轮重发完整历史 → 成本与 context 排列强耦合"。请把它和第 (8) 条"稳定在前、易变在后"接起来，说清为什么 <current_time> 只精确到日期。

两条经验是一根链子的两端。第 (2) 条讲根因：模型无状态，每一轮都得把全部历史 + 工具定义重新发一遍，所以厂商用前缀缓存来省重复计算——但缓存是按字节严格前缀匹配的（读命中只花约 0.1×、未命中要按写入 1.25× 重算）。

第 (8) 条是它的直接推论：既然改动越靠前、后面失效越多，那凡是每次都变的东西都不能放前面。时间戳恰恰是最"易变"的——如果 <current_time> 精确到秒，那每一次请求这个字节都不同，会把它后面所有缓存全部击穿，账单直接翻倍量级。mino_server 的解法就一行：context.go 里 <current_time> 只用 Format("2006-01-02")——降精度到"日期级"，让它在一整天内保持稳定，缓存就留得住了。一个看似随手的格式选择，背后是"成本 = context 排列"这条铁律。

全书到此收口。你现在手里有的，不是 13 个孤立的知识点，而是一条能自己复述、能钉到真实代码上的问题链：从"只会聊天的大脑"到生产级 Agent，每一层都知道自己为什么存在。这就是这门发育学课想给你的东西。
