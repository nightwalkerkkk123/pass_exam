Agent 的记忆：短期 Context 与长期 Memory

上一课我们让 agent 学会了"聊得太久就自己压缩"。但压缩是有损的——它会把旧对话糊成一段摘要。这一课解决一个更根本的问题：有些东西不能糊、不能忘。我们给 agent 装上第二种记忆：会落到硬盘、日后还能捞回来的长期记忆。

上一课 summarize 让 context 不再撑爆，代价是有损遗忘：旧对话被压成一段摘要，细节抹掉了。可"用户叫什么、他长期的偏好、上次我们一起定下的结论"这类事，一旦忘了下次就得重问一遍，体验直接崩。

问题于是很尖锐：压缩注定会忘，那怎么让 agent 跨会话记住真正重要的东西？

一、先分清两种记忆（白板 vs 日记本）

先用生活化的方式想（ELI5）：你今天开会，脑子里"正在想的那些话"是工作记忆——会议一散、注意力一转，大半就模糊了。但你会把关键结论写进笔记本，那是长期记忆——哪怕一年后翻开，字还在。

Agent 一模一样，只是名字换成术语。第一次出现，先解释清楚：

  • 短期记忆＝当前这次请求发给大模型的那一整包消息（我们前几课一直在拼的 messages 数组，即"当前 context 窗口"）。它会被上一课的 summarize 压缩，是块随时会被擦的白板。
  • 长期记忆＝被单独抽出来、写到沙盒硬盘上的一个 markdown 文件，还会进"向量库"（一个能按意思、而不只是按关键词去搜的数据库）。它是那本擦不掉的日记本。
一句话抓住区别：短期记忆会被压缩、装在每轮请求里、越攒越贵；长期记忆不会被压缩、静静躺在硬盘上、只在需要时才被捞回来。所以真正重要的事，不能只留在短期——要在它被压缩糊掉之前，先"抄"一份到长期。这个"抄下来"的动作，就是这节课的主角。

二、把记忆写下来：flush 这个动作

mino_server 里"抄进日记本"这件事叫 flush（冲刷、落盘）。触发点很讲究——就在压缩发生之前调一次（函数名直接就叫 memoryFlushBeforeCompression）。逻辑上顺理成章：既然压缩会糊掉细节，那就赶在糊之前先抢救一份出来。

flush 做三件事，都在 memory_flush.go 里：

  1. 抽取：把整段对话（历史 originalMessages + 当前轮 newMessages）丢给一个便宜的小模型（Gemini3Flash，模型简称；真实符号 common.ModelNameGemini3Flash，值 "gemini-3-flash-preview"），让它只挑出"值得跨会话记住"的东西——用户偏好、拍板的决定、项目事实、未完成的任务、人名关系。产出是一小段 JSON：{"memories": "..."}。配置是低温度、关掉思考（Temperature 0.2、ThinkingBudget 0、MaxTokens 4096）——因为这是"摘录"而不是"创作"，不需要它发挥。
  2. 落盘：把抽出来的记忆追加写进当天的文件 memory/YYYY-MM-DD.md（resolveMemoryFlushPath）。注意是 append-only（只往后加，不覆盖），像日记一天一页。为防两个请求同时写坏文件，用了一把"弱锁"（weaklock：锁的 TTL / 被保护的写操作最长跑 10s（LockTimeout=10s）；拿不到锁时最多重试 3 次、每次隔 200ms（RetryCount=3、RetryDelay=200ms））。
  3. 入库：文件写完，再把这段记忆异步塞进向量库（IndexMemoryFileToVector），这样日后能"按意思"搜到它，而不是靠精确关键词。

还有一条"更慢的整理"链路顺带一提：每满 15 轮用户对话（memoryMDUpdateRoundInterval = 15），系统会再用 Gemini3Flash 把零散的日记增量重写进一份总的 MEMORY.md——相当于每隔一阵把散记整理成"目录"。本课先聚焦按天 flush 这条主线，知道有这么条整理链路即可。

三、最关键的设计决策：为什么是 SafeGo + WithoutCancel

这一节是本课的"为什么"最深的地方，慢慢来。落盘要调小模型抽取，是有耗时的。如果让用户干等着落盘完成才看到回复，体验就毁了。所以第一个决定很自然：异步——开一个后台协程（Go 里叫 goroutine）去写，主流程立刻返回。mino_server 用的是团队封装的安全启协程工具 concurrency.SafeGo（比裸 go func() 多了 panic 兜底）。

但异步会踩一个所有新手都会踩的坑。用 ELI14 讲：

在服务端，每个请求都拴着一根"引线"叫 context（ctx）——请求一结束，系统就拉断这根引线（cancel），拴在上面的一切正在进行的活儿都会被喊停。这本是好事：用户关掉页面，就别再白白烧算力了。可我们的落盘协程也拴在这根引线上——用户一看到回复、请求一收尾，引线被拉断，落盘里那个抽取记忆的 LLM 调用就被中途掐死，记忆没写成，agent 又"失忆"了。

解药只有一句代码，但它是这节课最值得记住的一行——用 context.WithoutCancel 从原 ctx 派生出一根"剪断了取消信号、但保留其它信息的新引线"，让落盘协程拴在新引线上：

// 使用 WithoutCancel 创建独立 context，避免请求结束后
// context canceled 导致 LLM 调用失败
detachedCtx := ctx.CopyFromCtx(context.WithoutCancel(ctx))
concurrency.SafeGo(detachedCtx, func() {
    // ... runMemoryFlush 在这里安心跑完抽取 + 落盘 ...
})

类比就一句话：你下班走了（请求结束），但托付实习生留下来把今天的日记写完（脱钩的 detachedCtx），他不会因为你走了就停笔。 注意 WithoutCancel 只剪掉"取消信号"，用户身份、追踪信息这些还照常带着——所以落盘协程仍知道"这是谁的记忆、该写到哪"。

四、写下来，还要找得回来（闭环）

光写进硬盘还不算"有记忆"——得能在下次用上。这就是为什么落盘后还要 IndexMemoryFileToVector：把记忆送进向量库，日后可以"按意思"检索（比如用户问"我上次说想用什么数据库"，即使措辞完全不同也能捞到那条记录）。

捞回来之后走哪条路回到对话？还记得第 04 课讲的 context_info 吗——那条 developer 消息里有一个专门的 <user_memory> 块。相关记忆就是从这里重新回注进短期 context 的。于是闭环合上了：

这次对话会被压缩遗忘
▶
flush 落盘存进日记本
▶
下次会话<user_memory> 回注
↻

这里藏着一个精妙的成本设计：记忆不是每轮都全量塞回 context（那会白白烧 token，还触发上一课的压缩）。它平时躺在硬盘和向量库里不占预算，只在需要时把相关的一小段经 <user_memory> 带回来。省钱和"记得住"这两件看似矛盾的事，就是靠"短期 / 长期分离"同时拿到的。

五、关键定义

短期记忆（short-term context）
本次请求发给模型的 messages 数组，即当前 context 窗口。会被上一课的 summarize 有损压缩，越攒越贵。一块随时被擦的白板。
长期记忆（long-term memory）
被单独抽取、落到沙盒硬盘的 markdown 文件 memory/YYYY-MM-DD.md，并索引进向量库。不会被压缩，只在需要时按语义检索回注。一本擦不掉的日记本。
flush（记忆落盘）
在压缩之前触发（memoryFlushBeforeCompression）：用 Gemini3Flash 从对话里抽出值得长期保存的信息，追加写入当天 md 文件并入向量库。核心函数 executeMemoryFlush / flushConversation。
context.WithoutCancel
从一个 ctx 派生出"剪掉取消信号、但保留其余信息"的新 ctx。用它让异步落盘协程不因请求结束被 cancel 而中途失败。配合 concurrency.SafeGo 使用。

六、自测题

Q1：落盘明明用 SafeGo 开了后台协程异步跑了，为什么还非得套一层 context.WithoutCancel？直接把请求的 ctx 传进协程不行吗？

不行，会失忆。 关键在于 ctx 会传播"取消信号"：用户一看到回复、请求一收尾，这根 ctx 就被 cancel。异步协程虽然还活着，但它手里拿的如果是原 ctx，那么它正在做的抽取记忆的 LLM 调用会因为 ctx 被取消而立刻失败——记忆没写成。

context.WithoutCancel 派生的新 ctx 剪断了取消信号（但保留用户身份、trace 等信息），落盘协程拴在它上面就能越过"请求结束"这条线把活儿干完。一句话：SafeGo 解决"不阻塞用户"，WithoutCancel 解决"别被请求结束连累"，两个坑要分别堵。

Q2：既然记忆都抽出来了，为什么不干脆每轮请求都把整份 MEMORY.md 塞回 context 里，省得还要向量库检索这么麻烦？

因为那会同时踩中前几课的两个坑。第一，烧钱：记忆是长期累积的，全量塞回意味着每一轮都为一大堆"这次用不上"的旧记忆付 token 费。第二，反而更容易被压缩：记忆越堆越多，短期 context 越快撑到上一课的压缩红线，结果又被有损压缩糊掉——绕了一圈回到原点。

所以正确做法是"分层 + 按需检索"：长期记忆平时躺在硬盘/向量库不占预算，只在与当前问题相关时，把那一小段经 <user_memory> 回注。这才是"既记得住又负担得起"。

好了，到这一课为止，你亲手装的这台 agent 已经会说（messages）、会做（工具）、不失控（编排与可靠性）、还记得住（压缩 + 长期记忆）。它看起来相当完整了。

可是——你今天往 prompt 里改了一句话、给记忆抽取换了个更小的模型，你怎么知道它没因此变笨？ 服务端代码有 CI 帮你把关每次改动，那 agent 这种"输出天生带随机性"的东西，它的"CI"又长什么样？下一课就来装最后两个零件里的第一个：评估。
