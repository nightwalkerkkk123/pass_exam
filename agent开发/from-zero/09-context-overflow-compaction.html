Context 会爆炸：聊着聊着为什么就崩了

一个能连续干活的 agent，聊得越久越危险。这节课只讲一个零件：当对话长到装不下时，agent 怎么主动把旧对话读一遍、缩成一小段摘要，好腾出地方继续聊——而不是傻等它撑爆。

上一课我们让 agent 学会了自愈：超时会重试、报错会回退、坏 JSON 会被修好再回注。但有一个更慢、更隐蔽的问题一直没解决——

因为大脑（大模型）是无状态的，每走一步，agent 都得把"从开头到现在的完整对话"重新打包发过去（还记得 buildConductorMessages 吗）。对话越长，这个包越大。早晚会超过模型一次能读进去的上限；而且远没到上限，模型就开始读得不认真、变笨了。这一课就来拆掉这个雷。

一、先打个比方：白板会写满

想象你和一个记性为零的助理合作（ELI5：他每次开口前，脑子都会被清空一次）。为了让他知道来龙去脉，你把整段对话抄在一块白板上，每次都让他从头读一遍白板，再回答。

问题来了：白板就这么大。你们聊得越久，白板上的字越多。token（可以粗略理解成"字/词的计费与计量单位"）一轮一轮往上堆——终有一刻，白板写满了，新的话再也挤不进去。

关键认知：上下文窗口（模型一次能读进去的 token 总量上限）是一种有限资源，像白板的面积、像书包的容量。它不会自己变大。所以只要 agent 想活得久，就必须有人来擦白板——问题只是"什么时候擦、怎么擦"。

二、别等撑爆：为什么要主动画一条红线

新手的直觉是："等白板真写满了、报错了，再处理不就行？" 这恰恰是错的。有两个原因让 agent 必须提前动手。

原因一（硬的）：真撑爆会直接报错。包太大发过去，模型会甩回一个 ErrLLMPromptTooLong——这一步直接失败。你不想让用户的任务卡死在这里。

原因二（软的，更阴险）：远没写满，字就开始看不清了。白板这个比方有个漏水点：真白板写到 99% 才装不下，前 98% 的字一样清楚。可模型不是——token 越多，它分给每个字的"注意力预算"越薄，越靠后、越拥挤的信息越容易被读漏。这个现象叫 context rot（上下文腐烂）：还没报错，答案质量就先偷偷滑坡了。

两个原因合起来，结论只有一个：不要等满，要在还清醒的时候就压。所以 mino_server 画了一条红线——不是画在 100%，而是画在窗口的 80%。

为什么取"80% 和一个上限里更小的那个"？因为有些模型窗口特别大（比如上百万 token）。就算它装得下，塞到几十万 token 时 context rot 早就发作了。所以用一个绝对上限（20 万 / 聊天 10 万）兜底：装得下 ≠ 用得好。

这条线也不是唯一防线。万一某一步一次性灌进来太多东西、直接越过红线撑爆了，主循环还有个兜底急救：请求真的抛出 ErrLLMPromptTooLong 时，恢复路径不读取任何 token 阈值——只要还没有现成摘要，就 summarizeForce（force=true，跳过 80%、绝对上限、防抖等一切检查）强制压一遍，再 applySummaryImmediately 立刻把摘要换上、continue 重试，全程无条件（这正是上一课"自愈"的一个分支）。红线是主动预防，兜底是无条件的被动急救，两者都在。

那 thresholdToken = 120_000 又是干嘛的？它跟错误恢复没关系，是每一步的内容级机制：某步读回的 TotalTokens() 超过 12 万时，就把这条消息标成"大上下文"（IsBigContext），从而提高该消息的文本截断等级、把里面的长文本压得更狠。别把这条"大上下文判定线"和上面的错误急救混为一谈。

三、压缩到底怎么压：读一遍，写成一页

很多人以为"压缩"就是"删掉前面几条旧消息"。不是。删除是无脑丢弃，会把重要的前因一起丢掉。真正的做法更聪明，也更像人：

ELI14：期末复习时，你不会把一学期的笔记原样背下来（装不下），也不会随手撕掉几页（会漏考点）。你会把厚厚一摞笔记通读一遍，浓缩成一页纲要，然后靠这页纲要复习。压缩就是让一个小模型替 agent 做这件事。

两个设计细节值得记住：

  • 压缩是有损的。摘要必然丢掉细节——这是"腾地方"的代价，也是它和"记忆"的根本区别（下一课的伏笔）。所以压缩前先把原文归档存盘，万一以后要翻账还找得回。
  • 刚压过就别反复压。压缩本身要多花一次模型调用，很浪费。所以有个防抖：距上次总结 ≤2 步（summaryReuseStepThreshold = 2，含正好 2 步）就跳过本轮再总结、复用上次的摘要。别一激动把白板擦个不停。

四、更省的小算盘：先问一句"还是同一个话题吗"

上面讲的是对话进行中的压缩。还有一种场景：聊天品类（IM）里，用户可能晾了你一整天才回来说句话。这时如果他开的是全新话题，那把一大堆老上下文原样带上去纯属浪费钱。

mino_server 在这里加了一道"冷启动前预压缩"，而且动手前先花一分钱问一句：这条新消息，到底是接着老话题，还是开了新话题？

这里藏着一条贯穿全课的成本哲学：按任务难度选模型。判断"是不是同一个话题"是个极简任务，用最便宜的 nano 模型、只让它吐 50 个 token 就够；真正的对话才配得上贵的大模型。省下的每一分，都是从"不必要的 token"里抠出来的。

五、关键定义

上下文窗口（context window）
模型一次能读进去的 token 总量上限。有限、不会自增——像白板的面积。GetModelContextWindow() 返回当前模型的这个值。
context rot（上下文腐烂）
token 越多，模型分给每段信息的注意力越薄，读漏、读错的概率上升。所以"还没撑爆"不等于"还好用"——这是提前压缩的根本理由。
上下文压缩 / compaction
把一长段旧对话交给一个小模型读一遍，浓缩成一段摘要（analysis + summary），用它替换原始旧对话来腾出 token。有损，且不是简单删除。
压缩触发线（summaryContextLine）
min(0.8 × 窗口, 上限)，上限默认 20 万 / 聊天品类 10 万。累计 token 到这条线才压，兼顾"别撑爆"和"别降智"。

六、自测题

Q1：既然模型窗口能装下，为什么触发线要卡在 80%，还要再压一个绝对上限（20 万 / 10 万），而不是等接近 100% 再压？

两层原因，都指向"装得下 ≠ 用得好"：

① 软退化（context rot）：远没到 100%，模型的注意力就已经被稀释，越靠后的信息越容易被读漏，答案质量先偷偷滑坡。等到 100% 时它早就变笨了。所以要在它还清醒时就压——这是卡 80% 的理由。

② 大窗口模型更需要绝对上限：有的模型窗口上百万 token，80% 也是几十万，context rot 在那之前就重度发作。所以再叠一个绝对红线（min 取更小者）兜底，聊天品类甚至收到 10 万，宁可早压也不硬塞。

顺带：真被一步撑爆时还有一层兜底急救——它不依赖任何 token 阈值，只要没有现成摘要就无条件强制压一遍再重试；这是"救火"，不该常态依赖。（thresholdToken = 120_000 是另一回事，只是"大上下文"判定线，与这里的错误急救无关。）

Q2：压缩用的是便宜的 Gemini-3-Flash / GPT-5.4-nano，而主对话用的是贵的大模型。让小模型去总结、可能丢细节，这样安排合理吗？为什么？

合理，而且是刻意的成本设计。

任务难度不匹配就是浪费。"把一段对话读懂、浓缩成纲要"是相对简单、结构化的活儿，不需要顶级推理能力；用小模型足够，还快、还便宜。而"话题相关性判断"更简单，就用最小的 nano、只吐几十个 token。反过来，如果拿贵的大模型去做每一次压缩，压缩本身的开销会高到抵消掉它省下的钱——本末倒置。

至于"可能丢细节"：这是压缩作为有损操作的固有代价，不是选错模型的锅。真要防丢，靠的是压缩前把原文归档（saveContextToFile）和保留最近几条原文，而不是把总结任务升级成大模型。

压缩的本质是"把旧对话忘掉，换成一段摘要"。对"刚才在干嘛"这够用了——但它是有损的。

可有些事根本不该忘：用户是谁、他长期的口味和偏好、上个项目定下的规矩。这些一旦被摘要模型判成"不重要"揉掉，甚至等这次会话一结束、白板一擦，就彻底没了。agent 需要的不只是"记得住这次对话"的短期 context，还得有一种能跨会话捞回来的长期 memory。它存在哪、什么时候写、又怎么在下次读回来？下一课就装这个零件。

这一课你装上了 agent 的"擦白板"能力：把有限的上下文窗口当资源来经营，在还清醒时主动把旧对话压成有损摘要，并用最便宜的模型精打细算。下一课 → 10 Agent 的记忆：短期 Context 与长期 Memory，我们让它记住那些不该被压掉的东西。读到卡壳，回会话里问你的 AI 老师，可要求 ELI5 / 看真实代码。 · 🏛 返回门户
