编排：什么时候再走一步、什么时候停、能不能并行

上一课我们把工具的说明书写好了，大脑终于会挑工具、会填参数了。可它一旦开始干活，问题就来了：一轮想调好几个工具，调完还想再调……这个循环，到底谁来喊停？一次又能同时开几路？这节课只讲清楚 agent 循环的"油门与刹车"。

上一课的大脑，一轮可能同时要求调用好几个工具，调完看了结果还想再调下一批。那么：这个循环要一直转下去吗？会不会停不下来、转成死循环？会不会一口气并发太多工具，把下游的浏览器、代码沙箱、搜索服务全给打爆？

一、先打个比方：agent 循环就是"边问边做"的厨房

想象你是一个什么都不会、只会照着说做的帮厨（这就是执行循环的代码），身边站着一位大厨（这就是大模型）。你们的合作方式特别死板，就三句话：

  • 你问大厨："接下来干啥？"
  • 大厨要么说"去切菜、去烧水"（这就是"要求调用工具"），要么说"好了，可以上菜了"（这就是"不再要任何工具"）。
  • 你照做，把结果端回去给大厨看，然后再问一遍"接下来干啥？"

这一"问一做"就叫一步（step）。厨房里的一整顿饭，就是这样一步一步问出来、做出来的。什么时候散伙回家？只有两种情况：①大厨说"上菜吧"（活干完了），或者 ②你们已经来回问了太多次（比如约好最多问 100 次），再问下去就该打烊了，强制收工。

这个厨房，在 mino_server 里真真切切就是一个 for 循环。它有个很形象的名字——心跳循环：只要还在转，agent 就活着；一旦停下，这次对话就结束。

看图里那个红框 runConductorStep——它就是"一步"的主体：问一次大模型（拿到这轮回复）→ 执行它要的工具 → 把结果塞回消息。塞回这一下最关键：工具的输出会被包成一条消息（上一课讲过的 Tool 角色），追加进对话历史。这样下一圈再问大模型时，它就"看得见"上一步工具返回了什么——这就是循环能滚下去的燃料：这一步的输出，变成下一步的输入。

二、什么时候该停？两个出口，一个都不能少

循环最怕的就是"停不下来"。所以它必须有明确的刹车。mino_server 给了两个出口，正好对应厨房那两种散伙情况：

出口①　正常完成（软刹车，大厨说"上菜"）
某一圈里，大模型的回复不再要求调任何工具——代码里就是判断这轮回复的工具列表长度 len(ToolCalls) == 0。这说明大脑觉得活干完了，于是把 agent.done 置为 true，循环 return nil 干净收工。
出口②　撞上步数上限（硬刹车，防死循环）
如果大脑一直有工具要调、迟迟不说"上菜"，转的圈数达到上限 maxStepCount（默认 defaultMaxStepCount = 100；hooks 可覆写，比如 AnyClaw 场景放宽到 200）还没 done，循环就强制结束，返回一个报错 ErrAgentMaxStepCountReached。这是防止大脑陷入"调工具→再调→再调"死循环、无限烧钱的保险丝。

注意区分两个词，很多人在这里绕晕：一步（step） ≠ 一个工具。一步是"问一次大模型 + 把它这轮要的全部工具都跑完"。大模型一次可以同时点名好几个工具，这一整批算同一步；只有等结果塞回、重新去问大模型，才进入下一步。所以 runStepCount 记的是"问了大模型几个回合"，不是"跑了几个工具"。这也解释了为什么上限是 100 步就够用——一步里可能已经并发跑了好几个工具。

三、一次能同时干几件事？并发闸门：5 和 3

回到厨房。大厨这一轮如果说"同时去切菜、烧水、预热烤箱、洗米、备酱料、摆盘"——六件事一起来。你家帮厨只有几个人，不可能六件事真的同时开工。于是需要一道闸门：一次最多放几个进去，多的排队等空位。

为什么要限？因为每个工具背后都是一个真实的下游服务（浏览器实例、代码沙箱、搜索接口）。如果大模型一口气要 20 个工具、后端就真开 20 路并发，下游瞬间被打爆——这就是 chain.in 里担心的"把下游打爆"。mino_server 的做法是：允许并行，但卡一个上限。

两个数字，两条真实规则：

  • 普通工具：一次最多同时跑 5 个（maxParallelToolCount = 5）。够快，又不至于把下游打爆。
  • 子 agent 工具：一次最多同时跑 3 个（maxParallelSubAgentsToolCount = 3）。子 agent 是"派另一个 agent 去做一整件事"，本身极重（它内部又有自己的心跳循环、又要调大模型），所以开太多会拖垮整个系统，闸门收得更紧。

代码里的判断很直白：执行一批可并行工具前，先探一眼这批里有没有子 agent 类任务（hasSubAgent）——有就用 3，没有就用 5，然后建一个这个上限的任务池并发跑。注意，这个"并行"是大模型主动要来的：上一课提到，每轮请求都带了 WithEnableParallelToolCalls(true)，允许模型一次吐出多个工具让后端并发执行。不是所有工具都能并行——有些必须串行（比如互相依赖、或会改同一份文件的），它们会被单独分到串行组，一个一个来。

一句话记住这节课：agent 循环 = 问大模型 → 干活 → 结果塞回 → 再问，两个刹车（大模型不再要工具 = 软停；转满 100 步 = 硬停），一道闸（普通并发 5、含子 agent 收到 3）。油门是大模型给的，刹车和闸是工程写死的。

四、自测题

Q1：某次对话里 runStepCount 一路涨到 100，最后循环报了 ErrAgentMaxStepCountReached。这说明大模型在这 100 步里，每一步都做了什么？如果它中途某一步"想通了"，本该发生什么？

说明：这 100 步里每一步大模型都要求了至少一个工具（len(ToolCalls) > 0），也就是它从没说过"上菜"，agent.done 一直是 false。于是循环条件 runStepCount < maxStepCount 一直满足地转，直到撞上默认上限 100，触发硬刹车报错退出。

本该发生的：只要中途任意一步大模型返回的回复里没有工具调用，runConductorStep 就会把 agent.done 置 true，循环立刻 return nil 干净收工——根本轮不到硬刹车。所以撞上限往往是个信号：要么任务确实太复杂，要么大模型陷进了"反复调同类工具却不收敛"的圈里（这正是下一课要处理的可靠性问题）。

Q2：大模型这一轮同时要求调 8 个工具，其中有 2 个是子 agent 工具。后端这一步实际最多几路并发？为什么不是 5、也不是 8？

最多 3 路。因为 executeParallelTools 在开跑前会先探测这批里有没有子 agent 工具（hasSubAgent）。只要命中哪怕一个，整批的并发上限就从 5 收紧到 maxParallelSubAgentsToolCount = 3——不是"5 个普通的 + 3 个子 agent 分开算"，而是整批一起走 3 这条更保守的闸。8 个工具里先放 3 个进去,其余 5 个排队等空位。

为什么这么设计：子 agent 每个都很重（自带完整心跳循环、还要调大模型），一次开 5 个足以拖垮系统；宁可牺牲一点并发度，也要护住下游。这是"可用性优先于速度"的典型取舍。

刹车装好了、闸门也卡住了——理想情况下循环该稳稳收敛。可真实世界不讲道理：工具会超时卡住不返回，会报错，大模型还会吐出语法坏掉的 JSON 当工具参数。循环一旦撞上这些脏东西，会当场崩掉、还是能自己爬起来接着干？这就是下一课要拆的可靠性与自愈。
