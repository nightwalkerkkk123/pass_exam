工具描述就是 Prompt：写不好，Agent 就变笨

上一课我们给模型装上了"手脚"。但模型看不到工具里面的一行代码——它决定"要不要用、怎么用"，靠的只有你贴在工具上的那张标签：名字、描述、参数说明。这节课只讲清一件事：那张标签本身，就是写给模型看的 prompt。写得含糊，再强的模型也会变笨。

第 05 课结尾我们发现：模型完全靠工具的名字、描述、参数说明来判断"该不该用这个工具、该怎么填参数"。它看不到函数体，看不到实现。那这些描述写得好不好，就直接决定了 agent 聪不聪明。今天就来拆：一段好的工具描述长什么样，写砸了会怎样。

一、在模型眼里，工具只是一张"标签"

先讲个直觉（ELI5）。想象你带一个刚入职、看不到公司代码的实习生（这就是模型）。你不能手把手教他，只能在每件工具上贴一张便利贴：这叫什么、干什么用、要填哪几个空。他能不能把活干对，全看这张便利贴写得清不清楚。贴"读图工具：读取一张图片的内容，注意不会把图展示给用户"，他就用得准；只贴"图片"两个字，他就懵了——是读图？画图？还是删图？

这不是比喻凑数，而是机制的字面事实。回忆第 05 课的一个关键点：工具不写进对话消息里，而是带外作为 API 参数递给模型的。代码里就一行——把当前所有工具收集好，用 model.WithTools(toolInfos) 挂进这一轮请求。模型收到的每个工具，本质就是一个 schema.ToolInfo 结构，其中三个字段是模型可见的：名字（name）、描述（desc）、参数说明（params，即内嵌的 *ParamsOneOf）。这个结构体另外还带一个 Extra map[string]any——那是工具侧自己用的元信息，不会发给模型。所以在模型眼里，一个工具就只有 name / desc / params 这三样。

那第三格"参数说明"具体长什么样？在 viewimageservice/tool.go 里，view_images 工具的参数是用 schema.NewParamsOneOfByOpenAPIV3(...) 写出来的，一个字段一个字段地标注类型和含义：

ParamsOneOf: schema.NewParamsOneOfByOpenAPIV3(&openapi3.Schema{
    Type: openapi3.TypeObject,
    Properties: map[string]*openapi3.SchemaRef{
        "urls": { Value: &openapi3.Schema{
            Type:        openapi3.TypeArray,
            Description: "The URLs of the images to retrieve, which must start with image/",
            Items: ...string,
        }},
        "label": { Value: agenttypes.LabelToolSchema },
    },
    Required: []string{"label", "urls"},
})

注意那句 Description: "...which must start with image/"——这不是给人看的注释，它会被序列化进发给模型的 schema，是模型填参数时唯一能读到的"填写须知"。约束写在这里，模型才知道 URL 必须以 image/ 开头。参数的 Description，就是参数的 prompt。

二、坏描述 vs 好描述：同一个工具，两种命运

既然描述就是 prompt，那"好"和"坏"的差别就很具体了。一段好的工具描述回答三个问题，缺一不可：

要素❌ 坏描述✅ 好描述（view_images 真实文本）
做什么 "处理图片"
是读？画？删？模型只能猜		 "Read the visual content of an image."
动词明确：读取
什么时候用 / 别用 （没写） "Important: This does not display the image to the user."
主动划出边界，防止误用
参数怎么填 "urls: 图片地址" "...which must start with image/" + Required: [label, urls]
带格式约束 + 哪些必填

坏描述那一列不是我瞎编的稻草人，而是每个人第一次写工具时的默认样子——名字取个 img，描述写"处理图片"，参数写"地址"。它能编译、能注册，看起来一切正常。可模型读到它时，只会做两件事之一：要么不敢用（怕用错，干脆绕开），要么乱用（把画图的请求也塞给它）。工具本身没坏，是那张标签让模型变笨了。

好描述的黄金三问：① 一句话说清动词（读/写/搜/画…）；② 明确什么时候该用、什么时候千万别用（view_images 专门用一段 Important: 说"我不会把图给用户看"）；③ 每个参数标类型 + 约束 + 是否必填。三问答全，模型就"懂事"了。

三、工具太多会互相干扰，而且每轮都要重发一遍

写好一个描述还不够。真实的 agent 手里工具很多——tools.go 里的 GetAllToolServices 是一张大注册表，登记了五十多个内置工具服务（浏览器、代码执行、搜索、画图、子代理、幻灯片……还有一个 browser_sub_agent 被注释停用了）。如果用户开了 MCP，运行时还会用 NewMCPServiceWithSessions 再往里塞更多。

工具一多，两个新麻烦冒出来。

麻烦一：工具会互相干扰。如果你有 search 和 web_search 两个名字相近、描述又都含糊的工具，模型会在它俩之间犹豫、选错。所以 mino_server 不是把五十多个工具无脑全给模型，而是过两道滤网，只把这次场景真正用得上的那批递过去：

全部工具服务　GetAllToolServices()　→ 五十多个 + 运行时 MCP
▼
第 1 道滤网：hooks.GetToolServices(ctx, features, …)
按当前子类型 / 功能开关（features）筛出这次要用的那批服务
▼
第 2 道滤网：每个服务 service.AsTools(...)
把一个服务展开成 0～多个具体工具，还能按 features 再裁剪
▼
toolMap：按工具名建索引，这才是这次 agent 真正能用的"手"

麻烦二：每轮都要把工具定义原样重发一遍。还记得第 03 课的"无状态"吗？模型不记事，每一轮请求都得把它需要的一切重新带上——工具定义也不例外。每一步开口问模型前，getCurrentToolInfos() 会从 toolMap 里把当前所有工具取出来，按工具名字典序排好（a.ToolInfo.Name < b.ToolInfo.Name），再用 model.WithTools 一次性挂进这轮请求。

为什么要排字典序？为了顺序稳定。顺序每轮都一样，才能命中厂商的前缀缓存（缓存要求"字节级严格前缀匹配"，这是进阶课第 0001 课的核心）。但即便命中缓存，工具定义仍然每轮都在那儿——它是这次请求 token 预算里雷打不动的"底盘"。工具越多，底盘越厚，这就是固定的 context 税：可增（加工具）不可减（每轮都得带）。

四、关键定义

工具 schema（schema.ToolInfo）
模型对一个工具的全部认知：名字 Name、描述 Desc、参数 ParamsOneOf。它就是"模型面向的 API"——你写的这段 schema，就是模型能读到的 API 文档。
OpenAPI v3 序列化（ToOpenAPIV3）
工具参数用 NewParamsOneOfByOpenAPIV3 描述，发给模型前调 ParamsOneOf.ToOpenAPIV3() 变成一段标准 OpenAPI v3 JSON schema。参数里的每个 Description 字段都会进这段 schema——所以参数说明也是 prompt。
两道滤网（GetToolServices → AsTools）
第一道按 features 从五十多个服务里选出要用的服务；第二道把每个服务展开成一个个具体工具。目的：只给模型这次用得上的工具，减少相似工具互相干扰。
固定 context 税
因为模型无状态，工具定义每一轮都要用 WithTools 原样重发，构成每次请求里不可省的 token 底盘。工具越多，底盘越厚——加工具是有"每轮复利"成本的。

五、自测题

Q1：你给 agent 新加了一个工具，名字叫 search，描述写"搜索"。可它手里已经有一个 web_search（描述：搜索公开网页）。上线后模型的行为可能出什么问题？根因是什么？

可能的问题：模型在 search 和 web_search 之间反复犹豫、或干脆选错——该搜网页时调了那个语义模糊的 search，拿回一堆无关结果。

根因（两层）：① 描述"搜索"没回答"黄金三问"——搜什么？和 web_search 有何区别？什么时候用哪个？模型只能读这段标签，读不出区别就只能猜（本课第二节）。② 两个功能相近的工具同时挂在请求里，本身就构成"互相干扰"——这正是 GetToolServices 那道滤网要按场景裁掉冗余工具的原因（第三节）。正确做法：要么删掉重复工具，要么把描述改到能一眼分清（"search: 只搜内部知识库；web_search: 只搜公开网页"）。

Q2：同事说"工具定义在第一轮发一次就行了，后面几轮模型自己记得住，别浪费 token 重发"。这个说法对吗？为什么？

不对。模型是无状态的（第 03 课），它不会"记住"上一轮发过什么——每一轮都是从零读入全部输入。所以代码里每一步开口问模型前，都会重新 getCurrentToolInfos() 取出全部工具、按字典序排好、再 model.WithTools 挂上去（agent.go）。不重发的后果：这一轮模型就"看不见"任何工具，会退化成纯聊天，无法调用工具。

那字典序排序是干嘛的？不是为了省钱本身，而是让工具顺序每轮完全一致，好命中厂商的前缀缓存——重发的 token 大部分能按缓存价（约一折）计。但"重发"这个动作省不掉，它就是无状态架构下的固定成本。

好，现在我们会写清工具描述、也知道工具是怎么被筛选和重发的了。可你有没有注意到第三节埋的一根线——每轮请求都显式打开了 WithEnableParallelToolCalls(true)：模型一次可以要求调好几个工具，甚至这一步的结果会引出下一步的连环调用。那问题来了：工具越加越多、模型一口气吐出五个工具调用、还想接着往下走——谁来决定"还要不要再走一步"？"这几个能不能并行、还是必须排队"？"什么时候该收手停下"？这就是编排。下一课见。
