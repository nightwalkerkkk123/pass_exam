可靠性：超时、报错、坏 JSON，Agent 怎么自愈

上一课我们让循环"听话"了——知道什么时候再走一步、什么时候停。可现实世界会捣乱：工具会卡住不动、大模型会累到罢工、它偶尔还会吐出你根本看不懂的乱码参数。这一课只讲一件事：一个生产级 agent 怎么不被这些意外打死，反而能自己爬起来。

上一课我们把循环控制住了：能多步、能并行、也知道何时停。但那套逻辑有个隐含前提——每一步都顺利跑完。真实生产里这个前提天天被打破：工具会卡在半路、大模型会过载返回 429、模型给工具填的参数 JSON 是坏的。循环控制得再漂亮，只要一步崩了没人接住，整个任务就断在那里。

一、先认清：会出岔子的一共就三类

先讲个生活里的画面。你开了一家小小的调度中心，接到客人的活儿之后，派"跑腿小哥"（工具）出门去办。办事的路上，会出三种岔子：

  • 🐢 小哥卡住了——出门后半天没回音，你不知道他是在路上还是出事了。（工具超时）
  • 🥵 你自己的大脑累瘫了——派活儿要先动脑子想"下一步派谁"，可这个"大脑"（大模型）此刻过载，直接跟你说"忙，等会儿再来"。（模型 429 / 过载）
  • ✍️ 派工单写成了鬼画符——大脑给小哥写的任务单是一堆乱码，小哥根本看不懂该干嘛。（坏 JSON 参数）

一个业余的调度中心，遇到任何一种岔子都会当场瘫痪。一个靠谱的，会为每一种岔子准备一套"接住 + 补救"的动作。mino_server 的 agent 正是这么干的。我们先看它把工具结果送回大模型之前，是怎么逐个分诊的——这张图是这一课的地图。

下面三节，我们一种岔子一种岔子地拆。

二、超时：不是"跑太久"，而是"太久没动静"

先纠正一个几乎所有人的第一直觉。你以为"超时"是这样："这个工具跑超过 5 分钟，就掐掉。" 错。mino_server 用的是另一种，更聪明。

ELI5：你点了外卖，平台不会说"下单超过 40 分钟自动取消"——万一这单本来就远呢？它盯的是骑手还动不动：只要地图上小蓝点在挪、状态在更新，就一直等；要是小蓝点卡在原地好几分钟一点动静都没有，才判断"出事了"。

工具也一样。工具在干活时会不停往外吐"进度事件"（心跳）。系统盯的不是"总共跑了多久"，而是"距离上一个事件过了多久"。只要还在吐事件，就说明它活着，继续等。这叫事件间隔超时。真实实现是：转发工具事件的那段代码 forwardToolEvents，每次去等下一个事件时，都给它套一个带超时的 context；一旦在超时窗口内没等到任何事件，就抛出 ErrToolCallEventIntervalTimeout。

事件间隔超时（心跳超时）
不看工具的总耗时，只看"两次进度事件之间的静默时长"。只要工具还在吐事件就一直等；静默超过阈值才判超时。生产 agent 用它，是因为工具的正常耗时天差地别，用固定总时长很容易冤枉长任务。

分层：不同任务，给不同的耐心

既然是"多久没动静才算死"，那对"本来就长、中途长时间静默很正常"的任务，就得给更长的耐心。mino_server 按任务类型分了好几档超时值——普通工具最短，重活儿最长：

注意这条隐藏原则：耐心只放大，不缩小。为什么？看最典型的桌面文件传输——它是"一次性长等待"：传输那几分钟里整条链路阻塞、根本不发心跳。如果还用 5.5 分钟的普通超时，会被误判成"卡死"而掐掉一个其实正在正常传输的任务。所以它必须把超时独立设到 30 分钟——让服务端的事件间隔超时与客户端的总传输超时对齐（均 30 min），两侧任一调整都要同步对方，宁可多等，不可错杀。

三、大模型过载：够条件就第一时间换模型，否则不原地重试

第二类岔子最反直觉：出问题的不是工具，是你的"大脑"自己。派活儿之前得先问大模型"下一步干嘛"，可大模型是共享的、有并发上限的资源。高峰期它会直接返回 429（请求太多）或 overloaded（过载）——相当于客服热线占线。

ELI5：电话占线时有一条"VIP 专线"约定——只有那位特定的 VIP（slides 编辑 + 当前用 Gemini3.1Pro-Preview）占线，总机才会第一时间把他转到备用坐席（换成更轻的 Gemini3Flash）并用新线重打；不是这位 VIP 的话，占线就是占线，这一次直接失败、不原地重拨。真正"挂了再拨几次"的重拨，留给线路本身抖了一下（连接类错误）那种情况。mino_server 就是这个顺序：先看能不能换模型，不能换也就不重试。

这里最值得学的，是 tryModelFallback 的克制。你可能以为"过载就自动换个模型顶上"是通用策略，但代码里它有两道硬门槛：必须是 slides 编辑模式，且当前模型恰好是 Gemini3.1Pro-Preview，两条都满足才降级到更轻的 Gemini3Flash。为什么这么保守？因为换模型会悄悄改变输出质量和风格，不是能随便做的事——只有在"这个特定重活儿已知会顶不住、而轻模型够用"的场景里才开这个口子。换过之后还会把选择写进 Redis，避免同一次任务里反复横跳。这就是工程和"想当然"的区别：自愈动作本身也有代价，得挑场景。

还有个细节值得记：PromptTooLong（上下文太长）在这里被特意排除在重试之外——重试同样长的上下文只会同样失败，纯属浪费。它被直接交给主循环去做压缩。这正好接上第四类岔子和下一课。

四、坏 JSON：不硬着头皮执行，而是"退回让它自己改"

第三类岔子最微妙。大模型给工具填参数，是靠"生成一段 JSON"完成的。可它偶尔会生成坏掉的 JSON——常见原因是这轮输出被 max output tokens 截断了，JSON 写到一半就断了。

ELI5：学生交上来的作业，格式全乱、有一半没写完。一个坏老师会硬按乱七八糟的内容去批改，结果错上加错；一个好老师会把作业退回去，附一张纸条写清"你这里格式错了、可能是写超了被截断，请改对再交一次"。mino_server 选的是好老师这条路。

具体机制分两步，都在 categorizeToolCalls 里：

  1. 先试着修：参数为空就补成 {}；否则用 jsonrepair.RepairJSON 尝试把坏 JSON 补全。
  2. 修好了也不执行：对非流式工具，即便修复成功，也把这次调用标成 IsInvalid，拒绝执行，并附上一段很长的理由。

关键在第二步的"拒绝执行"。为什么修好了还不跑？因为自动修复是启发式的猜测——它能补全括号，但猜不准你到底想传什么值。拿一份"猜出来的参数"去执行真实工具（可能是删文件、发消息），比不执行危险得多。而且坏 JSON 往往是"输出被截断"的信号，盲目执行等于拿半截指令干活。

那不执行怎么往下走？靠回注。系统把那段理由包成一条 Role=Tool 的工具结果（InvalidToolTaskData，错误码 ErrAgentInvalidToolCall），通过 AddToolResult 塞进消息列表。下一轮，模型就能读到自己上次错在哪，据此改对、重发。整段理由大意是：

"工具参数 JSON 格式有误（可能是上一次调用触发了 max output tokens 截断）。系统已成功修复 JSON，但为避免执行错误的调用，并未执行。请检查并改正原始参数后重试——可以减少输出、拆成多次工具调用，或用别的方式修好参数 JSON。"
自愈回注（error re-injection）
出错时不直接终止任务，而是把"错在哪、怎么改"写成一条工具结果，回灌给模型当下一轮的输入，让模型自己纠正后重来。坏 JSON、坏工具名、被锁工具走的都是这条路——它们统一被标 IsInvalid、串行处理、IsFinish=false（不结束对话）。
可增不可减（超时）
分层超时的调整方向只能变长、不能变短：因为把长任务的超时调短会误杀正在正常运行的任务（尤其是传输期不发心跳的一次性长等待）。

坏工具名、以及"已经执行过又被重复调用的 doc agent"，走的是同一条回注路——理由分别是 "invalid tool name, please check the tool name" 和"别再重复调 doc agent"。规律很清楚：凡是模型自己能改对的错，都退回去让它改，而不是让整个任务崩掉。

五、还剩一个错，它谁也接不住

你可能注意到：工具执行时冒出的错，绝大多数都被就地吞掉了（记个日志、变成一条失败结果回注给模型），唯独一个会一路向上抛——就是 ErrLLMPromptTooLong（错误码 100007009，定义在 biz/pkg/errcode/const.go）。

为什么它特殊？因为它不是"某个工具的锅"，而是"整段对话已经太长，塞不进模型的上下文窗口了"。这不是重试或改参数能救的——上下文长度是整场对话攒出来的账。所以它冒泡到 runConductorStep，记一句日志 "context too long, will compress via summary"，然后交给主循环去做压缩。前面重试那节我们也见过它：PromptTooLong 被特意排除在重试之外，正是同一个原因。

六、自测

Q1：模型吐了个坏 JSON，系统已经用 jsonrepair 修好了。既然修好了，为什么还坚持"不执行"，而是回注让模型自己改？

因为"修好格式"不等于"猜对意图"。 jsonrepair 能补全括号、闭合字符串，让 JSON 在语法上合法，但它无法知道模型原本想传什么值——被截断丢掉的那部分内容，修复器只能瞎猜。

拿一份"语法对但内容是猜的"参数去执行真实工具（可能是删文件、发消息、花钱调下游），后果比"不执行"严重得多。更重要的是：坏 JSON 常常是 max output tokens 截断的信号，盲目执行等于拿半截指令干活。所以正确动作是标 IsInvalid 不执行，把理由回注，让真正知道意图的模型（可能拆成多次调用、减少输出）自己改对重发。这就是"自愈回注"胜过"硬着头皮跑"的地方。

Q2：桌面文件传输为什么要单独给 30 分钟超时，还要求这个值不能往小调？普通工具 5.5 分钟不行吗？

不行。关键在于 mino_server 的超时是"事件间隔超时"——盯的是"多久没吐进度事件"，不是"总共跑了多久"。而文件传输是一次性长等待：传输那几分钟里整条链路阻塞、根本不发心跳。

如果沿用 5.5 分钟的普通阈值，系统会因为"5.5 分钟没收到任何事件"而误判它卡死，掐掉一个其实正在正常传输的任务。所以它必须独立设成 30 分钟：让服务端事件间隔超时与客户端总传输超时对齐（均 30 min），两侧任一调整都要同步对方，宁可多等，不可错杀。这就是"耐心可增不可减"原则——把长任务的超时调短，只会制造冤案。

Q3：一个普通（非 slides）对话，请求大模型时连续撞上 429 过载。系统会自动换一个模型来救场吗？最终会怎样？

不会换模型。 tryModelFallback 有两道硬门槛：必须是 slides 编辑模式、且当前模型恰好是 Gemini3.1Pro-Preview。普通对话两条都不满足，函数直接返回 false，不做任何模型降级。

它也不会原地重试。RetryIf 遇到 429/overloaded 会先调 tryModelFallback，非 slides 返回 false；而 429/overloaded 又不属于 IsErrLLMCall/IsErrLLMConnection 那类连接错误，于是 RetryIf 返回 false——第一次失败就停，只尝试 1 次，随即被包成 ErrAgentMaxConsecutiveExceptionsReached，这一步失败。所谓"重试 3 次（maxConsecutiveExceptions=3）/ slides 5 次 + 100ms 间隔"只在 RetryIf 返回 true 时才生效，即连接类抖动（IsErrLLMCall/IsErrLLMConnection），或 slides 编辑 + Gemini3.1Pro-Preview 触发换模型成功的情况。

这道题想让你记住的，是自愈动作的克制：换模型会改变输出质量，不是免费的，所以只在极窄的已知场景里才开。别把"过载就换模型"当成通用银弹。

这一课的一句话：可靠性不是"永不出错"，而是为每一类错都准备一个接得住、还能爬起来的动作——工具卡住用分层的心跳超时兜、模型过载按场景处理（够条件就第一时间克制地换模型、不够条件就不重试，真正的原地重试留给连接类抖动）、坏参数退回让模型自愈。绝大多数错都被就地消化成"下一轮的输入"，唯独上下文太长这一个，得靠压缩。

有了这三套自愈机制，agent 的单轮终于稳了：一步崩了有人接、能重来。可你注意到那个甩不掉的例外了吗——ErrLLMPromptTooLong。每接住一次错、每回注一段修复提示、每塞回一次工具结果，消息数组就又长一截。对话越拉越长，这个数组只增不减，迟早撑爆模型的上下文窗口。到那时，光靠"接住错误"已经不够了，得动手把历史压缩掉。这就是下一课要解决的痛。
