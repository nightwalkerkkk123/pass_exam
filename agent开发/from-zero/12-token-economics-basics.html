Token 是新的钱：成本、延迟与前缀缓存

前十一课我们把一个 Agent 的零件拆了个遍。这一课换一个视角：每一次调用大脑，你都在花钱、花时间。我们来看 mino_server 用什么手艺，让一个真跑起来的 Agent 不被账单和延迟拖垮——核心只有一个词：前缀缓存。

上一课我们发现：大脑是无状态的，它不记得上一轮说过什么。所以每一轮，我们都得把「完整对话历史 + 全部工具定义」重新打包，原封不动地再发一遍。第 10 轮时，第 1 轮那句「你好」还在被重新发送、重新计费、重新处理。这很浪费。一个真上线的 Agent，怎么才能不为「重复发送同样的开头」反复付全价？

一、先建立直觉：token 就是计价的水表

第一次接触大模型计费的人常有个误解：以为「问一次收一次钱，像点一杯咖啡」。不是。它更像自来水表——按流过的量收费。你灌进去多少字（输入 token）、它吐出来多少字（输出 token），都逐个 token 计量。

token（词元）是模型眼里的最小计量单位，大致是「一个字 / 半个英文单词」。一次调用的账单，就是「输入 token 数 × 输入单价 + 输出 token 数 × 输出单价」。这份单价表在 llms/common/model.go 里，每个模型一行。以 Claude Sonnet 为例（ModelNameClaudeSonnet）：

  • 输入 PriceInputTokens: 3（每 100 万 token 3 美元）
  • 输出 PriceOutputTokens: 15（输出比输入贵 5 倍——生成比阅读费劲）

结账逻辑在同文件的 getPrice 函数里，ModelFamilyAnthropic 那一支：把输入、输出、缓存读、缓存写各自乘上单价再相加。关键点来了：这里出现了两个新词——「缓存读」和「缓存写」。它们就是省钱的门道。

二、前缀缓存：把「不变的开头」存起来，别再付全价

想象一个剧组每天拍同一间客厅的戏。如果每天开拍前都把整套布景拆了重搭，那太蠢了——布景不变，只有台词在变。聪明的做法是：布景搭一次留着，第二天只补拍新台词。

大模型厂商也这么干，叫 前缀缓存（prompt caching）：你这次请求的开头，如果和上次一模一样，厂商就不必从头再算一遍，直接复用上次的中间结果。省下的算力，它以大幅折扣回馈给你。

读这张图要抓住三个倍率：

命中读 ≈ 0.1×
缓存里已经有的部分，这次读出来只花基础输入价的 十分之一（Sonnet：$3 → $0.3）。这是省钱的主战场。
写入 5 分钟 ≈ 1.25×
第一次把开头「存进」缓存，要多付 25% 手续费（$3 → $3.75）。这是一次性投资。
写入 1 小时 ≈ 2×
想让缓存活得更久（1 小时而不是 5 分钟），预付得更多——两倍价（$3 → $6）。

算笔账就懂了：一段 10 万 token 的稳定开头，第一次写入多花点（1.25×），但从第二轮起每轮都只按 0.1× 读。轮数越多，越划算。这就是为什么一个「多轮 Agent」几乎必然要用缓存。

理想与现实的差（一个真实的教学点）：内部虽然区分了 5m / 1h 两档 TTL（llms/common/common.go 里 CacheTTL5Minutes = "5m"、CacheTTL1Hour = "1h"、NewCacheControl1Hour()），但在 llms/claude/convert.go 的 convertCacheControl 里，把内部标记翻译成 Anthropic 的 ephemeral 参数时，TTL 字段被注释掉了，注释写着「GCP 目前不支持 1h 缓存」。结果：1h / 5m / 默认三个分支发出去的都只是 Type: "ephemeral"——实际上全按 5 分钟走。代码里的价格表准备好了 1h，线上却还没放开。这就是工程：理想的接口先建好，现实的开关慢慢开。

三、代价：缓存是「严格前缀」，一个字节变了，其后全部重算

缓存不是白给的，它有一条铁律：严格前缀匹配。厂商是从头开始逐字节比对的——从你发送内容的第一个字节起，一路命中，直到遇到第一个不一样的字节。从那个分歧点往后，全部作废，按全价重算。就像磁带：前面对得上就快进，一旦对不上，剩下的只能重录。

这条规则有个可怕的推论：它反过来把 context 的排列顺序焊死了。既然「越靠前的内容越容易被缓存复用」，那么设计原则只能是一句话——稳定的放前面，易变的放后面。

mino_server 处处都在遵守这条铁律。看这张请求版面图，每一段的位置都不是随便排的：

三个细节，三条同样的原则在起作用：

  • 工具定义按名字排序（conductorservice/agent.go 的 getCurrentToolInfos）：如果每次工具顺序随机，前缀就每次都不同，缓存全废。排序让它稳定。
  • system 挂 1 小时缓存点（reactagentservice/agent.go 的 NewLLMMessageWithCache1Hour(common.System, systemPrompt)）：系统提示词整个会话都不变，值得为它多付一次写入费。
  • 时间只精确到「天」（conductorservice/context.go）：如果写成 15:04:05，每一秒都是新前缀，缓存命中率归零。写成日期，一整天稳定。

四、延迟：流式让人「感觉」它更快

省钱之外还有一个维度：延迟。模型生成一长段回答要几秒甚至十几秒。如果你憋到全部生成完才一次性显示，用户会盯着空白屏干等。流式（streaming）的思路很朴素——像打字机，生成一个字就吐一个字。总时间没变，但用户第一眼就看到东西在动，感知延迟大大降低。

这在 llms/stream.go 里。两个真实设计值得记：

  • 自动续写：streamWithContinue 默认开启续写，一段回答如果撞到 max_tokens（一次能输出的上限）被截断，它会自动再发一轮接着写，把碎片拼起来——最多续 maxRounds := 10 轮，超了就记一条 Reached max continue rounds (10) 的告警。对上层来说，一段完整回答无缝出现。
  • session 粘性：_stream 会把用户 ID 注入成 session_id（common.WithSessionID）。中台按这个 ID 做账号粘性路由——让同一个用户的连续请求尽量落到同一台机器，那台机器上才有你上次的 prompt cache。路由对了，缓存才命中。你看，延迟优化又绕回了缓存。

五、账单撑不住时：两道保命闸（浅尝）

缓存是「省」，但当整个系统被挤爆时，还需要「保」。mino_server 有两道闸，这里只点到为止——它们是通往进阶课「成本支柱」的桥。

过载限流（loadshedding/policy.go）
系统被打爆时主动丢弃一部分请求，用一条 sigmoid 曲线算丢弃比例（ThrottlePercent，defaultSlope = 0.11、defaultMidpoint = 22.0）：错误越多，丢得越狠。但 DecideForMessage 只对免费用户下手——付费用户（subItem.Type != Free）直接放行。而且默认 Enabled = false，靠 TCC 配置临时打开，是个应急阀门，不是常态。
PTU 准入（ptuquota/admission.go）
PTU 是预付的「包月算力」，便宜但有限；PayGo 是「按量现付」，贵但无限。Admit 在发请求前看令牌桶：够就走 PTU，快耗尽（触达 SoftRatio）就排队，等超时再降级 PayGo——宁可多花钱，也不让用户等到断线。

六、自测

Q1：某工程师为了「让日志更精确」，把 <current_time> 从「日期」改成了精确到秒，并且它位于 system 提示词内部（靠前）。上线后账单暴涨、延迟变大。为什么？

因为他亲手击穿了前缀缓存。时间精确到秒，意味着每一次请求的这个字段都不同；而它位于靠前的可缓存区内，根据「严格前缀匹配」，从这个分歧点往后——包括后面所有工具定义、system 剩余部分、历史——全部作废，按全价重算。原本 0.1× 的读缓存，退化成 1.25× 的重新写入。既贵（账单暴涨）又慢（重算耗时，延迟变大）。

正解就是代码里的做法：时间只写到日期粒度（Format("2006-01-02")），放在稳定区里一整天都不变；真需要秒级信息就放到最末尾的易变区，让它只作废它自己后面那一点点。原则不变：稳定在前，易变在后。

Q2：写 1h 缓存要付 2× 的价，比不缓存（1×）还贵一倍。既然这么贵，为什么还要用它？什么情况下反而不该用？

因为 2× 是一次性的写入投资，换来的是之后每一轮都只按 0.1× 读。假设一段开头 10 万 token，会话里被复用 10 轮：不缓存 = 10 × 1× = 10 份成本；缓存 = 1 × 2×（写）+ 9 × 0.1×（读）≈ 2.9 份。轮数越多，回本越快。缓存赌的是「这段开头会被反复读」。

反过来，如果这段内容只会用一次（比如一次性的单轮请求），那 2× 的写入费永远没机会靠 0.1× 的读回本——纯亏，这时就不该缓存，或只用便宜的 5m 档（1.25×）。另外别忘了现实约束：本仓库里 1h 档的 TTL 在 convertCacheControl 里被注释掉了（GCP 不支持），线上实际都按 5m 走——所以现在这个选择题，答案暂时是「想用也用不上 1h」。

我们现在集齐了全部零件：context 是主轴、工具是手脚、评估是 CI、token 是钱。但十二课都是「一个零件看一个零件」。真正的问题是——把这十二个零件同时塞进一次真实请求，它们会互相打架吗？稳定前缀要求「别动」，但工具会动态增删、历史会被压缩、时间会推进……当所有约束一起生效，一台完整的 Agent 到底长什么样？下一课我们把它拼起来看一遍。

这一课记住一句话就够：token 是钱，前缀缓存是省钱的唯一大杠杆，而它的代价是把「稳定在前、易变在后」焊进你的每一个设计决策。有不懂的地方，回会话里问你的 AI 老师，可以要求 ELI5 或直接看 model.go / convert.go 的真实代码。
