一个 Agent 到底是什么

一句话：Agent = 一个大脑（大模型）＋ 一颗心跳（不停问“下一步干嘛”的循环）＋ 一副手脚（能真的动手的工具）。这一课我们只把这三件套讲清楚，别的都往后放。

一、先用生活场景想一遍（ELI5）

想象你有一个特别聪明、但被关在一个小房间里的朋友。房间只有一道投信口：你写一张纸条塞进去，他看完、在纸条背面写好答案，再塞出来。仅此而已——他没有记忆（下次你塞纸条，他完全不记得上次聊过啥），没有手脚（不能上网、不能打电话、不能帮你订咖啡），只会“读文字 → 写文字”。

这个“关在房间里、只会读写文字”的朋友，就是一个裸的大模型（LLM）。你平时用的普通聊天机器人（chatbot），玩法就是最朴素的一种：你塞一张纸条，他回一张，一个来回，结束。

现在我们给这个房间做两个改造，神奇的事情就发生了：

  • 装一颗心跳：门外站一个助理，拿到朋友的答案后不急着交给你，而是看看“事儿办完没？没办完就再塞一张纸条进去问‘接下来呢’”。这样朋友就能连续想好几步，而不是一锤子买卖。
  • 装一副手脚：在房间里装上电话、电脑、打印机。朋友现在可以在纸条上写“帮我搜一下 X”，助理照做，把搜到的东西塞回去给他看——他就能真的动手办事了。

一个能自己连续走好几步、还能真的动手的东西，就叫 Agent（智能体）。这就是它和普通 chatbot 的根本区别。

二、三件套，逐个认门牌号

上面的房间比喻，在真实的 mino_server 里都有对应的代码零件。我们不深挖（那是后面每一课的事），先认个脸、记住门牌号。

🧠 大脑：大模型，一次调用只认这一次的文字

大脑就是大模型本身。它有一个关键性格：无状态（stateless）——每次调用它，它都像失忆一样，只看你这一次发过去的全部文字，不记得任何“上一次”。所以“连续性”不能指望大脑自己，得靠外面那颗心跳把历史一遍遍重新递给它（这正是下一课要解的谜）。

在 mino_server 里，Conductor 主循环每走一步、要“开口问大脑”时，都走 conductorAgent.streamCallLLM（biz/domain/dmservice/agents/conductorservice/agent.go:1545）这个函数。记住这个名字：主循环里每一次“问大脑”都从这道门出去。不过它并不是全仓库唯一的 LLM 调用——生成摘要、标题、分类器、各子 agent 等别处另有多处 LLM 调用，只是不在这条主循环上。

💓 心跳：一个 for 循环，转一圈就是“一步”

心跳是让 Agent “活起来”的东西。它其实朴素得让人意外——就是一个 for 循环。这个循环写在 conductorAgent.streamChat（agent.go）里，每转一圈叫走了一步（step）：问一次大脑、把大脑要的工具跑一遍、看看事办完没有，没办完就再转一圈。

为什么循环要有上限？
万一大脑犯轴、无限地“再来一步、再来一步”怎么办？所以循环设了一道保险丝：单次运行最多走 defaultMaxStepCount = 100 步（agent.go）。到顶还没干完，就主动停下报错，而不是无限烧钱空转。

🖐 手脚：工具，让文字变成动作

光会想没用，得能动手。工具（tools）就是 Agent 的手脚：一个浏览器工具让它能上网，一个代码工具让它能跑程序，一个消息工具让它能把结果发到飞书或 Discord 上。大脑在回复里写“我要调 X 工具、参数是这些”，后端就在 conductorAgent.executeToolCallGroups（toolcall.go）里真的把它跑起来。

这里有个让循环能“滚下去”的关键动作：工具跑完后，结果会被 AddToolResult（message.go）包成一条消息，塞回给大脑当下一轮的输入。工具的输出，变成大脑的输入——这就是心跳能一圈圈转起来的秘密。

三、Agent ≠ Chatbot：差别到底在哪

很多人以为 Agent 就是“更强的聊天机器人”，这其实抓错了重点。差别不在于“答得多聪明”，而在于能不能自己多走几步、能不能真的动手。

用一句话记住：chatbot 是“一问一答”，agent 是“接了活儿自己去干，干完才回来”。多步（心跳）＋ 动手（工具），缺一个都还只是聊天机器人。

四、先看成品：Conductor 长什么样

说了半天三件套，来看一个真家伙。mino_server 里跑在生产环境的这个 Agent 叫 Conductor（“指挥”的意思，很贴切）。用户的一个请求打进来，第一站是它的大门 conductorService.StreamChat（biz/domain/dmservice/agents/conductorservice/service.go）：先 newConductorAgent 把这台 Agent 组装出来，再调 agent.streamChat(ctx) 真正开跑——也就是启动上面那颗心跳。

五、关键定义

Agent（智能体）
一个能自主连续多步、并能调用工具动手的程序。三件套：大脑（LLM）＋ 心跳（loop）＋ 手脚（tools）。对应 mino_server 的 Conductor。
大模型 / LLM
Agent 的“大脑”。无状态：每次调用只认这一次发过去的全部文字，不记得上一次。Conductor 主循环里“问大脑”走 streamCallLLM（agent.go:1545）；仓库别处另有多处 LLM 调用。
工具（tool）
Agent 的“手脚”，把大脑的文字意图变成真实动作（上网、跑代码、发消息…）。在 GetAllToolServices 注册，共 54 个。
心跳循环（agent loop）
驱动 Agent 连续行动的 for 循环（agent.streamChat），转一圈 ＝ 一步(step)，上限 defaultMaxStepCount = 100。
无状态（stateless）
指大模型不自带记忆：连续性必须由外部把历史每次重新递进去，而不是模型自己记住。

六、自测题

Q1：下面哪个才算 Agent，而不只是普通 chatbot？
A. 你问“今天天气咋样”，它凭训练知识答一句就结束
B. 你说“查一下明天北京天气，如果下雨就发飞书提醒我带伞”，它自己去查、判断、再发消息

答案：B。关键不在“答得聪不聪明”，而在两点：能自己多走几步（查→判断→发消息，不是一个来回就停）和能真的动手（调了搜索工具和发消息工具）。A 只走了一步、没动任何工具，就是典型 chatbot——对应“大脑答一次就 agent.done”。B 才是心跳（agent.streamChat 转了好几圈）＋ 手脚（调了 search、message 工具）齐活的 Agent。

Q2：Agent 每走一步，工具跑出来的结果去哪了？为什么这一步是“循环能转起来”的关键？

工具结果会被 AddToolResult（message.go）包成一条消息，塞回消息列表，成为下一轮发给大脑的输入。为什么关键？因为大脑是无状态的——它不会记得上一步工具返回了啥。只有把结果回灌进去，下一圈大脑才“看得见”刚才办事的结果，从而决定下一步干嘛。工具的输出变成大脑的输入，这个回灌动作就是心跳能一圈圈滚下去的燃料；不回灌，循环转第二圈时大脑就是睁眼瞎。

Q3：为什么心跳循环要设 defaultMaxStepCount = 100 这个上限？不设会怎样？

这是一根保险丝。大脑有可能犯轴、陷入“再来一步、再来一步”却永远不收工的死循环。每一步都要真金白银地调用一次大模型（花 token、花时间）。不设上限，一个失控的请求就能无限空转、无限烧钱、还占着资源不放。设成 100 步后，到顶还没干完就主动停下报错——宁可失败，也不失控。这也预告了后面“可靠性 / 编排”那几课要处理的问题。

我们说大脑是无状态的：每次调用它，它只认这一次发过去的文字，完全不记得上一步发生了啥。可上面又说心跳能让它“连续做好几步”——这不矛盾吗？一个失忆的大脑，凭什么能接着上一步往下干？

答案藏在“每一圈到底把哪些文字递给大脑”这件事里。下一课我们就把心跳循环拆开，一圈一圈看它 6 个动作怎么转、又是怎么把历史重新喂回去，让一个失忆的大脑装出“有连续性”的样子。
