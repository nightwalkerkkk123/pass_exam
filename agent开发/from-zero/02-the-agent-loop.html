最小内核：Agent 的心跳循环

上一课你记住了一句话：agent = 大脑 + 循环 + 工具。这节课只干一件事——把中间那个"循环"拆开给你看。看完你会有一个顿悟：大脑其实很笨（每次都失忆），真正让它"会连续干活"的，是它外面那个一圈一圈转的循环。

上一课我们知道了 agent = 大脑 + 循环 + 工具。可"循环"到底怎么转？

更让人困惑的是：为什么普通聊天是你问一句、它答一句，而 agent 却能自己"想一步、做一步、再想一步"，连着做好几件事都不用你插手？这中间到底藏了什么机关？

一、先打个比方：一个记性只有 7 秒的天才厨师

想象你雇了一位料理天赋点满、但记性只有 7 秒的厨师（我们叫他"大脑"）。你只要把菜谱和眼前的食材给他看，他就能准确告诉你"下一步该干嘛"。但你一转身，他就把刚才发生的一切忘得干干净净。

用这样一个"每次都失忆"的厨师，怎么做出一整桌菜（也就是完成一个多步骤的任务）？答案很朴素——你在旁边当那个不失忆的循环：

  1. 你把菜谱 + 现在桌上已有的东西，一起端到他面前（组装）；
  2. 问他："下一步做什么？"（问大脑）；
  3. 他要么说"齐活，上菜！"，要么说"去切葱"（看他要不要你动手）；
  4. 如果要动手，你就真去把葱切了（执行）；
  5. 把切好的葱端回桌上，让他下次能看见（把结果摆回台面）；
  6. 回到第 1 步，再问一遍"下一步做什么"。

厨师永远只回答"下一步"这一个问题，是你这个循环把无数个"下一步"串成了一道完整的菜。这就是 agent 和聊天的根本差别：聊天只跑一次第 1、2 步就停了；agent 把这 6 步圈起来自动转，直到厨师说"做完了"。

本课要焊进脑子的一句话：无状态的大脑（每次调用都从零读起、答完就忘）+ 一个在外面反复调用它的循环 = 一个会自己干活的 agent。大脑负责"想"，循环负责"记住 + 反复问 + 动手"。智能来自大脑，能干活来自循环。

二、把比方翻译成真代码：一圈 = 一个 step

现在把厨房搬进 mino_server。心跳循环的本体是一个函数 conductorAgent.streamChat（文件 biz/domain/dmservice/agents/conductorservice/agent.go）。它的核心就是一个 for 循环，每转一圈，计数器 runStepCount 加一——这个"一圈"就叫一步（step）：

for agent.runStepCount = 0; agent.runStepCount < agent.maxStepCount; agent.runStepCount++ {
    // (a) 余额够不够？（第 2 圈起才查，第 1 圈放行）
    if agent.runStepCount > 0 && enforceCreditsInterception {
        if err := agent.checkCreditsExhausted(ctx); err != nil { return err }
    }
    // (b) 上下文太长就先压缩一下（下一课的话题，这里先放着）
    if err := agent.summarize(ctx); err != nil { return err }
    // (c) 真正跑一步：问大模型 + 执行它要的工具
    if err := agent.runConductorStep(ctx); err != nil { /* 按错误类型 continue / 压缩 / return */ }
    // (d) 大脑说"做完了"？收工
    if agent.done { return nil }
    // (e) 把用户中途插进来的新消息接上，进入下一圈
}
// 转满上限还没 done：熔断报错
if !agent.done {
    return errno.NewWithCtx(ctx, errcode.ErrAgentMaxStepCountReached, "max step count reached")
}

看清楚了吗？比方里那 6 步，和这段 for 循环是一一对应的。而每一圈里真正"想 + 做"的活，都被塞进 (c) 那一行的 runConductorStep 里。我们把它再拆成两半：

前半"开口问"——主循环里只有一个函数真的在跟大模型说话。在 agent 的主推理循环里，每一步真正把消息发给大模型的就是 conductorAgent.streamCallLLM（agent.go）；仓库别处（上下文压缩、记忆抽取、标题生成等）另有各自的 LLM 调用。它长这样：

options := []model.Option{
    common.WithThinking(thinkingTokens),
    common.WithEnableParallelToolCalls(true),   // 允许模型一次要好几个工具
}
toolInfos := agent.getCurrentToolInfos()
options = append(options, model.WithTools(toolInfos))   // 工具清单"带外"传进去，不写进 messages
llmStream, err := agent.llmClient.Stream(ctx, messages, options...)   // ← 真正发给大模型就这一行
// ...一块块（流式）接收：思考内容 / 正文 / 工具调用，最后合并成一条完整回复...

这里先埋两颗种子（后面的课会展开）：第一，工具清单是用 model.WithTools 当作 API 参数带外递给模型的，不塞进 messages 数组里——这正是 agent 和普通聊天在接口上的关键差别。第二，回复是流式一块块回来的，最后由 StreamingMessageProcessor 合并成一条完整消息返回。

中间"看回复"——循环滚不滚得下去，全看这一句。拿到大模型的回复后，runConductorStep 做一个极简单却极关键的判断：

msg, err := agent.determineNextStep(ctx)   // 内部：buildConductorMessages → streamCallLLM → AddMessage
if err != nil { return err }
agent.lastAgentMessage = msg

if len(agent.lastAgentMessage.ToolCalls) == 0 {   // 大脑这次没要任何工具
    agent.done = true                             // = 它认为事情办完了
    return nil
}
// 否则：把这一批工具分组、过策略、真正执行
groups, err := agent.categorizeToolCalls(ctx, agent.lastToolCalls)
groups = agent.applyToolCallPolicies(ctx, groups)
err = agent.executeToolCallGroups(ctx, groups)   // 执行 + AddToolResult 把结果塞回消息

翻译成人话：模型的回复里还带着"我要调工具"的请求，就继续；不带了，就说明它觉得任务结束了。这就是 agent 判断"该不该再走一步"的全部逻辑——朴素得让人意外。

后半"动手做 + 把结果摆回台面"——这是循环能闭合的秘密。工具跑完，结果不会凭空进入下一轮。AddToolResult（文件 message.go）把每个工具的输出包成一条 Role = common.Tool 的消息，append 进 agent.newMessages：

toolMessage := &minomessage.Message{
    Role:     common.Tool,
    Contents: []minomessage.MessageContent{ messageContent },
}
agent.newMessages = append(agent.newMessages, toolMessage)

下一圈开头 buildConductorMessages 重新拼消息时，会把 agent.newMessages 里这些新东西一起读回去（转换逻辑在 context.go 的 newConductorBasicMessages）。于是上一步的工具结果，变成了下一步大脑的输入——就像切好的葱被端回了台面。这正是那个"失忆厨师"能连续做菜的唯一原因：不是他记住了，是循环替他把台面维护住了。

三、循环怎么知道该停？两道保险

如果只会"继续、继续、继续"，agent 就成了停不下来的机器——要么陷入死循环无限切葱，要么把你的账单烧穿。所以循环设了两道明确的出口，外加几个中途急刹。

出口一（正常）：大脑说"够了"。只要某一步的回复里 len(ToolCalls) == 0，runConductorStep 就把 agent.done 置为 true；回到循环体，if agent.done { return nil } 直接优雅退出。这是绝大多数任务的收尾方式。

出口二（兜底）：转太多圈了。for 循环的条件是 runStepCount < maxStepCount。这个上限来自常量 defaultMaxStepCount = 100（agent.go），也就是说一个任务最多允许大脑"想 + 做"100 个来回。特殊品类可以由 hooks 覆写（代码注释里举了 AnyClaw = 200 的例子）。转满了还没 done，循环外那句 if !agent.done 就会抛出 ErrAgentMaxStepCountReached。这不是 bug，是保险丝：宁可报错停下，也不让一个跑飞的 agent 无限烧钱。

除了这两道正门，中途还有几个"急刹"：余额被 checkCreditsExhausted 判定耗尽会直接 return；上下文太长（ErrLLMPromptTooLong）不会崩，而是先压缩再 continue 重来一圈——这些"自愈"手法留到第 08、09 课细讲。此外，工具本身也可以是"终结型"的：executeToolCallGroups 里如果某个工具结果带 IsFinish，同样会把 agent.done 置 true，让循环收尾。

四、关键定义

心跳循环（agent loop）
Agent 的最小内核。一个反复"组装消息 → 问大模型 → 看要不要工具 → 执行工具 → 结果塞回消息"的外层循环，本体是 conductorAgent.streamChat 里的 for。它让"每次失忆的大脑"能连续完成多步任务。
一步（step）
循环转的一整圈，计数器 runStepCount 加一。一步 = 一次大模型对话回合（一次 streamCallLLM）+ 这次回合里它要求的全部工具的执行。模型一次可以要多个工具（开了 WithEnableParallelToolCalls），这一批工具都在同一步内跑完。
done
一个布尔开关。当某一步大模型不再要求任何工具（len(ToolCalls) == 0），或某个终结型工具返回时，agent.done 被置 true，循环随即正常退出。它就是循环的"停止信号"。
maxStepCount / ErrAgentMaxStepCountReached
循环的硬上限，默认 defaultMaxStepCount = 100（可被 hooks 覆写，如 AnyClaw 200）。转满还没 done 就抛 ErrAgentMaxStepCountReached——防跑飞的保险丝。

五、自测题

Q1：同样是调用大模型，为什么普通聊天"你问一句它答一句就停"，而 agent 能自己连做好几步？机关到底在哪一层？

机关不在大模型里，而在大模型外面那个循环。聊天应用调用一次模型、拿到回复、显示、结束——只跑了"组装 + 问"这两步。agent 把这套动作放进 streamChat 的 for 循环：每拿到一次回复就检查"里面还要不要调工具"，要就执行、把结果塞回消息、再问一遍，直到模型说"不用了"。

所以"会连续干活"是循环的功劳，不是模型变聪明了。模型始终是那个无状态、答完就忘的大脑；是循环替它维护台面（消息列表）、反复调用它。这就是本课最该带走的顿悟：无状态大脑 + 外部循环 = 会干活的 agent。

Q2：某一步，大模型一口气要求同时调用 3 个工具（搜索、读文件、算数）。这算走了几步（step）？工具结果什么时候能被模型"看见"？

算一步。一步 = 一次大模型回合 + 这次回合里它要的全部工具的执行。因为开了 WithEnableParallelToolCalls(true)，模型可以在一次回复里要求多个工具；这一批工具会被 categorizeToolCalls 分成并行/串行组，在同一步内全部跑完。

3 个工具的结果都由 AddToolResult 塞进 agent.newMessages 后，循环才 runStepCount++ 进入下一步——也就是下一次 streamCallLLM。所以模型要到下一步开口时，才第一次"看见"这 3 个工具返回了什么。记住这条时序：本步执行工具 → 塞回消息 → 下一步大脑才读到。

Q3：如果一个 agent 因为任务太复杂，转了整整 100 圈还没得出"done"，会发生什么？为什么要这么设计？

for 循环条件 runStepCount < maxStepCount（默认 100）不再成立，循环结束；循环外的 if !agent.done 命中，抛出 ErrAgentMaxStepCountReached。任务以报错告终。

这么设计是因为大模型不保证一定会收敛——它可能陷入"调工具→再调工具→再调工具"的死循环，或钻进无意义的兜圈子。每一圈都要真金白银地调用一次大模型（花 token = 花钱）。maxStepCount 是一根保险丝：宁可停下来报错，也不让一个失控的 agent 无限消耗资源与预算。这也是"能干活"之外，生产级 agent 必须补上的"不失控"——正是后面模块 3 的主题。

循环每转一圈的第 1 步都是"组装消息"（buildConductorMessages）——把该带给大脑的东西拼成一份发过去。可这份"东西"里到底装了什么？系统人设、历史对话、上一步的工具结果……它们按什么顺序摆放，大脑才读得懂、读得对？

这份被反复组装、又决定一切的东西，有个名字叫 messages。下一课我们就钻进 buildConductorMessages，看清喂给大脑的到底是什么。

这一课你拆开了 agent 的最小内核：一个在 streamChat 里转圈的循环，靠 done 停、靠 maxStepCount 兜底。下一课（03）：从聊天到 messages——喂给大脑的那份"消息清单"到底长什么样。读到卡壳，回会话里问你的 AI 老师，可要求 ELI5 或直接看真实代码。返回 课程门户。
