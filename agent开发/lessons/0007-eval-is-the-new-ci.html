eval 是新的 CI：统计性 ≠ 确定性

你已经有了正确的地图："eval ≈ CI"。这个类比帮你想到"回归测试""判分""通过率"，方向没错。这节课只做一件事：带你走进 cmd/memory_eval 的真实代码，指出这条类比在四个地方漏水——那四个地方，是服务端世界里根本不存在、只有 Agent 世界才冒出来的现象。

一、先承认类比对的部分

把 memory_eval 当成 memory 子系统的回归测试，是完全正确的起点。它固定一组输入、跑被测系统、给输出判分，通过率退步就报警——这套骨架和 CI 一模一样。

但先看清它不是什么。它不是 go test，不测单函数。它是一个独立的命令行程序：go build -o output/memory_eval，入口 cmd/memory_eval/main.go 整个只有一行实质代码——

func main() {
	if err := eval.Run(context.Background()); err != nil {
		fmt.Fprintf(os.Stderr, "Error: %v\n", err)
		os.Exit(1)
	}
}

cmd/memory_eval/main.go:11-15

eval.Run（runner.go:49）做的事，服务端工程师一眼就懂：读配置 → loadDataset 反序列化数据集 → filterCases 过滤 → 建 gateway 客户端 → runAll 串行跑批。真正会打脸的，是"跑批"里每一条到底在干什么。

二、四个漏水点

漏水点 1：被测对象不确定

CI 里被测的是你的确定性代码：同一输入进函数，一万次出同一个值。memory_eval 打的却是真实的 Conductor 对话——它建独立 chat、发 /new 让 AI 从 VFS 重载 MEMORY.md、再发真正的 user_message，然后轮询等回复（sendAndCollect）。被测的是 claude-sonnet-4-5，同一句话两次跑，回复可能不同，甚至调用不同的工具。测试的输入端就已经不可复现了。

这还带来一个 CI 里没有的成本结构：为保证用例间隔离，每条用例先 clearAllMemory，再硬 sleep 60 秒等消息队列把向量索引异步删干净——

	// Clear all existing memory before each test case for isolation
	if err := r.clearAllMemory(); err != nil { ... }

	// Wait for vector index deletion to complete (MQ event processing has delay)
	vectorDeleteWait := 60 * time.Second
	...
	time.Sleep(vectorDeleteWait)

cmd/memory_eval/eval/runner.go:110-121

单条用例 = 60s 删向量 + 5s 等索引传播 + 两轮 gateway 往返 + 一次裁判 HTTP，耗时以分钟计。eval 天生是慢速的持续观测，不是 CI 里那种秒级红绿。（顺带一提：config.go 定义了 concurrency 字段且默认 1，但 runAll 是纯串行 for 循环，从未用它——所以它现在就是顺序跑。）

漏水点 2：判分器本身是一个模型

CI 的判分是一行 assertEqual：确定、廉价、可复现。memory_eval 的判分核心却是另一次模型调用。score（scorer.go:33）算三个分量：containsScore（确定性子串匹配）、toolCallScore（是否调了写文件工具）、llmJudge（现场 POST 给 gemini-2.5-flash，要它只回 {score, reasoning} 的 JSON，解析出 0.0–1.0）。裁判是模型，裁判的输出也会漂。

三个分量怎么合成一个分？看 weightedScore——这是全课最该盯住的十行代码：

func (s *scorer) weightedScore(category string, detail scoreDetail) float64 {
	switch category {
	case "recall":
		return detail.ContainsScore*0.3 + detail.LLMJudgeScore*0.7
	case "store":
		return detail.ToolCallScore*0.5 + detail.LLMJudgeScore*0.5
	case "update":
		return detail.ToolCallScore*0.4 + detail.LLMJudgeScore*0.6
	case "rejection":
		return detail.ContainsScore*0.2 + detail.LLMJudgeScore*0.8

cmd/memory_eval/eval/scorer.go:333-343

为什么用跨厂商的 gemini-2.5-flash 给 claude-sonnet-4-5 的输出打分，而不是自己评自己？为了避开自评偏袒——同家模型给自己打分容易手软。这本身就是一个 CI 里不存在的设计决策：你得担心"裁判可不可信"。

漏水点 3：PASS 是"阈值化"，不是"断言成立"

CI 里一条 assert 挂了，build 就红——判定是二值的，成立或不成立。memory_eval 的 PASS 是在一个连续分布上切一刀： 
		status := "PASS"
		if result.Score < 0.6 {
			status = "FAIL"
		}
		fmt.Printf("[%s] %s (%.2f) - %s\n", status, tc.ID, result.Score, tc.Description)

cmd/memory_eval/eval/runner.go:91-95

0.6 是一个人为选的阈值，不是自然分界。为什么不要求 score == 1.0 的精确相等？因为分数是个混合浮点量：contains_all 按 matched/total 给部分分，裁判给 0.0–1.0，还可能兜底成 0.5。在这种量上谈"精确相等"没有意义——你能做的只是选一个阈值把分布切成 pass/fail 两桶。阈值本身是可调的政策，不是真理。

漏水点 4：产出是统计量，且判分链路可降级

CI 的产出是一个比特：build 红或绿。memory_eval 的产出是一批用例的统计聚合——每类的均分与通过率： 
		avg := sum / float64(len(catRes))
		r.Categories[cat] = categoryStats{
			Count:    len(catRes),
			AvgScore: avg,
			PassRate: float64(pass) / float64(len(catRes)),
		}

cmd/memory_eval/eval/report.go:60-65

更彻底的漏水在这里：判分链路是软逻辑，可以降级。没有裁判 API key 时，llmJudge 不报错，而是直接兜底成 0.5——

	if s.cfg.apiKey == "" {
		return 0.5, "LLM judge skipped (no API key)"
	}

cmd/memory_eval/eval/scorer.go:196-198

0.5，既不通过也不失败。断言的世界里根本没有"0.5 通过"这种状态——assert 不会因为"环境没配好"就给你半分。这一行把 eval 和 CI 的本质差异钉死了：eval 度量的是一个会漂移的分布，CI 断言的是一个确定的事实。

三、但他们没有丢掉 CI 式的确定性

注意一个容易看漏的克制：CI 那套没有被抛弃，只是被降权了。containsScore 那层纯确定性的子串断言（contains_all/contains_any/not_contains）仍然在跑，只是权重被压到 0.2–0.3。为什么留着？因为对"关键实体有没有出现"这种事，字符串精确匹配又快又可复现，是最靠谱的信号。真正判不了的是"语义对不对""有没有幻觉"——那部分才交给裁判。确定性能判的继续用断言，判不了的才上统计。

所以正确的心智模型不是"用 eval 替换 CI"，而是"在确定性断言之外，叠一层统计性判分来覆盖语义与幻觉"。这也解释了数据集里 expected 为什么同时长着两种字段：确定性的 contains_all/should_call_write_file，和统计性的 judge_prompt/exact_answer（dataset.go:10-32）。

五、关键定义

eval（评估集）
一组固定输入 + 判分逻辑，用来持续观测模型输出质量。与单元测试的根本差异：判分是统计性的（浮点分布 + 阈值分桶），不是确定性 assert。
判分器（scorer）
把模型输出映射成分数的逻辑。memory_eval 用三类合成：确定性的包含匹配（containsScore）、代码判分（toolCallScore，是否调对工具）、模型裁判（llmJudge）。载体：cmd/memory_eval/eval/scorer.go。
LLM-as-judge（模型裁判）
用一个 LLM 给另一个 LLM 的输出打分。核心风险是裁判可信度与自评偏差；memory_eval 用跨厂商的 gemini-2.5-flash 评 claude-sonnet-4-5 来规避偏袒。
通过率漂移（pass-rate drift）
同一 eval 集的通过率随模型/prompt/代码变化而移动，是 Agent 系统的"回归信号"。你监控的是 AvgScore 与 PassRate 这条线，不是单条用例的红绿。

六、自测题

Q1：把 recall_fact_001（问"我的项目 PSM 是什么"，期望回复含 lark.mino.server）用 memory_eval 连跑两次，两次的最终 score 一定相等吗？结合具体代码说。

不一定相等。最终分 = weightedScore("recall") = ContainsScore*0.3 + LLMJudgeScore*0.7（scorer.go:336-337）。其中 ContainsScore 是对回复做子串匹配（命中 lark.mino.server 即 1），这层是确定性的；但占 70% 的 LLMJudgeScore 是现场调 gemini-2.5-flash 打出来的，而且被测的 claude-sonnet-4-5 回复本身也会变。所以那 0.7 的部分两次很可能不同，最终浮点分会漂。这正是为什么要看 PassRate/AvgScore（report.go:60-65）这个分布量，而不是把一次跑当成 assert。

Q2：为什么 PASS 被定义成 score >= 0.6（runner.go:92），而不是 score == 1.0 的精确相等？

因为分数是一个连续的混合浮点量：既有按 matched/total 给部分分的关键词匹配，又有 0.0–1.0 的裁判分，还可能在无 key 时兜底成 0.5（scorer.go:196-198）。像"不许幻觉""语义有没有覆盖十几条应捕获的观察"这种判据，用字符串精确相等根本判不了。0.6 只是把一个分布切成 pass/fail 桶的政策阈值——eval 度量的是通过率这个统计量，不是给出唯一正确答案的证明。这就是"eval ≈ CI"这条类比最深的漏水处：CI 断言二值事实，eval 阈值化一个分布。

本课的一手来源，值得完整读一遍：Your AI Product Needs Evals — Hamel Husain。带着一个问题去读：文章反复强调"先做错误分析（error analysis），别先堆基础设施"——回到 memory_eval，它落盘的完整 JSON 报告（含每条的 Response、ToolCalls、JudgeReasoning）和 results/index.html 人工翻看界面，正是"人工看几十条真实输出"的地基。eval 不是先证明正确，而是先看清楚在哪里错。
