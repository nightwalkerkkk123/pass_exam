四支柱耦合总复盘 + 项目改进清单

前面几课我们分别拆过质量（注意力/位置）、成本（prefix cache 排列）、评估（context 可复现性）、工具（返回值占预算）。这节课收网：证明这四条线不是四个并列的问题，而是同一个中心——conductorAgent 的 context 组装——投影出的四张脸。然后把 mino_server 里那几处代码亲手写下的 FIXME / TODO 收成一张改进清单。

一、中心装置：streamChat 主循环的一个 step

先把中心机制看清楚。整个 Conductor 的运行就是 streamChat（agent.go:900）里的一个 for 循环（agent.go:916），每一步固定三拍：先主动压缩，再跑一步，出错就紧急压缩后重跑。

拍③执行完工具后，工具返回值会被塞回 newMessages，成为下一次 LLM 调用输入的一部分。如果这一塞把 context 撑爆，就会抛出 PromptTooLong，主循环在拍④接住它：若还没有可用摘要就 summarizeForce（强制压缩，跳过条件检查），再 applySummaryImmediately 把 originalMessages 清空、用摘要替换历史（summary.go:406），然后 continue 用瘦身后的 context 重跑这一步。

注意一个刻意的设计：requestAgentMessage 的 retry 策略遇到 PromptTooLong 特意 return false 不重试（agent.go:1038–1041），把它让给主循环去压缩。这条注释写得很直白：

retry.RetryIf(func(err error) bool {
    agent.lastAgentErr = err
    if errcode.IsErrLLMPromptTooLong(err) {
        // context too long: return error immediately,
        // the main loop will handle summary compression
        return false
    }

↑ agent.go:1036–1041：重试对 PromptTooLong 主动放行，交给主循环压缩。

二、四支柱都是这张图的投影

质量：压缩线不是"窗口满"，而是 80% 且封顶

质量支柱的量化锚点在 summary.go:86–92。压缩触发线不是等窗口填满，而是取"窗口 80%"和"品类上限"的较小值——注释明说是为了防"降智"：

// 取上下文窗口 80% 和上限的较小值，避免超大上下文窗口的模型使用了太大的上下文窗口导致降智
// IM 品类上限 100K，其他品类上限 200K
summaryContextCap := 200_000
if subType == enums.SubType_AXON_IM {
    summaryContextCap = 100_000
}
summaryContextLine := min(int(float64(agent.llmClient.GetModelContextWindow())*0.8), summaryContextCap)

↑ summary.go:86–92。触发判断在 summary.go:105：TotalTokens < summaryContextLine 才放行不压。

成本：一个动态字段就打破 prefix cache

成本支柱的锚点是 prefix cache 的严格前缀匹配。buildContextPayload 里有一段自白式注释，亲口承认 context 组装里一个动态字段 <current_plan> 会毁掉缓存命中：

// 计划相关信息（仅在启用了计划工具时才注入）
// 注意：<current_plan> 在第 1→2 次 LLM 调用间会变化（IsLastMessageFromUser 从 true→false），
// 导致 context_info 改变、prefix cache miss。不需要 plan 提示的 Agent（如 Slides）
// 应在 CustomizeContextPayload hook 中清除这些字段。

↑ agent.go:1524–1527。赋值在 agent.go:1530：IsLastMessageFromUser = len(agent.newMessages) == 0；渲染在 context.go:270：只有它为 true 才写 <current_plan>。

从 Redis 读一个 key，命中与否只由 key 决定，值放在哪不影响命中。这里的 prompt cache 命中与否，由 context 里一个 <current_plan> 动态字段决定——而这个字段的值取决于 step 状态（len(newMessages)）。所以缓存问题的根因在组装逻辑，不在缓存层。修法不是去调缓存，而是把 plan 提示挪到 cache break 之后、或在 CustomizeContextPayload hook 里清掉它（代码注释 agent.go:1526–1527 自己也是这么建议的）。

评估：裁判是另一个厂商的非确定性模型

评估支柱的锚点是离线 harness cmd/memory_eval。它的判分器三维打分：ContainsScore（关键词）+ ToolCallScore + LLMJudgeScore（scorer.go:14–18）。关键在配置：被测模型和裁判模型来自不同厂商。

fs.StringVar(&cfg.model, "model", "claude-sonnet-4-5", "model to evaluate")
fs.StringVar(&cfg.judgeModel, "judge-model", "google/gemini-2.5-flash", "model for LLM judge (OpenRouter model ID)")

↑ config.go:41–42：被测方 claude-sonnet-4-5（Anthropic），裁判 google/gemini-2.5-flash（Google，走 OpenRouter）。report.go:89–90 也只有在 r.Model != r.JudgeModel 时才多打一行 Judge——代码本身就承认"被测方与裁判方常常不是同一家"。

裁判用 OpenAI 兼容格式打到 OpenRouter，system prompt 强制要求只返回 {"score", "reasoning"} 的 JSON（scorer.go:151–159）。跨厂商是刻意的：避免自家模型给自家输出打分的自评偏袒（见 GLOSSARY 的 LLM-as-judge 条）。代价是——eval 的可复现性同时依赖钉死 context 和钉死 judge model，任何一个动了，通过率就会漂。

工具：返回值占预算，压缩链是唯一的回收阀

工具支柱的锚点在 toolcall.go。这里藏着本课最锋利的一处证据：串行和并发两条路径对 PromptTooLong 的处理不一致。串行路径把它当一等公民——保留已产出的 partial 结果，把原始 PromptTooLong 原样上抛：

result, err := agent.executeToolCall(ctx, toolCall, agent.eventStream)
if errcode.IsErrLLMPromptTooLong(err) {
    if result != nil {
        toolResults = append(toolResults, result)
    }
    return toolResults, err
}

↑ toolcall.go:285–291（executeSerialTools）。原样上抛 PromptTooLong，让主循环的紧急压缩接住。这是"正确姿势"。

并发路径做不到。goroutine 里遇到同样的错误，只能转成 ErrInternalServer——这就是那条 FIXME：

if errcode.IsErrLLMPromptTooLong(err) {
    // FIXME: 并发调用时要处理这种报错非常麻烦，且目前暂时没有并发工具可能出现这种报错，暂不处理
    return nil, errno.Errorf(errcode.ErrInternalServer, "unexpected context too long error: %+v", err)
}

↑ toolcall.go:333–335（executeParallelTools）。PromptTooLong 被掩埋成 ErrInternalServer，压缩链再也识别不出它。并发度常量：maxParallelToolCount=5、maxParallelSubAgentsToolCount=3（toolcall.go:305–306）。

在 RPC 世界里，工具 ≈ 一次远程调用。子调用的返回值再大，也只是一个响应体，它不会反过来撑爆调用方的输入预算——调用方读完就丢。可 Agent 里工具返回值要塞回 context，参与下一次 LLM 调用；撑爆时唯一的回收办法是压缩整段 context。整个错误恢复系统的本质就是"context 溢出 → 压缩 context → 重试"，而这条中心机制靠识别 PromptTooLong 这一个错误类型来触发。并发路径把错误类型改成了 ErrInternalServer——于是它成了整张压缩图唯一够不到的角落。这就是为什么它是 FIXME，而不是随便一个 TODO。

三、类比漏水点总账

把四个漏水点并排放。你会发现它们不是四个独立缺陷，而是同一张 context 组装图上的四个投影点——每一处都能从"context 怎么组装/怎么压缩"推导出来。这正是这门手艺叫 context engineering 的原因。

支柱（你的服务端类比）类比对的部分漏水点（真实锚点）
工具 ≈ RPC 幂等、超时、参数校验、错误回注——都成立 子调用返回值会反噬调用方的输入预算，撑爆只能压整段 context；并发路径接不进压缩链（toolcall.go:334 FIXME）
context ≈ 缓存 无状态、随请求带全量、可淘汰——都成立 命中与否由 context 里一个动态字段决定，根因在组装逻辑不在缓存层（agent.go:1524–1527）
token ≈ 计费单位 按量计费、缓存读写有折扣——都成立 价目表自认"不准确""分段计费未处理"，成本地基可疑（model.go:501 TODO、model.go:530 FIXME）
eval ≈ CI 固定输入 + 判分 + 回归信号——都成立 裁判是另一厂商的非确定性 LLM，可复现性同时依赖钉死 context 和 judge model（config.go:41–42）
本课唯一要带走的判断：遇到 Agent 系统的任何一类问题——质量抖、账单涨、eval 飘、工具炸——先回到那张 context 组装图问一句"这份 context 是怎么拼的、怎么压的"。四个支柱的大部分决策都会从这张图上自动推导出来。

五、关键定义

紧急压缩重试（emergency compaction retry）
主循环捕获 PromptTooLong 后 summarizeForce + applySummaryImmediately 压缩 context 再 continue 重跑该 step 的恢复机制（agent.go:937–949）。它靠识别错误类型触发——所以任何把 PromptTooLong 改写成别的错误类型的路径，都会绕过它。
主动压缩线（proactive compaction line）
min(0.8×模型窗口, cap)，cap 其他品类 200K、IM 品类 100K（summary.go:88–92）。不等窗口满，提前压，目的是防"降智"。
前缀漂移（prefix drift）
step 间"稳定前缀"因动态内容而 byte 不一致，导致 prompt cache miss。典型源头是依赖 step 状态的字段（如 <current_plan>）进了缓存区（cache_prefix_stability_test.go 头注）。
跨厂商裁判（cross-vendor judge）
LLM-as-judge 里刻意让裁判模型与被测模型来自不同厂商，规避自评偏袒（config.go:41–42：Claude 被测 / Gemini 裁判）。

六、自测题

Q1：串行 executeSerialTools 遇到 PromptTooLong 会返回"已产出的 partial 结果 + 原始错误"（toolcall.go:285–291），并发 executeParallelTools 遇到同样错误却返回 ErrInternalServer（toolcall.go:333–335）。为什么这个差异恰好证明了"context 组织是中心"？

因为主循环的紧急压缩恢复（agent.go:937–949）是靠识别 PromptTooLong 这个错误类型触发的——捕获到它就 summarizeForce + applySummaryImmediately 压缩 context 后重跑。串行路径原样上抛，恢复链能接住；并发路径把错误类型改成了 ErrInternalServer，恢复链识别不出，压缩不会发生，用户直接吃到 500。

整个错误恢复系统本质上就是"context 溢出 → 压缩 context → 重试"这一条中心机制，它唯一够不到的角落正是并发工具——所以那里才是 FIXME。这个差异证明四支柱的行为全从 context 组装/压缩这张图推导出来。

Q2：buildContextPayload 里 IsLastMessageFromUser = len(agent.newMessages) == 0（agent.go:1530），只有它为 true 时 context.go:270 才渲染 <current_plan>。为什么这一行会伤害"成本"支柱，而这又说明缓存问题的根不在缓存层？

step 0 时 newMessages 为空 → IsLastMessageFromUser=true → context_info 里有 <current_plan>；step 1 工具跑完后 newMessages 非空 → 翻成 false → <current_plan> 消失。这段文本处在 cache break point 之前的"稳定前缀"里，前缀一变 byte 就不再 identical，prompt cache miss，成本和延迟双升。

根因是 context 组装逻辑往稳定前缀里塞了依赖 step 状态的动态内容，而不是缓存本身失效——所以修法是把 plan 提示挪到 cache break 之后、或在 CustomizeContextPayload hook 里清掉（代码注释 agent.go:1526–1527 也是这么建议的）。这就是"缓存问题的根在组装层"的活例。

Q3（综合）：产品同学反馈某个新上线的 Agent"账单比预期贵一截、偶发 500、eval 通过率还比测试时低"。用本课的中心图，你会先查哪三处代码，各对应哪个支柱？

账单贵 → 成本支柱 → 查 buildContextPayload 是否往稳定前缀塞了动态字段（agent.go:1524–1530），以及价目表是否踩中 model.go:501/530 的不准价。

偶发 500 → 工具支柱 → 查是否触发了并发工具的 PromptTooLong 掩埋（toolcall.go:333–335），恢复链没接住。

eval 通过率低于测试时 → 评估支柱 → 查 eval 的 context 与 judge model 是否被钉死（config.go:41–42）；judge 换了或 context 组装变了，通过率就会漂。三处根子都在同一张 context 组装图上——这正是本课的中心判断。
