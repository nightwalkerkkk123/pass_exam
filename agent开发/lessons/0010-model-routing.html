模型路由：主模型、fallback 兜底、廉价模型门控

你熟悉的 RPC 路由：按服务身份、负载、权重把请求分发到实例，路由表是静态配置，规则可穷举，失败时按注册的实例列表重试或熔断。模型路由长得很像——但它多出一维你的路由表里从来没有过的东西：任务难度判断。这一维不是查表能得到的确定答案，而是被 eval 和灰度实验反复校准的经验产物。这节课就沿着 conductorservice 里真实的三层路由，看这维"难度判断"如何把 RPC 类比一处处顶漏。

一、先承认类比对的部分

把模型当成一组可替换的后端实例，是完全正确的起点。请求要分发到某个后端（选哪个模型）；某个后端过载或挂了，要按预先注册的列表切到备用（fallback）；不同后端有不同成本，你想把便宜的请求路到便宜的后端。这三件事——选型、兜底、成本分层——在 RPC 世界里你天天做。

代码结构也确实这么组织：路由决策收敛在 utils/model_policy.go 和 llms/common/model.go 两处，前者管"选谁"，后者是模型元数据的事实源。model_policy.go:11 的注释把这条纪律说得很直白：

// 业务调用点不应硬编码具体模型名。各档模型的选型 / 灰度 / 后续降级逻辑统一收敛到本文件。

这跟"不要在业务代码里写死下游 IP，走服务发现"是同一个工程直觉。方向没错。但接下来是四个 RPC 路由里不存在的现象。

二、第一层：主模型不是查表选出来的，是灰度实验选出来的

PickPrimaryModel（model_policy.go:23-62）决定"这个用户这次用哪个主推理模型"。它的优先级链是：

优先级条件选中的模型代码位置
1命中 AB 实验 cohort，且返回的模型名合法AB 决议的模型L31-37
2IsEnableGPT55 FG 开GPT-5.5L40-42
3付费/免费桶命中 IsEnableGPT54PTUPaid/Free FGGPT-5.4L50-59
4（默认）以上都不命中，或依赖缺失（ctx/user/fgItems 为空）GPT-5.2（fail-closed）L24-26, L61

注意这条链的形状：没有一条规则是"某类请求永远落到某个模型"这种静态映射。每一档都挂在一个 feature gate 或 AB 实验上。哪个用户用 GPT-5.5、哪个用 GPT-5.4、新用户和长期免费用户走什么，全部由灰度配置控制（L21 注释：新用户/长期免费的差异化"完全由 AB 实验控制…不在本函数里写硬规则"）。PickPrimaryModel 是一个把"运营/实验此刻的决议"翻译成模型名的函数，不是一张路由表。

RPC 路由表是确定且可穷举的：给定服务名和参数，你能静态推出请求落到哪类后端。模型主档不是——"这个用户该用哪个模型"是一个正在被 AB 实验持续调整的经验问题，答案随灰度桶天天变。所以这里的"路由表"本质是 feature gate 的投影，而不是拓扑配置。这也是为什么代码要 fail-closed 回 GPT-5.2（L47、L61）：AB 抖动、运营 typo、依赖 RPC 失败都可能让"该选谁"变得不确定，此时宁可回到一个已知稳定的档，也不把奇怪字符串透传给 LLM 调用层（L18-19 注释）。

三、第二层：便宜模型门控——省钱的旁路，但便宜不是自动便宜

主推理循环之外有一大批小任务：判断一个事件"该不该触发 Agent"、把一段内部 thinking 改写成给用户看的进度文案、生成标题/摘要/用户画像。这些任务不值得动用贵的主模型，代码里直接硬选廉价档。先看价格为什么值得这么做——同样每 1M output token：

实例 1 · trigger 语义过滤。判断"该不该触发 Agent"是个二分类，用最便宜的 Nano。但代码里最锋利的一行是它关掉了推理（trigger_judge.go:19-21）：

// triggerJudgeModel 触发器语义过滤统一模型。nano 是推理模型，judge 为二分类任务，
// 必须显式关闭推理（reasoning effort none），否则延迟与输出 token 抵消价格优势。
var triggerJudgeModel = llmcommon.ModelNameGPT54Nano

对应的调用处显式把 reasoning 关成 none（trigger_judge.go:84）：

Reasoning: &shared.ReasoningParam{Effort: shared.ReasoningEffort("none")},

而且任何失败都返回 (false, "")——"宁可漏过，不为错误事件跑完整 Agent"（trigger_judge.go:33 注释）。这是 fail-open：门控出错时选择放过，而不是误触发烧掉一整个 Agent 回合。

实例 2 · slides thinking 脱敏改写。slides 模式下 Conductor 的 thinking 会直接讨论幻灯片底层协议（SVG 命名空间、私有属性），不能原样透给用户。方案是用 Nano 把整段 thinking 改写成一句安全的进度文案，MaxTokens 只给 256（slides_thinking.go:64-95）。设计注释（slides_thinking.go:14-17）：

// slides 模式下，conductor 的 thinking 会直接讨论幻灯片底层实现协议（SVG slide:* 命名空间...
// 原始 thinking 推给前端，而是把整段 thinking 用便宜模型改写成一句「在做什么」的进度文案，
// 并对协议关键词做兜底过滤——即使小模型改写不彻底，也不会把协议泄漏出去。

关键在最后半句：改写完还不算完，还要用一张关键词黑名单（slidesProtocolMarkers，slides_thinking.go:28-33：slide:、<svg、xmlns、viewbox…）做确定性过滤，命中任一就丢弃改写结果、退回通用文案（slides_thinking.go:91）。

RPC 里把请求路到便宜后端，便宜就是便宜——单价降了，账单就降了，行为不变。模型不是：
（1）便宜模型不是自动便宜。Nano 是 reasoning model，不显式关掉推理，它默认会做推理，额外的延迟和 reasoning output token 会把每-token 的价格优势吃回去（trigger_judge.go:20）。"路由到便宜档"只是第一步，还得按任务形态把模型配置对。
（2）你不敢完全信小模型的输出。所以 slides 改写后面还焊了一层确定性关键词过滤兜底——贵模型和便宜模型的能力是非线性的，省下的钱要用一张安全网来对冲质量风险。RPC 后端不会给你"这个实例返回的数据可能不够可信，请再加一层校验"这种问题。

四、第三层：失败降级——两套并存的机制，两种粒度

请求真发出去、provider 过载或失败时，才轮到降级。这套系统里降级不止一处、不止一种粒度，有两套并存的机制，很容易只看到一套。

机制 A · registry 级 FallbackModels 链（通用、LLM 客户端内部）

每个模型在 registry 里可以登记一条静态兜底链。例如 GPT-5.5（model.go:171-184）：

ModelNameGPT55: {
    ...
    CostTier:       ModelCostTierStandard,
    ...
    FallbackModels: []ModelName{ModelNameGPT54, ModelNameGPT52},
},

这条链的唯一消费点在 openainext/client.go 的 Stream（L186-194）：把主模型和它的 fallbacks 拼成一个列表，然后 for 逐个 model 尝试。

机制 B · Conductor 级 tryModelFallback（专用、硬编码一对）

主循环 determineNextStep（agent.go:1018-1079）用 retry.Do 包住每次 LLM 调用。retry.RetryIf（agent.go:1036-1061）里，命中限流或过载错误时调 tryModelFallback（agent.go:1048-1052）：

if errcode.IsErrLLMTooManyRequests(err) || errcode.IsErrLLMOverloaded(err) {
    if agent.tryModelFallback(ctx) {
        return true   // 换成功 → 触发重试
    }
}

而 tryModelFallback（agent.go:1200-1222）被死死圈住——两道 guard 之后只做一件事：

if agent.currentModelName != common.ModelNameGemini31ProPreview {
    return false
}
fallbackModel := common.ModelNameGemini3Flash
ctx.Log().Warnf("[Conductor] Model %s rate limited, falling back to %s", agent.currentModelName, fallbackModel)

if err := agent.switchModel(ctx, &agenttypes.ModeModelConfig{
    ModelName: string(fallbackModel),

它只在 slides 模式（agent.go:1202 的 if !agent.activeModes[...Slides] { return false }）、且当前模型正是 Gemini 3.1 Pro Preview 时，把模型换成 Gemini 3 Flash。换成功后还 SetSlidesEditFallback 落 Redis（agent.go:1218），让整个 slides-edit 会话都留在降级档，避免反复撞限流。

RPC 熔断/兜底是确定的实例拓扑决策：注册的备用实例列表在那儿，切过去就是切过去，语义不变。模型兜底要先回答一个非确定的问题：降级会不会掉质量？正因为答案不确定，tryModelFallback 才被硬编码成"只对 Gemini 3.1 Pro → Gemini 3 Flash 这一对、只在 slides 场景"——团队只 eval 验证过这一对模型在这一个任务上降级后质量可接受，不敢做成"过载就降到便宜档"的通用规则。这就是类比最深的漏水点：RPC 的兜底是拓扑，模型的兜底是被 eval 圈养的经验判断。

五、旁路：loadshedding 是限流，不是切模型

还有一条容易被误认成"降级"的路径。每次 LLM 调用后，llms/stats.go 会把 overload 错误计入一个滑动窗口（loadshedding.RecordModelError）。处理新消息前调 DecideForMessage（policy.go:190-221）：按累积错误数走一条 sigmoid 曲线算出 throttle 百分比（ThrottlePercent，policy.go:151-169），命中就返回一句 busy 文案。

看清楚这里降级的是什么：请求准入，不是模型。它不换任何模型，只是在系统过载时，按 messageID 哈希分桶（ShouldThrottleByMessageID，policy.go:171-179）概率性拒绝一部分免费用户的请求，返回文案并引导升级。真正的"过载时换模型"是上一节的机制 A/B。把这条路径当成模型降级，是读这块代码时最常见的误解。

六、关键定义

模型路由（model routing）
按任务/mode 选择不同模型（贵主模型 vs 廉价辅助模型），并在失败时降级兜底。载体：model_policy.go:PickPrimaryModel（选型）、agent.go:tryModelFallback 与 registry FallbackModels 链（兜底）；事实源 llms/common/model.go。
廉价模型门控（cheap-model gating）
主动把某类小任务硬路由到便宜小模型以省钱，且常配合确定性兜底（关 reasoning、关键词过滤、fail-open）来对冲小模型的质量与延迟风险。载体：trigger_judge.go（Nano + reasoning none）、slides_thinking.go（Nano + MaxTokens 256 + 关键词过滤）。
registry 级 FallbackModels 链
在 model.go 里给每个模型静态登记的降级序列（如 GPT-5.5→GPT-5.4→GPT-5.2），由 openainext/client.go 的 Stream 逐个 model 尝试；粒度在 LLM 客户端内部，Conductor 不感知。
tryModelFallback
Conductor 级的过载降级，被硬编码限定为"仅 slides 模式、仅 Gemini 3.1 Pro → Gemini 3 Flash"，成功后落 Redis 使整个 slides-edit 会话保持降级档。与 registry 链是两套并存机制。
loadshedding（smart load shedding）
过载时按 sigmoid 曲线对免费用户请求做概率性 throttle 的准入控制，返回 busy 文案。降级的是请求准入，不是模型。

七、自测题

Q1：tryModelFallback（agent.go:1200）为什么被硬编码成"只在 slides 模式、只把 Gemini 3.1 Pro 降到 Gemini 3 Flash"，而不是做成对所有模型通用的"过载就降到便宜档"？

因为"哪个任务降到哪个小模型不掉质量"是需要 eval 支撑的经验判断，不是通用规则。团队只验证过 slides 编辑这个具体任务上 Gemini 3 Flash 的质量可接受，所以把降级严格圈在这一对模型 + 这一个场景（agent.go:1202 的 slides guard、L1205 的 currentModelName != Gemini31ProPreview 直接 return false）。

这正是类比漏水点：RPC 兜底是确定的实例拓扑，切过去语义不变；模型兜底要先回答"降级会不会掉质量"这个非确定问题，所以只能对 eval 验证过的组合放行。另外它成功后 SetSlidesEditFallback 落 Redis（agent.go:1218），让整个 slides-edit 会话都留在降级档，避免反复撞限流。

Q2：trigger judge 已经选了最便宜的 GPT-5.4-Nano，为什么代码还要显式把 Reasoning.Effort 设成 "none"（trigger_judge.go:84）？不设会怎样？

Nano 是 reasoning model，默认会做推理。对"该不该触发"这种二分类任务，推理带来的额外延迟和 reasoning output token 会把 Nano 每-token 便宜的价格优势抵消掉（trigger_judge.go:20 注释原话："否则延迟与输出 token 抵消价格优势"）。

所以"路由到便宜模型"只是第一步，还要按任务形态把模型配置对——便宜模型不是自动便宜。这是类比漏水的另一面：贵/便宜模型能力非线性，用错配置照样烧钱或变慢，而 RPC 里把请求路到便宜后端，便宜就是便宜。

Q3（判断题）：某工程师说"我们有 loadshedding，所以系统过载时会自动降级到便宜模型扛住流量"。对吗？

不对。loadshedding（policy.go）不换任何模型。它按累积 overload 错误数走 sigmoid（ThrottlePercent，L151-169），只对免费用户（L211 显式跳过付费用户）按 messageID 哈希概率性拒绝请求并返回 busy 文案。降级的是请求准入，不是模型。

过载时真正的模型降级在另外两处：Conductor 级的 tryModelFallback（slides 专用一对）和 registry 级的 FallbackModels 链（openainext/client.go Stream 逐个试）。把这三条路径混为一谈，是读这块代码最常见的错误。

八、主推资源

本课的一手来源，值得完整读一遍：Building Effective AI Agents — Anthropic Engineering。读的时候带着本课的问题：文中反复强调"用最简单能解决问题的方案，不要为了用而用复杂模式"——把这条原则套到模型路由上，你会发现 tryModelFallback 的硬编码、trigger judge 的 fail-open、slides 改写的关键词兜底，全都是同一种克制：只在被验证过的窄场景里引入复杂度，其余一律回退到已知稳定的档。这正是 Agent 工程与 RPC 工程气质上的分野。
