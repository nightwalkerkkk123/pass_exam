指标体系与统计门禁

你在 0013 立了两根柱子——类型分层和多层评判，并留下一个悬而未决的追问：既然单跑一次只是一次采样，那到底跑几次、拿什么当上线门槛、一个通过率的提升是真进步还是随机噪声？这节课往下走一步，只回答这一个问题：怎么把"感觉好不好"变成"可量化、可比较、可回归"的门禁。它拆成两半——先讲指标体系怎么分级，再讲版本对比为什么必须配统计检验。

一、服务端的 SLA 仪表盘，先承认它对的部分

你在服务端见过监控大盘：P99 延迟、错误率、QPS，每个都挂一条红线（比如"错误率 > 1% 就告警"）。这套直觉搬到 Agent 评测上，方向完全正确——指标体系确实就是"把业务目标拆成一组可观察、可打分、可追踪的检查项，每项配一条阈值线"。0013 那条五步骨架跑完出的通过率，就是这块大盘上的一个读数。

但请先记住指标体系不止用来打分。它真正的价值，是让每次评测不只吐一个总分，而是能明确回答三件事：好在哪、差在哪、下一步该改什么。一个只会亮红灯的大盘救不了你——它得告诉你该去修哪根管子。这是指标体系区别于"一个分数"的分水岭，也是它能驱动"可落地、可验证的持续优化"的原因。

服务端大盘的每条红线背后，是一个确定的、可复现的物理量：错误率 1% 就是 1%，你多测一分钟它不会变成 3%。Agent 指标的读数却是从一个会漂移的分布上采来的——同一个通过率，今天跑是 82%，明天同样的代码同样的用例集再跑可能是 78%。所以"读数过线就发布"这个在服务端天经地义的动作，在 Agent 世界里缺了一环：你还得先问"这个读数本身稳不稳、这次的高低是真信号还是采样抖动"。这一课后半段全在补这一环。

二、五大类指标 + 三级门禁

指标不能拍平成一堆并列的数字，否则你面对二十个分数根本不知道哪个能拦住发布、哪个只是参考。原始方法论把 Agent 评测指标建议分成五大类，再用三个门禁等级给它们定"话语权"：

门禁等级作用不达标会怎样
P0 上线门禁发布的硬约束——安全、合规、资损、关键任务成功率这类底线不达标不能发，一票否决
P1 版本比较与工程优化用来判断新版本比基线是好了还是坏了，指导工程迭代方向不阻断发布，但要解释清楚为什么退步
P2 体验改善与长期观察体验类、长期趋势类，慢慢磨、长期看只观察，不做发布判断

为什么要分级？因为不分级的指标体系会退化成两种失败：要么什么都拦（每个指标都当 P0，一个体验小回退就卡住发布），要么什么都不拦（全是参考值，最后还是拍脑袋发）。分级就是把"这个指标到底有没有一票否决权"提前说死。

Skill 密集系统：五大维度下再挂一组子指标

如果系统大量依赖 Skill 完成工具调用、数据处理、报告生成，那就在上面五大类之下再补一组 Skill 子指标——正是 0013 那张表的"该不该触发 → 流程对不对 → 参数对不对 → 产物可不可用 → 异常能不能降级"。它们不是另起炉灶，而是挂在五大维度下面的一层更细的粒度。你在这里又看到那条贯穿全系列的线。

三、一致性：至少一次成功率 vs 连续成功率

现在触到这节课真正的新东西。对任务执行型和对话型（客服、营销、导购、售后）Agent，光有分级指标还不够——因为系统是非确定性的，同一个任务跑一次成功，不代表它稳。方法论的建议是：同一个任务重复运行多次，然后盯两个截然不同的率。

至少一次成功率（at-least-once success rate）
同一任务跑 N 次，只要有一次成功就算它"行"。度量的是能力上限——这个 Agent 到底具不具备完成该任务的可能性。
连续成功率（consecutive / all-pass success rate）
同一任务跑 N 次，必须每次都成功才算它"行"。度量的是稳定可靠——生产环境能不能把它交给用户。

这两个率可以差得极大。看一个任务跑 5 次的例子：

CI 的绿灯是"跑一次绿了就发"——因为确定性代码跑一次绿，就永远绿。把这个习惯搬到 Agent 上是致命的：你看到 Run 1 PASS 就发布，等于用"至少一次成功"的证据去承诺"连续成功"的可靠性。生产系统关心的几乎总是连续成功率：用户不接受"多试几次总有一次成功"——退款、支付、合规场景里，第一次错就是一次真实的资损或投诉。CI 那盏"单次绿灯"在这里没有统计效力。

什么时候用哪个？探索能力边界时看至少一次成功率（"它到底会不会做这件事"）；决定能不能上生产时看连续成功率（"它每次都能做对吗"）。把两者混为一谈，是非确定系统里最常见的自欺。

四、版本对比：为什么单点分数会骗你

指标分了级、一致性也测了，还剩最后一个坑：版本对比。你把新版本 B 和基线 A 各跑一遍，B 的通过率 82%、A 的 80%，于是宣布"B 提升了 2 个点，发布"。这几乎一定是错的——因为你拿两个单点在比，而每个单点都是从一个会漂的分布上采来的。A 再跑一次可能是 83%，那这"2 个点提升"就凭空蒸发甚至反转了。

所以版本对比必须配套统计检验。方法论建议在报告里同时给出三样东西：

统计工具它回答什么没有它会怎样
置信区间
(confidence interval)		这个指标的真实值落在哪个范围里，而不是一个点你以为 82% 是精确值，其实它是 82%±5%，和基线的区间大面积重叠
与基线的显著性判断
(significance test)		这次升/降是真实变化还是随机波动把噪声当成进步发布，下个版本"提升"又消失，团队反复被骗
最小可感知变化阈值
(minimum detectable effect)		提前定义：至少升/降多少才值得做发布判断为 0.3 个点的"提升"疲于发版，或漏掉真正有意义的退步
核心口号（这节课背下来这一句就够）：

门禁不只看"差了多少"，还要看"这个差异是否超过统计噪声"。一个升降只有既方向对（超过最小可感知变化阈值）又统计上站得住（显著、区间不重叠到失去意义），才值得据它做发布判断。

五、落到你的项目：mino 的门禁处在哪一格

诚实映射（严格区分"mino 真有"和"方法论层、mino 未实现"）：

mino 真有的：一个最粗糙的 P0 门禁雏形。report.go:60-65 把一批用例聚合成 AvgScore 和 PassRate（单点聚合值），runner.go:92 用一句 if Score < 0.6 { FAIL } 给每条用例切一刀。你完全可以把"通过率过某条线"类比成一个最原始的上线门槛——这部分是真实存在、0007/0008 已逐符号核实的。

mino 未实现的（方法论层，明写）：这节课讲的统计能力，mino 一样都没长出来，而且根因是硬的——runAll 是纯串行 for 循环，config.go 里的 concurrency 字段定义了却从未被使用，每条用例只跑一次。既然每条只跑一次：

  • 没有"同任务跑 N 次"的重复运行，所以算不出至少一次成功率，也算不出连续成功率——一致性这一整块无从谈起。
  • 没有多次采样，就没有分布，所以没有置信区间——那个 AvgScore 就是个赤裸的单点数。
  • 没有分布，就没有与基线的显著性检验，也没有最小可感知变化阈值——两个版本比通过率，只能比两个单点谁大。
  • 门禁分级（P0/P1/P2）也没有：0.6 是唯一一条硬编码线，不区分"安全一票否决"和"体验长期观察"。

可核实的诚实结论：mino 的门槛是"单点分数过 0.6"，而不是"统计上显著优于基线"。它有一个门禁的形，还没有统计门禁的实。

CI 里"跑一次绿了就发"之所以成立，是因为那盏绿灯背后是确定性事实。搬到 mino 的 Score >= 0.6 上就失效了：单次通过在非确定系统里没有统计效力。runAll 每条只跑一次给出的 PASS，本质是"至少一次成功"里最弱的那种证据——一次采样，你却拿它当"连续成功、稳定可发布"来用。要补上这节课的能力，第一步不是换算法，而是让每条用例跑 N 次（把那个躺着没用的 concurrency 真正用起来只是并发，重复运行是另一件事），有了分布，置信区间和显著性检验才谈得上。

六、关键定义

指标体系（metric system）
把业务目标 + 专家经验，拆成一组可观察、可打分、可追踪的检查项。核心价值不止打分，而是让每次评测能回答"好在哪、差在哪、下一步改什么"，驱动可落地、可验证的持续优化。
三级门禁（P0 / P1 / P2）
P0 = 上线门禁（不达标不能发，一票否决）；P1 = 版本比较与工程优化（相对基线的升降判断）；P2 = 体验改善与长期观察（只看趋势，不做发布判断）。分级的意义是把"哪个指标有一票否决权"提前说死。
至少一次成功率 / 连续成功率
同一任务跑 N 次：有一次成功即算成功 = 度量能力上限；每次都成功才算成功 = 度量稳定可靠。生产系统主要关心连续成功率，因为用户不接受"多试几次总有一次成功"。
版本对比三件套
置信区间（指标真实值的范围，不是单点）、与基线的显著性判断（真实升降 vs 随机波动）、最小可感知变化阈值（至少升降多少才值得做发布判断）。三者共同支撑"门禁不只看差多少，还看差异是否超过统计噪声"。

七、自测题

Q1：一个退款任务，你让它跑 10 次：8 次 PASS、2 次 FAIL（两次都是"漏调订单查询工具就承诺退款"）。有人说"至少一次成功率 100%，能力没问题，可以上生产"。这个结论错在哪？该用哪个率来做上生产的判断？

错在用错了率。"10 次里有成功"给出的是至少一次成功率——它只证明 Agent 具备完成退款的能力（能力上限 OK），不证明它每次都做对。

退款是资损/合规场景，用户不接受"多试几次总有一次成功"，所以上生产的判断必须看连续成功率：10 次里有 2 次 FAIL，连续成功率不满足，不能上生产。更严重的是那 2 次 FAIL 是"漏调订单查询就承诺退款"——这是 0013 讲的假阳性类风险，每一次都是一笔真实资损。把至少一次成功率当成上线依据，是非确定系统里最典型的自欺。

Q2：版本 B 相对基线 A，某核心通过率从 80% 涨到 82%。产品经理说"涨了 2 个点，赶紧发"。用这节课的三件套逐条反驳，并说明什么情况下这 2 个点才值得据以发布。

三件套逐条问：

(1) 置信区间：80% 和 82% 都是从会漂的分布上采的单点。若 A 是 80%(75–85)、B 是 82%(77–87)，两个区间大面积重叠，这 2 个点很可能落在噪声里——单点相减毫无意义。

(2) 显著性判断：必须做与基线的显著性检验，判定这 2 个点是"真实提升"还是"随机波动"。没做检验就发，等于赌博。

(3) 最小可感知变化阈值：得提前定义"至少升多少才值得发布判断"。若阈值定为 3 个点，那么 2 个点本就在"不值得据以决策"的范围内。

只有当这 2 个点既超过最小可感知变化阈值、又通过显著性检验（区间分离到有意义）时，才值得据它发布。核心口号：门禁不只看差了多少，还要看这个差异是否超过统计噪声。

Q3（延伸）：有人看了 mino 的 memory_eval 说："它有 PassRate 聚合、有 Score >= 0.6 阈值，这不就是统计门禁吗？"结合白名单里的真实代码事实反驳，并指出补齐这节课能力的第一步是什么。

不是。它只是"单点分数过 0.6"的门禁形，不是统计门禁。关键在一条硬事实：runAll 是纯串行 for 循环，concurrency 字段定义了却从未使用，每条用例只跑一次。

由此推出它缺的全部：每条只跑一次 → 没有重复运行 → 算不出至少一次/连续成功率；没有多次采样 → 没有分布 → report.go:60-65 的 AvgScore/PassRate 只是赤裸单点，没有置信区间；没有分布 → 没有显著性检验和最小可感知变化阈值；0.6 是唯一硬编码线，也没有 P0/P1/P2 分级。

补齐的第一步：让每条用例重复跑 N 次，先把"单次采样"变成"一个分布"。有了分布，置信区间、显著性、连续成功率才谈得上——这是把"门禁的形"填成"门禁的实"的地基。（注意：把 concurrency 用起来只是并发提速，和"重复运行取分布"是两件事，别混。）

八、承上启下

这节课把 0013 的"多层评判"往前推到了"怎么把读数变成一条站得住的门禁线"：指标要分级（P0/P1/P2），一致性要分至少一次成功率和连续成功率，版本对比要配置信区间、显著性检验和最小可感知变化阈值。一句话——门禁不只看差多少，还看差异是否超过统计噪声。但这一切都建立在一个还没被质疑的前提上：你手里那套评测集，本身可信吗？如果用例集是从线上随机抽样来的，它可能被幸存者偏差骗了——正常流程占绝大多数，真正会让 Agent 出错的边缘场景几乎没被覆盖。那样的话，再漂亮的置信区间也是在错误的样本上算出来的。下一课就拆这件事。
