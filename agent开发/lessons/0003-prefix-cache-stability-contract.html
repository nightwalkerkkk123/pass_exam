prefix cache 稳定性契约：为什么 <current_plan> 是个雷

上一课的结论是"成本"这根支柱会塌缩进"context 排列"。这一课把它钉死在一处真实代码上：conductorservice 里一个藏在 <context_info> 中间的布尔量，在两次 LLM 调用之间悄悄翻转，就能击穿整条前缀缓存。你会看到一段亲自写下"这里会 cache miss"的警告注释——以及仓库怎么把这个雷焊成了 CI 卡点。

一、先把物理背景讲清楚：每个 step 都重拼一份完整请求

在 conductorservice 里，一个 turn 不是一次调用，而是一个循环：step 0 → tool call → step 1 → tool call → …。每进一个 step，buildConductorMessages 都会从头重新拼一整份消息数组发给 Claude。它不是增量补丁，是整份重发。装配顺序固定（agent.go:1471-1485）：

为什么"整份重发 + 严格前缀"能省钱？因为厂商在 GPU 上缓存了已经算过的 KV。只要你这次请求的前缀字节和上次完全一致，它就直接读缓存（便宜），不重算。这就是 prefix cache 的全部动机：省 token 成本、降延迟。前提只有一个——前缀必须 byte-identical。

二、这颗雷长什么样：一段自己写下的死亡注释

看 buildContextPayload（agent.go:1492）末尾这段。注释不是我加的，是代码里就有的，它精确点名了本课的标题：

// 计划相关信息（仅在启用了计划工具时才注入）
// 注意：<current_plan> 在第 1→2 次 LLM 调用间会变化（IsLastMessageFromUser 从 true→false），
// 导致 context_info 改变、prefix cache miss。不需要 plan 提示的 Agent（如 Slides）
// 应在 CustomizeContextPayload hook 中清除这些字段。
if agent.agentFeatures.PlanEnabled {
	payload.CurrentPlan = agent.currentPlan
	payload.IsLastMessageFromUser = len(agent.newMessages) == 0
}

— agent.go:1524-1531

关键在最后一行：IsLastMessageFromUser = len(agent.newMessages) == 0。这个布尔量不是配置、不是常量，它是从 agent 的运行时状态推导出来的——而这个状态每个 step 都在变：

  • step 0：还没有任何工具回填，newMessages 为空 → len == 0 → IsLastMessageFromUser = true。
  • step 1 起：上一步的 appendToolResultToNewMessages 已经往 newMessages 追加了 assistant + tool 两条 → len > 0 → IsLastMessageFromUser = false。

翻转的物理后果在 context.go 的渲染分支里。这个布尔量直接门控整块 XML 是否写入字符串：

// 计划提示（只有当最后一条消息是用户消息时才显示）
if payload.IsLastMessageFromUser {
	sb.WriteString("<current_plan>\n")
	if payload.CurrentPlan != nil {
		sb.WriteString("The user has provided new input. ...")
	} else {
		sb.WriteString("No plan exists yet. ...")
	}
	sb.WriteString("\n</current_plan>\n")
}

— context.go:269-278（buildDeveloperContextMessage 内，最终以 Role=Developer、Content=整串 <context_info> 返回，见 context.go:356-359）

step 0 时这段 WriteString 全跑，<context_info> 里多出一整块 <current_plan>...</current_plan>；step 1 时这个 if 不进，那一块凭空消失。同一条 Developer 消息，两个 step 的字节不同了。而它排在装配顺序里的 System / Mode / Hook 之后、History 之前——正处在缓存前缀内（context.go:76-88 装配入口）。

三、类比在这里漏水

你会本能地把 prompt cache 当成 Redis / CDN / 本地 LRU 来推理。这个类比对的部分要先承认：两者都是"把算过的结果存下来，下次命中就省"。但接下来是服务端世界里不存在的三层新现象。

服务端缓存是你自家的：省的是你自己的 CPU 和 DB 压力，缓存 key 的匹配规则由你定——你可以 normalize、hash、忽略无关字段、给不同前缀分桶，怎么命中你说了算。

prompt cache 省的不是你的算力，是厂商 GPU 上的 KV 计算，所以匹配规则由厂商定死，你无权更改：字节级严格前缀匹配，顺序固定为 tools → system → messages，前缀里任何一个字节变，其后全部失效。于是三重漏水：
  1. 你以为能调 key，其实只能被动保持前缀 byte-identical。本仓库干脆放弃"调 key"的幻想，把"前缀必须一致"写成了 CI 卡点（下一节）。
  2. 波及面远超一条缓存项。一个藏在 <context_info> 中间的布尔量翻转，就让它之后的断点全部 miss——不是丢一个 key，是从分歧点起整条链塌方。
  3. "成本"支柱塌缩进"context 排列"支柱。省钱不再靠缓存策略，而是靠你能不能保证 <current_plan> 这类字段不在 step 间抖动。

四、仓库怎么防：把"前缀必须 byte-identical"写成 CI 卡点

既然规则由厂商定死、你只能被动保持一致，那"保持一致"这件事就必须被自动化守卫，否则任何人改一行 buildContextPayload 都可能在无声中击穿缓存。cache_prefix_stability_test.go 就是这道守卫。它顶部自带一张消息布局图，直接把每一段标注了"是否应 step 间稳定"：

//     [System Prompt]                   ← 应 step 间稳定
//     [Mode Developer Messages]         ← mode 激活后应稳定（首次激活允许变化）
//     [Hook Developer Messages]         ← 应 step 间稳定
//     [Context Info]                    ← 应 step 间稳定（已知问题：PlanEnabled 会导致变化）
//     [History Messages]               ← 应 step 间稳定（风险：refreshMessages 改 distance 可能影响截断）
//     [Pre-User Hooks]                 ← 应 step 间稳定
//     [User Request]                   ← 应 step 间稳定
//     -------- 以上为 "稳定前缀"，cache 命中的关键 --------
//     [New Messages + cache_control]   ← 每 step 增长（预期行为）

— cache_prefix_stability_test.go:12-20

注意 [Context Info] 那行的括注：应 step 间稳定（已知问题：PlanEnabled 会导致变化）。本课的雷被明明白白写进了契约文档里。测试的机制很直白（assertPrefixStable，cache_prefix_stability_test.go:288）：

  1. step 0：refreshMessages + newConductorBasicMessages 构一次消息，extractStablePrefix 取到最后一条 User 消息为止的前缀（test.go:99-116），算一个 SHA256。
  2. 模拟推进：runStepCount++ 且 appendToolResultToNewMessages 往 newMessages 追加工具结果（test.go:298-299）。
  3. step 1：再构一次、再取前缀、再算一个 hash。
  4. 断言 hash0 == hash1。不等就 t.Errorf，并逐条 diff 报告哪条消息变了（test.go:305-320）。

还有一个反向测试 TestCachePrefixStability_DetectsKnownDrift_PlanEnabled（test.go:413）：它故意开 PlanEnabled，断言两次 hash 应该不同——用来证明"检测管线确实抓得到这种漂移"。这是一种自证：守卫本身也被守卫着。

正确的修复方向不是"把 <current_plan> 换成等长占位串骗过匹配"（严格前缀是 byte-identical，不看长度，见自测题 Q2），而是让抖动字段根本不进前缀：不需要 plan 提示的 Agent（如 Slides）在 CustomizeContextPayload hook（context.go:86 调用点）里把 CurrentPlan、IsLastMessageFromUser 清零；真正需要动态内容的，就挪到前缀之外的 New Messages 段——那里本就每 step 增长，是预期行为。

五、更深一层的漏水：连你选的 TTL 都没真正发出去

你可能已经注意到断点①用的是 NewLLMMessageWithCache1Hour（agent.go:1471、common.go:403），字面意思是"打 1 小时缓存"。内部类型也确实区分两档：CacheTTL5Minutes = "5m" 与 CacheTTL1Hour = "1h"（common.go:600-601）。但看它落到 Anthropic SDK 的那一步 convertCacheControl（convert.go:54）：

switch cacheControl.TTL {
case common.CacheTTL1Hour:
	return anthropic.BetaCacheControlEphemeralParam{
		Type: "ephemeral",
		// TTL:  anthropic.BetaCacheControlEphemeralTTLTTL1h,
	}
case common.CacheTTL5Minutes:
	return anthropic.BetaCacheControlEphemeralParam{
		Type: "ephemeral",
		// TTL:  anthropic.BetaCacheControlEphemeralTTLTTL5m,
	}

— convert.go:59-70（default 分支同样把 TTL 行注释掉了，convert.go:71-76）

你以为你在控制缓存的生命周期——选 1h 还是 5m。但真实 TTL 字段（BetaCacheControlEphemeralTTLTTL1h / ...TTL5m）在 convert.go 里全被注释掉了，TODO 写着"GCP 目前不支持 1h 缓存"。当前所有分支都只返回 {Type:"ephemeral"}、不带 TTL，退化为 Anthropic 的默认——也就是 5 分钟。NewLLMMessageWithCache1Hour 表达的 1h 意图，此刻并没有真正传到厂商。这是又一处"你以为你在掌控，其实规则/实现不在你手里"的漏水。

教学附注（关于"行号会漂移"）：反向测试的文档注释里把漂移根因标成了 agent.go:1426（test.go:405-407），但当前真实代码在 agent.go:1528-1530。注释的行号已过期——这本身就是"断言必须以实际 Read 到的行号为准，不能信任二手引用"的活教材。

六、关键定义

prefix cache / prompt caching（前缀缓存）
厂商按"字节级严格前缀匹配"缓存已处理的 prompt 前段，顺序为 tools → system → messages；命中约 10% 计价、写入加价约 25%，未命中即从分歧点起全额重算。省的是厂商的算力，故匹配规则由厂商定死。
分歧点（divergence point）
一次请求的 prompt 与已缓存前缀第一个字节不同的位置；其后所有 token 缓存全部失效。本课的分歧点就是 <context_info> 里 <current_plan> 块存在与否造成的那几个字节。
IsLastMessageFromUser
conductorAgent 的运行时派生量，= len(agent.newMessages) == 0（agent.go:1530）。step 0 为 true、step 1 起为 false。它门控 <current_plan> 是否写入 <context_info>，因此是这颗雷的引信。
cache write / cache read
首次写入缓存（比基础输入贵约 25%）与后续命中读取（约基础输入价的 10%）。usage 里以 CacheCreationInputTokens / CacheReadInputTokens 区分，与断点后的 InputTokens 相加还原 PromptTokens（convert.go:88-93）。
前缀稳定性契约（prefix stability contract）
"稳定前缀必须 step 间 byte-identical"这条要求被固化为可执行断言的做法。载体：cache_prefix_stability_test.go，取到最后一条 User 消息为止的前缀算 SHA256 并断言 step0/step1 相等。

七、自测题

Q1：conductorAgent 开启 PlanEnabled 后，为什么恰恰是 step 0 → step 1 之间发生 prefix cache miss，而不是 step 1 → step 2？

因为引信是 IsLastMessageFromUser = len(agent.newMessages) == 0（agent.go:1530）。step 0 时 newMessages 为空 → true，<context_info> 里带 <current_plan> 块；step 1 起，上一步的 assistant+tool 回填让 newMessages 非空 → false，那一块从 context.go:269-278 的分支里消失。这一次 true→false 的翻转让 Context Info 这条 Developer 消息的字节改变，而它在缓存前缀内，导致 miss。

step 1→2 时 IsLastMessageFromUser 一直是 false、该块一直不存在，字节稳定，就不再因这个字段 miss。（这也是为什么另有 TestCachePrefixStability_BasicStepProgression 额外检测 step1→2 的累积漂移——那是为了抓 distance 等别的渐进漂移源，不是这个布尔量。）

Q2：服务端同学说"那我把 <current_plan> 换个等长的占位串，字节数不变不就不 miss 了？" 这个想法哪里没理解 prompt cache 的匹配规则？

错在把它当成自家缓存、以为可以靠"长度对齐"或自定义规则骗过匹配。Anthropic 是 byte-identical 的严格前缀匹配，不是按长度、也不是按语义匹配——只要有一个字节不同（哪怕总长完全相同），从该位置往后的缓存断点全部失效。

正确做法不是"伪装稳定"，而是真正保持前缀不变：要么根本不注入会抖动的字段（Slides 等在 CustomizeContextPayload hook 里把 CurrentPlan、IsLastMessageFromUser 清零），要么把动态内容挪到前缀之外的 New Messages 段（那里本就每 step 增长，是预期行为）。这正是"成本"支柱塌缩进"context 排列"支柱的含义——省钱靠的是排版纪律，不是缓存技巧。

Q3（进阶）：如果只看 NewLLMMessageWithCache1Hour 这个名字，你会以为 System Prompt 的缓存能活 1 小时。为什么当前这是错的？在哪一行被"截胡"了？

因为意图态与行为态脱节了。内部确实建了一个 TTL="1h" 的 CacheControl（common.go:618-623），但 convertCacheControl（convert.go:54）把它映射到 Anthropic SDK 时，case CacheTTL1Hour 分支里真正的 TTL: anthropic.BetaCacheControlEphemeralTTLTTL1h 被注释掉了（convert.go:64），TODO 说明"GCP 目前不支持 1h"。于是实际发出的是 {Type:"ephemeral"} 不带 TTL，退化为默认 5m。名字承诺 1h，链路只交付 5m——读代码时不能只信函数名，要跟到映射的最后一跳。
