三维判分与裁判可信度：为什么 judge 跨厂商选 Gemini

你已经把 eval 类比成 CI 了——一组固定输入 + 判分逻辑，红了就回滚。这个类比在 containsScore 这类规则判分上一滴不漏。可一旦判据本身是一个 LLM，它就漏了：断言本身会犯错。这节课只做一件事——看 memory_eval 如何把"裁判可信不可信"从一个免费假设，变成一个必须动手设计的工程问题。

一、先承认类比对的部分

服务端测试里，oracle（判定对错的那把尺子）是被你构造出来就天然可信的。assertEqual(x, 42) 里的 42 不会自己算错，schema 校验不会看走眼，状态码断言不会犯困。测试红了，你怀疑的是被测代码，从不怀疑断言。这个"断言免费可信"的假设深到你根本不会意识到它是个假设。

memory_eval 的两个维度完全符合这个直觉：

  • containsScore（scorer.go:45-88）：把 Agent 回复转小写做子串匹配。ContainsAll 按命中比例给分，ContainsAny 有一个不中就乘 0，NotContains 命中禁词就乘 0。纯字符串规则，没有模型参与。
  • toolCallScore（scorer.go:90-125）：扫 result.ToolCalls，用 isWriteTool 判断有没有写文件类工具，再按 WriteFileContains 关键词命中比例给分。也是确定性判断。

这两个维度可信度不需要论证——它们错了就是你规则写错了，和 assertEqual 同一个性质。方向没错。但第三个维度会把这个免费假设整个掀翻。

二、三个维度，一次加权合成

一条 testCase 进来，score()（scorer.go:33-43）算出三维明细 scoreDetail{ContainsScore, ToolCallScore, LLMJudgeScore}，然后 weightedScore() 按类目把它们压成一个总分：

func (s *scorer) weightedScore(category string, detail scoreDetail) float64 {
	switch category {
	case "recall":
		return detail.ContainsScore*0.3 + detail.LLMJudgeScore*0.7
	case "store":
		return detail.ToolCallScore*0.5 + detail.LLMJudgeScore*0.5
	case "update":
		return detail.ToolCallScore*0.4 + detail.LLMJudgeScore*0.6
	case "rejection":
		return detail.ContainsScore*0.2 + detail.LLMJudgeScore*0.8

— cmd/memory_eval/eval/scorer.go:334-343（其余类目走三者均值，见 L345）

盯着权重看。recall 类里 LLM 裁判占 0.7，rejection 类占到 0.8。总分几乎押在那个你还没证明可信的维度上。

合成完，判定极其朴素——runner.go:92 一句 if result.Score < 0.6 { status = "FAIL" }，report.go:54 统计 PassRate 用同一条 res.Score >= 0.6。0.6 是硬编码的通过线，不是可配置项。

三、漏水点：判据本身是个 LLM

服务端的断言你从不校准——它天然是对的。可 llmJudge（scorer.go:192-237）的判据本身是一个 LLM，而 LLM 是不可靠的调用方，也是不可靠的裁判：它可能吐出解析不了的输出、打分会漂移，最阴险的是自评偏差——自家模型给自家 Agent 的输出打分，会系统性地偏袒。于是"判据可信吗"从一个你想都不会想的免费假设，变成了一个必须专门设计的工程问题。

代码怎么应对？三招，招招都在承认"裁判会失灵"。

招数一：裁判默认跨厂商，与被测模型解耦

被测模型和裁判模型是两个独立字段、两个不同厂商的默认值：

fs.StringVar(&cfg.model, "model", "claude-sonnet-4-5", "model to evaluate")
fs.StringVar(&cfg.judgeModel, "judge-model", "google/gemini-2.5-flash", "model for LLM judge (OpenRouter model ID)")

— cmd/memory_eval/eval/config.go:41-42

被测的是生产模型 claude-sonnet-4-5（Anthropic），裁判固定走 google/gemini-2.5-flash（Google，经 OpenRouter，baseURL 默认 https://openrouter.ai/api/v1，config.go:50）。用非自家的模型给自家 Agent 的输出打分——这就是"跨厂商裁判"的代码落点。

诚实边界（务必分清"代码写了"和"我推断的"）："跨厂商是为了规避自评偏差"这句话，代码里没有任何注释明文写出。它是从配置默认值强推理出来的——model 默认 claude、judgeModel 默认 gemini，且 config.go:42 的注释只说 "model for LLM judge (OpenRouter model ID)"。自评偏差（self-bias）是评估领域公认的动机，但本仓库没留下说明这一意图的注释或文档。正确说法是"从配置可以看出裁判被刻意设成跨厂商"，而不是"注释解释了这是为规避自评偏差"。

招数二：强约束裁判输出，并 clamp 到 [0,1]

裁判的 system 提示写死了"只返回 {score, reasoning} 的 JSON，别的什么都别说"（callOpenRouterJudge，scorer.go:151-189）。返回后代码从文本里抠出第一个 { 到最后一个 } 再解析，拿到 score 还要 clamp 回 [0,1]（scorer.go:229-234）。这是在承认：裁判可能话痨、可能越界，你不能拿它的原始输出当真。

招数三：失灵时回退中性分 0.5

三处兜底（无 key、裁判调用出错、解析失败），都退回一个"不表态"的中性分，下面列出其中两处：

if s.cfg.apiKey == "" {
    return 0.5, "LLM judge skipped (no API key)"
}
// ... 调用裁判、抠出 JSON 之后 ...
if err := json.Unmarshal([]byte(jsonStr), &judgeResult); err != nil {
    return 0.5, fmt.Sprintf("failed to parse judge response: %s", responseText)
}

— cmd/memory_eval/eval/scorer.go:196-198, 225-227

没有 API key、裁判调用本身出错（HTTP/网络，scorer.go:208-210）、或返回的东西解析不成 JSON，都不报错崩溃，而是给 0.5。这是对"裁判是 oracle"这一假设的最直白否认：真正的 oracle 不需要"失灵退路"，因为它不会失灵。你得给 LLM 裁判准备退路，正因为你知道它会失灵。

配置层的一个坑（如实记录，非本课主线）
apiKey 加载优先级是 OPENROUTER_API_KEY > 项目 .env > ANTHROPIC_API_KEY > .env > TCC 的 AnthropicAPIKey（config.go:75-87）。但 baseURL 默认指向 OpenRouter——若最终只拿到一个 Anthropic key 却往 OpenRouter 发请求，会失败。这是配置层面的潜在错配，值得知道。

四、另一套判分器：对抗类根本不请裁判

别把两套系统搞混。上面讲的是 Go 侧（eval/）对端到端 Agent 记忆行为的评估，含 LLM 裁判。仓库里还有一套 Python 侧（memory_eval_suite/），评的是向量检索召回质量，完全不用 LLM 裁判，纯统计判据——token 级 F1/recall（scorer.py:42-63）。

最值得学的是它处理"对抗类"的方式。对抗样本的正确行为是系统应该说"我不知道"——检索本就不该返回任何相关结果。这种"该沉默"的正确性，它没交给一个 LLM 去主观判断，而是压成了一条机械可判的规则：

# Adversarial: expect NO relevant results
if category == 5:
    high_score = [r for r in results if r.score >= adversarial_threshold]
    detail.num_high_score_results = len(high_score)
    detail.passed = len(high_score) == 0
    detail.recall = 1.0 if detail.passed else 0.0
    return detail

— cmd/memory_eval/memory_eval_suite/scorer.py:86-92

对照着看，这一节的教训就立体了：能用确定性规则判的，就别请 LLM 当裁判。把"正确行为"翻译成一条可机械检验的判据，你就把裁判可信度问题从根上消掉了——对抗类做到了，containsScore 做到了。只有当行为的"好坏"实在无法规则化时（回答是否准确、是否有幻觉），才不得不动用 llmJudge，也才不得不承担它带来的全部可信度负担。

五、关键定义

oracle（判定 oracle）
测试里那把"判定对错"的尺子。服务端里它天然可信（assertEqual 的期望值、schema、状态码）；LLM-as-judge 打破了这个免费假设——尺子本身会出错。
LLM-as-judge（模型裁判）
用一个 LLM 给另一个 LLM 的输出打分。核心风险是裁判可信度与自评偏差。载体：eval/scorer.go:192-237 的 llmJudge，默认 google/gemini-2.5-flash 走 OpenRouter。
自评偏差（self-bias）
模型倾向于给"和自己同源/同厂商"的输出打高分。跨厂商裁判就是为规避它（本仓库这一动机靠配置推断，非注释明写）。
中性回退分（neutral fallback）
裁判失灵（无 key / 调用出错 / 解析失败）时退回的"不表态"分 0.5（scorer.go:196-198, 208-210, 225-227）。它的存在本身即证明裁判不是可信 oracle。
对抗样本（adversarial case）
正确行为是"拒绝/沉默"的用例，判据为"无高分即通过"（scorer.py:86-92）——把主观正确性变成机械判定，绕开 LLM 裁判。

六、自测题

Q1：memory_eval 的三个判分维度里，哪一个的可信度本身需要被专门设计/校准？为什么？代码用了哪些手段应对？

llmJudge。 containsScore（子串匹配）和 toolCallScore（工具名判断）是确定性规则，判据天然可信，错了只可能是规则写错，和 assertEqual 同性质。而 llmJudge 的判据本身是一个 LLM——不可靠且有自评偏差。

三招应对：
(1) 裁判默认 google/gemini-2.5-flash 走 OpenRouter（config.go:42, 50），与被测生产模型 claude-sonnet-4-5（config.go:41）跨厂商解耦，规避自评偏袒；
(2) 系统提示强制裁判只返回 {score, reasoning} JSON，并把 score clamp 到 [0,1]（scorer.go:229-234）；
(3) 没 key、调用出错或解析失败时回退中性分 0.5（scorer.go:196-198, 208-210, 225-227）作为失灵兜底。

Q2：一个 rejection 类用例，Agent 正确地拒绝了没有记忆支持的问题——containsScore = 1.0。但这一次 gemini 裁判抽风，返回了一段解析不成 JSON 的文本。最终 Score 是多少？判 PASS 还是 FAIL？这说明了什么？

Score = 0.6，判 PASS（勉强压线）。

裁判解析失败，llmJudge 回退 0.5（scorer.go:225-227）。rejection 权重是 contains*0.2 + judge*0.8：
= 1.0 × 0.2 + 0.5 × 0.8 = 0.2 + 0.4 = 0.6。
runner.go:92 是 Score < 0.6 才 FAIL，0.6 不小于 0.6，所以 PASS。

这恰好是漏水点的活体标本：一个本该满分的正确行为，因为裁判失灵被硬生生拖到刚好及格线。裁判不可靠会直接污染总分——它不是旁观者，而是占了 0.8 权重的当事人。这就是为什么"裁判可信度"必须当成一等工程问题来对待，而不是像 assertEqual 那样当作免费假设。

Q3（延伸）：产品经理说"对抗类用例也让 gemini 裁判来打分，更智能"。用这节课的逻辑反驳他。

对抗类的正确行为是"系统该沉默/说不知道"，这件事能被确定性规则完全刻画——"没有任何结果分数 >= 阈值"即通过（scorer.py:86-92）。既然能机械判定，就不该引入 LLM 裁判：引入裁判 = 平白给判据注入不可靠性、自评偏差和失灵回退的全部成本，换不来任何精度。原则是"能规则化的绝不请裁判"——只有当"好坏"无法规则化（如回答是否有幻觉）时，才不得不动用 llmJudge 并承担它的可信度负担。
