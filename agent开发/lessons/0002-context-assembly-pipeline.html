context 组装管线：分层、顺序与位置

上一课的结论是：context 组织是主轴，其余支柱是它的投影。这一课把主轴打开——看 conductorservice 里一次 LLM 调用的输入到底是怎么按固定分层顺序拼出来的。你会看到一件在服务端世界里荒谬、在这里却是硬约束的事：一个字段放在哪个位置，直接决定它被模型读得认不认真、以及这轮请求要不要付全价。

一、先看骨架：一次调用的 messages 是四段 append 出来的

在 conductorservice 里，每次 LLM 调用的输入不是随手拼的 payload，而是由 buildConductorMessages（agent.go:1418）按固定顺序组装的 []*common.LLMMessage。骨架只有四段 append，先后顺序就是发给模型的消息顺序：

messages := []*common.LLMMessage{common.NewLLMMessageWithCache1Hour(common.System, systemPrompt)}

// 添加所有激活模式的 developer messages（行为约束、相位规则等）
modeDeveloperMessages := agent.buildModeDeveloperMessages()
messages = append(messages, modeDeveloperMessages...)

// 添加 hooks 注入的补充参考文档（animation_patterns、chinese_fonts 等），紧贴对话以提高注意力权重
developerMessages := agent.hooks.BuildDeveloperMessages(ctx, agent.agentFeatures, &payload)
messages = append(messages, developerMessages...)

chatMessages = agent.hooks.OverrideReplayChatMessages(ctx, chatMessages)
messages = append(messages, chatMessages...)

— agent.go:1471-1485

三点先记住：

  • System Prompt 不是一整块字符串。它由 populateSystemPrompt（prompts.go:126）把 HITL / Space / Search / Plan / Notebook / Trust / Reply / Research / Preliminary / SuperAgentMemory / Safety 等组件分别渲染（prompts.go:202-255），塞进 SystemPromptData（prompts.go:257-270），再用某个 mode 的模板 tmpl.Execute 合成（prompts.go:272-276）。用哪个模板由 hooks.GetSystemPrompt 分发。
  • Hook Developer 是真有其事的注入。比如 HTML Slides 模式的 BuildDeveloperMessages（htmlslides/state.go:93-98）返回两条 Developer 消息：GetAnimationPatternsSection() 和 GetChineseFontsSection()——正是 agent.go:1477 注释里点名的 animation_patterns / chinese_fonts。
  • 组装期一次定序，发送期不再重排。streamCallLLM（agent.go:1545）拿到已排好序的 messages，只算 thinking 预算（agent.go:1547-1558）、拼 options、在 agent.go:1591 把 messages 原样交给 agent.llmClient.Stream。顺序在组装期就冻结了。

二、放大 chatMessages：稳定前缀在哪里断开

chatMessages 的内部顺序由 newConductorBasicMessages（context.go:76）决定。非群聊主线（context.go:139-168）用 gslice.Concat 依次拼：

return gslice.Concat(
    // 上下文信息 Developer 消息
    wrapContextMessage(contextMsg),

    // 历史消息（User 消息包裹 <user_request>、…与当前轮格式一致以提升 prefix cache 命中率）
    agent.convertOriginalMessages(slidesPatchEditEnabled),

    // 用户请求前的Hook消息
    utils.WrapInSystemHintMessage(preUserMsgs...),

    // 用户请求
    agent.populateUserRequest(agent.latestUserMessage, /* … */),

    // 当前轮次的消息
    utils.MarkAsCachePointSpecified(minomessage.MTransferToLLMMessages(
        agent.newMessages, /* … */
    ), common.Tool),

    utils.WrapInSystemHintMessage(
        gslice.Concat(
            populateRunStepCount(agent.runStepCount, agent.maxStepCount),
            populateLastErr(agent.lastAgentErr),
        )...,
    ),
)

— context.go:139-168（非群聊路径）

关键一行是 MarkAsCachePointSpecified(..., common.Tool)：它从后往前找到最后一条 Tool 消息，在它的最后一个内容块上打一个 NewCacheControl() 缓存断点（utils/prompts.go:24-28）。NewCacheControl() 的默认 TTL 是 5 分钟（common.go:610-613，CacheTTL5Minutes = "5m"，见 common.go:600）。这个断点就是稳定前缀的尾巴：断点之前是每步都该 byte-identical 的前缀，断点之后（新消息继续增长、system_hint 里的 runStepCount / lastErr 每步变）是预期会变的区域。

这不是我编的分层——团队把它写成了 CI 卡点。cache_prefix_stability_test.go（文件头 :1-35）直接画出这张布局并断言它稳定：

[System Prompt]            ← 应 step 间稳定
[Mode Developer Messages]   ← mode 激活后应稳定
[Hook Developer Messages]   ← 应 step 间稳定
[Context Info]           ← 应 step 间稳定（已知问题：PlanEnabled 会导致变化）
[History Messages]       ← 应 step 间稳定
[Pre-User Hooks]         ← 应 step 间稳定
[User Request]           ← 应 step 间稳定
-------- 以上为 "稳定前缀"，cache 命中的关键 --------
[New Messages + cache_control] ← 每 step 增长（预期）
[System Hints (stepCount/err)] ← 每 step 变化（预期）

— 据 cache_prefix_stability_test.go:12-21 的布局注释转述（非逐字）。这个测试模拟 step 0 → step 1，对比两次"稳定前缀"的 hash，不同就 fail——它守着前缀不许抖。

三、类比在这里漏水（一）：位置本身是有语义的

你的服务端直觉是对的一半：拼一次 RPC 请求 = 把需要的数据收集齐塞进 payload。字段放 payload 里哪个位置无所谓，服务端反序列化成结构体后按 key 取值，一个值存哪、读它和它的物理位置无关。KV 存储读 key=X 的开销也与 X 在库里的排布无关。

在 context 里，同一段信息放开头、中间、结尾，模型的实际利用率不同——中间最差，这就是 lost in the middle。把 payload 当"无序字典"的直觉在这里失效：位置是一个影响输出质量的变量。代码里那句注释 紧贴对话以提高注意力权重（agent.go:1477）就是在对抗这一点——它故意把 Hook 参考文档往后放，靠近对话，因为越靠近末端注意力权重越高。

四、类比在这里漏水（二）：顺序被 prefix cache 绑架

服务端缓存是为了省你自己的计算，命中规则你说了算。prompt 的 prefix cache 省的是模型厂商的计算，命中规则由厂商定：请求前缀 byte-identical 才命中，任何一个字节变了，分歧点之后的整段前缀全部 miss、按全价重算。

于是"字段放哪"从质量问题升级成钱的问题：稳定内容必须在前并打 cache_control（System Prompt 打 1 小时 NewLLMMessageWithCache1Hour（common.go:403）→ 内部调 NewCacheControl1Hour（common.go:618）→ CacheTTL1Hour = "1h"（common.go:601）），任何每步会变的东西（runStepCount、lastErr、current_plan、秒级时间）必须挪到缓存断点之后，否则整段前缀 cache miss、这轮付全价。在"读值与位置无关"的存储世界里，你永远不会因为"读它会破坏缓存"而去改一个字段的内容。这里会。

证据 1：为了缓存，把时间精度砍到只剩"天"

// 当前时间（只保留日期粒度 + 时区，避免秒级精度导致跨 session prompt cache miss）
if !payload.StartTime.IsZero() {
    sb.WriteString("<current_time>\n")
    fmt.Fprintf(&sb, "%s (%s)",
        payload.StartTime.Format("2006-01-02"),
        payload.StartTime.Location().String(),
    )
    sb.WriteString("\n</current_time>\n")
}

— context.go:252-260

current_time 位于 <context_info> 里、在缓存断点之前的稳定前缀区。如果保留秒级，每次请求这一字节都不同，会击穿它之后的整段前缀。砍到 2006-01-02（Go 的日期格式串）之后，同一天内这一字段稳定，前缀可复用。这是最锋利的漏水证据：普通存储里一个时间字段精度高低与读取成本无关；在 context 里，精度直接换缓存命中率。

证据 2：一个动态块被点名会击穿缓存，对策是让不需要它的 Agent 主动清空

// 注意：<current_plan> 在第 1→2 次 LLM 调用间会变化（IsLastMessageFromUser 从 true→false），
// 导致 context_info 改变、prefix cache miss。不需要 plan 提示的 Agent（如 Slides）
// 应在 CustomizeContextPayload hook 中清除这些字段。
if agent.agentFeatures.PlanEnabled {
    payload.CurrentPlan = agent.currentPlan
    payload.IsLastMessageFromUser = len(agent.newMessages) == 0
}

— agent.go:1524-1531

"位置/内容影响缓存"不是抽象担忧，而是线上已知问题：<current_plan> 放在稳定前缀里的 context_info 块，会在同一 turn 的第 1→2 次调用整体翻转（IsLastMessageFromUser 从 true 变 false），从这里往后全部 cache miss。团队的对策：让不需要它的 Agent（如 Slides）在 CustomizeContextPayload hook 里把这两个字段清掉（htmlslides/state.go:85-88 正是这么做的）。

证据 3：连 mode 的输出顺序都被排序钉死

// 收集并排序激活的 mode，确保输出顺序一致，避免影响缓存命中率
var activeModes []agenttypes.ConductorMode
for mode := range agent.modeDeveloperMessages {
    if agent.activeModes[mode] && agent.modeDeveloperMessages[mode] != "" {
        activeModes = append(activeModes, mode)
    }
}
sort.Slice(activeModes, func(i, j int) bool {
    return string(activeModes[i]) < string(activeModes[j])
})

— agent.go:1381-1390

Go 的 map 遍历顺序是随机的。如果直接按遍历顺序输出 mode developer 消息，同一组激活 mode 每次可能排出不同的字节序列，前缀就抖了。所以这里显式 sort.Slice 按 mode 名排序——只为让相同激活集合每次产出 byte-identical 的前缀。"稳定内容必须放前面且顺序不能抖"这条约束，被直接编译进了代码。

六、关键定义

context 组装管线（context assembly pipeline）
把 System Prompt、developer messages、hooks 注入文档、chatMessages 按固定分层顺序拼成一次调用输入的过程。载体：buildConductorMessages（agent.go:1418）的四段 append。
稳定前缀（stable prefix）
一次请求里从开头到缓存断点之间、要求跨 step byte-identical 的那一段。命中 prefix cache 的前提。在本仓库由 cache_prefix_stability_test.go 用 CI 断言守护。
cache break point（缓存断点）
用 cache_control 标记的"缓存到此为止"的位置。本轮新消息尾部由 MarkAsCachePointSpecified（utils/prompts.go:24）在最后一条 Tool 消息上打点，TTL 默认 5 分钟。
context_info
当轮上下文信息块，一条 Developer 消息。内部按 space_info → user_identity → current_time → main_language → current_plan → user_personalization → user_memory → mobile_app 顺序拼装（context.go:236-305）。

七、自测题

Q1：团队为什么把 <current_time> 的精度从秒级砍到只剩日期（2006-01-02）？如果保留秒级会怎样？

因为 prompt prefix cache 命中要求前缀 byte-identical。current_time 位于稳定前缀区（在缓存断点之前）。若保留秒级，每次请求这一字节都不同，会让它之后的整段前缀 cache miss、请求付全价。砍到天级后，同一天内该字段稳定，前缀可复用。

这正是漏水点：在普通存储里，一个时间字段精度高低与读取成本无关；但在 context 里，精度直接换缓存命中率——你是在拿信息粒度换钱。见 context.go:252-260。

Q2：System Prompt 用 NewLLMMessageWithCache1Hour（TTL 1h），而本轮新消息尾部用 MarkAsCachePointSpecified（默认 NewCacheControl，TTL 5min）。为什么两处 TTL 不同、且分别放在最前和最后？

因为它们的稳定性不同、在前缀里的位置也不同。System Prompt 跨多轮甚至跨 session 几乎不变，放在最前（位置 0）且给长 TTL（1h），让复用窗口最大；本轮新消息（tool 结果等）每 step 增长，只在同一 turn 的循环里短时间复用，放在稳定前缀之后作为 cache break point，给短 TTL（5min）即可。

一句话：越稳定的内容越靠前、TTL 越长；越易变的越靠后、TTL 越短。见 agent.go:1471 与 context.go:153、TTL 常量 common.go:600-601。

Q3：buildModeDeveloperMessages 在输出 mode developer 消息前做了一次 sort.Slice（agent.go:1388）。如果删掉这次排序，功能上消息内容一模一样，会出什么问题？

会周期性地击穿 prefix cache。Go 的 map 遍历顺序是随机的，删掉排序后，同一组激活 mode 每次可能排出不同的字节序列。Mode Developer 段位于稳定前缀内（在 System Prompt 之后、chatMessages 之前），它一抖，其后的整段前缀就 miss。

这就是为什么注释写"确保输出顺序一致，避免影响缓存命中率"——排序不是为了正确性，是为了字节稳定性。这类 bug 在服务端几乎不存在：那里字段顺序变了，反序列化照样按 key 取值，没有代价。见 agent.go:1381-1390。

八、一个边界：这条主线只讲非群聊路径

上面 chatMessages 的顺序（context_info → 历史 → 用户请求 → 本轮新消息 → system_hint）严格适用于非群聊路径（context.go:139-168）。群聊路径（context.go:95-137）结构不同：它不单独拆"用户请求"，而是把历史与最新消息统一当群聊上下文，再在其后插 pre-user hint 和本轮消息。分层与缓存的原理一致，具体切分不同——碰到群聊场景时回来看这段。
