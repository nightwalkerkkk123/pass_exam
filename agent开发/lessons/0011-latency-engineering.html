延迟工程：流式 TTFT、事件间隔超时、PTU 配额准入的三个漏水点

你对"延迟"有一套成熟的服务端直觉：延迟就是请求发出到响应返回的墙钟时间，timeout 就是"超过 X 秒判失败"，rate limit 就是"保护本服务别被打垮"。这三条直觉在 Agent 世界都会漏水。这节课在 mino_server 里各挑一处真实代码，把漏水点戳穿。

一、先承认类比对的部分

把 LLM 调用当成一次远程请求来做延迟治理，是完全正确的起点。请求有 TTFT（首字节时间）和总时长，你会想到缓存、并发、超时、限流——方向没错。三条服务端直觉分别是：

  • 延迟 = 墙钟时间，优化延迟就是把这个数字压小。
  • timeout = 总时长超时，超过 X 秒还没返回就判失败。
  • rate limit = 保护本服务，超限就 429 拒绝，别让自己被打垮。

接下来是三处这些直觉不成立的地方。

二、漏水点一：感知延迟 ≠ 实际延迟（流式 / TTFT）

服务端里，一次响应通常一次性返回，感知延迟约等于实际延迟，你几乎不区分这两者。LLM 生成慢是硬约束——总时长 ≈ token 数 × 每 token 生成时间，你改不动它。但流式实现能把首 token 时间（TTFT）从总时长里拆出来：底层 provider 每吐一个 chunk，后台 goroutine 立刻转发给下游，用户在第一个 token 就看到输出滚动，体感"秒回"。总时长没变，感知延迟塌了。"感知延迟 ≠ 实际延迟"这个维度，传统后端几乎不存在。

在 biz/infrastructure/llms/stream.go 里，streamWithTracing（72–137 行）先拿到底层 provider 的 originalStream，然后新建一个带 1000 缓冲的中转流：

outputStream := stream.New[*common.LLMMessage](1000)   // stream.go:91

再用 concurrency.SafeGo 起后台 goroutine（stream.go:97）跑转发循环。核心就这几行：

for {
    msg, err := originalStream.Receive(ctx)
    if err != nil {
        outputStream.SendError(ctx, err)
        ...
        return
    }
    outputStream.Send(ctx, msg)      // 每个 chunk 一到就转发给下游 → TTFT
    messages = append(messages, msg) // 同时留存，收尾合并 / 计费
}

— biz/infrastructure/llms/stream.go:121-133

顺带一提，同文件的 streamWithContinue（184 行起）还处理 max_tokens 截断的自动续写：默认最多 10 轮（maxRounds := 10，stream.go:195），把已生成内容追加回去再发一轮。这是"总时长"这一侧的工程，与 TTFT 是两码事。

三、漏水点二：超时不是"总时长"，而是"事件间隔"（心跳）

服务端 timeout 几乎都是总时长超时：一个请求超过 X 秒还没结束就判失败。但 mino_server 的工具超时不是这么算的——它衡量的是相邻两个 event 之间的间隔（心跳超时）。只要工具持续吐事件，一个跑 30 分钟的工具也永远不超时。"超过 X 秒即失败"的直觉在这里彻底失效。

四、漏水点三：限流不为保护自己，而为调度共享 PTU + 成本路由

服务端 rate limit 是为了保护本服务：超限就 429 拒绝，宁可少服务也别被打垮。PTU 配额准入反过来——它保护的是全公司共享的、按预留吞吐量（Provisioned Throughput Unit）付费的昂贵推理资源。所以令牌桶和等待表都跑在 Redis 里、跨所有 pod 共享；而且超限时不是拒绝，而是"降级到按量付费 PayGo"继续服务。限流从"保护自己"变成了"调度稀缺共享资源 + 成本路由"。

六、关键定义

TTFT（首 token 时间，time to first token）
从发起请求到收到第一个输出 token 的时间。流式的价值在于压低 TTFT（感知延迟），而非压低总生成时长（实际延迟）。载体：stream.go 的后台转发 goroutine。
事件间隔超时 / 心跳超时（event-interval timeout）
超时衡量的是"相邻两个 event 之间的间隔"，而非工具总运行时长。计时器在每次 Receive 前用 WithTimeoutCause 重建、收到 event 后 cancel 归零。载体：toolcall.go:forwardToolEvents。
PTU 配额准入（PTU quota / admission）
在发起昂贵推理前，用 Redis 令牌桶 + 等待表决定"放行 / 排队 / 降级 PayGo"，以保护跨 pod 共享的预留吞吐量资源。载体：llms/ptuquota/（admission.go / waitlist.go / bucket.go）。
SoftRatio（近耗尽软阈值）
令牌桶使用率的软阈值，默认 0.8。未越过时直接放行且不打点；越过后才进入受限判定与排队。载体：bucket.go:clampSoftRatio。
降级到 PayGo（downgrade）
排队超时后不报错，而是从 provider 列表里剔除 PTU provider、改走按量付费继续服务（decisionDowngraded）；剔完为空则保留原列表（decisionDowngradeSkipped）。载体：admission.go:waitForAdmission。

七、自测题

Q1：一个 LLM 请求生成 2000 个 token 要 40 秒。mino_server 的流式实现能把这 40 秒的总时长缩短吗？用户的体感为什么会变快？

不能缩短总时长。40 秒是 token 数 × 每 token 生成时间的硬约束，stream.go 改不动它。变快的是感知延迟：streamWithTracing 用后台 SafeGo goroutine 把底层 provider 的每个 chunk 一 Receive 到就 outputStream.Send 给下游（stream.go:131），所以用户在第一个 token（TTFT，可能只需几百 ms）就看到输出开始滚动，而不是干等 40 秒看空白。这就是"感知延迟 ≠ 实际延迟"——服务端世界很少区分这两者，因为传统响应是一次性返回的。

Q2：PTU 令牌桶已接近耗尽、用户又被判定为受限，Admit 会直接拒绝这个请求、返回错误吗？

不会拒绝。它进 waitForAdmission 排队（ZSet 等待表 + Top-K 原子 Lua 判定），最长等 MaxQueueWaitMs；期间桶变健康就 decisionQueued 放行走 PTU。若等到超时，不是报错，而是 removeProvider 把 PTU provider 从列表里剔掉、降级到 PayGo（按量付费）继续服务（decisionDowngraded）；只有剔完 provider 列表为空时才保留原列表（decisionDowngradeSkipped）。这正是漏水点：这里的限流是"调度共享 PTU 资源 + 成本路由（PTU→PayGo）"，而不是服务端那种"保护自己、超限即 429 拒绝"。

Q3（进阶）：一个子代理工具跑了 25 分钟才结束，全程每隔几十秒吐一个进度 event。它的超时上限是 30min，但为什么"25 分钟"这个总时长其实和会不会超时无关？

因为 forwardToolEvents 度量的是相邻 event 的间隔，不是总时长。WithTimeoutCause(ctx, timeout, ...) 在每次 Receive 前重建、收到 event 后 cancel（toolcall.go:693-697），计时器随每次心跳归零。只要没有任何一段间隔超过 30min，工具跑 25min、甚至跑几小时都不超时。真正会触发超时的是"卡住不吐事件"——比如死锁、或像 desktop_file_transfer 那样传输期不发心跳（此时间隔≡总时长，所以它的阈值被特意放大到 30min）。
