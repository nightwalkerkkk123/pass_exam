评测集是设计出来的质量资产

你在 0014（统计门禁：多次运行才有效力） 学过——单跑一次只是采样，得跑 N 次、看统计量、拿通过率当门槛，才有说服力。但那一整套统计机器都架在一个默认前提上：你手里那批用例，本身值得跑。这节课往下走一步，去质疑那个前提：如果评测集是"随手从线上捞一批"来的，那么你跑得再多次、统计得再严谨，量出来的也只是一个漂亮的幻觉。评测集不是采样得来的，是设计出来的。

一、服务端直觉：抓一批线上流量当测试样本

做服务端的人有个根深蒂固的好习惯：用真实流量回放来测。把线上请求录下来，重放到新版本上，看响应有没有变。这套"流量回放（traffic replay）"在确定性系统里近乎无敌——因为对确定性代码来说，每一条请求都同等有信息量：一条 GET /order/123 走的代码路径，和另一条 GET /order/456 几乎完全一样，多录一万条正常请求，覆盖率并不会掉，顶多是浪费机器。所以"多采样、随机采、按线上分布采"是免费的、甚至是美德。

一旦被测对象是 Agent，"按线上分布随机采样"这个美德会系统性地骗你。线上流量有强烈的幸存者偏差（survivorship bias）：正常流程占了绝大多数，真正能把 Agent 逼出错的边缘场景——罕见的订单状态组合、带情绪的刁难表达、诱导 Agent 越权承诺的问法——在真实分布里占比极低。你按这个分布随机采 200 条，边缘场景可能就摊到个位数，甚至一条不落。跑出来的报告"看起来很不错"，通过率漂亮，而那些真正会让你线上事故的场景，压根没进评测集。服务端里"多采点总没坏处"，在这里变成了"采得越随机，越把关键问题稀释没了"。

差别的根子在于：确定性系统里每条样本信息量均等，Agent 里信息量极度不均——一条精心构造的"已发货订单硬要退款"的对抗样本，抵得过一千条"你好/在吗/谢谢"的寒暄。评测集的价值不由条数决定，由它覆盖了多少高风险路径、关键逻辑和失败模式决定。所以这节课的第一句话就是原文的划重点：

评测集并非线上数据的随机抽样，而是围绕高风险路径、关键逻辑和失败模式设计出来的质量资产。

二、把评测集当"资产"意味着什么

"质量资产"这四个字不是修辞。资产有三个特征，评测集都得有：长期复用、持续增值、需要治理。对应到评测里，最核心的载体就是 Golden Set（黄金评测集）。

Golden Set（黄金评测集）
本文中默认作为基线评测集使用——一组相对稳定、长期复用的核心样本，主要用于版本对比与发布门禁。它是你每次改模型 / 改 Prompt / 改工具后都拿来回归的那把"不动的尺子"：正因为它稳定，两个版本在它上面的通过率才可比；正因为它覆盖核心路径与高风险边界，它红了你才敢拦住发布。
幸存者偏差（survivorship bias）
只统计"活下来 / 走完正常流程"的样本，导致失败样本被系统性忽略。线上流量天然带这种偏差：出错的会话往往被用户放弃、转人工或投诉，在"正常完成"的分布里占比极低，随机采样会把它们采没。
质量资产（quality asset）
把评测集当作需要长期建设、版本治理、持续回流增值的工程资产，而非一次性抓取的临时样本。判断标准：它能不能跨版本复用、能不能随 badcase 增长、有没有明确的期望行为与口径。

Golden Set 和 0014 的统计门禁是一对搭档：0014 解决"跑几次、怎么统计才算数"，这节解决"跑哪些才算数"。统计再严谨，样本选错了照样是精确地测量了一个无关的东西。

三、评测集的四类来源：定标准 + 扩覆盖 + 接真实 + 回流失败

一个成熟的评测集至少包含四类来源，各司其职、互相补位：

来源职责退款场景的例子
专家设计用例定标准：由业务/专家写出"正确行为长什么样"，钉死高风险路径的期望动作"用户要求退款但订单已发货，Agent 必须先查订单状态、再解释规则"
扩展用例扩覆盖：在专家用例的骨架上，扩出同一场景的不同说法/状态/情绪同一退款场景下，换不同用户措辞、不同订单状态、不同情绪强度
线上真实数据接真实：引入真实用户的自然表达与长尾情况，避免闭门造车来自真实客服会话记录、工单记录
badcase 回流回流失败：把线上翻过的车固化成用例，保证同一个坑不踩第二次"Agent 直接承诺一定退款成功"这类线上 badcase 被收编成一条必测样本

这四类不是并列平权的堆料，它们有明确的生成顺序与约束关系——尤其"专家用例"和"扩展用例"之间：

很容易顺手滑向一个偷懒版本："把退款场景描述一句话丢给 GPT，让它生成 100 条测试用例。"这在服务端"造测试数据"的直觉里似乎没问题——数据越多越好。但对评测集是灾难：LLM 自由生成会连期望行为一起编，它可能生成一条"已发货订单，Agent 应爽快退款"——把错误行为写成了正确答案。评测集一旦标准漂移，你就是在用一把被污染的尺子量东西，量得越勤越有害。所以扩展用例的铁律是：专家用例和业务规则先钉死"标准"，LLM 只许动"表达"，不许动"对错"。

五、含 Skill 的 Agent：用例按"触发 → 核心逻辑 → 产物 → 异常"四类组织

还记得 0013 那张 Skill 评测维度表吗？它在这一节以"数据集怎么组织"的形态复现。对包含 Skill 的 Agent，用例设计沿一条固定的诊断顺序铺开：

用例类别问的问题数量占比倾向
① 触发这个 Skill 在这个场景该被调起吗？（漏触发 / 误触发）适量
② 核心逻辑调起后步骤顺序、分支走对了吗？高风险分支覆盖了吗？通常最多
③ 产物质量生成的报告/文件/动作，业务能直接用吗？适量
④ 异常容错异常输入、工具失败、边界条件下稳得住、退得回吗？适量但不可省

六、落到你的项目：mino 有一批用例，还没有"资产"

诚实映射（严格区分"mino 真有"和"方法论层、mino 未实现"）：

mino 真有的（有代码符号锚）：默认数据集 memory_eval_cases.json 28 条手写用例，分 4 类（recall 12 / store 8 / update 4 / rejection 4），其中 20 条 requires_vector；另有两套替代集 locomo_recall_cases.json（57 条）与 locomo_store_cases.json（15 条）。特别值得点出：rejection 类考的正是"该拒绝 / 不许幻觉"——这本质就是本节说的对抗样本思路（正确行为是拒绝或不作答），和 Python 侧 scorer.py:86-92 那条"无高分即通过"的机械判据（见 0008）是同一个直觉：把"该沉默"做成可判的用例。也就是说，mino 已经无意识地实践了"设计对抗类用例"这一条。

方法论层、mino 未实现的（明写，不虚构）：

  • Golden Set 的版本治理——那 28 条只是一批固定文件，没有"基线集 / 版本对比 / 发布门禁"的治理机制。
  • 幸存者偏差感知的采样策略——用例是手写的，谈不上"从线上按分布采、并主动过采样高风险边缘"。
  • "专家定标准 + LLM 扩覆盖"的建设流程——没有"规则先定结构字段、LLM 只填自然表达"的扩展管线。
  • badcase 回流入库——没有把线上翻车样本自动固化回评测集的闭环。

诚实结论：mino 有一批质量不错的手写用例（含对抗类），但还没有把"数据集作为可治理质量资产"的建设与治理机制建起来。这不是缺陷指控，是"从一批 case 长成一份资产"还差的那几步。

七、自测题

Q1：团队说"我们直接把上周线上 5000 条客服会话随机抽 200 条当评测集，最真实"。用这节课的逻辑指出这套做法的根本问题，以及正确姿势。

根本问题是幸存者偏差。线上 5000 条里绝大多数是走完正常流程的会话（问候、查物流、常规咨询），真正会让 Agent 出错的边缘场景——罕见订单状态、带情绪的刁难、诱导越权承诺——占比极低。随机抽 200 条，几乎忠实复刻这个"正常流程占绝大多数"的分布，边缘场景可能只剩个位数甚至一条不落。结果：报告通过率很漂亮，但真正会导致线上事故的场景根本没进集合。这是"精确地测量了一个无关的东西"。

正确姿势：评测集是设计出来的，不是采样出来的。先由专家围绕高风险路径、关键逻辑、失败模式写 50–200 条 golden set（主动过采样高风险边界），线上真实数据只作为四类来源之一用来接真实表达和长尾，再叠 badcase 回流。线上流量该用，但用来补充覆盖、而非定义分布。

Q2：为快速扩量，同事把"退款场景"一句话丢给 LLM，让它"自由生成 100 条退款测试用例，包含输入和期望输出"。这批用例能直接进 golden set 吗？问题出在哪一步？正确的扩展流程是怎样的？

不能直接进。问题出在让 LLM 连"期望行为（对错标准）"一起生成。LLM 自由发挥时可能编出"已发货订单应爽快退款"这种把错误行为写成正确答案的用例，一旦入库，你就是在用一把被污染的尺子做回归——量得越勤越有害，且这种污染很隐蔽。

正确流程（两段分工）：① 先用规则（对齐专家用例与业务 SOP）确定结构字段——场景类型、订单状态、用户诉求、期望动作——把"考什么、正确行为是什么"钉死；② 再让 LLM 只负责把这套结构翻译成更接近真实用户的自然语言表达（不同说法/情绪/措辞）。规则管"标准不漂移"，LLM 管"表达更真实"，LLM 绝不许动"对错"。

Q3（延伸）：mino 的 memory_eval_cases.json 里有 4 条 rejection 类用例，考"该拒绝 / 不许幻觉"。这算不算这节课说的"设计出来的评测集"？它离一份真正的"质量资产"还差哪几步？

算是一个正确的雏形。rejection 类考"正确行为是拒绝/不作答"，这正是本节的对抗样本思路——主动构造"最容易让 Agent 幻觉/越权"的失败模式，而不是随机采样得来的。这说明 mino 的用例是围绕失败模式手工设计的，方向对。

但它离"资产"还差几步（都属方法论层、mino 未实现）：(1) 没有 Golden Set 的版本治理——它只是一批固定文件，缺"基线集 / 跨版本对比 / 发布门禁"的定位；(2) 没有幸存者偏差感知的采样与过采样策略；(3) 没有"专家定标准 + LLM 扩覆盖"的扩展管线；(4) 没有 badcase 回流入库的闭环。一句话：有对抗类用例这颗好种子，还没长成能跨版本复用、持续增值、有口径治理的那棵资产树。

八、承上启下

这节课立的柱子是：评测集是设计出来的质量资产，不是线上流量的随机快照——警惕幸存者偏差，以 Golden Set 为基线，按"专家定标准 / LLM 扩覆盖 / 线上接真实 / badcase 回流"四类来源建设，先做好 50–200 条再向外长，含 Skill 的用例按"触发→核心逻辑→产物→异常"四类组织。但一个新问题冒出来了：有了这批设计精良的用例，到底该用规则判、用 LLM Judge 判、还是交给人？什么样的样本必须捞出来给人看？——不同判据成本天差地别，全上 LLM 太贵、全靠人不现实、全靠规则又判不了语义。这就是下一课的主题。

有任何不清楚的地方，回到会话里问你的 AI 老师——可以要求 ELI5、ELI14，或让我带你打开 cmd/memory_eval/eval/ 下的 memory_eval_cases.json，对照这节的"四类来源 / 对抗样本"看那 28 条到底设计了什么、还缺什么。下一课：0016 评分的分层与人工路由——规则主判、LLM Judge 补语义、以及"什么样本必须进人工终判"的分层筛查与路由。
