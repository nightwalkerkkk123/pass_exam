评分的分层与人工路由

你在 0015 学过：评测集是要长期养护的质量资产，用例不是写一次就完事。这节课往下走一步——用例进来之后，谁来打分？0008 已经把"一把叫 llmJudge 的尺子会犯错、要跨厂商、失灵回退 0.5"讲透了。但 0008 只解决了"这一把尺子可不可信"。真正的评分系统面对的是更大的问题：什么时候该用规则、什么时候该请裁判、什么时候必须叫人来。这节课只讲这件事——把评分从"一把尺子"升级成"一套分层 + 路由 + 校准的判分体系"。

一、先承认服务端直觉对的部分

做过服务端质量的人对"分层拦截"并不陌生：一个请求进来，先过廉价的参数校验，再过中等成本的业务规则，最后才进昂贵的下游依赖。把便宜且确定的检查放前面，把贵且模糊的检查放后面——这个成本梯度的直觉，在评分系统里完全成立。规则 Scorer 就是那层参数校验，LLM-as-Judge 是业务规则，人工是那个你不想每次都惊动的下游专家。

0008 里的 memory_eval 已经天然带着这个梯度：containsScore（子串匹配）、toolCallScore（是否调写文件工具）是确定性规则，llmJudge（跨厂商 gemini）是模型裁判。规则在前、裁判在后、裁判失灵退 0.5——这已经是两层评分的雏形。这一节要做的，是把这个梯度补成完整的三层，并回答一个 0008 没碰的问题：那第三层"人"，到底在什么时刻、按什么信号被叫进来？

服务端的"分层拦截"有一个免费假设：每一层的判定本身都是可信的——参数校验不会看走眼，业务规则不会犯困，下游返回的就是真值。所以分层只是省成本，不改变"判得对不对"。评分系统里这个假设塌了：中间那层（LLM-as-Judge）判据本身会错、会漂、会偏袒自己（0008 已证）。于是评分的分层不只是省成本，它还多背了一个服务端从不需要的任务——给不可信的中间层配一个"什么时候不信它、改叫人"的路由。分层 + 人工路由，是一体两面。

二、三类评分器：优先级与分工

原则一句话：规则看"硬条件是否满足"，Judge 看"语义/策略是否合理"，能写成代码的项由规则主判。三者的边界不是模糊的品味问题，而是可以逐项划清的：

评分器判什么典型判据成本 / 可复现什么时候用
规则 Scorer
（优先级最高）		硬条件是否满足工具调没调、状态对不对、字段结构、有没有触碰禁用动作低 / 完全可复现凡能写成代码的，一律规则主判
LLM-as-Judge语义 / 策略是否合理解释质量、策略妥当性、是否礼貌、是否答非所问中 / 会漂规则判不了的语义维度；高风险、抽检、低置信、变更观察期追加
Human Scorer
（优先级最低、最贵）		业务口径 / 终判依赖内部政策、实时活动规则、需要人定口径的争议高 / 不可复现但权威不承担日常全量；只在四类环节出场（见第四节）

注意这里的用词是"主判"和"追加"：规则不是唯一评分器，而是能判的部分它先判死；Judge 不是无条件全量上，而是在规则覆盖不到、且样本值得花这个成本时才追加。能由规则和 Judge 稳定判断的，就不长期依赖人工——人是最后一道，不是第一道。

三、好的 LLM-as-Judge 不是"请从 1 到 5 打分"

0008 教了怎么不信任裁判（跨厂商、约束输出、失灵回退）。但那都是"裁判已经写好了，如何防它出错"。这一节补上更前置的一问：一个值得被信任的裁判，本身该怎么建？只写一句"请从 1 到 5 给这个回复打分"是最糟的裁判——标准全在模型脑子里，你既不知道它凭什么打这个分，也没法复现和校准。好的 LLM-as-Judge 至少要具备五件东西：

要素为什么必须有
明确评分标准（每档可执行）"4 分和 3 分差在哪"必须写成可判的条款，否则分数是玄学，无法复现也无法对齐
输出 reason不只给分，还要给判定理由——用于事后定位，以及第 07 章按现象把 badcase 聚类
few-shot 示例尤其要放边界样本和判定的 COT 逻辑，让裁判在"擦边球"上向人的口径收敛
周期性校准Judge 与人工复核一致率达到约 85%（参考值）后，才让它进入日常自动化
偏差治理同模型/同系列会偏爱自己风格的回答；引入多个不同 LLM 做对抗打分来稀释这种偏袒
服务端的断言写完就永久生效——你从不会"每季度校准一次 assertEqual"。LLM-as-Judge 不行：模型会升级、Prompt 会改、线上分布会漂，今天一致率 85% 的裁判，三个月后可能已经和人对不齐了。所以裁判不是一次性资产，而是要像监控一样周期性重新校准的活物。0008 的"跨厂商 + 回退 0.5"只防住了裁判失灵，防不住裁判慢慢跑偏——跑偏要靠这一节的金标校准集(见第六节)来抓。

四、人工不是全量兜底，是四个精确落点

新手最容易犯的错，是把人工评分当成"自动分不确定时随便叫人看看"。人贵、慢、不可复现，不适合承担日常全量打分。它只在四类高杠杆环节出场：

  1. 建集定标准：新建评测集时，由业务先定"期望行为"和打分口径。例如"订单已发货、用户拒收、商家参与运费险且商品属特殊类目"时，Agent 该解释退款路径、建议拒收、转人工，还是先发售后单——这种口径机器定不了，必须人先拍板。
  2. 校准 Judge：抽一批样本人工先判，反过来对齐裁判。Judge 能判"回复礼不礼貌、解释清不清楚"，但判不了"叠加优惠后的差价该不该补偿"这类依赖内部政策和实时活动规则的问题。
  3. 处理争议样本：规则和 Judge 结论打架、或多个 Judge 分歧明显的，由人做终判。
  4. 抽查高风险场景：赔付、隐私、合规、投诉升级——即使自动评分通过，也要定期人工复核。这类场景错一次的代价，远高于人工抽查的成本。

人工评分路由：不由单一 Judge 阈值决定

关键设计决策在这里：是否叫人，不应该只看 LLM-Judge 的分数阈值。"分数低于 0.6 就叫人"是错的路由——它既会放过那些分数不低但风险极高的样本，又会为大量分数偏低但其实明确失败的样本白白惊动人工。正确做法是一套多信号的"人工评分路由"：

触发条件为什么要人
自动评分不确定：分数落在通过/不通过边界附近、置信度低、reason 含糊、多个 Judge 分歧明显机器自己都拿不准，硬判就是赌博
变更观察期：新模型 / 新 Prompt / 新工具 Schema / 新业务活动上线抽样进人工，用来发现自动评分的盲区并顺手校准 Judge
规则与 Judge 冲突：确定性规则和裁判给出相反结论由人做终判，并把口径"回收"沉淀成新规则或 few-shot

看清楚：这三条里只有第一条和分数沾边，而且是"落在边界"而非"低于阈值"。变更观察期是按事件触发的，规则冲突是按一致性触发的——都和"分数高低"无关。这就是"路由 ≠ 阈值"的全部含义。

五、分层筛查：把 badcase 层层滤出来

把上面三类评分器串成一条流水作业线，就是评分执行阶段的"分层筛查"。目的有两个：一是省——先用低成本、确定性强的能力快速分流，别让每条样本都去挨一次昂贵的完整判定；二是控质量——控制进入第 07 章 RCA（根因定位）的样本纯度，别把一堆明显通过/明显失败的噪声也塞进去浪费人力。

把这张图和第一节那条"规则→Judge→人工"的横向管线对照：它俩是同一件事的两个视角。横向管线讲"一条样本该由谁判"，分层筛查讲"一批样本如何被逐层过滤"。三层的评分器，就是三层筛查的执行体。

六、Judge 金标校准集：让"可信"变成可测量的数字

"周期性校准"不能靠感觉。要维护一套 Judge 金标校准集（一批有人工标准答案的样本），并在每次 Judge Prompt / 模型版本升级时，固定产出三类统计指标——这是判断"新裁判能不能上线"的门禁：

指标测的是什么不达标意味着
与人工一致性（Kappa 系数）Judge 判定和人工标注的吻合程度（也可是多个大模型对抗打分的一致性）裁判和人的口径分家了，日常自动化的结论不可信
高风险样本漏判率合规、资损、隐私场景里"该拦没拦"的比例最危险的场景正在被放过——这类漏判的代价远超普通误判
边界样本稳定性同批样本重复评测时的分数波动幅度裁判在擦边球上"抛硬币"，同一样本今天过明天挂
服务端的回归测试，"稳定性"是免费的——同一断言重复跑一万次结果一致，你根本不会去测"断言的重复波动"。LLM-as-Judge 把这个也掀翻了：同一个裁判对同一个边界样本重复打分，分数会抖。于是评分系统要专门测一个服务端闻所未闻的指标——裁判自己的重复稳定性。这就是为什么用 Kappa 这种"扣除随机一致后的吻合度"来衡量，而不是简单看"对了几条"：随机也能蒙对一部分，Kappa 把那部分扣掉，才看得清裁判是真的和人对齐，还是碰运气。

七、评分输出：不是 pass/fail，是结构化诊断

最后一个升级，改的是评分的输出形态。0008 的评分终点是一个浮点分过 0.6 阈值得到 pass/fail——这对"能不能上线"够用，但对"上线后怎么修"完全不够。一个 FAIL 只告诉你"这条错了"，不告诉你"错在哪类现象"。所以评分结果应该输出四样东西：

问题分类（现象层）
如"事实性错误 / 答非所问 / 过度承诺"。注意：这是现象，不是根因，也不等于责任模块——它由产品、业务、研发提前协商定义，是给第 07 章根因定位当"现象入口"用的。
问题现象
具体表现的描述，帮人和后续 Agent 快速看懂这条错成什么样。
置信度
自动评分自己有多确信——这正是"人工路由"里判断该不该叫人的关键信号。
判定依据
规则命中了哪条、Judge 的 reason 是什么——让每个结论可追溯、可复核。

为什么值得多输出这三样？因为第 07 章做根因定位时，能基于"现象入口"收敛候选模块，而不是对所有模块无差别排查。"答非所问"和"过度承诺"指向的可疑模块根本不同——现象分类就是那张把排查范围先缩小的地图。0016 的结构化输出，是 0017 根因定位的入场券。

假阳性要追加过程型 + 风险型检查

还记得 0013 讲的假阳性吗——最终回复看起来正确，但执行路径偏离或带风险。只看输出的评分器抓不到它。对这类 case，评分要专门追加两类检查：过程型检查（该调的工具调没调、SOP 顺序对不对，即评 Trace）和风险型检查（有没有碰合规/资损/隐私的红线）。这就是为什么"结果对"不能直接判 PASS——过程和风险这两关不过，照样是 badcase。

八、落到 mino：它走到了哪一层

诚实映射（严格区分"mino 真有"与"方法论层、mino 未实现"）：

mino 真有的（承接 0008，只用已核实符号）：

  • 规则 + 单一跨厂商 Judge 的两层评分已经存在：containsScore（scorer.go:45-88 子串匹配）、toolCallScore（scorer.go:90-125 是否调写文件工具）是确定性规则层；llmJudge（scorer.go:192-237）是模型裁判层。这正是本节"规则 Scorer 在前、LLM-as-Judge 在后"的雏形。
  • 已用跨厂商规避自评偏差：被测 model=claude-sonnet-4-5（config.go:41），裁判 judge=google/gemini-2.5-flash 走 OpenRouter（config.go:42、config.go:50）。
  • 已有失灵回退中性分 0.5：无 key（scorer.go:196-198）、调用出错（scorer.go:208-210）、解析失败（scorer.go:225-227）。
  • 已把能规则化的对抗类交给机械判据：Python 侧 memory_eval_suite/ 对 category==5 用"无高分即通过"（scorer.py:86-92，adversarial_threshold=0.5 见 config.py:20），基于 LoCoMo。这是"能规则化就别请裁判"的活例。
  • 落盘每条完整 JSON（含 Response / ToolCalls / JudgeReasoning）+ results/index.html 人工翻看界面——这给"人工复核"提供了原始素材，但界面本身只是翻看，不是路由。

mino 还没有的（方法论层，未实现，绝不虚构）：

  • 人工评分路由：mino 靠 runner.go:92 一句 if Score<0.6 直接判 FAIL，没有"边界/低置信/冲突→叫人"的路由；results 界面能给人看，但不会主动把样本路由给人。
  • Judge 金标校准集（Kappa / 一致率 85% / 漏判率 / 边界稳定性）：没有任何"用人工标注反过来度量裁判"的机制。
  • 多 Judge 对抗打分：只有单一 gemini 裁判，没有多模型对抗稀释偏差。
  • 结构化问题分类输出：评分结果是分数 + JudgeReasoning，没有"现象分类 / 置信度 / 判定依据"的结构化字段。
  • 假阳性的过程型 / 风险型专项检查：toolCallScore 只判"有没有调写文件工具"，不是完整的 SOP 顺序与风险红线检查。

诚实结论：mino 有"规则 + 单一跨厂商 Judge"的两层评分（0008 已逐符号核实），还没有"三层筛查 + 人工路由 + 可校准 Judge + 结构化诊断输出"的完整评分体系。这节课讲的，正是从"两层评分"长成"评分体系"缺的那几块。

九、关键定义

三类评分器（rule / LLM-as-Judge / human）
优先级从高到低：规则判硬条件（工具/状态/字段/禁用动作，确定性），Judge 判语义与策略（会漂、要校准），人工只在建集、校准、争议、高风险四类环节出场。能写成代码的由规则主判。
人工评分路由（human routing）
决定"何时叫人"的多信号策略，不由单一 Judge 分数阈值决定：自动评分不确定（边界/低置信/reason 含糊/多 Judge 分歧）、变更观察期（新模型/Prompt/Schema/活动）、规则与 Judge 冲突。
分层筛查（tiered screening）
粗筛层（规则 + 轻量 Judge，切走明显通过/失败）→ 精判层（完整规则 + Judge，产出结构化诊断）→ 人工复核层（高风险/低置信/冲突/口径未固化）。既省成本，又控制进入 RCA 的样本纯度。
Judge 金标校准集（golden calibration set）
一批带人工标准答案的样本，每次 Judge Prompt/模型升级时固定产出三指标：与人工一致性（Kappa）、高风险样本漏判率、边界样本稳定性（重复评测波动）。是裁判能否进日常自动化的门禁。
结构化评分输出
不止 pass/fail，还输出问题分类（现象层，≠根因）、问题现象、置信度、判定依据——为第 07 章按"现象入口"收敛候选模块铺路。

十、自测题

Q1：一条客服用例，回复文本读起来完全正确、礼貌，Judge 给了 0.85 稳稳通过。但它根本没调订单查询工具就直接承诺了退款。按这节课，评分系统该怎么处理这条？光靠"规则→Judge"的两层够吗？

不够，这是典型假阳性，必须追加过程型 + 风险型检查。

Judge 看的是语义（回复对不对、得不得体），它对"回复文本"打 0.85 没错——错的是执行路径：跳过了"先查订单状态"这个必需步骤。只看输出的评分抓不到它，因为文本本身没毛病。

处理方式：(1) 过程型检查——用规则评 Trace，检查该调的订单查询工具调没调、SOP 顺序对不对；这是硬条件，能写成代码，属规则 Scorer 主判。(2) 风险型检查——退款涉及资损，属高风险场景，即使自动分通过也应进人工抽查路由。所以正确结论是把这条判为 badcase 并输出"过度承诺"的现象分类，而不是凭 0.85 放行。（这条会在 0017 根因定位里被现象入口接住。）

Q2：有人提议"把人工路由简化成一条规则——Score < 0.6 就叫人复核，>= 0.6 就自动通过，省事又清晰"。用这节课的逻辑指出它错在哪，正确的路由该看哪些信号？

错在把"人工路由"退化成了"单一分数阈值"——这正是本节明确反对的。

它会同时犯两个方向的错：

  • 漏叫人：一条赔付/隐私样本 Judge 给了 0.7，按这规则自动通过——但高风险场景即使自动分通过也该定期人工复核；一条规则判 FAIL、Judge 判 PASS 的冲突样本，若综合分恰好 >=0.6 也被放过，而冲突恰恰是最该叫人终判的。
  • 滥叫人：大量分数偏低但明显失败的样本（粗筛层就能一刀切走的）也被塞给人工，白白消耗最贵的资源。

正确路由是多信号的：① 自动评分不确定（分数落在边界而非单纯低、置信度低、reason 含糊、多 Judge 分歧）；② 变更观察期（新模型/Prompt/工具 Schema/业务活动上线，按事件抽样）；③ 规则与 Judge 冲突（按一致性触发）。三条里只有第一条和分数沾边，且看的是"边界"不是"低于阈值"。这就是"路由 ≠ 阈值"。

Q3（延伸）：mino 的裁判已经跨厂商用 gemini 评 claude 了（0008 讲过），为什么这节课还说它"没有可校准的 Judge"？跨厂商不就够可信了吗？

跨厂商解决的是自评偏差（不让模型给自己打分偏袒），它防的是"裁判系统性偏心"。但它不回答两个不同的问题：

  • 裁判和人对齐了吗？gemini 判得"公正"不等于判得"对"——它可能公正地、稳定地和业务口径差着 20%。要知道这一点，必须有 Judge 金标校准集去测 Kappa/一致率，看它和人工标注吻合到什么程度（参考值约 85% 才进日常自动化）。mino 没有这套。
  • 裁判跑偏了吗？模型会升级、Prompt 会改、线上分布会漂。跨厂商是静态配置，防不住裁判随时间慢慢和人对不齐。金标校准集要在每次 Judge 升级时重测三指标（一致性/漏判率/边界稳定性）——这是周期性动作，mino 也没有。

所以跨厂商是"防偏袒"，校准集是"证对齐 + 防漂移"，两者不能互相替代。mino 做到了前者，还没长出后者——这就是"有跨厂商 Judge，但还没有可校准的 Judge"的准确含义。

十一、承上启下

这节课把评分从 0008 的"一把会犯错的尺子"升级成一套体系：三类评分器分层（规则>Judge>人工，能规则化的绝不请裁判，能自动的绝不叫人）、人工路由（按边界/事件/冲突叫人，不按单一阈值）、金标校准（用 Kappa 等三指标证明裁判和人对齐、没漂移）、结构化输出（问题分类/现象/置信度/判定依据）。最后这一样不是终点——它是钥匙：评分吐出的"现象分类"，正是下一课把 badcase 收敛到"可疑模块"的入口。评出来一条"过度承诺"的 badcase，接下来怎么顺着现象反查到底是哪个模块、哪一步的锅？——那就是根因定位。
