工具即面向模型的 API：描述、namespacing、返回有意义的上下文

你有一个精准的类比：工具 schema ≈ Thrift/protobuf 的 RPC 接口定义。这节课要证明它在三个地方漏水——不是因为类比粗糙，而是因为工具 schema 的消费者不是编译器，是一个概率性的读者。在 conductorservice 里，约 50 个工具的 name / description / 参数 schema 不是给编译器看的接口文档，而是 prompt 的一部分。它们既决定模型能否选对工具、填对参数，又每一轮都实打实地占用 context 预算。这是"工具调用"支柱的地基。

一、先承认类比对的部分

把一个工具的定义看成 IDL 里的一条 RPC，是完全正确的起点。Name 是方法名，参数 schema 是入参结构，返回值是出参。在 mino_server 里这套结构确实存在：每个工具服务实现 AsTools()，产出一个内嵌 eino *schema.ToolInfo 的对象——Name / Desc / ParamsOneOf 三件套，加一个执行函数。参数 schema 用 kin-openapi 的 openapi3.Schema 手写；序列化成 OpenAPI v3 JSON 的机制确实存在（ParamsOneOf.ToOpenAPIV3()，见于 eval 导出路径 conductorservice/tools.go:120），而生产每轮推理的工具序列化由 eino 的 model adapter 在 model.WithTools 内部完成。

先看最朴素的一个工具，它长得就像一条 RPC——短名字、一句注释、两个字段：

agenttypes.NewSerialToolInfo(
    &schema.ToolInfo{
        Name: string(types.TaskTypeUpdateTodo),      // "update_todo"
        Desc: "Update the content of the todo",
        ParamsOneOf: schema.NewParamsOneOfByOpenAPIV3(&openapi3.Schema{
            Type: openapi3.TypeObject,
            Properties: map[string]*openapi3.SchemaRef{
                "label":   {Value: agenttypes.LabelToolSchema},
                "content": {Value: &openapi3.Schema{Type: "string", Description: "The content of the todo"}},
            },
            Required: []string{"label", "content"},
        }),
    },
    svc.UpdateTodo,
)

二、三个漏水点

漏水点 1：消费者是概率性读者，描述就是 prompt

IDL 里字段名起得含糊无所谓——编译器不读语义，调用方是确定性的代码，类型对得上就永远调得对。工具 schema 不是：它的读者是模型，一个按概率生成参数的读者。字段名含糊、描述缺策略，模型就选错工具、填错参数。所以工具描述根本不是接口文档，而是 prompt——带策略、带警告、带劝导。

看 search_web 的描述。它不是"一个 Google 搜索工具"一句话，而是一段 //go:embed 进来的 Go template，带 XML 风格标签：

A powerful Google-backed Internet search tool with multi-topic support and flexible filtering.

This is a Google-like search. Leverage Google search operators (site:, filetype:, OR, -, "...", etc.) to search more precisely.

<quick_answer>
Prefer quick_answer=true for simple factual queries (weather, prices, scores, definitions). Returns an answer with source citations.
Not for research tasks requiring multiple searches or full page reading.
</quick_answer>

searchservice/prompts/web_next.go.tmpl:1-8 — 后面还有 <search_strategy> / <research_for_content_creation> / <multi_language_strategy> 三段。这是 prompt，不是接口注释。

连一个枚举参数都要劝导。time 参数合法取值就是 anytime / past hour / … / past year，枚举本身已经限定了取值范围。但它的 Description 里塞了一整段行为引导（searchservice/tool.go:120-122）：

Time range filter. DEFAULT is "anytime" — ALWAYS prefer this.
WARNING: Setting a specific time range filters OUT official/authoritative pages
(docs, product pages) and mostly returns forum posts and blogs...
Only set a specific range when you explicitly need timeline-bound discussions.

同理 keywords 参数写着 Under 6 decisive keywords per query、One topic/entity per call（tool.go:90-96）。给人类同事的 RPC 永远不会这么写——人类会读文档、有判断力，字段名加一句注释就够，甚至字段名含糊也能靠上下文猜对。确定性调用方不需要劝导，概率性调用方需要。这正是"工具描述其实是 prompt 的一部分"的实锤。

漏水点 2：工具定义吃 context 预算，而且每一轮都重发

IDL 写在文件里，不占运行时开销——接口定义再长也只是编译期的事。工具 schema 不是：它每一轮推理都被完整序列化进 prompt。约 50 个工具的 name + 多行 desc + 完整 OpenAPI 参数 schema，是每 turn 都要付的固定 context 税。

税从哪里收？就这两行——工具信息进入模型的唯一入口：

toolInfos := agent.getCurrentToolInfos()
options = append(options, model.WithTools(toolInfos))

conductorservice/agent.go:1582-1583 — 每一轮 Stream 前都跑一遍。getCurrentToolInfos()（L1965-1979）遍历 toolMap、按 name 字典序排序求稳定顺序、抽出每个工具的 *schema.ToolInfo。全部工具的定义因此每 turn 重进 prompt 一次。

把这条和上一课连起来：工具定义占的每一个 token，都在花模型的注意力预算、都在推高每轮账单。过滤两次不是洁癖，是省预算——用不到的工具连进 prompt 的资格都没有。

漏水点 3：返回值必须限量、必须塑形

一条 RPC 返回 10 万行的 list，顶多是网络和内存问题，不影响"正确性"——调用方是代码，它不会因为 list 太长就读串行。工具返回值不是：返回 10 万 token 的 JSON 会挤掉 context 里别的信息，静默拉低后续所有轮次的质量。这是服务端世界不存在的失效模式——超大返回值在这里是"正确性"问题，不是性能问题。

所以护栏建在服务端，不信任模型自律。search_web 的 count 参数 schema 里已经写了 max: 50, default: 10（tool.go:129）——但那只是给概率性读者的建议，模型可能填 200。真正的护栏是服务端的硬校验：

if args.Count == nil {
    args.Count = gptr.Of(int32(10))
}
if args.Count != nil && *args.Count > 50 {
    return nil, errno.Errorf(errcode.ErrInvalidLLMOutput, "count should be no more than 50")
}

searchservice/service.go:143-148 — 默认 10 条、超过 50 直接报错。schema 里的 "max 50" 是建议，这里才是护栏。

限量之外，返回值还要塑形——回喂给模型的不只是数据，还有"如何解读这份数据"的元指令。在 slides 模式下（isSlidesContext 门控，并非每次 search_web 都加），搜索结果末尾会追加一段提醒，防止模型把 Google snippet 当正文：

const slidesFetchReminder = "<after_search_reminder>\n" +
    "The `Text` fields above are Google's engine-written snippets, not page bodies — treat them as a URL menu only. " +
    "Your next action is `get_web_page_contents` on the strongest URLs from this list; " +
    "do not write research_notes.md, an outline, or slide content from snippet text. " +
    "Parallel-fetching multiple URLs in one tool turn is encouraged.\n" +
    "</after_search_reminder>"

searchservice/service.go:31-36 定义、service.go:170-173 仅在 isSlidesContext 为真时注入 — 主动引导模型下一步去 get_web_page_contents，而不是拿 snippet 当研究材料。

三、namespacing：命名即契约

没有语言级的命名空间可用——工具名是一个扁平的字符串全局空间（就是 TaskType 常量）。于是 namespacing 靠命名约定兜底：全部 snake_case、结构是"动词_名词"、同族工具共用前缀。这样模型在一堆工具里靠前缀就能归类、缩小选择范围。

还有一个"藏在契约里的 UI 契约"值得一提：几乎每个工具都注入一个公共 label 参数（types.go:400-403 的 LabelToolSchema），描述里写着 Always output this parameter FIRST before all other parameters。这是把前端进度文案（"Searching for latest news"）硬塞进工具契约——让模型在真正干活之前先吐一句给用户看的话。工具 schema 不只是功能接口，它还兼着"产品交互"的活儿。

四、一张对照表

维度RPC / IDL（确定性调用方）工具 / 面向模型的 API（概率性调用方）
字段名与描述够用即可，含糊也能靠代码上下文调对是 prompt：需策略、警告、劝导，含糊就选错（tool.go:120-122）
定义的运行时成本写在 IDL 文件，不占运行时每 turn 完整序列化进 prompt = 固定 context 税（agent.go:1582-1583）
返回超大结果网络/内存问题，不影响正确性挤占 context、静默拉低后续质量 → 正确性问题（service.go:143-148）
命名空间语言级 package/namespace扁平字符串，靠 snake_case 前缀约定（event.go:622-677）
本课唯一要带走的判断：写工具时，脑子里的读者不是编译器、也不是你的同事，而是一个会读劝导、会忽略建议、会被超大返回值淹没的概率性读者。于是三件事变成硬约束——描述要当 prompt 写、定义要按预算裁、返回值要限量并塑形。

五、关键定义

工具（面向模型的 API）
提供给模型调用外部能力的接口定义，其 name / description / 参数 schema 本身就是面向模型的 API 文档，是 prompt 的一部分并占用 context 预算。载体：conductorservice/tools.go:GetAllToolServices，经 model.WithTools 传入。
namespacing（命名空间约定）
在没有语言级命名空间时，用 snake_case 的"动词_名词"+ 家族共前缀（如 search_*）给扁平的工具名空间制造结构，帮助模型归类与选择。载体：event.go 的 TaskType 常量。
不可靠调用方假设（unreliable-caller assumption）
把模型当成会传错参、会忽略建议、会填超限值的调用方来设计工具边界：服务端硬校验、默认值兜底、可读错误。载体：service.go:143-148 的 count 校验。
返回值塑形（response shaping）
返回给模型的不只是数据，还包括"如何解读这份数据"的元指令——限量、过滤、追加提醒，主动把模型推向正确的下一步。载体：service.go:31-36 的 slidesFetchReminder（slides 模式专属）。

六、自测题

Q1：search_web 的 time 参数是个普通枚举（anytime / past hour / …），但它的 Description 里塞了一整段 "DEFAULT is anytime — ALWAYS prefer this. WARNING: Setting a specific time range filters OUT official/authoritative pages…"（tool.go:120-122）。为什么一个枚举参数需要这么长的劝导文字？换成给人类同事的 RPC 会这样写吗？

因为这个 schema 的消费者是概率性读者（模型），不是确定性的调用方代码。枚举本身只能限定"合法取值"，却拦不住模型在不该用时主动选 past week——它是按概率生成参数的。所以描述里必须做 prompt 级的行为引导：默认值 + 反面警告 + 何时才该偏离。给人类同事的 RPC 不会这么写：人类会读文档、有判断力，字段名加一句注释就够，甚至字段名含糊也能靠上下文猜对。这正是"工具描述其实是 prompt 的一部分"的直接体现——类比在这里漏水。

Q2：count 参数的 schema 描述已经写了 "max: 50"，为什么 service.go:146-147 还要在服务端再判一次"> 50 就报错"？既然模型能读到 schema，两处校验不是冗余吗？

不冗余，因为两处的信任级别不同。schema 里的 "max 50" 是写给概率性读者的建议，模型可能忽略、可能填 200。服务端的硬校验（L147 直接返回 ErrInvalidLLMOutput）才是真正的护栏。它保护的不是数据结构合法性，而是 context 预算——一次返回上百条结果会挤占 context、静默拉低后续所有轮次的质量。这是服务端世界没有的失效模式：传统 RPC 里返回超大 list 顶多是性能问题，在 Agent 里它是"正确性"问题。同理 count 默认 10（L143-144）也是主动的返回值节制。
