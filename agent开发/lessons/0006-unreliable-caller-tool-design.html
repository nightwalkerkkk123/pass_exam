不可靠调用方假设：分层超时、JSON 修复回注、并发 context-too-long

你已有正确的地图："工具 ≈ RPC 边界"。这节课只做一件事：站在 Conductor 主 Agent 工具调用层的"防线"上，看清 RPC 类比在哪三处漏水——因为这一次，调用方是模型，一个会传坏 JSON、会跑三十分钟、会因为自己上下文太长而调用失败的调用方。

先定位战场。模型每一步推理吐出的 assistant message 带着 ToolCalls，在真正执行前要过一道关卡：先 categorizeToolCalls 按"名字/JSON 合法性/是否可并行"分诊成若干组，再由 executeToolCallGroups 逐组下发；每个工具的执行又用 errgroup 拆成两条 goroutine——一条真跑工具，一条转发事件并计时。这层代码位于 biz/domain/dmservice/agents/conductorservice/toolcall.go，夹在 LLM 推理循环（agent.go）与 40+ 具体工具实现之间。它的职责只有一句话：把"不可靠的模型输出"翻译成"可控的工具执行"。

一、先承认类比对的部分

把工具调用看作一次 RPC，是完全正确的起点。工具有名字、有参数 schema、有返回值、有超时——和你写了十年的服务间调用同构。于是你自然会想到那套成熟的老问题：参数校验、超时、重试、幂等、背压、并发上限。方向没错，代码里这些机制一个都不少。

但 RPC 世界有一个你从不需要说出口的前提：调用方是确定性、可信的。参数结构由编译期 / IDL（Thrift、protobuf schema）保证合法，一个语法坏掉的请求根本到不了你的 handler。错误处理也简单：把 error code 返回给调用方，调用方按代码里写死的分支处理。

现在把调用方换成一个语言模型。三个前提同时塌陷。下面逐个看。

二、漏水点一：它会传语法坏掉的 JSON——于是系统"修好了也不执行"

分诊的第一件事，是拿工具名去 toolMap 里查（查不到就建一个 IsInvalid 组，原因 "invalid tool name"，toolcall.go:84-95）。第二件事，是把参数字符串反序列化。这里出现了 RPC 世界不可能出现的分支——反序列化会失败：

// toolcall.go:98-109
m := map[string]any{}
if err := codec.UnmarshalFromString(toolCall.Function.Arguments, &m); err != nil {
    ctx.Log().Warnf("[Conductor] tool call json format error: %v ...")
    // 修复JSON格式
    if len(toolCall.Function.Arguments) == 0 {
        toolCall.Function.Arguments = "{}"
    } else {
        repairJson, err := jsonrepair.RepairJSON(toolCall.Function.Arguments) // TODO
        if err == nil {
            toolCall.Function.Arguments = repairJson
        }
    }
    ...

空参数补一个 "{}"；否则请出第三方库 jsonrepair.RepairJSON 试着把坏 JSON 修回合法。这段代码本身就是漏水的产物——传统 RPC 里没有任何一行是为"调用方传了半个 JSON"准备的。

真正扎眼的不是"修 JSON"，而是修好之后的选择。对非流式工具（toolInfo.NewStreamingState == nil），即使 RepairJSON 成功，系统仍然拒绝执行，转而把一整段自然语言错误当作 tool result 回注给模型： 
// toolcall.go:117（InvalidReason，节选）
"Tool call JSON argument format error. Possible causes: previous API call
 encountered max output tokens truncation ... System successfully repaired JSON
 format, but to avoid executing incorrect calls, the system did not execute this
 tool call. Please check and correct the original JSON arguments and retry ...
 Correction methods: reduce output tokens, split into multiple tool calls ..."
为什么修好了还不执行？因为 jsonrepair 修的是语法，救不了语义。坏 JSON 的常见成因是上一次 API 调用撞上 max output tokens 截断——被截断处补全出来的值很可能是错的。执行一个"猜出来的调用"，比拒绝执行危险得多。于是系统宁可把错误说明喂回给这个不可靠的调用方，让它自己改原始参数重发。

这就是 RPC 世界不存在的东西：error message 成了下一轮 prompt 的一部分。这条 InvalidReason 会经 executeInvalidTool 写进 InvalidToolTaskData.Message（toolcall.go:628），再由 AddToolResult（toolcall.go:207）变成模型可见的 tool result。你从没在 RPC 里"把错误塞回调用方让它自我修复"——因为 RPC 的调用方是死代码，不会读你的错误说明改自己。

顺带记住那个 NewStreamingState==nil 的判断：流式工具不走这条拒绝逻辑。流式工具的参数是随 token 增量拼接的，中途"看起来不合法"是正常状态，用一次快照去否决它没有意义。

三、漏水点二：超时不是"总时长"，而是"事件间隔"心跳

看到超时常量表，服务端工程师的第一反应是"每个工具的最长执行时间"。看清楚注释：

// toolcall.go:28-33
var defaultToolCallTimeout = time.Minute*5 + time.Second*30 // 事件间隔超时时间 5.5min

// TODO: 也许不需要 timeout
var subAgentToolCallTimeout = time.Minute * 30       // 子代理工具调用超时时间 30min
var slidesGenerateToolCallTimeout = time.Minute * 10 // 幻灯片生成工具超时时间 10min
var acpExecToolCallTimeout = time.Minute * 30        // ACP 执行工具超时时间 30min，与 acpTaskTimeout 对齐

注释白纸黑字写着"事件间隔超时时间"。这个数字量的不是工具跑多久，而是两个事件之间的空档——心跳。证据在 forwardToolEvents 里：

// toolcall.go:692-707（节选）
for {
    _ctx, cancel := context.WithTimeoutCause(ctx, timeout, timeoutErr)   // 每轮重置计时
    ...
    event, err := innerEventStream.Receive(eventCtxWithMinoCtx)
    cancel()
    if err != nil {
        if errcode.IsErrStreamClosed(err) { return nil }               // 流正常关闭
        if errcode.IsErrToolCallEventIntervalTimeout(err) { return timeoutErr }
        return err
    }
    outerEventStream.Send(ctx, event)   // 转发事件
}

for 循环里，每收到一个事件就 cancel() 旧计时、下一轮用 context.WithTimeoutCause（toolcall.go:693）重新起表。所以一个工具跑 20 分钟，只要每隔 1 分钟吐一个事件，defaultToolCallTimeout=5.5min 永远不会触发。它测的是"卡住了多久没动静"，不是"一共跑了多久"。

"心跳超时"这个设计本身是对的——工具是长流式的、可能真跑 30 分钟但持续吐事件。可它引出了一个隐藏漏水：如果一个工具在长等待期间根本不发心跳，它的"事件间隔超时"就被迫等于"工具总超时"。 desktop_file_transfer 就是这样的极端例子，代码注释直接点破： 
// toolcall.go:34-37
// desktop_file_transfer 是"一次性长等待"工具：传输期间不发心跳，整条链路阻塞在
// PopResult 等终态。事件间隔超时 ≡ 工具总超时，必须 ≥ fileTransferDefaultTimeoutSeconds
// 的客户端总超时（当前 30min）。独立常量与 ACP 解耦，避免后者调时长影响这边。
var fileTransferToolCallTimeout = time.Minute * 30
RPC 里你设一个超时，含义永远是"总时长"，不会因为"这个工具中途发不发心跳"而语义漂移。这里同一个 timeout 变量，对 subAgent 是"心跳间隔"、对 file_transfer 却退化成"总超时"——同名不同义，全靠工具行为决定。所以给它一个独立常量、必须 ≥ 客户端总超时，否则会在传输中途被误杀。

四、漏水点三：并发 context-too-long 的 FIXME——新问题还没解干净的活证据

模型调用可能因为一个 RPC 从不会有的理由失败：调用方自己的上下文太长了。这个错误由 errcode.IsErrLLMPromptTooLong 判定（对应错误码 ErrLLMPromptTooLong=100007009）。它有两条命运，取决于工具走的是串行还是并行路径。

先看单工具执行的默认策略：toolErr 默认不上抛，只有一种错误例外——

// toolcall.go:590-593
// toolErr 不需要上抛，除非是 context 超长错误
if toolErr != nil && errcode.IsErrLLMPromptTooLong(toolErr) {
    return execResult, toolErr
}

串行路径把这个错误连同已有的 result 一起上抛（executeSerialTools，toolcall.go:286-291），一路经 executeToolCallGroups 冒到 agent.go 的主循环。在那里它被自愈：

// agent.go:937-949（节选）
if errcode.IsErrLLMPromptTooLong(err) {
    // 如果没有 summary，强制生成（跳过条件检查）
    if agent.lastSummary == nil {
        if summaryErr := agent.summarizeForce(ctx); summaryErr != nil { return summaryErr }
    }
    agent.applySummaryImmediately(ctx)   // 立即应用压缩上下文
    continue                             // 压缩后重来
}

强制 summarizeForce 压缩历史、applySummaryImmediately 换上更短的上下文，然后 continue 重跑这一步。上下文太长？那就把它压短再来——串行路径能自愈。

但并行路径遇到同一个错误，是这样的：

// toolcall.go:333-336
if errcode.IsErrLLMPromptTooLong(err) {
    // FIXME: 并发调用时要处理这种报错非常麻烦，且目前暂时没有并发工具可能出现这种报错，暂不处理
    return nil, errno.Errorf(errcode.ErrInternalServer, "unexpected context too long error: %+v", err)
}
这条 FIXME 是最诚实的漏水证据。传统 RPC 从不会因为"调用方自己的上下文太长"而失败——调用方的内部状态与 handler 无关。而在 Agent 世界，模型的上下文就是它的全部状态，长到一定程度调用直接失败，这是 Agent 独有的新失败模式。

更值得咀嚼的是团队对它的态度：串行路径已经有了优雅解（压缩重试），并行路径却"非常麻烦"，于是坦白"目前暂时没有并发工具可能出现这种报错，暂不处理"，直接包成 ErrInternalServer 放弃。注意最后半句——这意味着这条放弃路径当前是理论隐患，尚未在线上被触发。它不是 bug，是一个被显式标注、被有意推迟的技术债。这正是 Agent 工程的真实质地：新问题冒出来，你先接住能接住的（串行），把接不住的诚实地写进 FIXME，而不是假装它不存在。

五、两条 goroutine：为什么"跑工具"和"计时"要分开

回到 executeToolCall 的核心结构。它用 errgroup 起了两条 goroutine，一条跑工具、一条转发事件并计时，最后 g.Wait() 等两者。为什么不写成一条？因为工具执行是阻塞在流上的——只有独立的第二条 goroutine 在旁边盯着"多久没来事件"，才能在工具卡死时把它掐掉。

并发多个工具时，外面还套了一层有上限的并发：executeParallelTools 用 concurrency.NewGroupTaskRunnerG 建 runner，AddTask 挂任务，StartAndWait 后按 Promises() 顺序收结果——结果顺序与 toolCalls 顺序严格一致（toolcall.go:351-352 的 NOTE 特意强调了这点）。上限就是开头那张图里的 5 / 3。

六、关键定义

不可靠调用方假设（unreliable-caller assumption）
把模型当成一个会传错参数、会超时、会重复调、会因自身上下文过长而失败的调用方来设计工具边界：参数修复、可读错误回注、分层超时、并发上限，都是这个假设的产物。
错误回注（error feedback into context）
工具（或分诊）失败时，不把错误当异常吞掉，而是写成人话作为 tool result 回注给模型让它自我修复。error message 因此成为下一轮 prompt 的一部分。载体：toolcall.go:117 的 InvalidReason → InvalidToolTaskData.Message（:628）→ AddToolResult（:207）。
分层超时 / 事件间隔超时（tiered / event-interval timeout）
按工具预期行为设不同超时；且这里的超时量的是"两个事件之间的间隔"（心跳），不是工具总时长。forwardToolEvents 每轮用 context.WithTimeoutCause 重置计时（toolcall.go:693）。default 5.5min / slides 10min / subAgent 30min / ACP 30min / desktop_file_transfer 30min。
context-too-long 自愈 vs 放弃
模型上下文过长（ErrLLMPromptTooLong=100007009）时：串行路径上抛到主循环触发 summarizeForce 压缩后 continue 自愈；并行路径因 FIXME 直接包成 ErrInternalServer 放弃。

七、自测题

Q1：模型发来一个 JSON 参数损坏的非流式工具调用，系统调 jsonrepair.RepairJSON 成功修好了。系统会执行这个工具吗？为什么这样设计？

不会执行。toolcall.go:112-120：即使 RepairJSON 成功，只要 toolInfo.NewStreamingState==nil（非流式工具），系统仍把它标为 IsInvalid、不执行，而是把 :117 那段错误说明作为 tool result 回注给模型，让模型自己修正原始 JSON 重试。

设计动机：jsonrepair 修的是语法不是语义。坏 JSON 常常来自上一次 max output tokens 截断，被补全处的值可能是错的；执行一个"猜出来的调用"比拒绝执行更危险——宁可让不可靠的调用方自己重发。（流式工具因为参数是增量拼接的，不走这条拒绝逻辑。）

Q2：defaultToolCallTimeout 是 5.5min。一个工具跑了 20 分钟，但每隔 1 分钟就吐一个事件，会触发超时吗？为什么 desktop_file_transfer 要单独给 30min？

不会超时。forwardToolEvents（toolcall.go:692-693）每轮循环都用 context.WithTimeoutCause 重置计时，它测的是"两个事件之间的间隔"（心跳），不是工具总时长——只要 5.5min 内有下一个事件就永远不触发。

desktop_file_transfer 给 30min（toolcall.go:34-37 注释）是因为它是"一次性长等待"工具：传输期间根本不发心跳、整条链路阻塞在 PopResult 等终态，所以它的"事件间隔"实际 ≡ 工具总时长，必须 ≥ 客户端总超时（当前 30min），否则会在传输中途被误杀。

Q3（进阶）：为什么同样是 context-too-long，串行工具能自愈而并行工具直接放弃？如果你要补上并行路径的处理，难点在哪？

串行路径一次只跑一个工具，出错时上下文状态是干净、单一的，可以安全地整体压缩后 continue 重跑（agent.go:937-949）。并行路径同时在跑多个工具、各自可能已经产生了部分结果和事件，一旦其中一个报 context-too-long，你没法简单地"压缩上下文重跑"——其它并发任务的状态怎么办？已发出的事件如何回滚？这正是 FIXME 里"非常麻烦"的所在（toolcall.go:334）。团队的务实选择是：既然当前没有并发工具会触发它，就先包成 ErrInternalServer 放弃，把它记为技术债而非假装解决。
