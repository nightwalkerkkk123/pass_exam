Badcase 根因定位

你在 0016（评分输出结构化：分层筛查 + 问题分类=现象入口）学过——评分不只吐 pass/fail，还要给出问题分类、问题现象、置信度、判定依据，让 badcase 带着"现象入口"进下一站。这节课往下走一步：现象入口只告诉你"看起来像哪类问题"，它不等于责任模块。本课只做一件事——把一条错例稳定地追到"责任模块 + 可修复原因"，而不是停在"这条答错了"。

一、直觉先行：服务端"看堆栈定位"，在 Agent 里失灵

服务端排障有一个你用惯了、深到不会去质疑的动作：程序崩了，翻 stack trace，第一行就写着 NullPointerException at OrderService.java:142——责任模块和责任行是运行时免费给你的。你不需要"推断谁的错"，异常传播链把它直接摊在你面前。

Agent 的端到端失败没有堆栈。用户问"已发货的订单能退款吗"，Agent 答"可以，马上帮您办"——这条会话既没抛异常、也没 500，它只是业务上错了（跳过了订单状态校验、违反了"已发货需先协商"的 SOP）。你手里只有一个"端到端结果不对"的表象，没有任何一行日志自动告诉你责任落在意图识别、槽位抽取、工具选择、还是回复生成。责任模块在 Agent 世界里不是被抛出来的，是要靠 RCA 反推出来的。"看堆栈就知道哪一行崩了"这个服务端直觉，在这里整个失效。

所以"根因定位"（RCA，Root Cause Analysis）在 Agent 里是一件独立的、必须专门设计的工程活儿：把"端到端错了"这个表象，一步步收敛成"哪个模块、什么可修复原因"。

二、badcase 从哪来：不止离线评测集

第一个容易踩的窄化，是以为 badcase 只从离线评测集里跑出来。真实的 badcase 来源是两条通路、多个入口：

通路入口举例特点
离线评测评测集不通过、版本回归失败可复现、有 golden、批量
线上运营线上真实会话覆盖长尾、不可控
人工质检抽样有人工标注、质量高
投诉工单 / 低满意度样本用户负反馈驱动
监控告警指标异常触发

关键设计（首图已画）：这两条通路在入口就汇入同一张 badcase 任务表、同一套 RCA Pipeline。不要为线上和离线各建一套分析口径——否则同一个根因会在两套系统里被拆成两个标签，既没法统计，也没法聚类。

三、端到端失败只是表象：真因散在十来个环节

为什么不能"端到端错了就说是模型不行"？因为一条会话是一串环节的接力，任何一环偏一点，都可能在终点被放大成错误结论（这正是 0013 讲的错误级联）。真因可能落在：

CI/服务端里，"哪个函数错了"和"哪个断言红了"几乎是同一件事——断言红在哪，锅就在那附近。Agent 的评分结论和责任模块是脱钩的：0016 的评分说这条现象是"过度承诺"，但"过度承诺"可能是风险拦截没拦、也可能是回复生成擅自加戏——现象入口只缩小了怀疑范围，不等于定责。把现象直接当根因，是 RCA 最常见的偷懒。

四、RCA 五步通用链路（本课核心）

把"表象 → 责任模块 + 可修复原因"这段路，拆成一条可复用的五步链路。记住它的骨架节奏：先收集证据 → 再收敛范围 → 最后定责落盘。

① 证据汇总：把链路"摊开"

按 sessionId / chatId / traceId / 评测运行 ID 找到对应链路日志，汇总：用户输入、Agent 回复、各功能模块的输入输出、prompt、工具调用、异常、耗时、中间产物。工程约束：状态以持久化任务记录为准，运行时工作区只存中间产物——这样一条 badcase 分析崩在半路，可以恢复接着跑，而不是从头再来。

② 范围收敛：现象 × 模块映射表

不要对全部模块无差别排查（成本高、噪声大、结论散）。维护一张"问题现象 × 功能模块"的映射：命中就走确定性路径直接缩圈，未命中再让 LLM 兜底缩圈。

问题现象（0016 给的入口）优先排查的候选模块
答非所问意图识别 · Query 改写 · 知识筛选
过度承诺风险拦截 · 回复生成
订单未澄清槽位抽取 · 上下文判断
事实性错误FAQ 检索 · 知识检索 · 回复生成

这张表只缩范围，不等于最终定责。最终责任仍要结合 Trace、模块输入输出、工具返回和规则/Judge 结论确认。

③ 分模块诊断：逐个读 IO，输出结构化结论

对候选模块逐个读 input / output / prompt / 工具返回 / 异常日志，每个模块产出：问题摘要 + 关键证据 + 改进建议 + 可归因的结构化字段（状态 PASS / SOFT_PASS / FAIL，SOFT_PASS = 基本通过但有轻微瑕疵）。

举例（"事实性错误"样本）：先看 FAQ 检索是否召回正确政策 → 再看知识检索是否保留关键片段 → 最后看回复生成是否忠实使用这些证据。若前两段都通过、最终回复仍说错，就把回复生成标 FAIL，前两个模块标 PASS/SOFT_PASS。诊断到此，责任已经被证据夹逼出来了。

④ 责任判定：消费诊断结果，定责 + 分类

这一步消费"问题现象 + 分模块诊断结果 + 根因知识库"，输出四样东西：

输出字段含义上例（事实性错误）
责任模块锅落在哪个模块（主责/次责）回复生成模块
问题分类面向报告/看板的大类模型指令遵循（语义理解）
问题枚举更细的失败类型证据使用错误
修复建议指向具体可改动的东西回复生成 Prompt / 引用约束 / 忠实性校验

注意区分两个层：问题分类是给人看报告的大类（语义理解 / 知识召回 / 工具执行 / 风险防控），问题枚举是更细、能绑定处理动作的失败类型。定责用三层策略（下一节）而非全交给 LLM。

⑤ 结构化落盘与渐进展示

根因结果不能只躺在一次性报告里，要结构化写入任务记录，至少含：问题分类 / 问题现象 / 问题枚举 / 责任层 / 责任模块 / 置信度 / 详细报告 / 修复建议。前端渐进展示：

五、定责策略：确定性优先 → 模型辅助 → 人审兜底

责任判定不该全交给一个 LLM（那会重蹈 0008 里"裁判不可信"的覆辙）。落地按三层，和评分同一套哲学：

三层定责对应到"责任判定"内部就是：严重模块结论直接定责 → 规则引擎匹配确定性模式 → LLM 汇总复杂链路中的责任传递关系。既保留确定性，又能处理多模块耦合。

六、根因标签体系与 badcase 聚类

RCA 的产物不是一堆零散描述，而要沉淀成稳定、可统计、能指向明确 owner 与处理动作的根因标签体系。标签稳定了，两件事才成立：

能绑定解决角色（owner）
每个问题枚举可以映射到处理角色：运营可配置 / 算法需优化 / 工程需修复 / 业务需定口径。例："用户申请已发货订单退款，Agent 没先查订单状态就承诺可退" → 问题分类"工具覆盖与执行"、问题枚举"关键工具未调用"、责任模块"退款 Skill / 退款流程执行模块"。
能聚类（clustering）
同一根因 / 同一业务场景 / 同一工具 / 同一知识点的失败，自动聚成问题簇。报告就不再是"列 100 条 badcase"，而是一句可行动的结论："退款已发货场景中，Agent 23 次跳过订单状态校验，主要集中在 v1.8 Prompt。"

为什么聚类这么重要？因为 100 条散装 badcase 没法排优先级，而"23 次同一根因、集中在某版本 Prompt"直接就是一张可派单的工单——这正是下一课"从根因到行动项"的原料。

七、落到你的项目：mino 处在哪一格

诚实映射（严格区分"mino 真有"和"方法论层、mino 未实现"）：

mino 没有 badcase RCA 子系统。这一整节讲的现象×模块映射、分模块诊断、责任判定、根因标签体系、聚类——都属于方法论层，mino 未实现，代码里没有任何对应符号，本课不虚构。

mino 真有的，只是"证据的最原始地基"这一小格：memory_eval 落盘时每条用例写一份完整 JSON（含 Response / ToolCalls / JudgeReasoning），并生成 results/index.html 供人工翻看。对照 RCA 五步，这恰好只覆盖了第①步"证据汇总"里最底层的一层——原始报告 JSON，而且还是单一 memory 能力、非多模块链路的证据。

诚实结论：mino 有 badcase 的原始证据（报告 JSON + 人工翻看界面），还没有把证据变成"责任模块 + 可修复原因"的 RCA 流水线——没有现象×模块映射、没有分模块诊断、没有责任判定、没有根因标签体系、没有聚类。它停在"人能一条条看"，还没到"系统能自动定责"。

八、关键定义

根因定位 / RCA（Root Cause Analysis）
把一条 badcase 的端到端表象，稳定追溯到责任模块 + 可修复原因的过程。核心不是"判它错了"（那是评分的活），而是"错在谁、怎么修"。
现象 × 模块映射表
由"问题现象"（0016 给的现象入口）指向"优先排查的候选模块"的映射。作用是缩小范围，命中走确定性路径、未命中让 LLM 兜底缩圈；它不等于最终定责。
分模块诊断（PASS / SOFT_PASS / FAIL）
对候选模块逐个读 IO/prompt/工具返回/异常，输出结构化结论。SOFT_PASS = 基本通过但有轻微瑕疵。这一步用证据把责任夹逼出来。
问题分类 vs 问题枚举
问题分类=面向报告/看板的大类（语义理解 / 知识召回 / 工具执行 / 风险防控）；问题枚举=更细、能绑定处理动作的失败类型（如"关键工具未调用""证据使用错误"）。
根因标签体系 + 聚类
稳定、可统计、能指向 owner 与处理动作的标签集合；聚类把同根因/同场景/同工具/同知识点的失败聚成问题簇，让报告从"列 N 条"变成"一句可行动的结论"。

九、自测题

Q1：0016 的评分把某条 badcase 的"问题现象"标成了"过度承诺"。有人说"那责任模块就是回复生成，直接定责就行"。用本课逻辑指出他错在哪，并说明正确该怎么走。

他把"现象入口"当成了"最终定责"——这正是本课 leak 点破的偷懒。"过度承诺"这个现象，候选模块至少有两个：风险拦截（该拦没拦）和回复生成（擅自加戏承诺）。映射表只负责缩小怀疑范围，不负责下结论。

正确路径：走完 RCA 后三步。③ 分模块诊断——读风险拦截模块的 IO：它到底有没有识别到"已发货订单"这个风险点并尝试拦截？再读回复生成：它拿到的证据/约束里有没有"不得无条件承诺退款"？④ 责任判定——如果风险拦截压根没触发，主责在风险拦截；如果风险拦截给了约束、回复生成却无视约束照样承诺，主责在回复生成。证据决定定责，现象只决定从哪几个模块开始查。

Q2：一条 badcase 的分模块诊断结果是：FAQ 检索 PASS、知识检索 PASS、回复生成 FAIL（正确政策被解释错）。为什么这条应该走"规则/严重模块直接定责"而不是"扔给 LLM 汇总责任传递"？什么样的 badcase 才需要 LLM 那一层？

因为这条的责任已经被证据夹逼成单点了：前两段检索都通过（证据齐全且正确），唯独回复生成 FAIL——责任传递关系是确定的、无歧义的，主责直接落回复生成即可。此时上 LLM 纯属给判据平白注入不确定性（0008 的教训），换不来任何精度。这属于"确定性优先"里的"严重模块结论直接定责"。

需要 LLM 汇总的是多模块耦合、责任传递链复杂的情形：比如检索召回了错的片段、回复生成又"忠实地"用了这个错片段——到底该怪检索给错料，还是怪生成没做忠实性/合理性二次校验？这种责任在多个模块间弥散、规则难以一刀切的链路，才交给 LLM 做语义归类与责任汇总，且结论若低置信/与规则冲突，再进人工兜底。

Q3（延伸）：团队已经能把每条 badcase 定责了，报告里老老实实列出了 100 条"责任模块 + 修复建议"。leader 看完说"没法排期"。缺了本课哪个能力？补上它之后报告会变成什么样？

缺的是聚类（clustering）与根因标签体系。100 条散装 badcase 即使每条都定了责，也无法回答"先修哪个、影响多大、归谁"——因为没有把"同一根因/同一场景/同一工具/同一知识点"的失败聚成问题簇，也就没有量级和 owner。

补上之后，报告从"列 100 条"变成"一句可行动的结论"，例如："退款已发货场景，Agent 23 次跳过订单状态校验，集中在 v1.8 Prompt。"——有场景、有量级（23 次）、有版本定位、能绑定 owner（工程需修复 / 退款 Skill 负责人）。这样才排得出优先级，也正好是下一课把"根因"翻译成"行动项"的输入。这也要求根因标签稳定、可统计：标签一乱，同类问题被拆成零散标签，聚类就失效了。

十、承上启下

这节课把一条 badcase 从"端到端表象"稳定追到了"责任模块 + 可修复原因"：统一入口（两通路汇一表）→ RCA 五步（收集证据→收敛范围→定责落盘）→ 三层定责（确定性优先→模型辅助→人审兜底）→ 沉淀成可统计、可聚类、可绑 owner 的根因标签。但 RCA 的终点"修复建议"仍然只是一句话——"优化 Prompt""加强训练"这种泛化表述修不了任何东西。下一课要解决的是：怎么把"原因"翻译成带失败范围、证据、具体动作、owner、验收方式和优先级的行动项，并让它闭环回评测集，形成全链路运转。

有任何不清楚的地方，回到会话里问你的 AI 老师——可以要求 ELI5、ELI14，或让我带你把 mino 的 memory_eval 落盘 JSON（Response/ToolCalls/JudgeReasoning）摊开，对照本课 RCA 第①步"证据汇总"看它到底只覆盖了哪一层。下一课：0018 从根因到行动项与全链路闭环——把根因翻译成可验收的动作，并让 badcase 变成 bug/工单/回归用例回流评测集。
