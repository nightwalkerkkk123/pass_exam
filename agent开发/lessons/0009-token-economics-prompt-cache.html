token 经济学：输入/输出/缓存读写的非对称计价

你把 prompt cache 类比成 Redis / CDN：命中省钱、读近乎免费、写不过是多存一份。方向对。但这本账不是你记的，是厂商记的——一次调用被拆成四类 token，每类单价不同，而且读写不对称到会直接颠覆你的成本直觉。这节课用 mino_server 里真实的单价表和成本函数，逐字把这本账拆开。

一、先承认类比对的部分

把模型调用看作无状态服务是对的：每次调用都得把全部输入随请求带过去，厂商从零读一遍。prompt cache 的动机也确实像 CDN——"别让厂商重新处理你上次已经处理过的前缀"。这个视角让你自然想到 TTL、前缀命中、写一次读多次摊薄。方向没错。

问题出在计量权：服务端缓存省的是你自己机器的 CPU/内存，账你自己心里有数；prompt cache 的账由厂商记，而且厂商把一次调用拆成四类分别计价。要谈成本，先得认识这四类。

二、四类 token：一次 usage 被拆成四路

Anthropic 回传的 usage 有四个字段，mino_server 在 common.go 里用一大段注释把它钉死了口径（common.go:146-156）：

//   "input_tokens": 2328,                // 仅表示请求中最后一个缓存断点之后的 token, 并非发送的所有输入 token
//   "cache_creation_input_tokens": 2200, // 写入缓存
//   "cache_read_input_tokens": 37687,    // 从缓存读取
//   "output_tokens": 176
// 总输入 = input_tokens + cache_creation_input_tokens + cache_read_input_tokens

第一个坑就在这里：Anthropic 的 input_tokens 不是"总输入"，它只是最后一个缓存断点之后的那一小截。服务端工程师最容易在这对不上账——你以为拿到的是总输入，实际只是零头。真正的总输入要把三者相加。代码正是这么合并的（convert.go:86-93）：

// Claude 的 input_tokens 仅表示最后一个缓存断点之后的 token
// 需要加上 cache_creation_input_tokens 和 cache_read_input_tokens 才是总输入
usage := &common.TokenUsage{
	PromptTokens:        int(e.Message.Usage.InputTokens + e.Message.Usage.CacheCreationInputTokens + e.Message.Usage.CacheReadInputTokens),
	CompletionTokens:    int(e.Message.Usage.OutputTokens),
	CacheCreationTokens: int(e.Message.Usage.CacheCreationInputTokens),
	CacheReadTokens:     int(e.Message.Usage.CacheReadInputTokens),

合并进内部的 common.TokenUsage（common.go:184-205）后，有四个计费字段：PromptTokens（含缓存的总输入）、CompletionTokens（输出）、CacheCreationTokens（写缓存）、CacheReadTokens（读缓存），外加一个 CacheCreation{Ephemeral5mInputTokens, Ephemeral1hInputTokens} 明细。这四路就是全部记账基础。

厂商差异不能忽略：同一段注释还写明，OpenAI 的 prompt_tokens 与 Gemini 的 promptTokenCount 已经把缓存 token 含在总输入里（common.go:158-180），只有 Anthropic 是分离计量的。所以"总输入怎么算"这件事，跨厂商没有统一答案——这是后面成本函数要分 provider 分支的根因。

三、非对称计价：四类不是同一个价

拆成四类只是记账，真正的冲击在单价。看 mino_server 里 ClaudeSonnet（claude-sonnet-4-5）的真实定价（model.go:288-291、298-300，单位 USD / 1M tokens）：

PriceInputTokens:         3,
PriceOutputTokens:        15,
PriceCacheCreationTokens: 3.75, // 5分钟缓存价格
PriceCacheReadTokens:     0.3,
// CacheCreationPricing: Price5Minutes: 3.75, Price1Hour: 6

把倍率读出来，就是三个反直觉的事实：

  • 输出是输入的 5 倍（15 vs 3）。一次长回答的钱可能盖过整段 prompt。
  • 缓存写比普通输入还贵：5m 写是 1.25×，1h 写是 2×。第一次建缓存不是免费，反而要多掏 25%~100% 的溢价。
  • 缓存读不是零成本，仍要付输入的 0.1×（10%）。近乎免费，但不是免费。

四、类比在这里漏水

服务端世界里，缓存是为省你自己的钱：算力是你自家服务器的成本，命中缓存 = 少跑一次昂贵计算，缓存读近乎零成本，缓存写不过是多写一份数据。成本按"请求数"或 QPS 大致就能估。这套直觉在 token 经济学里会漏得很惨。

token 成本由厂商计量、按四类分别计价、且读写非对称。以 ClaudeSonnet 真实单价为证（model.go:288-300）：
（1）缓存写不是免费，反而更贵——首次建缓存要多掏 25%（5m）到 100%（1h）的溢价。指望靠后续多次命中把它摊薄；只被读一次就过期，反而比不加缓存更亏。
（2）缓存读不是零成本——仍要付输入价的 10%（$0.3 / 1M）。
（3）输出通常比输入贵几倍（这里 5 倍）——省 prompt 省不到点子上，控制输出长度才是大头。
所以成本归因必须像 getPrice（model.go:745-752）那样把四类 token 各乘各的单价分开算，绝不能笼统按"请求数"估——这正是服务端类比彻底漏水的地方。

五、成本函数：四路分账在代码里的样子

四路分账最终落在 getPrice（model.go:708-777）。它按 provider family 分支，Anthropic 分支（model.go:745-752）长这样：

case ModelFamilyAnthropic:
	price = info.PriceInputTokens*float64(stats.InputTokens())/1000_000 +
		info.PriceOutputTokens*float64(stats.OutputTokens())/1000_000 +
		info.PriceCacheReadTokens*float64(stats.CacheReadTokens())/1000_000
	if info.CacheCreationPricing != nil && (stats.CacheCreation5mTokens() > 0 || stats.CacheCreation1hTokens() > 0) {
		price += info.CacheCreationPricing.Price5Minutes*float64(stats.CacheCreation5mTokens())/1000_000 +
			info.CacheCreationPricing.Price1Hour*float64(stats.CacheCreation1hTokens())/1000_000

读出来就是那句公式：3·input + 15·output + 0.3·cacheRead + (3.75·5m + 6·1h)。四类 token 各乘各的单价再相加。算出的 cost 连同四类 token、cache_hit_rate 一起进 tea 埋点（stats.go:630-638）和 trace span 的 usageMap（infrastructure/llms/stats.go:89-99），成为成本归因的最终落点。其中 CacheHitRate = cacheReadTokens / RealInputTokens，而 RealInputTokens() 直接返回 inputTokens（即 PromptTokens，已含缓存）（stats.go:206-215）。

⚠ 一处口径存疑（观察，非已证实的 bug）
注意 Anthropic 分支用的是 PriceInputTokens × stats.InputTokens()，而 InputTokens() 返回的 inputTokens = PromptTokens——这个数已经包含了 cache_read 和 cache_creation（见 convert.go:90）。随后该分支又另外加上 0.3 × cacheRead 和缓存写成本。对照 Gemini / default 分支（model.go:758、772）会先做 InputTokens - CacheReadTokens 再乘输入价，Anthropic 分支没有这个扣减。结果是缓存 token 疑似被同时按"输入价 + 缓存价"计，内部成本估算偏高。这是本节埋点（不是 Anthropic 真实账单），把它当作"成本归因很容易算错"的活教材——但请注意这是观察到的疑点，仓库里未找到确认修复的记录。

六、写缓存标记：1h vs 5m，与一个现状 gap

前面都是响应侧记账。缓存标记则在请求侧由业务 Agent 构造消息时打上：稳定不变的 system prompt 用 common.NewLLMMessageWithCache1Hour（common.go:403，挂 NewCacheControl1Hour()，TTL=1h，common.go:617-623），默认的 NewLLMMessageWithCache 则是 5m（common.go:395）。这也解释了为什么 1h 缓存只用在几十处稳定的 system prompt 上，而不用在易变内容上——只有反复命中才能把写入溢价赚回来。

到 Claude 客户端时，convertCacheControl（convert.go:53-77）把内部 TTL 映射成 Anthropic SDK 的 BetaCacheControlEphemeralParam。但 5m 与 1h 两个 case 里真正的 SDK TTL 字段都被注释掉了（convert.go:64/69/74，注释写 TODO: GCP 目前不支持 1h 缓存，后续再放开）。也就是说：NewLLMMessageWithCache1Hour 在内部标了 1h，但当前实际发给厂商的其实是不带显式 TTL 的裸 ephemeral 标记，1h vs 5m 的差别在请求侧当前并不生效。响应侧仍会分别读回 5m/1h 明细并分层计价——这属于"内部意图与当前实际下发之间的 gap"，讲 1h 缓存时务必注明此现状。

七、关键定义

四类 token（four billable token classes）
Anthropic 把一次调用的 usage 拆成 input_tokens（最后缓存断点之后）/ cache_creation（写缓存）/ cache_read（读缓存）/ output_tokens，各自单价不同，各自记账。载体：common.TokenUsage（common.go:184-205）。
非对称计价（asymmetric pricing）
四类 token 单价不成比例：ClaudeSonnet 输出=5×、缓存写=1.25×~2×、缓存读=0.1×（相对普通输入）。成本必须四类分开算，不能按请求数估。载体：model.go:288-300。
cache creation vs cache read
首次把 prompt 写进厂商缓存（比普通输入更贵）与后续命中读取（约为普通输入的 10%）。溢价靠多次命中摊薄，只读一次即亏。载体：CacheCreationTokens / CacheReadTokens。
CacheHitRate
cacheReadTokens / RealInputTokens，其中 RealInputTokens = inputTokens = PromptTokens（已含缓存）。载体：stats.go:206-215。

八、自测题

Q1：ClaudeSonnet 一次调用：uncached input=100、cache 写(5m)=400、cache 读=1000、output=500 tokens。只看"缓存读那 1000 个 token"这一项，按单价表要花多少钱？为什么不是按普通输入价算？

1000 / 1e6 × 0.3 = $0.0003。因为 PriceCacheReadTokens = 0.3（model.go:291），是普通输入价（PriceInputTokens = 3）的 1/10——缓存读单独按 0.1× 计价。这正是四类分账的意义：命中缓存的 token 不能按输入价算，否则会把成本高估 10 倍。（对照：同样 1000 token 若走普通输入是 $0.003。）

Q2：为什么"缓存写"(cache creation) 会比"普通输入"还贵？在 mino_server 里贵多少，设计上指望靠什么把这笔溢价赚回来？

Sonnet 缓存写 5m = 3.75 是输入（3）的 1.25×，写 1h = 6 是 2×（model.go:290/300，CacheCreationPricing）。首次把 prompt 写进厂商缓存要做额外的持久化，所以收溢价；设计上指望后续多次命中——每次命中缓存读只花 0.1×，用足够多次命中把首写的 25%~100% 溢价摊薄。若一段缓存只被读一次就过期，反而比不加缓存更亏——这就是 NewLLMMessageWithCache1Hour（common.go:403）只用在稳定不变的 system prompt 上、而不用在易变内容上的原因。
