压缩三策略：触发线、Nano 门控、flush 落盘

你已经知道 context 会 rot，也知道逼近窗口上限就得回收 token。那就把它当缓存淘汰来做——容量到了，删最冷的 entry。这节课要戳破的正是这个类比：在 conductorservice 里，回收 context 的动作不是删除，是有损总结；没有后端源库可回源。整套代码的复杂度，几乎全在为"删了就没了"这件事兜底。

一、先承认类比对的部分

把 context 逼近上限时的回收，类比成缓存淘汰，起点是对的。容量有限、成本随占用上升、需要一条触发线——这些都成立。你甚至能想到"用一个便宜模型先判断要不要回收"，这就是 cheap-model gating，和"先查布隆过滤器再回源"是同一个直觉。

方向没错。但 LRU 有一个你从不言说的前提：删除是免费的。entry 淘汰了，数据还在后端 DB，要用再按 key 回源。淘汰只是省内存的纯性能操作，不丢信息。接下来的一整节，都在说这个前提在 Agent 世界里塌了。

二、类比在哪里漏水

context 压缩不是删除，是有损总结。一旦 summarize，几十轮工具调用的细节被 Gemini 3 Flash 压成一段 {analysis, summary}，原始 token 就永久没了——没有"后端源库"可回源。LRU evict 之后你还能 GET key，compaction 之后那些 token 不存在于任何地方。

正因为有损，代码在每次压缩前都做了两件 LRU 永远不需要的事：

其一，先把持久记忆抽出来落盘（memory flush）。压缩前调 memoryFlushBeforeCompression，用 LLM 把对话里值得跨会话记住的东西抽成 markdown，append 到 memory/YYYY-MM-DD.md，再索引进向量库。flush 的提示词开头一句话就把理由讲透了：

// memory_flush.go:319
systemPrompt := `Pre-compaction memory flush. The session is near auto-compaction;
capture durable memories to disk before context is truncated.

"在 context 被截断前，把持久记忆落盘"——这是压缩有损性的自证。如果压缩无损，何须抢救？

其二，把原始 context 归档成文件。saveContextToFile 把这次要被压掉的原始消息存成 context_*.json（弱依赖，失败不阻断压缩）。这是给人看的黑匣子，不是给模型回源用的——模型继续跑的是那段有损 summary。

第二处漏水：触发线设在 80% 而不是等填满。LRU 可以撑到 100% 容量才淘汰，因为读取质量和占用率无关。context 不行——远在填满前，读取质量就随占用连续退化。触发线本体： 
// summary.go:86-92
// 取上下文窗口 80% 和上限的较小值，避免超大上下文窗口的模型使用了太大的上下文窗口导致降智
// IM 品类上限 100K，其他品类上限 200K
summaryContextCap := 200_000
if subType == enums.SubType_AXON_IM {
    summaryContextCap = 100_000
}
summaryContextLine := min(int(float64(agent.llmClient.GetModelContextWindow())*0.8), summaryContextCap)

注意注释里那个词——"导致降智"。这就是 context rot 的中文口径：质量在窗口填满之前就已退化，所以不能等满。80% 不是随手取的裕度，它背后压着两个理由：

  1. context rot：注意力预算被 token 数摊薄，"用了太大的上下文窗口"本身就降智。
  2. 压缩自己要吃余量：压缩是一次要把整段对话当输入的 LLM 调用，产出的 summary 还要塞回窗口继续跑。等到 99% 才压，压缩这一步自己就 PromptTooLong 了。

还有那个 cap：它是变量，不是写死的字面量。默认 200K、AXON_IM 品类 100K。为什么封顶？因为有些模型窗口特别大，0.8×window 能撑到几十万 token——那正是注释担心的"太大的上下文窗口导致降智"。min 取两者较小值，等于给"别喂太满"再加一道保险。

三、三条策略：同一个目标，三种时机

三条路径都在守 context 上限，区别只在什么时候触发、由谁决定要不要压。

策略①：Nano 门控预压缩（preCompactSummarize）

这是三条里唯一带"要不要压"语义门的一条，跑在循环之前（agent.go:906-911），且只在 agent.preCompact 开关打开时执行。开关由上游 shouldPreCompact 在 channel/IM 路径设置——条件是缓存里的 TotalTokens ≥ 50_000（preCompactTokenThreshold）且距上次用户消息 > 12h（preCompactIdleThreshold，channel_app_service.go:5455-5484）。注意 token 数取自缓存 cache.GetConductorTotalTokens，不是实时算的——这是给"长时间不活跃的 IM 对话"准备的省钱路径。

进了 preCompactSummarize 也不一定真压。它先用 GPT-5.4-Nano 判断新消息是否延续旧话题：

// summary.go:270-277
// 话题相关性判断：新消息和历史上下文是否在同一话题，相关则跳过压缩
related, err := agent.checkTopicRelevance(ctx)
if err != nil {
    ctx.Log().Warnf("[Conductor] preCompactSummarize: topic relevance check failed, proceeding with compact: %+v", err)
} else if related {
    ctx.Log().Infof("[Conductor] preCompactSummarize: new message is related to previous topic, skipping compact")
    return nil
}

逻辑很干净：用户离开很久回来发了条新消息——如果和之前是同一件事（related=true），历史还有用，return nil 不压；如果是全新话题，历史就是包袱，压掉换 token。注意判定出错时是 fail-open——"proceeding with compact"，宁可多压一次，也不因为一个便宜判断卡住主流程。

checkTopicRelevance（summary.go:236）把历史 user 消息按轮包成 <message round="N">，连同新消息喂给 ModelNameGPT54Nano（gpt-5.4-nano-2026-03-17），MaxTokens: 50，JSON schema 只要一个 {related: bool}。50 个 token 的判断，省下的是一次几十万 token 的主模型调用——这就是 cheap-model gating 的经济学。判定要压时，用 newPreCompactClient（同样是 Nano，MaxTokens: 2000）把 originalMessages 总结成一条 summary 替换掉，latestUserMessage 留给正常处理。

策略②：触发线压缩（summarize）

常态路径。主循环每步开头调 summarize → summarizeWithOptions(force=false)（agent.go:925）。它只对 filesystem subtype 生效（summary.go:75），要过三道闸才真压：

常态压缩用的是 Gemini 3 Flash（gemini-3-flash-preview，newSummaryClient at summary.go:35，MaxTokens: 10000，结构化输出 {analysis, summary}）——不是 Nano。这点要记牢：Nano 只用在策略①的话题门控和预压缩总结上；触发线压缩和 memory flush 用的都是 Gemini 3 Flash。别被"三策略"混成"主压缩用 Nano"。

策略③：紧急压缩（applySummaryImmediately）

兜底路径，异常触发。runConductorStep 报 PromptTooLong 时（agent.go:937-949）：若还没有 lastSummary，就 summarizeForce（force=true，跳过策略②的全部三道闸——都撑爆了还检查什么）；随后 applySummaryImmediately 就地重排消息缓冲（summary.go:406）：originalMessages 清空、latestUserMessage 换成 summary、newMessages 从最近 stepsSinceLastSummary 个 Assistant 轮切开保留，然后 continue 重试这一步。

四、flush 落盘链路：压缩前的抢救如何不阻塞主流程

策略①②在真正压缩前都先调 memoryFlushBeforeCompression；策略③只有在没有可复用的 lastSummary、需要强制新总结时才会走到这一步（复用旧 summary 时它早已 flush 过，不再重复）。这条 flush 链路有两个值得资深工程师停下来看的设计决策。

决策一：用 context.WithoutCancel 派生独立 ctx 异步跑

flush 要调 LLM、要写文件、要建索引，都不快。但它是压缩的前置，如果同步跑，主压缩流程就被它拖住了。更麻烦的是：请求随时可能结束，一旦请求 ctx 被 cancel，正在飞的 LLM 调用会直接失败。代码的解法（memory_flush.go:118-124）是用 context.WithoutCancel(ctx) 派生一个脱离请求生命周期的 detachedCtx，丢进 concurrency.SafeGo 异步跑。请求可以先返回，flush 在后台自己跑完。

这是服务端"请求-响应生命周期"直觉的漏水点：在 Web 服务里，请求结束 = ctx cancel = 所有派生工作停止，天经地义。但抢救记忆这件事不能随请求死——它的价值恰恰在请求之后的下一次会话。所以这里要主动切断 cancel 传播。WithoutCancel 保留 ctx 上的 value（trace、用户信息），只砍掉 cancel 信号。

决策二：三重去重，防止同一记忆被反复落盘

三条压缩策略都会触发 flush，一个 session 里可能压很多次；一次请求里也可能走到多条路径。如果每次都 flush，同样的记忆会被抽出来 append 很多遍。shouldFlushMemory（memory_flush.go:47-75）设了三层闸：

层判据挡什么位置
1hooks == nil 或 sandboxToken 为空不支持 flush 的 subtypememory_flush.go:50-56
2请求内 agent.memoryFlushed 标志同一请求重复 flush（快速路径，不读文件）memory_flush.go:59-61
3session.jsonl 里已有 memory_flush 标记跨请求重复 flushmemory_flush.go:63-72

第 3 层是关键，它把去重状态持久化到了 session 文件。flush 成功后往 yyyy-MM-dd-{pageID}.jsonl 追加一行标记：

// memory_flush.go:131-135
if runMemoryFlush(agentCtx, llmService, vfsService, sandboxToken, agentID, messages, timezone, "compression") {
    // flush 成功后写入标记，防止同一 session 后续请求重复 flush
    if sessionPath != "" {
        appendSessionFlushMarker(detachedCtx, vfsService, sandboxToken, sessionPath)
    }

标记内容是 {"type":"memory_flush","timestamp":...}（memory_flush.go:454-463，常量 sessionFlushMarkerType="memory_flush" at :39）。下次请求进来，第 3 层 hasSessionFlushMarker 读这个文件发现标记就直接跳过。落盘本身走 weaklock（memory_flush.go:370，锁超时 10s、重试 3 次）防并发覆写，append-only 到 /home/user/workspace/{agentID}/memory/YYYY-MM-DD.md。成功后再 SafeGo 一次 IndexMemoryFileToVector（:284-288）把记忆索引进向量库，供后续检索。

一个收尾细节：策略①和策略③结束时都会把 agent.memoryFlushed 重置为 false（summary.go:355 和 summary.go:434），允许下一次压缩再 flush 一次——因为压缩之后又积累了新对话，新一轮的记忆值得再抢救。

本课要带走的判断：当"回收"的对象是有损且不可回源的，回收动作的复杂度不在回收本身，而在它的前置抢救和触发时机。LRU 的代码只有淘汰逻辑；compaction 的代码里，淘汰只是中间一步，两头包着 flush 落盘、context 归档、80% 提前触发、cheap-model 门控、三重去重、异步不阻塞——每一样都是在补"删了就没了"这个漏洞。

五、关键定义

compaction / 压缩（summarize）
历史逼近上下文窗口时，用模型把旧历史总结成摘要以回收 token 的策略。有损、不可回源。载体 summary.go，触发线 min(0.8×window, cap)，cap 默认 200K、AXON_IM 100K。
触发线（summaryContextLine）
min(int(float64(GetModelContextWindow())*0.8), summaryContextCap)。设在 80% 而非满窗，因为 context rot（提前降智）+ 压缩自身要吃输入/输出余量。
廉价模型门控（cheap-model gating）
用便宜小模型先做判断，再决定要不要动用贵主模型。载体 preCompactSummarize 用 GPT-5.4-Nano（MaxTokens: 50）判话题相关性，related=true 就跳过压缩。
memory flush
压缩前把对话中的持久记忆异步落盘到 memory/YYYY-MM-DD.md（append-only + weaklock），并在 session.jsonl 打去重标记，避免有损压缩丢掉该记住的信息。载体 memory_flush.go。
context.WithoutCancel 派生
从请求 ctx 派生一个只砍 cancel 信号、保留 value 的独立 ctx，让异步落盘不随请求结束被 cancel。载体 memory_flush.go:123。

六、自测题

Q1：服务端直觉说"从缓存里淘汰一条 entry 是免费的——数据还在 DB，要用再回源"。为什么 context 压缩在这套代码里不是免费的？代码用哪两个前置动作弥补？

因为压缩是有损总结不是删除：summarize 把几十轮细节用 Gemini 3 Flash 压成一段 {analysis, summary}，原始 token 永久丢失，没有"后端源库"可回源。代码用两个前置动作兜底：

(1) memoryFlushBeforeCompression——压缩前异步（SafeGo + context.WithoutCancel）用 LLM 把持久记忆抽出来 append 到 memory/YYYY-MM-DD.md，再索引进向量库；提示词原话 "capture durable memories to disk before context is truncated"（memory_flush.go:319）。

(2) saveContextToFile——把原始 context 归档成 context_*.json（弱依赖，失败不阻断压缩）。这两步 LRU 淘汰都不需要。

Q2：触发线为什么设在 min(0.8×GetModelContextWindow(), 200_000)（IM 为 100_000）而不是等窗口填满才压？给出代码里能佐证的两个理由。

理由一（context rot）：summary.go:86 注释直说"避免超大上下文窗口的模型使用了太大的上下文窗口导致降智"——模型质量在窗口填满前就随占用连续退化，所以不能等满。

理由二（压缩自身要余量）：压缩是一次要把整段对话当输入的 LLM 调用，产出的 summary 还要塞回窗口继续跑，必须留头寸；等到 99% 才压，压缩这一步自己就 PromptTooLong 了。

补充：cap（200K / IM 100K）额外封顶，防止窗口特别大的模型把 0.8×window 撑到几十万 token 反而降智；cap 是变量不是写死的字面量。真正撑满触发的是 PromptTooLong → applySummaryImmediately 的紧急兜底（策略③），那是异常路径不是常态。

Q3：为什么 memory flush 要用 context.WithoutCancel 派生独立 ctx，而不是直接复用请求 ctx？这戳破了哪个服务端直觉？

因为 flush 要调 LLM + 写文件 + 建索引，都不快，且它是压缩的前置。若同步跑会拖住主压缩；若复用请求 ctx 异步跑，一旦请求结束、ctx 被 cancel，正在飞的 LLM 调用会失败（memory_flush.go:122 注释"避免请求结束后 context canceled 导致 LLM 调用失败"）。

它戳破的是"请求-响应生命周期"直觉：Web 服务里请求结束就该停掉所有派生工作，天经地义。但抢救记忆的价值恰恰在请求之后的下一次会话，所以要主动切断 cancel 传播——WithoutCancel 保留 value（trace/用户信息），只砍 cancel 信号，丢进 concurrency.SafeGo 后台跑完。
