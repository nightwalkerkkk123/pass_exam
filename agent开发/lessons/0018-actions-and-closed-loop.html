从根因到行动项与全链路闭环

你在 0017（根因定位）学过怎么把一条 badcase 稳定追到责任模块——但追到责任模块只是把手指点在了伤口上。这节课往下走一步，也是本系列的最后一步：把"原因"翻译成能被系统消费的行动项，再把行动项接进一条从线上 Bug 到生产反馈、再回流成研发资产的全链路闭环。做完这一步，前面十一节讲的一切才真正开始复利。

一、先承认服务端直觉对的部分——然后看它在哪漏水

在服务端世界，一次质量活动的收尾动作非常干脆：跑完测试、出一份报告（覆盖率、失败清单、性能数字），把报告发到群里或贴到 PR 上，这件事就算交付了。报告是终点，因为下游都是确定性的——失败的断言指向确定的代码行，谁修、怎么修几乎不需要额外翻译。

这个"报告即交付"的直觉，正是 0007「eval ≈ CI」那副地图最后一处、也是最隐蔽的一处漏水。

服务端里"报告发出去就算交付"——因为报告里的每条失败都自带确定的修复入口。Agent 评测里这句话直接失效：一份写着"rejection 类通过率从 82% 掉到 71%"的报告，没有告诉任何人该改哪个 Prompt、加哪条规则、谁来改、改完怎么验。评测的真正交付物不是报告，是回流进研发流程的资产——行动项、回归用例、根因标签、Judge 校准样本。停在报告上的评测，只是给团队增加了一份没人消费的 KPI 报表。这也是 0008 结尾埋的那句"这套 eval 下一步该长出的东西"的答案：下一步长出的，就是这条闭环。

二、把"原因"翻译成"动作"：结构化行动项（§08）

根因定位（0017）给出的是一句判断，比如"责任模块 = 退款 Skill，问题枚举 = 关键工具未调用"。但如果优化建议只写到"优化 Prompt""加强训练""提升准确率"这种泛化表述，它就无法被任何人、任何系统消费——没人知道范围多大、证据在哪、具体动作是什么、归谁、怎么算修好了。

正确做法是把建议产出成结构化行动项：一个能被工单系统、看板、配置平台、研发流水线直接读取的标准任务对象。它至少要钉死六个字段：

字段回答的问题反例（泛化表述）正例（结构化）
失败范围影响哪一类场景、多少样本？"退款有问题""已发货退款场景，v1.8 Prompt 下 23 例跳过订单状态校验"
证据凭什么这么判？"感觉不太对"Trace 显示未调订单查询工具，直接承诺退款
具体动作到底改什么？"优化 Prompt"在退款 Skill 加"先查订单状态"硬前置 + 未调用则拦截
owner谁来改？（空）算法 / 工程 / 运营 / 业务，按根因绑定角色
验收方式怎么算修好？"再看看"该场景连续成功率 ≥ 阈值，且入回归集不再复现
优先级多急？"尽快"P0（资损风险）→ 阻塞发布

记住这句定性：建议是质量工程里的标准任务对象，不是报告里的一段自然语言总结。它和你熟悉的 bug 单、需求卡是同一类东西——可分派、可追踪、可关闭。

三、全链路闭环：终点不是报告，是持续运转（§09）

0017 解决"为什么错"，§08 解决"怎么修"，§09 要解决的是第三个、也最容易被跳过的问题——如何持续运转。一套完整评测系统不能停在报告和建议上，而要把每一条 badcase 变成下面这一串东西，并在每个版本周期里持续跑：

为了让反馈可持续生产，平台可以把它组织成三条生产线（原文），大致对应"快修类 / 知识与规则类 / 模型与数据类"三种反馈的不同处理节奏——轻的当天配置上线，重的进训练周期。三条线共享同一个 badcase 任务表与同一套根因 Pipeline，避免线上长尾和研发主动排查各搞一套。

四、发布门禁 × 线上灰度：至少三类信号联动

闭环里有一个必须联动、不能各看各的关键接缝：离线门禁和线上灰度。离线评测通过，绝不等于线上一定变好。发布阶段至少要同时跟踪三类信号：

信号类别看什么典型指标
离线质量信号发布前的门禁核心场景通过率、P0 风险数、关键工具参数正确率
线上体验信号用户在真实流量里的感受转人工率、重复追问率、投诉率、满意度变化
业务结果信号钱和事有没有办成任务完成率、工单闭环率、退款/赔付成功率
服务端的发布判据是单向的：CI 绿 → 部署 → 完事，绿灯之后没有回头路要监控。Agent 的发布判据必须双向对账：如果离线质量提升、但线上关键信号反而恶化，应当触发回滚或降级。"离线分涨了就发、发完不管"这个 CI 式直觉在这里会害你——离线集永远只是真实分布的一个切片，模型可能恰好过拟合了你的 golden set 而在长尾上退步。门禁和灰度必须联动成一个闭合回路，而不是两个各自为政的关卡。

要让这套联动可信，还有一条工程前提：Trace 必须当作被测 Agent 的标准能力，而不是出事后临时补的日志。Trace 至少要稳定记录工具名、入参、返回值、时间戳、耗时和错误；复杂 Agent 还应记录计划、检索、Guardrail。同时执行环境要隔离——多个用例共享状态会导致分数不可复现，那样整条闭环采集到的都是脏数据。

五、反馈生产：闭环里最容易被低估的一环（§09）

原文把话说得很重：反馈生产是评测闭环里最容易被低估的一环。发现问题本身不会自动带来收益——只有把线上失败沉淀成可复用资产，下一轮迭代成本才会下降。而"无差别地把所有失败塞进回归集"是另一个极端：回归集会无限膨胀、跑批越来越慢、信噪比越来越低。

所以反馈生产要有入库标准，满足条件才入库：

入库门槛为什么
失败可复现，或至少有稳定 Trace + 人工确认一次性环境抖动不该污染回归集
期望行为明确，能写成规则 / Judge 标准 / 人工验收标准写不出判据的样本没法自动回归
根因标签清楚，能归到具体能力域或业务流程无标签样本无法统计、无法分派
样本有代表性，能覆盖一类问题覆盖"一类"而非"一条"，才有回归价值
已脱敏，符合隐私与数据合规线上样本入库的合规底线

入库之后还要持续做样本治理，控制回归集规模与新鲜度：

  • 同一问题簇只保留代表例，避免同类样本堆积；
  • 关键 P0 / P1 风险样本长期保留，不因通过就下线；
  • 已稳定多版本通过且风险低的样本，降级为抽样集，不必每次全量跑；
  • 线上出现新分布时，用主动学习优先抽取低置信、高影响、新意图、新工具、新知识点的样本进入人工确认。

六、终点站：可复用的质量资产库

闭环转够多圈，最终会沉淀出一套可复用的质量资产库——这才是评测平台真正要产出的东西：

资产是什么本系列哪一节在造它
用例库覆盖核心路径与高风险边界的样本0015 数据集建设
Trace 库可复现的执行轨迹记录本课 §四（Trace 作标准能力）
根因标签库稳定、可统计、能指向 owner 的标签体系0017 根因定位
修复建议库结构化行动项的沉淀本课 §二
Judge 校准集用人工标注反向校准裁判的金标样本（Kappa）0008 / 0016
回归集下一版必跑、防止旧病复发的样本本课 §五

一句话点透价值：质量资产越厚，Agent 迭代越不靠个人经验和临时救火。没有资产库，每次出问题都是从零开始查、从零开始救；有了它，团队的排查能力、判据、标准全部被固化下来、可被新人和自动化直接复用。

七、难点与解法（§10）：闭环的成败判据

原文用整整一节，把评测体系的成败收敛成一句极其克制的话：

评测体系的成败判据
取决于它能否在跨团队协作中长期、稳定地产出可执行行动项。必须把低分稳定转成行动项，否则再精细的分数也只是报表。

拆开看，这句话有三个负重的词。跨团队：行动项要分派给算法、工程、运营、业务四类不同 owner，标签和字段必须是他们都认的口径。长期稳定：不是发布前突击一次，而是每个版本周期持续运转、资产持续累积。可执行行动项：回到 §二那六个字段——范围、证据、动作、owner、验收、优先级，缺一格就退化成"报表"。这三个词，恰好也是本课前六节的浓缩。

八、落到你的项目：mino 处在闭环的哪一格

诚实映射（严格区分"mino 真有"与"方法论层、mino 未实现"）：

方法论层、mino 未实现（整节几乎都在这一层，明写，不虚构任何闭环符号）：mino 没有闭环运营子系统——没有离线门禁 × 线上灰度联动，没有把 badcase 翻译成结构化行动项的产出、没有反馈生产线、没有入库标准、没有样本治理、也没有质量资产库（用例库 / Trace 库 / 根因标签库 / 修复建议库 / Judge 校准集 / 回归集）。本课 §二~§七讲的整条链，在 mino 代码里都不存在，别去代码里找它们。

mino 真有的，只有闭环里"离线出分"这一格，外加一个 Trace 雏形：

  • 离线出分（真有）：cmd/memory_eval 有那条"执行用例 → 采集回复 → 评分 → 聚合"的骨架——runner.go 的 runAll/runCase、sendAndCollect 取回复、scorer.go 的 score 算三维明细、report.go 的 AvgScore/PassRate 聚合。这一格结结实实存在，0007/0008 已逐符号核实。
  • Trace 雏形（部分符合本课方向）：memory_eval 已把 ToolCalls 记录采下来，并落盘每条完整 JSON（含 Response、ToolCalls、JudgeReasoning），还有 results/index.html 供人工翻看。这符合本节"把 Trace 当标准能力、留下可复现证据"的方向——但它没有接进任何线上信号联动，也没有回流进回归入库流程。

诚实结论：mino 只点亮了闭环里"离线评测出分"这一格，后面"根因 → 行动项 → 反馈生产 → 回归入库"整条链尚未长出。这不是缺陷指控，而是一个可核实的坐标：它是一个能出分的评测程序，还不是一套能持续运转的评测体系——而本系列从 0013 到 0018，讲的正是从前者长成后者缺的每一块。

九、关键定义

结构化行动项（structured action item）
把根因翻译成的、能被工单/看板/配置平台/研发流水线直接消费的标准任务对象，至少含失败范围、证据、具体动作、owner、验收方式、优先级六个字段。是质量工程里的标准任务对象，不是报告里的自然语言总结。
全链路闭环（closed loop）
把 badcase 持续转化为 bug/工单/配置变更/知识库修订/训练数据/回归用例，并在每个版本周期持续运行的流程。评测平台的终点不是报告，而是这条闭环产出的可复用研发资产。
门禁 × 灰度联动
发布阶段把离线门禁与线上灰度绑成一个回路，至少跟踪离线质量 / 线上体验 / 业务结果三类信号；离线提升但线上关键信号恶化时触发回滚或降级。
入库标准 / 样本治理
反馈进回归集的准入条件（可复现、期望明确、根因清楚、有代表性、已脱敏）与规模控制策略（同簇留代表例、P0/P1 长期保留、稳定低风险降抽样、新分布主动学习优先）。
质量资产库
闭环最终沉淀的可复用资产：用例库、Trace 库、根因标签库、修复建议库、Judge 校准集、回归集。资产越厚，迭代越不靠个人经验和临时救火。

十、自测题

Q1：根因定位得出结论"责任模块 = 退款 Skill，问题枚举 = 关键工具未调用"，工程师照着写了一条建议："优化退款 Skill 的 Prompt。"按本课标准，这条建议为什么还不能算行动项？把它补成合格的结构化行动项，至少要补哪几格？

因为它停在了"泛化表述"，无法被任何系统或人消费。"优化 Prompt"没说清失败范围多大、证据是什么、具体改什么、归谁、怎么算修好、多急——工单系统读不了，owner 也认领不了。它是报告里的一句话，不是"质量工程里的标准任务对象"。

补成结构化行动项至少要钉死六格：失败范围（如"已发货退款场景、v1.8 Prompt 下 23 例"）、证据（Trace 显示未调订单查询工具就承诺退款）、具体动作（在退款 Skill 加"先查订单状态"硬前置 + 未调用则拦截）、owner（算法或工程）、验收方式（该场景连续成功率达标且入回归集不再复现）、优先级（涉资损 → P0，阻塞发布）。补齐这六格，它才能被工单/看板/流水线直接消费。

Q2：一次发布，离线评测核心场景通过率从 88% 涨到 93%，团队据此判定"质量提升，可以全量发布"。灰度期间却发现转人工率和投诉率同时上升。按本课的"门禁 × 灰度联动"，正确处置是什么？这暴露了哪个服务端直觉的失效？

正确处置：触发回滚或降级，而不是全量发布。本课的判据是三类信号必须联动对账——离线质量信号涨了，但线上体验信号（转人工率、投诉率）恶化，说明"离线提升但线上关键信号恶化"，这正是原文明确规定要回滚/降级的情形。

失效的服务端直觉是"CI 绿了就发、发完不用管"。服务端发布是单向的，绿灯之后没有回头路要监控；但离线集永远只是真实分布的一个切片，模型可能过拟合了 golden set 而在长尾退步。所以 Agent 的发布必须双向：门禁看离线，灰度看线上，两者绑成闭合回路。只看离线分做发布判断，等于用一个切片替真实流量下结论。

Q3（延伸）：有人说"我们每次评测都产出一份很详细的报告发到群里，还配了根因分析，这已经是完整的评测体系了。"用本课 §一的漏水点和 §十的成败判据反驳他。

他混淆了"报告"和"资产"。§一的漏水点说得很直白：评测的交付物不是报告，是回流进研发的资产。一份再详细、再带根因的报告，如果没有被翻译成带 owner 和验收方式的结构化行动项、没有沉淀成回归用例和根因标签库，那它就只是一份"没人消费的 KPI 报表"——下一个版本出同样的问题，还得从零再查一遍，成本一分没降。

§十的成败判据把这一点钉死：评测体系的成败取决于能否在跨团队协作中长期、稳定地产出可执行行动项。"发报告 + 附根因"顶多完成了"出分"和"定位"两格，缺了"行动项 → 反馈生产 → 回归入库"这条让改进复利、让能力沉淀成资产的闭环。所以它是一个能出分、能定位的评测程序，还不是一套能持续运转、跨团队产出行动项的评测体系——这也正是 mino 现在所处的坐标。

十一、系列收束

到这里，评测这条支线走完了。回看这一路：0007 把 eval 摆成新的 CI，0008 学会不轻信那把叫 llmJudge 的尺子，0013 把"跑几条 case"抬升成分层、多层评判的体系雏形，0014 补上统计门禁，0015 把评测集当质量资产来建，0016 给评分分层并路由到人工，0017 把 badcase 追到责任模块，0018 把责任翻译成行动项、再接进全链路闭环。0013 给评测体系下的定义——"把不稳定行为持续收敛成可发布质量的闭环"——到这节才算被填满。

但评测从来不是孤岛。它是 Agent 四支柱里的一根，和上下文工程、工具与 Skill、记忆系统紧紧咬合：你评的是它们的产物，你的行动项又反过来改造它们。所以本系列的最后一步，是把这根"评测支柱"接回四支柱的全景里去看。
