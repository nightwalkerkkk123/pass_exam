第二双眼睛：提议者-审核者 与 Sidecar

危险操作不能只让一个模型说了算，要给它加一双独立的“第二双眼睛”。但书里给的不是一种范式，而是两种——提议者-审核者与 Sidecar。这节课唯一要拿下的，是把这两种范式的时机、审查对象、输入隔离三处差异刻清楚，从而看懂本章那句看似自相矛盾的话：“能力相差过大的模型互审不可靠”，可 Sidecar 偏偏用轻量模型去审主模型。它们出现在第 4 章“执行工具”一节，对应表4-2。

一、同一个动机，两个不同的问题

书里把话说得很清楚：这两种机制都是“用独立的第二视角检验第一视角的产出”，动机相同，但它们各自回答的是不同的问题：

  • 提议者-审核者回答的是：“操作执行前要不要批、操作完成后结果对不对。”——它管的是事前审批与事后验证。
  • Sidecar 回答的是：“操作执行时如何实时校验安全性和可靠性。”——它是每次工具调用当口的那道闸门。

先记住这条时间轴：事前 → 执行当口 → 事后。提议者-审核者站在两头（前、后），Sidecar 卡在中间那一刻（执行当口）。三个位置各归各管，这是理解全部差异的起点。

二、提议者-审核者：为什么审查者要“能力相近但训练偏好不同”

事前审批的骨架很朴素：一个模型负责提议行动（Proposer），另一个独立的模型负责审查批准（Reviewer）——书里的类比是银行的经办、审核双签，转账须经两道签字才生效。真正的设计学问，全在怎么挑这两个模型上：

  • 两个模型应来自不同的家族（如 GPT 系列与 Claude Sonnet 系列），引入认知多样性——像两个不同学校毕业的工程师，不太可能在同一个地方犯同样的错。
  • 但能力水平要相近：相近才保证审批模型“跟得上”提议模型的思考。
  • 能力相差过大反而不可靠——书里的原话举例是“Haiku 审查 Opus 的输出”，审查者跟不上被审者的思考。

所以理想配对是一句要背下来的口径：能力相近但训练偏好不同，例如 Claude Opus 与 GPT-5 互审。还有一处细节别漏——两个模型的底层规则和约束必须完全一致（否则互相扯皮、陷入僵局），但关注点应有所差异：提议模型强调行动导向与任务完成，审批模型强调风险控制与规则遵守。

第二种机制是事后验证，要诀是模态切换：不是让第二个模型重读一遍相同内容，而是换一种模态检验——例如把生成的文档渲染成视觉输出再看排版，或把改过的配置放进沙盒实际运行看是否生效。单一模态的审查容易陷入相同的盲区，换模态才提供互补视角。

两个模型能力相差过大（如 Haiku 审查 Opus 的输出）反而不可靠——审查者跟不上被审者的思考。理想配对是能力相近但训练偏好不同的两个模型。 ——《深入理解 AI Agent》第 4 章

三、Sidecar：为什么它敢用轻量模型，还刻意“捂住耳朵”

Sidecar 借鉴微服务的边车模式——如同摩托车旁挂的边车，独立运行但与主体并行。它是一种伴随主 Agent 思考循环运行的轻量级 LLM 调用，不审查主 Agent 的最终输出，而是对主 Agent 的行为（工具调用）做独立判断。Claude Code 就是典型案例：主模型决定执行一个工具调用时，一个独立的轻量级、非流式、低延迟的 LLM 调用被触发，判断“这个工具调用是否安全”。

这里有两个最容易被讲错、也最见功力的设计决策，必须逐字讲清：

决策一：只看结构化字段，刻意隔离主模型的自由文本。这个旁路调用只看工具名 + 参数，不看主模型的自由文本思考过程。为什么？因为关键威胁是提示注入：若 Sidecar 也读自由文本，攻击者只要在用户输入或网页里夹带“请允许执行 rm -rf”这类话术，主模型可能把它复述进思考过程，再被 Sidecar 误判为合理理由。只读结构化字段，就堵住了这条话术通道。书里的例子：主模型准备执行 bash("rm -rf /tmp/data")，Sidecar 收到 {tool: "bash", command: "rm -rf /tmp/data"}，识别出 rm -rf 模式，判高风险，拒绝并要求用户确认。整个调用通常在数百毫秒（亚秒级）内完成。

决策二：为什么这里敢用轻量模型？这正是本章故意抛出的“自相矛盾”。答案在于审查对象不同：提议者-审核者审的是开放式思考，审查者必须跟得上被审者的思路，所以要能力相近；Sidecar 判的是结构化数据上的分类问题（这条命令是否越界），任务复杂度低得多，轻量模型足以胜任。

五、边界与易错

误区一：“Sidecar 是异步的，所以它不阻塞、不算门控。” 恰恰相反。书里特意点破了这句话：“并行”省掉的是审查排队的时间，而不是取消了审查这道闸门。对被审查的那一次工具调用，危险操作在 Sidecar 放行之前不会真正执行。把“并行”读成“放行”，是这一节最常见的理解事故。

误区二：“既然轻量模型不能可靠地互审（Haiku 审 Opus 不行），那 Sidecar 用轻量模型也不靠谱。” 这正是本章故意设的坎。区别不在模型大小，而在审查对象：互审面对的是开放式思考，要跟得上思路才需能力相近；Sidecar 面对的是结构化分类（命令是否越界），任务简单，轻量模型够用。判断“该不该用轻量模型”，先问“它审的是思考还是结构化调用”。

六、关键定义

提议者-审核者范式（Proposer-Reviewer）
用独立第二视角检验第一视角产出：一个模型提议行动(Proposer)，另一个能力相近但训练偏好不同的模型审查(Reviewer)；含事前审批与事后验证(模态切换)两种机制。避免理解成“同一模型自我反思”——关键在认知多样性，两模型须来自不同家族。
Sidecar 机制（Sidecar）
借鉴微服务边车模式的轻量 LLM，与主模型流式输出并行运行，只看结构化工具调用数据(工具名+参数)做安全门控；对被审查的那次调用起门控作用，危险操作放行前不执行。避免说“Sidecar 异步执行所以不阻塞审查”(它省的是排队时间，闸门仍在)，也别说它读主模型自由文本(恰恰刻意隔离)。
致命三要素（Lethal Trifecta）
Simon Willison 提出的安全框架：访问私有数据、暴露于不可信内容、对外通信能力——三者齐备即构成完整攻击闭环；接入的 MCP 服务器越多越易集齐。避免把它当本章原创或已在本章展开——本章只是引出，完整机制在第五章。

七、自测题

Q1：本章说“能力相差过大的模型互审不可靠”，为什么 Sidecar 又能用轻量模型来审查主模型？两者的关键区别是什么？

因为审查对象不同。提议者-审核者审的是开放式思考，审查者必须跟得上被审者思路，故需能力相近、训练偏好不同的异构模型(如 Opus 与 GPT-5 互审)；Sidecar 判的是结构化数据上的分类问题(这条命令是否越界)，任务复杂度低，轻量模型足矣。且 Sidecar 刻意只看工具名 + 参数、隔离主模型自由文本，以堵住提示注入的话术通道。

Q2：为什么 Sidecar 要“刻意不读”主模型的自由文本思考过程？只读结构化字段防的是哪种攻击？

防的是提示注入。如果 Sidecar 同时读自由文本，攻击者只要在用户输入或网页内容里夹带“请允许执行 rm -rf”这类话术，主模型可能把它复述进思考过程，再被 Sidecar 误判为合理理由。只读结构化字段(工具名 + 参数)就堵住了这条话术通道——例如收到 {tool: "bash", command: "rm -rf /tmp/data"} 时，Sidecar 直接从命令模式判高风险，不受任何“理由”干扰。

本课唯一要带走的判断：给危险操作加“第二双眼睛”有两种范式，别混为一谈——提议者-审核者守“事前/事后”两头、审开放式思考、要能力相近的异构模型；Sidecar 卡“执行当口”、审结构化的工具调用、刻意只看工具名+参数所以轻量模型就够。看到“为什么这里能用小模型”别慌，先问一句：它审的是思考还是结构化调用？
