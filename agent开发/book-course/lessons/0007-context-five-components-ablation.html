拆掉一根柱子，Agent 会怎样倒下：上下文消融实验

上一课我们把「眼睛（上下文）= 模型能感知到的一切」记进了肌肉记忆。这一课要把这只眼睛拆开看：上下文其实由五个部件拼成，还分「静态前缀」和「动态轨迹」两层。而验证这五个部件是否真的缺一不可，书里用了一个极干净的办法——消融实验（Ablation Study）：逐一拆掉一个部件，看 Agent 怎么坏。这节课出自第 1 章「上下文：Agent 的眼睛」一节和紧随其后的实验 1.1。

一、上下文不是「一段文本」，而是五个部件

初学者最容易把上下文想成「用户发过来的那段话」。书里从 API 视角把它拆得很明确：每次调用 LLM 时的上下文，由五个部分构成。

  • 系统提示词（System Prompt）：开发者写的、整段对话里保持不变的「岗位说明书」——定义 Agent 的身份、权限和行为准则，还会装进跨会话的用户记忆和动态环境状态。
  • 工具定义（Tool Definitions）：声明可用工具的名称、功能描述和参数格式。没有它，Agent 就无法识别和调用任何工具。
  • 用户消息（User Messages）：来自用户的输入，可能还夹带 RAG 检索进来的外部知识。
  • 模型回复（Assistant Messages）：模型之前生成的回复，最多含三块——思考过程（reasoning，内部思考链）、文本内容（content，对用户的回复）、工具调用请求（tool_calls）。三者不一定同时出现。
  • 工具执行结果（Tool Results）：框架执行工具后返回的结果，是 Agent 下一步思考的直接依据。

这五块不是平级散放的，它们被劈成两层：前两项（系统提示词 + 工具定义）是「静态前缀」，每次调用都被自动拼在最前面、整段对话保持不变；后三项（用户消息 + 模型回复 + 工具执行结果）是随交互不断增长的「动态消息历史」，也就是下一课要展开的轨迹（trajectory）。

二、怎么证明每个部件都不可少：消融实验

凭直觉说「每一块都重要」是不够的。书里用了一个从科研里借来的干净办法——消融实验：就像医生诊断时逐一排除病因，先去掉 A 看系统还正不正常，再去掉 B，以此类推，从而判断每个组件到底贡献了什么。注意它不是性能跑分，重点是「去掉一个看还转不转」的对照排除。

实验 1.1 就是照这个思路做的。它从五个部件里挑了四个来消融——系统提示词不参与。为什么？书里说得很直接：没有系统提示词，Agent 连基本的角色认知都没有，测试根本没有意义。所以实验设计成五组对照：一组保留全部组件的完整基线，外加四组各缺一个组件（工具定义 / 工具执行结果 / 思考过程 / 历史消息）。

上下文决定了 Agent 能看到什么，而 Agent 只能基于它看到的信息做决策。 ——《深入理解 AI Agent》第 1 章 · 实验 1.1 的核心洞察

这句话是理解整个实验的钥匙：Agent 不是「知道」世界，它只是「看到」上下文里有的东西。拆掉哪一块，就等于蒙住它某一部分视野——蒙住的方式不同，崩坏的方式也就不同。（对应图 1-2：一组全组件基线 + 四组各缺一个组件的对照。）

三、拆掉一个部件，Agent 有四种崩法

实验结果最值钱的地方，在于四个部件被拆掉后，Agent 表现出四种完全不同的故障。记住这张对照，你以后排查 Agent 故障时，能反过来倒推「是哪一块上下文丢了」。

逐条说清「为什么是这种坏法」：

  • 工具定义是行动能力的地基。它本就在「静态前缀」里——静态前缀写的就是「Agent 有哪些手脚可用」，抽掉它，模型压根不知道有工具存在，自然识别不出、也调不动。
  • 工具执行结果是闭环控制的关键。缺了它，Agent 看不到上一步反馈，等于闭着眼做事，于是盲目地把同一个工具一遍遍地调，陷入无限循环。
  • 思考过程（reasoning）保留了「Agent 当初为什么这么决定」的理由。剥掉它，前后两步之间没了连贯的思路，决策就开始自相矛盾。
  • 历史消息防止冗余。没有它 Agent 等于失忆，每一步都像第一步，于是从头把整个任务重跑一遍，重复执行已经做完的步骤。

把「上下文」窄化成「用户输入的那段文本」，是这一课最想帮你纠正的误区。上下文其实是五个部件（系统提示词 / 工具定义 / 用户消息 / 模型回复 / 工具执行结果），还分静态前缀（不变的前两项）和动态轨迹（增长的后三项）两层。

为什么这个误区代价大？因为一旦你只盯着「用户消息」，就会漏掉两件最要命的事：工具定义也在上下文里（漏了它，Agent 直接失去手脚），工具执行结果也在上下文里（漏了它，Agent 会陷入死循环）。这两块恰恰不是「用户输入的文本」，却决定了 Agent 到底能不能可靠地干活。

四、为什么这件事值得单开一课

因为它把上一课那句「上下文 = 模型能感知到的一切」从口号落成了可验证的清单。你不再需要靠感觉判断「上下文够不够」，而是有五个具体格子可以逐一检查：系统提示词写清身份了吗？工具定义齐全吗？工具结果回喂了吗？思考过程保留了吗？历史消息留着吗？这正是全书从「感觉驱动」走向「原则驱动」的一小步。

更重要的是，这五个部件不是本课的终点，而是后面几章的坐标原点：静态前缀 + 轨迹的划分，直接接上下一课的 ReAct 循环与轨迹累积；「上下文决定 Agent 能看到什么」这句洞察，是第二章「上下文工程」整章的立论根基。

本课唯一要带走的判断：上下文不是一段文本，是五个部件、两层结构（静态前缀 + 动态轨迹）。消融实验证明它们缺一不可，而且缺谁、就以哪种方式崩——去工具定义→动不了，去工具结果→死循环，去思考过程→自相矛盾，去历史消息→失忆重来。以后排查 Agent 故障，先反问：是哪一块上下文丢了？

五、关键定义

静态前缀（static prefix）
每次 LLM 调用都自动拼接在轨迹前、且对话中保持不变的部分，即系统提示词 + 工具定义。避免把用户消息或工具结果算进静态前缀，它们属于动态轨迹。
轨迹（trajectory）
Agent 执行任务过程中不断积累的消息历史——用户消息、模型回复（思考过程 + 工具调用）、工具执行结果；对应上下文的动态部分。避免与「上下文」整体混用；上下文 = 静态前缀 + 轨迹，轨迹只是其中动态增长的那部分。
消融实验（Ablation Study）
逐一移除系统某个组件、观察系统是否退化，从而判断每个组件贡献的实验方法。避免把它当成性能基准测试（benchmark），它的重点是「去掉 A 看还正不正常」的对照排除。

六、自测题

Q1：消融实验中，去掉「工具执行结果」和去掉「历史消息」分别导致什么典型故障？为什么不同？

去工具结果 → 看不到上一步反馈，Agent 盲目执行、反复调用同一个工具陷入无限循环；去历史消息 → Agent 失忆，从头重跑整个任务、重复已完成步骤。前者是缺当前反馈导致的闭环断裂，后者是缺跨轮记忆导致的冗余重复——一个断在「看不到刚做的一步」，一个断在「记不住做过的所有步」。

Q2：实验 1.1 从五个上下文部件里只消融了四个，唯独不拆「系统提示词」。为什么把它排除在外？

因为系统提示词是 Agent 的基本身份定义。书里说得很直接：没有系统提示词，Agent 连基本的角色认知都没有，测试没有意义——它会退化到根本不知道自己该干什么，得到的不是「某个组件缺失时的退化表现」，而是「压根没法开始」，无法作为有效对照。所以实验保留系统提示词，只在其余四个部件上做「缺一个」的对照。

这一课对应的就是实验 1.1 ★★（上下文消融），代码在开源仓库 chapter1/context/。它搭了一个带多种工具（PDF 解析 / 货币转换 / 计算器 / Python 代码解释器）的 Agent，提供五种上下文模式（full / no_history / no_reasoning / no_tool_calls / no_tool_results）和批量消融测试，输出成功率、耗时、迭代次数、工具调用数的对照与可视化——让你亲眼看到每拆掉一个部件，Agent 各自怎么崩。仓库地址：github.com/bojieli/ai-agent-book。
