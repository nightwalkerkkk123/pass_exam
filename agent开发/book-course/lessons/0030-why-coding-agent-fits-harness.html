任务四象限：为什么 Coding Agent 是最成熟的 Agent 类型

这节课只押一张表——第 5 章「Harness 工程在 Coding Agent 中的实践」一节里的表5-1。它用任务清晰度 × 验证自动化程度两个维度，把所有 Agent 任务切成四格。你要拿下的唯一判断是：Harness 的全部目标，就是把尽可能多的任务推向「目标明确 + 验证自动化」那一格；而代码之所以是当前最成熟的 Agent 领域，正是因为它天生就坐在这一格的核心——不是模型特别强，而是软件工程几十年积累的基础设施，恰好构成了一套现成的强 Harness。

一、两个维度，切出四种任务状态

作者没有按「行业」或「难度」给任务分类，而是挑了两个更本质的维度——它们直接决定 Agent 干不干得动：

  • 任务清晰度：目标是明确的，还是模糊的？（Agent 知不知道该往哪走）
  • 验证自动化程度：结果能自动验证，还是需人工验证？（Agent 能不能自己判断对错）

二、为什么这两个维度决定一切（深挖「为什么」）

为什么偏偏是「清晰度」和「验证自动化」这两个维度，而不是别的？把四格逐个读一遍，你会发现它们各自卡在了 Agent 工作循环的不同关节上：

  • 目标明确 + 可自动验证（最佳区域）：目标清楚，Agent 知道往哪走；结果能自动判对错，Agent 能自我校验、自己闭环。这正是「修复有测试用例的 bug」——写完跑测试，绿了就是对。
  • 目标明确 + 需人工验证（吞吐量受限）：方向不缺，缺的是自动的对错信号。原文一句话点破——「吞吐量的天花板就是人的审查速度」。代码重构就是典型：Agent 能大批量产出，但每一版都得人来盯。
  • 目标模糊 + 可自动验证（高效地跑偏）：这一格最阴险。有自动反馈，Agent 跑得飞快，可目标是模糊的——于是它会「高效地往错误方向跑」。拿 linter 分数去优化「代码质量」就是这样：分数一路涨，代码未必更好。
  • 目标模糊 + 需人工验证（难以启动）：两样都缺。既没有明确的成功标准，也没有自动的对错判断，Agent 无从知道该往哪走、也无法自我校验，基本派不上用场。「让 UI 更好看」就落在这里。

所以 Harness 工程的着力点非常清晰：要么把模糊的目标钉成明确的（往左搬），要么把人工验证换成自动验证（往上搬）。原文原话——「Harness 的目标就是把尽可能多的任务推向『目标明确 + 验证自动化』这个象限」。

三、为什么代码天然坐在核心象限

这才是全书这一节最反直觉、也最重要的一句话。Coding Agent 是当前所有 Agent 类型里成熟度最高的——但作者特意强调：「不是因为代码生成模型特别强，而是因为软件工程几十年积累的基础设施天然构成了一套强大的 Harness」。

代码编写是所有 Agent 任务中可验证性最高的一类，约束、验证和纠正都有现成的基础设施可以依托。 ——《深入理解 AI Agent》第 5 章「Harness 工程在 Coding Agent 中的实践」

四、最容易读错的一格

初学者最容易把「有自动化验证」直接等同于「适合交给 Agent」——于是掉进左下角那格：「高效地跑偏」。典型场景是拿一个能自动打分、但并不真正代表目标的指标去驱动 Agent，比如用 linter 规则数、代码行数、某个覆盖率数字去优化「代码质量」。反馈很快、Agent 很勤快，指标一路上涨——可目标本身是模糊的，于是它只是又快又稳地朝着错误的方向狂奔。

教训：自动化验证只有在目标明确时才是加分项。目标还没钉死就先接上一个自动指标，等于给跑偏装了个加速器。四象限里「往上搬（接自动验证）」和「往左搬（钉死目标）」是两件独立的功课，顺序反了会更糟。

本课唯一要带走的判断：任何 Agent 任务，先用两个维度定位它落在哪一格——目标明确吗？结果能自动验证吗？ Harness 工程的全部意义，就是把任务往「目标明确 + 验证自动化」的左上角搬。Coding Agent 最成熟，不是模型的功劳，而是测试、Linter、Git 这套软件工程老基建，天生就把代码任务钉死在了那一格。下次评估「这件事该不该交给 Agent」，先把它丢进这四格里。

五、关键定义

任务四象限（清晰度 × 验证自动化）
用「任务清晰度（目标明确／模糊）」和「验证自动化程度（结果可自动验证／需人工验证）」两个维度把任务切成四格：最佳区域、吞吐量受限、高效地跑偏、难以启动。对应表5-1。避免只看「有没有自动验证」就下结论——目标模糊时自动验证反而助长跑偏。
Harness（Agent = Model + Harness）
围绕模型搭建的基础设施，包含核心公式里的上下文和工具，外加约束、验证与纠正机制。在 Coding Agent 场景落地为验收基线、执行边界、反馈信号、回退手段四类组件。它的目标就是把任务推向四象限的左上角。避免把「能不能稳定跑」全归因于模型强弱——很多时候取决于 Harness 扎不扎实。
元能力（meta-capability）
一种「能创造其他能力」的能力：Agent 用它当场写出新工具、新约束、新表达形式来完成任务，而不必事先把所有能力都预制好。代码生成正是这样的元能力。避免把它等同于「一个很强的能力」或「万能工具」；关键在「创造能力的能力」而非能力本身强弱。

六、自测题

Q1：「让 UI 更好看」这个任务落在四象限的哪一格？为什么它对 Agent 最难？

落在「目标模糊 + 需人工验证」那一格——原文命名为「难以启动」。因为它既没有明确的成功标准，也没有自动化的对错判断：Agent 无从知道该往哪走，也无法自我校验，所以基本派不上用场。这一格与「最佳区域」正好在对角线两端。

Q2：作者说 Coding Agent 是当前成熟度最高的 Agent 类型。真正的原因是「代码生成模型特别强」吗？

不是。原文明确否定了这个解释：成熟度高「不是因为代码生成模型特别强，而是因为软件工程几十年积累的基础设施天然构成了一套强大的 Harness」——测试套件提供明确的验收标准（目标明确），Linter 和类型检查器提供即时的自动化验证（验证自动化），Git 提供完美的版本控制和回退。这三样正好把代码任务钉在四象限的核心「最佳区域」里。赢在 Harness，不在模型。

把这套 Harness 组件看成活的，最好的办法是读一个真跑起来的 Coding Agent。配套仓库里的 chapter5/coding-agent/ 是一个纯 Python 实现的生产级 Coding Agent，把 tools.json 里全部 17 个工具落地（Read/Write/Edit、纯 Python 的 Grep、Glob、持久 Bash 会话、TodoWrite 等），并集中演示了本课提到的反馈信号与纠正机制——工具调用计数（3 次以上告警防死循环）、详细错误信息、Write/Edit 后自动 lint 检测。边读边对照四象限：看它是如何用这些基建把任务往「最佳区域」按的。仓库入口：github.com/bojieli/ai-agent-book。
