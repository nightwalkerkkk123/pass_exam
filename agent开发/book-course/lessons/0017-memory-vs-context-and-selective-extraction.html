记忆不是流水账：一次会话结束后，Agent 到底该记住什么

这节课只押一件事：用户记忆不是把对话逐字存下来，而是主动投入一次额外的算力，把冗长的对话压缩成一个关于用户的“预测模型”。它落在第三章开篇——上一章刚讲完单次会话的上下文管理，这一章要跨过“会话结束”这道坎。你要在本课把提取的三个特征——选择性、抽象化、结构化——刻进直觉，并且分清它和“把东西塞进上下文”这种临时学习的根本区别。

一、是什么：记忆是一次“主动的学习”，不是录音

作者一上来就给了一个一秒就懂的类比。我们和朋友相处，从不记住每次对话的原始内容——我们是在持续交往中，在脑海里逐渐形成一个关于对方的生动模型：他的爱好、习惯、价值观。这个模型让我们能够理解、甚至预测他的需求。

Agent 的用户记忆应当照此办理。原文把它的本质说得很硬：

用户记忆系统的本质是一个主动的、持续的学习过程，其目标是构建一个关于用户的简洁而有效的预测模型。它投入额外的算力（通过专门的 LLM 调用来分析、总结和结构化信息），将分散在冗长对话历史中的关键信息进行显式提取和压缩。 ——《深入理解 AI Agent》第三章 · 用户记忆系统

拆开这句话，有两个容易滑过去的词：“主动”和“额外的算力”。记忆不是对话的副产品，不是顺手留下的日志——它需要一次专门的、独立的 LLM 调用，在一段对话结束后，回过头去分析、总结、结构化。你是特地花了一份算力，去换一个更小、更好用的用户模型。

二、怎么做：一段对话如何被榨成几条记忆

作者给了一个订机票的例子。用户说：帮我订下周五去东京的航班，我喜欢靠窗座位，我吃素、需要特殊餐食；随后又补了一句：用我的 United MileagePlus 会员号 12345678。中间 Agent 调了搜索工具，返回了 3 个选项。

三、为什么这样设计：记忆 vs 上下文学习，差在“持久”与“临时”

这里是最容易糊涂、也最值得深挖的地方。你可能会想：把用户偏好直接塞进 prompt / 上下文，不也能让模型“记住”吗？——能，但那是另一回事。原文用一句话划清了界：

这与上下文学习形成对比——用户记忆是持久的、可审查的，上下文学习则是临时的、会话结束就消失。 ——《深入理解 AI Agent》第三章

两个维度上的对立：

  • 持久 vs 临时：上下文学习活在这一次会话里，窗口一关就烟消云散；用户记忆被显式写入持久存储，跨会话、跨实例都在。
  • 可审查 vs 不可见：用户记忆是一条条被提炼、被打了类型的显式条目，你能看、能改、能删；上下文学习藏在模型对当前 token 的隐式处理里，无从审查。

所以“投入额外算力做一次专门提取”不是浪费——正因为要跨越“会话结束”这道坎、要让记忆可被人和系统检查与维护，才值得把对话主动压缩成一份稳定、显式的档案，而不是指望模型把整段历史一直背在上下文里。

四、边界与易错：别把“轨迹”当成“记忆”

初学者最容易犯的错，是把轨迹（Trajectory）和用户长期记忆混为一谈，觉得“把这次会话的完整记录存下来，就是记忆了”。作者专门把这两者对立起来讲：

轨迹是单次会话的完整原始记录，按时间顺序追加、只增不改（append-only）——它是流水账。用户长期记忆则是跨会话提炼出的稳定信息，会被反复改写、合并、淘汰——它是档案。

把流水账当档案，你会遇到两个必然的麻烦：一是越攒越长、根本喂不进上下文；二是里面全是「返回 3 个选项」这类当场就作废的临时信息，检索时净是噪声。记忆的价值，恰恰在于它不是流水账——它是被选择、被抽象、被结构化之后剩下的那一小撮。

本课唯一要带走的判断：用户记忆 = 花一次额外的 LLM 调用，把一段对话选择性地留下有用事实、抽象成通用结论、结构化地打上类型，得到一份持久、可审查的用户档案——而不是把对话流水账原样存下、也不是靠上下文临时“记一下”。判断一条信息该不该进记忆，就问一句：它对未来还有用吗？没用的（如「返回 3 个选项」），扔。

五、关键定义

用户记忆（User Memory）
针对单个用户 ID 的个性化持久记忆，跨会话提炼出的稳定信息，会被反复改写、合并、淘汰；本质是投入额外算力构建关于用户的简洁预测模型。避免等同于“对话历史”或“轨迹”——轨迹是单次会话按时间追加不可改的流水账，用户记忆是跨会话提炼的档案。
轨迹（Trajectory）
一次 Agent 运行的完整历史记录（用户消息 + 模型回复 + 工具执行结果），按时间顺序 append-only 追加、只增不改，为当前决策提供即时上下文。避免与“工作记忆”混用——工作记忆是经筛选激活的动态子集（按相关性裁剪），轨迹是不可变的完整事件序列。

六、自测题

Q1：订机票对话里出现了“搜索返回了 3 个选项”和“用户偏好靠窗座位”两条信息。记忆提取应保留哪一条？为什么？体现了哪个提取特征？

保留“用户偏好靠窗座位”。“3 个选项”是临时信息、对未来无用（选择性原则丢弃它）；“偏好靠窗”被从这次具体航班抽象为通用偏好（抽象化），并标记为“偏好”类型（结构化），下次订票无需再问。

Q2：既然把用户偏好塞进上下文也能让模型“记住”，为什么本书还要专门花一次 LLM 调用去做记忆提取？记忆和上下文学习差在哪两点？

差在持久与可审查两点。上下文学习是临时的、会话结束就消失，且藏在模型对当前 token 的隐式处理里、无从检查；用户记忆是持久的（跨会话写入存储）、可审查的（一条条显式、带类型，能看能改能删）。要跨越“会话结束”这道坎、并让记忆可被人和系统维护，就值得主动投入额外算力做一次专门提取，而不是指望模型把整段历史一直背在上下文里。

配套代码在 chapter3/user-memory/——它用统一接口实现了记忆的生成与检索，并把对话 Agent 与后台记忆处理器分离（正是本课说的“专门的 LLM 调用”落地的样子）。实验 3-1 就在这个项目里：用三层次框架评估记忆系统，你可以在其中查看每层测试用例的完整定义，亲眼看提取出来的记忆条目长什么样。整本书实验都在开源仓库：github.com/bojieli/ai-agent-book。
