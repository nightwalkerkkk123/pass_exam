想→做→看：ReAct 循环与轨迹的累积

前面几课把 Agent 拆成了大脑、眼睛、手脚。这一课回答那个自然的追问：它们如何协同转起来？答案就是本章「ReAct 循环」一节的核心——思考 → 行动 → 观察的三环节循环，加上一条随迭代不断变长的轨迹（trajectory）。唯一要拿下的判断是一句等式：上下文 = 静态前缀 + 轨迹，以及这条轨迹如何在「多币种收入汇总」那个 3 次迭代、4 次工具调用的例子里一层层累积起来，驱动一个多步任务走到终点。

一、ReAct 是什么：名字只有两个词，循环却有三环

作者把话说得很明确：Agent 执行任务的核心模式叫 ReAct（Reasoning + Acting）。名字里只有「思考」和「行动」两个词，但实际循环包含三个环节——这是初学者第一个容易漏掉的地方：

  • 思考（Reasoning）：模型先想清楚当前应该做什么；
  • 行动（Acting）：调用工具去做（发出 tool_calls）；
  • 观察（Observation）：看工具返回的结果，再据此继续思考下一步。

这个「想→做→看→想→做→看」的圈不断重复，直到任务完成。它不是凭空设计出来的循环，而是第 0006 课里工具调用四步流程的自然延伸——「结果被追加到上下文中，模型据此决定下一步行动」这一步，接回开头，就闭成了环。

二、轨迹：循环转起来后，上下文在长大

循环转一圈，会往哪里留下痕迹？留在轨迹（trajectory）里。作者的定义很干脆：轨迹是「Agent 在执行任务过程中不断积累的消息历史」——用户消息、模型回复（思考过程 + 工具调用）、工具执行结果。

关键的一步在这里：每次调用 LLM 时，它接收的完整上下文由两部分拼成——静态前缀（系统提示词 + 工具定义，对话中不变）和轨迹（动态消息历史，随交互不断增长）。这就揭示了本课要刻进肌肉记忆的等式：

上下文 = 静态前缀 + 轨迹
静态前缀 = 系统提示词 + 工具定义（前两个组件，不变）；轨迹 = 用户消息 + 模型回复 + 工具执行结果（后三个组件，只增不减）

注意作者特意提醒的一点：伪代码里的轨迹不显示系统提示词和工具定义——它们作为静态前缀，在每次 LLM 调用时都会被自动拼接在轨迹前面。所以你看到的轨迹只是「长大的那半截」，前缀那半截一直安静地贴在最前面。

三、把循环走一遍：多币种汇总的 3 次迭代

光有定义还是抽象。作者用一个「多币种收入汇总」的任务把循环走了一遍，这是本课最值得盯着看的例子。任务是：Q1 2.5M 美元、Q2 2.1M 欧元、Q3 1.8M 英镑、Q4 380M 日元，求年度总收入与季度平均。看它怎么用 3 次迭代、4 次工具调用完成：

这里的「精妙之处」，作者点得很清楚，叫上下文的累积性：每次 LLM 调用都能看到完整的轨迹，于是它知道自己处在任务的哪个阶段、之前试了什么、拿到了什么结果——就像人解题时不断回顾。轨迹的结构化还带来两个附赠品：可解释、可调试，因为用户消息、模型回复、工具结果被清清楚楚地分开了。

最常见的想错：把「上下文」当成只是用户输入的那段文本。按书里的定义，这是把眼睛蒙掉了一大半。上下文有五个组件——系统提示词、工具定义、用户消息、模型回复、工具执行结果；而且它们分属两类：静态前缀（前两项，不变）和动态轨迹（后三项，只增不减）。

为什么这个区分不是学究气？因为它直接决定你怎么想成本和缓存：静态前缀每轮原样重发（这正是后面 KV Cache / prompt caching 能省钱的根基），而轨迹是那个越滚越大的雪球。把上下文窄化成「用户那句话」，你就既看不懂 Agent 为什么越跑越贵，也看不懂它为什么能记住三步之前干过什么。

四、为什么这条循环值得单独讲：它是全书的运行引擎

因为几乎所有后面的东西，都挂在这条循环上。作者在「自主 Agent」一节里把话挑明：自主 Agent「本质上就是在一个循环中使用工具的 LLM」——这正是前面介绍的 ReAct 循环。工作流是把这条循环的路径用代码写死，自主 Agent 是让循环自己决定下一步走哪。轨迹数据还能「总结到知识库中，或者通过强化学习来训练更好的 Agent 模型」——这是第八章自我进化和第七章后训练的伏笔。

更狠的一层在思考题里：既然每次 LLM 调用都要看到完整轨迹，而轨迹随迭代线性变长，那么处理的 token 总量就会二次方增长。这不是危言耸听，而是 ReAct 这种「累积式上下文」设计与生俱来的代价，也是后面上下文压缩、Sidecar 旁路等一切工程手段要对付的那个敌人。先记住这条循环长什么样，才能理解它们在省什么。

本课唯一要带走的判断：Agent 干活 = ReAct 循环（想→做→看）不停转，每转一圈就往轨迹里追加消息；而 LLM 每次看到的上下文 = 静态前缀（系统提示词+工具定义，不变）+ 轨迹（用户消息+模型回复+工具结果，只增不减）。理解了这条「累积式上下文」，你就同时理解了 Agent 为什么能多步推进、为什么可调试、以及为什么会越跑越贵。

五、关键定义

轨迹（trajectory）
Agent 执行任务过程中不断积累的消息历史——用户消息、模型回复（思考过程 + 工具调用）、工具执行结果；对应上下文的动态部分。避免与「上下文」整体混用：上下文 = 静态前缀 + 轨迹，轨迹只是其中动态增长的那部分。
静态前缀（static prefix）
每次 LLM 调用都自动拼接在轨迹前、且对话中保持不变的部分，即系统提示词 + 工具定义。避免把用户消息或工具结果算进静态前缀，它们属于动态轨迹。
ReAct（Reasoning + Acting）
Agent 执行任务的核心模式。名字只体现思考与行动两词，但实际循环有三环节：先思考该做什么，再调用工具行动，再观察结果并继续思考，「想→做→看」循环往复直到任务完成。

六、自测题

Q1：为什么说 ReAct 循环的成本随轨迹增长是二次方的？（提示：每次 LLM 调用看到什么）

因为每一次 LLM 调用都要看到完整的历史轨迹，轨迹长度随迭代线性增长，而每次调用处理的 token 量正比于当前轨迹长度，累加 N 轮就是 1+2+…+N ≈ N²，故总成本二次方增长。这正是本章思考题 2 的出发点，也是后续上下文压缩等 Harness 手段要打破的瓶颈。

Q2：在多币种汇总的例子里，Agent 一共用了几次迭代、几次工具调用？第一轮为什么能一次发出多个工具调用？

3 次迭代、4 次工具调用。第一轮 Agent 分析任务后并行调用三个 convert_currency 工具（欧元/英镑/日元各转 USD），第二轮基于转换结果调用代码解释器算总和与平均，第三轮确认完成后生成最终答案。第一轮能一次发多个工具调用，是因为这三笔货币转换互不依赖，可以并行——模型在一条 assistant 消息的 tool_calls 里同时列出三个调用即可。

配套实验：实验 1.1 上下文消融（仓库目录 chapter1/context/）。它正是用本课这套「多工具 + ReAct 循环」的 Agent（PDF 解析 / 货币转换 / 计算器 / Python 代码解释器）跑多币种这类任务，并提供 full / no_history / no_reasoning / no_tool_calls / no_tool_results 五种上下文模式做批量对照，输出成功率、耗时、迭代次数、工具调用数——你可以亲手把「轨迹」里的某一层抽掉，看循环怎么当场崩坏。强烈建议边读边跑：github.com/bojieli/ai-agent-book。
