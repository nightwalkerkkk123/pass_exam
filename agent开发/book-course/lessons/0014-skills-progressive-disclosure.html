别把手册全堆桌上：Agent Skills 的三层渐进式披露

系统提示词不能无限膨胀——客服的退款规则、编程的代码规范、文档的格式要求，全塞进一个 prompt，既浪费 token又稀释注意力。第 2 章「动态提示词与 Agent Skills」一节给出的答案，就是本课唯一要拿下的东西：Skills 的三层渐进式披露——先给目录、按需取书。而这句话真正难的不在「是什么」，在于两个更狠的工程判断：description 为什么要写成路由条件，以及完整内容到底该放进上下文的哪个位置。

一、三层结构：先给目录，需要哪本再去取

作者用一个一秒就懂的类比开场：你不会把公司所有部门的操作手册都堆到新员工桌上，而是先给一份总目录，需要哪本再去取。这就是渐进式披露（Progressive Disclosure）的设计哲学——不是把所有知识一次性塞给 Agent，而是让它按需加载。落到工程上，一个 Skill 分成三层：

  • 第一层（元数据）：每个 Skill 必含一个 SKILL.md，开头是 YAML frontmatter，含 name 和 description 两个字段。框架启动时扫描所有已安装 Skill，把这两个字段（仅占数百个 token）注入上下文，让 Agent 在不消耗大量上下文的前提下知晓自己有哪些专业能力。
  • 第二层（核心流程）：当 Agent 判断某任务需要某个 Skill 时，通过专用 Skill 工具加载完整的 SKILL.md，内容作为 tool result 出现在对话历史中。书里的例子是 PPTX Skill：如何用 markitdown 提取文本、如何解压 PPTX 访问 XML、关键文件路径约定。
  • 第三层（细则）：主文件再通过文件引用深入更细的子文档，如 html2pptx.md、reference.md，Agent 按具体需求选择性阅读。Skill 不止是文档，还可捆绑可执行代码工具与模板文件。

二、为什么 description 要写成「路由条件」而非「功能介绍」

这是本节第一个反直觉的设计决策。元数据里的 description 常驻上下文，是路由决策的关键——它既要足够短（控制常驻 token），写法上却要像路由条件，而不是功能介绍。作者给的最直接写法是「Use when / Don't use when」再加上几条反例（明确列出「不该触发此 Skill」的场景）。

为什么「何时该用我」比「我能做什么」重要得多？因为描述太宽泛（如 “help with backend”）等于任何后端相关的工作都能触发它，路由就会失准。书里说得很硬：反例不是可选项，而是 Skill 路由能否准确触发的关键——缺少反例会让路由准确率明显下降，宽泛描述会在不相关任务上频繁误触发；补上反例后准确率显著回升。

真正有效的描述是路由条件——“何时该用我”比“我能做什么”重要得多。 ——《深入理解 AI Agent》第 2 章 · 动态提示词与 Agent Skills

三、放哪里：两种朴素方案都会疼，方式三如何两头回避

知道了 Skills 是什么，接下来是更具体、也更根本的工程问题：Skill 内容放在上下文的什么位置？这直接关系到 KV Cache 效率与模型的指令遵循效果。理论上有两种朴素方案，但都有明显代价；生产实现（如 Claude Code）走的是回避了两者痛点的第三种。

方式一（注入 system 消息）：把 Skill 内容直接追加进 system prompt。模型对 system 位置的指令遵循能力最强，执行效果最好；但每次加载新 Skill 都改变 system 内容，导致 KV Cache 前缀失效。频繁切换 Skill（先搜索、再文档）时缓存反复失效，延迟与成本显著增加。

方式二（当普通文件读，内容落在上下文中间）：用通用文件读取工具读 Skill，内容作为 tool result 落在对话历史中间。完全不动 system prompt，对 KV Cache 友好；但对指令遵循（instruction following）提出更高要求——模型要在长上下文中间位置准确识别并遵循指令，而不是当作普通工具输出「参考」一下。不同模型对此差异很大：Claude 因训练中大量使用中间位置的指令遵循数据，表现最可靠。

方式三（生产实现）把「路由」和「执行」两步拆开，分别回避前两者的痛点：

  • 元数据列表（所有 Skill 的 name + description，合计仅数百 token）以一条 user 角色的 meta 消息注入上下文末尾，外层用 <system-reminder> 标签包裹。它既不改 system（不破坏 KV Cache 前缀），又位于末尾（注意力位置最优）。且采用增量发送：每个 skill 只在首次出现时发，已发过的不再重复——稳态下每轮元数据增量为零，对缓存极友好。
  • 完整内容通过专用 Skill 工具按需加载。模型从元数据列表识别出合适的 Skill 后，调用形如 Skill(skill: "pdf")，工具内部读 SKILL.md 并返回为 tool result。这绕过了方式二的指令遵循风险——模型对「自己刚刚主动调用的工具的输出」执行倾向远强于对上下文中间一段普通文件内容的遵循。

为什么这套设计对缓存这么友好，可以用一条缓存前缀的示意看清：方式一每次更新都让下游整条 trajectory 失效重算，而方式三只在末尾追加、静态前缀纹丝不动。

四、边界与易错：末尾的注意力优势不是永久的

最容易想当然的一点：以为「末尾注入 → 注意力位置最优」这个好处会永久成立。作者在原文里紧接着就打了补丁——「末尾」的注意力优势只在注入的当轮成立。增量发送的元数据会永久留在轨迹里，随着会话增长，它会逐渐滞留到上下文中部，位置优势随之衰减。

这其实是一对权衡：「只发一次、节省缓存」 与 「每轮置底、保住注意力」 不可兼得。方式三选了前者。下一课讲 Agent 状态栏的持久追加式更新时，会再一次撞上同一个取舍——记住这对张力，比记住任何单一结论都有用。

第二个坑：把「对 KV Cache 友好」误读成「零成本」。原文明确澄清——首次 emit 那几百到几千 token 终归要付一次写入代价（Prompt Cache 的缓存写入还是加价计费的）。它的准确含义是「一次性写入、永久受益」：要让模型知道某个 Skill 存在，至少得让它进缓存一次；Claude Code 做到的是只付这一次。真正「不友好」的是方式一——每次更新让下游整条轨迹进 cache_creation，量级是数万到数十万 token。

想亲眼看「细微改动如何摧毁缓存」，跑实验2-3（仓库 chapter2/kv-cache/）：用 Kimi K2 ReAct Agent 系统对比六种实现模式（一种正确 + 五种错误，含动态系统提示词、动态用户配置、工具动态排序等），量测 TTFT / 缓存命中 / token 用量——正是方式一「改写 system 让前缀失效」的实测版。想看 Skills 三层加载全过程，则可对照实验 2-6（Claude Code + PPTX Skill，从论文 PDF 生成演示文稿）。完整代码见 github.com/bojieli/ai-agent-book。

五、关键定义

渐进式披露（Progressive Disclosure）
Agent Skills 的设计哲学：先给模型看一份 Skill 名称 + 描述的目录摘要（数百 token），需要时再通过专用工具加载完整 SKILL.md 及细则子文档。避免说成「一次性加载全部知识」或普通的懒加载缓存。
KV Cache
模型内部优化：缓存前文 token 在注意力机制中已算出的键值对（Key-Value pairs），下一轮只算新增 token，前提是前缀完全不变。避免与 Prompt Cache 混为一谈——KV Cache 在推理引擎内、单次请求内。
Prompt Cache
构建在 KV Cache 之上的 API 服务层跨请求缓存：服务商对请求前缀匹配，相同则复用已算好的 KV，读取成本约为首次计算的 1/10 到 1/2。避免以为它自动命中——需显式设 cache_control 断点、有写入加价与 TTL。

六、自测题

Q1：把 Skill 完整内容直接追加进 system prompt（方式一），和让 Agent 用通用文件读取工具把它读进上下文中间（方式二），各自的痛点是什么？Claude Code 的方式三如何两头回避？

方式一执行效果最好（system 位置指令遵循最强），但每加载一个 Skill 就改写 system 消息、KV Cache 前缀失效，频繁切换 Skill 时缓存反复失效。方式二完全不动 system prompt、对缓存友好，但要求模型在长上下文中间准确遵循指令，不同模型差异大。方式三把「路由」和「执行」分离：几百 token 的元数据以 user 角色 meta 消息增量注入上下文末尾（不改前缀、位置最优、稳态增量为零），完整内容通过专用 Skill 工具按需加载（模型对「自己刚调用的工具输出」执行倾向远强于对中间普通文件内容）。

Q2：为什么 Skill 的 description 要写成「Use when / Don't use when + 反例」，而不是介绍功能？缺少反例会怎样？

因为 description 是常驻上下文的路由条件——决定模型何时触发该 Skill。「何时该用我」比「我能做什么」重要得多。描述太宽泛（如 “help with backend”）会让任何沾边的任务都触发它，路由失准。原文强调：缺少反例会让路由准确率明显下降、在不相关任务上频繁误触发；补上反例后准确率显著回升。反例不是可选项，而是路由能否准确触发的关键。

本课唯一要带走的判断：Skills = 渐进式披露（目录常驻、正文按需）+ description 写成路由条件（Use when / Don't use when + 反例）+ 生产实现把「路由」和「执行」拆开——元数据以 user meta 消息增量置底（保前缀、稳态零增量），完整内容靠专用工具按需拉取（借「刚调用的工具输出」拿到最强执行倾向）。而末尾的注意力优势只在当轮成立，会随会话增长而衰减。
