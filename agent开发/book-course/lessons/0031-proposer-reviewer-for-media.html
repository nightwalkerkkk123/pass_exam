提议者-审核者机制：让 Agent 看见自己生成的效果

这节课只押一件事：Agent 写完 Slidev 代码，其实并不知道它渲染出来长什么样——内容挤不挤、文字溢不溢、图片尺寸对不对，只有真正渲染出来才看得见。第 5 章“代码驱动的多媒体生成”一节给出的解法，是把“写”和“看”拆成两个 Agent：Proposer 生成代码，Reviewer 渲染成图、用 Vision LLM 出结构化反馈，一轮轮迭代（对应图5-5）。而这套双 Agent 设计真正的甜头，不在“分工”本身，而在上下文管理。

一、问题：代码写完了，效果却是个盲区

先看这节课嵌在书里的位置。第 5 章后半部分讲“代码是通用 Agent 的元能力”——代码能当场造出新工具、新约束、新表达形式。多媒体生成是这个元能力的第三个发挥方向：内容呈现。作者的洞察是，PPT、报告、界面这些复杂文档，本质都是结构化数据的组织与呈现，底层都由代码定义（HTML 描结构、CSS 控样式、JS 做交互）。让 Agent 用代码生成它们，就绕开了 GUI 所见即所得编辑里最难的视觉定位与坐标点击。

以 PPT 为例。一份学术报告 PPT 动辄数十页，逐页都要设计布局、提炼要点、选配图表。现代框架如 Slidev 用 Markdown + HTML 定义演示内容，对会写代码的 Agent 极其友好——写一页幻灯片只需写一小段标记，框架自动处理渲染、布局、动画。

但作者随即点出一句要命的话：

仅能生成代码还不够。Agent 编写完代码后并不知道实际渲染效果：内容是否太挤、文字是否溢出、图片尺寸是否合适，这些只有真正渲染出来才能发现。 ——《深入理解 AI Agent》第 5 章

这就是缺口所在——代码是符号，效果是像素。Agent 手里只有符号，它对最终呈现出来的像素是盲的。要补上这只“眼睛”，就得让某个环节真正把代码跑出来、看一眼、再把看到的东西喂回去。

二、解法：Proposer 写，Reviewer 看

作者的做法是把“代码编写”和“质量评审”解耦成两个独立 Agent（如图5-5）：

  • Proposer Agent：负责生成 Slidev 代码，理解内容的逻辑结构，把它分解为合理的页面。
  • Reviewer Agent：运行代码，把每一页渲染为图片，用 Vision LLM（能“看”懂图片的多模态大模型）从内容密度、可读性、布局合理性、视觉美感等维度分析渲染结果，产出结构化的改进建议。

关键在“结构化”三个字。Reviewer 给的不是模糊的“不好看”，而是具体可执行的指导——书里的原话举例是“第 3 页：内容过多，建议拆分”“第 7 页：代码块字体过小，建议增大到 14pt”，每条都带页码、问题类型、严重程度等字段。

然后就是循环：Proposer 接收反馈、理解意图、修改代码，新版本再次交给 Reviewer 审查，迭代直到质量达标，或达到最大次数（如 5 轮）。

三、为什么是“两个 Agent”，而不是一个 Agent 自己写自己看？

这是本课最该深挖的“为什么”。一个 Agent 既写代码又看渲染图、自己迭代，听上去更简单，为什么书里坚持要拆成两个？作者给的答案很硬：核心优势在于上下文管理。

把两条路径摆在一起对比就清楚了：

拆开看两个好处：

  • Reviewer 侧：每次只处理最新一版的渲染图片，不被历史版本的旧图干扰——它不需要记得三轮前那版长什么样。
  • Proposer 侧：只累积结构化文本反馈（几行带页码的建议），token 消耗少，而且纯文本比图片更易于推理。

反过来，单 Agent 若想自审自改，就得把每一轮、每一页的渲染图都堆在同一个上下文里。图片是 token 大户，数十页乘上多轮迭代，上下文迅速超限。所以这里的“双 Agent”不是为了组织上的分工好看，而是一个被上下文预算逼出来的工程决策。

还要看清它的“血统”。作者特意点明：本章的提议者-审核者迭代循环，和第四章的事前审批是同源的——都是提议者-审核者范式的实例，共享同一批设计原则：共享目标约束、使用不同模型家族以降低同类错误概率、把反馈作为特殊事件加入 Proposer 轨迹。差异只在目标与形态：第四章用它做不可逆操作的安全审查（审核者对单次操作给批准/否决）；本章用它做内容质量的迭代改进（多轮循环，且审核者能接触到提议者看不到的新信息——渲染结果）。

四、边界与易错

最常见的想当然：“不就是让 Agent 检查一下自己写的代码嘛。” 错在两处。其一，Reviewer 检查的不是代码，而是代码渲染出来的图片——它靠 Vision LLM“看”像素，专门抓那些从代码里看不出来的问题：文字溢出、内容拥挤、图片尺寸不当。检查代码本身根本发现不了“这一页挤不挤”。

其二，也是更深的一点：拆成两个 Agent 的动机不是“两个脑子比一个准”，而是上下文管理。如果你以为单 Agent 只是“效果差一点”，就会漏掉真正的失败模式——单 Agent 会因为把数十页图片乘以多轮迭代全塞进一个上下文而直接超限、根本跑不下去。省 token、可持续迭代，才是这套机制的立身之本。

这套机制也不是 PPT 专用。作者明确说它会在后续视频编辑（Proposer 生成 Blender 脚本，Reviewer 渲染关键帧再用 Vision LLM 检查）和日志可视化实验中重复使用；第十章还会进一步探讨提议者-审核者之外的其他多 Agent 协作模式。记住这条循环，后面几节能少走弯路。

配套实验：实验 5-4 ★★ 基于论文的 PPT 自动生成（参考实现见开源仓库 chapter5/coding-agent/）。Proposer 阅读论文 PDF、规划结构、逐页生成 Slidev 代码；Reviewer 渲染每页截图、用 Vision LLM 查文字溢出/内容拥挤/图片尺寸问题、出结构化建议，迭代到达标。验收标准里专门要求你对比“单 Agent 自我审查”与“提议者-审核者分工”在上下文消耗和生成质量上的差异——这正是本课第三节结论的动手验证。相邻的实验 5-5（论文讲解视频）、实验 5-6（Blender 视频剪辑）会把同一机制迁移到视频。

五、关键定义

提议者-审核者机制（Proposer-Reviewer）
把“生成”与“审查”解耦为两个独立 Agent 的范式：Proposer 生成产物（如 Slidev 代码），Reviewer 用独立视角评估并给出反馈，多轮迭代至达标或触顶。与第四章事前审批同源，共享“不同模型家族、反馈作为特殊事件入轨迹”等原则。避免把它理解成“Agent 检查自己写的代码”——审查的是渲染出的效果，且分工动机是上下文管理。
元能力（meta-capability）
一种“能创造其他能力”的能力：Agent 用它当场写出新工具、新约束、新表达形式来完成任务，而不必事先把所有能力都预制好。代码生成正是这样的元能力；多媒体生成是其“内容呈现”方向的发挥。避免等同于“一个很强的能力”或“万能工具”——关键在“创造能力的能力”，而非能力本身强弱。
结构化反馈
Reviewer 产出的可执行改进建议，带页码、问题类型、严重程度等字段（如“第 3 页：内容过多，建议拆分”），而非模糊的“不好看”。它是纯文本、轻量、易推理，因此可以低成本地累积进 Proposer 的轨迹。

六、自测题

Q1：为什么用两个 Agent 分工，而不是让一个 Agent 自己写代码又自己看渲染图迭代？

核心是上下文管理。单 Agent 要在同一上下文里累积数十页、多轮的渲染图片，上下文迅速超限；双 Agent 下 Reviewer 每次只看最新版渲染图、不受历史干扰，Proposer 只累积轻量的结构化文本反馈，token 少且更易推理。

Q2：Reviewer 靠什么“看见”问题？它能接触到而 Proposer 接触不到的关键信息是什么？

Reviewer 先运行代码把每页渲染为图片，再用 Vision LLM 从内容密度、可读性、布局合理性、视觉美感等维度分析。它能接触到、而 Proposer 接触不到的关键信息，正是渲染结果（图片）本身——这也是本章提议者-审核者相较第四章事前审批的独特之处：审核者拿到了提议者看不到的新信息，据此产出带页码/问题类型/严重度的结构化建议。

本课唯一要带走的判断：Agent 写完代码看不见效果，所以要引入提议者-审核者——Proposer 写、Reviewer 渲染成图用 Vision LLM 看、结构化反馈回喂迭代。而拆成两个 Agent 的真正理由不是“两个更准”，而是上下文管理：图片留在 Reviewer 且用完即弃，Proposer 只背轻量文本反馈，才让多轮迭代不至于撑爆上下文。
