五类工具：用「谁发起 × 作用于什么」两个坐标钉死定位

第 4 章一上来不讲任何一个具体工具，而是先摊开一张地图——表4-1。这节课唯一要拿下的东西，就是这张地图的两根坐标轴：调用方向（这次交互由谁发起）和作用对象（这次交互作用于什么）。用这两个特征，感知 / 执行 / 协作 / 事件触发 / 用户沟通五类工具的定位被一次性钉死。把这张图记牢，后面整章逐类深入时你都不会迷路。

一、先给意象：Samantha 的「手脚和感官」

作者借科幻电影《Her》里的 AI 助手 Samantha 开场：她能主动整理邮件、识别情感复杂的信件并提议润色回复，能代表主角处理出版事宜，还能在不同沟通渠道间无缝切换。她的智能之所以动人，是因为她拥有强大的工具——连接语言「大脑」与真实数字世界的「手脚和感官」。

二、两根坐标轴，一张表

要理解这五类工具的设计差异，作者建议从两个特征去审视它们：

  • 调用方向——这次交互由谁发起。是 Agent 自己主动去调，还是外部世界把它叫醒？
  • 作用对象——这次交互作用于什么。是拿信息、改世界、驱动别人，还是唤醒自己、告知用户？

这就是表4-1 的两列。下面这张信息图把表4-1 逐行画开——左看「谁发起」，右看「作用于什么」：

逐类对号入座，各给几个书里的例子：

  • 感知工具（主动 / 获取信息）：web_search、knowledge_base_search、fetch_url、find_file、grep_file、read_file。设计关键在粒度权衡与输出信息量的控制。
  • 执行工具（主动 / 改变世界）：shell_exec、code_interpreter、write_file、edit_file、send_email。与感知不同，它的错误代价可能极高，安全约束是设计核心。
  • 协作工具（主动 / 驱动其他 Agent 或人类）：spawn_subagent、send_message_to_subagent、cancel_subagent。最简单的动机是并行执行互不相关的多个任务。
  • 事件触发工具（注册+外部触发 / 驱动 Agent 开始执行）：set_timer、monitor_shell、connect_channel。
  • 用户沟通工具（主动 / 向用户传递信息）：reply_to_user、send_card_to_user、send_user_notification。

三、为什么「事件触发」这一行长得不一样（深挖）

盯着上图，会发现只有事件触发工具的箭头是反的。这不是画着好看，而是它内在的双时刻结构：

  • 注册时——由 Agent 主动调用工具，声明「我关心什么事件」（比如设一个定时器、连上一个邮件通道）。
  • 触发时——由外部事件异步回调，把 Agent 从「没在运行」的状态里唤醒，让它开始处理。

这正是表4-1 里那句「Agent 注册、外部触发」的全部含义。它为什么值得单独成一类？因为没有它，Agent 就只能等用户开口才能动——无法在指定时间自主行动，也无法对新邮件、系统告警这类外部事件做出反应。Samantha 之所以能「主动」整理邮件，靠的就是这一类。也正因为它和用户沟通工具都依赖「Agent 在没有会话保持时被唤醒 / 主动说话」，这两类工具的设计离不开本章后半部分的事件驱动异步架构——而前三类由 Agent 主动调用的工具，则会先逐类展开。

误区一：把这两列当成一张交叉分类矩阵去填。初学者一看「两个特征」，本能地想画一个「调用方向 × 作用对象」的九宫格，然后去每个格子里塞工具。作者在原文里专门打了预防针：这两列并不构成一个交叉分类框架——每类工具在「作用对象」上各有专属取值。它们只是帮你快速把握每类工具的定位，不是让你去做笛卡尔积。

误区二：靠「用什么渠道发」来归类。「发通知」这个动作，用邮件还是 Slack 发，跟它属于哪一类无关。归类看的是作用对象，不是渠道：通知一个协作者 / 审批者去推进任务，是协作工具（驱动其他 Agent 或人类）；通知最终用户本人，才是用户沟通工具（向用户传递信息）。同一个「发通知」，对象不同，归类就不同。

四、这张地图为什么重要（宏观背景）

第 4 章的野心是把「工具」讲透——通用设计原则、MCP 生态、执行工具的多层安全、支撑主动服务的事件驱动架构，内容很密。作者不先给这张地图，你读到后面就会被一堆工具名淹没。这两根坐标轴的价值，是让你随时能回答一个定位问题：我现在读的这个设计，是在解决「Agent 怎么伸手」还是「世界怎么叫醒它」的问题？调用方向一分，全章的结构就立起来了——前三类主动工具讲「怎么把手伸得准、伸得安全」，后两类被动 / 异步工具讲「怎么让 Agent 活在一个随时会打断它的真实世界里」。

本课唯一要带走的判断：拿到任何一个工具，先问两句——这次交互谁发起（调用方向）、作用在什么上（作用对象）。四类是 Agent 主动伸手，只有事件触发是「注册在先、外部叫醒」。归类永远看作用对象，不看用什么渠道。这两列不是矩阵，别去填九宫格。

五、关键定义

调用方向（Invocation direction）
审视一类工具的第一个特征：这次交互由谁发起。取值只有两种——Agent 主动调用，或 Agent 注册后由外部异步触发。避免把它和「作用对象」拼成一张交叉分类矩阵。
作用对象（Target of action）
审视一类工具的第二个特征：这次交互作用于什么。五类各有专属取值：获取信息 / 改变世界 / 驱动其他 Agent 或人类 / 驱动 Agent 开始执行 / 向用户传递信息。归类以此为准，不看使用的渠道。
事件触发工具（Event-trigger tool）
外部世界驱动 Agent 行动的方式，涉及两个时刻：注册时 Agent 主动声明关心的事件，触发时外部异步回调唤醒 Agent。避免把它和 Agent 主动调用的工具混为一谈——它的箭头是反的。
ACI（Agent-Computer Interface）
对标 HCI（人机交互）提出的概念：研究 Agent 如何与计算机交互，核心是让工具对 Agent 而非对人友好——工具应对应 Agent 的目标而非底层 API 操作。这是分类之后逐类设计工具时的通用透镜。避免把它等同于「API」或「函数调用规范」；ACI 是设计理念层面的，不是某种具体协议。

六、自测题

Q1：「新邮件到达时唤醒 Agent」和「Agent 回复用户消息」分别属于哪类工具？为什么发通知这个动作可能既属协作又属用户沟通？

前者是事件触发工具（Agent 注册、外部触发）；后者是用户沟通工具（Agent 主动调用、向用户传递信息）。发通知的归类不看渠道而看对象：通知审批者 / 协作 Agent 去推进任务，归协作工具；通知最终用户本人，才归用户沟通工具。同一个动作，作用对象不同，归类就不同。

Q2：感知工具和执行工具都是「Agent 主动调用」，那它们靠哪一列区分？各自的设计核心又是什么？

它们的调用方向相同，区别全在作用对象：感知是「获取信息」，执行是「改变世界」。设计核心也随之不同——感知工具的关键在粒度权衡与输出信息量的控制；执行工具因为错误代价可能极高，安全约束才是它的设计核心。

本章的五类工具各有可跑的 MCP 服务器实现，正好一类一验：感知工具见实验4-1（chapter4/perception-tools，搜索 / 多模态理解 / 文件系统 / 公开与私有数据源）；执行工具见实验4-2（chapter4/execution-tools，带自动 linter 验证的写入、沙盒 code_interpreter、危险命令检测、LLM 事前审批）；协作工具见实验4-3（chapter4/collaboration-tools，浏览器自动化、HITL 审批、多渠道通知、定时器管理）；事件触发与用户沟通两类则在实验4-4 / 4-5（chapter4/agent-with-event-trigger，FastAPI 原生 async，自动加载 3 个 MCP 服务器共 42 个工具，响应 web 消息 / 邮件 / GitHub / 定时器等事件）里合并演示。建议边读边跑：github.com/bojieli/ai-agent-book。
