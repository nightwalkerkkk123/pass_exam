稠密 vs 稀疏 vs 重排序：生产级 RAG 为什么三管齐下

检索器的质量直接决定 RAG 的效果——检索不到相关片段，LLM 再强也无米之炊。这节课只押一件事：看懂稠密嵌入与稀疏 BM25 各自的盲区，以及混合检索三个阶段（并行检索 → 融合 → 重排序）为什么各自不可替代。它对应第三章“检索器的两大技术路线”与“混合检索：两全其美的艺术”一节，是全书 RAG 底层技术的地基。

一、两条路线，两种盲区

文档进了知识库，检索器要从里面把最相关的几块捞出来。作者给了两条技术路线，它们的世界观截然不同：

  • 稠密嵌入（Dense Embedding）：用深度学习把文本映射到向量空间，语义相近的内容向量距离也近。判断“有多近”用余弦相似度——比较两个向量夹角，值越接近 1 语义越相似。它懂语义，同一个词“苹果”在“苹果公司”和“买两斤苹果”里会得到不同向量。
  • 稀疏嵌入（Sparse Embedding）：根植于传统信息检索，核心是精确的关键词匹配。经典代表 BM25 靠倒排索引（从词到文档的反向映射表）做字面命中，在 TF-IDF 基础上引入 k1（词频饱和度）和 b（文档长度归一化）两个参数。它精确，但读不懂同义表达。

二、混合检索三阶段：各司其职、层层递进

混合检索的思路很简单——两个引擎都跑，结果合并——难点在于把分布迥异的两组得分整合成一个有意义的排序。作者把典型流水线拆成三个阶段，三者各司其职、层层递进：

① 并行检索：系统同时向稠密和稀疏两个引擎发送查询，各自召回一部分候选文档。

② 结果融合：把两路结果合成一个统一候选池。难点在于两路得分不可直接比较——稠密的余弦相似度（归一化后通常落在 0 到 1）和 BM25 得分（0 到几十的任意值）尺度、分布完全不同。作者给了两种常用融合法：一是各路得分分别归一化后加权求和；二是倒数排名融合（RRF）——完全抛开原始得分、只看排名：

RRF 得分 = Σ 1 / (k + rank)
k 是平滑常数，常取 60，用于压低排名最靠前几个位置之间的得分差距

RRF 简单鲁棒，但只利用排名信息，丢失了原始得分中蕴含的相关性信号；改用加权归一化融合则保留得分，代价是两路尺度对齐本身不好调。这是一个真实的工程权衡，没有免费的午餐。

③ 神经重排序：对融合产生的候选池中排名靠前的 N 个候选（如前 50 个）逐一精细打分，产生最终排序。这一步用的是跨编码器——下一节的主角。

三、双编码器 vs 跨编码器：换范式，不是补救

为什么第三阶段值得单独存在？作者给了一个精准的比喻：求职者把简历交给猎头快速筛选，是双编码器；面试官与每位候选人深谈，是跨编码器。

  • 双编码器（Bi-Encoder）：为查询和文档各自独立生成向量，再靠向量运算算相似度。速度极快，适合从海量数据中初筛——这正是检索阶段（稠密引擎）用的范式。
  • 跨编码器（Cross-Encoder）：把查询和候选文档拼接成一段完整的文字送入模型，让模型逐词比对、输出一个综合相关性得分。慢得多，但判断更准，能捕捉双编码器无法感知的细微语义关联。常用模型如 BAAI/bge-reranker-v2-m3。

常见误区：以为重排序是在“补救 RRF 丢掉的得分”。初学者看到“RRF 丢失了原始得分信号”，很自然会推断——那第三阶段就是把丢的分再找补回来吧？错。

作者专门强调：重排序不是为补救融合而存在的。无论前一步用 RRF 还是加权归一化，重排序都值得加，因为它换用了一种更强的匹配范式——从“各自独立编码再比向量”换成“拼在一起逐词交互”。它的收益来自范式升级，而不是修补上一阶段的损失。把这两件事混为一谈，你就会误以为“融合做好了就能省掉重排序”，从而在生产系统里漏掉最能提升排名质量的一环。

还要记住重排序并不替代融合：融合负责从两路结果产生统一候选池，重排序负责在这个候选池上精排。两者是串联的分工，不是二选一。作者的结论很直白：没有哪种单一检索策略在所有场景下都可靠，把稠密、稀疏和重排序组合起来，才是构建生产级 RAG 系统的正确做法。

本课唯一要带走的判断：稠密漏关键词、稀疏漏同义词，两者盲区互补，所以要并行都跑；融合（RRF 或加权归一化）解决“两路得分没法直接比”的尺度问题，产出候选池；重排序用跨编码器逐词精排，是换范式而非补救融合——因此无论融合怎么做，它都值得加。三管齐下，才是生产级 RAG。

四、关键定义

稠密嵌入（Dense Embedding）
用深度学习把文本映射到向量空间，语义相近的内容向量距离也近；用余弦相似度（比较向量夹角，越接近 1 越相似）衡量。懂语义，但可能漏掉关键词。避免把它当成唯一检索手段。
BM25
稀疏检索的经典概率排序算法，在 TF-IDF 基础上引入 k1(词频饱和度)和 b(文档长度归一化)两参数，靠倒排索引做精确关键词匹配，至今仍是搜索引擎核心组件。避免说它懂语义——它只按词频和文档频率算分、读不懂同义表达。
倒数排名融合（RRF, Reciprocal Rank Fusion）
抛开原始得分、只看排名的融合法，综合得分 = Σ 1/(k + rank)，k 常取 60。简单鲁棒，代价是丢失原始得分蕴含的相关性信号。避免误以为它需要两路得分尺度对齐——它恰恰绕开了这个问题。
跨编码器 / 双编码器（Cross-Encoder / Bi-Encoder）
双编码器对查询与文档各自独立编码再比向量，快、用于检索初筛；跨编码器把两者拼成一段文字送入模型逐词交互打分，慢但准，用于重排序。避免把重排序理解成“补救融合”——它是更强的匹配范式。

五、自测题

Q1：搜“kitty”时只写了“cat”的文档搜不到，搜“HTTP-403”时返回泛泛的“服务器错误”讨论。这两种失败分别是稠密还是稀疏检索的盲区？混合检索里重排序用的是双编码器还是跨编码器，为什么更准？

“kitty→cat”搜不到是稀疏检索盲区（只能字面匹配，同义词失效）；“HTTP-403”返回泛泛讨论是稠密检索盲区（懂语义但漏掉关键词）。重排序用跨编码器——把查询和文档拼成一段文字送入模型逐词比对（对比双编码器各自独立编码再算相似度），能捕捉深层匹配所以更准，代价是慢。

Q2：RRF 公式里的 k 常取多少、作用是什么？RRF 相比“加权归一化求和”的融合法，牺牲了什么？

k 是平滑常数，常取 60，作用是压低排名最靠前几个位置之间的得分差距。RRF 完全抛开原始得分、只看排名，因此丢失了原始得分中蕴含的丰富相关性信号；而加权归一化融合保留了得分，代价是两路尺度对齐本身不好调——两种方法各有取舍。
