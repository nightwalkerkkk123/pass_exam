录制-回放：重复任务提速数倍，但必须装上验证闸门

这节课只押一件事：把一次成功的操作序列录下来、以后直接回放，能把重复任务提速数倍——但只有当你给它装上两道验证闸门时，这套机制才不会腐坏。它出自第 8 章「自我进化」里的实验 8-2（browser-use-rpa），是「工作流录制」（Workflow Recording）从工程技巧上升为可靠机制的关键一节。

一、是什么：像 Excel 宏一样，把一次成功录成可回放的工作流

作者给的直觉类比一秒就懂：Excel 的「宏录制」——第一次手动操作时把步骤录下来，以后只需点一下「回放」就自动重复。很多浏览器里的重复操作（发报告邮件、查特定网站信息）每次参数不同（收件人、查询关键词），但核心流程是固定的。让 Agent 每次都用昂贵的多模态大模型从头「重新发现」这个流程，是极大的资源浪费——本质上是只依赖上下文学习，而没有把成功经验外部化为可复用的工具。

机制分两个阶段。学习阶段（Learning Phase）：Agent 首次执行时像人一样跑一遍多模态 LLM 的观察-思考-行动循环；每次决定动作，系统就从 browser-use 的历史记录里抽出被操作元素的精确定位——网页是一棵 DOM 树，每个按钮/输入框/链接是一个节点，XPath 用类似文件路径 /html/body/div[2]/button[1] 的写法指向它。每步录成结构化条目：操作类型、目标元素 XPath、操作参数、以及执行后的验证信息（如 URL 是否变化、预期元素是否出现）。任务成功后，LLM 生成语义标签（如「发送电子邮件」）和描述，与步骤序列一起存入知识库，成为一条参数化的「工作流」。

回放阶段（Replay Phase）：新任务到达，系统用语义相似度（嵌入向量）加关键要素检查去匹配已有工作流；命中就按步高速执行，靠 Playwright 的等待机制确保元素加载，用轻量 LLM 调用从当前指令里抽出实际参数值填进模板（如「在收件人字段输入 {{email}}」），无需完整的视觉思考。

二、为什么这样设计：录制省掉的，是「重复的多模态思考」

为什么值得专门录制？因为重复任务里真正贵的东西不是点击本身，而是每次都要重新跑一遍多模态视觉思考——用大模型去「看懂」这个页面、认出哪个是发送按钮。回放阶段把这一段整个省掉：命中缓存时一次模型调用都不需要，参数靠一次轻量 LLM 提取就够。作者据此给出量化结论：命中缓存的重复任务能快上 8.5–13 倍，LLM 调用成本大幅降低，成功率也更稳定。

这套「探索→沉淀→复用」并非孤例。作者点明它背后是更通用的方法论：NVIDIA 的 Voyager 在 Minecraft 里把循环系统化为「执行任务 → 验证成功 → 存入技能库 → 类似任务检索复用」。实验 8-2 就是这套思路在浏览器自动化的落地——学习阶段对应「探索」，工作流库对应「技能库」，回放与失效回退对应「检索复用」和持续改进。注意这里 Voyager 循环里也有「验证成功」这一环，绝非可有可无。

三、边界/易错：两道验证闸门，缺哪一道都会腐坏

作者说得很直白：实验 8-2 暴露了「录制-回放」最脆弱的两个环节，把它们处理干净，机制才真正可靠。

环节一：什么时候敢信回放。更稳妥的做法是把一次成功序列编译成一个小小的状态机程序，每个状态带一个「验证谓词」——一段必须在当前真实屏幕上成立的界面模式。回放时每一步动作之前，先拿谓词去核对实时屏幕，即「先看清、再动手」；一旦谓词不成立或动作报错，就交还给完整的 Agent 重来，并把新轨迹重新编译成程序。

环节二：别把坏程序存进去。编译完成后立刻重置环境、从头回放一遍，用基准自带的评判器确认「这次是真的做成了」才准入库——这就是「存前验证」闸门。

初学者最容易把录制-回放当成纯工程技巧，觉得「回放能 100% 覆盖录下来的每一步，就等于任务办成了」。这恰恰是最坑的地方。作者举的反例：流程全走完、Save 也点了，但某个字段其实是空的——回放覆盖率 100%，事却没办成。如果没有「存前验证」这道关，这种「假成功」的坏程序就会被存进库；随着有毛病的程序越攒越多，程序库会逐渐退化，之后命中它的每个任务都被带偏。

记住那条被作者压成一句话的原则：流程记忆也要有验证闸门，自我改进的循环才不会腐坏。回放覆盖 ≠ 任务成功，中间隔着一个必须由评判器来跨过的鸿沟。

本课唯一要带走的判断：录制-回放靠省掉重复的多模态思考换来 8.5–13 倍加速，但它天生会腐坏——必须装两道验证闸门：回放时每步先用「验证谓词」核对实时屏幕（先看清再动手），入库前用评判器做一次「存前验证」确认真做成。缺哪一道，程序库都会被「假成功」的坏程序拖垮。

四、配套实验

实验 8-2 ★★：从重复任务中学习——工作流录制与回放，对应仓库 chapter8/browser-use-rpa/。它封装 browser-use：学习阶段用多模态 LLM 完成任务并捕获操作步骤（XPath + 参数），应用阶段识别相似任务后直接回放工作流、免去 LLM 调用，并持续记录指标改进。验收场景就是在 Gmail 网页版发邮件——先跑一遍学习阶段，再换参数看回放阶段的耗时与调用次数骤降，最后模拟网页改版让某按钮 XPath 变化，验证 Agent 能检测失效并回退重学。完整开源仓库：github.com/bojieli/ai-agent-book。

五、关键定义

外部化学习（Externalized Learning）
把知识沉淀到模型之外的文件、知识库和工具中的学习范式，持久、可解释、可随时修正；含「经验外部化为知识/Skill」与「能力外部化为工具」两个维度。避免等同于「记忆」或泛泛的「持久化」——它特指不改权重的第三条学习路径。工作流录制属于「能力外部化为工具」这一侧。
验证谓词（Verification Predicate）
把成功轨迹编译成状态机程序后，每个状态附带的一段「必须在当前真实屏幕上成立的界面模式」。回放时每步动作前先用它核对实时屏幕——先看清再动手；不成立则交还完整 Agent 重学。
存前验证闸门
把编译好的程序准入工作流库之前的一道关：立刻重置环境从头回放一遍，用基准评判器确认「真的做成了」才入库。用来挡住「回放 100% 覆盖却其实没办成事」的坏程序，防止程序库退化。

六、自测题

Q1：录制-回放机制最脆弱的两个环节是什么？为什么去掉「存前验证」闸门会让程序库退化？

两个环节：(1) 什么时候敢信回放——应把成功序列编译成带「验证谓词」的状态机，每步动作前先拿谓词核对实时屏幕（「先看清再动手」），谓词不成立或报错就交还完整 Agent 重来；(2) 别把坏程序存进去——编译后立即重置环境从头回放一遍、用评判器确认真做成了才准入库。去掉存前验证，会放进「回放 100% 覆盖却其实没办成事」（如某字段实际为空）的坏程序，程序库随坏程序增多而退化。命中缓存的重复任务因无需模型调用可快 8.5–13 倍。

Q2：回放阶段命中已有工作流后，系统还需要一次「轻量 LLM 调用」，它是用来干什么的？为什么这不违背「回放无需完整视觉思考」？

那次轻量 LLM 调用用于从当前任务指令中提取实际参数值，填进参数化模板（如把「给 another@example.com 发邮件」里的地址填进「在收件人字段输入 {{email}}」）。它不做完整的多模态视觉思考——不需要重新「看懂」页面、认出哪个是发送按钮，那部分早在学习阶段就被录制成 XPath 步骤了。省掉的正是这段昂贵的视觉思考，所以命中缓存能快 8.5–13 倍。
