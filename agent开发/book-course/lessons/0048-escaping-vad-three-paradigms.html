三种语音范式，一根轴：摆脱 VAD 的轮次假设

级联、Omni、全双工——语音 Agent 的三代架构听起来像三种彼此无关的技术。这节课只押一件事：它们其实是同一条轴上的三个刻度，那根轴叫「如何摆脱 VAD 对轮次的猜测」。把这根轴握在手里，你就能一眼看穿一件反直觉的事——Omni「把三个模型合成一个」，却没有解决用户报电话号码时被抢话的问题。这条主线出自第九章开篇的「语音架构的三种范式」，是整章的坐标系。

一、三种范式：是什么

要理清语音 Agent 的技术演进，作者借用了 OpenAI 在 2026 年发布 GPT-Live 时给出的三分法——它恰好对应 ChatGPT 语音自身走过的三代架构：

  • 级联（Cascaded）：把 VAD、ASR、LLM、TTS 串成一条流水线，一棒接一棒（图9-1）。模块清晰、可独立调优，但延迟逐段累积。
  • 端到端全模态（Omni）：用单一模型直接「听音频、想回复、说出来」，把三段合一（图9-4）。延迟更低，韵律与情感等非文字信息得以保留。但它仍然假设「轮流说话」——要等用户停顿才开口，切换靠静音判断。OpenAI 称其为「轮次式（turn-based）」，工业界按能力叫「全模态（Omni）」，两个名字指同一类东西（如 Qwen3-Omni）。
  • 全双工 / 交互式（Full-Duplex）：模型边听边说，同时处理输入与输出，每秒做许多次「该说、该听、该停、该打断、还是该调用工具」的决策，彻底取消了「轮流」这个假设。2024 年 Kyutai 的 Moshi 是研究先声，2026 年 OpenAI 的 GPT-Live 把它带到了 1.5 亿用户的规模。

二、一根轴：摆脱 VAD 的轮次假设

作者在这里下了一句关键判断——贯穿这三代的其实是同一条主线：如何摆脱「轮流说话」这个假设，摆脱 VAD（语音活动检测）对轮次的猜测。理解了这句话，三种范式就从「三种技术」塌缩成「一根轴上的三个点」。

为什么 VAD 是这根轴的原点？因为它的核心设计——结束点检测（End-of-Speech Detection）——只能靠一个笨办法猜「用户说完没有」：设一个 500-800ms 的连续静音阈值，停顿超过这个数就判定说完。这引入了第一层延迟，而且两头不讨好：阈值太短，用户思考时顿一下就被截断句子；太长，说完了还得干等大半秒。轮次，从一开始就是猜出来的。

沿这根轴看，级联一节里的流式语音感知就是把判断依据从「静音时长」升级到「语义层面」——用一个流式识别模型综合语义和声学判断「这句话是不是已经表达完一个完整的意思」，因为轮内停顿（报电话号码时顿一下）常常比轮间间隔还长，光靠静音阈值必然两头不讨好。但它升级的是「怎么判断轮次」，仍在轮次框架内。只有到了全双工，模型一直在听，「打断」不再是一个需要专门处理的事件，barge-in 处理链在架构上被省去了大部分环节——这才是「替代 VAD」这条叙事线的终点。

三种范式并非简单的新旧替代，而是不同延迟与成本约束下的设计取舍，在 2026 年的生产系统里长期并存。 ——《深入理解 AI Agent》第九章 · 语音架构的三种范式

三、为什么「合三为一」没解决抢话

这是本课最容易想错、也最值得停下来的地方。直觉会说：Omni 把 ASR-LLM-TTS 三个模型合成了一个，中间那些接力交棒的缝隙没了，抢话问题总该解决了吧？

错在把「合三为一」当成了「取消轮流」。作者说得很直白：Omni 无论多强，本质上也只是把三个模型合成了一个，并未取消「轮流说话」这一假设。它依然依赖 VAD 来划分发言权——一旦检测到用户出声便停下，用户一静音便随即开口。

于是那个熟悉的问题再度浮现：用户报出一串数字、中途略作停顿，Omni 便判定对方已说完而强行插话。合并的是「听/想/说」这三件事，动的是延迟和韵律；而抢话的病根在「谁有发言权」这个更底层的假设上，合并根本没碰它。前文的流式语音感知能把轮次判断从静音时长升级到语义层面，大幅缓解这类误判——但那终究只是在「轮次」框架内的局部修补，并未取消轮流本身。

要从根本上跳出这一困境，就不能再在「轮次」框架内修补，而须让模型边听边说、自主决定何时开口，不再存在「该谁说话」的硬性切换。这正是全双工做的事：Thinking Machines Lab 把主张挑明——「要让交互性随智能一同扩展，交互性就必须成为模型本身的一部分」，落到架构上便是微轮次（micro-turn）：模型以约 200ms 为一段，持续地「读入 200ms、生成 200ms」，让静音、重叠、打断都作为连续流保留在上下文里，不再有人为的轮次边界需要迁就。

四、为什么这根轴值得单开一课

因为它是读懂整章的参考系。作者刻意给语音最多篇幅，理由是：在实时多模态这条演进线上，语音是走得最完整的一个——从「串行流水线延迟太高」这个问题出发，经过端到端、全双工、边想边说，一路走到相对成型的终局，问题→方案→终局全程跑通。后面的 Computer Use 和机器人两个场景，都可以对照语音这条脉络来看：它们各自走到了这条演进线的哪一段、卡在了哪里。

握住「摆脱轮次假设」这根轴，你还能顺手接住第二条主线的开头：GPT-Live 把「实时交互」与「深度思考」解耦——遇到需要搜索或复杂推理时，交互模型把任务委派给后台的前沿模型，自己继续维持对话。那条「快慢分工」正是下一课要展开的主题。

本课唯一要带走的判断：级联、Omni、全双工不是三种并列的技术，而是「摆脱 VAD 轮次假设」这一根轴上的三个刻度——判断依据从静音阈值→语义→无轮次。Omni「合三为一」动的是延迟与韵律，没碰「轮流」这个假设，所以照样会在你停顿时抢话；只有全双工取消了轮次本身。而三者是并存取舍，不是新旧替代。

五、关键定义

级联流水线（Cascaded Pipeline）
把 VAD、ASR、LLM、TTS 四个独立模型串成一条接力式流水线的语音架构，模块清晰可独立调优，但延迟逐段累积。避免与「自级联 self-cascade」混淆——后者是同一模型先转录再推理。
端到端全模态模型（Omni）
用单一模型直接听音频、想回复、说出来，把 ASR-LLM-TTS 三段合一，延迟低且能保留韵律情感，但仍假设轮流说话、依赖 VAD 划分轮次。避免：不等于全双工；OpenAI 称其为 turn-based，与工业界的 Omni 是同一类东西。
全双工 / 交互模型（Full-Duplex / Interaction Model）
模型同时持续地听和说，以约 200ms 微轮次交织多路流、每秒多次决策该说该听该停该打断，彻底取消轮次假设（Moshi、GPT-Live、TML-Interaction）。避免与 Omni 的「支持打断」混为一谈——那仍在 VAD 框架内做优化。
结束点检测（End-of-Speech Detection）
VAD 的关键设计，通常用 500-800ms 连续静音阈值判断用户是否说完，阈值太短截断句子、太长干等，两难。避免简称为「静音检测」而忽略它其实要区分轮内停顿与轮间间隔。

六、自测题

Q1：用户报电话号码时中途停顿了一秒，Omni 模型抢话插嘴了。为什么 Omni「把三个模型合成一个」也没能解决这个问题？哪种范式才能从根本上解决？

因为 Omni 只是把听/想/说合一，并未取消「轮流说话」假设，它仍靠 VAD 静音时长划分发言权，停顿就被判定为说完。流式语音感知能把判断升级到语义层缓解误判，但仍在轮次框架内；只有全双工（边听边说、每秒多次决策该说该听该停）才从根本消解「切换」本身。

Q2：作者说「摆脱 VAD」是一条叙事线。流式语音感知和全双工都在「摆脱 VAD」，两者的层次有什么本质区别？

流式语音感知是把轮次判断的依据从静音时长升级到语义层面（综合语义和声学判断一句话是否表达完整），但它仍在「轮次」框架内做优化——照样要判断「该谁说话」。全双工则取消了「轮次」这个框架本身：模型一直在听，「打断」不再是一个需要专门处理的事件，barge-in 处理链在架构上被省去大部分环节。前者是在轴上前移一格，后者是走到了轴的终点。

配套代码见开源仓库的 chapter9/live-audio（对应实验 9-1：构建传统语音 Agent）：前端 Next.js 捕获麦克风、后端 Node.js 通过 WebSocket 实时通信，后端跑 Silero VAD 检测约 500ms 连续静音后切分音频，ASR/LLM/TTS 三级均可切换多家提供商（Whisper/SenseVoice、GPT-4o/Gemini/豆包、Fish Audio），并做实时延迟监控。跑一遍，你会亲手感受到本课那根轴的「零刻度」——级联流水线的串行数据流，以及 VAD 静音等待带来的第一层延迟。仓库地址：github.com/bojieli/ai-agent-book。
