Agent 眼中的文件系统：一棵挂载四类区域的目录树

上一课解决了「多 Agent 何时值得用」，这一课解决「多 Agent 怎么把产物交给彼此」。本课唯一要拿下的东西：Agent 访问的不是单一存储，而是一棵虚拟目录树，上面挂着来源、生命周期、权限各异的四类区域——你要能默画出这棵树，并从可见性 / 生命周期 / 读写 / 并发控制四个维度把四类区分开。它出现在第 10 章「不共享上下文的显式协作」一节，是三种协作拓扑共同踩着的数据平面底座（对应图10-3与表10-3）。

一、是什么：一个统一接口，底下挂着四种完全不同的存储

书里把话说得很干脆：在实际系统中，Agent 访问的并非单一存储，而是一个虚拟文件系统（virtual filesystem）。来源、生命周期与权限各异的存储被挂载（mount）到同一目录树下，Agent 通过统一的 read_file / write_file / list_dir 接口访问，底层则可能是本地临时盘、持久对象存储、第三方云盘的 API，或只读的系统资源包。

  • ① Agent 专属工作区（Scratchpad）：每个 Agent 实例独享的私有目录，存放中间产物、临时文件、草稿与调试日志，生命周期与实例绑定，对其他 Agent 和用户不可见。
  • ② 多 Agent 共享空间（Shared Workspace）：多个 Agent 共同读写、且用户可见的协作区域，是不共享上下文架构下 Agent 间交换产物的主要媒介（Glossary Agent 写术语表，Translation Agent 读它）。生命周期与整个任务绑定，需持久化。
  • ③ 外部挂载资源（Mounted External Resources）：用户授权接入的第三方信息源——Google Drive、Notion、Dropbox、企业 Wiki——通过适配器（adapter）映射为挂载点（如 /mnt/gdrive），Agent 以读文件的方式访问，底层由适配器调对方 API。
  • ④ 系统内置资源（Built-in System Resources）：系统预置、对所有 Agent 只读共享的资源包，典型代表是 Skills——以文件形式组织、按渐进式披露取用，挂载于 /skills 等路径。

书里给了一张检查表，从可见性、生命周期、读写权限、并发控制四个维度对比四类区域。这张表就是本课的骨架，值得背下来——它可作为文件系统布局设计的检查表。

读这张表有个诀窍：并发控制的需要与否，是「可写」与「多方共享」两个条件的交集。① 私有所以不需要；④ 只读所以不需要；③ 的一致性由外部源负责；唯独 ② 既可写又被多方读写，才是并发冲突的高发处——乐观锁、工作副本隔离（worktree）等机制都作用在这一层。

三、为什么这样设计：文件路径作为通用接口

为什么要费劲把四种毫不相干的存储硬塞进同一棵目录树？因为这样做换来了一个极其廉价的协作原语——文件路径作为通用接口。

将四类区域统一至同一目录树，正是"文件路径作为通用接口"这一设计的价值所在：Agent 间传递产物、主 Agent 向子 Agent 交接输入、乃至跨组织 A2A 协作交换 Artifact，传递的均为轻量的路径字符串，而非将内容载入上下文窗口。 ——《深入理解 AI Agent》第 10 章

把这句话咬碎了看：如果没有统一接口，主 Agent 要把一份 50 页的译稿交给审校 Agent，就得把整份内容塞进对方的 context——既烧 token，又撑爆窗口。有了这棵树，交接的只是一个路径字符串 chapter1_zh.md，内容留在盘上，谁需要谁自己 read_file。这也是第 4 章「子 Agent 返回结构化摘要而非全量轨迹」在存储层面的体现，与第 5 章「文件系统作为 Agent 的中枢」一脉相承——那里讲单 Agent 如何以文件系统承载记忆与能力，这里把同一抽象扩展到多 Agent。

另一半设计动机是隔离。书里强调：明确每一区域的可见性与生命周期，是多 Agent 协作设计的前提，因为「相当一部分并发冲突与信息泄露，源于将本应隔离的区域混置」。四类分区不是分类学趣味，而是把「谁能看到什么、什么时候消失、谁能写」这三件事在存储布局上一次性钉死。

四、边界与易错

误区一：把试错草稿直接写进共享空间。初学者常觉得「都是 Agent 产出的文件，放一起方便」。错。书里点明隔离 scratchpad 有两重作用：避免多个 Agent 的临时文件相互覆盖，以及保持主 Agent 上下文的精简——子 Agent 的试错过程留存于自身工作区，仅将最终产物提交至共享空间。把草稿混进共享空间，既会让并发的临时文件互相踩踏，又会用一堆中间垃圾污染主 Agent 的上下文。

误区二：以为乐观锁能兜住一切冲突。乐观锁只能防同一文件的写入冲突。对于跨文件的语义冲突（如图片编号被一处重编、另一处仍引用旧编号），乐观锁无能为力——那需要更高层的语义校验：在编排层面避免有依赖关系的文件被并行修改，或写入后跑一遍全局一致性检查。

误区三：把外部挂载资源当本地盘用。统一的文件接口让 Agent 无需为每个数据源定制工具，但也掩盖了性能与安全差异：访问受外部权限约束、延迟更高且一致性更弱（每次读取是一次网络往返）、以按需只读为主（写回外部源可能污染用户真实数据）。所以要在挂载层面显式管理只读 / 可写、超时与凭证边界。

本节讲的是概念底座，没有独立配套代码目录；但共享空间这一层在书中有活生生的实现——第 4 章以卷挂载 /workspace/shared 连接主 Agent、虚拟电脑与虚拟手机，正是「② 多 Agent 共享空间」的典型落地。想边读边跑，整本书的实验都在开源仓库：github.com/bojieli/ai-agent-book。

五、关键定义

Agent 专属工作区（Scratchpad）
每个 Agent 实例独享的私有目录，存放中间产物、临时文件、草稿与调试日志；生命周期与实例绑定，对其他 Agent 和用户不可见。作用是隔离——避免临时文件互相覆盖、保持主 Agent 上下文精简。
多 Agent 共享空间（Shared Workspace）
多个 Agent 共同读写、且用户可见的协作区域，是不共享上下文架构下 Agent 间交换产物的主要媒介；生命周期与任务绑定、需持久化，是并发冲突的高发处。
乐观锁（Optimistic Locking）
共享文件系统的并发控制策略：每个文件维护版本号，Agent 读时记录版本、写时检查是否仍一致，不一致则写入失败并被迫基于最新版本重做。只能防同一文件的写入冲突，不解决跨文件语义冲突，也不同于「打开即锁定」的悲观锁。
文件路径作为通用接口
把四类存储统一挂载到同一目录树后，Agent 间交换产物、主 Agent 向子 Agent 交接、A2A 交换 Artifact，传递的均为轻量路径字符串而非载入上下文的内容。是多 Agent 协作的存储底座。

六、自测题

Q1：子 Agent 的试错草稿和调试日志应该放在四类区域的哪一类？为什么不直接写进共享空间？

放 Agent 专属工作区（Scratchpad），随实例销毁、对他人不可见。放进共享空间会让多个 Agent 的临时文件互相覆盖，还会污染主 Agent 上下文——正确做法是试错留在自身工作区，只把最终产物提交到共享空间。

Q2：四类区域里，唯独哪一类真正需要并发控制？用「可见性 + 读写」两个维度解释为什么另外三类不需要。

唯独多 Agent 共享空间需要（乐观锁 / worktree），因为它同时满足「多方可见」和「可读写」。另外三类各自缺一个条件：专属工作区是私有的（不多方共享），系统内置资源是只读的（不可写），外部挂载资源的一致性由外部源负责（并发控制不在本系统这一层）。并发控制的必要性 = 可写 ∩ 多方共享。

本课唯一要带走的判断：Agent 面对的是一棵虚拟目录树，上面挂着四类区域——专属工作区（私有、随实例销毁）、共享空间（多方可见、需持久化、要并发控制）、外部挂载资源（经 adapter、多为只读）、系统内置资源（全局只读、跨会话稳定）；统一到一棵树的价值，是让协作只需传路径字符串而非搬运内容。设计或排查多 Agent 存储时，先按这四格 + 四维度对号入座。
