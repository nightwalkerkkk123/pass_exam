七个核心工具：极简却完整的 Coding Agent 工具箱

这节课只拿一件事：一个能真正干活的 Coding Agent，底层只需要七个核心工具——它们从何而来、如何配合完成“整理所有 TODO”这类真实任务。这是第 5 章开篇“Coding 是 Agent 的基础能力”一节的骨架。记住它，你才能在后面的沙盒、安全、元能力等重头戏里，始终知道 Agent 手里到底握着哪几样东西。

一、七个工具从哪来：先看一个真实任务

作者不是凭空列一张工具清单，而是从一个典型任务倒推出来的：“整理仓库中所有遗留的 TODO 注释，按优先级分类并生成 issue”。完成这件事需要哪些动作？

浏览目录结构（ls/glob）、读取代码（read）、修改文件（edit/write）、运行命令（bash）、查找模式（grep/search）——这五类操作覆盖了几乎所有 Coding Agent 的核心动作。作者特意提醒：这五类操作自然对应六个工具；第七个 Code Interpreter 对应的是“执行代码/计算”这类操作，在有的实现里干脆与 Bash 合并。所以“七个工具”是一个规范化的参考集，不必与五类操作严格一一对应。

三、为什么这样切分，而不是更多或更少

你可能会问：为什么恰好是这七个？作者给出的判断链条值得深挖。

第一层，这七个工具主要覆盖了感知和执行两类操作。它不同于第四章按调用方向和作用性质划分的五类通用工具分类（感知 / 执行 / 协作 / 事件触发 / 用户沟通）——那么协作、事件触发、用户沟通这三类需求呢？作者的回答是：在 Coding Agent 中，这三类通常由 Agent 框架（而非工具层）处理。例如子 Agent 委托由框架的编排逻辑管理，而不是通过一个专用的“协作工具”。所以工具层能保持极简，是因为职责被有意地分层了。

第二层，也是更深的“为什么”：coding 能力的价值，不在于这七个固定工具本身，而在于它让 Agent 能在遇到新需求时动态扩展自己的能力边界。遇到数学推理，可以写段代码交给求解器算出精确答案；需要固化业务规则，代码比自然语言精确得多；缺少某个工具，可以临时写一个；数据格式变了，动态生成解析逻辑。正因如此，即使工具箱里只有这七个简单工具，一个具备基本 coding 能力的 Agent 也不会被工具数量卡死。

最容易犯的错，是把“七个核心工具”当成一张要死记硬背、且必须与操作类别严格一一对应的清单。作者早就打了预防针：五类操作自然对应六个工具，第七个 Code Interpreter 对应“执行代码/计算”，有的实现里干脆与 Bash 合并——七个工具是“规范化的参考集”，不是铁律。

更深一层的误读：以为一个 Coding Agent 之所以强，是因为“工具够多、够强”。恰恰相反——它强在 coding 是一种能在运行时动态创造新工具和能力的手段（第 0028 课会把这个“元能力”讲透）。盯着七个工具数数，你就错过了真正的重点。

四、关键定义

Code Interpreter（代码解释器）
提供隔离的沙盒环境（sandbox，与主系统隔离的安全运行空间，代码即使出错也不影响宿主机），安全执行 Python 代码。它对应“执行代码/计算”这类操作，是七个工具里区别于纯文件操作的那一个——统计、绘图、精确求解都靠它。
Glob（搜索文件名工具）
通过模式匹配快速定位文件系统中的目标文件，例如用 **/*.py 找出项目中所有 Python 文件。避免与 Grep 混淆：Glob 搜的是文件名。
Grep（搜索文件内容工具）
在文件内容中搜索特定的文本模式，例如搜索所有调用了某个函数的代码行。避免与 Glob 混淆：Grep 搜的是内容。
元能力（meta-capability）
一种“能创造其他能力”的能力：Agent 用它当场写出新工具、新约束、新表达形式来完成任务，而不必事先把所有能力都预制好。代码生成正是这样的元能力。避免等同于“一个很强的能力”或“万能工具”——关键在“创造能力的能力”，而非能力本身强弱。

五、自测题

Q1：用户要求“统计每个模块的 TODO 数量并画柱状图”，在七个核心工具中，除了 Grep 和 Write，还必须用到哪一个？为什么？

必须用 Code Interpreter（代码解释器）。因为统计计数和画柱状图属于“执行代码/计算”操作，需要在沙盒里跑 Python 完成，这正是它区别于纯文件操作工具的价值——Grep 只能找到 TODO 行、Write 只能落盘清单，都算不出数量、也画不了图。

Q2：七个核心工具覆盖了第四章五类通用工具中的哪两类？其余三类（协作 / 事件触发 / 用户沟通）在 Coding Agent 里由谁负责？

主要覆盖感知和执行两类。其余三类（协作、事件触发、用户沟通）在 Coding Agent 中通常由 Agent 框架（而非工具层）处理——例如子 Agent 委托由框架的编排逻辑管理，而不是通过一个专用的协作工具。这正是工具层能保持极简的原因：职责被有意分层了。

本课唯一要带走的判断：一个能干活的 Coding Agent，底层只需七个核心工具——代码执行、命令行、读/写/编辑文件、搜文件名、搜内容。它们从“整理 TODO”这类真实任务倒推而来，是规范化的参考集而非死记的铁律；真正让 Agent 不被工具数量卡死的，是 coding 这种能在运行时动态创造新能力的“元能力”。
