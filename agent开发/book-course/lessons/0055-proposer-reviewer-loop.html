提议者-审核者循环：为什么不能自己审自己

这节课只押一件事：Reviewer 的独立价值，不来自「换个视角再想一遍」，而来自它能接触到 Proposer 生成时根本拿不到的新信息——测试结果、渲染截图、编译错误、外部搜索结果。这是第十章「对等协作模式」里最经典的范式（图10-4），也是全章那条总判据（表10-2：协作是否引入生成时不存在的新信息）落到具体架构上的第一个样板。把这一课想透，你就有了判断一切「双 Agent 迭代」值不值得做的尺子。

一、是什么：最经典的对等协作范式

提议者-审核者（Proposer-Reviewer）是对等协作里最经典的一种。对等协作通常只涉及 2-3 个平等身份的 Agent，通过多轮迭代互相提供反馈——相比管理者模式和去中心化模式，它实现复杂度低得多，只需定义好两个角色、通信机制和迭代终止条件就能跑起来，是快速验证想法、构建原型的理想选择。

它的骨架很简单：Proposer Agent 负责生成，Reviewer Agent 评估质量、给出结构化改进建议，两者反复迭代直到效果达标。书中第五章已经在 PPT 生成、视频编辑、日志可视化三个实验里详细演示过它——Proposer 写代码，Reviewer 渲染执行结果并用 Vision LLM 打分。除了写代码，这套范式还适用于一大类审查场景：

  • 安全审查：Proposer 生成操作方案，Reviewer 检查合规性和潜在风险。
  • 内容审核：Proposer 起草回复，Reviewer 检查业务规则和用语规范。
  • 代码审核：Proposer 编写代码，Reviewer 检查安全性和最佳实践。

二、为什么这样设计：新信息，而不是新视角

范式的核心问题是一句反问：为什么不能让一个 Agent 自己生成、再自己审查？ 直觉上「重读一遍总能挑出毛病」，但书里用三篇研究把这条直觉按死了。

  • Huang et al., ICLR 2024《Large Language Models Cannot Self-Correct Reasoning Yet》：让 GPT-4 在没有外部反馈的情况下审查并修正自己的回答，准确率反而下降——模型把正确答案改错的次数，比把错误答案改对的次数还多。
  • TACL 2024 综述《When Can LLMs Actually Correct Their Own Mistakes?》(arXiv:2406.01297)：进一步确认，除非提供可靠的外部反馈（测试用例执行结果、外部工具验证输出），纯靠模型自身的「自我纠正」几乎不起作用。
  • ICLR 2024 CRITIC 论文：让模型用外部工具（搜索引擎、Python 解释器）验证自己的回答，效果显著提升；但一旦移除工具验证、只留模型的自我评估，大部分提升就消失了。

三篇指向同一个结论：审查的价值不在于「让模型再想一遍」，而在于引入了模型生成时不具备的新信息——测试结果、渲染截图、编译错误、外部搜索结果。回到第五章的 PPT 生成实验：Reviewer 的价值不是「用同一个模型再看一遍代码」，而是渲染了 PPT 并截取了屏幕截图——这张截图里的视觉信息，Proposer 在写代码时完全无法获得。同理，代码生成里执行测试用例产生的通过/失败结果，也是编写代码时并不存在的新信号。Reviewer 的独立价值，正来源于它能接触到 Proposer 拿不到的这些外部反馈。

三、边界与易错：别把它当「多 Agent 天然更强」的证据

提议者-审核者好用，但它的有效性有严格前提。最常见的误解，是把「用了两个 Agent」本身当成变强的理由。

误区一：以为「多 Agent 天然更强」。 全章反复强调，真正的判据只有一条——协作是否引入了生成时不存在的新信息。等算力预算下，辩论式的多 Agent 并不优于单 Agent：2026 年 Tran 与 Kiela 的研究在多跳推理任务上发现，当思考 token 预算被严格控制为相同时，单 Agent 的表现与多 Agent 持平甚至更好。信息论里的数据处理不等式给了解释：辩论中多个 Agent 处理的是完全相同的文本，Agent 之间每串行传递一次中间结论，只可能丢信息、不可能凭空造信息。

误区二：忽视成本。 Anthropic 曾披露，其多 Agent 研究系统的 token 消耗约为普通对话的 15 倍，且 token 用量本身就能解释约 80% 的性能差异。所以多 Agent 的收益必须大到能覆盖数倍乃至一个数量级的额外开销，否则一个调校得当的单 Agent 往往更划算。

误区三：混淆两种「能力配置」论断。 「审查者能力应与被审者相近」说的是审查场景——审查者必须跟得上被审者的推理才能发现破绽，落差太大就根本审不动；而「最强模型给规划者」说的是另一件事——规划与执行的分工（下一课的主题）。二者语境不同，不矛盾。

需要划清 Tran-Kiela 论证的边界：它针对的是「多 Agent 串行传递中间结论」造成的信息瓶颈，并不否定另外两类做法——对同一问题多次独立采样再聚合（如 self-consistency、多数投票），或利用生成与验证的难度不对称（写答案难、验答案易）来做生成-验证分工。这两类要么引入了额外的独立采样、要么利用了任务本身的不对称结构，都不落在数据处理不等式的适用范围内。提议者-审核者恰好属于「有效」的一侧——因为 Reviewer 那条外部工具支路真的注入了新信息。

本课唯一要带走的判断：判断一个「提议-审核」循环值不值得做，只问一个问题——Reviewer 是否借外部工具拿到了 Proposer 生成时不存在的新信息（测试结果、渲染截图、搜索输出）？ 拿到了，它就是实质性的增益；没拿到，它只是让同一个模型再想一遍，通常无效甚至有害，还白烧十几倍的 token。

四、关键定义

共享上下文 / 不共享上下文（Shared / Non-shared Context）
多 Agent 的第一设计维度：共享指后一 Agent 继承前一 Agent 的完整轨迹（隐式协调、信息零损耗、上下文膨胀）；不共享指各 Agent 维护独立上下文，仅通过工具参数/共享文件/消息总线三种显式机制通信。提议者-审核者是「不共享上下文」下的对等协作形态。避免与「多轮对话历史」混同——这里区分的是 Agent 之间是否继承彼此的完整轨迹。
新信息判据
本章总判据：多 Agent 相对单 Agent 是否有实质价值，只看协作过程是否引入了单个 Agent 在生成时无法获得的新信息（执行反馈、视觉反馈、工具反馈）。对应表10-2。
数据处理不等式（Data Processing Inequality）
信息论结论，本章用它解释辩论式多 Agent 的局限：Agent 之间每串行传递一次中间结论，只可能丢失信息、不可能凭空创造信息。因此在等算力预算下，处理同一段文本的辩论并不优于单 Agent。

五、自测题

Q1：CRITIC 实验中，实验者移除工具验证步骤、只保留模型的自我评估后，大部分提升消失了。这说明审查的价值来自哪里？

说明审查的价值不在于「让模型再想一遍」，而在于引入了模型生成时不具备的新信息——搜索引擎结果、Python 解释器的执行结果等外部工具反馈。移除工具后没有新信息注入，提升自然消失。

Q2：都是「两个 Agent 来回讨论」，为什么提议者-审核者（Reviewer 渲染截图打分）能显著提升质量，而辩论模式在等算力下却和单 Agent 持平？

差别在于是否引入了生成时不存在的新信息。提议者-审核者里，Reviewer 借外部工具（渲染 PPT 截图、执行测试）拿到了 Proposer 写代码时完全无法获得的视觉/执行反馈，是真正的新信号；而辩论模式里两个 Agent 处理的是完全相同的文本，没有外部反馈注入。按数据处理不等式，串行传递中间结论只会丢信息、不会造信息，所以等算力下辩论并不优于单 Agent。判据始终是那一条：协作是否引入了新信息。

本课的三个落地实验都在第五章（PPT 生成、视频编辑、日志可视化），Proposer 写代码、Reviewer 渲染截图打分的完整代码可在开源仓库对照阅读：github.com/bojieli/ai-agent-book。读的时候盯住一件事——Reviewer 到底通过哪个工具拿到了 Proposer 拿不到的信息。
