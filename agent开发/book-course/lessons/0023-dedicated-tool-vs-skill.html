专用工具还是 Skill？先于粒度的第一性问题

后面几节会教你怎么把工具的粒度切好、怎么把描述写清——但它们都偷偷预设了一件事：这个能力已经决定要做成一个专用工具。本课要拿下的，就是把这个预设翻上台面：在讨论粒度和描述之前，先问一个更基本的问题——Agent 的能力究竟该以什么形态来表达？作者给了两种基本形态和一把三维标尺。这一节在第 4 章开头“能力表达形式的选择：专用工具还是 Skill + 通用执行器”一节。

一、两种基本形态：结构化的手，还是一张会读的说明书

作者把话说在最前面：“在讨论具体的工具类型之前，首先需要回答一个更基本的设计问题：Agent 的能力应该以什么形式来表达？”——他明确指出，后续关于粒度、通用性、描述艺术的讨论，“都建立在‘应该做成专用工具’这一假设之上”。也就是说，本课要拆的，正是那条一直没被说破的地基。

二、三把标尺：凭什么在两者之间选

“选择哪种形态取决于三个维度。”作者只给了三条，不多不少：参数复杂度、变更频率、模型能力。关键在于，这三条不是并列的口诀，而是三根可以让你把任意一个能力“定位”到 Skill 侧还是专用工具侧的坐标轴。

逐条把作者的判据说清——注意每一条都是“什么时候该往专用工具那边靠”：

  • 参数复杂度：涉及嵌套对象、多字段联合校验、复杂类型约束的操作，专用工具的结构化 schema 能更好地引导模型正确传参；而参数简单的操作，通过 CLI 命令传参“同样可靠”——简单参数上 schema 是白费力气。
  • 变更频率：频繁变化的能力用 Skill 来维护，成本远低于专用工具——“改一段文本远比改代码、测试、部署要轻松得多”；而稳定的底层操作更适合固化成专用工具。
  • 模型能力：SOTA 模型可以用 Skill + 通用执行器表达更多能力、减少工具数量；较弱的模型则需要结构化的工具 schema 来引导正确调用。（作者预告：第八章会讲 Agent 在自我进化中沉淀新能力时，如何做同样的选择。）

三、为什么这是“第一性问题”——它决定了后面所有讨论的地基

把这一节放在第 4 章开头，不是排版随意。作者点破：粒度权衡、通用性设计、描述艺术这些后续话题，全都默认“这个能力已经是一个专用工具”。如果你没先回答形态问题，就一头扎进“这个工具该切多细、描述该怎么写”，可能整段功夫都花在了一个本就不该存在的工具上——它更该是三行 Skill 文档。

这条框架的生命力还在于它跨越 MCP 与 Skills 两套体系仍然成立。作者在本章后文再次确认：一项具体能力究竟做成专用 MCP 工具，还是 Skill + 通用执行器，“本章开头‘能力表达形式的选择’一节给出的三维决策框架（参数复杂度、变更频率、模型能力）仍然适用”。换句话说，这三把尺子不随你用哪种协议而失效。

误区：默认“能力就该做成专用工具，越结构化越可靠”。初学者顺着后面几节的语气，很容易把“做成专用工具”当成不言自明的起点，于是每加一个能力就写一个函数、配一套 schema。错在两点：其一，作者明说参数简单时 CLI 传参“同样可靠”，硬套 schema 并不会更稳；其二，专用工具是有代价的——“每个工具会占据数百个 token，且数量膨胀会破坏 KV Cache”。工具越堆越多，不是能力越强，而是上下文越贵、模型选择越容易出错。真正该先问的，是这三根轴把这件能力指向了哪边。

本课唯一要带走的判断：拿到一项新能力，别默认写成专用工具。先用三把尺子量一遍——参数复杂度、变更频率、模型能力：参数简单、变更频繁、模型够强，就交给 Skill + 通用执行器；参数结构复杂、要强约束防呆、且稳定，才值得做成专用工具。工具不是越多越可靠，每个都在吃 token、挤 KV Cache、增加选错的概率。这套判断跨 MCP 与 Skills 两套体系都成立。

四、关键定义

专用代码工具
把一项能力实现为结构化的函数调用：确定性高、可测试。代价是每个工具会占据数百个 token，且数量膨胀会破坏 KV Cache。参数复杂、底层稳定、或模型较弱时更倾向它。
Skill + 通用执行器
用自然语言编写的 Skill 文档描述操作流程，Agent 通过终端或代码解释器执行，只需少量通用工具就能覆盖大量场景（第五章论证的七个核心工具）。参数简单、变更频繁、或模型够强时更倾向它。
ACI（Agent-Computer Interface）
对标 HCI（人机交互）提出：研究 Agent 如何与计算机交互，核心是让工具对 Agent 而非对人友好——工具应对应 Agent 的目标而非底层 API 操作。避免把它等同于“API”或“函数调用规范”；ACI 是设计理念层面的，不是某种具体协议。
MCP（Model Context Protocol）
Anthropic 于 2024 年底发布的开放标准，用客户端-服务器架构统一 AI 模型与外部工具、数据源的通信。避免说成“Anthropic 的私有 API”——它是被多家框架/IDE 采纳的开放标准；无论走 MCP 还是 Skills，本课的三维框架都仍适用。

五、自测题

Q1：一个每两周就改一次流程、参数只是几个字符串的“部署应用”能力，该做成专用工具还是 Skill 文档？为什么？

做成 Skill 文档 + 通用执行器（bash）。因为变更频率高——改一段文本远比改代码、测试、部署轻松得多；且参数简单，CLI 传参同样可靠，不需要结构化 schema 引导。只有当参数嵌套复杂、底层稳定、或模型较弱时，才倾向做成专用工具。

Q2：有人主张“工具当然是越结构化越好，能力都做成专用工具最稳”。按本节，这句话忽略了专用工具的哪两项代价？

忽略了作者明说的两点代价：每个专用工具会占据数百个 token，以及工具数量膨胀会破坏 KV Cache。此外参数简单时 CLI 传参“同样可靠”，多此一举的 schema 并不会带来额外可靠性。所以“越多越结构化”常常换来的是上下文更贵、模型在众多工具间更容易选错。

想亲手摸一摸“通用执行器”这一侧长什么样，去看实验 4-2 的执行工具 MCP 服务器（chapter4/execution-tools/）：里面的沙盒 code_interpreter 与 virtual_terminal 正是“少量通用工具 + 让 Agent 自己写命令/代码来执行”的落地——对照本课，体会为什么强模型加一个通用执行器，往往胜过十几个专用工具。完整代码见 github.com/bojieli/ai-agent-book。
