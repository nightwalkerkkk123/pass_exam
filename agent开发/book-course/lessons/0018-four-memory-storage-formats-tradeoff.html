四种记忆存储格式：简单性与表达力的根本张力

上一课解决了“会话结束后该记住什么”，这一课解决“记下来的东西怎么存”。作者在“用户记忆的四种存储格式”一节里，沿 Simple Notes → Enhanced Notes → JSON Cards → Advanced JSON Cards 摆了一条递进的阶梯（对应图3-2）。本课唯一要拿下的，是看穿这条阶梯背后那条贯穿始终的张力——简单性 vs 表达力：每往上走一格，都在用某种代价换某种能力，没有免费的格式。

一、四种格式：一条“越来越会记事”的阶梯

同一条用户信息，可以用不同的粒度和结构来表示。作者给的四种，是按记忆粒度和结构复杂度递进排列的——不是四个并列选项，而是一步步“加东西”：

  • Simple Notes：每条记忆是一个最小的、不可再分的事实（如“用户邮箱：john@example.com”）。开销极低，是 O(1) 操作（耗时固定、不随数据量增长）。代价是信息关联性完全丢失——“在 TechCorp 担任高级工程师，负责推荐系统”会被拆成三条互不相关的原子事实，内在联系被割裂。
  • Enhanced Notes：把每条记忆存成一段包含完整上下文的自然语言段落，保留叙事结构与语义完整性，适合“基于我的背景推荐新项目”这类需要细微推断的场景。
  • JSON Cards：采用三层嵌套结构（类别→子类别→键值对，如 work.position.title），模拟人类分类认知，支持部分更新（改职位不动公司名），可预测、可扩展。
  • Advanced JSON Cards：在 JSON Cards 之上，每张卡片除事实外，再加 backstory（获取上下文）、person（主体身份）、relationship（与用户关系）和时间戳。作者称它是“从信息存储到知识管理的范式转变”。

二、为什么必然有这条张力（深挖“为什么”）

作者把四种格式并到一起，是为了逼出一句结论：这里没有绝对优劣，只有简单性与表达力之间的根本权衡。原文把每种格式“换到了什么、又赔了什么”一句话点破：

Simple Notes 选择了极致简单，牺牲语义完整性；Enhanced Notes 选择叙事完整性，牺牲结构化和可更新性；JSON Cards 选择了结构化，牺牲灵活性；Advanced JSON Cards 选择全面性，牺牲简单性。 ——《深入理解 AI Agent》第 3 章

为什么这不是“工程师偷懒没做出完美格式”，而是结构性的？因为这几个目标本身互相拉扯：要“原子化、可 O(1) 存取”，就必须把一段叙事拆碎，关联随之丢失；要“保留完整叙事”，就没法再精确定位、部分更新某一个字段；要“强结构、可部分更新”，就得预先假设信息能被清晰地归入某一类，而现实里“周末用 Python 开发个人项目”同时是时间偏好、技术偏好和活动类型，硬塞进单一类别就丢了多维性。你按下一个目标，另一个目标就翘起来——这才是它被称为“根本张力”的原因。

正因为没有银弹，作者给出的不是“选一个最好的”，而是混合选型标准：

关键且少量 → Advanced JSON Cards | 大量且非关键 → Simple Notes
关键人物关系、核心偏好用 Advanced JSON Cards 保证可检索与消歧；海量普通对话事实用 Simple Notes 压成本。多数生产系统采用混合模式，同一 Agent 内不同类信息走不同路径。

三、Advanced JSON Cards 凭什么能消歧

阶梯最顶端那一格，值得单独拆开看，因为它解决的是一个前三种都解决不了的真实痛点：同名实体消歧。作者的例子——用户有两位“张医生”：一位是自己的牙科医生，一位是父亲的心脏科医生。脱离具体情境，这条信息根本无法被正确理解。

Advanced JSON Cards 的解法是给每条事实挂上三个额外维度：backstory 记录“为什么”会存这条信息（获取上下文），person 与 relationship 建立“为谁”存的实体模型。于是两位“张医生”不再是两条会打架的键值，而是两张各自自洽的卡片：

误区一：把“怎么存”和“放哪里 / 存什么”混成一锅。本章其实同时引入了三套正交的分类体系，表3-1 专门把它们一次厘清：记忆层次（放哪里）、存储格式（怎么存，即本课的四种）、认知类型（存什么，情景/语义/程序）。初学者常把它们当成一回事，其实它们可以自由组合——一条“靠窗座位偏好”的语义记忆可以用 Simple Notes 存，一段“先搜直飞→确认座位”的程序记忆也可以用 Advanced JSON Cards 存。选哪种格式看工程需求（简单性 vs 表达力），存哪种类型看业务场景。

误区二：以为 Advanced JSON Cards 做的消歧和“一词多义”是同一件事。它做的是实体消歧（区分现实中两个同名的张医生，需要维护关于实体本身的知识），不是词义消歧（判断“bank”指河岸还是银行，靠上下文感知的嵌入就能解决）。后面 GraphRAG 一节会让这种实体消歧变成图结构的原生能力，这里先把两者的边界记牢。

本课唯一要带走的判断：四种存储格式是同一条“简单性↔表达力”轴上的四个刻度，每往上一格都要赔一样东西（关联 / 可更新性 / 灵活性 / 简单性）。所以正确的问题从来不是“哪种最好”，而是“这条信息关键且少量吗”——是就用 Advanced JSON Cards 换消歧与可检索，不是就用 Simple Notes 换成本，其余走混合。

四、关键定义

Advanced JSON Cards
最高表达力的记忆存储格式，每张卡片除事实外还含 backstory（获取上下文）、person（主体身份）、relationship（与用户关系）和时间戳，实现同名实体消歧，代表从信息存储到知识管理的范式转变。避免与普通 JSON Cards 混淆——后者只是三层嵌套分类键值对，不含消歧字段。
用户记忆（User Memory）
针对单个用户 ID 的个性化持久记忆，跨会话提炼出的稳定信息，会被反复改写、合并、淘汰；本质是投入额外算力构建关于用户的简洁预测模型。避免等同于“对话历史”或“轨迹”。
轨迹（Trajectory）
一次 Agent 运行的完整历史记录（用户消息+模型回复+工具结果），按时间顺序 append-only 追加、只增不改，为当前决策提供即时上下文。避免与用户记忆混用——用户记忆是跨会话提炼、可改写的档案，轨迹是单次会话不可变的流水账。

五、自测题

Q1：用户有两位“张医生”（自己的牙医、父亲的心脏科医生）。哪种存储格式能可靠区分？靠什么字段？为什么 Simple Notes 做不到？

Advanced JSON Cards。靠 person（主体身份）与 relationship（与用户关系）建立实体模型、backstory 提供获取上下文来消歧。Simple Notes 是原子键值事实、信息关联被割裂，无法表达“这个张医生是为谁存的”，也就无从区分两位同名医生。

Q2：一句话说清 JSON Cards“牺牲了什么”，以及为什么这个牺牲是结构性的、躲不掉。

JSON Cards 选择了结构化（可部分更新、可预测、可扩展），牺牲的是灵活性。它的刚性三层结构假设信息能被清晰归入单一类别；可现实里像“周末用 Python 开发个人项目”这样的信息同时涉及时间偏好、技术偏好和活动类型，强制归入一类就丢了多维性。要“强结构”就得先“预设可分类”，二者绑定，所以这个牺牲躲不掉——这正是简单性与表达力根本张力的一个切面。

配套项目 chapter3/user-memory/ 在统一接口下实现了上述四种记忆模式的记忆生成与检索，运行时改配置即可切换。建议在实验 3-1 的三层次评估集上逐一跑一遍，亲手对比同一组测试会话在四种格式下提取出的记忆形态与最终得分：你会看到 Simple Notes 以最低成本通过第一层“基础回忆”，却在需要综合多条信息、区分同名实体的第二、三层频繁失分，而 Advanced JSON Cards 在消歧和跨会话关联上表现最好、代价是记忆维护调用明显更贵更慢。差异在具体例子面前一目了然。完整代码见开源仓库 github.com/bojieli/ai-agent-book。
