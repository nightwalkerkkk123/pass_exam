实践在前，命名在后：别等名词流行才动手

这节课只押一件事：Skill、harness、loop engineering 这些名词，不是先被发明、再被采用的；而是大量 Agent 早就在这么做，头部公司事后才把它们提炼、命名成架构原则。这就是作者在引言里下的判断——“实践在前，命名在后”。搞清这个顺序，你才能听懂它对企业级开发那句很实际的告诫：等名词流行了再动手，你已经慢了一步。

一、顺序被搞反了：名词是实践的影子

作者在引言里回头审视一年前总结的那些 Agent 架构原则，有个发现让他“既欣慰又惊讶”：这些原则不但没过时，反而越来越经典。可业界后来陆续冒出了 Skill、harness、loop engineering 等新名词——很容易让人以为是这些概念先被造出来，Agent 才跟着用。

作者把真实的因果顺序讲得很直白：“真实的顺序恰恰是反的。” 不是 Anthropic 这些公司先发明了概念、众多 Agent 才跟着用；相反，是大量 Agent 早就在这么做了，Anthropic 才把它们提炼、总结成了架构设计原则。一句话——实践在前，命名在后。

三、为什么这个顺序对企业开发是致命的

作者点破了“实践在前，命名在后”对企业级 Agent 开发的实际含义：如果你每次都要等到业界开始流行某个 Agent 名词才去实践，就已经慢了一步。因为名词流行的时候，头部公司往往早已把对应的问题趟过一遍了。

那么，怎样才能赶在名词流行之前就知道该怎么做？作者认为最关键的有两点——这也是本课要你记死的核心：

抢在名词前 = 极高要求的真实业务（持续反馈） + 评估机制
缺一不可：没有真实业务，你没有动力打磨；没有评估，你分不清是真进步还是运气

第一，拥有一个对 Agent 能力上限有极高要求的真实业务，并能持续获得真实的业务反馈。以 Pine 为例：处理一件事往往耗时数小时甚至数周，其间要打好几个小时电话、在电脑上填好几页复杂表单、来回发数封邮件，全程既不能在任何数字上出错，又要谨慎维护用户利益。只有置身这样足够复杂的场景，实践才会自然地把你倒逼着去构建 harness、去解决那些模型当下做不到、业务上却必须完成的事。反过来，业务对能力上限要求不高、模型稍一升级就够用，你就没有动力去打磨这些架构原则。

第二，必须建立评估（Evaluation）机制。作者反复强调：没有评估，就没有进步。评估让你能分辨一次改动究竟是真的变好了，还是只是运气，从而让迭代方向不再依赖直觉——这是“用科学方法论做工程”的地基。（这套方法第六章会专门展开。）

四、这个顺序，凭什么“穿越模型周期”

为什么值得为“顺序”单开一课？因为它支撑着全书的底层信念：好的设计原则本就应该穿越模型的迭代周期。作者的理由很硬——这些原则描述的不是某个模型的用法，而是智能系统与世界交互的基本模式。GPT、Claude 半年一迭代，产品形态不断翻新，但几乎所有成功的 Agent 系统都遵循相同的架构模式，这不是巧合。

作者还借强化学习之父 Richard Sutton 把这件事抬到宇宙尺度：宇宙演化经历尘埃→恒星→生命→智能体四个阶段。生物进化是盲目的（随机变异、自然选择，多数生物并不理解自己怎么运作）；而 Agent 是全新的存在——它能通过生成代码实现自举（bootstrap）与自我进化，“就像一个程序员编写了另一个程序员，新程序员又能继续编写下一个”。既然 Agent 能理解并改进自身，那么掌握“为什么这样设计”的原则，就比追逐任何一个热名词都更持久。

最容易搞反的，就是因果顺序：以为“Anthropic 先发明了 Skill / harness / loop engineering，Agent 圈才跟着用”。按这个错误图景，你会本能地等名词流行、等最佳实践成文再动手——而作者恰恰说这已经慢了一步。真实顺序是反的：大量 Agent 早就在做，名词是事后的提炼。名词是果，不是因；把它当成起跑信号，你永远在追影子。

另一处顺带的坑：本书对“思考（reasoning）”与“推理（inference）”做了刻意区分——reasoning 指模型“想”的中间推导过程（思维链、思考模型、思考 token），inference 指模型的前向计算与部署运行（推理时、推理成本、推理栈）。别按日常习惯把两者都读成“推理”而错失作者的区分；仅“逻辑推理、多跳推理、空间推理、时序推理”等固化词例外保留“推理”二字。

本课唯一要带走的判断：名词流行是实践的影子而非源头——实践在前，命名在后。所以别等名词火了再动手；抢在前面的唯一办法，是握住一个对能力上限有极高要求的真实业务，并为它建立评估机制。真实业务给你压力，评估给你方向，二者缺一不可。

五、关键定义

提议者-审核者（proposer-reviewer）
让 Agent 审阅自己输出的交付件（artifact）并迭代改进的方法，用于解决模型过早认为任务完成的问题。避免把它等同于泛泛的“自我反思”。
思考（reasoning）
本书统一译法，指模型展开中间推导、“想”的过程，包括思维链（Chain-of-Thought）、思考模型、思考 token、思考过程。避免译作“推理”。
推理（inference）
本书统一译法，指模型的前向计算与部署运行，如推理时、推理成本、推理栈、推理时扩展。避免与 reasoning“思考”混用。
自举（bootstrap）
Agent 通过生成代码实现自我进化，就像一个程序员编写了另一个程序员，新程序员又能继续编写下一个。避免与编译器 bootstrap 的狭义含义混淆。

六、自测题

Q1：作者说要赶在名词流行前就知道怎么做，关键的两点是什么？

第一，拥有一个对 Agent 能力上限有极高要求的真实业务，并持续获得真实业务反馈（如 Pine 处理涉及金钱的长程任务，逼你去构建 harness）；第二，必须建立评估（Evaluation）机制——没有评估就没有进步，它让你分辨一次改动是真变好还是只是运气。

Q2：“实践在前，命名在后”对企业级 Agent 开发的实际含义是什么？为什么等名词流行才动手就已经慢了？

含义是：如果你每次都要等到业界流行某个 Agent 名词才去实践，就已经慢了一步。因为真实顺序是——大量 Agent 早就在这么做，头部公司事后才把它提炼成概念、命名。所以名词流行的时候，头部公司往往早已把对应问题趟过一遍了；你若把名词当起跑信号，等于在追已经跑完的人的影子。

本课属引言，尚无独立配套代码；但作者强烈建议边读边跑书中实验（“很多设计上的直觉需要在动手调试的过程中才能真正建立起来”）。全部实验组织在配套开源仓库，项目名与书中“实验 X-Y”一一对应：github.com/bojieli/ai-agent-book。这正是“实践在前”的读法——先动手，名词自然长在你手上。
