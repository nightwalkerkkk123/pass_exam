数据和环境比算法更重要：后训练最值钱的一条判断

这节课只押一个反直觉的判断：决定后训练成败的，不是你调 PPO 还是 GRPO，而是仿真环境的保真度和训练数据的质量。它出自第 7 章「数据与环境：比算法更重要的事」——作者亲口说这是「全章我最想让你记住的一节」。如果这门课你只带走一条工程直觉，就是这一条。

一、优先级被搞反了：先验 > 环境 > 算法

初学者接触 RL，第一反应几乎都是「我该用哪个算法」。但工业界一线的经验恰恰相反。作者把话说得很硬：算法的重要性，远不及三个更基础的要素——仿真环境的保真度、训练数据的质量、基础模型的能力。现成算法（PPO、GRPO 等）早已封装进 veRL、TRL 这类框架，你会用、能选对就够了；真正拉开差距的，是环境和数据做得好不好。

这不是一句口号，而是有历史依据的。本章 7.2 节记录了 OpenAI 的「认知反转」（图7-6）：几十年 RL 研究者一直默认的优先级是算法 > 环境 > 先验——第一阶段（2015–2016）算法中心主义，换个环境就得从头训；第二阶段（2016–2018）转向环境的重要性，试图把整个互联网变成训练场；直到第三阶段（2018 至今）「先验的觉醒」，GPT 系列证明语言预训练积累的先验知识可以通过与 RL 完全无关的方式获得。于是真实的排序被彻底颠倒过来：

二、环境：为什么真实 API 不能直接拿来训练

RL 的本质是「试错学习」，而试错必须有个试错的场地——这就是仿真环境（simulation environment）。模型在环境里一遍遍跑任务、拿反馈、调策略。环境的保真度（跟真实部署场景有多像）直接决定训出来的策略能不能用。这里有两条作者反复强调的判断：

  • 环境失真，策略必废。如果仿真里的客服总按固定套路回话、错误信息跟生产环境对不上，模型就会学到一套只在仿真里管用的「应试策略」，一上线就露馅。这是 RL 项目最常见的翻车方式——不是算法不行，是练习场跟考场不是一回事。
  • 构建高保真环境，常常比训练本身更贵、更难。一个能大规模并行、可复现、反馈真实的环境，往往要投入比调模型多得多的工程。

那为什么不干脆连上真实的搜索、浏览器、打电话 API 来训练？因为真实 API 有速率限制、会封号、有副作用，根本没法直接拿来训练。RL 要成千上万次可复现地反复试错，真实世界既扛不住这个并发，也不允许你把不可逆的副作用重放一万遍。所以你必须先花大力气造一个稳定可控可重放的「影子世界」——本章后面的 AWorld MCP 沙盒、ReTool 代码解释器沙盒，之所以花那么大力气搭环境，正是这个原因。

三、数据：质量胜过算法，SFT 到位甚至不必上 RL

如果说环境是场地，数据就是教材，而且是三要素里最关键的一环。SFT 阶段的数据指示范样本（输入—输出对），RL 阶段指任务分布和奖励信号。无论哪个阶段，一条铁律贯穿始终：数据质量胜过算法。再精巧的算法，喂进去脏数据、覆盖不全的数据、有系统性偏差的数据，学出来的也只能是脏策略——SFT 会一字不差把噪声和偏见固化进参数，RL 则会朝着有偏差的奖励拼命优化（这正是 reward hacking 的温床）。Garbage in, garbage out，在后训练里体现得淋漓尽致。

由此引出一个很多团队没想通、却极其省钱的判断：很多场景下，只要 SFT 的数据质量到位，你根本不需要做 RL。RL 又贵又不稳定，常是 SFT 的几十到上百倍成本，大家却往往一上来就想上 RL。但只要任务分布可预期、能拿到足够多样且高质量的示范数据，一个扎实的 SFT 往往就够用。作者给的行业例证是 Anthropic：2025 年之前它的后训练配方主要是「海量高质量数据做 SFT + RLAIF」，并不怎么依赖如今做代码、推理已成标配的 RLVR，可它当时的 Coding 模型质量就已经非常出色——靠的不是算法花哨，而是把数据质量做到了极致。

数据决定你能到哪，RL 决定你还能再高多少。 ——《深入理解 AI Agent》第 7 章「数据与环境」

那「数据质量」具体指什么？作者给了三个维度：覆盖面（是否覆盖到部署时会遇到的长尾与边界情况）、多样性（示范里的说话者、风格、解法够不够丰富，否则模型会塌缩到单一模式）、标注准确性（示范答案本身对不对，尤其思维链蒸馏里，错误的思考过程会被学生一并模仿）。这三点的投入产出比，通常远高于换一个更花哨的算法。

初学者最容易犯的错，是把全部注意力放在「调哪个算法」上。翻开论文看到 PPO、GRPO、DPO 一堆名字，就以为选对算法是胜负手，于是一上来就纠结超参、纠结要不要上最新的 RL 变体——却对「我的仿真环境像不像真实世界」「我的示范数据覆盖全不全」这两个真正决定成败的问题视而不见。

为什么这样想会错？因为现成算法早被 veRL、TRL 封装好，调用通常只改几行配置；而环境失真会让策略「练习场管用、考场露馅」，脏数据会被 SFT 一字不差固化进参数、被 RL 越优化越偏。算法是塔尖那一小块，环境和数据才是地基。把顺序摆正：先把环境和 SFT 数据做扎实，再判断到底需不需要 RL。

本课对应的两个「花大力气搭环境」的实战，都在开源仓库里可跑：

  • 实验 7-16（chapter7/AWorld-train/）：在 MCP 服务器沙盒（26 服务器 / 126 工具函数）上用 Qwen3-4B 跑通 GAIA 多工具「从实践中学习」——正因为真实 API 的速率限制、服务波动、账号封禁使直接在生产环境训练不可行，才必须先构建这个可控可重放的沙盒。
  • 实验 7-15（chapter7/retool/）：SFT 预热 + 基于 veRL 改造的 PPO，把代码解释器沙盒整合进 RL 思考循环，对 <interpreter> 反馈 token 做损失屏蔽，AIME2024 从 25% 提升到 67%。

四、关键定义

仿真环境（simulation environment）
RL 试错学习的「场地」，模型在其中反复跑任务、拿反馈、调策略。核心指标是保真度——跟真实部署场景有多像，它直接决定训出的策略能否上线。避免把它当成可有可无的配套，它常常比训练本身更贵、更难。
监督微调（SFT / Supervised Fine-Tuning）
用几千到几万条「问题—标准回答」示范对继续做「预测下一个词」、且只在回答 token 上算损失的后训练方法，把指令遵循、格式、风格、流程等协议性知识固化进参数。避免泛称为「微调」或等同于「训练」——它特指有标注输入-输出对、带 loss masking 的方法。
可验证奖励强化学习（RLVR）
奖励来自规则验证器（测试是否通过、答案是否正确）而非学习到的奖励模型的 RL；此时「奖励模型」退化成一段确定性代码。如今做代码、推理已成标配，但 Anthropic 早期恰恰不怎么依赖它也训出了顶尖模型。避免与 RLHF 混淆。

五、自测题

Q1：为什么真实 API（搜索、打电话、浏览器）几乎不能直接拿来做 RL 训练？这跟「环境比算法重要」有什么关系？

因为真实 API 有速率限制、会封号、有不可逆副作用，无法大规模并行、可复现地反复试错。所以必须先花大力气造一个稳定可控可重放的高保真「影子世界」（如 AWorld 的 MCP 沙盒、ReTool 的代码沙盒）——环境保真度直接决定策略能不能上线，这份工程投入往往比调算法回报高得多。

Q2：按本章的判断，什么情况下你「甚至不需要做 RL」？作者举了谁作为例证？

当任务分布可预期、能拿到足够多样且足够高质量的示范数据时，一个扎实的 SFT 往往就够用——RL 又贵又不稳定（常是 SFT 的几十到上百倍成本），不必一上来就上。例证是 Anthropic：2025 年之前主要靠「海量高质量数据做 SFT + RLAIF」，并不怎么依赖 RLVR，Coding 模型质量却已非常出色。一句话概括：数据决定你能到哪，RL 决定你还能再高多少。

本课唯一要带走的判断：后训练拉开差距的从来不是算法，而是仿真环境的保真度和训练数据的质量。动手前先问两句——我的环境像不像真实世界？我的 SFT 数据覆盖全、够多样、标注准吗？把这两件事做扎实，很多场景连 RL 都不用上。真实优先级是：先验 > 环境 > 算法。
