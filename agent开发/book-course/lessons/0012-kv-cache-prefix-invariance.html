改一个空格，账单翻倍：KV Cache 为何只认稳定前缀

这一课只押一件事：前缀里改一个字节，后面的缓存就全废。书里第 2 章「KV Cache 友好的上下文设计」一节，用一个真实的生产事故把这条铁律钉死——某客服 Agent 只是在系统提示词开头加了一行时间戳，首 token 延迟从 0.5 秒涨到 3-5 秒，月度推理账单几乎翻倍。读完这一课，你要能亲口把这个故事讲清楚，记住三条实践铁律，并且分得清 KV Cache 和 Prompt Cache 是两个层级的东西。

一、故事：一行时间戳，为什么能让账单翻倍？

书里的场景是这样的：某团队的客服 Agent 每天处理 10 万次对话，原本一切正常。某天工程师为了让 Agent「知道」当前时间，在系统提示词里加了一行 Current time: {{now}}，把时间戳实时注入进去。第二天监控告警：所有对话的首 token 延迟（TTFT）从 0.5 秒涨到 3-5 秒，月度账单几乎翻了一倍。代码看起来完全没问题，模型也没换——问题出在哪里？

先把 KV Cache 的直觉建立起来。模型每生成一个 token，都要回头看一遍前文所有 token 的中间计算结果（注意力机制里的键 Key、值 Value）。如果每轮都从头算一次，开销会随上下文长度爆炸式增长。KV Cache 的做法是：把前文的中间计算结果缓存下来，下一轮只算新增 token 的部分。前提是前缀完全不变——只要前缀里有一个字符被改写，缓存就全部作废，模型不得不从头重算。

那行时间戳，恰恰改的就是前缀的最前面。系统提示词每次都不同，模型每次请求都要从头重新计算前缀对应的所有键值对——这就是那笔多出来的延迟和账单。作者把这类成本叫作 Agent 系统里的「无形成本」：一行看似无害的代码，能让整条推理链路慢一个量级。

二、为什么改一个 token 会让「所有层」的缓存都失效？

这是本课最需要深挖的「为什么」。答案藏在 Transformer 的结构里：大语言模型由多层 Transformer 串联堆叠而成（现代大模型通常有数十到上百层），每一层都独立生成自己的 K、V 缓存，而这些层是像流水线一样一环扣一环的。

关键在于两点叠加：

  • 层内是「看全前文」的：第 1 层在处理每个词时，会综合考虑该词及其前面所有词的信息，然后输出一个中间结果。
  • 层间是「串联」的：第 1 层的输出喂给第 2 层作为输入，第 2 层的输出再喂给第 3 层，层层向下传递。

于是，如果你改了第 1 个 token（比如系统提示词开头改了一个字），第 1 层处理它得到的输出就变了 → 第 2 层的输入随之改变 → 逐层向下传导 → 所有层的缓存都必须重算。这就是为什么书里反复强调「系统提示词一旦定下来就不要改」。

反过来看无缓存的代价就懂了：无缓存时 prefill 阶段（正式生成回复前一次性处理全部输入 token 的阶段）的注意力计算量随上下文长度平方级（O(N²)）增长；有了 KV Cache，历史 token 不必重算，每步解码的增量只随长度线性（O(N)）增长。对一个动辄几十轮工具调用的 Agent 来说，前者根本不可接受——所以缓存不是锦上添花，是刚需。

三、三条铁律：不懂原理也能照做

书里明确说：这一节是全书技术密度最高的部分之一，如果你不熟悉底层机制，可以跳过原理，只记住三条核心结论。

逐条说清楚「为什么」：

  • 前缀不改：任何改动，哪怕多一个空格，都会让缓存全部失效，延迟成倍增加、成本上升。
  • 动态信息追加末尾：想让 Agent 知道当前时间、用户余额这类会变的东西，把它作为新消息追加到对话末尾，而不是去修改已有的系统提示词——这正是本课第一张图下半部分画的做法。
  • 用标准 API 格式：结构化的 role-content 消息会被 Chat Template 翻译成模型训练时见过的固定 token 序列。这里要格外小心一个误解：自己拼 "USER: … ASSISTANT: …" 字符串，根本问题不在缓存（缓存只认 token 字节序列，只要拼出的前缀字节级稳定照样命中），而在于偏离了训练格式、会削弱模型的多步思考能力。

初学者最直觉的想法是：「在系统提示词里加动态信息（时间戳、用户余额、按使用频率给工具排序）多方便啊。」——这恰恰是全章反复警告的隐形陷阱。这些信息都在前缀里，每次一变就让整段缓存作废。书里的 ★★ 实验 2-3 里，「工具定义动态排序」就是典型：工具定义常占上下文很大一块，重排一次整段缓存就废，而实验表明固定顺序对模型选工具的能力几乎没影响——白白付出代价却换不来收益。

另一个更隐蔽的错，是把 KV Cache 和 Prompt Cache 当成一回事。它们利用的都是「前缀不变」，但层级不同：KV Cache 是推理引擎内、单次请求内的优化；Prompt Cache 是API 服务层、跨请求的缓存，而且不会自动命中——比如 Anthropic 要显式设 cache_control 断点、有约 1.25 倍的写入加价、有最小可缓存长度（如 1024 token）和默认约 5 分钟的 TTL；OpenAI 则是自动前缀缓存。混为一谈，就会在计费和启用方式上踩坑。

四、宏观背景：为什么这条铁律值得单开一课

因为它是整个上下文工程的物理约束。第 2 章把 Agent 每次调用的上下文拆成「静态前缀 + 轨迹」两段（对应图2-4）：系统提示词和工具定义构成不变的前缀，用户消息、模型回复、工具结果构成不断增长的轨迹。理解了 KV Cache，你就理解了那句贯穿全章的话——「前面不能动、后面可以压缩」。

更深一层：在生产级系统里，缓存已经不只是性能优化，而是架构约束。书里举 Claude Code 的例子——系统提示词被一个缓存边界一分为二，所有动态元素（操作系统类型、当前模式、语言偏好）都被严格归到边界之后，因为放到边界之前每多一个二值条件就会让缓存键的变体数量翻倍（N 个条件产生 2ᴺ 种组合）；子 Agent 的前缀甚至要和父 Agent 逐字节对齐才能命中 Prompt Cache。缓存经济性不是事后优化，而是前置约束——越早纳入架构，后续工程代价越小。

本课唯一要带走的判断：静态前缀（system + 工具定义）一旦定下就一个字节都别动；所有会变的东西一律追加到上下文末尾。做到这一点，KV Cache / Prompt Cache 才能持续命中——否则延迟成倍、账单翻番，而你的代码「看起来完全没问题」。

五、关键定义

KV Cache
模型内部优化：缓存前文 token 在注意力机制中已算出的键值对（Key-Value pairs），下一轮只算新增 token，前提是前缀完全不变；加速单次请求内的 token 生成。避免与 Prompt Cache 混为一谈——它在推理引擎内、单次请求内。
Prompt Cache
构建在 KV Cache 之上的 API 服务层跨请求缓存：服务商对请求前缀匹配，前缀相同则复用已算好的 KV，缓存读取成本约为首次计算的 1/10 到 1/2。避免以为它自动命中——Anthropic 需显式设 cache_control 断点、有写入加价与 TTL。
Chat Template
把结构化 role-content 消息转换为模型训练时见过的线性 token 流的模板（用 <|im_start|>、<|im_end|> 等特殊标记划分角色与边界），由 API 服务端自动完成。避免理解为开发者需手写的东西，也不要与 system prompt 混淆。

六、自测题

Q1：开发者想让 Agent 知道「当前时间」，把 Current time: {{now}} 写进系统提示词开头。为什么这会让整段 KV Cache 每次都失效？正确做法是什么？

因为 Transformer 层层串联，第 1 层处理每个词都综合了它和前面所有词的信息；系统提示词开头的时间戳每次都不同，第 1 个 token 一变，第 1 层输出就变、逐层向下传导，所有层缓存全部作废、必须从头重算。正确做法是把时间信息作为消息追加到对话末尾（动态信息追加末尾），或只在真正需要时通过工具调用获取——保持静态前缀字节级稳定。

Q2：KV Cache 和 Prompt Cache 都靠「前缀不变」，它们到底差在哪一层？

层级不同。KV Cache 是模型内部、推理引擎里的优化，作用在单次请求内，加速 token 生成。Prompt Cache 构建在它之上，是 API 服务层跨请求的缓存：不同请求前缀相同就复用之前算好的 KV，读取成本远低于首次计算（约 1/10 到 1/2）。两者都要求前缀稳定，但 Prompt Cache 的经济影响更大（直接影响 API 计费），且不一定自动命中——如 Anthropic 需显式设 cache_control。

★★ 实验 2-3「常见的错误上下文管理模式」（仓库目录 chapter2/kv-cache/）：用 Kimi K2 ReAct Agent 系统性对比六种实现模式（一种正确 + 五种错误：动态系统提示词、动态用户配置、工具动态排序、滑动窗口、文本格式化），实测 TTFT、缓存命中率与 token 用量，让你亲眼看到「细微改动如何摧毁缓存与 Agent 能力」。想先建立 TTFT 直觉，可配合 ★ 实验 2-1（chapter2/local_llm_serving/）：保持系统提示词不变连发两次、再改开头几个字符发一次，对比首 token 延迟。开源仓库：github.com/bojieli/ai-agent-book。
