工作流还是自主 Agent：一条“从简单到复杂”的选型梯

这节课只拿下一件事：确定性工作流和动态自主 Agent 的本质区别、各自的优势与局限，以及作者给的那条铁律般的选型顺序——先提示词，再工作流，最后才自主。它出现在第 1 章「编排模式：工作流与自主」一节，是 Harness 工程里「上下文与工具怎么组织」这一层的核心决策。记住：Agent 系统用延迟和成本换性能，这笔交换值不值，就是选型的全部张力所在。

一、两种模式：路径是写死的，还是现算的？

编排模式决定了三件事：上下文如何在多次 LLM 调用之间流动、工具如何被调度、以及 Agent 的执行路径是预先设定还是动态生成。最后这半句，就是工作流与自主 Agent 的分水岭。

工作流（Workflow）：通过预定义的代码路径来编排 LLM 和工具。每一步做什么、下一步去哪里，都是开发者用代码写死的；LLM 只在每个节点内部负责理解和生成。作者用订机票举例——四个固定节点：核实身份 → 搜索航班 → 完成付款 → 确认预订。节点内部可以用 LLM（比如理解用户的出行需求），但节点之间的流转顺序是代码固定的：系统不会在付款前预订，也不会在核实身份前搜航班。

自主 Agent（Autonomous Agent）：执行路径不是预先定义的，而是 Agent 根据环境反馈实时决定的。同样订机票，它不需要四个固定节点——用户说「帮我订下周三去上海的机票」，Agent 自行决定先搜航班、发现要登录、于是先核实身份、再回来搜、发现最便宜的要转机、主动问用户接不接受……路径是现场长出来的。从实现看，自主 Agent 本质上就是在一个循环里使用工具的 LLM，靠持续获取环境反馈推进任务——这正是前面讲过的 ReAct 循环。

二、为什么是「从简单到复杂」，而不是一上来就自主

作者引用 Anthropic 与数十个团队合作构建 LLM Agent 的经验，给出一条反直觉但极务实的原则：最成功的实现往往不是复杂的框架，而是简单、可组合的模式。落到操作上，就是一条单向的选型梯——能用低一档解决，就绝不跳到高一档。

为什么要压着自己别升档？因为Agent 系统通常用延迟和成本换取更好的任务性能。一个任务如果要 20 轮推理，每轮多等 2 秒就是多等 40 秒；自主 Agent 还会带来复合错误（compounding error）的风险——一步错，后面基于错误状态继续决策，越滚越偏。所以这笔「用延迟和成本换性能」的交换，必须谨慎权衡值不值。

那自主 Agent 的价值在哪？在于开放式问题——这类问题难以甚至不可能预测需要多少步。作者举的典型场景：Coding Agent 解 SWE-bench、Computer Use Agent 像人一样操作界面、需要反复搜索与分析的研究任务。这些场景里，你根本画不出一张固定的流程图，只能让 Agent 边走边看。

三、边界与易错：安全性的取舍，与「非此即彼」的误区

工作流不是「低级」，自主 Agent 也不是「高级」——它们是两种权衡。把这两点的取舍看清楚，才不会选错。

工作流有两个核心优势。一是严格的流程控制：像「付款前不能预订」这类业务规则由代码强制执行，不依赖 LLM 的判断。二是安全性：因为路径确定，提示注入或模型犯错最多只能影响当前节点内部，无法让 Agent 跳到不该执行的分支——攻击面被限制在单个节点里。代价则是缺乏变通：遇到预设流程没覆盖的情况（临时改签、航班取消需推荐替代），固定路径只能走预设异常分支或交还人类。

根据 Anthropic 与数十个团队合作构建 LLM Agent 的经验，最成功的实现往往不是使用复杂的框架，而是采用简单、可组合的模式。 ——《深入理解 AI Agent》第 1 章，引 Anthropic《Building Effective Agents》

自主 Agent 这一侧，最关键的边界是：自主 ≠ 无限制。必须设计明确的停止条件（任务完成、达到最大迭代次数、或遭遇不可恢复的错误），否则 Agent 容易陷入死循环或过度执行。常见退出条件：调用最终输出工具、模型返回没有任何工具调用的响应、遇错、达到最大轮次。部署时还要在沙盒里充分测试、设护栏与监控、在关键决策点加人机协作检查点。

误区一：过早跳到自主 Agent。「自主更智能，那就直接上自主」——这是初学者最常犯的错。作者的顺序是先提示词 → 再工作流 → 最后自主，理由不是保守，而是自主 Agent 会用延迟、成本和复合错误风险换性能。如果任务能清晰拆成固定子任务，工作流不但更快更便宜，还更安全（攻击面被锁在单节点内）。跳档要有明确理由，而不是「感觉更高级」。

误区二：以为两者非此即彼。实践中很多系统混合使用：合规要求严的关键流程用工作流保可靠，需要灵活决策的部分切自主。例如开源框架 n8n 允许在同一系统里同时拖拽工作流节点和自主 Agent 节点（书中图 1-6）。选型不是二选一，而是「哪一段用哪种」。

四、框架只是壳：为什么「模型越强，Harness 越关键」

看主流框架对比表（OpenAI Agents SDK、Claude Agent SDK、LangChain/LangGraph、n8n、Dify、CrewAI、OpenClaw）时，别被「哪个框架更强」带偏。随着「模型即 Agent」趋势深化，框架的核心价值不再是「编排 LLM 调用」——模型越来越能自主决策，反而是围绕模型的上下文管理、工具生态、安全约束和错误恢复这些 Harness 工程变得更重要。所以作者说：选框架的关键不在框架本身多复杂，而在它能否以最小的抽象层让你专注于业务逻辑。

本课唯一要带走的判断：编排选型是一条单向梯——能用单次 LLM 就别上工作流，能用工作流就别上自主 Agent；每升一档，你都在用延迟、成本和失控风险换灵活性。工作流的隐藏红利是安全（攻击面锁在单节点），自主 Agent 的必答题是停止条件，而真实系统往往是两者混合。不存在通用最优解，只有「这一段该用哪种」。

五、关键定义

工作流（Workflow）
通过预定义的代码路径来编排 LLM 和工具的系统，执行路径是确定性的、由开发者预先设计；LLM 只在每个节点内部负责理解与生成。优势是严格的流程控制与安全性，局限是缺乏变通。
自主 Agent（Autonomous Agent）
执行路径不预先定义、而由 Agent 依据环境反馈实时决定的系统，本质是「在循环中使用工具的 LLM」。需自主规划、能识别失败并调整策略，且必须配明确的停止条件。避免把「自主」等同于「无限制」。
模型即 Agent（Model as Agent）
先进模型通过后训练（尤其 RL）把工具调用内化为原生能力，何时调、调哪个、传什么参数都由模型自主决定，无需人工编排层的新范式。避免理解为「框架不再重要」——恰恰相反，模型越强 Harness 越关键。
Harness（工程外壳/马具）
模型之外的全部基础设施，核心是上下文与工具，外加约束、验证、纠正三层保障机制；编排模式正是其中「上下文与工具怎么组织」的一层。避免简单等同于「框架」或「SDK」。

六、自测题

Q1：订机票场景里，工作流模式在「安全性」上的优势具体来自什么？它的主要代价是什么？

优势来自执行路径确定：提示注入或模型犯错最多只能影响当前节点内部，无法让 Agent 跳到不该执行的分支——攻击面被限制在单个节点里。代价是缺乏变通：遇到预设流程未覆盖的情况（如临时改签、航班取消需推荐替代方案），固定的节点路径无法灵活应对，只能走预设异常分支或将控制权交还人类。

Q2：作者主张「从简单到复杂」的选型顺序是哪三档？为什么不建议一上来就用自主 Agent？

三档顺序：① 单次 LLM 调用（能靠优化提示词+上下文示例解决就别引入 Agent 系统）→ ② 工作流（可清晰拆成固定子任务时）→ ③ 自主 Agent（只有需要动态决策和灵活执行路径时）。不建议直接上自主，是因为 Agent 系统通常用延迟和成本换取更好的任务性能，自主 Agent 还带来更高成本与复合错误风险；应谨慎权衡这笔交换是否值得，能用低一档解决就别升档。

想亲手摸到「自主 Agent = 循环中用工具的 LLM」这句话，跑一下实验 1.2（对应仓库 chapter1/web-search-agent/）：它用 Kimi K2 内置的 $web_search 工具做自主迭代式网络搜索（最多 5 次迭代）并综合答案——由模型自己决定何时搜、搜什么、信息是否已经足够。这正是「模型即 Agent」的自主模式最小可跑样例。完整代码见开源仓库 github.com/bojieli/ai-agent-book。
