工具爆炸了怎么选：主动发现 vs Skills 按需查阅

本课只拿下一件事：当可用工具从十几个涨到成百上千，为什么“把所有工具的 schema 一次性塞进 system prompt”会失效，以及书里给出的两条解法各自省掉了什么、又付出了什么。这是第 8 章「主动工具发现」一节（在 8.5 之后、「从工具使用者到工具创造者」之前）的正面战场。记住：找到工具和后一节的创造工具是两件事，本课只管前者。

一、问题：全量注入为什么会“塞爆”上下文

所有从经验中学习的机制都有一个前提——Agent 得先有合适的工具才能干活。工具少的时候，传统做法很省事：把所有工具的 schema 一次性写进系统提示词。但作者把话说得很直接：当工具数量上千，这种做法迅速失效。

为什么？不是内存不够，而是上下文被“工具说明书”塞满，模型的选择精度随之下降。第四章讲过的检索式预筛选（按语义相似度先筛出一批候选工具）能缓解，却有一个内在局限：它按用户的初始查询做一次性匹配。可像“Debug the file”这类看似简单的请求，实际会牵出文件访问、代码分析、命令执行等多步骤、跨领域的工具链——任务开始时根本无法预见所有需求。

二、解法一：主动发现——从“被动选择”到“主动声明”

更进一步的思路，是让 Agent 从被动接受者变成主动发现者：在执行过程中意识到能力缺口时，主动用自然语言声明“我需要什么能力”，系统再动态匹配并注入。代表工作是 MCP-Zero——系统提示词里不预置任何工具 schema，Agent 在思考中生成结构化请求块（如“GitHub 服务器：搜索仓库并返回元数据”），系统通过服务器级 → 工具级的两层语义路由从数千候选中匹配注入。论文报告：在约 2800 个工具上，比全量注入节省约 98% 的 token。

工程上更常见的等价方案，是系统提示词里只保留少数基础工具（web search、code interpreter），外加一个“工具搜索工具”，Agent 用自然语言描述需求即可检索并加载——Anthropic 在 Claude API 中提供的 Tool Search Tool 即属此类。两者的共同点都是：Agent 声明缺口、系统按需注入。

为什么要分“两层”？

高效匹配的关键，在于工具组织本身就有层次结构。在 MCP 等协议里，工具按服务器分组（类似手机上的 App，每个 App 提供一组相关功能）。于是匹配可以分两层：先按能力描述定位相关服务器，再在服务器内匹配具体工具——把搜索空间从“数千个工具”缩小成“数十个服务器 × 每个服务器数十个工具”，既省算力，也减少跨领域的语义混淆。这依赖一个离线构建、支持增量更新的嵌入索引；若两层匹配的候选相似度都低于阈值，就应明确返回“未找到”，让 Agent 改写需求重试、用基础工具手工实现，或干脆去创造一个新工具（那是下一节的事）。

动态加载的隐藏代价：KV Cache

主动发现有一个微妙的工程代价：动态加载工具会破坏 KV Cache。如果把工具列表放进 system prompt，每加载一个新工具就使整段缓存失效。破解思路与第二章讨论 Skill 注入位置时一致——把会变动的部分（新工具的完整 schema）作为 user 消息追加到对话末尾，让 system prompt 前缀保持稳定、KV Cache 完全复用，只在 Agent 状态栏维护一份简短的工具名列表。

但这里有个前提要盯住：在主流 function-calling API 中，合法工具集由请求的 tools 参数决定——模型无法调用“只出现在对话文本里、却不在 tools 参数中”的工具。因此这套模式依赖框架或 API 的专门支持（如前述 Tool Search Tool）。此外，动态工具环境对模型能力要求更高：能力较弱的模型既难以理解“工具定义出现在上下文中间”这种非标准位置，也容易生成非法调用格式（JSON 括号不匹配、参数缺失），往往要通过强化学习专门训练（详见第七章）。

误区一：“检索预筛选”就等于“主动发现”。不是。第四章的检索式预筛选是按用户初始查询做一次性匹配，任务一开始就把候选定死；而主动发现是在执行过程中随需要多次触发——Agent 走到哪、发现缺什么，才声明什么。“Debug the file”能牵出文件访问、代码分析、命令执行一整串工具，这些在开局那一刻是预见不到的。把两者划等号，就会以为“开头筛一次”足够了。

误区二：“上下文够长，全塞进去让模型自己挑就行。”恰恰相反。作者点明：上下文被工具 schema 塞满时，模型选择精度会下降。实验 8-4 里，4B 模型面对 50K+ token 的全量注入，指令遵循严重退化，甚至“忘记”列表中的某些工具。多≠好。

三、解法二：Skills——把“发现工具”变成“翻工具书”

近来更流行的一种思路来自 Skills 机制。第二章从上下文工程角度介绍过它的渐进式披露（Progressive Disclosure）；这里换个角度，把它当成一种工具发现范式来看。它和上一节最大的不同，是不再需要那套“嵌入索引 + 语义匹配”的基础设施。

像 MCP 这样的协议倾向于把工具的完整 schema 一次性摆在模型面前（要么全量注入、要么靠检索预筛先选出一批）；Skills 则相反：Agent 启动时只看到一份薄薄的目录——每个 skill 的 name 与 description（合计数百 token）。当当前上下文真的需要某种能力时，模型才去读对应的 sub-skill，并顺着其中的引用再往下一层，读取具体的脚本或子文档。“发现”由模型在上下文里的实际需要驱动，而不是在任务开始时对初始查询做一次性预匹配。

没有人会把一本工具书或整个维基百科从第一页读到最后一页，而是顺着索引和目录，按当下的需要一个词条、一个词条地精确查阅。 ——《深入理解 AI Agent》第 8 章「主动工具发现」

关键在这句：Agent 靠通用的文件阅读能力（grep、读取文件）翻阅 skill 目录即可，既不必维护向量索引，也不必把“发现工具”单独建模成一次特殊的语义检索。上一节那一整套“主动声明—语义匹配—动态注入”虽然有效，工程上却相当繁琐：要离线维护嵌入索引、处理 KV Cache 失效、依赖特定 API 支持、还要为弱模型做专门训练。它们共同的前提，是把每个工具都当成一份面向模型的正式定义，先注册、再检索、再注入。Skills 换了一种更轻、也更省心的思路——用到哪条查哪条。

那 Skills 场景下 KV Cache 怎么办？加载一个 sub-skill，本质上就是往上下文里插入一段内容，同样可以用第二章的“注入位置”方法把它放到末尾、复用前缀。但 Skills 有个新特点：同一批 skill 会被反复、且在不同位置加载（跨会话、跨用户）。若每次都随对话历史从头 prefill，成本不小。第二章末尾介绍的“可编辑、可组合的 KV Cache”正是为此而生：把每个 skill 的 KV 表示预编译并缓存一次，之后用 RoPE 重定位“粘贴”到任意上下文位置，以 O(L) 而非 O(L²) 的代价拼接进来。这样 skill 就从“一段每次都要重新 prefill 的文本”，升级为“一个可复用、可组合的缓存对象”。

本课唯一要带走的判断：工具一多，全量注入必然塌方——因为塞满上下文会拉低模型的选择精度。两条解法的分水岭是要不要养一套检索基础设施：主动发现走“声明→两层语义路由→动态注入”，精准但要维护嵌入索引、处理 KV Cache、依赖特定 API；Skills 走“薄目录→逐层查阅”，靠通用文件阅读（grep、读文件）就够，省掉了向量索引和“把发现单独建模成一次语义检索”这一整层工程。判断该用哪种，先问一句：你手上有没有、愿不愿意养那套索引？

四、关键定义

主动工具发现（Active Tool Discovery）
Agent 执行中意识到能力缺口时，用自然语言声明需求，系统按服务器级 → 工具级两层语义路由动态匹配并注入工具 schema。区别于第四章的“一次性检索预筛选”——主动发现是执行过程中随需要多次触发。
渐进式披露（Progressive Disclosure）
Skills 机制启动时只暴露每个 skill 的 name 与 description（数百 token），需要时才逐层读取 sub-skill 及脚本子文档。不要说成“检索”——它由模型上下文中的实际需要驱动，靠通用文件阅读而非向量索引。
检索式预筛选（第四章）
按用户初始查询的语义相似度，先筛出一批候选工具。缓解了全量注入的塞爆问题，但只做一次性匹配，无法覆盖任务中途才浮现的跨领域工具链需求。

五、自测题

Q1：把上千个工具的完整 schema 一次性注入 system prompt 有什么问题？Skills 的渐进式披露相比“主动发现 + 语义匹配”省掉了什么基础设施？

全量注入会用“工具说明书”塞满上下文，模型的选择精度随之下降（实验 8-4 中，4B 模型在 50K+ tokens 下指令遵循严重退化）。主动发现（如 MCP-Zero）用服务器级 → 工具级两层语义路由按需注入，但工程上要离线维护嵌入索引、处理 KV Cache 失效、依赖特定 API 支持。Skills 则靠通用的文件阅读能力（grep、读文件）顺着目录一层层查阅，不必维护向量索引、也不必把“发现工具”单独建模成一次语义检索——更省心。

Q2：层次化匹配把搜索空间从什么缩小到了什么？如果两层匹配的候选相似度都低于阈值，系统应该怎么办？

把搜索空间从“数千个工具”缩小为“数十个服务器 × 每个服务器数十个工具”——先按能力描述定位相关服务器，再在服务器内匹配具体工具。若两层候选相似度都低于阈值，应明确返回“未找到”，让 Agent 改写需求重试、用基础工具手工实现，或干脆创造一个新工具。

实验 8-4 ★★★ 主动工具发现：用 Qwen3-4B 访问前面构建的 MCP 服务器里的 120+ 工具。对照组把全部工具 schema 一次性注入 system prompt（超 50K tokens），4B 模型在这么长的上下文下指令遵循严重退化（该选 Yahoo Finance 却错选 Web Search，或“忘记”某些工具导致任务失败）；实验组只在 system prompt 保留 web_search、code_interpreter 和一个 discover_tools 元工具，按需检索并把新工具定义追加为 user message。预期：准确率与任务完成率显著提升——主动发现尤其让小参数量模型在上百工具的场景下保持可用。配套代码见开源仓库 github.com/bojieli/ai-agent-book 的 chapter8。
