给模型装一个状态栏：把隐式状态提炼成可查的显式知识

这节课只押一句判断：上下文窗口是一台只有一半的检索引擎——有检索，没提炼。它能从上万个 token 里把相关记录捞出来，却从不会自动把它们数一遍、总结成一条结论。Agent 状态栏就是补上缺失那一半：把分散的隐式状态提前算好、以键值对形式贴在上下文末尾，让模型“瞥一眼”就知道现在到哪了。位置在第 2 章《上下文工程》讲完 KV Cache、提示工程、Skills 之后，是把前面所有约束（缓存、注意力、末尾注入）收拢到一处的落地机制。

一、问题：一台只有一半的检索引擎

作者给了一个很狠的比喻。上下文学习更像检索而非推理——模型擅长从已有内容里查找信息，但不擅长主动归纳统计。所以：

上下文窗口是一台只有一半的检索引擎。“检索”那一半非常强……但它缺了另一半：没有“提炼层”。任何“关于这些内容的结论”——一共多少条、有没有超标、进展到哪一步——模型每次要用，都得从原始记录里现算一遍。 ——《深入理解 AI Agent》第 2 章

书里的实际场景一秒说清代价：系统提示词要求“拨打每个商家不超过 3 次”，但打了 3 次之后，Agent 经常数不清到底打了几次，又打了第 4 次，甚至陷入循环反复拨打。根源不是模型笨——是“已经打了几次”这个知识从没被提炼出来，而是以原始通话记录的形式散落在 KV Cache 里；模型每次决策都得花思考 token 去重新扫描统计，而“现算一遍”的代价，会随上下文堆积量（记作 N）一起往上涨。

二、状态栏是什么：贴在末尾的一条 user 消息

最好的类比是手机顶部状态栏：时间、电量、信号——不是 App 的主界面内容，但你随时能瞥一眼掌握设备状态。Agent 状态栏对模型起完全相同的作用：它不属于用户消息、模型输出或工具结果，而是 Agent 框架在上下文末尾持续注入的状态摘要——“你已经打了 3 次电话”“当前时间 10:30”“TODO 还剩 2 项”。

看图里那条最末的消息：它的 role 是 user，但内容是框架自动生成的元信息。这里要澄清一个极易混淆的点：这个 user 角色只是 API 协议层面的技术槽位，不等于“来自真实终端用户的输入”。Harness 只是借用 user 这个消息格式，把系统状态挂到上下文末尾。它跟系统提示词（System Prompt）的分工也很清楚：系统提示词是入职发的员工手册，一旦定下不变；状态栏是贴在屏幕边缘的实时仪表盘，随任务推进不断更新。

三、为什么放末尾、为什么用 user 角色（深挖“为什么”）

这两个选择不是随意的，全都被前面几节的约束逼出来：

  • 为什么不改 system 消息？因为 KV Cache：修改开头的 system 会让整个前缀缓存作废——这正是本章反复警告的“动态时间戳”陷阱。追加到末尾，则只影响最近几轮。
  • 为什么放末尾有效？结构化的元信息放在末尾，在空间上更接近即将生成的新 token，因而获得更高注意力权重——一种“强制性的注意力引导”。这也是它绕开了模型不擅长的“长上下文注意力衰减”的原因。
  • 为什么一定写成键值对？书里点名：状态要写成 衣物: 9 件（合格 7、次品 2） 这样能一眼定位的键值对，而不是一段大白话。用散文写同样的状态，效果明显更差——模型还得先把散文读一遍、解析出来，等于又回到了“扫描”。

作者和合作者用一个专门基准把这套“提前算好、直接查一眼”的做法量化了一遍（它有个统一名字叫上下文蒸馏，Context Distillation，状态栏是它最日常的形态）：三类任务、11 个模型、近 2.4 万次评测。结论很干净——弱模型补回来的是准确率（最弱的几个涨 40 到 54 个百分点，一个 2B 本地小模型直接追平不带状态栏的前沿大模型）；强模型省下来的是效率（思考量、延迟、花费各降约一个数量级）。最本质的变化是：不带状态栏时思考量随上下文变长而持续增长，带上后基本恒定——不管上下文堆多长，模型都只是瞥一眼那几格。

“提前算好”这件事，做对和做错是天壤之别。书里给了三条能直接照着做的经验：

最诱人也最错的念头：“再叫一个大模型去读完整段历史、帮我总结出状态栏。”直觉上这很省事，实际结果恰恰相反。让前沿大模型一次性读完整段历史、吐出统计结果，反而在大多数格子上出错，把下游准确率拖得比根本不用状态栏还低；而一个 20 行的正则函数就能达到“标准答案”级准确度。

为什么？因为让 LLM 批量统计长历史，等于把“扫描整段上下文做归纳”这个原始难题原封不动搬了个家——问题一点没解决。正确姿势是：能用代码确定性地算就用代码算；实在要用 LLM，也要逐条抽取、再由代码汇总，绝不让它一次性批量统计。

另外两条同样是生产红线：

  • 想删原文，先确认状态栏覆盖了所有会被问到的问题。状态栏是对原始上下文的一次有损投影——它只算了“你预想会被问到”的维度。只要有一个问题落到没算过的维度，就会翻车：论文里状态栏只存了“两两组合”的计数、却去问“三者交叉”，只留状态栏时连 Claude 都从 100% 掉到 7.6%——一条像样却答非所问的状态栏，会变成理直气壮带偏模型的“假权威”。所以新增一种问法要当成“改数据库表结构”来对待：要么先给状态栏加字段，要么这次就别删原文。
  • 把状态栏准确率当一线生产指标来盯。模型几乎无条件相信状态栏——你写“打了 3 次”它就当真是 3 次，既不核对也不重算。这既是它有效的原因，也意味着状态栏一旦写错，错误会原样传进最终答案。这也接上了状态栏投毒风险：状态栏里的信息越是来自对真实世界的可靠观测越好，绝不能来自可被外部污染的数据源。
本课唯一要带走的判断：上下文只有“检索”没有“提炼”，Agent 状态栏就是把隐式状态提前算好、以键值对贴在末尾的“提炼层”。它有效有三重根据（提炼隐式状态、末尾操纵注意力、喂进模型自己想不出来的外部观测），但做对的前提也有三条铁律——用代码维护而非叫大模型总结、删原文前先确认覆盖、把准确率当生产指标盯。

六、关键定义

Agent 状态栏（Agent Status Bar）
Agent 框架在上下文末尾以 user 角色消息持续注入的结构化状态摘要（工具计数、当前时间、TODO 等），把分散的隐式状态提炼为可直接检索的显式知识。对应图2-14、图2-15。注意：这里的 user 角色只是 API 协议槽位，不等于“来自真实终端用户的输入”，也不要与系统提示词混淆。
上下文蒸馏（Context Distillation）
“提前算好、直接查一眼”这套做法的统一名称——把需要模型现场归纳统计才能得到的结论提前算好放进上下文；Agent 状态栏是它最日常的形态。避免与模型蒸馏（把知识蒸馏进权重）混淆，这是推理时的上下文层做法。
KV Cache
模型内部优化：缓存前文 token 在注意力机制中已算出的键值对，下一轮只算新增 token，前提是前缀完全不变。状态栏“放末尾、用 user 角色、只追加不改 system”的所有设计，都是为了不破坏这段缓存。避免与 API 服务层的 Prompt Cache 混为一谈。

七、自测题

Q1：有人提议“再叫一个大模型去读完整段历史、帮我总结出状态栏”。为什么论文实验发现这样做反而比根本不用状态栏还糟，而一个 20 行正则函数却能达到标准答案级准确度？

因为让 LLM 一次性批量统计长历史，等于把“扫描整段上下文做归纳”这个原始难题原封不动搬了个家——问题一点没解决，还会在多数格子上出错，把下游准确率拖到比不用状态栏更低。状态栏的价值恰恰是绕开模型不擅长的归纳统计：能用代码确定性地算就用代码算（正则计数不会错）；实在要用 LLM 也要逐条抽取、再由代码汇总，绝不让它一次性批量统计。

Q2：状态栏为什么被实现成“末尾追加的一条 user 消息”，而不是把动态信息写进开头的 system 提示词？说出协议和缓存两个层面的原因。

缓存层面：修改开头的 system 消息会让整个前缀的 KV Cache 作废（就是本章批评的“动态时间戳”失效机制）；追加到末尾只影响最近几轮。协议层面：借用 user 角色只是 API 的技术槽位选择，用来把框架自动生成的状态挂到上下文最末尾——它紧邻即将生成的新 token，注意力权重最高。它并不代表这条消息来自真实终端用户。

配套实验直接对着代码跑最有感觉：实验 2-8 的 chapter2/system-hint/（agent-status-bar 框架）实现了五种状态栏技术——时间戳跟踪、工具调用计数器、TODO 列表、详细错误信息、系统状态感知，每种可独立开关，能亲眼看到组合使用的涌现效应（如 TODO 让平均迭代从 21 降到 15）。想验证“注意力集中到状态栏那几格”的直觉，再配上 实验 2-7 的 chapter2/attention_visualization/ 看真实模型的注意力热力图。仓库：github.com/bojieli/ai-agent-book。
