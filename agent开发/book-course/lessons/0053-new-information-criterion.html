多 Agent 何时真赢过单 Agent：唯一判据是「有没有新信息」

这节课只押一条判据：协作过程是否引入了单个 Agent 在生成时无法获得的新信息？它出现在第十章《多 Agent 协作》正文“多 Agent 何时真正优于单 Agent”一节，是全章所有工程方案的总体参照。学会用这一条尺子量任何一种协作，你就能一眼分清哪些多 Agent 是真本事、哪些只是“让模型再想一遍”还多花了十几倍的钱。

一、是什么：一条判据，把学术与工程的争吵一次说清

书里先摆出一个看似矛盾的现象：学术研究常说“单 Agent 就够了”，工程实践里多 Agent 却确实更好。作者的回答不是和稀泥，而是给出一条清晰的判据——两派讨论的根本不是同一种“多 Agent”。

多 Agent 是否值得 ⟺ 协作是否引入了生成时不存在的新信息
第十章 · “多 Agent 何时真正优于单 Agent”的核心判据

学术研究里比较的，多是“多个 Agent 看着同一段文本互相讨论”（如辩论）——没有引入新信息；工程实践里有效的多 Agent，往往带一条外部反馈环路（代码执行、视觉渲染、工具调用）——引入了新信息。前者无效，后者显著提升。书里用表10-2 把五种模式一次列清：

二、为什么这样设计：把判据画成一次分流

为什么一条判据就够？因为它把“要不要上多 Agent”从模糊的直觉，变成一个可以当场执行的二分判断。任何一个协作步骤，先问它一句：这一步让某个 Agent 看到了它自己生成时根本拿不到的信息吗？答案分流如下——

“否”这一侧为什么无效，书里给了信息论层面的硬解释。2026 年 Tran 与 Kiela 在多跳推理任务上对比单 Agent 与五种多 Agent 架构，发现当思考 token 预算被严格控制为相同时，单 Agent 与多 Agent 持平甚至更好。依据是信息论里的数据处理不等式：辩论中各 Agent 处理的是完全相同的文本，Agent 之间每一次串行传递中间结论，只可能丢信息、不可能凭空造信息。辩论在一些论文里的收益，很可能只是因为它偷偷消耗了更多总计算量。

对等协作那一节里，作者还把“自我审查”这一行拆开讲透：Huang 等人在 ICLR 2024《Large Language Models Cannot Self-Correct Reasoning Yet》中发现，让模型在没有外部反馈时审查并修正自己的回答，准确率反而下降——把正确答案改错的次数，比把错误改对的次数还多。ICLR 2024 的 CRITIC 论文给了对照：接上外部工具（搜索、Python 解释器）验证时效果显著提升，一旦移除工具、只留模型自评，提升大部分消失。这就是提议者-审核者范式的设计原理：Reviewer 的价值不在“再看一遍”，而在它渲染了截图、跑出了测试结果——这些是 Proposer 写代码时根本拿不到的新信号。

三、边界与代价：判据之外，还有一张账单

就算判据是“是”，也别急着上多 Agent——书里把成本摆在所有设计之前。Anthropic 曾披露，其多 Agent 研究系统的 token 消耗约为普通对话的 15 倍，而 token 用量本身就能解释其中约 80% 的性能差异。换句话说：多 Agent 的收益必须大到能覆盖数倍乃至一个数量级的额外开销，否则一个调校得当的单 Agent 才是更划算的选择。

初学者最容易想错的一句：“多 Agent 天生更强，人越多越好。” 这句直觉恰恰踩空了判据。让三个 Agent 围着同一段文本辩论、或让同一个模型生成后再自己审一遍，都没有引入任何新信息——按数据处理不等式，它们至多持平、常常还越改越差，却实打实地烧掉了十几倍的 token。判据管的不是“几个 Agent”，而是“这次协作有没有让谁看到新东西”。

另一个易混点：这条数据处理不等式只否定“串行传递中间结论”的辩论式多 Agent，并不否定另外两类做法——对同一问题多次独立采样再聚合（self-consistency、多数投票），或利用生成与验证的难度不对称（写答案难、验答案易）来做生成-验证分工。这两类要么引入了额外的独立采样、要么利用了任务本身的不对称结构，都不在数据处理不等式的射程之内。别把“辩论无效”错误地推广成“所有多路做法无效”。

本课唯一要带走的判断：评估任何一种多 Agent 协作，先问一句——它引入了单个 Agent 生成时拿不到的新信息吗？没有（自我审查、等算力辩论），那多半是白花十几倍 token 的错觉；有（执行 / 视觉 / 工具反馈），才是真价值，且收益还得大到能盖过那张账单。

四、关键定义

新信息判据
判断多 Agent 相对单 Agent 是否有实质价值的唯一核心标准：协作过程是否引入了单个 Agent 在生成时无法获得的新信息。有效来源包括执行反馈、视觉反馈、工具反馈；无效的典型是自我审查与等算力下的辩论。对应表10-2。
数据处理不等式（Data Processing Inequality）
信息论结论：处理同一份信息的串行链路上，每一次传递中间结论只可能丢失信息、不可能凭空创造信息。书中用它解释“多 Agent 辩论同一段文本”为何在等 token 预算下不优于单 Agent。避免把它推广到独立多次采样或生成-验证不对称的场景。
提议者-审核者（Proposer–Reviewer）
最经典的对等协作范式：Proposer 生成，Reviewer 渲染 / 执行结果并用外部反馈评估、给出结构化改进建议，反复迭代。其价值不在“让模型再想一遍”，而在 Reviewer 能接触 Proposer 生成时拿不到的新信息（截图、测试结果、外部搜索）。避免误以为换个 Agent 重读就能生效。

五、自测题

Q1：「让同一个模型生成代码后再自己审查一遍」和「Reviewer 渲染 PPT 截图后审查」，哪个属于有效的多 Agent 协作？为什么？

后者有效。前者没有引入任何新信息，只是让模型再想一遍，研究表明纯自我纠正常常越改越差；后者的截图包含 Proposer 生成代码时完全无法获得的视觉信息，属于引入了新信息，因此能显著提升。

Q2：有人读到“多 Agent 辩论在等 token 预算下并不优于单 Agent”，就断言“所有多路 / 多次做法都没用”。按本书，这个推广错在哪？

错在越界了。数据处理不等式否定的只是“多个 Agent 串行传递中间结论”的辩论式协作——同一份文本反复转手，只会丢信息。它不否定另外两类：其一是对同一问题多次独立采样再聚合（self-consistency、多数投票），引入了额外的独立采样；其二是利用生成与验证的难度不对称（写答案难、验答案易）做生成-验证分工，利用了任务本身的不对称结构。这两类都不在数据处理不等式的适用范围内。

本节是第十章的“判据”总论，没有独立配套代码；判据的正例会在后面的对等协作与管理者实验里反复落点（如书籍翻译 Agent、边打电话边用电脑的双 Agent）。想现在就把“新信息”看成分水岭，最直接的动手方式是跑一遍提议者-审核者：对比“Reviewer 只重读代码”与“Reviewer 跑测试 / 看渲染截图”两种设置，亲眼看提升如何随新信息的有无而出现。整本书实验都在开源仓库：github.com/bojieli/ai-agent-book。
