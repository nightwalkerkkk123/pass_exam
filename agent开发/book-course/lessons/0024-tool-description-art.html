工具描述的艺术：先讲清"不能做什么"

这节课只押一句判断：多数工具调用失败，不是模型不知道工具"能做什么"，而是不知道它"不能做什么"、参数长什么样。所以当 Agent 频繁选错工具时，第一反应应该是去改工具描述，而不是怀疑模型、换一个更强的模型。这一课出自第 4 章"工具描述的艺术"一节——它是 ACI（Agent-Computer Interface）原则里最便宜、投产比最高的一环。

一、工具描述到底在描述什么？

作者开门见山："工具描述的质量直接决定了 Agent 使用工具的准确性。"但初学者写描述时，脑子里想的往往是"这个工具是干嘛的"——于是写出"搜索相关内容"这种句子。书里说得很直白：这远不如写"当需要获取实时信息或查找未知事实时使用"。

差别在哪？前者只描述功能（能做什么），后者帮 LLM 做调用决策（什么时候该用它）。工具描述的核心，是让模型知道"什么时候用"，而不只是"能做什么"。这是本节的第一个转向：描述的读者不是人类文档读者，是一个正在多个工具之间做选择的模型。

二、为什么"讲边界"比"讲能力"更重要

这是全课的分量所在。书里的例子：一个文件搜索工具，如果不写清楚它只能按文件名匹配、不能搜索文件内容，LLM 就会去猜——然后拿它去搜文件内容，失败。

清晰列出工具的边界条件——做不到什么、不接受什么输入——往往比描述能力本身更重要，因为大多数工具调用失败的根因不是模型不知道工具能做什么，而是不知道工具不能做什么。 ——《深入理解 AI Agent》第 4 章·工具描述的艺术

为什么"反例"比"正面能力"更值钱？因为模型的正面能力覆盖面本来就很宽——它见过无数搜索工具，大致猜得到能干嘛；真正让它翻车的，是它猜不到的那条隐形红线。你不写边界，模型就会用它对"搜索"二字的通用理解去脑补，脑补错了就反复失败。写清"不能做什么"，等于提前把它会踩的坑标出来。

三、参数：用"例子"代替"规范"

第二个反直觉点在参数描述上。写规范术语（"RFC3339 格式""E.164 格式"）看起来专业，但作者说：用具体的例子代替抽象的规范。

  • 不写"timestamp：RFC3339 格式"，而写"timestamp：RFC3339 格式，例如 2024-03-15T14:30:00Z"。
  • 不写"phone：使用 E.164 格式"，而写"phone：电话号码，使用 E.164 格式（国家代码+号码，无空格或特殊字符），例如 +861388888888（中国）或 +12025551234（美国）"。

为什么例子更有效？书里给了一个很精准的解释：LLM 在专注处理一个问题时确实能理解这些术语；但在执行复杂任务时——要同时处理多个工具、从历史轨迹里提取信息、权衡多个决策——"确认参数格式"只占它注意力的一小部分，就容易出错。具体的例子让 Agent 直接套用，无需额外的思考步骤。这也是为什么书里进一步建议：给每个工具附 1-5 个真实调用示例。JSON Schema 只能描述参数类型，却表达不了调用方式和典型参数组合（时间戳到底是秒还是毫秒、过滤条件如何嵌套），这些隐式约定靠例子最好传达。

别忘了返回值和代价也要写。"返回 JSON 数组，每个元素包含 title、url、snippet 三个字段"能减少后续解析出错；对耗时工具注明代价（"下载完整网页，大型网站可能需要 5-10 秒；如果只需元信息，请考虑用 get_page_metadata"）能帮模型合理规划调用顺序。

四、一条调试铁律：先改描述，别怪模型

把前三节收成一条可操作的原则——这是本课要带走的东西：

当 Agent 频繁选错工具时，应优先检查工具描述而不是怀疑模型能力……修正工具描述的投入产出比，通常远高于更换一个更强的模型。 ——《深入理解 AI Agent》第 4 章·工具描述的艺术

大多数工具选择错误的根因，是描述不准确：边界不清、缺少反例、参数含义模糊。这一整套做法，在书的"工具设计的演进"里被归为第二代——ACI（Agent-Computer Interface）原则：工具应对应 Agent 的目标，而非底层 API 操作；核心是让工具对 Agent 而非对人友好。你写描述时要时刻记得，读者是那个正在多任务里分心的模型。

初学者会怎么想错？看到 Agent 老是拿"文件搜索"去搜文件内容、老是选错工具，第一反应是"这模型不行，换个更强的"或"再多写几句 prompt 哄一哄"。

为什么错？因为问题根本不在模型的能力，而在它拿到的描述里没有那条红线。工具描述没写"只能按文件名匹配、不能搜索文件内容"，模型就只能靠对"搜索"二字的通用理解去猜。换更强的模型，它照样猜；真正该做的是回到工具定义，把"不能做什么"和参数示例补上。修描述的投产比，远高于换模型。

五、关键定义

ACI（Agent-Computer Interface）
对标 HCI（人机交互）提出的概念：研究 Agent 如何与计算机交互，核心是让工具对 Agent 而非对人友好——工具应对应 Agent 的目标，而非底层 API 操作。粒度权衡、通用性设计、描述规范都属于这一层。避免把它等同为"API"或"函数调用规范"，它是设计理念层面的，不是某种具体协议。
边界条件（做不到什么 / 不接受什么输入）
工具描述中显式写出的反例说明，例如"只能按文件名匹配，不能搜索文件内容"。书中判断：清晰列出边界条件，往往比描述能力本身更重要，因为多数调用失败源于模型不知道工具"不能做什么"。
参数传递的保真性（Parameter fidelity）
模型感知到的世界与工具操作的世界之间，不得存在系统性偏差；工具不得在模型不知情时静默转换输入（如弯引号→直引号）或静默注入参数。避免窄化为"参数校验"——它强调的是透明性：即使做规范化，也必须在描述与返回中告知模型。

六、自测题

Q1：文件搜索工具老被 Agent 误用来搜文件内容。按本章原则，你会先改什么？再举一个"参数用例子代替规范"的改法。

先改工具描述，而不是怀疑模型。显式写出边界反例——"只能按文件名匹配，不能搜索文件内容"——把这条隐形红线标出来，模型就不会去猜。

"参数用例子代替规范"的改法：不写"timestamp：RFC3339 格式"，而写"例如 2024-03-15T14:30:00Z"；不写"phone：E.164 格式"，而给出 +861388888888（中国）/ +12025551234（美国）的具体样例。再附 1-5 个真实调用示例，能把准确率从约 72% 提到 90%（因任务而异）。

Q2：书里说，为什么"给具体例子"比"写术语规范"对 LLM 更有效？

因为 LLM 在专注处理单个问题时确实能理解 RFC3339、E.164 这类术语；但执行复杂任务时，它要同时处理多个工具、从历史轨迹提取信息、权衡多个决策，"确认参数格式"只占注意力的一小部分，就容易出错。具体的例子让它直接套用、无需额外的思考步骤。同理，JSON Schema 只能表达参数类型，表达不了"时间戳是秒还是毫秒""过滤条件如何嵌套"这类隐式约定——这些靠例子最好传达。

配套实验 4-1 的感知工具 MCP 服务器里就有搜索与文件系统工具（chapter4/perception-tools/），正是本课"文件搜索只能按文件名"这个例子的落地场景。建议对着仓库里这些工具的 description 逐个读：看它们怎么写"何时用"、怎么标边界、参数里给没给示例。整本书实验都在开源仓库：github.com/bojieli/ai-agent-book。

本课唯一要带走的判断：工具描述的读者是那个正在多任务里分心的模型。写描述时，"讲清不能做什么"（边界反例）和"用例子代替规范"（参数样例 + 1-5 个调用示例），比堆砌功能说明更管用。所以 Agent 选错工具时，先改描述，别急着换模型——这是投产比最高的一步。
