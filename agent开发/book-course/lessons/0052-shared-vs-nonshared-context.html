共享 vs 不共享上下文：多 Agent 的第一个岔路口

动手搭多 Agent 系统，第一个必须拍板的问题不是"用几个 Agent"，而是它们之间要不要共享上下文。第十章一开头就把这条定为"最基础的架构决策"。本课的唯一目标：让你拿到一个新场景，能从子任务数、窗口占用、并行度、信息隔离、成本五个角度当场判断该走哪条路，并记住那条"50% 窗口"经验法则和"阶段切换式"折中方案。这是理解后面所有协作拓扑的地基。

一、是什么：两种传信息的方式

作者把多 Agent 系统拆成两个正交的设计维度，第一维就是"上下文是否共享"——它决定了多个 Agent 之间如何传递信息。

  • 共享上下文：后一个 Agent 接收前一个 Agent 的完整对话历史和轨迹（第一章定义的 trajectory）。每个阶段切换系统提示词和工具集后就变成了一个新的 Agent（身份、职责、能力都变了），但它保留了前任的全部记忆。优势是信息不丢失；挑战是上下文可能快速膨胀。
  • 不共享上下文：每个 Agent 维护完全独立的上下文和对话历史，彼此无法直接访问对方的"思考过程"。模块化和隔离性更好，也更易扩展维护——增加新 Agent 不必改动现有 Agent 的内部逻辑，只需定义好接口和数据格式。

正因为不共享上下文的 Agent 看不到彼此的内心活动，它们必须靠显式的通信机制传信息。书里列了三种：工具调用的参数（结构化数据当参数传）、共享文件系统（读写共享目录里的中间产物）、消息总线（Message Bus，一个专门在 Agent 间中转消息的站点，发送方不直接调用接收方）。消息总线天然支持异步通信——像公司内部邮件，发送方和接收方不需要同时在线。

那到底怎么快速拍板？书给了一条极简判断，也是本课最该背下来的一句：

若预期累计上下文会超过窗口的 50%（这是一条经验法则，而非精确阈值），应不共享；若信息零损耗对任务正确性是硬约束，应共享。 ——《深入理解 AI Agent》第 10 章

三、折中：多数真实系统走"阶段切换式"

现实里很少非黑即白。作者点明：多数实际系统采用"阶段切换式"方案——前几个 Agent 先共享上下文（信息零损耗、隐式协调最省事），到信息饱和点后再切换为"不共享上下文 + 显式 handoff（移交）"。所谓 handoff，就是由上游 Agent 主动决定把哪些信息交接给下游，而不是把整条轨迹一股脑倒过去。

补一层"为什么"：共享上下文里，每个阶段其实都是一个独立的 Agent（有自己的系统提示词和工具集），只是它继承了前序 Agent 的完整轨迹——"就像接班的同事能翻阅前任留下的所有工作日志"。这种继承式协作的红利是信息零损耗，代价则是——如何让当前 Agent 专注自己的核心职责，而不被继承来的大量历史信息干扰。饱和点的存在，正是这份代价累积到临界的信号。

误区一：以为"多 Agent = 必须不共享上下文"。 不是。书里强调两种架构都是真正的多 Agent 系统——只要每个阶段的系统提示词和工具集不同，它就是不同的 Agent。共享上下文只是让它们靠"隐式协调"（继承轨迹）通信而已。

误区二：把"上下文是否共享"和"协作拓扑"当成一回事。 二者概念上独立、实践中相关：共享上下文的系统一样有拓扑，只不过拓扑往往退化成一条角色切换序列、没多少架构决策可做；一旦选择不共享，"信息如何流动、由谁协调"才成为必须显式设计的问题。别把这一维直接读成"要不要拆多个 Agent"。

四、宏观背景：为什么这一刀值得单开一课

因为它是整章的分叉总闸。第十章后面讲的对等协作、管理者、去中心化三种拓扑，作者明说"只详细展开不共享上下文的三格"——也就是说，你先在这一维上做对了选择，后面的拓扑设计才有意义。选错了这一刀（该隔离时硬共享、该共享时强行拆开），后面无论怎么调拓扑都是在错误的地基上加盖。

它也直接连着成本这条硬约束。表10-1 里"不共享总 token 高出数倍到一个数量级"不是随口一说——这与后文"多 Agent token 消耗约为普通对话 15 倍"的数据是同一件事的两个侧面。隔离和并行都要花钱，所以这一刀本质上是在替整个系统的账单做第一次预算。

本课唯一要带走的判断：拿到多 Agent 场景，先做一件事——沿五个角度（子任务数、窗口占用、并行度、信息隔离、成本）扫一遍，用"预期累计上下文超窗口 50% 就不共享、信息零损耗是硬约束就共享"当默认判据；拿不准就走阶段切换式：先共享吃零损耗红利，到饱和点切成"不共享 + 显式 handoff"。这一刀切在前，拓扑设计才在后。

五、关键定义

共享上下文 / 不共享上下文（Shared / Non-shared Context）
多 Agent 的第一设计维度：共享指后一 Agent 继承前一 Agent 的完整轨迹（隐式协调、信息零损耗、上下文膨胀）；不共享指各 Agent 维护独立上下文，仅通过工具参数 / 共享文件 / 消息总线三种显式机制通信。避免与"多轮对话历史"混同——这里区分的是 Agent 之间是否继承彼此的完整轨迹。
移交（handoff / 移交包）
由上游 Agent 主动决定把哪些信息交接给下游。不共享上下文下的"移交包"通常含任务描述、已确认的事实与约束、结构化产物的引用（文件路径而非内容），刻意不传全量轨迹。避免理解成"把完整历史一并交出"——那是共享上下文下的形态。
消息总线（Message Bus）
专门在 Agent 间中转消息的基础设施，Agent 发布 / 订阅消息而不直接调用彼此，天然支持异步、多方并行；常见实现有 Redis Pub/Sub（不持久化）与 RabbitMQ（持久化）。避免与点对点通信混用——总线的意义正是解耦收发双方、避免连接数平方增长。

六、自测题

Q1：一个翻译几百页技术书的多 Agent 系统，为什么不适合让所有 Agent 共享上下文？用本章的判据说明。

因为全书术语表 + 已译章节 + 翻译思考 + 工具结果会累积到远超上下文窗口（预期超窗口 50% 就应不共享），且过长上下文会让 Agent"迷失"——忘记术语约定、重复检查、甚至幻觉出不存在的术语规则。这类任务子任务多、需并行、单窗口装不下，应选不共享上下文 + 显式 handoff。

Q2：不共享上下文下的三种显式通信机制分别是什么？其中哪一种天然支持"发送方和接收方不必同时在线"，为什么？

三种是：工具调用的参数（把结构化数据当参数传给下游 Agent 的工具）、共享文件系统（读写共享目录里的中间产物）、消息总线（Agent 把消息发到总线、由它转发）。其中消息总线天然支持异步——像公司内部邮件系统，你发邮件不要求对方此刻在电脑前，消息先存在服务器上，等对方上线再处理，因此收发双方无需同时在线，特别适合多个 Agent 并行工作、彼此需要协调的场景。

本节侧重架构判据，配套代码在后面的实验里落地（如管理者模式下的"实验 10-3 书籍翻译 Agent"就是"共享装不下 → 改不共享"的活样本）。建议边读边跑开源仓库：github.com/bojieli/ai-agent-book——先把这一维的取舍看懂，再去看拓扑怎么在"不共享"这一格里展开。
