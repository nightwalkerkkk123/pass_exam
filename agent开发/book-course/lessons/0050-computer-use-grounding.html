点哪个按钮：Computer Use 视觉定位的三条路线

Computer Use Agent 每转一圈都要回答同一个问题：“这个按钮在截图的哪个坐标？”——这就是视觉定位（Grounding）。本课唯一要拿下的，是三条定位路线各自怎么做、什么时候选哪条，以及贯穿其中的那句设计直觉：让模型答选择题，比让它答填空题靠谱得多。这一段在书里紧跟“感知-思考-行动循环”，是《Computer Use》一节的技术核心（原文“视觉定位（Grounding）”小节）。

一、Grounding 到底是什么问题

先把它放回循环里。Computer Use 的核心是一个感知-思考-行动的循环（图9-7）：截图 → 多模态模型看着截图和任务指令、输出一段思考和一个动作 → 执行层去点/去输入 → 等界面响应后再截图，进入下一轮。

循环里有三个关键设计维度：动作空间（能执行哪些操作）、视觉定位（如何在截图中找到目标元素）、模型架构（如何从截图生成正确动作）。这节课只钻中间那个——因为它是最容易翻车、也最能拉开系统好坏的一环。原文把话说得很直白：模型要回答的是“搜索框在哪里？”“提交按钮的坐标是什么？”

二、三条路线，逐一拆开

路线一 · 结构化元素索引（DOM / Accessibility Tree）。现代网页在渲染前就已经定义好完整的元素结构（DOM 树）和语义角色（哪个是按钮、哪个是输入框），无障碍接口（Accessibility Tree）为许多桌面应用提供类似信息。与其让模型在像素里猜“哪块是按钮”，不如直接问界面本身“你有哪些可点的元素？”。以 browser-use 为代表的方案分四步走：用 CDP（Chrome DevTools Protocol）拿到 DOM 与无障碍信息 → 自动检测可交互元素 → 给每个元素标 ID 并在截图上画框 → 同时生成一份 ID 到元素的文本清单（图9-9）。模型只需吐一个 ID 号，系统用该元素的中心坐标去点。代价是不省 token（所有标注都要发给模型），换来的是定位准确稳定，还免掉了分割模型的漏检误检。

路线二 · Set-of-Mark（SoM，纯视觉标注）。SoM 由微软研究院 2023 年提出，最初是为了释放 GPT-4V 的定位能力。它是纯视觉的：用图像分割模型（SAM、SEEM 等）在截图上切出候选区域，给每块叠一个编号，模型看到的是一张带编号的图，只报编号，由系统换算成区域中心坐标。它不需要 DOM、不需要任何界面内部结构——所以原生桌面软件、游戏界面照样能用，只要分割模型能把区域切出来。可以把路线一理解成“SoM 思想在 Web 上的结构化实现”：同样是“先标号、再选号”，只是标号的活儿从分割模型换成了界面自带的结构。

路线三 · 纯坐标预测。这条路线不做任何标注，直接让模型输出坐标。代表是 SeeClick 和 Claude 的 computer use：在海量“GUI 截图 + 元素位置”配对数据上训练，让模型学会把“点击提交按钮”这样的自然语言直接映射到截图里的精确坐标——像人一样纯靠“看”。它省掉了标注这一步，但把最难的开放式定位全压回了模型身上。

为什么路线三非配这层缩放不可？因为模型对坐标的理解高度依赖训练时用的分辨率。Claude 训练只用过三档：XGA（1024×768）、WXGA（1280×800）、FWXGA（1366×768）。截图分辨率一旦不匹配，预测坐标就会系统性偏移——“就像在小地图上量了距离，直接拿到大地图上用”。反向映射的算法很朴素：

真实x = 模型x × (2560 / 1366)　　真实y = 模型y × (1440 / 768)
(683, 384) → (683×2560/1366, 384×1440/768) ≈ (1280, 720)

三、为什么“选择题比填空题好答”

这句直觉是三条路线背后的设计主轴，也是路线一、二共享的红利。开放式定位（“在整张截图里找到那个按钮并报出精确坐标”）对模型极难；而选择题思路先把界面元素标好编号，把问题压成封闭式——“从这堆标好的元素里挑一个”。用原文的对比说：模型只需说“点击 [123]”，而不是“点击屏幕左上角偏右大约 200 像素处的蓝色按钮”。就像考试里选择题比填空题更容易答对，把开放定位关进有限选项，正确率自然更稳。

反过来也就看清了路线三的代价：坐标预测本质是填空题——没有选项兜底，全靠模型“看”准。所以它只对专门做过 GUI 定位训练的模型才直接好用，对没训练过的通用模型则不友好。这就是为什么同样一件“点按钮”的事，会分裂出三条工程路线。

误区一：以为“没有结构就只能用坐标预测”。不对。结构化信息不可得时（原生桌面软件如 Photoshop、Canvas/WebGL 渲染的界面、游戏），视觉标注（原始 SoM）和坐标预测两条都能走。SoM 把定位变成选择题，对未经专门训练的通用模型更友好；坐标预测省去标注，对做过 GUI 定位训练的模型更直接。二选一是按“你手里的模型训没训练过”来定，不是“没结构就只剩坐标”。

误区二：随便把截图塞进模型就报坐标。坐标预测最隐蔽的坑是非等比拉伸。若硬把 16:9 的屏幕拉进 4:3 的 1024×768，画面被横向压扁，模型预测的坐标就会系统性偏移。正确做法是先按宽高比挑最接近的档位（16:9 挑 FWXGA 1366×768），等比缩放后再送入，输出坐标再反向映射回去。

四、选择逻辑：一句话收口

作者把三条路线的取舍浓缩成一条判断链：结构化信息可得时，优先用 DOM / Accessibility Tree 索引，定位最精确稳定；不可得时，视觉标注或坐标预测都行，按“模型有没有做过 GUI 定位训练”来挑。而且要老实承认边界：两者在小元素和密集界面上的精度都还有差距——这不是已经解决的问题，而是仍在推进的战线。

本课唯一要带走的判断：视觉定位先看“界面能不能自己交出结构”。能，就读 DOM / AX（最准）；不能，就在 SoM 标注和坐标预测里按“模型训没训练过”二选一。贯穿始终的直觉只有一句——能把定位变成选择题，就别让模型去答填空题；只有非用坐标预测（填空题）时，才必须补上按宽高比缩放的双向映射。

五、关键定义

视觉定位（Grounding）
在感知-思考-行动循环的每一轮里，模型要在截图中准确定位目标元素（“搜索框在哪”“提交按钮坐标是多少”）的问题。三条路线都是它的不同解法。
Set-of-Mark（SoM）
微软研究院 2023 年提出的纯视觉标注法：用分割模型（SAM/SEEM）在截图上切出候选区域并叠编号，模型只报编号、系统换算成区域中心坐标。不依赖 DOM，桌面软件与游戏同样适用。避免把它当成只能用于网页。
结构化元素索引（DOM / Accessibility Tree）
直接从界面自带的结构里枚举可交互元素并编号（以 browser-use 为代表，经 CDP 读取），是 SoM 思想在 Web 上的结构化实现。可得时定位最精确稳定，代价是不省 token。
双向坐标缩放
坐标预测路线的工具层机制：截图按宽高比选目标档位、等比缩放后送入模型，模型输出坐标再反向按比例映射回真实分辨率。避免非等比拉伸——那会让坐标系统性偏移。

六、自测题

Q1：真实屏幕 2560×1440（16:9），要送入只在 XGA/WXGA/FWXGA 三档训练过的 Claude，该选哪档？模型输出坐标后要怎么处理？

选 FWXGA（1366×768），因为它宽高比最接近 16:9，避免非等比拉伸把坐标判断带偏。把截图等比缩放到 1366×768 送入；模型输出坐标后要反向按比例映射回真实分辨率，如 (683,384) → (683×2560/1366, 384×1440/768) ≈ (1280,720)。

Q2：界面是个 Canvas/WebGL 渲染的游戏，拿不到任何 DOM 结构。此时用哪条路线？决定“选哪条”的关键因素是什么？

结构不可得，所以 DOM/AX 索引出局，剩下视觉标注（原始 SoM）或纯坐标预测两条都可行。选哪条的关键是手里的模型有没有做过 GUI 定位训练：没训练过的通用模型更适合 SoM（把定位变成选择题），做过定位训练的模型可以直接用坐标预测。两者在小元素/密集界面上精度都仍有差距。

这一节有两个可上手的实验，都在开源仓库里 github.com/bojieli/ai-agent-book：实验 9-7 用 browser-use 做自然语言驱动的浏览器操作，启用 SoM 可视化模式后，每次决策前都会保存带红色标注框和 ID 号的截图——你能亲眼看到“选择题”是怎么标出来的（打开 Google 查旧金山天气那个例子，全程 5 步约 20 秒）。想先看纯坐标预测那一路，可跑 实验 9-6 的 Anthropic Computer Use Demo，留意它每个动作间隔 2–5 秒、明显慢于人类。
