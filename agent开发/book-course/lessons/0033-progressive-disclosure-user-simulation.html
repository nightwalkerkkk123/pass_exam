别把答案一次性喂给 Agent：渐进式信息透露

这节课只押一件事：渐进式信息透露（Progressive Information Disclosure）——它是第 6 章「人机交互型评估环境」一节里，作者点名的「人机交互型评估与传统 benchmark 的根本区别」。搞懂它，你就明白了为什么 τ-bench 要专门再拉一个 LLM 来扮演一个「话说一半」的用户，以及这么做到底在考察 Agent 的哪块肌肉。

一、是什么：把需求分几口喂，而不是一次性倒出来

先看它落在书里的哪个位置。第 6 章把评估环境分成两种范式（对应图6-2）：工具调用型和人机交互型。前者像 Verifiers 框架那样，Agent 调用预定义工具、用可执行标准（测试是否通过、答案是否匹配）来验证，不依赖人；后者要面对一个绕不开的难题——如何在自动化环境中模拟真实用户。

作者的观察很朴素：现实中用户很少能一上来就清晰描述需求。他们往往只会说「我的航班好像有问题」「网络连不上了」。所以评估里定了一条硬规矩——绝不能一开始就把模拟用户的所有信息暴露给 Agent，信息要按需、渐进地在对话中透露。这就是「渐进式信息透露」。

二、怎么做：用另一个 LLM 照着固定脚本演用户

τ-bench 的解决方案叫用户模拟（User Simulation）：用另一个 LLM 扮演用户，根据预定义指令与 Agent 对话——逐步透露必要信息、回应询问，任务完成后发出终止信号。提示词里明确要求模拟用户「不要一次性透露所有信息，只提供当前步骤必要的内容」「不要编造指令中未提供的信息」。

下面是书里给出的一个 6 轮渐进式信息透露对话示例（用户模拟器按固定脚本行动）。注意看每一轮：用户手里其实早就攥着全部答案，但闸门只在 Agent 问对问题时才开一格。

为什么非要「固定脚本」？因为用户模拟本身要在真实性和可控性之间权衡——行为要接近真实用户（表达模糊、信息不完整、偶尔情绪波动），但又必须遵循剧本，否则每跑一次结果都不一样，评估就没法复现。脚本＝已知信息＋透露规则，正是这个平衡点。

关键设计原则是渐进式信息透露，这是人机交互型评估与传统基准测试的根本区别……绝不能一开始就把模拟用户的所有信息暴露给 Agent，信息应按需、渐进地在对话中透露。 ——《深入理解 AI Agent》第 6 章

三、为什么这样设计：主动澄清本身就是要考的能力

这是本课最该深挖的「为什么」。传统 benchmark 大多一开始就把完整需求全盘托出——但那等于替 Agent 把最难的一步做掉了。Agent 需要通过主动提问来澄清需求，这个过程本身就是能力的重要体现。如果用户开口就把「Delta 123 / 想改签 / 下午都行」一次性说全，Agent 不问一句也能猜对答案，评估就测不出「谁更会追问」——区分度归零。

再往深一层：τ-bench 的检查是组件级、多维度的——一面查数据库最终状态对不对（如预订记录变成「已取消」），一面查 Agent 在对话里有没有输出必要的关键信息（如退款金额、到账时间，靠搜索特定字符串验证）。这套「双重验证」同时考操作准确性和沟通有效性。可到了任务层面，它们最终汇总成零或一的二元奖励：全部检查通过才得 1 分，任何一项不过就是 0 分。好处是便于统计 Pass^k 这类可靠性指标；代价是「操作准确但漏掉某个非关键字段」和「完全失败」拿到同样的 0 分。

四、演进：τ²-bench 补了什么——双控环境

改进版 τ²-bench 的核心增量不在评分粒度，而在两点。一是双控环境（Dual-Control）：不再只有 Agent 一方能调工具，用户模拟器也能操作同一个共享环境——比如 Agent 指导用户切换飞行模式，用户这一下真的改变了环境状态。这更贴近技术支持这类「需要用户动手配合」的场景。二是更精确的任务规范与组合式任务生成：成功条件歧义更少、具体任务实例可以参数化批量生成。

作者还点了 τ-bench 初版的三处毛病，正好反衬渐进式透露为什么要做扎实：用户指令过于简单（Agent 能猜对答案）、成功条件不够精确（导致误判）、用户模拟器过于机械。τ²-bench 的对症下药之一，就是「更真实的用户模拟器行为规范：渐进式信息透露、自然的情绪波动」。

初学者最容易想歪的地方：把「渐进式信息透露」当成「就是多轮对话嘛」。不是。多轮对话只描述了「聊了好几个来回」，而渐进式信息透露的要害在信息按脚本受控释放——用户手里明明有全部答案，却被规则约束着一格一格放。少了这层「受控」，你要么退回一次性给全（考不出追问能力），要么让模拟用户随机乱答（评估不可复现）。

连带的第二个坑：以为「双控环境」只是「对话更长」。双控的本质是用户侧也能真正改变共享环境状态，于是验证多出一维——Agent 必须通过工具调用观察到用户那一下操作的结果，再据此继续排查。这是「谁能动手」的变化，不是「聊几轮」的变化。

五、关键定义

渐进式信息透露（Progressive Information Disclosure）
人机交互型评估的核心原则：不一开始把模拟用户的全部信息暴露给 Agent，而按需、渐进地在对话中透露，考察 Agent 主动澄清需求的能力。避免把它当成简单的多轮对话——关键在信息按脚本受控释放。
双控环境（Dual-Control）
τ²-bench 引入的机制：不再只有 Agent 一方能调用工具，用户模拟器也能操作同一共享环境（如用户切换飞行模式真正改变环境状态），更贴近技术支持等需用户动手配合的场景。
数据泄漏（Data Contamination）
评估数据被纳入模型训练数据的现象，导致评估测的是记忆力而非泛化能力；防范手段含答案独特性、动态参数生成、时间新鲜度、canary GUID 等。避免简称为「过拟合」——泄漏特指评估题进了训练集。

六、自测题

Q1：传统 benchmark 一开始就给出完整需求，而 τ-bench 让模拟用户只说「我的航班有个问题」。这个设计差异是为了考察 Agent 的什么能力？为什么用固定脚本很重要？

为了考察 Agent 主动提问、澄清模糊需求的能力——现实用户很少一上来就说清需求，主动澄清本身就是核心能力。固定脚本（已知信息 + 透露规则）保证评估可复现，同时模拟真实用户的渐进式表达。若一开始暴露全部信息，Agent 不需提问就能猜对，评估就失去了区分度。

Q2：τ-bench 的检查明明是「组件级、多维度」的，为什么最终只给一个 0 或 1 的分数？这样做的好处和代价各是什么？

因为在任务层面，所有组件级检查（数据库最终状态是否正确、对话中是否输出了退款金额/到账时间等关键信息）被汇总为二元奖励：全部通过才得 1 分，任何一项不过就是 0 分。好处：便于统计 Pass^k 这类可靠性指标。代价：「操作准确但漏掉某个非关键字段」与「完全失败」得到相同的分数，丢失了细粒度区分。

本课唯一要带走的判断：人机交互型评估的分水岭是「谁掌握答案、什么时候放」——渐进式信息透露让模拟用户攥着全部信息、按固定脚本一格一格放，逼 Agent 用「主动追问」自己把需求挖出来；这才是它区别于传统 benchmark 的根本，也是它真正在测的那块能力。
