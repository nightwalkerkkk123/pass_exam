一个公式看懂 Agent：大脑 + 眼睛 + 手脚

第 1 章开篇就把整本书的坐标系钉死在一句话上：Agent = LLM + 上下文 + 工具，换成直觉版就是大脑 + 眼睛 + 手脚。这节课唯一要拿下的东西，是把这个公式变成一副可复用的拆解眼镜——以后随手拿起 Cursor、Deep Research、Manus、豆包、Pine 里的任意一个，你都能立刻把它拆到感知 / 行动 / 策略三个维度上。作者在“现代 Agent = LLM + 上下文 + 工具”一节里把这套心智模型交给你，本章其余部分全是它的展开。

一、公式：三个词，每个都要做“广义理解”

作者的原话是：“现代 Agent 系统的本质可以用一个简洁的公式来表达。”但紧接着他补了一句很重要的限定——“其中每个词都需要做广义的理解”。这句限定不是客套，它决定了你会不会一开始就把公式读窄。

Agent = LLM + 上下文 + 工具
= 大脑 + 眼睛 + 手脚 · 三者缺一不可

三个词分别做广义理解后是这样：

  • LLM = 大脑：不只是一组模型参数，而是整个决策内核——理解意图、思考规划、做出判断。它的能力来自两部分：预训练积累的世界知识与语言能力，加上后训练固化的决策策略（后者第 7 章展开）。
  • 上下文 = 眼睛：不只是喂给模型的那段文本，而是 Agent 在每个决策点能看到的全部信息——环境信息、用户记忆、领域知识、自身状态和任务进展。
  • 工具 = 手脚：不只是几个可调用的 API 函数，而是 Agent 能做的所有事情的集合——从预定义工具调用、按需加载的 Skills，到动态生成代码创造新能力、委托子 Agent 协作、主动与用户沟通、响应外部事件。

作者给的一句话总结最好记：大脑负责思考和决策，眼睛提供思考所需的全部信息，手脚将决策转化为对现实世界的改变。

二、为什么要拆成三维：它接上了强化学习的学术骨架

这套“大脑 / 眼睛 / 手脚”的直觉，不是作者临时编的顺口溜——它恰好对应强化学习（RL）里的三个核心概念（详见第 7 章）。作者特意说明：下面这层映射是“可选阅读”，没有 RL 背景可以跳过、不影响理解；它只是帮有 RL 背景的读者把已有知识和本书术语对齐。

为什么值得把直觉和学术挂钩？因为它告诉你这三个维度不是随意切的三刀，而是任何一个“在环境里行动的智能体”都逃不掉的三件事：它看到什么、它能做什么、它据此怎么决策。这也是为什么作者说“理解这三者的作用及其相互关系，是构建有效 Agent 系统的基础”。

三、把眼镜戴上：五个产品，同一副骨架

公式的威力在于它可迁移。作者用它一口气拆了五个形态完全不同的产品——从写代码的 Cursor 到打电话砍账单的 Pine，全都落进同一张三维表：

作者还点出这五个系统有三个共同特征，而它们全都是三个组件协同作用的结果：

  • 开放式的动作空间——不是从有限的几个按钮里选，而是能生成任意自然语言和代码（来自“手脚/工具”）。
  • 能内部思考——采取行动前先思考和规划（来自“大脑/LLM”）。
  • 能持续交互——根据环境反馈不断调整策略（三者转成一个循环）。

最常见的翻车：把三个词读窄。初学者最容易犯的错，是不理会作者“每个词都要做广义理解”的提醒，于是——把 LLM 读成“就是一堆模型参数”，把 上下文 读成“就是我输入的那段 prompt”，把 工具 读成“就是几个 API 函数”。

为什么这是错的、且代价很大？因为一旦读窄，你就会漏掉公式真正的解释力。比如把“手脚”窄化成“几个 API”，你就想不到动态生成代码、委托子 Agent、响应外部事件也都算工具；把“眼睛”窄化成“对话历史”，你就会漏掉工具定义、用户记忆、环境状态同样是上下文的一部分——而这正是下一课消融实验要坐实的东西。公式的价值全在“广义”二字，读窄了它就退化成一句正确的废话。

四、为什么值得为一个公式单开一课（宏观背景）

因为作者明说了本章的定位：它是全书的“概念地图”。这个三维公式是一副不随版本翻篇的坐标系——当下一代模型、下一个热门框架冒出来时，你不必从零理解它，只要问三个问题：它能看到什么（上下文/眼睛）、能做什么（工具/手脚）、大脑（LLM）多强、据此怎么决策（策略）。作者甚至提醒：初读本章不必逐一记住所有概念，先建立整体印象，后面每一章都会回来展开其中一个方面。

本课唯一要带走的判断：任何 Agent，不管形态多花哨，都能拆成三个维度——眼睛（上下文/看到什么）、手脚（工具/能做什么）、大脑（LLM/据此如何决策）；三者缺一不可，且每个词都要做广义理解。以后读任何 Agent 系统、排查任何 Agent 故障，先把它套进这三个格子。

五、关键定义

Agent 核心公式（Agent = 大脑 + 眼睛 + 手脚）
全书主线：Agent = LLM（大脑，思考决策）+ 上下文（眼睛，能看到的全部信息）+ 工具（手脚，能做的所有事），三者缺一不可。关键：每个词都要做广义理解，别读窄成“参数/prompt/几个 API”。
策略（Policy）
“大脑/LLM”对应的学术概念：Agent 决定“下一步做什么”的决策逻辑——面对当前看到的信息，从所有可选行动中挑出最合适的一个。（RL 背景可选映射。）
观察空间（Observation Space）
“眼睛/上下文”对应的学术概念：Agent 能看到的所有信息——能看到什么、读到什么、记住什么、能访问哪些系统。本章反复强调它是决定性因素。
动作空间（Action Space）
“手脚/工具”对应的学术概念：Agent 能做的所有事情的集合——从发消息到执行代码再到操控界面。本章五个产品的动作空间都是开放式的。

六、自测题

Q1：把 Cursor 这类 Coding Agent 拆到「眼睛 / 手脚 / 策略」三个维度，各自对应什么？

眼睛（感知）=需求文档、代码库、终端环境；手脚（行动）=开放式动作（内部思考、代码搜索、文件读写、执行命令）；策略=增量开发（理解需求→搜索相关代码→编辑代码→测试验证→调试修复）。这正是“戴上公式眼镜”拆一个产品的标准动作。

Q2：作者反复强调公式里“工具”要做广义理解。除了“调用预定义的 API 函数”，工具还包括哪些？

按原文，“工具”是 Agent 能做的所有事情的集合，广义地包括：预定义的工具调用、按需加载的专业技能（Skills）、动态生成代码创造新能力、委托子 Agent 协作、主动与用户沟通、以及响应外部事件。把工具窄化为“几个 API 函数”，就会漏掉后面几种，也就看不懂后续章节里 Agent 的很多能力从哪来。

配套实验（强烈建议边读边跑）：实验 1.1 上下文消融，对应仓库目录 chapter1/context/。它做的正是把公式里“眼睛”这一维拆开验证：用一个带多工具（PDF 解析 / 货币转换 / 计算器 / Python 代码解释器）的 Agent，切换 full / no_history / no_reasoning / no_tool_calls / no_tool_results 五种上下文模式做批量消融，用成功率、耗时、迭代次数实证“每个上下文组件都不可或缺”。这是下一课（第 0007 课）的主戏，先把仓库拉下来：github.com/bojieli/ai-agent-book。
