管理者模式：为什么最强的模型要给规划者

这节课只押一件事：管理者模式（Orchestration）靠一个统一抽象立住——Manager 把每个专门 Agent 当成一个可调用的工具。理解这句话，你就顺手拿到它的可扩展性与异构性；再叠上 Plan-and-Act 论文的实证结论——弱规划者是整个系统最关键的瓶颈——你就知道资源该往哪儿倾斜。这是第十章“不共享上下文”三种拓扑里的第二种（对等协作、管理者、去中心化），承接上一课的提议者-审核者循环。

一、是什么：Manager 把 Agent 当工具调用

当任务涉及五个以上子任务、需要动态调度、或者子任务之间存在复杂依赖时，对等协作就力不从心了，需要引入管理者模式。Manager Agent 的职责就像一个项目经理：先理解整体任务，再拆解为可分配的子任务，选择合适的 Agent 去执行，跟踪进度并处理异常（重试、换 Agent、调整计划），最后把各 Agent 的输出整合为最终结果。

关键的设计一招在这里：从系统设计角度看，管理者模式把每个专门 Agent 建模为 Manager 可调用的工具。Manager 的工具集中不仅有传统的外部工具（如搜索、文件操作），还包含其他 Agent 的调用接口。从 Manager 的视角看，调用一个 Agent 和调用一个普通工具没有本质区别——都是发出请求、获得响应。

二、为什么这样设计：统一抽象换来可扩展与异构

“把 Agent 当工具”不是修辞，它直接兑现两个工程红利：

  • 可扩展性：新增能力只需开发对应 Agent 并注册为工具，Manager 的核心逻辑无需修改。想加一种子任务，就多挂一个工具，编排大脑本身不动。
  • 异构性：它天然支持异构——不同 Agent 可以用不同的模型、提示词、工具集，甚至运行在不同的硬件环境上。因为 Manager 只关心“发请求、拿响应”，不关心工具内部是谁在跑。

这个抽象并非本章新造。“Agent 互为工具”在第四章“协作工具”一节已经建立（spawn_subagent / send_message / cancel_subagent 的接口设计）。第四章解决“Manager → 子 Agent”方向传什么，本章补上对称的一半：“子 Agent → Manager”方向返回什么——答案就是上文那句结构化摘要而非全量轨迹。两章分工是：第四章讲机制（工具接口与上下文传递的实现），本章讲架构（拓扑结构与职责如何划分）。

三、边界与瓶颈：最强的模型要给规划者

中心化的好处伴随一个固有代价——Manager 成为系统的单点瓶颈：它必须理解所有子任务的性质、选对 Agent、准确传递上下文，任何决策偏差都会影响整体流程。这不是空谈，2025 年的 Plan-and-Act 论文对此做了实证分析。

在 Planner-Executor 双 Agent 架构中，弱规划者是整个系统最关键的瓶颈。当 Planner 的规划质量足够高时，即使 Executor 比较简单也能取得好结果；反之，如果 Planner 的任务分解有误，后续所有 Executor 的工作都建立在错误的前提上。由此得到本课的核心处方：应当将最强的模型和最精心设计的提示词分配给 Manager（规划者），而不是平均分配给所有 Agent。

四、一个看似矛盾之处（易错点）

你会立刻想起上一课的论断：提议者与审核者的能力应当相近。现在又说“最强模型给规划者”——初学者很容易觉得这两条自相矛盾，于是随手挑一条丢掉。这是误读，它们讨论的是两件不同的事。

前者针对审查场景：审查者必须跟得上被审者的推理，才可能发现其中破绽，能力落差太大就根本审不动。后者针对规划与执行的分工：规划者一旦把任务分解错，执行者再强也无从补救，所以最强的模型和最精心的提示词应当优先给规划者。至于执行者之间是否需要能力均衡，则取决于子任务的耦合程度——当多个执行者的产物最终要拼装成一个整体时，最弱的一环往往会拖累整体质量。分清“审查”与“分工”两个语境，矛盾就消失了。

本课唯一要带走的判断：管理者模式的全部力量来自一个抽象——Manager 把每个 Agent 当工具调用（换来可扩展性与异构性）；而它的全部风险都集中在 Manager 这个单点上——弱规划者是瓶颈，最强的模型和最精心的提示词要优先给规划者，别平均分。“审查者能力要相近”与此不矛盾，前者说审查、后者说分工。

五、关键定义

共享 / 不共享上下文（Shared / Non-shared Context）
多 Agent 的第一设计维度。共享指后一 Agent 继承前一 Agent 的完整轨迹（隐式协调、信息零损耗、上下文膨胀）；不共享指各 Agent 维护独立上下文，仅通过工具参数 / 共享文件 / 消息总线三种显式机制通信。避免与“多轮对话历史”混同——这里区分的是 Agent 之间是否继承彼此的完整轨迹。管理者模式属于不共享上下文一族。
移交（Handoff / 移交包）
由上游 Agent 主动决定把哪些信息交接给下游。移交包通常含任务描述、已确认的事实与约束、结构化产物的引用（文件路径而非内容），刻意不传全量轨迹。避免理解成“把完整历史一并交出”——那是共享上下文下的形态。子 Agent 向 Manager 回传的结构化摘要就是这一思想的落地。

六、自测题

Q1：本章说“审查者能力应与被审者相近”，又说“最强模型应给规划者”，这两条不矛盾吗？

不矛盾，它们讨论的是两件事。前者针对审查场景：审查者要跟得上被审者的推理才能发现破绽，能力落差太大就审不动。后者针对规划-执行分工：规划者一旦把任务分解错，执行者再强也无从补救，所以最强模型和最精心的提示词应优先给规划者。

Q2：从系统设计角度，管理者模式如何看待“调用一个子 Agent”？这个抽象带来了哪两个好处？

它把每个专门 Agent 建模为 Manager 可调用的工具——调用一个 Agent 和调用一个普通工具没有本质区别，都是发出请求、获得响应。两个好处：可扩展性（新增能力只需开发对应 Agent 并注册为工具，Manager 核心逻辑无需修改）与异构性（不同 Agent 可用不同模型、提示词、工具集，甚至跑在不同硬件上）。

本课属架构讲解，规格未绑定独立配套目录；但第十章的实验 10-3 书籍翻译 Agent正是管理者模式的标准范例——Manager 顺序调度 Glossary / Translation / Proofreading 三种子 Agent，且“只维护文件索引、不保存完整翻译内容”，亲手演示了 Manager 如何靠结构化摘要把自身上下文压在可管理范围内（对应图10-6）。建议边读边跑，全书实验都在开源仓库：github.com/bojieli/ai-agent-book。
