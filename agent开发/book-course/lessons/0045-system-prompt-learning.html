系统提示学习：一个边界案例，就改一条明文规则

本课只押一件事：为什么“直接改系统提示词”这种学习，数据效率会显著高于结果奖励型强化学习——不是因为“改文本比改参数简单”，而是因为反馈通道的信息带宽差了好几个数量级。它出自第 8 章“系统提示词的自动优化”一节，是 Karpathy 提出的“系统提示学习（System Prompt Learning）”。把这条带宽直觉吃透，你就懂了为什么它能从单个边界案例立刻长出一条明文规则。

一、是什么：被遗漏的第三种学习方式

作者引用 Karpathy 的观点：当前 LLM 的学习范式里漏掉了一种重要方式。已有的两种都要动模型参数——预训练是为了获取知识，微调是为了培养习惯性行为。但人类的很多学习其实更像“系统提示”的更新：想通了一件事，就用明确的语言给自己记下来，比如“下次遇到这种类型的问题，应该先试这种方法”。

Karpathy 打了个比方：LLM 就像电影《记忆碎片》的主角——每次醒来都不记得之前发生了什么，而我们还没给它们提供记录本。他读过 Claude 的系统提示词（约 1.7 万个单词，随版本变化）后发现，里面藏着大量通用问题解决策略，例如“计算单词、字母、字符时，先给每个字母分配数字再显式计数”——就是为了应对“strawberry 里有几个 r”这类问题。作者的判断是：这类知识不该由人类手工编写，而应来自系统提示学习。

系统提示学习直接修改系统提示词文本，强化学习则通过梯度下降调整模型参数。 ——《深入理解 AI Agent》第 8 章

换句话说：它和强化学习相通——都靠失败案例改进未来行为；但两者的“学习算法”完全不同。RL 是梯度下降改权重；系统提示学习是改一段文本，不碰任何参数。这也是它归在本章“不改权重的进化”这条主线下的原因。

二、为什么它数据效率碾压结果奖励型 RL

这是本课的核心“为什么”，也是最容易答偏的地方。原因不是“文本好改”，而是反馈通道的“维度”差异。这是 Karpathy 针对结果奖励型强化学习的批评：一次只给一个标量的结果奖励（如“对/错”），信息带宽远低于一整段自然语言复盘。

带宽差距直接翻译成样本需求：RL 要在海量数据上反复试错才能把权重挪动一点点；系统提示学习遇到一个失败案例，就能立即在系统提示里加一条明确规则，不需要收集数千个类似样本去微调。而且它完全可解释——每条规则都是明文，能审查、能修改、能删除。作者也用实证背书：2025 年的 GEPA 做的正是这种“系统提示学习”，它对失败轨迹做自然语言反思来进化提示词，在多项任务上超过了 GRPO 微调，而所用采样次数少一到两个数量级。

三、本质：靠边界情况让规则边界变清晰

作者进一步给出自己的判断：系统提示学习的本质，是通过边界情况（edge cases）让规则的边界更清晰。大多数规则在典型场景下都运作良好，真正的挑战在灰色地带——“在用户请求超出能力范围时转接人工”听起来很清晰，但“用户不满意政策”算不算超出范围？用户要求例外处理呢？正是这些边界情况，才定义了规则的真正含义。

于是随着边界情况不断积累，系统提示词就逐渐演化成一本详尽的“问题处理手册”，就像一位专家在工作中不断完善自己的笔记。

四、怎么落地，以及最容易想错的地方

如何自动化？作者的方案是引入 Coding Agent：系统提示词和工具描述本身就是文档和代码，散落在多个文件里。发现边界情况时，Coding Agent 阅读现有提示、分析规则结构，生成精确的代码级 diff（在哪个文件、哪个位置、改什么），并保证不引入矛盾或冗余；最终审核权仍在人类专家手中。作者还对比了 DSPy、OPRO、GEPA 等自动化框架：它们多在离线评估集上批量优化、端到端自动改写，而“diff + 人工审核”方案随生产环境的边界案例逐条演进，只需要单个失败案例加一段人类反馈——两者可以互补。

被问到“系统提示学习为什么数据效率高”，最常见的错误答案是：“因为改文本比改参数简单。”这答的是实现代价，不是学习效率。作者点明的根本原因是反馈通道的信息带宽：结果奖励型 RL 一次只吐一个标量（对/错，约 1 bit），而一段自然语言复盘承载的信息量丰富得多。同样一次失败，前者吸收 1 个比特，后者吸收一整段可执行的诊断——所以后者能从单个案例立刻学到东西。记住：差的是带宽，不是改法的难易。

另一个易混点：别把“往系统提示里写规则”当成纯收益。凡是被沉淀进长期规则/知识的东西，都要过人类审查这道闸门——否则恶意注入的指令一旦被写进规则，就从一次性攻击变成跨会话持久生效（记忆与经验投毒）。这正是作者坚持“diff 由人类审核”的原因。

与本课“反馈信息量 > 标量奖励”这条判断最直接呼应的配套实验是 chapter8/feedback-guided-sampling/（Feedback-Guided DAPO）：它用同一批内失败 rollout 的 critique 反馈去引导后续 rollout，从而提升 RL 的样本效率，并以“客服打电话、需从 CSR 反馈中学到认证要求”的场景做验证——把“自然语言反馈比一个标量更有信息量”这件事，直接搬进了 RL 采样里。想看系统提示学习本身的端到端流程，可读原文实验 8-3（★★，系统提示词的自动优化）：基于 tau-bench 航空客服，用人类反馈把“过度转接”这一边界案例改成一条明文规则。全书代码见 github.com/bojieli/ai-agent-book。

五、关键定义

系统提示学习（System Prompt Learning）
Karpathy 提出的学习方式：直接修改系统提示词文本，从边界案例快速学习明文规则，不改模型参数。避免与微调 / RL 混为一谈——它不做梯度下降，学习算法就是“改文本”。
外部化学习（Externalized Learning）
把知识沉淀到模型之外的文件、知识库和工具中的学习范式，持久、可解释、可随时修正。系统提示学习是它的一个分支（经验外部化为规则文本）。避免等同于“记忆”或泛泛的“持久化”，它特指不改权重的第三条学习路径。
记忆与经验投毒（Memory Poisoning）
Agent 接触到的网页文本 / 工具输出里的恶意注入指令，若未经审查被沉淀为长期记忆或规则，攻击就从一次性变为跨会话持久生效。避免与会话内的一次性提示注入混同——投毒污染的是 Agent 的“知识”本身，更隐蔽、更难清除。

六、自测题

Q1：Karpathy 说系统提示学习的数据效率显著高于结果奖励型强化学习，根本原因是什么？

是反馈通道的“维度 / 带宽”差异。结果奖励型 RL 一次只给一个标量（对 / 错），信息带宽极低；而一整段自然语言复盘（“这里应该先验证身份证再走退款流程”）信息量丰富得多。同样一次失败，系统提示学习能吸收的信息远多于一个比特，因此可从单个 / 少量边界案例立即添加明文规则，无需收集数千样本微调，且完全可解释、可审查修改。（注意：不是“改文本比改参数简单”。）

Q2：作者认为系统提示学习的“本质”是什么？为什么说“边界情况”才定义了一条规则的真正含义？

本质是通过边界情况（edge cases）让规则边界更清晰。大多数规则在典型场景下都运作良好，真正的挑战在灰色地带：一条“请求超出能力范围时转接人工”的规则，遇到“用户不满意政策算不算超范围”“用户要求例外处理”这类情形才暴露出它到底覆盖什么。所以每处理一个边界案例、补一条明文规则，系统提示词就更精确一点，最终演化成一本“问题处理手册”。

本课唯一要带走的判断：系统提示学习之所以数据效率碾压结果奖励型 RL，差的是反馈带宽（一段自然语言复盘 vs 一个标量比特），不是改法难易；正因带宽足够宽，它能从一个边界案例立刻长出一条明文、可审查、可回退的规则，让系统提示词逐步长成一本“问题处理手册”。

七、主推资源

本节的“系统提示学习”概念来自 Andrej Karpathy；配套的自动化框架有 DSPy、OPRO 与 GEPA（arXiv:2507.19457）——后者用自然语言反思进化提示词，超过 GRPO 微调且采样次数少一到两个数量级，是“反馈信息量”判断的实证。本章更宏观的“最小预定义、最大自我进化”原则出自 Alita（arXiv:2505.20286）；不改权重反思路线的经典是 Reflexion（arXiv:2303.11366）。一手来源仍是书的第 8 章 book/chapter8.md“系统提示词的自动优化”一节。
