从能跑的 Demo 到可靠产品：Harness 工程

前八课把 Agent = LLM + 上下文 + 工具 讲透了——那是「能跑一个 Demo」的最小配方。但书的下半章（“Harness 工程：模型之外的竞争力”）立刻泼了盆冷水：一个能跑的 Demo 和一个可靠的产品之间，还隔着一道鸿沟。这节课只押一个生产公式——Agent = Model + Harness——并把它拆成两句人话：「能做事」靠上下文 + 工具，「不做错事」靠约束 + 验证 + 纠正。这就是本章后半段全部内容的骨架。

一、是什么：同一个系统，两个视角

作者特意提醒：Harness 公式不是要替换前面的核心公式，而是换一个抽象层次去看同一个系统。Agent = LLM + 上下文 + 工具 描述的是 Agent 的内部组成（大脑、眼睛、手脚分别由什么承担）；而 Harness 工程给的是一个工程实现视角——把 LLM 当作一个核心组件（Model），围绕它构建的所有支撑代码统称为 Harness。之所以换用更通用的 “Model” 一词，是因为这套原则适用于任何具备推理和工具调用能力的模型，不绑定某一类模型。

Agent = LLM + [上下文 + 工具 + 约束 + 验证 + 纠正] = Model + Harness
最小公式是 Demo 视角，扩展公式是生产视角——后者完全包含前者，只是在外围加了一圈安全网

关键要读懂这个方括号：约束、验证、纠正不是三个新增的「独立模块」，而是围绕原公式里“上下文 + 工具”构建的一层保障。最小可工作的 Agent 只需 LLM + 上下文 + 工具就能跑；要在生产环境长期可靠运转，才需要补全外面这三层工程外壳。把它想成书里那个马具（Harness）隐喻：模型是一匹强壮但不可预测的马，Harness 是缰绳与挽具——不是限制它奔跑，而是把力量引导到正确方向。

书里把五个功能各自的一句话职责列成了表——记住这张表，就等于记住了 Harness 的全貌：

功能一句话职责与上下文/工具的关系
上下文 Context为模型提供感知信息核心能力
工具 Tools为模型提供行动手段核心能力
约束 Constrain设定行为边界——能做什么、不能做什么围绕上下文和工具构建的安全边界
验证 Verify自动判断操作结果的对错围绕工具执行结果构建的检查机制
纠正 Correct发现问题时自动修正或回退围绕工具调用失败构建的恢复机制

二、为什么这样切：一个退款例子看清「有无 Harness」

作者用同一个任务把这条鸿沟摆到台面上：让一个 Agent 帮用户退掉 3 天前的订单。同一个模型，有没有 Harness，结果天壤之别。

把这个例子读慢一点，你就能分清两组词的分工。作者原话把边界划得很清：“上下文中嵌入退款政策是上下文的范畴，而校验退款金额不超过订单金额则属于约束；工具执行 API 调用是工具的范畴，而 API 超时后自动重试则属于纠正。”上下文与工具让 Agent 能做事，约束、验证、纠正让 Agent 不做错事。

为什么书要花这么大力气强调后三者？因为在成熟度曲线上，两者的重要性是不对称的。早期框架主要在给模型工具、给上下文，让它“能做事”；而生产级系统的重心已经转向约束、验证、纠正。作者举了 Claude Code：它的 Harness 里绝大部分代码都是约束、验证与纠正（流程状态管理、多层上下文压缩、权限分类、熔断器、错误恢复），工具本身（文件读写、命令执行、搜索）只占一小部分。一句话——行业正在从“能做事”转向“可靠地做事”，Harness 工程因此成为核心竞争力。

当各家模型的能力越来越接近、不再是决定性的差异因素时，竞争优势就转移到了模型之外的工程实践。 ——《深入理解 AI Agent》第 1 章「从提示工程到 Harness 工程」

作者还给了一条清晰的演进弧线：软件工程 → 提示工程 → 上下文工程 → Harness 工程。这四层不是替代关系，而是层层包含——提示工程是上下文工程的子集，上下文工程是 Harness 工程的子集。每上一层，工程师的关注范围就从“模型能看到什么”扩展到“模型在什么样的系统里运行”。

三、边界与易错：最容易把五要素归错格子

误区一：把「校验退款金额」当成验证。很多人一看到“校验”两个字就归到验证。但按书的定义，校验退款金额不超过订单金额是「约束」——它在事情发生之前限定行为边界（能做什么、不能做什么）。而“验证”是事后检查操作结果对不对（比如查数据库确认退款是否真的成功）。判断口诀：约束是画红线（能不能做），验证是对答案（做得对不对）。

误区二：把「自动重试」当成约束。API 超时后自动重试属于「纠正」——它是工具调用失败之后的恢复机制，不是事前的边界限定。约束防越界、验证发现错误、纠正恢复异常，三者在时间线上各占一段，别混。

误区三：读到「模型即 Agent」就以为框架/Harness 不重要了。恰恰相反——书里反复强调：模型越强，Harness 越关键。因为模型自主决策的空间越大，出错时的影响面也越大，越需要精细的约束、验证和纠正来兜底。模型厂商的真正优势不是“让框架变薄”，而是能对模型与外围 Harness 做协同优化。

本课属概念地图，直接配套的动手实验是上一课已引入的实验 1.1 上下文消融（对应仓库 chapter1/context/）：一组全组件基线 + 四组各缺一个组件的对照，用成功率/耗时/迭代/工具调用数实证每个上下文组件的作用。它正是“能做事”那半边——上下文——的地基。整本书代码见开源仓库 github.com/bojieli/ai-agent-book，强烈建议边读边跑。

四、关键定义

Harness（工程外壳/马具）
模型之外的全部基础设施，核心是上下文与工具，外加约束、验证、纠正三层保障机制；把模型的能力引导、约束并放大为可靠的任务执行。避免简单等同于“框架”或“SDK”——它是围绕模型的整套工程外壳，不限于某个具体框架。
模型即 Agent（Model as Agent）
先进模型通过后训练（尤其 RL）把工具调用内化为原生能力，何时调、调哪个、传什么参数都由模型自主决定、无需人工编排层的新范式。避免理解为“框架不再重要”——恰恰相反，模型越强 Harness 越关键。
防呆（Poka-yoke）
源自丰田生产体系的设计思路：从设计上让错误无法发生（如 USB 只能单向插入），应用到工具接口（ACI）设计中主动防止模型误用。避免把它当成“错误处理/异常捕获”，它是从接口设计上消除出错可能。

五、自测题

Q1：让 Agent 帮用户退 3 天前的订单，「校验退款金额不超过订单金额」和「API 超时后自动重试」分别属于 Harness 五要素的哪一个？

「校验退款金额不超过订单金额」属于约束（限定能做什么、设定行为边界，是事前的红线）；「API 超时自动重试」属于纠正（工具调用失败后的恢复机制，是事后的补救）。注意别把前者误归为“验证”——验证是事后核对操作结果对不对，比如查数据库确认退款是否真的成功。

Q2：书里说“模型越强，Harness 越关键”，这和「模型即 Agent」趋势不是矛盾吗？为什么两者能共存？

不矛盾。「模型即 Agent」说的是模型把工具调用决策内化成了原生能力，不再需要人工编排层；但模型自主决策的空间越大，一旦出错，影响面也越大，因此越需要精细的约束、验证、纠正来保证可靠性。所以模型变强，恰恰把竞争优势推向了模型之外的工程实践——这也正是“各家模型能力趋同后，竞争优势转移到 Harness”这一判断的由来。模型厂商的真正优势是能对模型与外围 Harness 做协同优化。

本课唯一要带走的判断：Agent = Model + Harness。上下文 + 工具让它能做事，约束 + 验证 + 纠正让它不做错事——后三者不是独立模块，而是围绕上下文与工具长出的安全网。判断任何一个 Harness 要素时，先问它发生在时间线的哪一段：事前画红线是约束，事后对答案是验证，出错后补救是纠正。模型越强，这层外壳越是胜负手。
