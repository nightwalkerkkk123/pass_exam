读懂 Benchmark 报告：总分之下的结构性短板与三层假设

这节课只押一件事：Benchmark 报告的价值，不在“总体成功率”这个单一数字，而在它揭示的结构性短板。你要学会交叉读两张表——逐任务表格 + 能力标签矩阵——把散落的任务失败定位到少数几个能力缺陷；再把改进按表层 / 中层 / 深层三层假设分级（成本递增、深层条件化启动），最后用成本收益比而非“有效就上”来决定部署谁。这是第 6 章“从 Benchmark 报告到系统改进”一节的完整闭环，也是把 Agent 工程从经验驱动推向科学工程的关键一步。

一、先看病灶，别看总分：两张表怎么交叉读

书里用了一个示教性的假设案例（作者反复强调“数据均为假设值，旨在演示方法论”）：在 AndroidWorld 上评估某个 Agent，拿到两张核心报告表——逐任务性能列表和按能力标签划分的性能矩阵。

逐任务表格显示了一个清晰的模式：大部分常规任务（录音、拍照、联系人管理、笔记、文件操作、系统设置）成功率接近 100%，平均要十几步、最复杂可达数十步——证明 Agent 在标准场景下规划与执行没问题。失败高度集中在少数区域：短信回复、Wi-Fi 开关及状态验证、待办事项查询、Wi-Fi+蓝牙组合、VLC 播放列表创建。表面上这些任务彼此无关。

二、为什么要分三层：把改进按“验证成本”排队

诊断出病灶后，作者不是一股脑把所有改法都试一遍，而是把假设按验证成本从低到高组织成三层——这就是“构建改进路线图”的核心设计。为什么要分层？因为深层改动（换模型、加信息源）既贵又慢，只有当便宜的手段用尽、短板仍在时，才值得动它。

三层的具体内容（全部来自书中假设案例）：

  • 表层（成本低、独立、可并行）：H1 给 Wi-Fi 操作加系统设置导航提示；H2 给待办应用提供 UI 元素识别规则。
  • 中层（同样独立、可并行）：H3 修复多模态输入管道（回放轨迹发现图片被丢弃或转成文本，再强的多模态模型也无从转录）；H4 全局启用思考(reasoning)以解决计数失败。
  • 深层（验证成本高、条件启动）：H5 换更强视觉模型（GPT-5）；H6 在截图之外增加 UI 元素树信息（UI Automator 的结构化 DOM 与截图交叉验证）。两者组成 2×2 对比实验（Claude/GPT-5 × 仅截图/截图+元素树），回答“模型能力和信息丰富度哪个更关键、是否有协同”。

三、从结果到决策：为什么“有效”不等于“该部署”

每个配置在完整 116 个任务上跑 5 次（不同随机种子）。假设实验结果出来后，作者的决策逻辑不是“凡有效即采用”，而是逐项权衡收益与它拖累的护栏指标（延迟、成本、token）：

部署与否 = 目标指标收益 − 护栏指标代价 × 受影响任务占比
收益要摊到“真正受益的任务”上，代价却常常摊到“全部任务”上
  • H1+H3 明确部署：H1 成本低、收益高、无副作用（设置类 0%→75%，输入 token 仅 +8%）；H3 虽增加 15% vision token 和 1 秒/步延迟，但转录能力从无到有（0%→80%），且修复的是“输入管道丢多模态信息”这一架构性缺陷，还可能顺带改善其他视觉任务。
  • H6 优于 H5：H5(GPT-5) 每步延迟 4 秒飙到 15 秒，complex_ui 却只到 35%——说明瓶颈不在模型思考能力，而在输入信息是否充分；H6 用 +30% token、+2 秒换来 complex_ui 17%→52%（提升 35 个百分点），性价比高得多。H5+H6 组合成绩最高（68%）但时长在大规模部署中不可接受，只适合银行转账、医疗预约这类关键异步任务选择性启用。
  • H2 缓办：为每个非标准应用手写专门规则不可持续，只能作临时方案，长期应提升泛化能力。

最终实施 H1、H3、H6 三项（H4 未部署），成功率从 88% 提到 94%。重跑完整 benchmark，失败模式变了：转录/设置/复杂 UI 大幅改善，剩余约 6% 集中在未解决的计数、仍波动的 Wi-Fi 状态验证（0%→60% 但不稳定），以及少量新失败——可能是提示词变长或元素树信息过多导致注意力分散。这份新报告，就成了下一轮假设（H7 条件化思考、H8 扩展手势动作空间）的起点。

初学者最容易掉进的坑：把“提升了总体成功率”当成“该部署”的充分理由。看 H4——全局启用思考把总体成功率从 88% 提到 91%，看起来是净提升，谁不想要？

但能力标签分布显示：只有约 8% 的任务涉及计数。H4 却让全部任务都承担 3 倍延迟（每步 4 秒→12 秒）和 3 倍成本。这是典型的“杀鸡用牛刀”——护栏指标（延迟、成本）的恶化，远远盖过了目标指标那 3 个百分点的微弱收益。错在哪？错在只盯着一个聚合数字，没把收益摊回“真正受益的 8% 任务”上。正确的做法是让 H4 的洞察（思考对计数有效）演化成 H7 条件化启用：任务开始前用一次约 1-2 秒的快速 LLM 调用分析任务描述，只对涉及计数或复杂推理的任务开思考，把代价压在真正需要的地方。

四、宏观背景：这套方法论到底在改造什么

为什么值得为“怎么读一张报告”单开一课？因为它是本章“评估驱动的决策”这一层的落地形态。作者在章末点题：核心方法论是观察 → 假设 → 实验 → 验证 → 新认识 → 新假设，目的就是让 Agent 工程从经验驱动的“炼金术”转向数据驱动的科学工程。三层假设、成本收益决策、条件化启动，都是为了让“改哪里、先改哪、值不值得改”这三个问题有据可依，而不是凭手感。

还有一条容易被忽视、但书里专门强调的前置原则：看到 Agent 表现下降时，应先检查评测系统本身，再动 Agent。评测环境资源不足杀进程、评分器有 bug、测试用例与生产脱节——这些在结果数字上跟“模型退化”长得一模一样，只有审查完整轨迹才能区分。基于失真信号调方向，改的方向可能从一开始就是错的。

本课唯一要带走的判断：Benchmark 报告的价值是结构性短板，不是总分。交叉“逐任务表 + 能力标签矩阵”定位病灶，把改进按表层→中层→深层三层假设排队（深层条件化启动），再用成本收益比（收益摊给受益任务、代价摊给全部任务）而非“有效就上”做部署决策——H4“全局思考”被否、演化成 H7“条件化思考”，就是这条判断最锋利的一刀。

五、关键定义

古德哈特定律（Goodhart's Law）
当一个度量指标变成优化目标时，它就不再是好的度量指标；Agent 越在某评分系统上调优，越倾向钻系统漏洞而非真正提升能力。提醒：别把机制指标（如“缩短计划文件长度”）当成真正目标（“降低会话成本”）。
模型替换实验（Model Swap）
固定 Harness、只更换更强或更弱的模型，观察分数变化幅度，用以区分性能瓶颈在模型能力还是在 Harness 设计。避免与消融实验混用——消融是关闭 Harness 组件，模型替换是换模型。本课 H5 换 GPT-5 就是一次模型替换实验。
Pass^k（Pass hat k）
k 次尝试全部成功的概率，回答“Agent 是否稳定可靠”，用于回归测试等看重稳定性的场景。切勿与 Pass@k（至少一次成功）混淆，二者衡量方向相反。

六、自测题

Q1：AndroidWorld 假设案例中，全局启用思考(H4)把总体成功率从 88% 提到 91%，为什么最终被否决、演化成条件化启用(H7)？

因为能力标签分布显示只有约 8% 的任务涉及计数，H4 却让全部任务承担 3 倍的延迟和成本——典型的“杀鸡用牛刀”。护栏指标（延迟、成本）恶化超过了目标指标的微小收益。但 H4 证明了思考对计数任务有效，于是演化为 H7：任务开始前用一次快速 LLM 调用分析任务描述，只对需要计数/复杂推理的任务开启思考，把代价控制在真正需要的任务上。

Q2：为什么深层假设（H5/H6）不和表层、中层假设一起并行跑，而是设了一个“启动条件”？

因为深层假设验证成本高（换模型、加信息源，延迟和 token 开销都大），不值得无条件地投入。书里给的启动条件是：仅在表层和中层改进后，complex_ui 成功率仍低于 40% 时才启动。也就是说，先用便宜的手段把能解决的短板解决掉，如果复杂界面理解这个硬骨头还在，才动用昂贵的深层实验。这正是三层假设“按验证成本从低到高排队”的设计意图——把贵的实验留到确实必要时。
