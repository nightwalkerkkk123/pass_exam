一个 while 循环，就是 Agent 的全部

本课只押一件事：亲手推演 messages 列表怎么一条条长起来。用书里那个“温哥华现在几点、天气如何”的例子，走两轮工具调用，你要能说清三件事——四种消息角色各是谁写的、模型只决策而框架才执行、以及每次调用为什么必须把完整历史原样送回去。这是第 2 章《上下文工程》一上来就讲的“Agent 如何调用大模型”，是后面 KV Cache、压缩、状态栏所有技术的地基。地基不牢，上面全塌。

一、四种角色：一张消息列表，四个来源

大模型 API 的核心是一个消息列表（messages），每条消息带一个 role 标识，模型靠它判断这条话是谁说的、算什么。书里点名了四种角色：

  • system：系统提示词，开发者写，定义身份与规则，模型视为最高优先级。通常只有一条，放最前面。
  • user：终端用户的输入，是 Agent 要响应的请求。
  • assistant：模型之前的回复——可能是文本，也可能是一组工具调用请求（tool_calls）。多轮里它会被放回列表，让模型“记住”自己说过什么。
  • tool：工具结果，由 Agent 框架执行后生成，每条用 tool_call_id 与对应的调用请求关联。

记住这四个来源，你就能给 messages 列表里任意一行“认领作者”——这正是下面那张增长图要考你的。

二、核心循环：模型只决策，框架才执行

单轮问答（system + user → assistant 文本）太简单。真正的 Agent 场景是：用户问“温哥华现在几点、天气如何”，模型答不了（它不知道“现在”），于是不直接回答，而是返回两个工具调用请求（content: null，带 tool_calls）。书里把最关键的一句话加粗了：模型只是发出调用请求，真正执行工具的是 Agent 框架。模型负责决策（调哪个工具、传什么参数），框架负责执行（真的去调 API、跑代码）。对应图2-3那条完整交互序列。

四、为什么第二次调用要把整段历史原样送回（深挖“为什么”）

这是本课最反直觉、也最容易栽的一处。书里那句加粗结论是理解一切的钥匙：每次调用都是无状态的。模型不会在服务端“记住”你上一次说过什么——它没有记忆，只有“这一次请求里你给我看的这些消息”。

所以第二次请求必须包含第一次的全部历史：system、user、第一次那条 content: null 只带 tool_calls 的 assistant 消息，再加上新的 tool 结果。书里点出三个关键细节：完整历史都要回传；第一次的 assistant 消息要原样放回（让模型看到自己之前做了什么决策）；tool 消息靠 tool_call_id 认领它对应哪次调用。

初学者最常犯的错：觉得“工具结果我送回去不就行了吗，那条 content: null、看起来什么内容都没有的 assistant 消息，省掉呗”。错。

那条 assistant 消息虽然文本是 null，但它承载着“模型做了 get_current_time 和 get_weather 两个调用”这个决策本身，而且它带着 call_abc123、call_def456 这两个 id。把它删掉，后面两条 tool 结果就成了无主的孤儿——tool_call_id 指向一个列表里根本不存在的调用，模型无从知道这些数字对应谁，也就没法把“决策→观察”串成一条连贯的 ReAct 轨迹。无状态意味着：你不主动放回，历史就真的不存在。

把这三条内化，你就理解了全书的底层设定：Agent 框架的核心工作，就是管理这个 messages 列表——在对的时机往里追加对的消息，再把整个列表送出去。后面所有上下文工程技术（KV Cache 复用、压缩、状态栏），本质上都是在优化这一个列表的内容与结构。而“前缀不变、轨迹追加”的形状，正是下一课 KV Cache 能省钱的前提。

本课唯一要带走的判断：Agent 的运行 = 一个 while 循环反复给模型送整段 messages；模型只决策（返回 tool_calls 或文本），框架只执行（跑工具、往列表尾部追加结果）；因为调用无状态，每一轮都必须把完整历史（含那条只带 tool_calls 的 assistant）原样回传，tool 结果靠 tool_call_id 认领归属。看到任何 Agent，先问：它的 messages 列表这一刻长什么样、下一条会由谁追加。

五、关键定义

上下文工程（Context Engineering）
系统性设计、组织并向模型提供完成任务所需全部背景信息的工程实践。管理 messages 列表只是它最基础的一层。避免把它等同于“提示工程”或“往 prompt 里塞更多内容”——它是更上位的系统性信息供给设计。
无状态调用（stateless）
大模型 API 每次调用彼此独立，模型不在服务端保留上一次的对话；所有它需要的信息，必须在本次请求的 messages 列表里完整提供。这是“每次都要回传完整历史”的根本原因。
tool_call_id
模型在一条 assistant 消息里发出工具调用请求时给每个调用分配的 id（如 call_abc123）；框架执行后，把结果放进一条 tool 消息并用同一个 id 标注，模型据此知道哪个结果对应哪次调用。避免省略那条带 tool_calls 的 assistant 消息，否则 id 无处对应。
Chat Template
把结构化的 role-content 消息，转换成模型训练时见过的线性 token 流的模板（用 <|im_start|>、<|im_end|> 等特殊标记划分角色边界），由 API 服务端自动完成。避免理解为开发者要手写、或与 system prompt 混淆。

六、自测题

Q1：Agent 第二次调用大模型时，为什么要把第一次那条 content 为 null、只带 tool_calls 的 assistant 消息原样放回消息列表？如果省略它会怎样？

因为 API 调用是无状态的——模型不会“记住”上一次说过什么。原样放回第一次的 assistant 消息，是让模型看到自己之前做了哪个工具调用决策，而随后的 tool 消息通过 tool_call_id 与之关联。省略它，tool 结果就成了无主的孤儿，模型无法知道这些结果对应哪次调用，也无法把决策与观察串成连贯的 ReAct 轨迹。

Q2：在带工具调用的交互里，一条 tool 角色的消息是谁生成的？它凭什么和某一次具体的工具调用请求对应上？

由 Agent 框架生成——框架实际执行了工具（如调时间/天气 API）后，把返回结果包成一条 role: "tool" 的消息。它靠 tool_call_id 与对应的调用请求关联：模型在 assistant 消息里发出调用时给了每个调用一个 id，框架回填结果时带上同一个 id，模型据此对号入座。（注意：模型只发决策，执行始终是框架的活。）

配套实验 2-1（对应仓库 chapter2/local_llm_serving/）：用标准 OpenAI 兼容 API 在本地（vLLM/Ollama）跑一个 0.6B 小模型的工具调用，亲眼看到 API 层看不见的原始 token 流——<think> 思维链、特殊标记、工具调用格式，以及流式响应。它正好把本课的“请求→工具调用→执行→送回→再请求”循环跑给你看，还能顺带观察 KV Cache 对首 token 延迟（TTFT）的影响，为下一课埋线。完整代码见 github.com/bojieli/ai-agent-book。
