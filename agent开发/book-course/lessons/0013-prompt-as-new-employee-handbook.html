把系统提示词当新员工手册：流程驱动，别堆规则

这节课只押一个判断：系统提示词的价值不在“写了多少条规则”，而在“规则以什么结构被送到模型面前”。作者给的检验标准是——大语言模型是一位聪明但一无所知的新员工，如果它读完你的系统提示词还不知道怎么做，Agent 也一样不知道。本课取自第 2 章「提示工程：优化系统提示词」一节，用书里“账单谈判 Agent”的例子把“模糊规则 → 行为不可预测”这条因果讲穿，并给出流程驱动、业务规则细化、few-shot 何时用的判断标准。

一、系统提示词是什么：Agent 的“员工手册”

提示工程（Prompt Engineering）的核心对象，是 API 消息列表里那条 role: "system" 的消息——系统提示词（System Prompt）。它定义了 Agent 的身份、行为规则、约束条件和工作流程，模型把它视为最高优先级的指令。作者一句话点题：它就是 Agent 的“员工手册”。

作者从四个维度拆解如何写好它，本课先记住前两个作为地基：

  • 语气与风格（“人格”）：例如 “You MUST answer concisely with fewer than 4 lines”，并要求“不要解释为什么不能做某事”，避免 Agent 陷入冗长的自我辩护。大写的 “NEVER do X” 比 “Please avoid doing X” 更能引起模型注意——但过度使用会稀释效果，要留给真正关键的约束。
  • 结构化（“格式”）：现代模型对结构化输入高度敏感，因为训练数据里就有大量结构化内容。XML 标签名本身携带语义——<working_directory> 一眼就知道是工作目录，而纯文本“当前目录：/Users/project/src”还要模型自己解析冒号前后的关系。XML 负责机器可解析的精确语义，Markdown 负责人机共读的组织逻辑，两者协同成双层结构。

二、为什么“流程驱动 vs 规则堆砌”是分水岭

作者提出一条贯穿全章的原则：对人类降低认知负担的方法，对大语言模型同样有效——因为模型在训练中学习了人类的语言和思维模式。

三、业务规则细化：账单谈判 Agent 的三种计费模式

作者说，构建生产级 Agent 时最容易被忽视却最关键的一环，是业务规则的细化——这不是技术问题，而是产品设计问题，需要产品经理深度参与。他用一个“帮用户打电话砍账单”的 Agent 举例：产品经理的诉求是“办不成就退款、防止薅羊毛”，于是设计了三种计费模式。

如果规则模糊成一句“根据任务情况选择合适的计费类型”，Agent 的行为就会极不稳定：“帮我退掉上个月买的衣服”——这算“帮用户省钱”还是“取回本属于他的钱”？“帮我取消 Netflix”——取消让用户未来不再付费，算“省钱”吗？同样的任务在不同时间会被分到不同的计费模式，业务逻辑变得不可预测。

作者给出的解法，是把决策规则明确到可执行的程度：

  • 按提成计费仅限“通过谈判降低现有账单”的场景；退款和取消绝对不能按提成。提示词里要直接写死：NEVER use percentage_based_one_time for refunds and service cancellations. Use fixed_fee instead.
  • 成功率按固定流程分步评估，估出的概率直接映射到计费模式（如 >60% 用可退款模式、<30% 直接拒绝）。
  • 金额计算把粒度写死（如通话按每分钟 $0.05、汇总后四舍五入到整美元），并明确“节省只基于现有账单计算”——否则模型会自作聪明：“不砍价明年涨到 $180，我维持 $150 就省了 $30”，把避免未来涨价也算成省钱。

初学者最常想错的一步：“模型这么聪明，业务规则模糊点没关系，它会自己判断。”恰恰相反。作者的设计哲学是：大语言模型的优势在于遵循复杂指令、从长上下文提取信息，但不应在业务规则制定上被赋予过多的自由裁量权。把“退衣服算不算省钱”这种判断留给模型，等于把系统行为交给掷骰子。好的新员工培训不是“你很聪明，自己看着办”，而是给一套详细 SOP，让员工在明确框架内发挥能力。所以在优秀的 Agent 公司里，提示词一般由产品经理设计、基于线上数据迭代；工程师负责把规则准确编码进提示词，不应擅自决定业务逻辑。

四、few-shot 何时该给例子——以及它藏着的缓存陷阱

除了规则和流程，示例（few-shot examples）是系统提示词里另一类重要内容。判断标准很清晰：

  • 该用：当期望输出难以用规则精确描述时——特定风格的文案、结构化报告格式、客服回复的语气分寸——与其堆冗长的文字定义，不如直接给两三个高质量的输入-输出示例，让模型的上下文学习能力“临时学会”这些模式。
  • 不该用：对于模型本来就擅长、规则又容易说清的任务，示例只是浪费 token。数量也不是越多越好——两三个覆盖边界情况的精选示例，通常胜过十个大同小异的，后者还会稀释模型对规则本身的注意力。

一个反直觉的工程陷阱：按请求动态检索“最相关”的 few-shot 示例。听起来很智能，实则灾难。无论示例放在系统提示词里、还是伪造成首轮 user/assistant 消息，它都处于上下文靠前的区域；一旦每次请求都换一批示例，等于每次都改写前缀，KV Cache 会持续失效（上一课 0012 讲过：改前缀 = 缓存全废）。所以生产系统通常为每一类任务准备一组固定示例，一旦确定就保持字节级稳定，而不是逐请求挑选。

本课唯一要带走的判断：写系统提示词时，先问自己一句“一个聪明的新员工读完这段，知道该干什么吗？”——把散乱规则重组成编号 SOP、把每条业务规则细化到可执行、无歧义（模糊即不可预测）、把 few-shot 当作固定的静态前缀而非动态检索项。内容多少是次要的，组织结构才是成功率的分水岭。

五、关键定义

系统提示词（System Prompt）
API 消息列表里那条 role: "system" 的消息，定义 Agent 的身份、行为规则、约束与工作流程，被模型视为最高优先级指令。整个对话通常只有一条，放在消息列表最前面。作者的检验标准：聪明的新员工读完它若还不知道怎么做，Agent 也不知道。
流程驱动 / SOP（标准操作流程）
把系统提示词的规则组织成有优先级、有依赖、有编号步骤的操作流程，让模型任何时刻都知道自己处于哪一步、目标是什么、下一步去哪，而不必遍历所有规则找匹配项。与之相对的是“规则堆砌”——一堆无序、无优先级的规则集合。
Few-shot 示例
系统提示词里给出的少量输入-输出示例。当期望输出难以用规则精确描述时，靠模型的上下文学习能力从示例中临时习得模式，往往胜过等量篇幅的抽象规则。避免按请求动态检索示例——那会破坏 KV Cache 前缀稳定性。
上下文工程（Context Engineering）
系统性设计、组织并向模型提供完成任务所需全部背景信息的工程实践，是 Harness“上下文与工具”层的核心实现；既是技术问题，更是组织问题。本课的提示工程只是它的一个切面。

六、自测题

Q1：消融实验里，只是“保留全部规则内容但打乱组织结构、去掉标题层次”，成功率就掉了 30% 以上。为什么内容没少、只是组织乱了，模型表现却灾难性下降？

因为模型难以从无序规则集合中识别优先级和依赖关系。像“先验证身份再处理退款”这种有序流程被拆散后，模型不知道哪条先执行、哪条是前置条件，于是有时跳过身份验证直接退款。这印证“对人类友好的信息组织方式对模型同样友好”——流程驱动让模型任何时刻都知道自己在哪一步，而不必遍历所有规则找匹配项。

Q2：账单谈判 Agent 里，为什么“退款和取消服务”绝对不能按“省钱提成”计费？作者要求提示词怎么写死这条规则？

因为按提成计费的前提是 Agent 通过谈判技巧降低现有账单；而退款/取消只是取回或停止本属于用户的钱，不涉及谈判省钱，若也按比例抽成就是薅羊毛、且分类会随任务措辞漂移。作者要求提示词直接写死一条命令式约束：NEVER use percentage_based_one_time for refunds and service cancellations. Use fixed_fee instead.——把模糊裁量收敛成可执行规则。

配套实验 实验 2-4（chapter2/prompt-engineering/）：基于 Tau-Bench（航空/零售客服）的提示工程消融实验，沿三个维度量化各要素贡献——语气风格（影响有限，模型风格适应力强）、信息组织随机化（成功率下降 30%+）、去掉工具描述（错误率上升 45%）。它把本课“组织比内容重要”的判断变成可复现的数字。整本书实验都在开源仓库：github.com/bojieli/ai-agent-book。
