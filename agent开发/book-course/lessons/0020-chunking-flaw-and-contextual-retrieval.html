分块的原罪与上下文感知检索：给每块补一段背景

这节课只拿下一件事：分块（chunking）必然切断片段与原文的联系——“该公司第二季度收入增长了 3%”离开原文后，“该公司”指代谁、报告出自哪年，全都留在了块外。上下文感知检索（Contextual Retrieval）怎么修它？答案是在索引期用 LLM 给每块补一段前缀摘要，把孤立块重新锚定回原文语境。它在第 3 章“用户记忆和知识库”的“RAG 技巧”一节，是本章埋在“文档分块”那里的伏笔的正式回收（对应图3-14）。

一、分块是必要之恶：为什么非切不可

文档进知识库前有一道离线工序——分块，把长文档切成能独立检索的片段。作者给了两个“非切不可”的理由：

  • 嵌入模型有长度上限，而且一整篇文档压成一个向量时，多个主题混在一起，向量哪个主题都表达不精确——这和 Enhanced Notes 段落越长越抓不住重点是同源问题。
  • 检索只想注入相关的那部分。片段太大会连带大量无关内容，浪费窗口、稀释注意力。

于是块大小成了一对典型权衡：块太小，单块信息不完整、脱离上下文后语义模糊；块太大，一个块混杂多主题、向量被稀释、命中后还拖入无关内容。作者给的实践起点是每块 256–1024 token、相邻块重叠 10%–20%，再按检索质量实测调优。

别把“调块大小”当成能根治的旋钮。作者在“文档分块”一节就把话挑明：无论采用哪种分块策略，分块都会切断片段与其原始上下文的联系——“该公司”指代谁、这段话出自哪份报告，这些信息留在了块的外面。这是分块的固有缺陷，不是参数没调好。调 chunk size 只能在“信息不完整”和“主题被稀释”之间挪动矛盾，消不掉“语境被切走”这条原罪。

三、上下文感知检索：索引期给每块补前缀

Anthropic 的 Contextual Retrieval 核心思想非常直观：在对文本块向量化索引之前，先用 LLM 为它生成一段简短的、含核心上下文的“前缀摘要”，拼接到块前面再索引。例如系统给上面那块生成前缀：

「[本段节选自 ACME 公司 2025 年 Q2 财报的“关键业绩指标”章节]」 + 原块
前缀 + 原文块 → 一起送去索引（对应图3-14）

原本模糊的块，被重新“锚定”回它的原始语义环境。而这一步真正巧妙的地方在于：它同时增强稀疏和稠密两条检索路——一份改动，两头受益。

收益有实测数字撑腰：据 Anthropic 数据，此技术结合 BM25，可把检索失败率（即 top-20 未命中率，1 − recall@20）降低 49%；再叠加重排序器，降幅达 67%。代价是索引期每块多一次 LLM 调用，但靠 prompt caching（相同前缀的重复调用只需约 1/10 成本）完全可控，每百万文档 token 约 1 美元——一次典型的高回报工程决策。

四、易错点：和第二章“上下文感知压缩”划清界限

两个名字长得像的东西，作者专门划了界，初学者却常混成一个：上下文感知检索 ≠ 上下文感知压缩。

  • 发生时机：检索在索引期（文档离线入库时）；压缩在运行期（当前会话进行中）。
  • 作用对象：检索针对知识库里的文本块；压缩针对当前会话的对话历史。
  • 动作方向：检索在做加法（补前缀、加背景，提升可检索性）；压缩在做减法（按当前任务裁剪、丢弃无关，节省窗口）。

记法：一个在“存进去之前把话说全”，一个在“用的时候把话说短”。方向相反，别用一个去理解另一个。

还有一个反直觉点值得顺带记住：把同一技术反过来用到用户记忆上（索引对话历史前先给每个对话块补前缀），配合 Advanced JSON Cards 常驻上下文，就收敛成本章的双层记忆架构——结构化概览常驻、上下文感知检索取细节。这是第 0021 课之后会继续展开的主线。

本课唯一要带走的判断：分块的原罪是“把语境切到了块外”，调 chunk size 治不了它；上下文感知检索的解法是在索引期用 LLM 给每块补一段前缀把它锚定回原文，一举同时喂强 BM25（多关键词）和向量（多语义背景）。它是索引期对知识库文本块做加法，和第二章运行期对会话历史做减法的“上下文感知压缩”方向相反，别混。

五、关键定义

上下文感知检索（Contextual Retrieval）
Anthropic 提出：在向量化索引前用 LLM 为每个文本块生成含核心背景的前缀摘要并拼接后再索引，把孤立块重新锚定语义环境，同时增强稀疏与稠密检索。避免与第二章“上下文感知压缩”混淆——本技术在索引期对知识库文本块做加法补背景，压缩在运行期对会话历史做减法去冗余。
分块（Chunking）
把长文档切成适合独立检索的片段的离线预处理。必要性有二：嵌入模型有长度上限、一篇文档一个向量会混主题；检索只想注入相关那部分。固有缺陷：无论何种策略都会切断片段与原始上下文的联系。
BM25
稀疏检索的经典概率排序算法，在 TF-IDF 基础上引入 k1(词频饱和度)和 b(文档长度归一化)两参数，靠倒排索引做精确关键词匹配。避免说它懂语义——它只按词频和文档频率算分。

六、自测题

Q1：文本块“该公司第二季度收入增长了 3%”脱离原文后为什么难检索？上下文感知检索怎么修？它发生在索引期还是运行期，和第二章的上下文感知压缩有何本质区别？

代词“该公司”无指代、“第二季度”无时间参照，嵌入向量语义模糊、检索精度低。修法：索引前用 LLM 生成前缀如“[节选自 ACME 公司 2025 Q2 财报关键业绩指标章节]”拼接后再索引。它在索引期针对知识库文本块做加法(补背景)；上下文感知压缩在运行期针对当前会话历史做减法(去冗余)。

Q2：为什么说“给块补一段前缀”能同时帮到 BM25 和向量检索这两条完全不同的路？

因为前缀里既有关键词也有语义背景，两条路各取所需：对 BM25 这样的稀疏检索，前缀增加了可精确匹配的关键词（如“ACME”“2025 年第二季度”）；对向量嵌入这样的稠密检索，前缀注入了关键语义背景，使向量表示能更精确反映文本块的真实含义。所以一份前缀改动，两头受益。

实验 3-11 ★★（chapter3/contextual-retrieval/）：并行构建两个知识库——一个用传统无上下文分块，另一个用 LLM 生成上下文前缀，再用 compare_retrieval_methods 同一查询在两库同时检索、并排比较结果差异。对“ACME 公司最近的收入增长情况如何？”这类查询，日志会清晰显示：有上下文的结果在得分上显著高于无上下文，返回的块也更精准。想看它反过来用到用户记忆，见配套的实验 3-12（chapter3/contextual-retrieval-for-user-memory/）。整本书代码在 github.com/bojieli/ai-agent-book。
