经验该存成什么：知识库、代码工具，还是 Skill

上一课我们确认了一件事：上下文再长，Agent 也不会自动学会。那学到的经验总得落到某处。这节课只拿下一件事——一条可落地的判别法则，决定每条经验用哪种外部载体沉淀。它出现在第 8 章开篇为“外部化学习”定位之后、展开各种工程机制之前，对应书里的表8-1。把这条法则记牢，你读后面所有具体机制（策略摘要、工作流录制、Skills 生成）时，就能一眼看出它在往哪个格子里塞东西。

一、是什么：一条经验，三种去处

作者用一个客服 Agent 的场景把话说得很具体：它在帮客户处理退款时，可能同时学到三类不同性质的东西。注意——不是“三种存法随便挑”，而是这三类东西性质不同，天然对应三种载体。

书里给的三个例子，把三格填满了：

  • “A 公司退款必须验证信用卡后四位” —— 一条特定规则，是事实性知识，存入知识库即可。
  • “用 X API 自动查询订单状态” —— 一段稳定可复用的操作序列，沉淀为代码工具最划算。
  • “退款流程的完整 Skill / 处理保险理赔的最佳实践” —— 涉及策略判断且业务规则经常变动，更适合写成 Skill 文档。

三种产物的“使用方式”也各不相同，这一列往往被读者跳过，但它恰恰解释了为什么要分——见下图。

二、为什么这样分：判别法则的三根轴

作者把整段判断压缩成一句话，值得逐字记住：

纯事实 → 知识库　常用+参数复杂 → 代码　常变+要判断 → 文档
表8-1 的判别法则 · 三根轴：事实性、复用稳定性、变动与判断

为什么偏偏是这三根轴，而不是“简单/复杂”或“重要/次要”？因为它们各自对应载体的一个硬约束：

  • 纯事实 → 知识库：事实没有“执行逻辑”，把它写成代码是浪费；它需要的是被检索到，所以存成条目，用语义搜索或 grep 找出来即可。
  • 常用且参数复杂 → 代码：一段操作序列如果稳定（不常变）、频繁被用、参数还多，那每次让模型重新“想”一遍既慢又不可靠。固化成代码，就从“每次重新思考”变成“一次生成、多次复用”——作者的类比是：第一次手动部署服务器后，把步骤写成自动化脚本。
  • 常变且涉及策略判断 → 文档（Skill）：如果规则经常变，写成代码就要不停改代码；而策略判断本就没法用固定参数穷举。写成自然语言 Skill，更新快、可被人类专家直接审查修改、按需加载即可。

后两种（代码 + Skill）在书里被合称为“工具生成”——外部化学习的更高阶形式：它不只把“知识”外部化，更把“流程”外部化、代码化。至于代码工具与 Skill 之间更细的取舍框架，书里说明第四章已详细讨论过，本课不展开。

三、更宏观：这条法则挂在哪根主线上

这条法则不是孤立技巧，它是外部化学习（Externalized Learning）的落地出口。第 8 章的整章主张是：在不改动模型权重的前提下，把经验和能力外部化为可持久、可检索、可复用的知识库与工具库，让 Agent 从“聪明”走向“越来越能干”。三种学习范式里，后训练固化参数、上下文学习做临时适应，而外部化学习是开发者最容易忽略的第三条路——也是本章的主角。

作者引用 Karpathy 的比喻点出了动机：LLM 像电影《记忆碎片》的主角，每次醒来都不记得之前发生了什么，而我们还没给它们提供“记录本”。表8-1 回答的正是：这个“记录本”该分成哪几册、每条经验记进哪一册。

误区一：把“外部化学习”和第五章的“Agent 自举”搞混。它们方向完全不同：外部化学习是不改权重的能力增长（往知识库/工具库里沉淀经验）；第五章“代码创造代码：Agent 自举”讲的是 Agent 创建与自身同类的系统。前者让自己更能干，后者造出另一个自己，别混为一谈。

误区二：把第三章的“用户记忆”当成本章要沉淀的“能力”。用户记忆存的是“用户是谁”的信息（偏好、事实、习惯）；本章要沉淀的是解题策略、操作流程、失败教训乃至新工具这类能力。表8-1 的三种产物属于后者。

误区三：以为“存哪儿”看的是难易或重要程度。不是。判据是这条经验本身的性质——是不是纯事实、稳不稳定、变不变、要不要判断。载体跟着性质走，不跟着“它有多重要”走。

配套实验：chapter8/gaia-experience/（实验 8-1，策略摘要）演示这套“学习-应用”闭环最直接的落地——基于 AWorld + GAIA，成功轨迹经 LLM 反思总结为结构化经验、向量化存入知识库，新任务再语义检索最相似的成功案例注入系统提示词。它填的正是三格里的“知识库”这一格。想看“代码工具”那一格如何落地，见下一课要讲的 chapter8/browser-use-rpa/（实验 8-2，工作流录制与回放）。完整代码在开源仓库 github.com/bojieli/ai-agent-book。

四、关键定义

外部化学习（Externalized Learning）
把知识沉淀到模型之外的文件、知识库和工具中的学习范式，持久、可解释、可随时修正；含“经验外部化为知识/Skill”与“能力外部化为工具”两个维度。避免等同于“记忆”或泛泛的“持久化”——它特指不改权重的第三条学习路径。
策略摘要（Strategy Summary）
把一次成功解题过程浓缩为结构化经验笔记（方法、踩过的坑、关键步骤），向量化存入知识库供后续语义检索。避免与“上下文压缩”混同——它是跨会话可迁移的经验，不是单次会话的临时压缩。
渐进式披露（Progressive Disclosure）
Skills 机制启动时只暴露每个 skill 的 name 与 description（数百 token），需要时才逐层读取 sub-skill 及脚本子文档。避免说成“检索”——它由模型上下文中的实际需要驱动，靠通用文件阅读而非向量索引。

五、自测题

Q1：“A 公司退款必须验证信用卡后四位”、“用 X API 查询订单状态”、“处理保险理赔的最佳实践”——这三条各该沉淀为哪种产物？

第一条是纯事实性规则 → 知识库条目（语义搜索或 grep 检索）；第二条是稳定可复用、参数明确的操作序列 → 专用代码工具（固化为代码、参数调用）；第三条涉及策略判断且业务规则常变 → Skill 文档（自然语言、按需加载）。法则：纯事实存知识库，经常用且参数复杂写成代码，经常变且涉及策略判断写成文档。

Q2：判别法则用的三根轴分别是什么？为什么“变不变”决定了该不该写成代码？

三根轴：是不是纯事实、复用是否稳定且频繁（参数是否复杂）、是否常变且涉及策略判断。“变不变”之所以关键：代码固化的是稳定流程，一旦规则经常变动，写成代码就要不停改代码、维护成本高；这类常变且需判断的经验更适合写成 Skill 文档——更新快、可被人类专家直接审查修改、按需加载。所以“常用+参数复杂”才进代码，“常变+要判断”进文档。

本课唯一要带走的判断：给 Agent 沉淀经验前，先问这条经验是什么性质——纯事实就进知识库（靠查），稳定常用且参数复杂就写成代码工具（靠跑），常变且要判断就写成 Skill 文档（靠读）。载体跟着性质走，不跟着重要程度走。这就是表8-1 的全部。
