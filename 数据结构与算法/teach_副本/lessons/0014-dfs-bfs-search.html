第 14 课:DFS 与 BFS 搜索 —— 两种"穷举"策略的直觉与递归/队列实现

目标:唤醒 DFS/BFS 的旧记忆(是什么、靠什么数据结构驱动),把主要力气花在"递归最容易写错的三处边界"和"这题该用 DFS 还是 BFS"的判断上。约 20 分钟。
接着第 13 课往下讲。 上一课学了枚举法和标记法,知道了什么时候暴力法可以接受——本质都是"试遍所有可能"。 这节课把"试遍所有可能"这件事,升级成两套有结构的搜索策略:DFS(深度优先搜索)和BFS(广度优先搜索)。

想象走迷宫:DFS 是"一条路走到黑,走不通再退回来重新选一条";BFS 是"站在原地不动,一圈一圈往外扩散,像水波纹一样"。 提醒:这里的 BFS 和第 6 课《算法图解》学的 BFS 是同一个算法——数据结构、扩展顺序、"第一次到达即最优"的性质完全一致, 只是《啊哈!算法》这本书换了"解救小哈"这个迷宫例子来讲,不是新算法。

一、DFS —— 用递归"能走多深走多深"

DFS 的核心套路:写一个递归函数 dfs(当前决策点),只解决"当下这一步该怎么办"——用一层循环把当前决策点所有可能的分支都试一遍; 每尝试一个分支就打上标记,然后递归调用去处理"下一步",递归返回后再撤销标记(回溯),这样下一次循环才能干净地尝试别的分支。 到达终止条件时处理一个完整方案并返回。

DFS 的三类典型用途:

  • 生成全排列 / 全组合(比如小哼手持 1、2、3 号扑克牌往三个盒子里放);
  • 判断连通性、判断某条路径是否存在;
  • Flood Fill(洪水填充式的区域染色计数)——统计连通区域面积、统计地图中独立岛屿的个数。

驱动 DFS 的数据结构是递归(或显式栈)——递归调用栈本身就在替你记住"如果这条路走不通,该退回哪一步重新选"。

Flood Fill(染色法)是什么? 以某个格子作为"种子点",用 DFS(或 BFS)向四周扩展,把所有可达、满足颜色/地形条件的格子标记(染)成同一种颜色 (比如同一个负数编号)。不同种子点得到不同颜色,颜色种类数就是连通分量(岛屿)个数——这正是 Windows"画图"油漆桶、Photoshop 魔术棒工具背后的原型算法。

二、BFS —— 用队列"一圈一圈往外扩散"

BFS 用一个队列模拟"层层扩展":起点入队并标记为已访问;只要队列非空,就取出队首元素,向四个方向 (用方向数组 next[4][2] 统一枚举)尝试扩展——凡是未越界、非障碍物且未被访问过的新点,标记为已访问后入队, 步数记为父节点步数+1;处理完队首的所有扩展后,把队首出队(head++)。

因为总是先把"近"的点全部扩展完才轮到"远"的点,所以第一次抵达目标点时的步数必然是最短的。 这也是 BFS 相比 DFS 最大的效率优势:DFS 想求最短路径必须穷尽所有路径再比较大小,BFS 不需要。

再提醒一次:这就是第 6 课学过的那个 BFS。 队列驱动、"先近后远"的扩展顺序、"第一次到达即最优"的性质,原理和代码结构完全一样—— 本书只是用小哈迷宫这个例子重新讲了一遍,不是要你学一个新算法。

三、DFS vs BFS 对比表

维度DFSBFS
驱动的数据结构递归(或显式栈)队列
遍历顺序特点一条路走到黑,走不通才回头按"离起点的距离"分层扩展,先近后远
更适合解决的问题穷举全部方案(全排列/全组合)、连通性判断、染色计数最短路径、最少步数
是否需要撤销标记需要(回溯 book[i]=0)不需要,每个点只入队一次
"第一次到达即最优"性质不具备,需要比较完所有方案才知道最优解具备

注:上面"是否需要撤销标记"这一行,严格来说是针对全排列型 DFS 的通用结论——第四节的代码会讲一个更容易考的例外情况。

四、C 代码:DFS 实现 Flood Fill(连通区域染色/计数)

下面把"网格连通性判断"作为 DFS 的具体应用来写:给定一张地图 a[x][y](1 表示可走的陆地,0 表示海洋/障碍), 从种子点 (x,y) 出发,用 DFS 把整片连通的陆地标记(染色)并统计面积。

#define MAXN 50
int n, m;                       // 网格行数、列数
int a[MAXN][MAXN];              // 1=可走的陆地, 0=海洋/障碍
int book[MAXN][MAXN];           // 0=未访问, 1=已访问(染色标记)
int next[4][2] = {{0,1},{1,0},{0,-1},{-1,0}};  // 右、下、左、上
int area = 0;                    // 当前这片连通区域的格子数

void dfs(int x, int y) {
    if (x < 1 || x > n || y < 1 || y > m) return;   // ①
    if (book[x][y] == 1 || a[x][y] == 0) return;    // ②
    book[x][y] = 1;                                  // ③ 标记为已访问,且不撤销
    area++;
    for (int k = 0; k < 4; k++) {
        int tx = x + next[k][0];
        int ty = y + next[k][1];
        dfs(tx, ty);            // 递归四个方向,继续向外扩散染色
    }
}

三处最容易在考试里写错、混淆的地方:

  1. ① 越界检查必须是函数进入后第一件事,顺序在任何数组访问之前。 如果把顺序颠倒——先执行 book[x][y] 或 a[x][y],再判断 x、y 是否越界, 这两个数组访问本身在越界坐标上就已经是越界访问了。这正是"递归出口条件写错导致越界访问"最常见的写法错误。
  2. ② 已访问判断 book[x][y]==1 绝不能漏写。 少了这一行,格子 A 递归到邻居 B,B 因为"没被标记过"又会递归回 A,A 再递归回 B……无限递归直到栈溢出—— 这就是"递归前忘记标记 visited 导致无限递归"最典型的表现。
  3. ③ 这里的 book[x][y]=1 标记不能撤销,和全排列型 DFS 正好相反,是最容易考的易混淆点。 全排列问题里,标记要在回溯时撤销(book[i]=0),因为下一次循环要"腾出"这个位置去尝试别的分支; 而 Flood Fill/连通性问题的目标是给整片区域打上永久的"已访问"标记——如果手滑加了一句撤销, 同一个格子会被反复标记、反复递归探索,轻则重复计数,重则重新陷入无限递归。
常见陷阱(原书踩坑总结):
  • BFS 忘记 head++:队首元素扩展完所有方向之后必须出队,漏写这一步,队列永远不出队—— 扩展逻辑要么死循环,要么提前停滞,得不到正确结果。
  • 用 DFS 硬解最短路径问题:DFS 也能靠穷举所有路径、比较步数求出最短解,但它不具备"第一次到达即最优"的性质, 必须遍历完所有可能路径才能确定答案——效率远低于 BFS,这是选择题最爱考的"这题该用哪种搜索"陷阱。
  • 终点判断和越界判断的顺序颠倒:应该先判断"是否已经到达终点",再做越界等其他检查;顺序一旦反了, 当终点被特意定义在"边界之外一格"时(比如水管工游戏里终点是 (n, m+1)),会被越界检查提前拦截, 永远无法触发"到达终点"这个分支。
