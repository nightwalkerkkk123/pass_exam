目标:5 分钟激活"大O是什么"的旧记忆,把剩下时间都花在二分查找**最容易扣分的边界写法**上。约 20 分钟。
为什么从这节课开始? 你已经学过数组、链表、栈、队列,只是不熟练——所以这节课不重讲"什么是数组", 而是把大O表示法当复习一带而过,把主要力气花在二分查找的 C 代码边界条件上。 这是全书最容易的算法,却也是期末最容易因为一个符号写错而丢分的地方。

一、大O表示法 —— 30 秒复习

大O表示法回答的问题只有一个:输入规模 n 翻倍时,操作数涨得有多快? 不是"这段代码跑了几毫秒"。 两个算法在小数据量下测出来很接近,不代表规模变大后依然接近——100 个元素时二分查找只比逐个扫描快 15 倍, 10 亿个元素时能快 3300 万倍。

大O名称代表算法n=100 万时大约要几步
O(log n)对数时间二分查找~20 步
O(n)线性时间逐个扫描100 万步
O(n log n)线性对数快速排序(第4课)~2000 万步
O(n²)平方时间选择排序(第2课)1 万亿步
O(n!)阶乘时间旅行商暴力解天文数字,城市数超过 20 就没法算
期末简答题最爱问的角度不是"背这张表",而是: "为什么大O比较的是最坏情况,不是某次实测?" ——因为算法要在规模变大后依然可用,而最坏情况才是可靠的保证;某一次运气好命中不能代表算法本身变快了。

二、二分查找 —— 前提只有一条

二分查找(Binary Search)只对已排序的数组有效。核心动作:每次看中间元素, 和目标比较,能排除掉剩下一半的可能性——而不是像逐个扫描那样一次只排除一个。

常见陷阱:对无序数组用二分查找。 二分查找"往左还是往右收缩"的判断,完全依赖"中间值大就说明左边、 中间值小就说明右边"这个前提。数组一旦无序,这个判断直接失效,算法给出错误答案而不报错——这是最容易被 考试选择题当作"找茬"选项的坑。

C 代码(带边界条件详解)

int binary_search(int arr[], int n, int target) {
    int low = 0;
    int high = n - 1;              // 覆盖整个数组的初始区间

    while (low /*①*/ <= high) {
        int mid = low + (high - low) / 2;   // ②

        if (arr[mid] == target) {
            return mid;             // 命中,直接返回下标
        } else if (arr[mid] > target) {
            high = mid - 1;         // ③ 目标在左半,收缩右边界
        } else {
            low = mid + 1;          // ③ 目标在右半,收缩左边界
        }
    }
    return -1;                      // 区间收缩到空,说明没找到
}

三处最容易在考试里写错、丢分的地方:

  1. ① 循环条件必须是 low <= high,不是 low < high。 当区间只剩一个元素(low == high)时,这个元素还没检查过,写成 < 会漏掉它, 导致目标明明存在却返回"没找到"。
  2. ② mid = low + (high - low) / 2,不要写成 (low + high) / 2。 两者数学上等价,但 low + high 在 low、high 都很大时可能整数溢出—— 这是更严谨的写法,老师如果讲过这一点,选择题里会专门考这个细节。
  3. ③ 收缩边界必须是 mid - 1 / mid + 1,不能写成 mid。 如果收缩成 high = mid(丢了 -1),当 mid 恰好等于 low 又没命中时, 区间永远不会缩小——死循环,这是二分查找最经典的 bug。

三、通关前先自测

Q1. 数组 [1, 3, 5, 7, 9, 11, 13](下标 0~6)已排序,查找 target=7。第一次 mid 是下标几?比较结果是什么?
Q2. 如果把循环条件从 low <= high 改成 low < high,会发生什么?
Q3. 找茬题:下面这段改写过的二分查找,某一行有 bug,选出问题所在。

先自己在纸上模拟一遍 arr=[2,4,6,8]、target=99(不存在)再选。

while (low <= high) {
    int mid = low + (high - low) / 2;
    if (arr[mid] == target) return mid;
    else if (arr[mid] > target) high = mid;   // ← 这一行
    else low = mid + 1;
}
▸ 看模拟过程
轮次lowhighmidarr[mid]结果
103144<99,low=2
223266<99,low=3
333388<99,low=4
443——low>high,退出

这个例子恰好没死循环(因为一直往右收缩),但把 target 换成一个会让代码走 high=mid 分支、且 mid 会等于 low 的情形(比如 arr=[2,4]、target=1),就会卡在 low=0,high=1,mid=0 反复不动——这就是死循环。
