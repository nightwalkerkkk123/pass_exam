一、直觉开场:给每个数分配一个专属格子

桶排序(Bucket Sort,本章简化版)的直觉,书里是这样打比方的:想象有 11 个(或者 1001 个)按编号排好的桶, 每出现一个数,就往对应编号的桶里插一面小旗子;等所有数都插完,只要按编号从小到大数一遍旗子, 数出来的顺序天然就是排好序的结果——不需要比较任何两个数的大小,直接"数格子"就完成了排序。

这就是桶排序最核心的思路:把"排序"变成"计数"。只要数值范围提前已知、又不太大,这个方法几乎不需要比较, 接近线性时间就能出结果。

二、桶排序的原理与局限

具体做法只有三步:

  1. 申请一个大小为"数据范围+1"的数组 book[](书里最初叫 a[],后改名 book 表示"记录/标记"),全部初始化为 0。
  2. 依次读入每个数 t,执行 book[t]++,把 t 出现的次数记在下标为 t 的"桶"里。
  3. 按下标从小到大遍历 book[],每个下标 i 出现 book[i] 次就打印 book[i] 遍 i。

但这其实是"计数排序"的雏形,而不是真正意义上的桶排序,它有两条硬前提: 数值范围必须提前已知,而且不能太大——一旦数据范围极大(比如 0~21 亿),需要申请一个同样大的数组,现实中根本开不出这么大的内存; 如果数据里有小数,也无法直接分配整数下标。此外,这个简化版本只能对数值本身排序,没法携带"这个数原本对应谁"这类关联信息 (比如姓名对应的分数)——排完序,身份信息就丢了。

三、冒泡排序:相邻交换,一趟归位一个数

冒泡排序(Bubble Sort)的思路更朴素,靠"相邻比较+交换"完成排序:

  1. 一共进行 n-1 "趟"。
  2. 每一趟从第 1 位开始,依次比较相邻两个数 a[j] 与 a[j+1],顺序错了就交换。
  3. 每一趟结束后,会有一个数"归位"(滚到当前未归位区间的末尾,形似气泡上浮/下沉),所以第 i 趟只需要比较到 n-i 位,已经归位的数不再参与比较。

冒泡排序有一个桶排序做不到的优点:可以携带附加信息。只要把数组元素换成结构体 (比如 struct student{ name, score }),比较时只看 score 字段、交换时把整个结构体一起换, 就解决了桶排序"排完序丢身份"的问题——这是考试里"排序+保留对应信息"这类题目的标准套路。

四、快速排序:这本书里的双哨兵原地实现

快速排序(Quick Sort)在这本书里的讲法,核心是"选基准数 + 双哨兵探测 + 原地交换":

  1. 从序列中选一个基准数(书中取最左边的数,比如序列 6 1 2 7 9 3 4 5 10 8 中的 6)。
  2. 设两个哨兵:i 指向最左,j 指向最右。必须先移动哨兵 j:从右往左找第一个小于基准数的数,停下。
  3. 再移动哨兵 i:从左往右找第一个大于基准数的数,停下。
  4. 若 i<j,交换 a[i] 与 a[j],然后重复步骤 2、3。
  5. 若 i 与 j 相遇,把基准数与相遇位置的数交换——基准数正式"归位",左边全部 ≤ 基准数,右边全部 ≥ 基准数。
  6. 对左右两个子序列递归执行同样的过程,直到子序列长度为 1。

这套流程的本质仍然是"二分"思想——每一轮都把问题规模大致折半处理,这一点和第4课的说法完全一致。

和第4课对照一下:同一个快速排序,两本书讲法一致吗? 核心思想(选基准数 + 分治 + 二分)完全一样, 但具体实现方式不同:第4课《算法图解》的版本是"新建 less/greater 两个数组, 把基准值以外的元素分装进去,不是原地操作";这一课《啊哈!算法》的版本是"不新建数组,靠 i、j 两个哨兵在原数组内部直接交换元素,原地完成分区"。两者不是互相矛盾,而是同一个思想的两种工程实现—— 一个更直观易懂(适合第一次学),一个更省内存(适合工程实践),复杂度结论(平均 O(n log n)、 最坏 O(n²))是一致的。

C 代码:桶排序(计数版),带边界条件详解

#include <stdio.h>
#define MAX_VALUE 1000          // 数据范围上限,必须提前已知——这是桶排序最大的前提

int book[MAX_VALUE + 1];       // ① 大小 = 数据范围+1,全局数组默认初始化为0

int main(void) {
    int n, t, i, cnt;
    scanf("%d", &n);
    for (i = 0; i < n; i++) {
        scanf("%d", &t);
        book[t]++;              // ② 计数:t 出现一次就 ++,不是标记为1
    }
    for (i = 0; i <= MAX_VALUE; i++) {   // ③ 从小到大遍历,顺序不能跳着来
        for (cnt = 0; cnt < book[i]; cnt++) {
            printf("%d ", i);
        }
    }
    return 0;
}

三处最容易在期末考试里写错、丢分的地方:

  1. ① 数组大小必须是"数据范围+1",而且这个范围要提前确定、不能开得比实际数据范围小。 如果测试数据里出现了超过 MAX_VALUE 的数,book[t]++ 直接数组越界写—— 这正是"数值范围必须已知且不能太大"这条前提在代码层面的体现,范围没估对,程序不报错也可能悄悄写坏内存。
  2. ② book[t]++ 是"计数",如果题目要求"去重",必须改成 book[t] = 1。 这是"小哼买书"这道题的关键区别:计数(++)会告诉你某个数出现了几次,但不会自动去重; 只有把出现过的位置标记为 1(而不是累加),最后按下标遍历时每个下标最多输出一次,才能得到去重效果。 这两种写法看起来只差一个符号,考试里经常拿来"找茬"。
  3. ③ 遍历 book[] 必须严格按下标从小到大(或从大到小)走一遍,不能跳跃。 桶排序"自动排好序"的效果,完全来自"下标本身就是有序的"这个事实——如果遍历顺序乱了,或者漏掉了某个下标区间, 排序结果直接错误,而且原因很隐蔽(代码逻辑本身没有"崩溃",只是遍历范围不对)。
常见陷阱:桶排序在数值范围很大时的空间浪费问题。 如果把桶排序直接搬去处理 0~21 亿这样的范围(比如身份证号、大范围收入数据),需要申请一个同样大小的数组—— 现实中根本开不出这么大的连续内存,程序会直接失败。这和第5课散列表里"填装因子"讨论的其实是同一笔账: 桶排序默认"数组下标 = 数值本身"这种最朴素的空间换时间方式,而散列表通过散列函数多做了一层"离散化"映射, 让值域可以远大于实际占用的桶数、只要控制好填装因子就行——本质上都是在权衡"愿意花多少额外空间,换多少速度"。
  • 桶排序无法处理小数(下标必须是合法整数),这一点和范围过大是同一类"数据类型不匹配"的问题。
  • 快速排序哨兵顺序颠倒:必须先移动哨兵 j(从右向左找小于基准数的数),若先移动哨兵 i 会导致基准数归位错误——书中反复强调"顺序很重要"。
  • 混淆"趟数"与"比较次数":冒泡排序每趟需要比较到 n-i 位,不是 n-1 位;照抄成每趟都比较到底不影响结果正确性,但会做大量无意义的比较,拉低效率。
  • 误以为快速排序最坏情况也是 O(NlogN):实际最坏情况会退化到 O(N²)(比如基准数每次都恰好选到最大或最小值),只有平均复杂度才是 O(NlogN)。

Q3. 找茬题:下面这段快速排序的双哨兵探测代码被改了顺序,请找出问题。
while (i != j) {
    while (a[i] <= temp && i < j) i++;   // ← 先移动哨兵 i
    while (a[j] >= temp && i < j) j--;   // 再移动哨兵 j
    if (i < j) { t = a[i]; a[i] = a[j]; a[j] = t; }
}
▸ 为什么顺序不能颠倒

书里反复强调"顺序很重要":正确写法必须先移动哨兵 j(从右往左找一个小于基准数的数停下), 再移动哨兵 i(从左往右找一个大于基准数的数停下)。这样才能保证:当 i、j 相遇时, 相遇位置上的数一定是"小于等于基准数"的,基准数换到这个位置才是正确归位。如果颠倒顺序、先移动 i, 相遇位置的数性质就反了,基准数归位到错误的地方,导致"左边全部 ≤ 基准数、右边全部 ≥ 基准数"这个不变量被破坏。
