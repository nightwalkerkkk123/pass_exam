第 9 课:动态规划 DP —— 网格建模与两个易混点

目标:先分清动态规划和贪婪算法的分工边界,再把主要精力放在 0/1 背包二维数组填表的边界条件上。约 25 分钟。
为什么这节课紧接第8课? 上一课学的贪婪算法,面对"商品可以分割"的背包问题(比如粮食、扁豆)能给出局部最优的近似解; 但换成"商品不可分割"的背包问题——每件东西只能整拿或不拿——贪婪算法就只能给近似解,这时候才需要本课的 动态规划(Dynamic Programming, DP) 去求精确的最优值。 DP 能用的前提只有两条,缺一不可:①问题是在约束条件下求某个指标的最优值(比如容量4磅时能装下的最大价值); ②问题能拆解成一系列彼此独立、离散的子问题——某个子问题的答案不会因为你先做了另一个子问题而改变。 第②条最容易被忽略:一旦子问题相互依赖,动态规划照样会跑出一个答案,却是错的,而且不会报错提示你。

一、网格建模思路 —— 行是可选对象,列是约束的离散取值

动态规划的第一步永远是画一张网格:想清楚"每个单元格代表哪个子问题",坐标轴自然就浮现出来了—— 通常一维是可选对象(商品、字符),另一维是约束的离散取值(容量、另一个字符串的前缀)。 单元格里存的值,通常就是你要优化的那个指标本身(比如"最大价值")。

网格画好之后,逐行(或逐列,两者本质等价)去填,每个单元格的值由前面已经算出的相邻单元格, 通过一个问题专属的递推公式推出——书里自嘲这是"费曼算法(Feynman algorithm)":写下问题、好好思考、写下答案。 没有一个通用公式能套用到所有 DP 问题上,这是刚开始学 DP 时最容易产生的误解。

讲解示例:容量 4 磅的背包

可偷的商品:吉他(1磅,1500美元)、音响(4磅,3000美元)、笔记本电脑(3磅,2000美元)。逐行填表:

商品(重量,价值)\ 容量1磅2磅3磅4磅
吉他(1磅,1500美元)1500150015001500
音响(4磅,3000美元)1500150015003000
笔记本电脑(3磅,2000美元)1500150020003500

容量4磅这一格是关键:"不拿笔记本电脑"沿用上一行的 3000;"拿笔记本电脑"要占 3 磅、剩 1 磅, 1 磅容量下之前算出的最大价值是 1500(吉他),2000+1500=3500>3000,于是更新为 3500(吉他+笔记本电脑)。 如果再加入 iPhone(1磅,2000美元)重新填一行,容量4磅时"不拿 iPhone"沿用 3500,"拿 iPhone"占 1 磅、剩 3 磅 (笔记本电脑那一行算出的最大价值是2000),2000+2000=4000>3500,答案更新为 4000(iPhone+笔记本电脑)。

一个容易被忽略的细节: 行/列的填充顺序,对背包问题和最长公共子串问题的最终结果都没有影响—— 但换一个新问题时未必如此,不能想当然地认为"顺序永远无所谓",需要针对具体问题重新分析。

二、0/1 背包问题 —— 答案在"网格最后一格"

0/1 背包问题:在背包容量约束下,从一组不可分割的商品中选出使总价值最大的子集, 是动态规划最经典的入门场景。网格坐标轴是商品(行)× 容量(列),状态转移方程是:

dp[i][c] = max( 不拿本行商品的价值, 拿本行商品的价值 + 剩余容量下前几行的最大价值 )

暴力枚举法作为对照:n 件商品要试遍所有组合,一共 2ⁿ 种,时间复杂度 O(2ⁿ),商品一多就完全不可行—— 这正是引出动态规划的原因。

三、最长公共子串 —— 答案是"网格中的最大值"

最长公共子串(Longest Common Substring):两个字符串中都存在的、要求连续排列的字符序列的最大长度, 可以用来估计拼写相似度。网格坐标轴是单词A字符(行)× 单词B字符(列),递推规则是:

  • 匹配(word_a[i] == word_b[j]):cell[i][j] = cell[i-1][j-1] + 1(在左上角对角线格子基础上加一)
  • 不匹配:cell[i][j] = 0(直接清零,不是取某个较大值!)

因为不匹配就清零,最长的那段连续匹配可能出现在网格中间任意位置,所以答案要在整张网格里找最大值, 不能想当然地只看最后一格。举例:Alex 把 fish 误输成 hish,hish 与 fish 的最长公共子串是 "ish"(长度3), hish 与 vista 的最长公共子串只有 "is"(长度2)——所以更可能是 fish。

拓展(了解即可,不是本课重点):如果字符不要求连续排列,就是最长公共子序列(LCS), 它的"不匹配"规则不是清零,而是取 max(cell[i-1][j], cell[i][j-1]),能更真实地反映相似度—— 两者递推规则的差异,正是最容易在考试里被搞混的地方。

五、C 代码:0/1 背包二维数组填表(附三处踩坑点)

#define N_ITEMS  3     // 商品数量
#define CAPACITY 4     // 背包容量,单位:磅

/* dp[i][c] 表示"只从前 i 件商品中选、背包容量为 c"时能装下的最大价值 */
int dp[N_ITEMS + 1][CAPACITY + 1];    // ① 行、列都要开 (N+1)、(CAP+1)

int knapsack(int weight[], int value[], int n, int capacity) {
    // ② 边界初始化:没有商品 或 容量为0时,最大价值恒为0
    for (int c = 0; c <= capacity; c++) dp[0][c] = 0;
    for (int i = 0; i <= n; i++)        dp[i][0] = 0;

    for (int i = 1; i <= n; i++) {                 // 行:第 i 件商品 → weight[i-1], value[i-1]
        for (int c = 1; c <= capacity; c++) {      // 列:容量 c
            if (weight[i - 1] > c) {
                dp[i][c] = dp[i - 1][c];            // 装不下,沿用上一行同容量的解
            } else {
                int not_take = dp[i - 1][c];                                // 不拿这件商品
                int take     = value[i - 1] + dp[i - 1][c - weight[i - 1]]; // ③ 拿这件商品,必须引用"上一行"
                dp[i][c] = (take > not_take) ? take : not_take;             // 比较顺序不能写反
            }
        }
    }
    return dp[n][capacity];   // 答案在网格"最后一格"
}

三处最容易在考试里写错、丢分的地方:

  1. ① 二维数组要开 (N_ITEMS+1) × (CAPACITY+1),不是 N_ITEMS × CAPACITY。 第 0 行代表"还没考虑任何商品"、第 0 列代表"容量为0",这两行/列是后面递推的起点; 数组开小了会导致下标越界,商品下标又要用 weight[i-1]、value[i-1](从0开始), i 和 i-1 混着用是最容易漏算或越界的地方。
  2. ② dp[0][c]=0 和 dp[i][0]=0 必须显式初始化。 这对应"没有商品"或"背包容量为0"这两种边界情况,最大价值理应恒为0——如果用局部数组忘了初始化, 数组里是垃圾值,后面所有递推结果都会被污染,却不一定报错,很难排查。
  3. ③ "拿这件商品"必须引用上一行 dp[i-1][c-weight[i-1]],不能写成 dp[i][c-weight[i-1]](当前行)。 0/1 背包里每件商品只能选一次,"拿了这件商品之后剩余容量下的最大价值"必须来自"还没考虑当前商品"的上一行; 如果引用当前行,就等于允许同一件商品被反复装入背包,变成了完全背包问题,和 0/1 背包的语义完全不同。 同时 take 和 not_take 的比较顺序也不能写反,否则会在明明有更优解时错误地保留较小值。
常见陷阱:
  • 对存在相互依赖的子问题硬套动态规划——比如"伦敦+巴黎"多站旅行行程:去了埃菲尔铁塔后,联合套票让卢浮宫门票变便宜, "去卢浮宫的花费"因此依赖"是否先去过埃菲尔铁塔"。这类子问题不满足独立性前提,动态规划会得到错误答案而不自知。
  • 误以为动态规划能处理"拿走商品的一部分"——动态规划只做二元决策(整取/不取); 商品可以被分割(比如粮食、扁豆)时,应该改用第8课的贪婪算法,DP 回答不了"拿多少",只能回答"拿还是不拿"。
  • 误以为答案必然出现在网格最后一个单元格——这对 0/1 背包问题成立,但对最长公共子串问题, 答案是网格中的最大值,可能出现在任意一格,考试最容易把这两条记反。

提示:先想清楚"拿这件商品"应该引用的是哪一行的历史结果。

if (weight[i - 1] > c) {
    dp[i][c] = dp[i - 1][c];
} else {
    int not_take = dp[i - 1][c];
    int take     = value[i - 1] + dp[i][c - weight[i - 1]];   // ← 这一行
    dp[i][c] = (take > not_take) ? take : not_take;
}
▸ 看修正后的写法 
int take = value[i - 1] + dp[i - 1][c - weight[i - 1]];   // 改成 i-1

0/1 背包问题里"0/1"两个字的含义就是"每件商品只能选0次或1次"——一旦"拿了这件商品之后"的剩余容量查的是当前行 (也就是已经把这件商品的选择结果算进去的那一行),这件商品就有可能被反复叠加进同一个解里,答案会比正确的0/1背包结果更大,这是判分时最典型的隐蔽 bug。
