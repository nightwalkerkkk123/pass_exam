第 3 课:递归 —— 用"缩小规模的自己"解决问题

目标:第2课学了数组/链表的读写代价对比和选择排序 O(n²);这节课换个角度,不再比较"快慢", 而是学一种新的思考方式——递归,以及它最容易在期末考试里翻车的两处:漏写基线条件、忽视调用栈代价。约 20 分钟。
用俄罗斯套娃理解递归。 一层套娃里装着更小的一层,再里面还有更小的一层……你不需要一次性想清楚"里面到底有几层", 只需要重复问同一个问题:"这一层里面还有娃娃吗?" 有,就打开它,对里面那个更小的娃娃重复同样的问题; 没有(打开的是最里层那个实心娃娃),就停下来。这个"停下来"的时刻,就是基线条件(base case); 每一次"打开、往里传"的动作,就是递归条件(recursive case)——而打开到一半、还没合上的那些外层娃娃, 正对应着调用栈(call stack)里那些"暂停、等待"的函数调用。

一、基线条件 vs 递归条件

递归(Recursion)指函数调用自身,用它可以把"重复处理直到条件满足"的问题表达得比循环更清晰。 但一个正确的递归函数,必须同时具备两个部分:

  • 基线条件(base case):函数不再调用自己、直接返回结果的情形——防止无限循环的关键。
  • 递归条件(recursive case):函数调用自身、处理更小规模子问题的情形——必须让问题规模严格变小, 才能保证最终一定能触达基线条件。

每一次递归调用,都会在调用栈上压入一个新的栈帧(stack frame),保存这次调用自己的局部变量; 基线条件触发后,栈帧从顶部依次弹出,逐层把结果往外传。递归的"清晰"和调用栈的"内存开销",是同一枚硬币的两面。

二、递归 vs 循环:不是快慢之争,是权衡

维度递归循环
清晰度表达嵌套/自相似结构更自然,不用自己维护"待处理列表"——调用栈自动替你记住 需要自己手动维护一个显式的待办堆/索引变量,逻辑更繁琐但过程直观
内存代价每次调用压入新栈帧,递归层数很深时栈会变得很高,可能耗尽内存 通常只用固定几个变量,不随处理规模增长而额外占用调用栈
性能没有性能优势,书中明确指出"使用循环的性能可能更好" 性能通常更好或至少不差
适用场景嵌套/树形/自相似结构(如盒中盒、遍历链表或树) 能清楚表达"重复直到条件满足"的线性重复任务
期末简答题最爱问的角度: "为什么说递归 vs 循环不是性能选择题?" —— 因为选递归从来不是为了跑得更快, 而是为了让"问题本身就是嵌套/自相似结构"这件事表达得更清楚;能用循环写清楚的问题,就不必强行套上递归。

C 代码(阶乘,带边界条件详解)

int fact(int x) {
    if (x == 1) {                    // ① 基线条件:不再递归,直接返回
        return 1;
    } else {
        return x * fact(x - 1);      // ② 递归条件:问题规模从 x 缩小到 x-1
    }
}

三处最容易在考试里写错、丢分的地方:

  1. ① 基线条件不能漏写。 如果把 if (x == 1) return 1; 整行删掉,只剩 return x * fact(x - 1);,函数永远不会"停下来返回"——这正是本章开头 countdown 例子的翻版:没有终止条件,会像 3...2...1...0...-1...-2... 一样无限递归下去, 最终耗尽调用栈内存。
  2. ② 递归条件必须让问题规模严格变小,不能只是"看起来在递归"。 如果不小心把 fact(x - 1) 误写成 fact(x)(忘记 -1),基线条件本身没写错, 但 x 永远不会变化、永远碰不到 x == 1,同样会无限递归——这比"忘写基线条件"更隐蔽, 因为代码表面上"看起来"有终止条件。
  3. ③ 递归深度过大时的栈溢出风险。 即使这段逻辑完全正确,如果 x 非常大(比如计算 fact(100000)),调用栈会压入等量的栈帧,可能耗尽内存导致栈溢出(stack overflow)—— 这时标准做法是改写成循环,或使用尾递归(如果语言支持,本章明确说这属于更高级、不在讨论范围内的技巧)。
常见陷阱:
  • 忘记写基线条件——没有终止条件会导致无限递归。
  • 误以为递归总是比循环快——递归只是让方案更清晰,并没有性能优势,循环的性能可能更好。
  • 忽视调用栈的内存代价——递归层数很深时,栈会变得很高,每层调用都占内存,可能带来严重的内存开销。

阶段调用x说明
