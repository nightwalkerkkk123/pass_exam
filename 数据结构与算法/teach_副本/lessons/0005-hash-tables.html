第 5 课:散列表 —— 把"查找"变成"计算"

目标:先建立"散列表把查找变成计算"这一层直觉,再把力气花在期末最爱考的两个点上—— 冲突怎么处理、填装因子怎么算、多高该扩容。约 20 分钟。
从"一个个找"到"直接算出来" 第4课学的快速排序,靠的是不断比较、不断分区来缩小范围——本质还是"找"。 这节课的散列表是一次跳跃:给一个键(key),散列函数直接算出它该待在数组的哪个格子里, 查找不再需要遍历,而是变成了一次函数计算。想象一个不知疲倦、过目不忘的杂货店收银员(书里的 Maggie): 不管问她苹果、牛奶还是鳄梨的价格,她都不用"翻账本"去找,而是心算出该看哪一格,直接报价—— 这就是散列表(hash table)比数组、链表快在哪里的直觉。散列表 = 数组 + 散列函数, 数组和链表都直接映射到内存,散列表则多了这一层"算出该存哪"的逻辑。

一、散列函数的两个前提

散列函数(hash function)接受任意数据作为输入,返回一个数字(索引)。它必须满足两个要求,缺一不可:

  1. 一致性(consistency):同样的输入必须始终映射到同样的输出。 做不到这一点,存进去的元素就再也找不回来——散列表直接失效。
  2. 均匀分布(uniform distribution):应把不同的输入尽量映射到不同的数字,分布越均匀越好。 做不到这一点,散列表会退化成事实上的一个大链表,查找退化为 O(n)。

此外,好的散列函数还应该知道数组大小,只返回有效索引范围内的数字——这一点在写 C 代码时最容易踩坑 (下面第四节会具体讲)。散列表的典型用途有三类:模拟"键→值"的映射关系(电话簿、DNS 解析——把域名映射为 IP 地址)、 防止重复(如投票去重:查一下这个人投过票没有)、缓存已经算过的结果(下次直接返回,不用重新计算)。

二、冲突(collision)与链表法

两个不同的键被散列函数映射到数组的同一个位置,就叫冲突——冲突不可避免, 好的散列函数只能降低概率,不能彻底消除。最常见的解决办法:在冲突的位置存一个链表。 把这个位置想象成一个小链表的"信箱":多个键共享同一格子,取信时要在这个小链表里从头逐个比对键名, 链表越短代价越小,链表越长(退化到只剩一个)代价越接近 O(n)。

书中的例子: 如果散列函数按字母表给 26 个位置分配索引,apple(首字母 a)和 avocado(也是 a 打头)会分到同一个位置——这就是冲突,查 apple 时要先定位到这个位置, 再在链表里逐个比较键名才能找到。如果这家店恰好只卖 A 开头的商品,所有元素都会挤进同一个链表, 散列表就整个退化成了一个链表,这正说明了"好的散列函数应让值均匀分布"的重要性。

三、填装因子(load factor)与扩容

填装因子 = 占用位置数 / 总位置数,用来衡量散列表的拥挤程度——把它当作拥挤程度的体温计: 数值越低,冲突概率越小,性能越接近理想的 O(1);一旦"发烧"就该调整长度(resizing)物理降温—— 创建一个更大(通常是原来两倍)的新数组,再用散列函数把所有元素重新插入进去。业界经验是超过 0.7 就该扩容。

操作数组链表散列表(平均情况)散列表(最糟情况)
查找O(1)O(n)O(1)O(n)
插入O(n)O(1)O(1)O(n)
删除O(n)O(1)O(1)O(n)

散列表兼具数组(查找快)和链表(插入删除快)的优点,但前提是要避开最糟情况—— 这依赖于低填装因子和良好的散列函数,而不是心存侥幸。

填装因子范围状态建议动作
远小于 0.7(如 0.1~0.4)冲突概率低,接近理想 O(1)无需操作
接近 0.7冲突概率明显上升密切关注,准备扩容
超过 0.7冲突概率高,性能明显下降应调整长度(通常翻倍)并重新插入所有元素

四、C 代码实现:简化散列表(链表法解决冲突)

#include <stdio.h>
#include <stdlib.h>
#include <string.h>

#define TABLE_SIZE 7        // 桶(bucket)数量,示例用小尺寸方便手算

typedef struct Node {
    char *key;
    int value;
    struct Node *next;      // 挂在同一个桶里的键值对,用链表串起来
} Node;

typedef struct {
    Node *buckets[TABLE_SIZE];
} HashTable;

void ht_init(HashTable *ht) {
    int i;
    for (i = 0; i < TABLE_SIZE; i++) {
        ht->buckets[i] = NULL;
    }
}

char *dup_key(const char *s) {
    char *copy = (char *)malloc(strlen(s) + 1);
    strcpy(copy, s);
    return copy;
}

// 散列函数:把字符串key的每个字符 ASCII 码累加,再对桶数量取模
unsigned int hash(const char *key) {
    unsigned int sum = 0;              // ① 用 unsigned,避免和负数取模纠缠
    int i;
    for (i = 0; key[i] != '\0'; i++) {
        sum += (unsigned char)key[i];
    }
    return sum % TABLE_SIZE;           // ① 保证返回值落在 0 ~ TABLE_SIZE-1 之间
}

void ht_insert(HashTable *ht, const char *key, int value) {
    unsigned int idx = hash(key);
    Node *cur = ht->buckets[idx];

    while (cur != NULL) {              // ② 插入前必须先遍历链表查一遍
        if (strcmp(cur->key, key) == 0) {
            cur->value = value;        // key已存在:更新value,不能再插一个新节点
            return;
        }
        cur = cur->next;
    }

    Node *n = (Node *)malloc(sizeof(Node));
    n->key = dup_key(key);
    n->value = value;
    n->next = ht->buckets[idx];        // 头插法,插到这个桶链表的最前面
    ht->buckets[idx] = n;
}

int ht_get(HashTable *ht, const char *key, int *out_value) {
    unsigned int idx = hash(key);
    Node *cur = ht->buckets[idx];
    while (cur != NULL) {
        if (strcmp(cur->key, key) == 0) {
            *out_value = cur->value;
            return 1;                  // 找到
        }
        cur = cur->next;
    }
    return 0;                          // 没找到
}

double load_factor(HashTable *ht) {
    int occupied = 0;
    int i;
    for (i = 0; i < TABLE_SIZE; i++) {
        if (ht->buckets[i] != NULL) occupied++;   // ③ 数"非空的桶",不是数元素总数
    }
    return (double)occupied / TABLE_SIZE;
}

三处最容易在期末考试里写错、丢分的地方:

  1. ① 散列函数取模要用 unsigned int,并保证返回值落在有效下标范围内。 如果用普通 int 累加,数值一旦溢出变负,C 语言里负数对正数取模的结果可能是负的—— 再用这个结果去当数组下标(ht->buckets[idx])就是数组越界访问。 "散列函数要知道数组大小、只返回有效索引"说的就是这件事。
  2. ② 插入前必须先遍历链表,检查这个 key 是否已经存在。 如果漏了这一步、直接头插一个新节点,同一个 key 会在链表里出现两份:ht_get 从头找, 只能拿到最新插入的那份,前一份变成再也访问不到的"僵尸节点"——既浪费内存,又和"键→值"这个映射关系本身矛盾 (一个键理应只对应一个值)。
  3. ③ 填装因子 = 占用位置数 / 总位置数,不是"元素总数 / 总位置数"。 链表法下,一个桶可能因为冲突挂了好几个键值对,"桶里有几个元素"和"占了几个桶"是两回事—— 这是期末最容易算错公式的地方。
常见陷阱(对应本章"常见陷阱"):
  • 散列函数用了 rand() 这类随机数:同一个 key 两次算出不同索引,存进去的元素再也找不回来,散列表彻底失效——这是破坏了一致性。
  • 散列函数恒定返回同一个值(比如永远返回 1):所有键都挤进同一个桶,散列表退化成一个大链表,查找退化为 O(n)——这是破坏了均匀分布。
  • 只关注平均情况、忽视最糟情况:散列表在最糟情况下(冲突严重、链表很长)所有操作都会退化为 O(n),必须靠"低填装因子 + 良好散列函数"主动避开,而不是心存侥幸。
  • 让填装因子无限增长、迟迟不扩容:填装因子越高,冲突概率越大,性能持续下降,超过 0.7 就该调整长度。

Q1. 用本节课代码里的散列函数 hash(key) = 所有字符ASCII码之和 % 7, 计算 hash("cat") 和 hash("act")。这两个 key 会不会冲突?
Q2. 一张散列表共有 10 个桶。经过多次插入后,有 8 个桶非空(其中 2 个桶因为冲突各挂了 2 个键值对, 链表里的键值对总数合计 12 个)。按本节课讲的填装因子定义,这张表的填装因子是多少?要不要扩容?
▸ 看填装因子怎么数
分子该数什么算式是否正确
非空的桶数(占用位置数)8 / 10 = 0.8✓ 正确定义
链表里的元素总数12 / 10 = 1.2✗ 常见误算
发生冲突的桶数2 / 10 = 0.2✗ 混淆了"冲突桶数"和"占用桶数"

填装因子定义死死咬住"占用位置数(桶数) / 总位置数(桶数)",和桶里链表挂了几个元素无关。0.8 > 0.7,应该扩容。

Q3. 找茬题:下面这个插入函数漏了一步。对同一个 key 先调用 ht_insert_buggy(ht, "apple", 10),再调用 ht_insert_buggy(ht, "apple", 20),会发生什么?
void ht_insert_buggy(HashTable *ht, const char *key, int value) {
    unsigned int idx = hash(key);
    Node *n = (Node *)malloc(sizeof(Node));
    n->key = dup_key(key);
    n->value = value;
    n->next = ht->buckets[idx];
    ht->buckets[idx] = n;              // ← 直接插入,没有先查是否已存在
}
