第 6 课:广度优先搜索 BFS —— 用队列把"找最短路径"变成机械操作

上一课学了散列表的冲突处理和填装因子;这节课换个赛道——从"表"转到"图",讲清楚 BFS 为什么必须靠队列、 以及它到底能不能回答"最短路径"。约 20 分钟。
为什么用"一圈圈往外扩散找人"来理解 BFS? 想象你要在人群里找一个符合条件的人,但你不认识他,只能靠"问朋友"打听。 最靠谱的做法不是随便乱问,而是先问自己所有的直接朋友(一度关系),如果都不符合,再让这些朋友去问他们各自的朋友 (二度关系),一圈一圈往外扩散——这样能保证你第一次问到符合条件的人时,中间经过的"传话人"数量一定是最少的。 广度优先搜索(Breadth-First Search,BFS)就是把这个直觉写成算法:用队列(先进先出) 保证"先加入名单的人先被检查",从而严格按"一度、二度、三度……"的层级顺序搜索。

一、队列(FIFO)在 BFS 里到底管什么用

BFS 维护一份"待检查名单",这份名单必须是队列(Queue,先进先出/First In First Out,FIFO)—— 先加入名单的先被检查。这不是随便挑的实现细节:如果换成栈(Stack,后进先出/LIFO),就会先检查 最近才加进去的"远房关系",而不是最早加进去的"近亲关系",搜索顺序一乱,BFS 保证的"最先找到=最短路径"就不再成立了。

也正因为严格按层级顺序检查,BFS 能同时回答两类问题:①"A 到 B 有没有路径"——队列耗尽都没碰到目标, 就说明不存在;②"A 到 B 最少要经过几步"——因为总是先处理离起点近的节点,第一次碰到目标时经过的边数,就是两点间的最短距离。

但这一切都建立在一个前提上:所有边的权重相同(或者压根不计权重),只关心"经过几条边"。这一点在 下面的表格和陷阱里会重点说。

二、BFS 的适用前提 vs 不适用场景

场景特征能否直接用 BFS 求"最短路径"为什么
边权重相同 / 不计权重,只关心"最少经过几条边"✅ 能 BFS 严格按入队顺序(先近后远)逐层扩展,第一次碰到目标时经过的边数必然最少
只想知道"A、B 之间有没有路径"(不关心几步)✅ 能 队列耗尽还没碰到目标,就说明两点不连通——这是 BFS 天然的副产品
边有不同权重(耗时、距离、费用不同)❌ 不能 BFS 只统计"经过的边数",边数最少不等于权重之和最小;必须换用能处理权重的算法 (下一课的狄克斯特拉算法)

C 代码(数组模拟队列 + 邻接矩阵,带边界条件详解)

#define MAX_NODES 100

int graph[MAX_NODES][MAX_NODES];  // 邻接矩阵:graph[i][j]==1 表示 i 到 j 有一条边
int visited[MAX_NODES];           // 标记数组:1 表示"已经入队过",不是"已经出队过"
int dist[MAX_NODES];              // 记录起点到每个节点的最短步数(边数)

int queue_arr[MAX_NODES];
int front = 0, rear = 0;          // ② front/rear 都从 0 开始,只增不减

void enqueue(int x) { queue_arr[rear] = x; rear++; }
int  dequeue(void)  { int x = queue_arr[front]; front++; return x; }
int  is_empty(void) { return front == rear; }   // ② 队空条件:front 追上 rear

int bfs(int start, int target, int n) {
    for (int i = 0; i < n; i++) { visited[i] = 0; dist[i] = -1; }

    enqueue(start);
    visited[start] = 1;           // ① 入队即标记,不要等出队再标记
    dist[start] = 0;

    while (!is_empty()) {
        int cur = dequeue();
        if (cur == target) return dist[cur];      // 命中,返回最短步数

        for (int nb = 0; nb < n; nb++) {
            if (graph[cur][nb] == 1 && !visited[nb]) {  // ① 必须查 visited,否则重复入队
                visited[nb] = 1;   // ① 同样是入队即标记
                dist[nb] = dist[cur] + 1;
                enqueue(nb);       // ③ 只从队尾追加 —— 先进先出,不是压栈
            }
        }
    }
    return -1;                    // 队列耗尽仍未命中 -> 不存在路径
}

三处最容易在考试里写错、丢分的地方:

  1. ① visited 标记的时机:必须在"入队"那一刻就标记,不能等"出队"才标记。 反例的常见后果:同一个节点如果有多条边指向它,会在它被真正处理之前反复被扫描进队列——轻则浪费时间, 重则在有环图上把队列越撑越大,表现得像死循环。这正对应"不标记已检查节点会导致重复检查甚至无限循环"这个经典坑。
  2. ② front/rear 边界:这里 front、rear 只增不减(不是循环队列),判空条件必须是 front == rear。 如果写反(比如写成 front > rear),或者忘记在 enqueue/dequeue 里递增 rear/front, 要么队列永远判断不为空导致死循环,要么读到还没写入的脏数据。
  3. ③ 别把 BFS 的队列写成栈。 enqueue 只在队尾(rear)追加、dequeue 只从队首(front)取出, 这才是"先进先出"。如果笔误把新元素也塞进 front 端,或者干脆用一个只在末尾操作的"栈式"结构模拟, 得到的就变成了 DFS(深度优先)的搜索顺序——代码依然能跑,但不再保证第一次碰到目标时经过的路径是最短路径。
常见陷阱:对带权图直接套用 BFS 求"最短路径"。 BFS 返回的是"经过的边数最少"的路径,前提是每条边的权重都相同 /不计权重。如果地图上每条路因为红绿灯、限速、拥堵而耗时不同(边权不同),BFS 依然会正常跑完,不会报错——但它给出的 "最少几步"完全可能不是"最快路线"。这一点最容易在期末简答题里被问到,也最容易被想当然地忽略(下一课的狄克斯特拉算法 专门解决"步数最少 ≠ 耗时最少"这个问题)。

三、通关前先自测

Q1. 有向图的边为:A→B、A→C、B→D、C→D、C→E。从 A 出发做 BFS,每次按字母顺序检查邻居。 按"出队 / 检查"的顺序,第 3 个被检查的节点是谁?
▸ 看队列变化过程
出队(检查)出队前 queue该节点未访问的邻居出队后 queue
A[A]B, C[B, C]
B[B, C]D[C, D]
C[C, D]D(已访问,跳过)、E[D, E]
D[D, E](无)[E]
E[E](无)[]

出队顺序就是"检查"顺序:A→B→C→D→E,第 3 个是 C。注意 C 检查邻居 D 时,D 早在处理 B 时就已经被标记 visited,所以不会重复入队——这正是"入队即标记"的作用。

Q2. 地图应用要算"从公司到家最快的路线",每条道路(边)因为限速、红绿灯耗时不同(边权不同)。 能不能直接对这张图跑本节课的 BFS,把它返回的"最少经过几条道路"当成"最快路线"?
Q3. 找茬题:下面这段 BFS 代码某一行有 bug,选出问题所在。
