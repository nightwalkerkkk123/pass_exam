第 19 课:面试收尾——从暴力到最优解(全课程最后一课)

上一课(第18课)学了 Kruskal/Prim 求最小生成树,以及割点割边、二分图匹配。这一课是全书收尾章,只讲一道题——但示范了"同一个问题怎么一步步逼近最优解"。约 20 分钟。
直觉类比:选举计票。 假如某个候选人的得票超过总票数的 50%,你其实不需要把所有选票一张张点清楚、排好队数出每个人的票数—— 你可以一边数一边"抵消":每遇到一张投给别人的票,就从这个领先候选人的票堆里划掉一张。 只要这个候选人真的过半,他的票"扛得住"所有抵消,最后剩下的候选人就是他。这正是本章最终答案——摩尔投票法——的直觉来源。

一、问题:找出出现次数超过 50% 的数

本章不引入新的数据结构,而是用一道真实的微软亚洲研究院(MSRA)面试题,示范"思路如何一步步逼近最优解": 给定一个序列,找出其中出现次数超过总数一半的那个数(如果存在的话)。全章的教学重点不是某个孤立算法, 而是"能不能更好"这条不断追问的思考链。

二、从暴力到最优:六次优化的关键一句话

下面每一步只说清楚"为什么比上一步快",不逐行展开代码——重点是记住这条追问链本身。

  1. 暴力计数法 O(n²): 对每个数,用双重循环和其余所有数逐一比较、统计出现次数,取次数最多的那个。这是面试第一版答案,也是后面所有优化的参照系。
  2. 排序法 O(n log n): 先把序列排序,相同的数就会彼此相邻;再线性扫一遍,用一个 count 统计当前数连续出现的次数即可——排序把"东一个西一个"的计数问题变成了"挨着数"的简单问题。
  3. 位置 n/2 定位法: 关键洞察——如果某个数出现次数一定超过 50%,排序后它必然站在第 n/2 位。于是不必扫描整个数组,只需对第 n/2 位的数再做一次计数确认即可,把工作量从"扫一遍排序结果"收窄到"验证一个候选"。
  4. 计数数组(类桶排序) O(n+m): 开一个下标对应数值的数组,直接累加每个值出现的次数,数组里最大值所在的下标就是答案。缺点是数值跨度很大(比如 1、10000、100000000)时会浪费大量空间,甚至开不出这么大的数组。
  5. 离散化 + 计数数组: 针对上面的空间浪费,先把原始数值按出现顺序或排序去重重新编号(比如 1→1,10000→2,100000000→3),再用小得多的数组计数——把"值域大小"的问题转化成"元素个数"的问题。
  6. 快速选择法(Quickselect): 借用快速排序"一次划分"的思路,划分完成后只递归进入包含第 n/2 位的那一侧,不需要把整个数组排完序——期望时间接近 O(n),比排序法更快。
知识补充说明(务必注意): 原书在这一章结尾只揭示了一条性质作为留白式的启发——"任意去掉两个互不相同的数, 若某数原本出现次数超过 50%,在剩余序列中它的占比依然超过 50%"——并没有给出完整的摩尔投票法代码。 下面关于摩尔投票法的具体步骤讲解和 C 代码,是知识补充,不是原书原文的逐字复现,但它是这条性质在教学上最自然的落地方式,也是这道题公认的最优解。

三、摩尔投票法:candidate + count 的计数抵消

摩尔投票法(Boyer-Moore Voting Algorithm)只维护两个变量:一个候选数 candidate,一个计数器 count。 扫描过程就是不断地"选票 vs 抵消":

  1. 初始 count = 0。
  2. 依次扫描序列中的每个数 x:
    • 若 count == 0,令 candidate = x,count = 1;
    • 否则若 x == candidate,count++;
    • 否则(x != candidate),count--——这一步等价于"同时去掉一个 candidate 和一个不同的数"。
  3. 扫描结束后,candidate 即为多数元素(前提是题目保证多数元素一定存在)。

用书中例子 3, 3, 1, 1, 3, 2, 3(n=7,超过 50% 需要出现次数 ≥4)手动演算一遍:

读入的数判断candidatecount
3count==0 → 立候选31
3与 candidate 相同32
1不同 → 抵消31
1不同 → 抵消30
3count==0 → 立候选31
2不同 → 抵消30
3count==0 → 立候选31

扫描结束,candidate = 3,正是出现次数超过 50% 的数(3 一共出现了 4 次)。

C 代码(摩尔投票法,带边界条件详解)

再次提醒:以下代码是知识补充讲解,不是原书原文复现。

int majorityElement(int a[], int n) {
    int candidate = a[0];          // ① 初始值随便设,不影响正确性
    int count = 0;

    for (int i = 0; i < n; i++) {
        if (count == 0) {           // ② 必须先判断 count==0,再更新 candidate
            candidate = a[i];
            count = 1;
        } else if (a[i] == candidate) {
            count++;
        } else {
            count--;                // 抵消:同时去掉一个 candidate 和一个不同的数
        }
    }

    // ③ 若题目未保证多数元素一定存在,此处必须再扫一遍统计
    // candidate 的真实出现次数,并与 n/2 比较,否则可能返回错误答案
    return candidate;
}

三处最容易在考试里写错、丢分的地方:

  1. ① 初始 candidate 怎么设都不影响最终正确性,但要理解为什么。 因为第一次循环时只要 count == 0(初始就是 0),第一个数一定会立刻把 candidate 覆盖成 a[0] 自己—— 所以初始值只是一个"占位",真正起作用的是后面的抵消逻辑,不是这个初始值本身。
  2. ② count == 0 时才更新 candidate,这个判断顺序不能颠倒。 如果写成先比较 a[i] == candidate 再看 count,或者把"立候选"和"抵消"两个分支顺序搞反, 会导致 count 已经归零却仍然在跟旧的 candidate 做无意义的比较,整个抵消逻辑就错了。
  3. ③ 如果题目不保证多数元素一定存在,扫描结束后必须再做一次计数校验,不能省略。 摩尔投票法只保证"如果多数元素存在,candidate 一定是它",但不保证"candidate 一定是多数元素"—— 序列里没有任何数超过 50% 时,算法照样会跑出一个 candidate,这个候选此时是错误答案,只有二次计数才能揪出来。
常见陷阱:以为找到候选就万事大吉。 这是本章最容易失分的一点——不管是"位置 n/2 法"还是"摩尔投票法", 它们扫描/定位出来的都只是一个候选,不是"证明"。只有在题目明确保证多数元素一定存在时,才能省略最后的二次计数确认; 否则必须再扫一遍数出 candidate 的真实出现次数,和 n/2 比较后才能下结论。默认题目"保证存在"就直接返回 candidate, 是考试里最常见的丢分点。

五、通关前先自测

Q1. 序列 2, 2, 1, 2, 3, 2, 2(n=7)用摩尔投票法扫描。读完前 4 个数 2, 2, 1, 2 后,candidate 和 count 分别是多少?
Q2. 序列 1, 2, 3, 4(每个数都只出现一次,不存在任何出现次数超过 50% 的数)用摩尔投票法扫描完,得到某个 candidate。下一步该怎么做?
Q3. 找茬题:下面这段改写过的摩尔投票法,某一处有 bug,选出问题所在。

先想想:一个数刚被立为新 candidate 之后,还应不应该再被下面的逻辑处理一次?

for (int i = 0; i < n; i++) {
    if (count == 0) {
        candidate = a[i];
        count = 1;
    }
    if (a[i] == candidate) {     // ← 这里少了 else
        count++;
    } else {
        count--;
    }
}
▸ 看模拟过程

用 1, 2, 2 模拟(正确答案应该是 2,因为 2 出现 2 次,占 2/3 超过 50%):

读入第一个 if(count==0)第二个 if(a[i]==candidate)candidatecount
1count==0,成立 → candidate=1,count=11==1,成立(又执行一次!)→ count++12(本该是1)
2count!=0,跳过2==1,不成立 → count--11
2count!=0,跳过2==1,不成立 → count--10

扫描结束 candidate=1,但真正的多数元素是 2——完全错误。根源在于少了 else: 第一轮把 candidate 刚设成 a[i] 后,第二个 if 又拿同一个 a[i] 去和刚设好的 candidate 比较, 结果必然相等,于是 count 被多加了一次(变成 2 而不是 1),后面两次抵消都追不平这个虚高的起点,最终给出错误答案。 这提醒我们:"立新候选"和"计数抵消"必须是互斥的一步,只能二选一执行,不能顺着往下再判一次。
