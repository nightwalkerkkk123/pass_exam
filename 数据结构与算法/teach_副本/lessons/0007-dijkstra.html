第 7 课:狄克斯特拉算法 Dijkstra —— 从"段数最少"到"总开销最小"

第6课学了 BFS 用队列做图的层级搜索,能求"步数最少"但不管边的权重。这节课升级到带权重的图:同样是找路径,但比的不再是"经过几条边",而是"总共要花多少代价"。约 20 分钟。
从"步数最少"到"总开销最小": 设想一张图——从起点 start 到 a 要 6 分钟,到 b 要 2 分钟; b 到 a 要 3 分钟,b 到终点 fin 要 5 分钟;a 到 fin 只要 1 分钟。 BFS 只看"经过几条边",会挑中 start→a→fin(只有2段),但实际耗时是 6+1=7 分钟。而 start→b→a→fin 虽然多了一段(3段),耗时却只有 2+3+1=6 分钟,反而更快。狄克斯特拉算法(Dijkstra's algorithm) 解决的正是这个问题:不管段数多少,只贪心地找"总权重最小"的路径——但前提是所有边的权重必须为正。

一、Dijkstra 维护的两张表:costs 与 parents

Dijkstra 用一张 开销表 costs 记录"起点到每个节点,目前已知的最小总代价"——刚开始只有起点的 直接邻居有真实数值,其余节点全部设为无穷大(表示"还不知道怎么到达它")。另一张 父节点表 parents 记录"到达某节点的最优路径上,紧邻的前一个节点是谁",用来在算法结束后回溯出完整路径—— costs 只回答"目前最便宜是多少",真正"怎么走"要靠 parents 表反向拼出来。

算法执行步骤(共 4 步,循环重复 2、3 直到所有节点处理完):

  1. 在所有未处理节点中,找出开销(costs 里的当前值)最低的那一个。
  2. 检查该节点的每个邻居:如果"经这个节点前往邻居"的开销比邻居当前记录的开销更低,就更新邻居的开销,并把这个节点记为邻居的父节点。
  3. 把该节点标记为已处理;重复步骤 1、2,直到图中所有节点都处理过。
  4. 从终点开始,沿 parents 链条反向回溯,还原出最终的最短路径。

这个"贪心地处理完一个节点就再也不回头"的做法,建立在一个假设上:只要所有权重为正,已处理节点就不可能再变得更便宜 (绕远路只会增加开销,不会减少)。这个假设一旦遇到负权边就会被打破——算法可能已经把某节点标记为"已处理",但后面出现的一条负权边 却能让抵达它的开销变得更低,而算法根本不会回头检查。遇到负权图,正确工具是贝尔曼-福特算法(Bellman-Ford algorithm)—— 本课不展开它的实现细节,只需要记住:它是专门为负权图设计的替代方案。

二、BFS 与 Dijkstra 的适用场景对比

算法适用图类型求解目标能否有负权边
广度优先搜索(第6课)非加权图边数最少的路径不涉及权重概念
狄克斯特拉算法(本课)加权图(近似 DAG)总权重最小的路径不能——负权边会导致错误结果
贝尔曼-福特算法(仅提及,不展开)含负权边的加权图总权重最小的路径可以——专门为负权场景设计
提醒:"最短路径"是个广义概念,可以指时间、金钱或任何要最小化的度量指标——不要被"路径"这个字面意思限制成 "物理距离最短"。书里 Rama 用乐谱换钢琴的例子就是在求"额外要多付多少钱最少",跟距离毫无关系。

三、C 代码:邻接矩阵版 Dijkstra(O(V²) 线性扫描)

不写优先队列优化,用最朴素的写法:邻接矩阵 graph[V][V] 存边权重(0 表示两节点间没有边), costs[V] 存当前最小开销,processed[V] 标记节点是否已处理过。

#define V 4                 // start=0, a=1, b=2, fin=3
#define INF 1000000         // 用足够大的数代表"无穷大",避免用 INT_MAX 相加时溢出

void dijkstra(int graph[V][V], int start) {
    int costs[V];
    int processed[V] = {0};      // 全部初始化为 0(false),表示还没有节点被处理过

    for (int i = 0; i < V; i++) {
        costs[i] = (i == start) ? 0 : INF;   /* ① 起点开销必须是0,其余节点必须是"无穷大" */
    }

    for (int round = 0; round < V; round++) {
        int node = -1, lowest = INF;
        for (int i = 0; i < V; i++) {
            if (!processed[i] && costs[i] < lowest) {   // 只在"未处理"节点里挑
                lowest = costs[i];
                node = i;
            }
        }
        if (node == -1) break;    // 剩下的节点都不可达,直接结束

        for (int n = 0; n < V; n++) {
            if (graph[node][n] == 0 || n == node) continue;  // 0表示node、n之间没有边
            int new_cost = costs[node] + graph[node][n];
            if (new_cost < costs[n]) {          /* ③ 松弛条件:"新路径更便宜"才更新 */
                costs[n] = new_cost;
                // parent[n] = node;  // 若要回溯完整路径,在此记录父节点
            }
        }

        processed[node] = 1;      /* ② 必须标记已处理,否则下一轮还会重复选中它 */
    }
}

三处最容易在考试里写错、丢分的地方:

  1. ① 初始化 costs[] 时,忘记把起点设为 0、其余设为无穷大。 如果偷懒把所有节点都初始化成 0,第一轮"找开销最小的未处理节点"会看到全是 0,可能随手选中一个根本还没算出真实距离的节点, 后面所有邻居更新都会建立在错误的起点开销上。
  2. ② 忘记在循环末尾写 processed[node] = 1。 如果不标记已处理,下一轮"找最小开销节点"还是会选中同一个刚处理过的节点——整个循环永远卡在同一个节点上, 陷入死循环,其他节点永远轮不到被处理。
  3. ③ 松弛条件写反,比如写成 if (costs[n] < new_cost)。 这个条件几乎永远不会成立(因为 costs[n] 初始就是 INF,天然比任何有限的 new_cost 大), 等价于整个更新逻辑失效——所有非起点节点的开销会一直停留在 INF,程序会把它们全部误判成"不可达"。
常见陷阱:对含负权边的图使用狄克斯特拉算法。 书中"用黑胶唱片换钢琴"的例子说明了这一点:一旦某个节点 (比如"海报")被标记为已处理,算法就认定"不会再有更便宜的路径能到它了",从此不再重新考虑它。但如果图里存在一条负权边, 它完全可能在"处理边界"已经向前推进之后,悄悄制造出一条更便宜的路径抵达这个已处理节点——而算法根本不会回头检查, 最终选出的就不是真正最优的路径。这不是"运气不好",而是贪心策略的假设本身被负权边戳破了。正确做法: 图中一旦出现负权边,换用贝尔曼-福特算法,不要硬用 Dijkstra。
