一、开场类比:小猫钓鱼——一个游戏,两套结构

"小猫钓鱼"是《啊哈！算法》里的一道纸牌游戏例题:两人轮流出牌,出的牌放到桌上;如果某人打出的牌,牌面和桌上已经放着的某张牌相同,就把这两张相同牌之间夹着的牌(含这两张)全部收进自己手牌的末尾;谁先把手里的牌出完,谁就输(对方获胜)。

这道题真正在考的不是"你会不会写队列/栈的代码",而是:看到一个具体场景,能不能拆出"这里该用队列、那里该用栈"——这正是这一章要练的抽象建模能力,不是新语法。

拆解: 每个人手里的牌,行为是"从最前面打出一张、赢回一沓牌之后接到最后面排队等待"——这是先进先出,建模成队列。桌面上摞的牌,行为是"每次出牌都盖在最上面、被赢走时也是从最上面一张一张往下收,直到收到那张匹配的牌为止"——这是后进先出,建模成栈。同一个游戏,两套结构各管一半。

二、核心概念:队列管手牌、栈管桌面

用队列/栈的语言重新说一遍上面的拆解:

  • 手牌 = 队列。 出牌 = 出队(从 head 位置拿走一张);赢牌之后,新赢到的牌要接到手牌末尾排队,而不是插到最前面下一轮优先打出——这就是入队(在 tail 位置追加)。
  • 桌面 = 栈。 出牌上桌 = 入栈(盖在最上面);被赢走 = 从栈顶开始连续出栈,直到弹出那张牌面匹配的牌为止。

判断"桌上是否已经有某个牌面"这件事,如果每次都从栈顶到栈底逐张扫一遍,是 O(n)。这里可以复用第1章桶排序的思路:开一个 book[] 标记数组,牌面为 t 的牌一上桌就令 book[t]=1,查询"桌上有没有牌面 t"直接看 book[t],降到 O(1)。

为什么考试里常用数组下标模拟指针,而不是真的用指针?

链表本身你已经学过:每个结点是 struct node { int data; struct node *next; },next 存下一个结点的地址,head 指向第一个结点,next==NULL 表示到了末尾;插入/删除只需要改 next 的指向,不用搬移已有数据。

但笔试、以及很多不想用指针的场合,会改用两个下标对齐的数组来模拟同样的逻辑:data[i] 存第 i 个结点的值,right[i] 存"第 i 个结点右边那个结点在 data 里的下标",right[i]==0 表示右边没有元素了——用数组下标 0 顶替"空指针"的角色。插入新值时,直接把新值追加到 data 数组末尾,再顺着原有的 right 链找到插入点,只改两个 right[] 的值就完成"接线"。这样既不需要 malloc,也不用担心指针语法(比如漏写 ->、忘记判空)在默写时出错——期末"不允许用指针,请用数组实现链表"这类题型,考的就是这套等价映射。

三、栈 vs 队列速查表

维度队列(Queue)栈(Stack)
规则FIFO 先进先出LIFO 后进先出
插入操作入队:data[tail++]=x入栈:data[++top]=x
删除操作出队:data[head++]出栈:data[top--]
判空条件head==tailtop==0
在小猫钓鱼里对应手牌(打出=出队,赢牌=入队排队)桌面(出牌=入栈,赢走=连续出栈)
典型考点场景排队叫号、QQ号解密、BFS广度优先搜索括号匹配、回文判断、DFS/递归调用栈

四、C代码:小猫钓鱼的核心模拟逻辑(数组实现)

下面用数组实现的队列/栈,拼出"某一方打出一张牌"这个核心动作。三处标号是期末最容易写错、丢分的地方。

/* 队列:两个下标 head/tail 代替搬移数据(FIFO) —— 对应"手牌" */
struct queue { int data[1000]; int head, tail; };
void q_init(struct queue *q)       { q->head = q->tail = 1; }     // ①
int  q_empty(struct queue *q)      { return q->head == q->tail; } // ①
void q_push(struct queue *q, int x){ q->data[q->tail++] = x; }    // 入队
int  q_pop(struct queue *q)        { return q->data[q->head++]; } // 出队

/* 栈:一个下标 top 代替搬移数据(LIFO) —— 对应"桌面" */
struct stack { int data[1000]; int top; };
void s_init(struct stack *s)       { s->top = 0; }                // ②
void s_push(struct stack *s, int x){ s->data[++s->top] = x; }     // ② 入栈:先加下标再存值
int  s_pop(struct stack *s)        { return s->data[s->top--]; }  // ② 出栈:先取值再减下标

#define MAXV 15                    // 牌面范围,按题目实际牌值调整
int book[MAXV];                    // book[t]==1 表示桌上已有牌面 t

/* 出一张牌 x:桌上没有就盖上去;桌上已有就把中间夹的牌全部赢回来 */
void play(struct queue *me, struct stack *table, int x) {
    if (book[x] == 0) {
        s_push(table, x);
        book[x] = 1;
    } else {
        while (table->data[table->top] != x) {
            int c = s_pop(table);
            book[c] = 0;              // ③ 牌收走了,标记要跟着清零
            q_push(me, c);            // 赢来的牌接到手牌末尾排队
        }
        int c = s_pop(table);         /* 收走牌面等于 x 本身的那张 */
        book[c] = 0;                  // ③
        q_push(me, c);
        q_push(me, x);                // 自己刚出的这张也一起收进末尾
    }
}

三处最容易在考试里写错、丢分的地方:

  1. ① 队列判空条件是 head==tail,初始化时两者必须设成同一个值。 这里 q_init 把 head、tail 都设为 1;如果两者初始值不一致(比如一个从 0、一个从 1 开始),空队列的判断从一开始就是错的,后面所有出队/入队都会跟着错位。
  2. ② 入栈是"先加下标再存值",出栈是"先取值再减下标",顺序不能颠倒。 s_push 写成 data[++top]=x,s_pop 写成 data[top--];如果哪一步顺序写反(比如出栈先减 top 再取值),读到的就是栈顶下面那一格,真正的栈顶元素被跳过,数据从此错位——这是"入栈先加后存、出栈先取后减"这条口诀最容易被默写反的地方。
  3. ③ book[] 标记数组必须和"牌被收走"同步清零。 牌从桌上(栈)被弹出去的同时,一定要把对应的 book[c]=0 一起改掉;漏了这一行,book 数组和桌面实际内容就不一致了,下一轮"桌上是否已有这张牌"的 O(1) 判断会直接给出错误答案。
常见陷阱(来自本章"常见陷阱"清单):
  • 用普通数组实现队列时,如果删除队首靠"所有元素依次前移一位"来补位,时间开销很大;正确做法是只移动 head 下标,用空间换时间,不搬数据。
  • 把队列的 tail 定义成"队尾本身"而不是"队尾的下一位置",会导致队列只剩一个元素时 head 和 tail 重合,和"空队列"的判定条件冲突——这也是为什么本章特意把 tail 定义成"下一位置"。
  • 链表插入代码如果用 t->next->data 判断插入点,必须先处理"插入到链表最末尾"的情况;这时 t->next 是 NULL,再去访问 t->next->data 会出错。
  • 像小猫钓鱼这种"双方轮流对战直到一方牌出完"的模拟题,如果没有额外规则限制,可能出现游戏永远无法结束的死循环——这是原书留给读者自己思考的开放问题,笔试遇到类似题目要留意这个边界。
