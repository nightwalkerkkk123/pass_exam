第 8 课:贪婪算法与 NP 完全问题 —— 复习 + 集合覆盖近似解的坑

上一课(第7课)学了 Dijkstra 算法求带权图最短路径,以及它对负权边的限制。这节课换一个思路:遇到根本没有已知高效精确解法的问题,怎么办?约 20 分钟。
直觉类比:没有"标准答案"的选择题,只能一步步赌"当下最好"。 Dijkstra 能算出唯一正确的最短路径,是因为这类问题有已知的高效精确算法。但很多问题不是这样——比如"用最少数量的广播台,覆盖全美 50 个州"。 这类问题精确解需要枚举幂集(power set),规模一大直接算不动。贪婪算法的思路很朴素:每一步都不去想"这样选对后面有没有影响",只问"现在这一步,哪个选项当下看起来最好",选完就往下走,不回头、不后悔。

一、贪婪算法:每一步都赌"眼前最好"

贪婪算法(Greedy Algorithm)反复执行同一条规则——"在当前所有可选项中挑出局部最优解,然后在剩余问题上重复",直到问题被完全解决。它不做全局回溯或穷举,不保证总能得到全局最优解,但实现极其简单、速度快,往往能得到与最优解足够接近的近似解。

先看一个贪婪策略"恰好就是最优解"的例子——教室排课(活动选择)问题:给定一组有时间冲突的课程,要选出数量最多、互不冲突的一组。贪婪解法是:

  1. 在所有课程中选结束时间最早的一堂,定为已选课程;
  2. 在剩余课程中,只保留与已选课程不冲突(开始时间晚于上一堂课结束时间)的课程,再从中选结束最早的一堂;
  3. 重复直到无课可选。

这个贪婪策略对这一问题恰好能得到全局最优解——这是本章里贪婪算法"运气好"的正例,但不能因此就以为贪婪算法处处都这么可靠。

二、NP 完全问题:暴力解不动的信号

NP 完全问题(NP-complete Problem)是一类以难解著称的问题(如旅行商问题、集合覆盖问题),目前没有已知的可以快速求解的算法。旅行商问题的可能路线数按阶乘函数增长,城市数一多就直接算不动:

判断 NP 完全问题没有充分条件,只有一些经验性的蛛丝马迹:

识别 NP 完全问题的蛛丝马迹
元素较少时算法运行速度快,但元素数量增加后速度骤降
涉及"所有组合"的问题
问题无法拆分成小问题,必须考虑各种可能情况
问题涉及序列(如城市访问顺序)且难以解决
问题涉及集合(如广播台覆盖)且难以解决
问题可转换为集合覆盖问题或旅行商问题
面对疑似 NP 完全问题该怎么办? 不要死磕精确算法——继续硬算只会让运行时间随规模爆炸式增长(10 个城市的旅行商问题就已经有超过 300 万条路径要比较)。应该及时转向近似算法(Approximation Algorithm),用速度换一个"足够接近"最优解的答案。贪婪算法就是最常用的近似算法之一。

三、集合覆盖问题的贪婪近似解法(C 代码)

集合覆盖问题(Set Covering Problem):需要从一批候选集合中,选出数量最少、并集能覆盖目标全集的子集组合(如用广播台覆盖 50 个州)。精确解需要枚举幂集,共 2ⁿ 个子集,运行时间 O(2ⁿ),n 稍大就完全不可行。贪婪近似算法的步骤:

  1. 在所有候选集合中,选出能覆盖"当前还未被覆盖的元素"最多的那一个(哪怕它同时覆盖了一些已覆盖的元素也没关系);
  2. 从"待覆盖集合"中减去它已覆盖的部分;
  3. 重复①②,直到待覆盖集合为空。

运行时间 O(n²),得到的是近似解,不一定是元素最少的组合,但非常接近。

下面把原书的 Python set 版本改写成 C:8 个州用一个 unsigned int 的 8 个二进制位表示(位掩码),集合的交集用 &、差集用 &= ~x 实现,对应 Python 里的 & 和 -=。

C 代码(带边界条件详解)

#define MT (1<<0)
#define WA (1<<1)
#define OR (1<<2)
#define ID (1<<3)
#define NV (1<<4)
#define UT (1<<5)
#define CA (1<<6)
#define AZ (1<<7)
#define NUM_STATIONS 5

int count_bits(unsigned int mask) {      // 数一个位掩码里有几个 1(几个"已覆盖"的州)
    int count = 0;
    while (mask) {
        count += mask & 1;
        mask >>= 1;
    }
    return count;
}

void set_cover_greedy(void) {
    unsigned int station[NUM_STATIONS] = {
        ID | NV | UT,   /* kone   */
        WA | ID | MT,   /* ktwo   */
        OR | NV | CA,   /* kthree */
        NV | UT,        /* kfour  */
        CA | AZ         /* kfive  */
    };
    const char *name[NUM_STATIONS] = {"kone","ktwo","kthree","kfour","kfive"};
    unsigned int states_needed = MT|WA|OR|ID|NV|UT|CA|AZ;  // 全部 8 个州待覆盖
    int i;

    while (states_needed != 0) {          // ② 终止条件:待覆盖集合是否已空
        int best = -1;
        int best_count = 0;
        for (i = 0; i < NUM_STATIONS; i++) {
            unsigned int covered = states_needed & station[i];  // ① 只看"新增覆盖"
            int c = count_bits(covered);
            if (c > best_count) {         // 严格 >:打平手时保留先遍历到的那个
                best = i;
                best_count = c;
            }
        }
        if (best == -1) break;            // 防御性退出:正常输入不会触发
        states_needed &= ~station[best];  // 差集:减掉刚选中台已覆盖的部分
        printf("选中 %s,剩余待覆盖州掩码 0x%02X\n", name[best], states_needed);
    }
}

三处最容易在考试里写错、丢分的地方:

  1. ① 每轮比较的必须是"新增覆盖数"count_bits(states_needed & station[i]),不是"该台元素总数"count_bits(station[i])。 如果误写成后者,算法会优先选"自身州最多"的台,而不是"对当前局面最有用"的台——这正是集合覆盖贪婪算法的核心,也是本章最容易被考试当作找茬题的地方。
  2. ② while 循环的终止条件是 states_needed != 0(是否所有元素都已覆盖),不是遍历完一遍候选集合就退出。 如果把 while 误写成对 station 数组只跑一遍的 for 循环,只会选出一个台就提前结束,后面还没覆盖的州会被漏掉。
  3. ③ 这里用 unsigned int 的位掩码代替 Python 的 set,前提是元素总数不超过这个整数类型的位数(这里 8 个州装进 32 位整数完全够用)。 如果换成"覆盖全国 3000 多个县"这种规模,一个 unsigned int 装不下,就要改用位数组或 unsigned long long,这是从 Python 的无限大 set 翻译成 C 时特有的边界,原书代码不需要考虑这一点。

按源书讲解示例逐轮走一遍

候选广播台:kone={id,nv,ut}、ktwo={wa,id,mt}、kthree={or,nv,ca}、kfour={nv,ut}、kfive={ca,az},目标覆盖 8 个州。

轮次待覆盖州各台新增覆盖数选中
1{mt,wa,or,id,nv,ut,ca,az}(8个)kone=3,ktwo=3,kthree=3,kfour=2,kfive=2kone(三者打平手,任选其一)
2{mt,wa,or,ca,az}(5个)ktwo={mt,wa}=2,kthree={or,ca}=2ktwo(与 kthree 打平手,任选其一)
3{or,ca,az}(3个)kthree={or,ca}=2,kfive={ca,az}=2kthree(与 kfive 打平手,任选其一)
4{az}(1个)只有 kfive={az}=1kfive(唯一选择,循环结束)

补充一句:第 2 轮其实 kfive 对剩余州的交集也是 2(覆盖 ca、az),与 ktwo、kthree 同样打平手——贪婪算法本来就允许平局时任选其一,不影响最终选出的电台总数。

最终 final_stations = {kone, ktwo, kthree, kfive},共 4 个广播台——与精确算法可能给出的最优解(如 {ktwo, kthree, kfour, kfive})站点数量相同、具体组合不同,这说明贪婪算法找到的是"足够好"的近似解,不保证与精确解完全一致。

四、常见陷阱:贪婪不是万能药

常见陷阱:误以为贪婪算法在所有问题上都能得到最优解。 本章用背包问题(35 磅背包)直接证伪:如果贪婪策略是"每步偷当前最贵的商品",会先偷最贵的音响,把背包塞满,导致放不下笔记本电脑和吉他,总价只有 3000 美元;而换一种组合能拿到 3500 美元。贪婪策略只是"每步局部最优",不代表全局最优,必须针对每个问题单独证明或验证其正确性——教室排课问题里贪婪恰好最优,背包问题里贪婪就会失效,两者不能一概而论。
