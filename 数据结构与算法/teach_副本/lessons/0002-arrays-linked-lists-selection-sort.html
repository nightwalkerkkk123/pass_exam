一、直觉类比:座位 vs 寻宝游戏

数组像电影院里连续的一排座位:知道第一个座位在哪、每个座位间距多少,想找第 37 号座位,直接心算一下地址就能走过去坐下——这就是O(1) 随机访问。但如果临时要在第 5 个座位后面加一个人,要么后面的人全体往后挪一位腾位置,要么干脆搬去更大的一排——这就是数组插入代价高的原因。

链表像一场寻宝游戏:每一张纸条上只写着"下一处宝藏在哪",你必须先摸到第一张纸条,才知道第二张在哪,一路摸到第 10 张——这是 O(n) 顺序访问,没有"公式"可以直接跳过去。但插入或删除,只需要改一张纸条上写的地址(把原本指向 B 的纸条 A,改成指向新纸条 C,再让 C 指向 B),其他纸条一张都不用动——这就是插入删除是 O(1) 的原因,前提是你已经拿到了需要修改的那张纸条。

一句话记住两者的对称性: 数组"读"快"改"慢(算地址不用找,但改动要挪位置);链表"读"慢"改"快(要顺着走不能跳,但改动只动一个指针)。两者互为镜像,不存在"全能选手"。

二、数组 vs 链表:核心概念

数组(Array):元素在内存里首尾相连地存放。知道起始地址和下标(下标从 0 开始计数,不是从 1 开始)后,用一次简单的加法+乘法就能算出任意元素的地址——数组读取是 O(1),因为它不需要"找",只需要"算"。

链表(Linked List):元素可以散落在内存任意角落,每个节点额外存一个"下一个节点地址"的指针,靠指针一个个串起来。想读第 k 个元素,只能从头节点开始顺着指针走 k 步,没有公式可以直接算出地址——所以是 O(n)。

插入和删除正好反过来。数组插入一个元素,后面的元素必须集体往后挪一位腾出空位——如果预留的空间已经用满,还得申请一块更大的内存、把所有元素整体搬过去,这是 O(n)。链表插入只需要改一个指针,其他节点一动不动,是 O(1);删除同理。

操作数组(Array)链表(Linked List)
读取(随机访问)O(1)O(n)(只能顺序访问)
插入O(n)(后面元素需整体后移,空间不足时还要整体搬迁)O(1)(仅需修改前一个节点的指针)
删除O(n)(后面元素需整体前移)O(1)(前提是已知待删除节点的位置)

三、选择排序:每轮从"剩下的"里挑一个最小的

选择排序(Selection Sort)的思路很朴素,重复"从未排序部分挑一个最小的,放到已排序部分末尾",直到挑完为止:

  1. 遍历当前未排序部分的所有元素,找出其中最小的一个;
  2. 把这个最小元素放到已排序部分的末尾(相当于"取出"再"接上");
  3. 重复第 1~2 步,直到没有未排序元素为止。

用书中同款数组 [5, 3, 6, 2, 10] 走一遍:第一轮要扫描全部 5 个元素才能找到最小值 2,第二轮只需扫描剩下的 4 个找到 3,以此类推。

轮次本轮未排序部分找到的最小值
1[5, 3, 6, 2, 10]2
2[5, 3, 6, 10]3
3[5, 6, 10]5
4[6, 10]6
5[10]10

每轮扫描次数依次是 5、4、3、2、1,加总 5+4+3+2+1=15≈n²/2——大O表示法只保留增长阶、丢掉常数系数,所以记作 O(n²) 而不是 O(n²/2)。

翻译成 C 之前的一点说明: 原书 Python 版本每轮把找到的最小值"取出"放进一个新数组,这样处理简单但要占额外内存;如果真用数组去做"取出",挪位置本身还要花 O(n),会让代码更啰嗦。C 语言更常见、也更省内存的等价写法是:不新建数组,直接在原数组内部,把找到的最小值和当前未排序区间的第一个位置交换——效果上仍然是"最小值归位、已排序区间逐步扩大",复杂度不变,还是 O(n²)。下面的代码就是这个等价版本。

四、C 代码与三处期末最容易翻车的边界

int find_smallest_index(int arr[], int start, int n) {
    int smallest_index = start;
    for (int j = start + 1/*②*/; j < n; j++) {
        if (arr[j] < arr[smallest_index]) {
            smallest_index = j;              // ③ 只记下标,先不交换
        }
    }
    return smallest_index;
}

void selection_sort(int arr[], int n) {
    for (int i = 0; i < n - 1/*①*/; i++) {
        int min_index = find_smallest_index(arr, i, n);
        int temp = arr[i];                   // 三步交换,借助 temp
        arr[i] = arr[min_index];
        arr[min_index] = temp;
    }
}

三处最容易在考试里写错、或者被"找茬题"盯上的地方:

  1. ① 外层循环边界是 i < n - 1,而不是 i < n。 但这里和二分查找的边界不一样——写成 i < n 并不会出错。原因是最后一轮 i=n-1 时,内层循环的 j 从 n 开始,条件 j < n 直接不成立,min_index 还是 i 自己,交换等于"自己和自己换"——不会越界也不会死循环,只是白跑一轮。真正的考点是:能不能判断出这次它是"多余但无害",而不是想当然地认定所有循环边界写错都会导致 bug。
  2. ② 内层循环起始位置必须是 j = start + 1(也就是 i + 1),不能写成 j = 0 或 j = i。 选择排序的核心前提是"只在未排序部分里找最小值"。如果内层循环从 0 开始,会把已经归位、排好序的前面部分也扫一遍——那部分永远包含当前的全局最小值,min_index 会被反复拉回到很早的位置,排序结果直接错掉,而且不会报错、不会崩溃,是最隐蔽的一类 bug。
  3. ③ 内层循环里只更新下标 smallest_index,不要提前交换值。 常见错误写法是一边扫描一边"发现更小就立刻交换",这会打乱"当前 i 位置的原始值"和"正在扫描的 j"之间的比较基准。正确做法是:内层循环只负责找到最小值的下标,等整个内层循环结束、确定了最终的 smallest_index,再在外层统一做一次交换。这一点在有重复元素时格外重要——如果记的是"最小值本身"而不是"下标",遇到多个相同的最小值,交换时可能换错具体位置。
常见陷阱(对应章节"常见陷阱"):
  • 以为"预留空位"能一劳永逸解决数组插入慢的问题。这只是权宜之计:预留的位置可能根本用不上(浪费内存,还挤占了别人可能需要的空间);一旦实际元素数超过预留额度,依然要重新申请一整块更大的内存、把所有元素搬过去。
  • 以为链表删除和数组插入一样"可能失败"。数组插入确实可能因为内存里找不到足够连续的空间而失败;但删除操作在任何情况下都能成功——无非是把后面元素往前挪(数组)或改一个指针(链表)。
  • 以为链表删除"永远"是 O(1)。只有当你已经拿到待删除节点本身(比如链表的第一个或最后一个节点)时,删除才是 O(1);如果还需要从头顺序查找这个节点,那查找本身就是 O(n),整体开销并不是 O(1)。
