第 13 课:枚举与暴力搜索 —— "试遍所有可能"也是一种算法

上一课(第12课)学了用栈/队列/链表建模"小猫钓鱼"类问题的抽象能力。这一课换个方向: 当你想不出巧妙解法时,枚举法教你如何"暴力但不蛮干"。约 20 分钟。
直觉类比:枚举是"笨办法",但笨办法也是算法。 想象你忘了保险箱密码,只记得是 3 位数字—— 你会怎么办?从 000 试到 999,总有一次能试中。这就是枚举(穷举)算法的全部直觉: 不知道怎么直接算出答案,就把所有可能的候选解一一列出,再逐个代入验证。 它看起来"笨",但依然是算法——因为它有明确的步骤、能保证找到正确答案、而且时间复杂度是可以分析和预测的。 "暴力"不等于"偷懒":本章真正的技巧,是分析清楚"到底要试多少种可能"(候选范围的上界), 以及能不能利用变量之间的关系,少试几层——这才是枚举法从"能用"到"好用"的关键一步。

一、枚举的基本写法:多重循环穷举候选解

枚举算法又叫穷举算法:把每一个可能的取值都试一遍,用条件判断筛出满足要求的解。最简单的情形只需一层循环, 比如"口×6528=口口×8256,两个方框填同一数字",for(i=1;i<=9;i++) 挨个试一遍即可。 稍复杂一点,比如"口口口+口口口=口口口,1~9 各用一次拼成加法等式",朴素做法就要写九层嵌套循环枚举 9 个变量, 再用一长串 && 判断两两互不相等、外加等式是否成立——这类代码又长又难维护。

标记法(book 数组):用一个数组记录状态,别再堆 &&

改进做法是引入标记数组:用一维数组 a[1..9] 存放 9 个变量的值, 再用 book[1..9] 标记 1~9 这九个数字分别有没有被用过。每轮枚举后统计 book 数组里 "1" 的个数,若恰好是 9,就说明这九个数字互不相同——不必再写九个变量两两比较的判断式。这不是降低了理论复杂度, 而是把"判断互不相同"这件事,从"堆叠一串 &&"变成"给取值范围建一个标记数组、数一数标记了几个", 代码大幅简化、也更容易扩展到更大规模。

网格候选点枚举:炸弹人问题

给定一张由墙 #、空地 .、敌人 G 组成的地图,只有一枚超远程炸弹, 求放在哪个空地能消灭最多敌人。做法是用二维字符数组存地图;双重循环枚举每一个空地作为候选放置点; 对每个候选点分别向上、下、左、右四个方向扫描,直到碰到墙 # 为止,边扫描边对遇到的 G 计数; 四个方向计数之和即该点可消灭的敌人数;全局维护最大值及对应坐标。这里书中坐标 (x,y) 指 "第 x 行第 y 列",不是数学坐标系里的 x 轴、y 轴,容易搞混。

炸弹人问题埋下的坑:枚举只看"能不能炸到",没看"能不能走到"。 这个算法只按几何直线和墙壁判断"能否炸到",却没有检查小人能不能走到这个候选空地—— 如果地图里某处墙改成空地,纯枚举算法可能选中一个小人根本走不到的位置。这是本章故意留下的伏笔, 指向下一课(DFS/BFS)要解决的"可达性"问题。

枚举范围的分析与降维:火柴棍等式

问题:用 m 根火柴棍拼出形如 A+B=C 的等式(0~9 每个数字的火柴棍拼法固定;加号、等号各占 2 根; A+B=C 与 B+A=C 视为不同解),一共有多少种拼法?

关键第一步是分析枚举上界。m 最多 24 根,减去加号等号占用的 4 根,剩余最多 20 根火柴棍; 数字 1 只需 2 根火柴棍最省,20 根最多拼出 10 个 1,所以 A、B、C 中任何一个数都不会超过 11111。 这给出了枚举的具体范围 0~11111,而不是漫无边际地枚举。

朴素做法是三层循环分别枚举 A、B、C,判断三者火柴棍总数是否等于 m-4, 时间复杂度约 O(m³)(m≈11112),实测要跑 1000 多秒,不可行。 降维优化:C 完全可以由 A+B 算出来,不需要单独枚举!只枚举 A、B 两层循环, C=A+B 直接算出,判断三者火柴棍数之和是否等于 m-4。 这一步把复杂度从 O(m³) 降到 O(m²),运行时间从 1000 多秒降到约 1 秒—— 这是本章"暴力枚举也需要仔细分析问题"的核心示范。

全排列:嵌套循环枚举的极限

求 1~n 的全排列(n=3 时为 123 的全排列)。做法是 n 层嵌套 for 循环,每层枚举一位, 配合一长串 && 判断各层变量两两不相等,满足则输出一种排列。n=3、n=4 时代码尚可接受, 但 n=9 时需要写九层嵌套循环加 36 个两两不等判断,明显不现实。这里留下一个"终极问题": 输入任意 n,如何用同一份代码输出 1~n 的全排列?嵌套循环的层数无法随输入参数变化, 这正是暴力枚举(本质是"层数固定的 DFS")的天花板,直接引出下一课"万能的搜索" (用递归/DFS 处理任意深度的枚举)。

二、枚举法什么时候能用、什么时候必须换算法

嵌套层数 / 候选范围典型例子大致复杂度期末怎么判断能不能接受
1 层循环,范围 ~10口×6528=口口×8256 填数字O(10)接受,一眼可写
2 层循环,范围数千~万级火柴棍等式(降维后 A、B 各枚举)O(m²),m≈1万时约1亿次接受(约1秒内跑完)
3 层循环,同等范围火柴棍等式(未降维前,A/B/C 都枚举)O(m³),约1000多秒不可接受,必须找降维关系
固定 9 层循环1~9 拼加法等式 / n=9 全排列接近 O(9!) 级判重代码勉强能写,但已经很丑,是信号而非终点
层数随输入 n 变化任意 n 的全排列O(n!)嵌套循环写不出来,必须换递归/DFS(呼应第1课大O表示法:规模一旦不确定就不能硬写死层数)

一句话判断标准:数据规模小、候选范围能事先分析出一个不太大的上界,枚举法可以接受; 数据规模大、或者嵌套层数得跟着输入变化,枚举法就必须让位给更好的算法 ——这正是第1课学的"大O衡量的是规模变大后还扛不扛得住",而不是"这次跑得快不快"。

三、C 代码:火柴棍等式(降维版)

下面是章节文件里给出的完整降维版实现——只枚举 A、B 两层循环,C 由 A+B 直接算出:

#include <stdio.h>

/* 0~9 每个数字需要的火柴棍根数 */
int matchCost(int x) {
    int cost[10] = {6, 2, 5, 5, 4, 5, 6, 3, 7, 6};
    int num = 0;
    while (x / 10 != 0) {          // ① 逐位取余、整除,直到只剩最高位
        num += cost[x % 10];
        x /= 10;
    }
    num += cost[x];                // 补上最后剩下的最高位
    return num;
}

int main() {
    int m, sum = 0;
    scanf("%d", &m);
    for (int a = 0; a <= 11111; a++) {      // ② 上界取 11111,不是拍脑袋定的
        for (int b = 0; b <= 11111; b++) {
            int c = a + b;          // ③ 用加法算出C,而不是再枚举一层C
            if (matchCost(a) + matchCost(b) + matchCost(c) == m - 4) {
                printf("%d+%d=%d\n", a, b, c);
                sum++;
            }
        }
    }
    printf("一共可以拼出%d个不同的等式\n", sum);
    return 0;
}

三处最容易在考试里写错、丢分或答错的地方:

  1. ① matchCost 里的循环边界:忘记 x/10==0 时还有最后一位没加。 如果把最后 num += cost[x] 这一行漏掉,单个数字或者取余到只剩最高位的那一位会漏算火柴棍数, 导致整体判断条件永远算错——这类"循环体外还有一步收尾"的写法,是枚举题里最容易漏写的收尾逻辑。
  2. ② 枚举上界写成 11111 是分析出来的,不是随手抄题目里的"最多24根"。 如果考试改了根数(比如把总根数从 24 变成别的值),上界也要跟着重新推导——直接照抄书上的 11111 而不理解"20 根火柴棍最多拼 10 个 1"这条推导逻辑,遇到改编题就会算错范围, 要么漏解(范围定小了)要么白跑很多轮(范围定大了没影响正确性但拖慢速度)。
  3. ③ 循环层数与判等条件容易混淆:忘记降维,或者降维后又把 C 也放进循环里重复枚举。 这题的核心得分点就是"C 由 A+B 算出、不需要单独一层循环";如果照着朴素思路写成三层循环 (for a、for b、for c 都枚举,再判断 a+b==c), 虽然结果依然正确,但复杂度多了一个数量级,是这道题最容易被扣"优化思路"分的地方。
常见陷阱:盲目枚举导致规模爆炸、直接超时。 本章反复强调的教训是:枚举前不分析范围上界, 直接机械地按题目给出的最大可能值(比如"火柴棍最多24根")设一个过大的循环上限,会浪费大量无效枚举; 该降维时不降维(三个变量存在运算关系却仍然三层都枚举),实际运行时间可能相差上千倍 ——火柴棍例子里就是从 1000 多秒降到 1 秒的差距。此外,遇到"若干变量互不相同"这类判断, 堆一长串 && 而不用标记数组,代码会随变量数增多而迅速失控; 在网格/路径类题目里只按几何或数值条件枚举候选位置、忽略"是否可达"的隐含约束 (炸弹人问题的经典反例),也是本章明确点出的坑。

Q3. 找茬题:下面这段改写过的 matchCost 函数,某处有 bug,选出问题所在。

提示:用 x=7(个位数)手动模拟一遍循环执行了几次、有没有把 cost[7] 真正加进 num。

int matchCost(int x) {
    int cost[10] = {6, 2, 5, 5, 4, 5, 6, 3, 7, 6};
    int num = 0;
    while (x / 10 != 0) {
        num += cost[x % 10];
        x /= 10;
    }
    return num;   // ← 这一行
}
▸ 看模拟过程
xx/10!=0?动作num
7否(7/10=0)循环体一次都不执行,直接 return num0(应为 cost[7]=3)

当 x=7 这种一位数时,while 循环条件从一开始就不满足, 循环体一次都不会执行——所有需要在循环体之外补的"最后一位"逻辑(原版里的 num += cost[x]) 如果被漏掉,单个数字的火柴棍数就永远算成 0。这是"循环体外还有一步收尾"这类代码最容易漏写的地方, 多位数(比如 x=23)也会漏掉最高位的那一位,只是不像单位数这么明显。
