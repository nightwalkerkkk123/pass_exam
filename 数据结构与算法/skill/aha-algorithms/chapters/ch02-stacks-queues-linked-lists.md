# 第2章：栈、队列、链表

## 核心思想 (Core Idea)

三种基础线性结构解决的都是同一个问题——"如何避免为了插入/删除而搬移一大堆数据"：队列和栈各用一到两个下标变量（head/tail、top）把"删除"变成下标移动而非数据搬移；链表则把"物理相邻"换成"指针相连"，插入/删除新数据时只需改指针，无需整体挪位。

## 引入的框架与算法 (Frameworks Introduced)

**队列（Queue）**：先进先出（FIFO, First In First Out）的线性结构，只在队首（head）删除、队尾（tail）插入。用数组 `data` 存内容，`head` 指向队首元素，`tail` 指向队尾元素的**下一个**位置（不是队尾本身，这样队首队尾重合时才能干净地表示"空队列"，不会和"只剩一个元素"混淆）。出队操作是 `head++`，入队操作是 `data[tail]=x; tail++`。适用场景：解密/排队类问题（书中例子：QQ号解密）、广度优先搜索（BFS）。

**栈（Stack）**：后进先出（LIFO, Last In First Out）的线性结构，插入和删除都只在同一端（栈顶 top）进行。用数组 `data` 存内容，`top` 指向当前栈顶。入栈是 `top++; data[top]=x`（可简写 `data[++top]=x`），出栈是先取 `data[top]` 再 `top--`。适用场景：括号匹配、回文判断、深度优先搜索（DFS）、函数调用栈（栈这个概念由图灵在1946年提出，最初就是为了解决子程序的调用与返回）。

**小猫钓鱼（队列+栈综合应用）**：这是一道把队列和栈组合起来建模的例题，不是新数据结构，而是"用哪种结构对应现实中的哪种行为"的建模训练。规则：两人轮流出牌，出的牌放桌上；若某人打出的牌与桌上已有某张牌牌面相同，则把两张相同牌之间夹的牌（含这两张）全部收入自己手牌末尾；一方牌出完则对方获胜。建模方式：每个人手中的牌 = 一个队列（出牌=出队，赢牌后新牌放回手中=入队）；桌面 = 一个栈（出牌上桌=入栈，被赢走=从栈顶连续出栈直到匹配的牌为止）。判断"桌上是否已有某牌面"若逐张枚举栈是 O(n)，可以像第1章桶排序那样开一个 `book[10]` 标记数组，牌面为 t 的牌在桌上就令 `book[t]=1`，查询降为 O(1)。

**链表（Linked List，指针实现）**：每个结点是一个 `struct node { int data; struct node *next; }`，`data` 存数值，`next`（后继指针）存下一个结点的地址；`head` 头指针指向第一个结点，`next==NULL` 表示链表末尾。建表时用 `malloc` 动态申请结点空间，通过 `->`（结构体指针运算符）访问指针所指结点的成员。插入一个新值时，只需遍历找到插入位置的前驱结点 `t`，令 `p->next = t->next; t->next = p` 两步改指针即可，不需要搬移任何已有数据。

**模拟链表（数组实现的静态链表）**：不用指针和 `malloc`，改用两个下标对齐的数组：`data[i]` 存第 i 个结点的值，`right[i]` 存"第 i 个结点右边那个结点在 data 数组中的下标"，`right[i]==0` 表示该结点右边没有元素（没有天然的"空指针"，用下标 0 代替 NULL）。插入新值时把新值直接追加到 `data` 数组末尾，再遍历原有的 `right` 链找到插入点，只改两个 `right[]` 的值完成"接线"。这是给不支持指针（或不想用指针）的场合准备的等价实现。

## 关键概念 (Key Concepts)

- **head / tail**：队列的两个下标，head 指队首，tail 指队尾的下一位置；`head==tail` 定义为空队列。
- **top**：栈顶下标，栈为空时 `top==0`；入栈先加后存，出栈先取后减。
- **FIFO / LIFO**：队列"先进先出"、栈"先进后出"，是区分二者应用场景的判据。
- **结构体（struct）**：把一组相关变量（如 data、head、tail）打包成一个新类型，成员用 `.` 访问（结构体变量）或 `->` 访问（结构体指针）。
- **指针（pointer）**：存储内存地址的变量，`*p` 取指针所指内存中的值，`&a` 取变量 a 的地址；`int *p` 定义的指针只能指向"存放整数的空间"。
- **malloc**：从内存中动态申请指定字节数的空间，返回 `void *`，需强制类型转换后使用；对应地有 `free` 释放（书中示例未释放，仅提示"不安全"）。
- **book 标记数组**：用一个数组把"某元素是否存在/出现过"记录为 0/1，把逐一枚举查找的 O(n) 降为一次数组访问的 O(1)，本质与第1章桶排序思想相同。
- **静态链表 / 数组模拟链表**：用两个数组（值数组 + "下一个"下标数组）模拟指针链表，用 0 充当 NULL。

## 思维模型 (Mental Models)

- 当问题需要"先来的先处理、后来的排在末尾等待"时，用队列（排队买票、BFS 扩展节点、QQ 解密规则）。
- 当问题需要"最后进入的最先被处理/撤销"时，用栈（括号匹配、回文判断、浏览器后退、DFS/递归调用栈）。
- 把链表的插入/删除理解为"改指针指向"，而不是"数据搬家"——这是链表相对数组的核心优势来源。
- 没有指针语言基础也不影响理解链表：把 `next` 指针换成"存下标的数组 right"，链表的逻辑完全不变，只是"地址"变成了"数组下标"。

## 常见陷阱 (Anti-patterns)

- 用普通数组实现队列时，若删除队首采用"所有元素依次前移一位"来补位，时间开销很大；正确做法是只移动 head 下标，用空间换时间。
- 把队列的 tail 定义成"队尾本身"而不是"队尾的下一位置"，会导致队列只剩一个元素时 head 和 tail 重合，与"空队列"的判定条件冲突，边界处理会变得别扭——书中特意选择 tail 指向队尾下一位置来避免这个歧义。
- 链表插入代码中若沿用 `if(t->next->data > a)` 判断插入点，必须先确认已经处理了"插入到链表最末尾"的情况（此时 `t->next` 为 NULL，再访问 `t->next->data` 会出错）；书中给出的示例数据凑巧没有触发这一情况，但自己实现和考试默写时要留意这个边界。
- 小猫钓鱼这类"双方轮流对战直到一方牌出完"的模拟题，如果没有额外规则限制，可能出现游戏永远无法结束的死循环（书中明确提出这是一个需要读者自己补充解决的开放问题）。

## 代码示例 (Code Examples)

```c
// 队列：用两个下标 head/tail 代替搬移数据（FIFO）
struct queue { int data[1000]; int head, tail; };
void q_init(struct queue *q)      { q->head = q->tail = 1; }
int  q_empty(struct queue *q)     { return q->head == q->tail; }
void q_push(struct queue *q, int x){ q->data[q->tail++] = x; }   // 入队
int  q_pop(struct queue *q)       { return q->data[q->head++]; } // 出队

// 栈：用一个下标 top 代替搬移数据（LIFO）
struct stack { int data[10]; int top; };
void s_init(struct stack *s)       { s->top = 0; }
void s_push(struct stack *s, int x){ s->data[++s->top] = x; }    // 入栈
int  s_pop(struct stack *s)        { return s->data[s->top--]; } // 出栈
```
这段代码演示了本章两种结构最核心的共同套路：都不真正移动数据本身，而是移动一两个下标变量来表示"当前有效范围"，从而把删除/插入操作的开销从 O(n) 降到 O(1)。

## 参考表格 (Reference Tables)

| 结构 | 插入 | 删除/查找端点 | 是否需连续内存 | 典型用途 |
|---|---|---|---|---|
| 普通数组 | 中间插入需搬移，O(n) | 任意位置随机访问 O(1) | 是 | 数据量固定、频繁随机访问 |
| 队列（数组实现） | 队尾 O(1) | 队首 O(1) | 是（但只搬下标不搬数据） | BFS、排队类模拟、Bellman-Ford 队列优化 |
| 栈（数组实现） | 栈顶 O(1) | 栈顶 O(1) | 是 | DFS、括号/回文匹配、函数调用 |
| 链表（指针实现） | 已定位插入点后 O(1) | 已定位后 O(1)，查找本身 O(n) | 否 | 需要频繁在中间插入/删除、数据量不确定 |
| 模拟链表（数组实现） | 同链表，但用下标 right 代替指针 | 同上 | 是（数组本身连续，但逻辑顺序不连续） | 不支持/不想用指针的场合，效果等价于链表 |

## 讲解示例 (Worked Example)

QQ号解密（第1节例题）：小哈给出加密串 "6 3 1 7 5 8 9 2 4"，解密规则是反复"删除队首的数，紧接着把新的队首的数移到末尾"，直到全部删完，按删除顺序拼起来就是 QQ 号。用队列模拟：初始 `head=1, tail=10`（9 个数已入队，tail 指向队尾下一位置）。

逐步操作（每轮：打印并删除队首，再把新队首移到队尾）：

1. 队列 `6 3 1 7 5 8 9 2 4` → 删除 6（输出6），把 3 移到末尾 → `1 7 5 8 9 2 4 3`
2. 删除 1（输出1），把 7 移到末尾 → `5 8 9 2 4 3 7`
3. 删除 5（输出5），把 8 移到末尾 → `9 2 4 3 7 8`
4. 删除 9（输出9），把 2 移到末尾 → `4 3 7 8 2`
5. 删除 4（输出4），把 3 移到末尾 → `7 8 2 3`
6. 删除 7（输出7），把 8 移到末尾 → `2 3 8`
7. 删除 2（输出2），把 3 移到末尾 → `8 3`
8. 删除 8（输出8），把 3 移到末尾 → `3`
9. 删除 3（输出3），队列为空

被删除的顺序拼起来就是 "6 1 5 9 4 7 2 8 3"——这就是解密出的 QQ 号。用代码实现时，每一轮的本质只是四行：`printf(q[head]); head++; q[tail]=q[head]; tail++; head++;`（先输出队首、head 前进，再把新队首复制到队尾、tail 前进，最后再让 head 前进跳过刚被复制走的那个数）。这个例子直接展示了"只移动 head/tail 两个下标就能模拟出队+入队的组合操作"，是队列思想最干净的落地演示。

## 学习要点 (Key Takeaways)

- 队列判空条件是 `head==tail`；tail 必须理解为"队尾的下一位置"而非队尾本身，否则空队列和单元素队列会分不清。
- 栈判空条件是 `top==0`；入栈是"先加下标再存值"，出栈是"先取值再减下标"，顺序不能反。
- book/标记数组是本章复用第1章桶排序思想的例子：用空间换时间，把"是否存在"的判断从遍历查找的 O(n) 降到数组下标访问的 O(1)，这是一个考试中容易被单独考察的优化点。
- 链表相对数组的优势只体现在"插入/删除点已经找到"之后（O(1) 改指针）；查找插入位置本身仍是 O(n) 线性遍历，链表并不提升查找效率。
- 数组模拟链表的核心映射关系：`data[i]` 存值、`right[i]` 存"下一个"的下标、`0` 扮演 NULL 的角色——这是笔试中"不让用指针，要求手写链表逻辑"时的标准应对方式。
- 栈用于 DFS 与函数调用（递归的底层机制），队列用于 BFS 与"队列优化的 Bellman-Ford"，这两条对应关系在后续图论章节会被直接使用，是本章与后面章节衔接的关键点。
