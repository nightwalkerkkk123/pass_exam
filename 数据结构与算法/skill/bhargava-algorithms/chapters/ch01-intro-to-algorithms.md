# 第1章：算法简介

## 核心思想 (Core Idea)
算法的价值不在于"能不能解决问题"，而在于"解决问题需要多少操作、随规模增长有多快"——大O表示法就是衡量这种增速的通用语言，二分查找（Binary Search）是本书第一个用来体会这件事的具体例子。

## 引入的框架与算法 (Frameworks Introduced)

**二分查找（Binary Search）**
- 何时使用：在一个**有序**的元素列表中查找某个值是否存在及其位置。
- 工作原理：
  1. 维护 `low` 和 `high` 两个指针，初始覆盖整个列表。
  2. 每次取中间元素 `mid = (low + high) / 2` 与目标值比较。
  3. 若中间值等于目标，返回该位置。
  4. 若中间值大于目标，说明目标在左半部分，令 `high = mid - 1`。
  5. 若中间值小于目标，说明目标在右半部分，令 `low = mid + 1`。
  6. 重复直到 `low > high`（没找到，返回 `None`）或命中。
- 每一步都排除掉剩余元素的一半，因此对 n 个元素最多需要 log₂n 步；而从头到尾逐个比对的"简单查找（Simple Search）"最多需要 n 步。

**大O表示法（Big O Notation）**
- 何时使用：不是用来算某段代码具体跑多少毫秒，而是用来描述"当输入规模 n 增大时，操作数按什么速度增长"，从而比较不同算法的可扩展性。
- 工作原理：数出算法在**最糟情况**下需要执行的操作次数，表达成关于 n 的函数，只保留增长最快的那一项（本章尚未展开"只留最高阶项"的运算规则，但已经通过例子暗示了这一点，严格的推导留到后续章节）。

## 关键概念 (Key Concepts)
- **简单查找（Simple Search）**：从头到尾逐个检查元素，最坏情况下需要检查全部 n 个元素，运行时间为 O(n)（线性时间）。
- **线性时间（Linear Time）**：操作数与输入规模成正比增长，记为 O(n)。
- **对数时间（Logarithmic Time）**：操作数以 log₂n 增长，记为 O(log n)；对数是幂运算的逆运算，log₂100 问的是"2的多少次方等于100"。
- **大O表示法（Big O Notation）**：描述算法运行时间随输入规模增长的速度，而非以秒为单位的绝对速度。
- **最糟情况运行时间（Worst-case Running Time）**：大O表示法默认给出的是最坏情况下的操作数上界，即便某次具体查找运气好一次命中，算法本身的复杂度标签不变。
- **阶乘时间（Factorial Time, O(n!)）**：操作数按 n 的阶乘增长，旅行商问题（Traveling Salesman Problem）的暴力解法即属此类，是本章提到的最慢增速。
- **O(n log n)**：比 O(n²) 快、比 O(log n) 慢的一类运行时间，快速排序（Quicksort，第4章）属于此类。

## 思维模型 (Mental Models)
- 当**列表已经有序**时，用二分查找代替逐个扫描——每猜一次就砍掉一半的可能性，而不是线性地排除。
- 把大O表示法当作"忽略具体秒数、只看增长曲线形状"的工具来理解：两个算法哪个更快，问的是"n 翻倍时操作数涨多少"，不是"这次跑了几毫秒"。
- 当需要判断"某个算法能不能在数据量暴涨后仍然可用"时，比较的应是增速（O(log n) vs O(n) vs O(n!)），而非在小数据量上测得的实际耗时——小规模下看似接近的算法，规模变大后差距可能是天壤之别（书中例子：100个元素时二分查找只比简单查找快约15倍，但10亿个元素时快3300万倍）。

## 常见陷阱 (Anti-patterns)
- **只比较小规模下的运行时间就下结论**：书中的 Bob 例子——他用100个元素的测试结果（简单查找100毫秒 vs 二分查找7毫秒，约15倍差距）线性外推到10亿个元素，得出"简单查找450毫秒也够用"的错误结论，实际上简单查找需要约11天。根源在于把"倍数关系"当成固定值，而没有意识到两种算法的增速（O(n) vs O(log n)）本质不同。
- **对无序列表使用二分查找**：二分查找依赖"取中间值可以判断目标在左半还是右半"，前提是列表有序，否则算法失效。

## 代码示例 (Code Examples)
```python
def binary_search(list, item):
  low = 0
  high = len(list) - 1
  while low <= high:
    mid = (low + high) // 2
    guess = list[mid]

    if guess == item:
      return mid

    if guess > item:
      high = mid - 1
    else:
      low = mid + 1

  return None

my_list = [1, 3, 5, 7, 9]
print(binary_search(my_list, 3)) # => 1
print(binary_search(my_list, -1)) # => None
```
这段代码展示了二分查找的完整实现：用 `low`/`high` 维护当前查找区间，每轮取中间下标 `mid` 比较，命中则返回位置，否则收缩区间继续二分，直到区间为空（`low > high`）时返回 `None` 表示未找到。原书排版为 Python 2 的 `print` 语句且存在 OCR 导致的运算符缺失，此处已还原为可直接运行的等价代码。

## 参考表格 (Reference Tables)
本章列出的五种常见大O运行时间（从快到慢）：

| 大O表示法 | 名称 | 代表算法 |
|---|---|---|
| O(log n) | 对数时间 | 二分查找 |
| O(n) | 线性时间 | 简单查找 |
| O(n log n) | 线性对数时间 | 快速排序（第4章） |
| O(n²) | 平方时间 | 选择排序（第2章） |
| O(n!) | 阶乘时间 | 旅行商问题的暴力解法 |

## 讲解示例 (Worked Example)
**场景**：在1~100之间猜一个数，每次猜测后会被告知"大了""小了"或"对了"，目标是用最少次数猜中。

- **简单查找式的猜法**：从1开始依次往上猜（1、2、3……），每猜一次只能排除一个数字。如果目标是99，需要猜99次才能中——这本质上就是逐个扫描，O(n)。
- **二分查找式的猜法**：
  1. 先猜50。若提示"小了"，则1~50全部排除，目标落在51~100。
  2. 再猜75（50和100中间）。若提示"大了"，则排除75~100，目标落在51~74。
  3. 再猜63（50和75中间）……如此每猜一次就排除剩余范围的一半。
  4. 无论目标是哪个数，在1~100范围内最多7次就能猜中（因为 2⁷ = 128 ≥ 100）。

**电话簿/字典的类比**：字典有240,000个单词，简单查找最坏要翻240,000次；二分查找每次排一半，只需约18步（log₂240000 ≈ 18）就能定位。

**旅行商问题（Traveling Salesman Problem）的暴力解法**：一位旅行商要走遍5个城市并使总路程最短。暴力做法是枚举所有城市访问顺序、逐一计算总路程再取最短的一个。5个城市有 5! = 120 种排列，需要120次操作；6个城市变成 720 次；7个城市变成5040次——操作数按阶乘暴涨，运行时间为 O(n!)。当城市数超过100时，即便动用最强算力也无法在合理时间内穷举完毕，这也是本章唯一给出的"O(n!) 确实存在"的证据。

## 学习要点 (Key Takeaways)
- 二分查找只对**有序列表**有效，最多需要 log₂n 步；简单查找不要求有序，但最多需要 n 步。
- 大O表示法衡量的是操作数随输入规模增长的**速度**，不是具体耗时的秒数；比较两个算法时应看增速曲线而非某次实测值。
- 大O表示法给出的是**最糟情况**下的运行时间保证，即使某次查找运气好、一次就命中，算法的复杂度标签仍以最坏情况为准。
- 记住从快到慢的五种常见运行时间：O(log n) < O(n) < O(n log n) < O(n²) < O(n!)，考试中判断算法优劣的第一直觉应从这张表出发。
- 需要搜索的元素越多，O(log n) 相对 O(n) 的优势就越大——不要用小规模测试结果线性外推到大规模场景（Bob 的错误）。
- 旅行商问题这类 O(n!) 问题在城市数较大时无法精确暴力求解，只能求近似解（呼应第10章 NP完全问题）。
