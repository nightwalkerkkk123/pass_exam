# 第5章：散列表

## 核心思想 (Core Idea)

把"数据存在哪里"这个问题交给一个函数去算，而不是靠遍历或排序去搜——散列函数（hash function）把任意输入映射成数组索引，让散列表（hash table）在平均情况下能以 O(1) 的速度完成查找、插入和删除。

## 引入的框架与算法 (Frameworks Introduced)

**散列表（hash table，又称散列映射、映射、字典、关联数组）**
- 何时用：需要模拟"键→值"的映射关系、需要快速判断某元素是否已出现过（去重）、需要缓存已计算过的结果时。
- 工作原理：内部结合一个数组和一个散列函数。存储时，先把键交给散列函数得到一个索引，再把值存到数组的这个索引处；查找时，同样对键调用散列函数得到索引，直接从数组该位置取值，无需遍历。散列表因此是数组的"增强版"——数组和链表都直接映射到内存，散列表则多了一层散列函数逻辑来决定元素的存储位置。

**散列函数（hash function）**
- 何时用：它是散列表的构建基石，任何要实现散列表的场合都要有一个散列函数。
- 工作原理：接受任意数据作为输入，返回一个数字（索引）。必须满足两个要求：（1）一致性——同样的输入必须始终映射到同样的输出，否则无法再找到已存入的元素；（2）尽量避免碰撞——应把不同的输入映射到不同的数字，分布越均匀越好。此外，好的散列函数还应知道数组大小，只返回有效索引范围内的数字。

## 关键概念 (Key Concepts)

- **散列函数（hash function）**：将任意输入映射为一个数字（索引）的函数，是散列表的核心组件。
- **散列表（hash table）**：由数组和散列函数组合而成的数据结构，用键（key）映射到值（value）。
- **键与值（key / value）**：散列表存储的基本单元，键经散列函数计算出索引，值存放在该索引处。
- **冲突（collision）**：两个不同的键被散列函数映射到了数组的同一个位置。
- **填装因子（load factor）**：散列表中已被占用的位置数除以总位置数，用来衡量散列表的拥挤程度，计算公式为「占用位置数 / 总位置数」。
- **调整长度（resizing）**：当填装因子过高时，创建一个更大（通常是原来两倍）的新数组，并用散列函数把所有元素重新插入其中。
- **常量时间（O(1)）**：不管数据规模多大，操作所需时间都相同，散列表在平均情况下的查找、插入、删除都是这个复杂度。
- **DNS 解析（DNS resolution）**：把域名映射为 IP 地址的过程，是散列表"用于查找"这一用途的现实案例。
- **缓存（cache）**：把已经计算/请求过的结果保存起来，下次直接返回而不必重新计算，缓存数据通常就存储在散列表中。

## 思维模型 (Mental Models)

- 当你需要"输入一个名字，立刻得到对应的值"（如商品名→价格、姓名→电话号码、URL→IP 地址）时，用散列表，因为它把"查找"这个动作变成了"计算索引"这个动作。
- 把散列函数当作一个不知疲倦、过目不忘的员工（书中的 Maggie）来理解：不管问她多少种商品的价格，她都能立刻答上来，因为她不需要"查找"，只需要"计算"。
- 当担心散列表变慢时，把"填装因子"当作拥挤程度的体温计——数值越低，冲突概率越小，性能越接近理想的 O(1)；一旦"发烧"（超过 0.7）就该调整长度物理降温。
- 当两个键算出同一个索引（冲突）时，把该位置当作一个小链表的"信箱"来理解：多个键共享同一格子，取信时要在这个小链表里逐个比对，链表越短代价越小。

## 常见陷阱 (Anti-patterns)

- 使用不一致的散列函数（如引入随机数 `rand()`）：同一个键两次得到不同索引，会导致存进去的元素再也找不回来，散列表因此彻底失效。
- 使用把所有输入都映射到同一个位置的散列函数（如恒定返回 1）：这会让散列表退化成一个巨大的链表，查找退化为 O(n) 的线性查找，失去了散列表存在的意义。
- 只关注"平均情况"而忽视"最糟情况"：散列表在最糟情况下（冲突严重、链表很长）所有操作都会退化为 O(n)，因此必须靠"低填装因子 + 良好散列函数"主动避开这种最糟情况，而不是心存侥幸。
- 让填装因子无限增长而不调整长度：填装因子越高，冲突概率越大，性能会持续下降，业界经验是超过 0.7 就该扩容。

## 代码示例 (Code Examples)

```python
voted = {}
def check_voter(name):
  if voted.get(name):
    print "kick them out!"
  else:
    voted[name] = True
    print "let them vote!"
```
这段代码展示了散列表在"防止重复"场景下的典型用法：用 `get` 检查键是否已存在（已投票），存在则拒绝，不存在则记录下来并放行，全程无需遍历名单，一次查找即可判定。

## 参考表格 (Reference Tables)

| 操作 | 数组 | 链表 | 散列表（平均情况） | 散列表（最糟情况） |
|---|---|---|---|---|
| 查找 | O(1) | O(n) | O(1) | O(n) |
| 插入 | O(n) | O(1) | O(1) | O(n) |
| 删除 | O(n) | O(1) | O(1) | O(n) |

散列表兼具数组（查找快）和链表（插入、删除快）的优点，但前提是要避开最糟情况——这依赖于低填装因子和良好的散列函数。

## 讲解示例 (Worked Example)

书中以"杂货店收银员查价格"为主线场景：假设有一个包含 5 个位置的数组用来存商品价格。散列函数计算后，`apple` 映射到索引 3，于是苹果的价格存进数组索引 3 处；`milk` 映射到索引 0，牛奶价格存进索引 0 处。之后要查 `avocado` 的价格时，不需要遍历数组，只需把 `avocado` 交给同一个散列函数，得到索引 4，直接去索引 4 处取值即可命中——全程没有"查找"这个动作，只有一次函数计算。

冲突场景的示例：假设散列函数按字母表顺序给 26 个位置分配索引，`apple`（首字母 a）分到位置 1，`avocado`（也是 a 打头）也会分到同一个位置——这就是冲突。解决办法是在该位置存一个链表：查 `banana` 之类字母靠后、不冲突的商品依然是 O(1)，但查 `apple` 时要先定位到位置 1，再在这个小链表里逐个比较键名，找到 apple 才算完成。如果杂货店恰好只卖 A 开头的商品，所有元素都会挤进同一个链表，散列表就退化成了一个链表，速度变得很慢——这正说明了"好的散列函数应让值均匀分布"的重要性。

填装因子的示例：数组一共 5 个位置，其中 2 个已被占用，填装因子为 2/5 = 0.4；若 100 种商品对应恰好 100 个位置，填装因子为 1（理想）；若只有 50 个位置却要存 100 种商品的价格，填装因子变成 2，说明位置已经严重不够用，必须扩容——新建一个两倍大小的数组，用散列函数把所有元素重新插入进去，例如某散列表调整后填装因子从接近 1 降到 3/8，冲突概率随之显著降低。

## 学习要点 (Key Takeaways)

- 散列表 = 数组 + 散列函数，散列函数负责把键映射成数组索引。
- 散列函数必须满足"一致性"（同输入同输出）和"均匀分布"（不同输入尽量映射到不同索引）两个要求，否则散列表要么找不回数据，要么退化变慢。
- 冲突（两个键映射到同一位置）不可避免，最常见的处理方式是在冲突位置存一个链表。
- 散列表在平均情况下所有操作（查找、插入、删除）均为 O(1)（常量时间）；但在最糟情况下会退化为 O(n)。
- 填装因子 = 占用位置数 / 总位置数，一旦超过 0.7 就应该调整（通常翻倍）散列表长度以降低冲突概率。
- 散列表三大典型应用：（1）模拟映射关系（电话簿、DNS 解析）；（2）防止重复（如投票去重）；（3）缓存/记住已处理过的数据（如 Web 服务器缓存页面）。
- 绝大多数编程语言都已内置散列表实现（Python 中就是 `dict`），一般无需自己实现，只需知道如何正确使用及其性能特征。
