# 第2章：选择排序

## 核心思想 (Core Idea)

本章表面讲的是"选择排序"这个具体算法，但真正的重点是数组（Array）与链表（Linked List）这两种基础存储方式的读写权衡——选择排序的效率高低，最终取决于你用哪种结构去实现"反复找最小值"这个操作。

## 引入的框架与算法 (Frameworks Introduced)

**数组（Array）**
- 何时使用：需要频繁随机访问（按索引直接取任意元素）时。
- 工作原理：元素在内存中首尾相连地存放，知道起始地址和索引后，通过简单的数学运算（起始地址 + 索引）即可算出任意元素的地址，因此读取是 O(1)。

**链表（Linked List）**
- 何时使用：需要频繁在任意位置插入或删除、且很少需要"跳跃式"读取时。
- 工作原理：元素可以散落在内存任意位置，每个元素额外存储"下一个元素的地址"，从而像寻宝游戏的纸条一样串联起来。插入/删除只需修改前一个元素记录的地址，不必搬动其他元素。

**选择排序（Selection Sort）**
- 何时使用：作为理解排序问题和大 O 增长规律的入门算法，实际工程中因 O(n²) 效率较低，很少用于大规模数据。
- 工作原理（步骤）：
  1. 遍历当前未排序部分的所有元素，找出其中最小（或最大）的一个；
  2. 将该元素从原数组中取出，加入到新数组的末尾；
  3. 重复步骤 1–2，直到原数组被取空，新数组即为有序结果。

## 关键概念 (Key Concepts)

- **数组（Array）**：元素在内存中连续存放的数据结构，支持通过索引直接定位任意元素。
- **链表（Linked List）**：元素分散存放、每个元素记录下一个元素地址的数据结构，插入删除代价低。
- **索引（Index）**：数组中元素的位置编号，从 0 开始计数，而非从 1 开始。
- **随机访问（Random Access）**：可以直接跳转到指定位置读取元素的访问方式，是数组的强项。
- **顺序访问（Sequential Access）**：必须从第一个元素开始、沿链接逐个读取才能到达目标元素的方式，链表只能这样访问。
- **大 O 中的常数被省略**：选择排序每轮比较次数依次为 n, n−1, …, 1，总次数为 n²/2，但大 O 表示法忽略常数系数，因此写作 O(n²) 而非 O(n²/2)。
- **预留空间（over-provisioning）**：为数组预先申请比当前所需更多的位置以延缓"搬家"，是一种权宜之计而非根本解决方案。

## 思维模型 (Mental Models)

- 把数组当作电影院里连续的一排座位来理解：找到座位坐下很快（随机访问 O(1)），但想临时加一个座位，要么后面没空位得挪全场，要么干脆换一整排（插入代价高）。
- 把链表当作寻宝游戏来理解：每张纸条只写着"下一处宝藏的地址"，插入/删除只需要改一张纸条（O(1)），但想找第十个宝藏，必须从第一张纸条开始一路找过去（O(n)）。
- 当需要频繁随机读取（如二分查找、按下标取值）时，用数组；当需要频繁在任意位置插入/删除、且很少乱序读取时，用链表。
- 把选择排序当作"每轮从剩余堆里挑出最小的放进新堆"来理解：n 轮操作、每轮一次 O(n) 的线性扫描，两者相乘得到 O(n × n) = O(n²)。

## 常见陷阱 (Anti-patterns)

- **误以为"预留空位"能一劳永逸解决数组插入慢的问题**：这只是权宜之计，存在两个缺点——预留的位置可能根本用不上（浪费内存，且别人也无法使用），一旦实际元素数超过预留额度，依然要重新分配整块内存并搬移所有元素。
- **误以为删除元素和插入元素一样可能失败**：插入可能因为内存中找不到足够连续空间而失败，但删除在任何情况下都能成功（无非是需要把后面的元素前移）。
- **误以为链表删除永远是 O(1)**：只有当能够立即访问到待删除元素时（例如已记录链表的第一个或最后一个元素），删除才是 O(1)；如果需要先顺序查找到该元素，那么查找本身仍是 O(n)。

## 代码示例 (Code Examples)

> 说明：原书页面在 PDF 抽取过程中丢失了部分行首缩进，导致抽取文本本身不是可运行的 Python；下面按标准 Python 缩进复原，逻辑与原书一致。

```python
def findSmallest(arr):
    smallest = arr[0]
    smallest_index = 0
    for i in range(1, len(arr)):
        if arr[i] < smallest:
            smallest = arr[i]
            smallest_index = i
    return smallest_index

def selectionSort(arr):
    newArr = []
    for i in range(len(arr)):
        smallest = findSmallest(arr)
        newArr.append(arr.pop(smallest))
    return newArr

print(selectionSort([5, 3, 6, 2, 10]))
```
这段代码演示了选择排序的完整实现：`findSmallest` 负责 O(n) 地找出当前数组中最小元素的索引，`selectionSort` 则调用它 n 次，每次把找到的最小元素从原数组弹出并追加到新数组末尾。

## 参考表格 (Reference Tables)

依据本章正文对数组、链表读写行为的描述整理：

| 操作 | 数组（Array） | 链表（Linked List） |
|---|---|---|
| 读取（随机访问） | O(1) | O(n)（只能顺序访问） |
| 插入 | O(n)（后面的元素需整体后移，空间不足时还需整体搬迁） | O(1)（仅需修改前一个元素记录的地址） |
| 删除 | O(n)（后面的元素需整体前移） | O(1)（前提是已知待删除元素的位置，如链表首尾） |

## 讲解示例 (Worked Example)

用书中同款数组 `[5, 3, 6, 2, 10]` 手动跟踪选择排序的执行过程：

| 轮次 | 待处理数组（原数组剩余部分） | 本轮找到的最小值（及其索引） | 取出后新数组 |
|---|---|---|---|
| 1 | [5, 3, 6, 2, 10] | 2（索引 3） | [2] |
| 2 | [5, 3, 6, 10] | 3（索引 1） | [2, 3] |
| 3 | [5, 6, 10] | 5（索引 0） | [2, 3, 5] |
| 4 | [6, 10] | 6（索引 0） | [2, 3, 5, 6] |
| 5 | [10] | 10（索引 0） | [2, 3, 5, 6, 10] |

最终得到有序数组 `[2, 3, 5, 6, 10]`。可以看到：第一轮要扫描全部 5 个元素才能找到最小值，第二轮只需扫描剩下的 4 个，以此类推，扫描次数依次为 5, 4, 3, 2, 1——这正是"每轮检查元素数递减，但总运行时间仍是 O(n²)"这一反直觉结论的直接来源：5+4+3+2+1 = 15 ≈ n²/2，大 O 只保留增长阶而丢弃常数 1/2，于是记作 O(n²)。

书中另给出一个更贴近应用场景的版本：把待办事项换成"乐队及其被播放次数"，目标是按播放次数从多到少排序——做法完全相同：每轮从剩余乐队中挑出播放次数最多的一个，加入新列表，重复该操作直至列表为空。

## 学习要点 (Key Takeaways)

- 选择排序的时间复杂度是 **O(n²)**：外层循环 n 次，每次内层查找最小值都是 O(n)，n × n = n²。
- 判断"该用数组还是链表"的核心标准是：**是否需要随机访问**（选数组）还是**是否需要频繁插入/删除**（选链表）；二者不可兼得。
- 数组的插入/删除是 O(n)（需要搬移或整体重新分配内存），链表的插入/删除是 O(1)（前提是已知插入/删除位置）。
- 链表只能顺序访问，因此读取第 k 个元素的开销是 O(n)；数组凭借地址计算可以 O(1) 随机访问。
- 数组要求所有元素类型一致（如都为 int 或都为 double），这是它能用简单数学公式计算地址的前提。
- "预留空间"只是缓解数组插入慢问题的权宜之计，不是根本解法；根本解法是改用链表，或使用数组+链表的混合结构（如按首字母分桶的链表数组）。
- 选择排序虽然直观、易实现，但速度不理想；后续会学到运行时间为 O(n log n) 的快速排序作为更优方案。
