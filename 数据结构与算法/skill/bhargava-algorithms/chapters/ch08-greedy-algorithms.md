# 第8章：贪婪算法

## 核心思想 (Core Idea)

贪婪算法（Greedy Algorithm）每一步都只选择当前看来最优的局部解，不做全局回溯或穷举；它不保证总能得到全局最优解，但实现极其简单、速度快，往往能得到与最优解足够接近的近似解——这在很多"完美是优秀的敌人"的现实场景中已经够用。

## 引入的框架与算法 (Frameworks Introduced)

**贪婪算法（Greedy Algorithm）**
- 适用场景：需要快速得到"足够好"的解，而精确算法运行时间不可接受（尤其是 NP 完全问题）。
- 工作方式：反复执行同一条规则——"在当前所有可选项中挑出局部最优的那个，然后在剩余问题上重复"，直到问题被完全解决。是否能得到全局最优解取决于具体问题的结构。

**教室排课问题（活动选择）的贪婪解法**
- 何时用：给定一组有时间冲突的活动/课程，要选出数量最多且互不冲突的一组。
- 步骤：①在所有课程中选结束时间最早的一堂，作为已选课程；②在剩余课程中，只保留与已选课程不冲突（即开始时间晚于上一堂课结束时间）的课程，再从中选结束最早的一堂；③重复直到无课可选。此贪婪策略对这一问题恰好能得到全局最优解。

**集合覆盖问题（Set Covering Problem）的近似算法**
- 何时用：需要从一批候选集合中，选出数量最少、并集能覆盖目标全集的子集组合（如广播台覆盖 50 个州）；精确解需要枚举幂集（power set），共 2ⁿ 个子集，运行时间 O(2ⁿ)，n 稍大就完全不可行。
- 步骤：①在所有候选集合中，选出能覆盖"当前还未被覆盖的元素"最多的那一个（哪怕它同时覆盖了一些已覆盖的元素也没关系）；②从"待覆盖集合"中减去它已覆盖的部分；③重复①②直到待覆盖集合为空。运行时间为 O(n²)，得到的是近似解，不一定是元素最少的组合，但非常接近。

## 关键概念 (Key Concepts)

- **局部最优解（Local Optimum）**：只考虑当前这一步能拿到的最好选择，不考虑对后续步骤的全局影响。
- **全局最优解（Global Optimum）**：整个问题在所有可能方案中真正的最优解。
- **近似算法（Approximation Algorithm）**：当精确解耗时过长时，用来快速得到"足够接近"最优解的算法；评价标准是速度和与最优解的接近程度。
- **幂集（Power Set）**：一个集合所有子集构成的集合，大小为 2ⁿ，是集合覆盖问题精确解法需要枚举的对象。
- **NP 完全问题（NP-complete Problem）**：以难解著称的一类问题（如旅行商问题、集合覆盖问题），目前没有已知的可以快速求解的算法。
- **阶乘函数（Factorial Function）**：旅行商问题中可能路线数的增长规律（n 个城市对应约 n! 条路线），增长速度远超指数。
- **交集 / 并集 / 差集（Intersection / Union / Difference）**：Python 中集合（set）的三种基本运算，分别记作 `&`、`|`、`-`，是实现集合覆盖贪婪算法的核心工具。
- **集合（Set）**：类似列表但不允许重复元素的数据结构，本章用它同时表示"待覆盖的州"和"每个广播台覆盖的州"。

## 思维模型 (Mental Models)

- 当问题可以被分解成"一连串独立的选择步骤"，且每一步的局部最优看起来都合理时，先尝试用贪婪算法探路——即使它不一定最优，也能快速给出一个基准答案。
- 把贪婪算法当作"每次只问一个问题：现在怎么选最好"来理解，它从不后悔、也不回头重新考虑之前的选择。
- 当算法的运行时间随规模增长呈阶乘或指数级爆炸（如枚举所有子集、所有路径排列）时，把这类问题当作 NP 完全问题的疑似对象，转而寻找近似算法而非死磕精确解。
- 把"背包问题"当作贪婪算法失效的反例来理解：贪婪策略在有些问题上退化为"看起来合理但结果次优"的陷阱，这提示我们贪婪算法的适用性必须针对具体问题验证，不能想当然地推广。

## 常见陷阱 (Anti-patterns)

- 认为贪婪算法在所有问题上都能得到最优解：本章用背包问题（35 磅背包）直接证伪——先偷最贵的音响会把背包塞满、放不下笔记本电脑和吉他，总价 3000 美元不如后者组合的 3500 美元。贪婪策略只是"每步局部最优"，不代表全局最优，必须针对每个问题单独证明或验证其正确性。
- 对可能是 NP 完全的问题坚持寻找精确算法：当问题涉及"所有组合"、无法拆分成独立子问题、涉及序列或集合且难以求解时，继续硬算只会让运行时间随规模爆炸式增长（如 10 个城市的旅行商问题已有超过 300 万条路径需要比较），应及时转向近似算法。

## 代码示例 (Code Examples)

```python
states_needed = set(["mt", "wa", "or", "id", "nv", "ut",
"ca", "az"])

stations = {}
stations["kone"] = set(["id", "nv", "ut"])
stations["ktwo"] = set(["wa", "id", "mt"])
stations["kthree"] = set(["or", "nv", "ca"])
stations["kfour"] = set(["nv", "ut"])
stations["kfive"] = set(["ca", "az"])

final_stations = set()

while states_needed:
  best_station = None
  states_covered = set()
  for station, states in stations.items():
    covered = states_needed & states
    if len(covered) > len(states_covered):
      best_station = station
      states_covered = covered
  states_needed -= states_covered
  final_stations.add(best_station)
```
这段代码是集合覆盖问题贪婪近似算法的完整实现：每轮 while 循环都遍历所有广播台，用集合交集 `&` 找出覆盖"剩余未覆盖州"最多的一个，选中后用差集 `-=` 更新剩余任务，直到 `states_needed` 为空——是本章"贪婪+集合运算"思路的最典型体现。

## 讲解示例 (Worked Example)

**广播台集合覆盖**：目标是用最少的广播台覆盖 8 个州 `{mt, wa, or, id, nv, ut, ca, az}`，候选广播台为：kone={id,nv,ut}、ktwo={wa,id,mt}、kthree={or,nv,ca}、kfour={nv,ut}、kfive={ca,az}。

贪婪算法逐轮执行：
- 第 1 轮：比较各广播台与"全部 8 个州"的交集大小，kone、ktwo、kthree 都覆盖 3 个未覆盖州，取其一（如 kone，覆盖 id、nv、ut）；剩余待覆盖州变为 `{mt, wa, or, ca, az}`。
- 第 2 轮：ktwo 与剩余州交集为 {mt, wa}（2 个），kthree 为 {or, ca}（2 个）；假设选中 ktwo，剩余变为 `{or, ca, az}`。
- 第 3 轮：kthree 覆盖 {or, ca}（2 个），kfive 覆盖 {ca, az}（2 个）；假设选中 kthree，剩余变为 `{az}`。
- 第 4 轮：只有 kfive 覆盖 az，选中 kfive，剩余变为空集，循环结束。

最终结果 `final_stations = {kone, ktwo, kthree, kfive}`，共 4 个广播台，与精确算法可能给出的最优解（如 {ktwo, kthree, kfour, kfive}）站点数量相同、但具体组合不同——这说明贪婪算法找到的是"足够好"的近似解，不保证与精确解完全一致。

## 学习要点 (Key Takeaways)

- 贪婪算法 = 每一步都选局部最优解，实现简单、速度快，但不保证得到全局最优解。
- 教室排课问题中，"每次选结束最早的课"这一贪婪策略恰好能得到全局最优解——这是贪婪算法适用的正例。
- 背包问题中，"每次偷最贵商品"的贪婪策略不能保证全局最优——这是贪婪算法失效的反例，考试中常用来考察"贪婪 vs 最优"的辨析。
- 集合覆盖问题的精确解需枚举幂集，运行时间 O(2ⁿ)；贪婪近似算法运行时间 O(n²)，用速度换取"足够接近"的解。
- NP 完全问题（如旅行商问题、集合覆盖问题）目前没有已知的多项式时间精确算法，面对它们应转向贪婪法等近似算法。
- 判断 NP 完全问题没有充分条件，只有一些经验性蛛丝马迹（组合爆炸、无法分解、涉及序列或集合、可归约为已知 NP 完全问题等）。
- Python 中集合（set）及其并集 `|`、交集 `&`、差集 `-` 运算是实现集合覆盖类贪婪算法的常用工具。
