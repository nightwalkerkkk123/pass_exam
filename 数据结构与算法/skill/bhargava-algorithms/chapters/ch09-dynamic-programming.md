# 第9章：动态规划

## 核心思想 (Core Idea)

动态规划（Dynamic Programming，DP）把一个难以直接求解的大问题拆成一系列彼此独立、离散的子问题，用一张网格记录每个子问题的最优解，再通过一个递推公式把子问题的解组合起来，逐步推出原问题的最优解。

## 引入的框架与算法 (Frameworks Introduced)

**动态规划（Dynamic Programming, DP）**
- 何时使用：需要在给定约束条件（如背包容量、时间预算）下让某个指标最优（最大化价值、最大化相似度等），且问题能拆解为彼此独立、不互相依赖的子问题。
- 工作原理：
  1. 画一张网格：想清楚"单元格代表哪个子问题"，坐标轴（行/列）自然浮现——通常一维是可选对象（商品、字符），另一维是约束的离散取值（容量、另一字符串的前缀）。
  2. 单元格中存放的值通常就是要优化的那个指标本身。
  3. 逐行（或逐列，两者本质等价）填充网格，每个单元格的值由前面已经算出的相邻单元格通过一个**问题专属**的公式推出，没有通用公式，需要针对具体问题反复尝试（书中戏称为"费曼算法（Feynman algorithm）"：写下问题→好好思考→写下答案）。
  4. 网格填满后从中读出答案——注意读取位置因问题而异（见下文"参考表格"）。

**暴力枚举法（brute force）**：作为对照出现——尝试所有商品组合再挑价值最高的，n件商品需要计算 2ⁿ 个组合，运行时间 O(2ⁿ)，商品一多就完全不可行，这正是引出动态规划的原因。

## 关键概念 (Key Concepts)

- **背包问题（Knapsack Problem）**：在背包容量约束下，从一组不可分割的商品中选出使总价值最大的子集，是动态规划最经典的入门场景。
- **动态规划网格**：行通常对应可选商品或字符，列对应约束的离散取值（如容量 1～4 磅），单元格存放对应子问题的最优解。
- **递推公式 / 状态转移方程**：描述如何用相邻已算出的单元格值推导当前单元格值的规则；因问题而异，没有放之四海而皆准的通用公式。
- **最长公共子串（Longest Common Substring）**：两个字符串中都存在的、要求连续的字符序列的最大长度，可用来估计拼写相似度。
- **最长公共子序列（Longest Common Subsequence, LCS）**：两个字符串中都包含、但不要求连续排列的字符序列的最大长度，比最长公共子串更能反映真实相似度。
- **离散子问题（discrete/independent subproblem）**：子问题彼此互不依赖，是动态规划能够正确工作的前提条件；一旦子问题相互依赖，动态规划就会失效。
- **编辑距离（Levenshtein Distance）**：衡量两个字符串相似程度的指标，也是通过动态规划计算的，广泛用于拼写检查、查重等场景。

## 思维模型 (Mental Models)

- 当问题可以"先解决小规模版本、再合并得到大规模版本的解"时，用动态规划网格来组织计算，而不是像暴力枚举那样穷举所有组合（O(2ⁿ) 太慢）。
- 把每个单元格当作一个独立的子问题来理解：先想清楚"这个格子代表什么容量/什么前缀组合"，坐标轴和公式自然就浮现出来了。
- 当子问题之间存在相互依赖（做了 A 之后 B 的代价/收益会变）时，把动态规划当作**失效**的工具，因为它的正确性建立在子问题彼此独立这一前提上。
- 当资源/商品可以被分割（连续量而非整取整不取）时，用贪婪算法替代动态规划——动态规划只能回答"拿整件还是不拿"，回答不了"拿多少"。

## 常见陷阱 (Anti-patterns)

- 对存在相互依赖的子问题（如"伦敦+巴黎"多站旅行：去了埃菲尔铁塔后卢浮宫会变"便宜"）硬套动态规划——这类子问题不满足独立性前提，会得到错误答案而不自知。
- 误以为动态规划能处理"拿走商品的一部分"——动态规划只做二元决策（整取/不取），可分割场景应改用贪婪算法。
- 误以为答案必然出现在网格最后一个单元格——这对 0/1 背包问题成立，但对最长公共子串问题，答案是网格中的**最大值**，可能出现在任意一格。
- 期待存在一个通用的单元格计算公式——不同问题的递推公式必须单独尝试推导，没有可以照搬的模板。

## 代码示例 (Code Examples)

```python
if word_a[i] == word_b[j]:
    cell[i][j] = cell[i-1][j-1] + 1
else:
    cell[i][j] = max(cell[i-1][j], cell[i][j-1])
```
这是最长公共子序列（LCS）的状态转移方程：两个字符匹配时在左上角对角线单元格的基础上加一；不匹配时取"左边单元格"与"上边单元格"中的较大值（对比最长公共子串问题中"不匹配则清零"的规则）。

## 参考表格 (Reference Tables)

| 问题 | 网格坐标轴 | 递推公式（匹配/命中 vs 不匹配/未命中） | 答案读取位置 |
|---|---|---|---|
| 0/1 背包问题 | 商品（行）× 容量（列） | max(不拿本行商品的价值, 拿本行商品价值 + 剩余容量下前几行的最大价值) | 网格最后一个单元格 |
| 最长公共子串 | 单词 A 字符 × 单词 B 字符 | 匹配：cell[i-1][j-1]+1；不匹配：0 | 网格中的**最大值**（未必在末格） |
| 最长公共子序列 | 单词 A 字符 × 单词 B 字符 | 匹配：cell[i-1][j-1]+1；不匹配：max(cell[i-1][j], cell[i][j-1]) | 网格最后一个单元格 |

## 讲解示例 (Worked Example)

**背包问题主例**：容量 4 磅的背包，可偷吉他（1 磅，1500 美元）、音响（4 磅，3000 美元）、笔记本电脑（3 磅，2000 美元）。

- 吉他行：容量 1～4 磅全部能装下吉他，所以每格最大价值都是 1500。
- 音响行：容量 1～3 磅装不下音响（太重），维持 1500；容量 4 磅能装下音响，比较"不拿音响（1500）" vs "拿音响（3000）"，更新为 **3000**。
- 笔记本电脑行：容量 1～2 磅装不下（维持 1500）；容量 3 磅可换成笔记本电脑（2000 > 1500，更新为 2000）；容量 4 磅是关键——"不拿笔记本电脑"沿用上一行的 3000，"拿笔记本电脑"则占用 3 磅、剩 1 磅，1 磅容量下之前算出的最大价值是 1500（吉他），2000+1500=3500 > 3000，更新为 **3500**（吉他+笔记本电脑）。

**加入 iPhone（1 磅，2000 美元）** 后重新填一行：容量 4 磅时，"不拿 iPhone"沿用 3500；"拿 iPhone"占 1 磅、剩 3 磅，3 磅容量下（笔记本电脑行）的最大价值是 2000，2000+2000=**4000**，成为新的最优解（iPhone + 笔记本电脑）。

**最长公共子串/子序列例**：Alex 把 fish 误输成 hish。hish 与 fish 的最长公共子串是 "ish"（长度 3），hish 与 vista 的最长公共子串是 "is"（长度 2）——所以更可能是 fish。当误输为 fosh 时，fosh 与 fish、fosh 与 fort 的最长公共子串长度打平（都是 2），改用最长公共子序列衡量后，fosh 与 fish 的相似度明显更高，从而正确判断用户想输入的是 fish。

## 学习要点 (Key Takeaways)

- 动态规划的两个适用前提缺一不可：①在约束条件下求最优值；②问题可拆解为彼此独立的离散子问题。
- 每种 DP 方案都先画网格：确定"单元格代表哪个子问题"，坐标轴随之明确；单元格里存的通常就是要优化的那个指标本身。
- 没有放之四海而皆准的递推公式，需要针对每个问题单独摸索（费曼算法式的自嘲）。
- 0/1 背包问题答案在网格最后一个单元格；最长公共子串问题答案是网格中的最大值，未必在最后一格——考试要分清两类问题的读取方式。
- 行/列的填充顺序对背包问题和最长公共子串结果无影响，但换一个问题时未必如此，需具体分析。
- 遇到子问题相互依赖（不独立）的场景（如多站旅行行程），动态规划失效，需换用其他算法思路。
- 商品可分割时用贪婪算法而非动态规划；动态规划只能做"整取或不取"的二元判断，这正是与第 8 章贪婪算法的分工边界。
