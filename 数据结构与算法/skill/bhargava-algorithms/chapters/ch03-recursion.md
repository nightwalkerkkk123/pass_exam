# 第3章：递归

## 核心思想 (Core Idea)
递归（Recursion）指函数调用自身，用它可以把"重复处理直到条件满足"的问题表达得比循环更清晰；但递归的清晰是以调用栈（call stack）的内存开销为代价的，二者是同一枚硬币的两面。

## 引入的框架与算法 (Frameworks Introduced)
本章没有引入具体排序/搜索算法，而是引入了一种**编程范式**：

- **递归（Recursion）**：函数在自己的定义体内调用自己。使用场景：问题可以自然地分解为"处理一小部分 + 把剩余部分交给同样的处理逻辑"，例如遍历嵌套结构（盒子中有盒子）、计算阶乘、遍历树/链表。工作原理：
  1. 写出**基线条件（base case）**——函数不再调用自己、直接返回结果的情形，防止无限循环。
  2. 写出**递归条件（recursive case）**——函数调用自身、处理更小规模子问题的情形。
  3. 每次递归调用都会在**调用栈**上压入一个新的栈帧，保存该次调用的局部变量；基线条件触发后，栈帧从顶部依次弹出，逐层返回结果。

## 关键概念 (Key Concepts)
- **递归（Recursion）**：函数直接或间接调用自身来解决问题。
- **基线条件（base case）**：递归函数中不再继续递归、直接返回的终止条件，是避免无限循环的关键。
- **递归条件（recursive case）**：递归函数中函数调用自身、把问题规模缩小后继续求解的部分。
- **调用栈（call stack）**：计算机用来管理函数调用的栈结构，每次函数调用都会分配一块内存（栈帧）存储该次调用的变量。
- **栈（Stack）**：只支持"压入（push）"和"弹出（pop）"两种操作的数据结构，后进先出（LIFO）。
- **栈溢出（stack overflow，本章隐含）**：调用栈过高、占用内存过多时可能耗尽内存，是递归相比循环的主要代价。
- **尾递归（tail recursion）**：一种可被编译器/解释器优化以避免栈增长的高级递归技巧，本章明确指出"不在讨论范围内"，且并非所有语言都支持。

## 思维模型 (Mental Models)
- 当问题是"在嵌套结构里查找某样东西，且不确定嵌套深度"时，用递归——每层嵌套对应一次函数调用，无需自己维护"还有哪些待查找的盒子"，调用栈会自动替你记住。
- 把每一次递归调用当作往一叠便条（栈）上压入一张新便条来理解：只关心最上面那张（当前调用），下面的便条（外层调用）在被弹出之前始终"暂停并处于未完成状态"，等待其内部调用返回。
- 当思路卡在"这样写会不会无限循环"时，先问自己两件事：这个函数的基线条件是什么？递归条件有没有让问题规模严格变小、最终一定能触达基线条件？
- 递归 vs 循环不是非此即彼的性能选择题，而是"清晰度"与"内存代价"的权衡：能用循环写清楚就不必强行递归；递归更适合表达本身就是嵌套/自相似结构的问题。

## 常见陷阱 (Anti-patterns)
- **忘记写基线条件**：如本章最初的 `countdown` 函数示例，没有终止条件会导致无限递归（表现为不断打印 `3...2...1...0...-1...-2...` 直至手动中断），必须显式加入 `if i <= 0: return` 之类的判断。
- **误以为递归总是比循环快**：书中明确指出"递归只是让解决方案更清晰，并没有性能上的优势，使用循环的性能可能更好"，选择递归应基于可读性而非性能预期。
- **忽视调用栈的内存代价**：递归层数很深时，栈会变得很高，每层调用都占用内存，可能带来严重的内存开销；此时应考虑改写为循环，或使用尾递归（如语言支持）。

## 代码示例 (Code Examples)
```python
def fact(x):
  if x == 1:
    return 1
  else:
    return x * fact(x-1)
```
计算阶乘的递归函数：`x == 1` 是基线条件，`return x * fact(x-1)` 是递归条件，每次调用把问题规模从 x 缩小到 x-1，直至触达基线条件。

## 讲解示例 (Worked Example)
**在嵌套盒子中查找钥匙**：假设有一堆盒子，盒子里可能还装着盒子，钥匙就藏在某个盒子里。

- 循环写法：维护一个"待查找盒子堆"，用 `while` 循环反复从堆中取出一个盒子；若里面还是盒子就放回堆里，若是钥匙就宣告完成。这里"盒子堆"是程序员手动维护的一个显式列表。
- 递归写法：定义 `look_for_key(box)`，遍历 box 中的每个 item；若 item 是盒子，直接递归调用 `look_for_key(item)`；若 item 是钥匙，宣告完成。这里完全没有显式的"盒子堆"——它其实被隐式地存储在了调用栈中：每一次尚未处理完的 `look_for_key` 调用，都对应着"还有没查完的盒子"这一事实。

**调用栈追踪 `fact(3)`**：调用 `fact(3)` 时，栈从底到顶依次压入 `fact(3)`（x=3，等待 `fact(2)` 返回）→ `fact(2)`（x=2，等待 `fact(1)` 返回）→ `fact(1)`（x=1，命中基线条件，直接返回 1）。之后栈帧从顶部依次弹出并回传结果：`fact(1)` 返回 1 给 `fact(2)`，`fact(2)` 算出 `2*1=2` 返回给 `fact(3)`，`fact(3)` 算出 `3*2=6` 作为最终结果。每个 `fact` 调用都有自己独立的 `x`，互不干扰——这正是栈帧隔离局部变量的体现。

同理，书中 `greet` → `greet2`/`bye` 的例子说明：调用另一个函数时，当前函数（如 `greet`）会暂停并保持"未完成"状态压在栈里，等被调用函数（`greet2`）执行完、其栈帧弹出后，才会回到 `greet` 中断的地方继续往下执行。

## 学习要点 (Key Takeaways)
- 每个正确的递归函数必须同时具备基线条件和递归条件，缺少基线条件会导致无限递归。
- 递归的价值在于**代码清晰度**，不在于性能；能用循环解决时，循环通常性能更好。
- 所有函数调用（无论是否递归）都会占用调用栈，栈帧保存该次调用的局部变量。
- 递归调用栈可以隐式地替代手动维护的"待处理列表"（如本章的盒子堆例子），这是递归常被用于遍历嵌套/树形结构的原因。
- 递归层数过深会导致调用栈过高、内存占用过大；解决办法是改写为循环，或使用尾递归（如果语言支持）。
- 阶乘函数 `fact(x)` 是理解递归调用栈如何逐层压入、逐层弹出并回传结果的标准范例，考试中常以画栈帧图的形式考察。
