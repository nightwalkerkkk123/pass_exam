# 《算法图解》模式速查表 (Patterns Cheat Sheet)

## 二分查找 (Binary Search)
- **使用时机**: 列表已排序，需判断某元素是否存在及其位置。
- **方法**: 维护 low/high 指针框定区间，每轮比较中间元素，收缩到左半或右半，直到命中或区间为空。
- **权衡**: 最坏情况从 O(n) 降到 O(log n)，但要求预先排序、需随机访问（对链表不友好）。
- **出自**: ch01

## 大O表示法 (Big-O Notation)
- **使用时机**: 判断算法在数据规模变大后是否仍可用，或对比两种算法孰优孰劣。
- **方法**: 统计最坏情况下操作次数，表达为关于 n 的函数，只关注增速曲线而非某次实测耗时。
- **权衡**: 能预测大规模表现，但忽略常数因子，小规模下增速慢的算法也可能更快。
- **出自**: ch01

## 选择排序 (Selection Sort)
- **使用时机**: 入门理解排序与 O(n²) 增长规律；不适合大规模数据的生产环境。
- **方法**: 每轮线性扫描剩余未排序元素找最小值，取出后追加到新数组末尾，重复 n 次。
- **权衡**: 实现简单直观，但 O(n²) 远慢于 O(n log n) 的快排/归并排序。
- **出自**: ch02

## 数组 vs 链表选型 (Array vs Linked List)
- **使用时机**: 设计需存储多元素序列的数据结构前。
- **方法**: 操作以随机读取/按索引访问为主选数组；以任意位置插入删除为主且很少乱序读取选链表。
- **权衡**: 数组读取 O(1) 但插入删除 O(n)；链表插入删除 O(1)（已知位置）但读取/查找 O(n)。
- **出自**: ch02

## 递归基线+递归条件 (Recursion: Base Case + Recursive Case)
- **使用时机**: 问题可自然分解为嵌套/自相似子问题，如遍历嵌套结构、树、链表，或数学递推。
- **方法**: 先定何时不再递归直接返回（基线条件），再写函数调用自身处理缩小规模的部分（递归条件），确保每次朝基线收敛。
- **权衡**: 代码更清晰但不提供性能优势；深层递归占用大量栈内存，必要时应改写为循环。
- **出自**: ch03

## 分而治之 (Divide and Conquer, D&C)
- **使用时机**: 面对不知如何下手的递归型问题时的通用解题策略。
- **方法**: 先找最简单的基线条件，再把原问题分解/缩小规模直到符合基线条件，最后合并子问题的解。
- **权衡**: 是思路而非具体算法，效率取决于如何分解与合并。
- **出自**: ch04

## 快速排序三步法 (Quicksort)
- **使用时机**: 需要通用、平均情况下高效的排序；C 标准库 qsort 即用此实现。
- **方法**: 选基准值 → 按大小分区为 less/greater 两个子数组 → 对子数组递归排序后与基准值合并。
- **权衡**: 平均 O(n log n) 常量小、实践中快；但最糟 O(n²)（如固定选首元素处理已排序数组），需随机化基准值规避。
- **出自**: ch04

## 散列表三大应用 (Hash Table: Mapping / Dedup / Cache)
- **使用时机**: ①需把一种数据映射到另一种（如姓名→电话）；②需判断元素是否已出现过（防重复）；③计算/请求代价高、结果会被重复访问（如网页生成）。
- **方法**: ①以待映射数据为键目标为值，O(1) 存取；②用 dict.get(key) 检查存在性，不存在则记录并放行；③以请求标识为键，先查缓存命中则直接返回，未命中则计算后存入。
- **权衡**: 均以内存换速度，依赖散列函数质量，冲突严重时退化为 O(n)；缓存还需考虑失效/更新策略。
- **出自**: ch05

## 广度优先搜索 (Breadth-First Search, BFS)
- **使用时机**: 判断图中 A 到 B 是否有路径，或求最少边数的最短路径（如社交网络找最近目标）。
- **方法**: 用队列存待检查节点，起点邻居先入队；循环出队检查，不满足目标则邻居入队并标记已检查节点；直到找到目标或队列为空。
- **权衡**: 保证边数最少的路径，需用队列（非栈）并记录已检查节点防重复/死循环；运行时间 O(V+E)。
- **出自**: ch06

## 狄克斯特拉算法 (Dijkstra's Algorithm)
- **使用时机**: 加权图（边权重均为正）中求起点到各节点的总权重最小路径。
- **方法**: 反复执行：选未处理节点中开销最低者 → 更新其邻居开销与父节点（若经它更短） → 标记已处理，直到全部处理完，再沿父节点链回溯出路径。
- **权衡**: 朴素实现每轮线性扫描效率不是最优（可用堆优化）；完全不能处理负权边，此时需换贝尔曼-福特算法。
- **出自**: ch07

## 贪婪算法 (Greedy Algorithm)
- **使用时机**: 需快速得到足够好的解，精确算法运行时间不可接受，尤其是 NP 完全问题。
- **方法**: 每一步都在当前可选项中挑局部最优的一个，然后在剩余问题上重复，直到问题解决。
- **权衡**: 实现简单、速度快，但不保证全局最优，正确性需针对具体问题验证（活动选择可得最优解，背包问题则不能）。
- **出自**: ch08

## 集合覆盖近似算法 (Set Covering Approximation)
- **使用时机**: 需从候选集合中选出数量最少、并集覆盖目标全集的组合，且候选数量大到无法枚举幂集。
- **方法**: 每轮选覆盖当前剩余未覆盖元素最多的候选集合，将其覆盖范围从待覆盖集合扣除，重复直到待覆盖集合为空。
- **权衡**: 运行时间从精确解 O(2^n) 降至 O(n²)，代价是可能得到次优（但接近）的解。
- **出自**: ch08

## 0/1背包动态规划 (0/1 Knapsack DP)
- **使用时机**: 商品不可分割、需在容量/预算约束下求价值最大的组合。
- **方法**: 建立商品行×容量列网格，逐行填充，每格取 max(不拿本行商品的价值, 拿本行商品价值+剩余容量下前几行的最大价值)，答案在最后单元格。
- **权衡**: 只能整取/不取，不能部分拿取；复杂度约 O(商品数×容量)，远优于暴力枚举 O(2^n)。
- **出自**: ch09

## 最长公共子串/子序列网格 (LCS Grid)
- **使用时机**: 比较两个字符串的相似度，如拼写纠错、DNA 比对、diff 工具。
- **方法**: 以两字符串字符为坐标轴建网格；子串：匹配则 cell[i-1][j-1]+1，不匹配则清零；子序列：匹配则 cell[i-1][j-1]+1，不匹配则取 cell[i-1][j] 与 cell[i][j-1] 的较大值。
- **权衡**: 子串要求连续，子序列允许跳跃；子串答案是网格最大值(不一定在末格)，子序列答案通常在末格。
- **出自**: ch09

## 子问题独立性判断 (Independence Check before DP)
- **使用时机**: 在决定是否使用动态规划前，先检查子问题是否彼此独立。
- **方法**: 若一个决策会改变另一子问题的代价/收益（如先到访A地会让B地变"便宜"），说明子问题相互依赖，DP 不适用。
- **权衡**: 误用会得到错误答案而不自知；此时需换用其他建模思路（如图论最短路径）。
- **出自**: ch09

## K最近邻 (K-Nearest Neighbors, KNN)
- **使用时机**: 分类——需把新样本归入已有类别，且"相似样本同类"直觉成立（如水果分类、OCR）；回归——需预测数值型结果（如评分、销量）。
- **方法**: 特征抽取 → 计算与所有已知样本的距离 → 取最近 K 个邻居 → 分类取多数类别，回归取平均值。
- **权衡**: 简单有效但强依赖特征选择质量；K 太小易受异常值影响，太大引入不相关邻居。
- **出自**: ch10

## 余弦相似度 (Cosine Similarity)
- **使用时机**: 比较对象的评分尺度不一致（有人打分宽松有人严格），导致欧氏距离误判品味相近的人为不相邻。
- **方法**: 比较两个特征向量的夹角而非直线距离。
- **权衡**: 更能反映"方向"上的相似度，但具体实现细节需读者课后深入。
- **出自**: ch10

## MapReduce (map+reduce)
- **使用时机**: 数据量大到单机数据库处理不动（数十亿行级别），需要分布式处理。
- **方法**: map 阶段将同一操作分发到多台机器并行施加于每个数据元素；reduce 阶段将各机器结果归并为最终值。
- **权衡**: 处理速度从数小时降到几分钟，代价是需要维护分布式基础设施（如 Hadoop）。
- **出自**: ch11

## 布隆过滤器 (Bloom Filter)
- **使用时机**: 需判断超大集合中是否包含某元素，能容忍偶尔误报但不能漏报，且要求极致省内存。
- **方法**: 用远小于散列表的存储空间给出概率性答案。
- **权衡**: 省空间但可能误报（假阳性），不能用于要求绝对精确的场景。
- **出自**: ch11

## Simhash 局部敏感散列 (Simhash / Locality-Sensitive Hashing)
- **使用时机**: 需判断两段内容的相似程度而非是否完全相同，如查重、反抄袭。
- **方法**: 让相似输入产生相似散列值，通过比较散列值接近程度衡量相似度。
- **权衡**: 与 SHA 设计目标相反，不能用于需要"防碰撞"的安全校验场景。
- **出自**: ch11
