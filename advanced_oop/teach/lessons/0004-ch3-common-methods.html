2 分钟脚手架:为什么 equals 和 hashCode 必须配对

  • 所有 Java 类继承自 Object —— 默认的 equals 是 ==(同一对象),默认的 hashCode 是内存地址导出的整数。
  • 条款 10 · equals 5 性质(破任一 → 集合行为诡异):
    1. 自反:x.equals(x) == true
    2. 对称:x.equals(y) ⇔ y.equals(x)(大小写忽略比较最常犯)
    3. 传递:x.equals(y) 且 y.equals(z) ⇒ x.equals(z)(子类加字段最常犯)
    4. 一致:信息不变,多次调用结果不变
    5. 非空:x.equals(null) == false(用 instanceof 自带这性质)
  • 条款 11 · hashCode 契约(核心,常被忽略):
    • 必备:equals 相等 ⇒ hashCode 必须相等
    • 反过来不要求:hashCode 相等 equals 可以不等(允许碰撞)
    • 反模式:只覆盖 equals 不覆盖 hashCode → 对象放进 HashMap / HashSet 就丢了(put 进的桶和 get 找的桶不同)
    • 标准实现:result = 31 * result + c,31 是奇素数,减少碰撞;Java 7+ 可用 Objects.hash(...) 一行搞定(性能稍差,可读性高)
  • 条款 12 · toString:默认 PhoneNumber@163b91 几乎不可调试;约定「简洁、信息丰富」,让日志和断言能直接读懂。

本课特殊关注(LR-0002 模式匹配风险)

你自评 条款 11 标「认识」,条款 10 标「模糊」。但条款 11 的全部意义建立在「你已经懂 equals 5 性质」上 —— 最可能是 「hashCode 必须跟 equals 一致」这句话字面熟了,契约级理解没建立。

条款 10 · 对称性反例
下面这个 equals 违反 5 性质中的哪一个?
public final class CaseInsensitiveString {
    private final String s;
    public CaseInsensitiveString(String s) { this.s = s; }
    @Override public boolean equals(Object o) {
        if (o instanceof CaseInsensitiveString)
            return s.equalsIgnoreCase(((CaseInsensitiveString) o).s);
        if (o instanceof String)
            return s.equalsIgnoreCase((String) o);   // ① 跟 String 比
        return false;
    }
}
答案:B(对称性)。 ① 处让 CaseInsensitiveString 跟 String 比可以,但 String.equals 不知道 CaseInsensitiveString 存在 → "hello".equals(cis) 是 false,反过来 cis.equals("hello") 是 true。对称性被破。HashSet 里行为会非常诡异:加 cis 进去,再用 "hello" 找可能找不到。条款 10 经典反例。修法:只跟同类型比。
条款 11 · hashCode 契约真伪
下面哪个说法正确?
答案:C。 这是条款 11 的核心契约。A 错(碰撞是允许的);B 错(同样允许碰撞,只是同一桶里再用 equals 区分);D 错(ArrayList 用 equals 不用 hashCode,不爆;但 HashMap / HashSet 必爆:put 进的桶用 hashCode 算,get 时再算一次,不等就找不到 → 对象「丢了」。
条款 11 · 真假 hashCode 实现
哪个 hashCode 实现是正确的?
class PhoneNumber {
    private final short area, prefix, line;
    PhoneNumber(short a, short p, short l) { area=a; prefix=p; line=l; }
    @Override public boolean equals(Object o) {
        if (!(o instanceof PhoneNumber)) return false;
        PhoneNumber pn = (PhoneNumber) o;
        return area==pn.area && prefix==pn.prefix && line==pn.line;
    }
    // 哪个 hashCode 正确?
}
答案:A。 A 标准(Objects.hash 用 31 * result 模式,可读性高,性能可接受)。B 合法但性能糟(HashMap 退化成链表)。C 错:碰撞率极高(碰撞率极高(area+prefix+line 易撞)),HashMap 性能崩。D 错:super.hashCode() 用内存地址作 hash → 两个相等的 PhoneNumber 散到不同桶,违反契约 ① → put/get 找不到对象。A 是「先满足契约,再考虑性能」的标准做法。
条款 11 · 运行时行为
运行下面代码,contains 打印什么?
class Point {
    final int x, y;
    Point(int x, int y) { this.x = x; this.y = y; }
    @Override public boolean equals(Object o) {
        if (!(o instanceof Point)) return false;
        Point p = (Point) o;
        return p.x == x && p.y == y;
    }
    // 注意:没覆盖 hashCode
}

Set<Point> set = new HashSet<>();
set.add(new Point(1, 2));
System.out.println(set.contains(new Point(1, 2)));
答案:B(false)。 两个 Point(1,2) equals 相等(都是 true),但 hashCode 用的是 Object 默认(内存地址)→ 两个对象散到不同桶。contains 算第二个 Point 的 hashCode,去另一个桶找,找不到 → 返回 false。这就是没覆盖 hashCode 的经典翻车现场:equals 说"是同一个",HashSet 说"我找不到"。条款 11 的实战意义就在这一题。
