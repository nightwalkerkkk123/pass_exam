90 秒脚手架：泛型 vs 数组

  • 类型擦除：泛型信息只存在于编译期。运行期 List<String> 和 List<Integer> 是同一个类。
  • 数组协变：String[] 是 Object[] 的子类（数组记着元素类型，运行期也知道）。
  • 泛型不变：List<String> 不是 List<Object> 的子类（擦除后类型没了，硬转就炸）。
  • 为什么混用不安全：数组具体化（记得类型） + 泛型擦除（不记得）= 同一套代码两套规则。所以 Java 直接禁止"创建泛型数组"。

条款 23/25/27/28 全是这套规则的不同侧面。下面 4 题各打一条。

条款 23 · 不要在新代码中使用原始类型
下面这段代码，跑起来会怎样？
public static void main(String[] args) {
    List list = new ArrayList();   // 原始类型 List,没有 <String>
    list.add("hello");
    list.add(42);                  // 编译居然通过!
    for (Object o : list) {
        String s = (String) o;     // 强转
        System.out.println(s.length());
    }
}
答案:B。原始类型 List 绕过了泛型检查,add(42) 编译期不报错。运行期类型擦除后全是 Object,循环到 42 时强转 String → ClassCastException。这正是条款 23 要避的坑——原始类型不报错是因为它没在做检查,不是它"能用"。
条款 25 · 列表优先于数组
下面这个工具方法,经典地违反/符合哪条?
public static <E> E[] toArray(Collection<E> c) {
    E[] arr = (E[]) new Object[c.size()];  // 这行是经典 hack
    int i = 0;
    for (E e : c) arr[i++] = e;
    return arr;                            // 返回数组
}
答案:A。(E[]) new Object[size] 是"骗"数组的经典 hack——编译期能过,运行期调方强转就 ClassCastException。条款 25 的核心:返回 List<E> 给调用者,让调用方自己决定要不要 toArray(用 Collection.toArray(T[]) 重载),既安全又灵活。
条款 27 · 优先考虑泛型方法
看这个静态工具方法,主要问题在哪?
public static Set union(Set s1, Set s2) {  // ← 原始 Set
    Set result = new HashSet(s1);
    result.addAll(s2);
    return result;
}
答案:B。参数和返回都是原始 Set → 接受任何类型、返回任何类型,调用方必须强转。条款 27 的标准修法:加类型参数 <E> + PECS 通配符 Set<? extends E>。A 错(原始类型编译通过),C/D 错(不是核心问题)。
条款 28 · 列表优先于泛型数组
这段代码编译会怎样?
public static <E> List<E>[] createLists(int n) {
    List<E>[] arr = new List<E>[n];   // ← 创建泛型数组
    return arr;
}
答案:B。Java 直接禁止创建参数化类型的数组:数组具体化(运行期记类型)+ 泛型擦除(运行期忘类型)冲突,无解。改法:返回 List<List<E>>。D 错(List[] 是原始类型数组,绕过泛型后类型信息全丢)。
