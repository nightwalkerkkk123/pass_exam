§0 先把"相等"两种意思分清楚

这部分是后面所有内容的地基。看不懂 == 和 equals 的区别,后面 4 个条款全是糊涂账。

Java 里两种"相等":
  • == 比的是引用(两个变量是不是指向同一块内存)
  • .equals() 比的是逻辑相等(内容/语义是不是一样)
String a = new String("hi");
String b = new String("hi");

System.out.println(a == b);        // false  ← 两次 new,两块内存
System.out.println(a.equals(b));   // true   ← String 重写了 equals,逐字符比较
关键:equals 是 Object 类的方法,默认行为和 == 一模一样(就是比引用)。

如果你写一个类不重写 equals,那 a.equals(b) 和 a == b 是一回事。

这就是为什么会有「条款 10」—— 想让两个对象按内容算相等,你必须自己写 equals。
练习 1 · == vs equals 入门
下面这段代码会输出什么?A、B、C、D 各填 true / false。
class Point {
    int x, y;
    Point(int x, int y) { this.x = x; this.y = y; }
    // 注意:没有重写 equals
}

Point p1 = new Point(1, 2);
Point p2 = new Point(1, 2);
Point p3 = p1;

System.out.println(p1 == p2);        // (A)
System.out.println(p1.equals(p2));   // (B)
System.out.println(p1 == p3);        // (C)
System.out.println(p1.equals(p3));   // (D)
(A) p1 == p2 (B) p1.equals(p2) (C) p1 == p3 (D) p1.equals(p3)
答案:A=false, B=false, C=true, D=true。
  • A:p1、p2 是两次 new,两块内存,引用不等 → false
  • B:Point 没重写 equals,继承自 Object,等价于 ==,所以也是 false
  • C:p3 = p1 是引用赋值,p3 和 p1 指向同一块内存 → true
  • D:同上,equals 默认 == → true
启示:Point 想让 (1,2) 和 (1,2) 算相等,就必须重写 equals。这就引出条款 10。

条款 10覆盖 equals 时请遵守通用约定

重写 equals 不是随便写,必须满足 5 性质。破任意一个,放进集合(HashMap/HashSet/List)就会出诡异行为。

性质含义最容易违反的场景
自反x.equals(x) == true很少违反,但用 ==/instanceof 时要小心
对称x.equals(y) ⇔ y.equals(x)⚠️ 让自己的类跟别的类(如 String)比时
传递x.equals(y) && y.equals(z) ⇒ x.equals(z)⚠️ 子类加新字段还想跟父类比时
一致多次调用结果不变(只要对象状态没变)用网络/可变字段做比较时
非空x.equals(null) == false用 instanceof 自带这性质,几乎不会违反
口诀:自-对-传-一-非。"自对传一非"。或者"对称、传递最容易翻车"。
练习 2 · 条款 10 · 对称性反例
下面这个 equals 违反 5 性质中的哪一个?
public final class CaseInsensitiveString {
    private final String s;
    public CaseInsensitiveString(String s) { this.s = s; }

    @Override public boolean equals(Object o) {
        if (o instanceof CaseInsensitiveString)
            return s.equalsIgnoreCase(((CaseInsensitiveString) o).s);
        if (o instanceof String)                            // ① 跟 String 比
            return s.equalsIgnoreCase((String) o);
        return false;
    }
}

CaseInsensitiveString cis = new CaseInsensitiveString("Polish");
String s = "polish";
cis.equals(s);    // → true
s.equals(cis);    // → ?
答案:B(对称性)。

① 处让 CaseInsensitiveString 跟 String 比可以(返回 true),但 String.equals 不知道 CaseInsensitiveString 存在 → "polish".equals(cis) 返回 false。一边 true 一边 false,对称性破了。

后果:放进 List、contains 时行为飘忽,跟你查的顺序有关。

修法:只跟同一类型比,删掉 ① 那段。这是条款 10 最经典的反例,几乎每年必考。

练习 3 · 条款 10 · 传递性陷阱
ColorPoint extends Point,加了一个 color 字段。下面 4 种 equals 写法,哪一种能完美满足传递性?
class Point {
    int x, y;
    @Override public boolean equals(Object o) {
        if (!(o instanceof Point)) return false;
        Point p = (Point) o;
        return p.x == x && p.y == y;
    }
}

class ColorPoint extends Point {
    Color c;
    // 4 种实现,哪种对?
}
答案:C。

这是条款 10 的核心结论之一:"无法在扩展可实例化类的同时,既增加新的值组件,又保留 equals 约定。"

A 看起来最自然,但破传递性:redPoint.equals(point) = true(忽略颜色), point.equals(bluePoint) = true,但 redPoint.equals(bluePoint) = false。

B 用反射(getClass())代替 instanceof 也行,但破里氏替换 —— 子类对象不能当父类用,违反 OOP 多态原则。

解药:不继承,改用复合(条款 18):class ColorPoint { private Point point; private Color c; ... }。这样 ColorPoint 不是 Point,equals 不冲突。

条款 11覆盖 equals 时总要覆盖 hashCode

这条考试出现率几乎是 100%。LR-0002 标记你"对条款 11 自评认识、对条款 10 自评模糊" — 现在我们用代码题验证这是不是真认识。

hashCode 契约(背下来):
  1. 必备:x.equals(y) == true ⇒ x.hashCode() == y.hashCode()
  2. 反过来不要求:hashCode 相等,equals 可以不等(允许"碰撞")
  3. 一致:对象状态没变,多次调 hashCode 结果不变
反模式(必考):只重写 equals,忘了重写 hashCode。

后果:对象放进 HashMap / HashSet 后就丢了。
(put 进去时按对象 A 的 hashCode 算桶号,get 时按对象 B 的 hashCode 算桶号 — 两个桶不一样,找不到。) 
// 标准实现 1:Java 7+ 一行版
@Override public int hashCode() {
    return Objects.hash(area, prefix, line);   // 简洁,可读
}

// 标准实现 2:经典 31 倍版(性能更好)
@Override public int hashCode() {
    int result = Short.hashCode(area);
    result = 31 * result + Short.hashCode(prefix);
    result = 31 * result + Short.hashCode(line);
    return result;
}
// 31 是奇素数,(31 * x) 可被 JVM 优化成 (x << 5) - x
练习 4 · 条款 11 · 真伪契约
下面哪个说法正确?
答案:C。这是契约本身。
  • A 错:碰撞是允许的(两个不相等的对象可以有相同 hashCode)
  • B 错:同样允许碰撞,同一桶里再用 equals 区分
  • D 错:ArrayList 用 equals不用 hashCode,不会爆;但 HashMap / HashSet 必爆
练习 5 · 条款 11 · 运行时翻车
运行下面代码,contains 打印什么?
class Point {
    final int x, y;
    Point(int x, int y) { this.x = x; this.y = y; }
    @Override public boolean equals(Object o) {
        if (!(o instanceof Point)) return false;
        Point p = (Point) o;
        return p.x == x && p.y == y;
    }
    // 注意:没覆盖 hashCode
}

Set<Point> set = new HashSet<>();
set.add(new Point(1, 2));
System.out.println(set.contains(new Point(1, 2)));
答案:B(false)。

两个 Point(1,2) equals 相等,但 hashCode 用的是 Object 默认实现(基于内存地址) → 两个对象散到不同的桶。

contains 算第二个 Point 的 hashCode,去另一个桶找,找不到 → false。

这就是没覆盖 hashCode 的经典翻车现场:equals 说"是同一个",HashSet 说"我找不到"。条款 11 的实战意义就在这一题。

条款 12始终要覆盖 toString

这条简单,但考试也常以"识别反例"形式出。

不重写 toString 的默认行为: 
PhoneNumber pn = new PhoneNumber(707, 867, 5309);
System.out.println(pn);
// 输出:PhoneNumber@163b91     ← 几乎没用,断点/日志都看不出内容
条款 12 的约定:
  • 简洁、信息丰富,让人(和日志)直接读懂这个对象是谁
  • 包含所有有意思的字段(密码这种敏感字段排除)
  • 给的格式要么文档化,要么明确说"格式可能变",避免别人解析后被你改格式坑死
// 反例:不重写
class PhoneNumber { ... }
new PhoneNumber(707, 867, 5309).toString()
  → "PhoneNumber@163b91"     // 没用

// 正例:重写
@Override public String toString() {
    return String.format("(%03d) %03d-%04d", area, prefix, line);
}
new PhoneNumber(707, 867, 5309).toString()
  → "(707) 867-5309"         // 一眼就懂
练习 6 · 条款 12 · toString 设计原则
下面 4 种 toString 哪一种最符合条款 12 的建议?
答案:D。
  • A 等于不重写,默认行为没用
  • B 太省 — 没"信息丰富",日志里全是 "User",看不出区别
  • C 违反"敏感字段排除"原则 — 密码进日志是安全事故
  • D 是教材标准答案模板:类名 + 主要字段 = 值,跳过敏感字段

条款 14考虑实现 Comparable 接口

把对象放进 TreeMap、TreeSet,或者用 Collections.sort(),都需要对象有"顺序"概念 — 这就是 Comparable 接口。

public interface Comparable<T> {
    int compareTo(T other);
    // 返回:负数 = 我比 other 小;0 = 相等;正数 = 我比 other 大
}
compareTo 契约(和 equals 类似但有差异):
  1. 反对称:sgn(x.compareTo(y)) == -sgn(y.compareTo(x))
  2. 传递:x.compareTo(y) > 0 && y.compareTo(z) > 0 ⇒ x.compareTo(z) > 0
  3. 一致(强烈建议):x.compareTo(y) == 0 应该等价于 x.equals(y) == true
实现陷阱(必考): 
// ❌ 错误:整数减法可能溢出
public int compareTo(Integer i) {
    return this.value - i.value;       // 如果一个很大、一个很小的负数,会溢出
}

// ✅ 正确写法 1:用比较运算符
return Integer.compare(this.value, i.value);

// ✅ 正确写法 2(Java 8+):用 Comparator 静态方法
private static final Comparator<PhoneNumber> COMP =
    Comparator.comparingInt((PhoneNumber pn) -> pn.area)
              .thenComparingInt(pn -> pn.prefix)
              .thenComparingInt(pn -> pn.line);
public int compareTo(PhoneNumber pn) { return COMP.compare(this, pn); }

第三条契约的现实意义:如果 compareTo == 0 但 equals == false,放进 TreeSet 时,集合会按 compareTo 算"相等" — 跟 HashSet 用 equals 行为不一致,Bug 难找。

练习 7 · 条款 14 · compareTo 实现陷阱
下面 compareTo 的写法,哪个有 Bug?
class Money implements Comparable<Money> {
    int cents;   // 可以是任意 int 值,包括 Integer.MIN_VALUE / MAX_VALUE
    // 4 种 compareTo 实现:
}
答案:A 有 Bug。

反例:this.cents = Integer.MAX_VALUE,other.cents = -1。期望返回正数(我大),但 MAX_VALUE - (-1) = MAX_VALUE + 1 = 溢出成 Integer.MIN_VALUE(负数!)→ 排序结果完全错。

B、C、D 都是安全的写法。记住:compareTo 永远不要用减法,用 Integer.compare(或对应类型的 compare 方法)。

练习 8 · 综合 · 哪些是 Bug?(多选)
下面这个类有几个违反条款 10/11/12/14 的问题?选出所有正确的。
class User implements Comparable<User> {
    String name;
    int age;
    String password;

    @Override public boolean equals(Object o) {
        if (!(o instanceof User)) return false;
        User u = (User) o;
        return name.equalsIgnoreCase(u.name) && age == u.age;
    }

    @Override public String toString() {
        return "User[name=" + name + ", age=" + age + ", pwd=" + password + "]";
    }

    @Override public int compareTo(User u) {
        return this.age - u.age;
    }

    // 没有 hashCode
}
答案:B、D、E 都对(本题最少要选这三项,选了 A 或 C 算错)。
  • B:✅ 重写了 equals 没重写 hashCode,经典翻车 — 条款 11
  • D:✅ 把 password 打印到 toString,安全事故 — 条款 12
  • E:✅ compareTo == 0 时(age 相等)equals 不一定为 true(name 可能不同),违反条款 14 第 3 条
  • A:❌ equalsIgnoreCase 在两个 User 之间是对称的(都做忽略大小写比较),不破对称性。易混淆:条款 10 那个对称性反例是 String 跟 CaseInsensitiveString 比 — 异类比才出问题
  • C:❌ age 是非负 int 时,a - b 不会溢出,但题目没说 age 一定非负,理论上还是可能;但本题 age 一般非负,不算 Bug(出题角度)
