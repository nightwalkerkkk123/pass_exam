Claude Code Hooks 全解 钩子 = 在 Agent 生命周期的 30 个固定切点，把控制权交给你的脚本。一套协议，30 个触发时机。本页用例子带你逐个吃透。

先建立心智模型 The Mental Model

不要把这 30 个 hook 当 30 个独立知识点去背。它们是同一套"控制反转"机制在不同时机的复现：Claude Code 跑到某个切点时暂停自己，运行你配置的外部脚本，然后根据脚本的退出码 + stdout 决定下一步。理解了这个回路，30 个 hook 只是"在哪一刻暂停"的区别。

① 切点触发

到达生命周期某一刻（如"工具执行前"），Claude Code 暂停，把一段 JSON 写到你脚本的 stdin。

② 你的脚本回话

脚本读 JSON，做判断，用退出码和stdout JSON 表态：放行 / 阻断 / 注入上下文。

③ Claude Code 据此行动

放行就继续；阻断就停下并把你的理由喂回模型；注入就把上下文塞进对话。

★ 关键洞察 ─────────────

  • 分水岭是"能不能拦"：PreToolUse/UserPromptSubmit/Stop 能真正改写或阻断控制流；而 SessionEnd/StopFailure 的返回值被直接忽略，它们只是"事后通知"。按这个维度分类，比按字母背诵有用得多。
  • "Pre" 能拦，"Post" 不能：动作发生之前的 hook（Pre*）才有否决权；动作发生之后的 hook（Post*）木已成舟，只能观测或追加上下文，无法撤销。
  • hook 不是 Claude 在跑：它是外部进程（shell 命令 / HTTP / MCP）。所以"自动化行为"必须用 hook，而非靠提示词——脚本由 harness 执行，确定性地发生，不受模型即兴发挥影响。

通用协议 所有 hook 共享的输入 / 输出契约

先把这套协议吃透，后面 30 张卡片就只剩"专属字段"的差异了。

① 输入：stdin 收到的 JSON

每次 hook 触发，Claude Code 都会把一段 JSON 写进你脚本的标准输入。所有事件都带的公共字段：

{
  "session_id": "2b4e26a2-...",        // 当前会话 ID
  "transcript_path": "/path/to.jsonl", // 完整对话记录文件
  "cwd": "/Users/you/project",        // 当前工作目录
  "hook_event_name": "PreToolUse",     // 哪个事件触发了你
  "permission_mode": "default"         // default|plan|acceptEdits|bypassPermissions...
}

每个事件再追加自己的专属字段——例如 PreToolUse 多出 tool_name 和 tool_input，UserPromptSubmit 多出 prompt。这些在下面每张卡片里列出。

② 输出（廉价版）：退出码

最简单的 hook 什么都不打印，只用退出码表态。这是三态信号：

退出码含义行为
0成功放行。如果该事件支持，解析 stdout 里的 JSON。stdout 内容（非 JSON 时）可作为上下文。
2阻断否决这个动作。stderr 的内容会按事件不同喂回给 Claude（让它换个做法）或显示给用户。
其他非阻断错误不阻断，但把 stderr 显示出来（脚本自身报错）。流程照常继续。

③ 输出（精细版）：stdout 的 JSON

需要更精细控制时，exit 0 + 打印一段 JSON 到 stdout。常用字段（不同事件支持子集，单字段上限约 1 万字符）：

{
  "continue": false,          // 是否让 Claude 继续；false = 整轮停止
  "stopReason": "...",       // continue=false 时给用户看的原因
  "suppressOutput": true,    // 不把本 hook 的 stdout 写进 transcript
  "systemMessage": "...",    // 给用户的一条提示信息
  "decision": "block",        // 部分事件用：block 阻断 / approve 放行
  "reason": "...",           // decision 的理由（喂给 Claude）
  "hookSpecificOutput": {        // 事件专属返回的容器
    "hookEventName": "PreToolUse",
    "permissionDecision": "deny",        // PreToolUse 专属：allow|deny|ask
    "permissionDecisionReason": "...",
    "additionalContext": "..."         // 注入到对话里的额外上下文
  }
}
"廉价版" exit 2 与 "精细版" JSON 往往二选一即可达到阻断效果。比如阻止一个工具：要么 exit 2 + stderr 写理由，要么 exit 0 + permissionDecision:"deny"。JSON 版能携带更结构化的信息（如修改后的输入 updatedInput）。

配置结构 hook 写在 settings.json 里

hook 不写代码——你在 settings.json 的 hooks 块里声明"哪个事件 → 匹配什么 → 跑什么命令"。

{
  "hooks": {
    "PreToolUse": [                         // ← 事件名作为 key
      {
        "matcher": "Bash",                 // ← 只对 Bash 工具触发（支持正则/竖线）
        "hooks": [
          {
            "type": "command",           // command | http | mcp_tool | prompt | agent
            "command": "bash ~/.claude/guard.sh",
            "timeout": 10                 // 秒；超时按非阻断错误处理
          }
        ]
      }
    ]
  }
}

★ 关于 matcher ─────────────

  • 不是所有事件都有 matcher（约一半有）。有 matcher 的事件，匹配维度各不相同：PreToolUse 匹配工具名，FileChanged 匹配文件名，SubagentStart 匹配子代理类型，SessionStart 匹配来源(startup/resume/clear/compact)。
  • matcher 的字符串规则：字母、数字、下划线、空格、|、, 当字面量；其余字符按 JavaScript 正则解释。所以 Edit|Write 匹配两个工具，^mcp__memory__.* 匹配某 MCP server 的所有工具。
  • 配置有优先级链：用户全局 ~/.claude/settings.json → 项目 .claude/settings.json → 项目本地 .local.json → 组织策略 → 插件 → Skill/Agent 元信息。同一事件多处配置会叠加触发。

实战配方 4 个最常用的真实用法

① PreToolUse — 拦住危险命令

最经典用途：在 Bash 执行前扫一眼，发现 rm -rf 之类就否决。

# ~/.claude/settings.json
{ "hooks": { "PreToolUse": [{ "matcher": "Bash",
    "hooks": [{ "type": "command", "command": "~/.claude/guard.sh" }] }] } }

# guard.sh — 读 stdin 的 tool_input.command，命中就 deny
#!/bin/bash
cmd=$(jq -r '.tool_input.command')
if echo "$cmd" | grep -qE 'rm -rf|:>|mkfs'; then
  echo '{"hookSpecificOutput":{"hookEventName":"PreToolUse","permissionDecision":"deny","permissionDecisionReason":"检测到破坏性命令，已拦截"}}'
  exit 0   # 或者干脆 exit 2，把理由写到 stderr
fi

② UserPromptSubmit — 每次提问自动注入项目上下文

在模型看到你的话之前，悄悄附上"当前分支 / 改了几个文件"等动态信息。

#!/bin/bash
branch=$(git branch --show-current 2>/dev/null)
dirty=$(git status --porcelain 2>/dev/null | wc -l | tr -d ' ')
echo "{\"hookSpecificOutput\":{\"hookEventName\":\"UserPromptSubmit\",\"additionalContext\":\"当前分支 $branch，有 $dirty 个改动文件\"}}"
# 配在 "UserPromptSubmit"（无 matcher）。也可直接 echo 纯文本 + exit 0，文本即上下文。

③ PostToolUse — 文件写完自动跑格式化 / lint

动作已发生（拦不住），但可以做善后：每次 Edit/Write 后自动 prettier，并把结果作为上下文反馈。

{ "hooks": { "PostToolUse": [{ "matcher": "Edit|Write",
    "hooks": [{ "type": "command",
      "command": "jq -r '.tool_input.file_path' | xargs -r prettier --write" }] }] } }

④ Stop — 不让 Claude 偷懒收工

模型想结束本轮时，检查测试是否还红；红就 decision:"block" 把它拽回来继续修。

#!/bin/bash
if ! npm test --silent >/dev/null 2>&1; then
  echo '{"decision":"block","reason":"测试仍未通过，请继续修复后再结束。"}'
fi
# 注意防死循环：stdin 里有 stop_hook_active=true 时应放行，否则会无限重入。
配 Stop 这类"逼模型继续"的 hook 时，务必检查输入里的 stop_hook_active 标志位——它为 true 说明本轮已是被 hook 拉回来的，再 block 就会无限循环。

口径与边界：本页"触发时机"以 Claude Code 官方事件描述为准；字段与行为细节由权威文档整理。部分较新/高级事件（如 PostToolBatch、TeammateIdle、ConfigChange、Elicitation*）官方文档信息较少，凡标 "文档未明确" 或 ? 处，请以你本机 Claude Code 版本（本会话为 2.1.191）实际行为为准验证，不要当成铁律。
