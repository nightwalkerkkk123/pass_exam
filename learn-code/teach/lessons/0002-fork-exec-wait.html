Lesson 0002 · 第 1 章 操作系统接口

fork + exec + wait
—— shell 如何造出并启动一个进程

🎯 对应使命:仍在第 1 章。上一课你看到 echo 用 write 求内核办事; 这一课回答上一课的预告——echo 这个进程是谁、怎么造出来的。答案藏在 user/sh.c 里。
🔁 先回忆(别往回翻 0001):进程想"碰硬件"唯一的办法是什么?…… 对,发起系统调用。今天的 fork / exec / wait 同样是系统调用—— 只不过它们操作的不是屏幕,而是进程本身。
这节课你将能做到一件事: 对着 shell 的主循环,讲清楚你敲下 echo hi 回车后, fork 把谁劈成了两个、哪条分支是孩子、exec 把孩子变成了什么、 wait 又让父亲做了什么。

1 · 三个系统调用,各管一件事

系统调用一句话关键怪点
fork()把当前进程整个复制一份,得到一个子进程调用一次,返回两次:父进程里返回子进程的 pid,子进程里返回 0
exec()把当前进程的程序整个替换成磁盘上的另一个程序成功就永不返回(原来的代码没了);只有失败才会返回
wait()父进程阻塞,直到它的某个子进程结束回收孩子,拿到它的退出状态

为什么造新进程要两步(先 fork 复制、再 exec 替换),而不是一步搞定? 因为这两步中间留出了一个宝贵的窗口:子进程可以先重新安排好自己的文件描述符(比如把输出重定向到文件), 然后再 exec 变身。这个"先复制、布置现场、再变身"的设计,是 Unix 最优雅的主意之一。

📖 依据:xv6 book (rev3) 第 1 章 1.1–1.2 节 — book-riscv-rev3.pdf。

2 · 看真东西:shell 的主循环

这是 ../xv6-riscv-20230207/user/sh.c 的 main,我去掉了 cd 特判,留下骨架:

160while(getcmd(buf, sizeof(buf)) >= 0){
161  // ...(cd 特判略)...
168  if(fork1() == 0)      // ← 这里劈成两条命运
169    runcmd(parsecmd(buf)); // 只有【子进程】走进来(fork 返回 0)
170  wait(0);               // 只有【父进程】走到这(继续往下)
171}

同一份代码,fork1() 之后却分裂成两个进程在跑。区别全在那个返回值:

敲下 "echo hi"，到达第 168 行 │ fork1() 复制 ┌─────────┴─────────┐ 子进程：返回 0 父进程：返回子进程 pid（≠0） if(...==0) 为真 if(...==0) 为假 进入第 169 行 跳过 169，直接到第 170 行 │ │ runcmd → exec wait(0) 阻塞等孩子 变身为 echo，打印 hi │ │ ←──── 孩子退出，wait 返回 exit 回到 160 行，等下一条命令

注意 fork1 只是对 fork 的一层薄封装(sh.c 第 182 行):调一次 fork(), 失败就 panic,否则原样返回 pid。所以理解 fork1 就等于理解 fork。

3 · 孩子如何"变身":runcmd 里的 exec

第 169 行的 runcmd 最终会走到 EXEC 分支(sh.c 第 75 行):

75case EXEC:
76  ecmd = (struct execcmd*)cmd;
77  if(ecmd->argv[0] == 0)
78    exit(1);
79  exec(ecmd->argv[0], ecmd->argv); // 成功 → 本进程变成 echo，下面两行永远到不了
80  fprintf(2, "exec %s failed\n", ecmd->argv[0]);
81  break;

这是理解 exec 语义的铁证:第 80 行那句 "exec failed" 平时永远打印不出来—— 因为 exec 成功后,这个进程的代码已经被 echo 整个替换掉了,根本不会再执行到第 80 行。 只有当 exec 失败(比如找不到这个程序)时,它才返回,第 80 行才有机会运行。 "成功不返回,返回即失败"——这就是 exec。

🔎 本地对照:../xv6-riscv-20230207/user/sh.c 第 59–81 行(runcmd)、第 146–173 行(main)、第 182–191 行(fork1)。
