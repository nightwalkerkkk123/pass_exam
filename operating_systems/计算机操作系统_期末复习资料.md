## 第1章 操作系统引论

### 1.1 操作系统的目标和作用

#### 1.1.1 操作系统的目标

操作系统（Operating System, OS）是一组有效控制和管理计算机系统的**硬件和软件资源**、合理地组织计算机工作流程以及**方便用户**的程序集合。

> **复习要点**：OS的三层含义——有效（efficiency）、合理（fair）、方便（convenience）

操作系统的主要目标：

| 目标 | 说明 |
|------|------|
| **方便性** | 提供用户接口，使计算机更易使用 |
| **有效性** | 提高系统资源利用率和吞吐量 |
| **可扩充性** | 支持模块化、层次化结构，便于功能扩充 |
| **开放性** | 遵循标准规范，实现软硬件的互操作性 |

#### 1.1.2 操作系统的作用

1. **从计算机系统组成观点**——OS是**系统软件**
   - 计算机系统 = 硬件（CPU、内存、I/O设备）+ 软件（系统软件 + 应用软件）

2. **从资源管理观点**——OS是**计算机系统资源的管理者**

| 系统资源 | 对应OS管理模块 |
|----------|---------------|
| 处理器 | 处理机（进程）管理 |
| 存储器 | 存储器管理 |
| I/O设备 | 设备管理 |
| 文件（数据和程序） | 文件系统 |

3. **从软件分层、扩充机器的观点**——OS是**扩充裸机的第一层系统软件**
   - 裸机 + 操作系统 → **虚拟机**（功能更强、使用更方便）

4. **从服务用户的观点**——OS是**用户与计算机硬件系统之间的接口**
   - **命令接口**：联机命令（键盘输入）、脱机命令（批处理JCL）
   - **程序接口**：系统调用（用户程序获取OS服务的唯一途径）
   - **图形接口**：GUI（图标、窗口、菜单）

#### 1.1.3 推动OS发展的主要动力

1. 不断提高计算机资源利用率
2. 方便用户
3. 器件的不断更新换代
4. 计算机体系结构的不断发展

---

### 1.2 操作系统的发展过程

> **复习要点**：各阶段的特征对比是考试重点！

#### 1.2.1 无操作系统的计算机系统

**1. 人工操作方式**（1945年 ~ 50年代中期）
- 用户（程序员）直接使用计算机硬件
- **缺点**：① 用户独占全机 ② CPU等待人工操作

**2. 脱机输入/输出（Off-Line I/O）方式**
- 利用磁带进行脱机输入输出
- **优点**：减少CPU空闲时间、提高I/O速度

#### 1.2.2 单道批处理系统

- **批处理技术**：先将一批作业输入磁带，由监督程序自动逐个处理
- **特征**：
  - **自动性**：作业自动逐个处理
  - **顺序性**：按顺序执行
  - **单道性**：内存中仅有一道作业

#### 1.2.3 多道批处理系统

**1. 多道程序设计的基本概念**

用户提交的作业存放在外存形成**后备队列**，由作业调度程序按算法选择若干作业调入内存，共享CPU和各种资源。

**2. 多道程序设计的好处**

- **提高CPU利用率**：当某程序因I/O暂停时，CPU可调度另一程序执行
- **提高内存和I/O设备利用率**
- **增加系统吞吐量**

**3. 多道批处理系统的特征**

- **多道性**：内存中有多道程序
- **无序性**：作业完成顺序与进入顺序无关
- **调度性**：包含作业调度和进程调度

**4. 优缺点**

| 优点 | 缺点 |
|------|------|
| 资源利用率高 | 平均周转时间长 |
| 系统吞吐量大 | 无交互能力 |

**5. 需解决的问题**：处理机管理、内存管理、I/O设备管理、文件管理、作业管理

#### 1.2.4 分时系统

**1. 产生动力**：用户需求（人机交互、共享主机、便于上机）

**2. 关键问题**：
- **及时接收**：系统能同时接收多个用户命令
- **及时处理**：用户命令能得到及时响应

**3. 分时系统的特征**

| 特征 | 说明 |
|------|------|
| **多路性** | 一台主机连接多台终端，宏观上多用户同时工作 |
| **独立性** | 每个用户独占一个终端，互不干扰 |
| **交互性** | 用户可通过终端与系统广泛对话 |
| **及时性** | 用户请求在很短时间内（通常1~2秒）获得响应 |

**4. 响应时间公式**：

```
T = n × q
q = S + T（处理）
```

其中：`T`为响应时间，`n`为用户数，`q`为时间片，`S`为对换时间。

#### 1.2.5 实时系统

**1. 定义**：系统能**及时**响应外部事件的请求，在**规定时间内**完成事件处理，并控制所有实时任务协调一致运行。

**2. 实时任务的分类**

| 分类依据 | 类型 | 说明 |
|----------|------|------|
| 按周期性 | 周期性实时任务 | 按固定周期执行 |
| | 非周期性实时任务 | 有截止时间（开始/完成截止时间） |
| 对截止时间要求 | **硬实时任务** | 必须满足截止时间，否则后果严重 |
| | **软实时任务** | 偶尔错过截止时间影响不太大 |

#### 各类型OS对比

| 特性 | 批处理系统 | 分时系统 | 实时系统 |
|------|-----------|----------|----------|
| 主要目标 | 资源利用率、吞吐量 | 交互性、响应时间 | 及时性、可靠性 |
| 交互能力 | 无 | 强 | 较弱 |
| 响应时间 | 长（小时级） | 短（秒级） | 极短（毫秒/微秒级） |
| 多路性 | 无 | 有 | 有 |
| 可靠性 | 一般 | 一般 | 很高 |

### 1.3 操作系统的基本特性

> **复习要点**：四大特征及其关系是必考点！

#### 1.3.1 并发（Concurrence）

- **并行性**：两个或多个事件在**同一时刻**发生（需多处理机）
- **并发性**：两个或多个事件在**同一时间间隔内**发生（单处理机通过分时交替实现）
- **并发是OS最基本的特征之一**

#### 1.3.2 共享（Sharing）

系统中资源可供内存中多个并发执行的进程共同使用。

**1. 互斥共享方式**
- **临界资源/独占资源**：一段时间内只允许一个进程访问（如打印机、磁带机）
- 进程A访问 → A释放 → 进程B才能访问

**2. 同时访问方式**
- 宏观上"同时"，微观上交替访问（如磁盘设备）

> **复习要点**：并发和共享是OS的**两个最基本特征**，它们**互为存在条件**——资源共享以并发为条件，资源共享管理影响并发程度。

#### 1.3.3 虚拟（Virtual）

通过某种技术把一个**物理实体**变为若干个**逻辑上的对应物**。

| 虚拟技术 | 说明 |
|----------|------|
| **虚拟处理机** | 通过多道程序设计，将一台CPU虚拟为多台逻辑CPU |
| **虚拟存储器** | 逻辑上扩充内存容量 |
| **虚拟设备** | 将一台物理I/O设备虚拟为多台逻辑设备（如SPOOLing） |

- 若物理设备对应n个虚拟逻辑设备，则虚拟设备平均速度为物理设备速度的 **1/n**

#### 1.3.4 异步性（Asynchronism）

- 进程以"**停停走走**"的方式运行（因资源限制不能一气呵成）
- 进程推进速度**不可预知**
- 只要运行环境相同，多次运行结果**相同**（结果可再现性）

### 1.4 操作系统的主要功能

#### 1.4.1 处理机管理功能

| 功能 | 说明 |
|------|------|
| **进程控制** | 创建、撤销进程，控制进程状态转换 |
| **进程同步** | 互斥方式（临界资源访问）+ 同步方式（执行次序协调） |
| **进程通信** | 合作进程间的信息交换 |
| **调度** | 作业调度（从后备队列选作业）+ 进程调度（从就绪队列选进程） |

#### 1.4.2 存储器管理功能

| 功能 | 说明 |
|------|------|
| **内存分配** | 静态分配（装入时确定）/ 动态分配（运行中可申请） |
| **内存保护** | 每道用户程序只在自己的内存空间运行，互不干扰（界限寄存器） |
| **地址映射** | 将**逻辑地址**转换为**物理地址** |
| **内存扩充** | 虚拟存储技术：请求调入 + 置换功能 |

> **复习要点**：逻辑地址（地址空间）vs 物理地址（内存空间）的区别！

#### 1.4.3 设备管理功能

| 功能 | 说明 |
|------|------|
| **缓冲管理** | 缓和CPU与I/O速度不匹配矛盾（单缓冲、双缓冲、公用缓冲池） |
| **设备分配** | 根据I/O请求和分配策略分配设备 |
| **设备处理** | 设备驱动程序，实现CPU与设备控制器通信 |

#### 1.4.4 文件管理功能

| 功能 | 说明 |
|------|------|
| **文件存储空间管理** | 为文件分配外存空间（以盘块为单位，512B~8KB） |
| **目录管理** | 按名存取、文件共享、快速目录查询 |
| **文件读写管理和保护** | 读/写操作 + 防止非法访问 |

#### 1.4.5 用户接口

| 接口类型 | 说明 |
|----------|------|
| **命令接口** | 联机命令（键盘输入）+ 脱机命令（JCL作业控制语言） |
| **程序接口** | 系统调用（用户程序取得OS服务的唯一途径） |
| **图形接口** | GUI（图标、窗口、对话框） |

### 1.5 操作系统的结构设计

#### 1.5.1 传统OS结构

| 结构类型 | 特点 | 优缺点 |
|----------|------|--------|
| **无结构OS** | 整体系统结构，过程相互调用 | 庞大杂乱，难维护 |
| **模块化OS** | 按功能划分模块和子模块 | 设计清晰，但接口规定不精确 |
| **层次结构** | 模块单向调用，自底向上 | 结构清晰，便于调试验证 |

#### 1.5.2 微内核技术

- **微内核**：精心设计的、实现现代OS核心功能的小型内核
- **特点**：更小更精炼、运行在核心态、开机后常驻内存
- **基本功能**：进程管理、存储器管理、进程通信管理、I/O设备管理
- **优点**：良好的扩充性、可靠性好、便于网络服务和分布式处理
- **缺点**：进程间通信开销增加，效率降低

> **复习要点**：OS结构演进——无结构 → 模块化 → 层次化 → 微内核

## 第2章 进程管理

### 2.1 进程的基本概念

#### 2.1.1 程序的顺序执行及其特征

**程序顺序执行**：仅当前一操作执行完后，才能执行后继操作。

**特征**：
- **顺序性**：严格按照规定顺序执行
- **封闭性**：执行结果不受外界因素影响
- **可再现性**：环境和初始条件相同，多次执行结果相同

#### 2.1.2 前趋图

- **前趋图（Precedence Graph）**：有向无循环图（DAG），描述进程之间执行的前后关系
- 记号：`Pi → Pj` 表示Pi必须在Pj开始前完成（Pi是Pj的**直接前趋**，Pj是Pi的**直接后继**）
- **初始结点**：没有前趋的结点
- **终止结点**：没有后继的结点
- **注意**：前趋图中**不能存在循环**

#### 2.1.3 程序的并发执行及其特征

**并发执行特征**：
- **间断性**："执行-暂停-执行"的间断性活动规律（因共享资源相互制约）
- **失去封闭性**：多个程序共享资源，资源状态由多个程序改变
- **不可再现性**：多次运行结果可能不同

> **复习要点**：由于程序并发执行时结果不可再现，所以引入**进程**来描述并发执行。

#### 2.1.4 进程的定义与特征

**进程（Process）定义**：

> 可并发执行的程序在一个数据集合上的运行过程，是系统进行**资源分配和调度的独立单位**。

其他定义：
- 进程是程序的一次执行
- 进程是程序及其数据在处理机上顺序执行时所发生的活动

**进程的五大特征**：

| 特征 | 说明 |
|------|------|
| **动态性** | 最基本的特征：创建→调度执行→暂停→撤销，有生命周期 |
| **并发性** | 多个进程同存于内存中，一段时间内同时运行 |
| **独立性** | 能独立运行的基本单位，独立获得资源和调度的单位 |
| **异步性** | 各自以不可预知的速度向前推进 |
| **结构特征** | 进程实体 = 程序段 + 数据段 + **PCB（进程控制块）** |

#### 2.1.5 进程的三种基本状态

```
        进程调度              I/O请求/等待事件
   ┌───────────→    ┌───────────→
   │              │              │
 就绪态 ───────→ 运行态 ───────→ 阻塞态
   ↑              │              │
   │              │              │
   └──────────────┘←─────────────┘
      时间片到/I/O完成/事件到达
```

| 状态 | 说明 |
|------|------|
| **运行态（Running）** | 进程正在处理机上运行 |
| **就绪态（Ready）** | 已获得除处理机外的所有资源，等待调度 |
| **阻塞态（Blocked）** | 等待某事件发生而暂停执行（也称等待态/挂起态） |

**状态转换**：

| 转换 | 条件 |
|------|------|
| 就绪态 → 运行态 | 处理机空闲，进程调度程序分配处理机 |
| 运行态 → 阻塞态 | 进程需要等待某事件发生（如I/O请求） |
| 阻塞态 → 就绪态 | 进程等待的事件已发生（如I/O完成） |
| 运行态 → 就绪态 | 时间片用完，或被更高优先级进程抢占 |

> **注意**：阻塞态 → 运行态 和 就绪态 → 阻塞态 **不可能发生**！

#### 2.1.6 挂起状态

**引入原因**：系统管理员暂停进程排除故障/减轻负荷；用户检查中间结果。

**五种状态模型**：

| 状态 | 说明 |
|------|------|
| 活动就绪 | 未被挂起的就绪状态 |
| 静止就绪 | 被挂起后的就绪状态（在外存） |
| 活动阻塞 | 未被挂起的阻塞状态 |
| 静止阻塞 | 被挂起后的阻塞状态（在外存） |
| 运行 | 正在执行 |

**挂起/激活操作**：
- 活动就绪 → 静止就绪（挂起）
- 活动阻塞 → 静止阻塞（挂起）
- 静止就绪 → 活动就绪（激活）
- 静止阻塞 → 活动阻塞（激活）

#### 2.1.7 进程控制块（PCB）

> **复习要点**：PCB是进程存在的唯一标志！

**PCB的作用**：使程序成为能独立运行的基本单位，OS根据PCB对并发进程进行控制和管理。

**PCB中的信息**：

| 类别 | 包含内容 |
|------|----------|
| **进程标识符** | 内部标识符（序号）+ 外部标识符（名字） |
| **处理机状态** | 通用寄存器、指令计数器、程序状态字PSW、用户栈指针 |
| **进程调度信息** | 进程状态、优先级、等待CPU时间、已执行时间、阻塞原因 |
| **进程控制信息** | 程序和数据地址、同步和通信机制、资源清单、链接指针 |

**PCB的组织方式**：
- **链接方式**：按状态将PCB链接成队列（就绪队列、阻塞队列等）
- **索引方式**：按状态建立索引表

### 2.2 进程控制

> **复习要点**：进程控制通过**原语**实现，原语操作具有**原子性**（不可分割）！

#### 2.2.1 进程的创建

**引起创建的事件**：用户登录、作业调度、提供服务、应用请求

**创建过程**：
1. 申请空白PCB
2. 为新进程分配资源
3. 初始化进程控制块
4. 将新进程插入就绪队列

#### 2.2.2 进程的终止

**引起终止的事件**：
- **正常结束**：执行到Halt指令或Logs off
- **异常结束**：越界、保护错、非法指令、特权指令错、运行超时、等待超时、算术运算错、I/O故障
- **外界干预**：操作员干预、父进程请求、父进程终止

**终止过程**：
1. 从PCB集合中检索该进程的PCB，读出状态
2. 若处于执行状态，立即终止执行，置调度标志
3. 终止所有子孙进程
4. 归还所有资源
5. 从队列中移出PCB

#### 2.2.3 进程的阻塞与唤醒

| 操作 | 说明 |
|------|------|
| **阻塞（block）** | 进程主动行为：停止执行→状态改"阻塞"→插入阻塞队列→转调度程序 |
| **唤醒（wakeup）** | 事件发生时被调用：从阻塞队列移出→状态改"就绪"→插入就绪队列 |

**引起阻塞/唤醒的事件**：请求系统服务、启动某种操作、新数据尚未到达、无新工作可做

#### 2.2.4 进程的挂起与激活

| 操作 | 说明 |
|------|------|
| **挂起（suspend）** | 活动就绪→静止就绪；活动阻塞→静止阻塞；可能需重新调度 |
| **激活（active）** | 静止就绪→活动就绪；静止阻塞→活动阻塞；可能触发抢占调度 |

#### Linux进程控制原语

| 系统调用 | 功能 |
|----------|------|
| `fork()` | 创建子进程，父进程返回子进程PID，子进程返回0 |
| `exec()` | 用目标文件正文段覆盖当前进程正文段 |
| `exit()` | 终止进程，返回状态给父进程 |
| `wait()` | 挂起调用进程，等待子进程暂停或终止 |

```c
#include <stdio.h>
void main() {
    int pid;
    pid = fork();  /* 创建子进程 */
    if (pid < 0) { 
        fprintf(stderr, "Fork Failed"); 
        exit(-1); 
    } else if (pid == 0) { 
        execlp("/bin/ls", "ls", NULL);  /* 子进程执行ls命令 */
    } else { 
        wait(NULL);  /* 父进程等待子进程 */
        printf("child Complete");
        exit(0);
    }
}
```

### 2.3 进程同步

#### 2.3.1 进程同步的概念

**1. 进程间制约关系**

| 关系类型 | 说明 |
|----------|------|
| **资源共享关系（互斥）** | 多进程共享临界资源，每次只允许一个进程使用 |
| **相互合作关系（同步）** | 进程间有先后执行次序的协调关系 |

**进程竞争资源的三个控制问题**：
- **互斥（mutual exclusion）**：多个进程不能同时使用同一资源
- **死锁（deadlock）**：多个进程互不相让，都得不到足够资源
- **饥饿（starvation）**：一个进程一直得不到资源

**2. 临界资源与临界区**

- **临界资源**：一次只允许一个进程使用的资源（如打印机、变量、缓冲区等）
- **临界区（critical section）**：访问临界资源的那段代码
- **互斥**：不允许两个以上的并发进程同时进入临界区

```
进程结构：
begin
    remainder section 1;   // 剩余区1
    进入区                  // 检查临界资源状态
    critical section;      // 临界区
    退出区                  // 释放临界资源
    remainder section 2;   // 剩余区2
end
```

**3. 进程同步机制应遵循的准则**

| 准则 | 含义 |
|------|------|
| **空闲让进** | 临界资源空闲时，允许请求进程立即进入 |
| **忙则等待** | 已有进程进入临界区时，其他进程必须等待 |
| **有限等待** | 保证进程在有限时间内进入临界区，避免饥饿 |
| **让权等待** | 不能进入临界区时，立即释放处理机，避免忙等 |

#### 2.3.2 信号量机制

**1. 整型信号量**

由Dijkstra定义，仅能通过两个原子操作P、V访问：

```
wait(S):   while S <= 0 do no-op;
           S := S - 1;

signal(S): S := S + 1;
```

> **缺点**：未遵循"让权等待"，进程处于"忙等"状态。

**2. 记录型信号量**

```
type semaphore = record
    value: integer;
    L: list of process;
end

procedure wait(S)
    S.value := S.value - 1;
    if S.value < 0 then block(S, L)  // 阻塞该进程
end

procedure signal(S)
    S.value := S.value + 1;
    if S.value <= 0 then wakeup(S, L) // 唤醒等待队列中的一个进程
end
```

**信号量取值的意义**：

| S.value | 含义 |
|---------|------|
| > 0 | 可供使用的资源数 |
| = 0 | 资源已被占用，无等待进程 |
| < 0（= -n） | 资源已被占用，还有n个进程在等待 |

**3. 信号量的分类**

| 类型 | 说明 |
|------|------|
| **公用信号量（互斥信号量）** | 用于互斥共享临界资源，初值=1，每个进程都可施加P/V操作 |
| **专用信号量（同步信号量）** | 用于同步协作，初值由资源数决定，拥有者施P，合作者施V |

**4. AND型信号量**

基本思想：将进程所需的所有资源**一次性全部分配**，使用完再一起释放。

```
Swait(S1, S2, ..., Sn):
    if S1 >= 1 and ... and Sn >= 1 then
        for i := 1 to n do Si := Si - 1;
    else
        将进程放入第一个Si < 1的等待队列
    endif

Ssignal(S1, S2, ..., Sn):
    for i := 1 to n do
        Si := Si + 1;
        唤醒Si队列中所有等待进程
end
```

**5. 信号量集**

```
Swait(S1, t1, d1, ..., Sn, tn, dn)
    if S1 >= t1 and ... and Sn >= tn then
        for i := 1 to n do Si := Si - di
```

特殊情况：
- `Swait(S, d, d)`：每次申请d个资源
- `Swait(S, 1, 1)`：蜕化为记录型信号量
- `Swait(S, 1, 0)`：可控开关（S≥1时允许多个进入，S=0时阻止进入）

#### 2.3.3 信号量的应用

**1. 利用信号量实现进程互斥**

```
Var mutex: semaphore := 1;
begin
    parbegin
        process 1: begin
            repeat
                wait(mutex);
                critical section;
                signal(mutex);
                remainder section
            until false;
        end
        process 2: begin
            repeat
                wait(mutex);
                critical section;
                signal(mutex);
                remainder section
            until false;
        end
    parend
end
```

> **关键**：mutex初值为1，P/V操作成对出现，紧靠着临界区。

**2. 利用信号量实现前趋关系**

对每条有向边设置一个信号量（初值为0），前趋进程执行完后V操作，后继进程执行前P操作。

### 2.4 经典进程同步问题

> **复习要点**：四种经典同步问题是考试重点，需掌握信号量设置和PV操作位置！

#### 2.4.1 生产者-消费者问题

**问题描述**：一组生产者向一组消费者提供消息，共享一个有界缓冲池（n个缓冲区）。

**信号量设置**：
- `mutex`：互斥信号量，初值=1（互斥访问缓冲池）
- `empty`：同步信号量，初值=n（空缓冲区数）
- `full`：同步信号量，初值=0（满缓冲区数）

**类PASCAL实现**：

```pascal
Var mutex, empty, full: semaphore := 1, n, 0;
    Buffer: array[0..n-1] of message;
    in, out: 0..n-1 := 0, 0;
begin
    parbegin
        Producer i: begin
            repeat
                Produce a new message m;
                P(empty);       // 申请空缓冲区
                P(mutex);       // 申请进入临界区
                Buffer[in] := m;
                in := (in + 1) mod n;
                V(mutex);       // 释放临界区
                V(full);        // 释放满缓冲区信号
            until false;
        end
        
        Consumer j: begin
            repeat
                P(full);        // 申请满缓冲区
                P(mutex);       // 申请进入临界区
                m := Buffer[out];
                out := (out + 1) mod n;
                V(mutex);       // 释放临界区
                V(empty);       // 释放空缓冲区信号
                Consume message m;
            until false;
        end
    parend
end
```

> **重要**：P(empty)和P(mutex)**不能交换顺序**！否则可能死锁。
> - 正确顺序：先P(同步信号量)，再P(互斥信号量)
> - V操作的顺序交换影响不大

**利用AND信号量实现**：

```pascal
Producer:
    repeat
        produce an item in nextp;
        Swait(empty, mutex);      // 同时申请
        Buffer[in] := nextp;
        in := (in + 1) mod n;
        Ssignal(mutex, full);     // 同时释放
    until false;

Consumer:
    repeat
        Swait(full, mutex);
        nextc := Buffer[out];
        out := (out + 1) mod n;
        Ssignal(mutex, empty);
        consume the item in nextc;
    until false;
```

#### 2.4.2 哲学家进餐问题

**问题描述**：5位哲学家围坐在圆桌旁，每人之间有一支筷子（共5支），哲学家需要同时拿到左右两支筷子才能进餐。

**信号量设置**：`chopstick[0..4]`，每个信号量初值为1。

**基本实现（会导致死锁）**：

```pascal
Var chopstick: array[0..4] of semaphore := (1,1,1,1,1);
process i:
    repeat
        wait(chopstick[i]);
        wait(chopstick[(i+1) mod 5]);
        eat;
        signal(chopstick[i]);
        signal(chopstick[(i+1) mod 5]);
        think;
    until false;
```

> **死锁原因**：每位哲学家同时拿起左边筷子，都等待右边筷子。

**三种解决方案**：

| 方案 | 说明 |
|------|------|
| **方案1** | 至多只允许4位哲学家同时拿左边筷子 |
| **方案2** | 仅当左右两只筷子均可用时才允许拿起 |
| **方案3** | 奇数号先拿左再拿右，偶数号先拿右再拿左 |

**利用AND信号量实现（方案2）**：

```pascal
Var chopstick: array[0..4] of semaphore := (1,1,1,1,1);
process i:
    repeat
        think;
        Swait(chopstick[i], chopstick[(i+1) mod 5]);
        eat;
        Ssignal(chopstick[i], chopstick[(i+1) mod 5]);
    until false;
```

#### 2.4.3 读者-写者问题

**问题描述**：多个读者可同时读，但写者写时排斥其他所有读者和写者。

**信号量设置**：
- `rmutex`：互斥访问readcount，初值=1
- `wmutex`：读者与写者、写者与写者之间互斥，初值=1
- `readcount`：记录正在读的读者数，初值=0

**读者优先实现**：

```pascal
Var rmutex, wmutex: semaphore := 1, 1;
    readcount: integer := 0;
begin
    parbegin
        Reader: begin
            repeat
                wait(rmutex);
                if readcount = 0 then wait(wmutex);  // 第一个读者
                readcount := readcount + 1;
                signal(rmutex);
                执行读操作;
                wait(rmutex);
                readcount := readcount - 1;
                if readcount = 0 then signal(wmutex); // 最后一个读者
                signal(rmutex);
            until false;
        end
        
        Writer: begin
            repeat
                wait(wmutex);
                执行写操作;
                signal(wmutex);
            until false;
        end
    parend
end
```

> **注意**：第一个读者负责P(wmutex)，最后一个读者负责V(wmutex)。

**利用信号量集实现**：

```pascal
Var L, mx: semaphore := RN, 1;  // RN为最大读者数
Reader:
    repeat
        Swait(L, 1, 1);     // 读者数限制
        Swait(mx, 1, 0);    // 检查是否有写者
        执行读操作;
        Ssignal(L, 1);
    until false;

Writer:
    repeat
        Swait(mx, 1, 1; L, RN, 0);  // 独占访问
        执行写操作;
        Ssignal(mx, 1);
    until false;
```

#### 2.4.4 吸烟者问题（补充经典同步问题）

**问题描述**：桌上有一个空盘，爸爸放苹果或桔子，儿子吃桔子，女儿吃苹果。

**信号量设置**：S=1（盘空），So=0（有桔子），Sa=0（有苹果）

begin

                else V(So);
            until false;
        end
        Son: begin
            repeat
                P(So);
                取桔子;
                V(S);
                吃桔子;
            until false;
        end
        Daughter: begin
            repeat
                P(Sa);
                取苹果;
                V(S);
                吃苹果;
            until false;

end

### 2.5 管程机制

#### 2.5.1 管程的定义

> **复习要点**：管程是为了解决信号量机制中临界区分散在各进程中的问题。

**管程（Monitor）**：定义了一个数据结构和能为并发进程所执行的一组操作，这组操作能**同步进程**和**改变管程中的数据**。

**管程的组成**：
1. **名称**：管程的标识
2. **共享变量说明**：局部于管程的变量（含条件变量）
3. **一组过程**：对数据结构进行操作的程序段（相当于临界区）
4. **初始化**：对局部数据设置初始值

```pascal
Type 管程名称 = monitor
    共享变量说明语句
    procedure entry P1(...)
        begin ... end;
    procedure entry P2(...)
        begin ... end;
    begin
        初始化语句
    end;
```

#### 2.5.2 管程的特点

- 进程访问临界资源**必须经过管程**
- 管程每次**只允许一个进程进入**（自动实现互斥）
- 使用**条件变量**实现同步：`x.wait`（挂起等待）、`x.signal`（唤醒）

#### 2.5.3 利用管程解决生产者-消费者问题

```pascal
Type PC = monitor
    Var in, out, count: integer;
        Buffer: array[0..n-1] of item;
        notfull, notempty: condition;   // 条件变量
    procedure entry put(var nextp: item)
        begin
            if count >= n then notfull.wait;  // 缓冲区满则等待
            Buffer[in] := nextp;
            in := (in + 1) mod n;
            count := count + 1;
            if notempty.queue then notempty.signal;
        end;
    procedure entry get(var nextc: item)
        begin
            if count <= 0 then notempty.wait; // 缓冲区空则等待
            nextc := Buffer[out];
            out := (out + 1) mod n;
            count := count - 1;
            if notfull.queue then notfull.signal;
        end;
    begin
        in := 0; out := 0; count := 0;
    end;
```

**进程描述**：
```pascal
procedure producer;
begin
    repeat
        生产一个产品nextp;
        PC.put(nextp);
    until false;
end;

procedure consumer;
begin
    repeat
        PC.get(nextc);
        消费一个产品nextc;
    until false;
end;
```

**管程的优点**：调用简单、进程结构清晰、易于查错。

### 2.6 进程通信

> **复习要点**：三种通信方式的对比是考试重点！

#### 2.6.1 进程通信的类型

| 通信方式 | 特点 | 适用场景 |
|----------|------|----------|
| **共享存储器** | 进程共享某数据结构或存储区，需自己设置同步机制 | 传送少量/大量数据 |
| **消息传递** | 以格式化的消息为单位，通过通信原语实现 | 最广泛使用的通信机制 |
| **管道通信** | 利用共享文件（pipe）连接读写进程 | 大量数据传送 |

#### 2.6.2 共享存储器系统

1. **基于共享数据结构**：低效，只适用传送少量数据
2. **基于共享存储区**：UNIX中通信速度最高，可传送大量数据
   - 进程通过`shmat`将共享存储区附接到各自虚地址空间
   - 需自己设置同步机制

#### 2.6.3 消息传递系统

**1. 直接通信方式**

```
Send(Receiver, message);    // 发送消息给目标进程
Receive(Sender, message);   // 接收消息
```

- 要求发送进程和接收进程**显式提供对方标识符**

**2. 间接通信方式（信箱通信）**

```
Send(mailbox, message);     // 将消息发送到指定信箱
Receive(mailbox, message);  // 从指定信箱接收消息
```

**信箱类型**：
- **私用信箱**：用户进程为自己建立，单向通信
- **公用信箱**：OS创建，所有核准进程可用，双向通信
- **共享信箱**：某进程创建并指明可共享

**发送与接收进程的关系**：一对一、多对一（客户/服务器）、一对多（广播）、多对多

**进程同步方式**：
1. 发送阻塞、接收阻塞（双方紧密同步，汇合通信）
2. 发送不阻塞、接收阻塞（发送方不等待，接收方等待）
3. 发送不阻塞、接收不阻塞（双方均不等待）

#### 2.6.4 管道通信

**管道（Pipe）**：用于连接一个读进程和一个写进程以实现通信的**共享文件**。

**管道通信的三方面协调能力**：
1. **互斥**：一个进程对pipe读写时，另一进程必须等待
2. **同步**：写进程写满后睡眠等待，读进程取走数据后唤醒写进程；读进程读空后睡眠等待，写进程写入后唤醒读进程
3. **确定对方存在**：只有确定对方存在才能通信

```c
#include <stdio.h>
main() {
    int x, fd[2];
    char buf[30], s[30];
    pipe(fd);                    /* 创建管道 */
    while((x = fork()) == -1);   /* 创建子进程 */
    if (x == 0) {                /* 子进程 */
        sprintf(buf, "This is an example\n");
        write(fd[1], buf, 30);   /* 写入管道 */
        exit(0);
    } else {                     /* 父进程 */
        wait(0);
        read(fd[0], s, 30);      /* 从管道读取 */
        printf("%s", s);
    }
}
```

#### 2.6.5 消息缓冲队列通信机制

**消息缓冲区结构**：
```pascal
type message_buffer = record
    sender;     // 发送进程标识符
    size;       // 消息长度
    text;       // 消息正文
    next;       // 指向下一个消息缓冲区的指针
end
```

**PCB通信相关数据项**：
```pascal
type process_control_block = record
    mq;         // 消息队列队首指针
    mutex;      // 消息队列互斥信号量
    sm;         // 消息队列资源信号量
end
```

### 2.7 线程

#### 2.7.1 线程的基本概念

**线程（Thread）**：进程中一个独立的执行流，是**调度和分派的基本单位**。

> **复习要点**：进程是资源分配的单位，线程是调度执行的单位！

**引入线程的目的**：减少程序并发执行时的时空开销，提高并发性。

**线程的属性**：

| 属性 | 说明 |
|------|------|
| **轻型实体** | 不拥有系统资源（仅有TCB和少量资源） |
| **独立调度和分派的基本单位** | 线程切换开销小 |
| **可并发执行** | 同一进程内的线程可并发，不同进程的线程也可并发 |
| **共享进程资源** | 同属一个进程的线程共享该进程的所有资源 |

**线程的状态**：执行状态、就绪状态、阻塞状态（与进程类似）

#### 2.7.2 线程与进程的比较

| 对比项 | 进程（Process） | 线程（Thread） |
|--------|----------------|---------------|
| **资源分配** | 资源分配的基本单位 | 不拥有系统资源，共享所属进程资源 |
| **调度单位** | 传统OS中是调度单位 | **调度和分派的基本单位** |
| **并发性** | 进程间可并发 | 线程间可并发，更高效 |
| **系统开销** | 创建/撤销/切换开销大 | 创建/撤销/切换开销小 |
| **独立性** | 独立地址空间 | 共享进程地址空间 |
| **通信** | 需OS支持（IPC） | 直接读写进程数据段 |
| **健壮性** | 一个进程崩溃不影响其他 | 一个线程崩溃可能导致整个进程崩溃 |

#### 2.7.3 线程的实现方式

| 实现方式 | 说明 | 特点 |
|----------|------|------|
| **内核支持线程（KST）** | 线程的创建、撤销、切换由内核实现，内核维护TCB | 切换慢，可发挥多处理机优势，阻塞不影响其他线程 |
| **用户级线程（ULT）** | 线程管理在用户空间完成，内核不知道线程存在 | 切换快，不需内核支持，一个线程阻塞整个进程阻塞 |
| **组合方式** | 内核支持多个内核线程，每个进程可创建多个用户线程 | 结合两者优点 |

**用户级线程的实现**：
- **运行时系统（Runtime System）**：管理和控制线程的函数集合
- **内核控制线程（LWP，轻型进程）**：作为用户级线程和内核的桥梁

#### 2.7.4 线程的同步和通信

| 机制 | 说明 |
|------|------|
| **互斥锁（mutex）** | 用于线程间对资源互斥访问，有开锁（unlock）和关锁（lock）两种状态 |
| **条件变量** | 与互斥锁配合使用，用于线程长期等待直至资源可用 |
| **信号量机制** | 私用信号量（同一进程内线程同步）+ 公用信号量（不同进程间线程同步） |

---

### 第2章重点总结

> **复习要点**：
> 1. **进程状态转换**：三种基本状态（就绪→运行→阻塞）及转换条件，五状态模型
> 2. **PCB是进程存在的唯一标志**，包含进程标识符、处理机状态、调度信息、控制信息
> 3. **信号量PV操作**：wait(S)=P(S)=S-1，signal(S)=V(S)=S+1；互斥信号量初值为1，同步信号量初值按需设置
> 4. **经典同步问题**：生产者-消费者（mutex, empty, full）、哲学家进餐（AND信号量防死锁）、读者-写者（readcount计数）
> 5. **管程**：封装同步机制，自动实现互斥，条件变量实现同步
> 6. **进程通信**：共享存储、消息传递（直接/间接）、管道通信
> 7. **线程vs进程**：进程是资源分配单位，线程是调度执行单位

# 第3章 处理机调度与死锁

## 3.1 处理机调度的基本概念

### 3.1.1 三级调度

处理机调度分为三个级别：**高级调度**（作业调度）、**中级调度**（内存调度）和**低级调度**（进程调度）。

#### 一、高级调度（作业调度/宏观调度）

- 决定允许哪些作业参与竞争CPU和其他系统资源
- 将一个或一批作业从**后备状态变为运行状态**
- 为作业分配**虚拟处理机**
- 主要功能：
  1. 根据调度算法和计算机当前状态，挑选一个或多个后备作业投入运行
  2. 为选中的作业分配基本的内存和设备资源
  3. 为选中的作业建立进程，将进程实体装入内存
- 调度时机：
  - 当前运行的作业数目少于系统允许的最大作业数目，且存在后备作业
  - 一作业运行终止被撤销后，且存在后备作业
  - 分时系统中，一用户被核准注册

#### 二、中级调度（进程交换调度）

- 决定哪些进程可参与竞争CPU
- 将进程从**活动态**（活动就绪、活动阻塞）变为**静止的挂起态**（静止就绪、静止阻塞），或相反
- 实际上是实现**"挂起"和"激活"操作**
- 通常仅用于**分时系统**

#### 三、低级调度（进程调度/微观调度）

- 决定哪个进程可获得CPU
- 从**活动就绪队列**中挑选一个进程，将它变为**运行态**，同时启动CPU执行该进程

**低级调度方式**：

| 方式 | 说明 |
|------|------|
| **非抢占方式** | 进程获得CPU后，一直执行完成或自动阻塞 |
| **抢占方式** | 允许其他进程依据一定原则抢占CPU资源 |

**抢占原则**：
1. **时间片原则**：时间片用完即被抢占
2. **优先权原则**：高优先级进程可抢占低优先级进程
3. **短作业（进程）优先原则**：短进程可抢占长进程

**低级调度时机**：
- 现行进程执行完成或自动阻塞
- 采用抢占方式的系统中，发生了某种抢占事件

### 3.1.2 调度队列模型

#### 仅有低级调度的调度队列模型

每个进程执行时有三种可能：
1. 在时间片内已完成 → 进入完成状态
2. 未完成 → 放在就绪队列后
3. 因某事件 → 进入阻塞状态

#### 具有高级和低级调度的调度队列模型

- 作业调度从后备队列中选择作业进入就绪队列
- 进程调度从就绪队列中选择进程执行

#### 具有三级调度的调度队列模型

- 包含后备队列、就绪队列（活动/静止）、阻塞队列（活动/静止）

## 3.2 调度算法

### 3.2.1 调度评估标准

#### 一、面向用户的原则

- **周转时间短**
- **响应时间快**
- **截止时间的保证**
- **优先权准则**

#### 二、面向系统的原则

- **系统吞吐量高**
- **处理机利用率好**
- **各类资源的平衡使用**

#### 三、评估指标公式

| 指标 | 公式 |
|------|------|
| **周转时间 Ti** | `Ti = 完成时刻 - 进入时刻` |
| **平均周转时间 T** | `T = (T1 + T2 + ... + Tn) / n` |
| **带权周转时间 Wi** | `Wi = Ti / Tsi`（周转时间/实际运行时间） |
| **平均带权周转时间 W** | `W = (W1 + W2 + ... + Wn) / n` |
| **等待时间** | 进程在就绪队列中等待的时间总和 |
| **响应时间** | 从提交请求到首次产生响应的时间 |

> **复习要点**：带权周转时间 `W = T/Ts`，反映作业等待的相对程度。W越小，系统性能越好。

---

### 3.2.2 先来先服务（FCFS）调度算法

**作业调度**：从后备队列中选择最先进入队列的作业。
**进程调度**：从就绪队列中选择最先进入队列的进程，分配处理机直至运行完或阻塞。

**特性**：
- 以**非抢占方式**工作
- 可用于作业调度与进程调度

| 优点 | 缺点 |
|------|------|
| 实现简单 | 对短作业不利（护航效应） |
| 有利于CPU繁忙型作业 | 对I/O繁忙型作业不利 |
| 有利于长时间作业 | 对紧迫作业不利 |

> **复习要点**：FCFS的平均周转时间可能很长，特别是当长作业先于短作业到达时。

---

### 3.2.3 短作业(进程)优先（SJF/SRTF）调度算法

**作业调度**：从后备队列中选择估计执行时间最短的作业。
**进程调度**：从就绪队列中选择估计需CPU时间最短的进程。

**SRTF**（最短剩余时间优先）：抢占式版本，当新到达进程的剩余时间比当前进程更短时发生抢占。

**特性**：
- 可以**抢占方式**（SRTF）和**非抢占方式**（SJF）工作
- 可用于作业调度与进程调度

| 优点 | 缺点 |
|------|------|
| 有利于短作业 | 对长作业不利（饥饿问题） |
| 抢占方式下短作业最快完成 | 未考虑作业紧迫程度 |
| 平均等待时间最小（理论上） | 运行时间是估计值 |

> **复习要点**：SJF在抢占方式下称为SRTF（Shortest Remaining Time First），可获得最优平均等待时间。

---

### 3.2.4 高响应比优先（HRN/HRRN）调度算法

**响应比公式**：
```
Rp = (等待时间 + 要求服务时间) / 要求服务时间 = 1 + 等待时间/要求服务时间
```

**作业调度**：选择响应比 Rp 最高的作业。
**进程调度**：选择响应比 Rp 最高的进程。

**特性**：
- 可用于作业与进程调度
- 是**FCFS和SF的折衷算法**
- Rp与需要运行时间**成反比** → 短作业Rp较高
- Rp与等待时间**成正比** → 等待时间长的作业Rp较高
- 一般采用**非抢占方式**

| 优点 | 缺点 |
|------|------|
| 兼顾短作业和长作业 | 每次调度需计算Rp，增加系统开销 |
| 有利于先来者 | 未考虑紧迫程度 |

> **复习要点**：HRN解决了SJF中长作业饥饿的问题，等待时间越长，响应比越高，最终一定会被调度。

---

### 3.2.5 最高优先权（HPF）调度算法

**作业调度**：从后备队列中选择优先权最高的作业。
**进程调度**：从就绪队列中选择优先权最高的进程。

**调度方式**：
- **非抢占方式**：一旦分配CPU，直到进程完成或阻塞
- **抢占方式**：新到达的高优先级进程可立即抢占CPU

**优先权类型**：

| 类型 | 说明 |
|------|------|
| **静态优先权** | 调度前赋值，之后不变。依据：进程类型、资源要求、用户要求 |
| **动态优先权** | 调度前赋初值，运行过程中动态改变 |

**特性**：
- 可用于作业与进程调度
- 动态优先权只用于进程调度
- 抢占方式可用于**实时系统**
- **特例**：优先权只依据到达时间→FCFS；只依据运行时间→SF；只依据Rp→HRN

---

### 3.2.6 时间片轮转（RR）调度算法

RR是一种**抢占式**的进程调度算法，依据**公平服务原则**，为每个进程轮转服务一次，每次执行一个**时间片**。

**时间片大小的确定**：
1. 系统对响应时间T的要求
2. 就绪队列中进程数目N：`T = N * q`（T：一次轮转时间，q：时间片大小）
3. 系统的处理能力：保证普通命令能在一个时间片内完成

**时间片类型**：
- **等长时间片**：时间片应尽量长以满足响应时间要求
- **不等长时间片**：根据优先权分配不同长度的时间片

**特性**：
- 用于**进程调度**，适合**分时系统**
- 时间片越长，越有利于缩短周转时间
- 时间片太长 → 退化为FCFS

| 优点 | 缺点 |
|------|------|
| 有利于交互性、事务性进程 | 调度开销较大 |
| 有利于I/O繁忙型进程 | 未考虑实时响应要求 |

> **复习要点**：RR的时间片选择很重要。q太小→上下文切换开销大；q太大→退化为FCFS。

---

### 3.2.7 多级队列调度算法

- 将就绪队列分成多种不同队列（**前台RR就绪队列**、**后台FCFS就绪队列**）
- 每个进程**固定**分属于一个队列
- 不同队列采用不同调度算法（前台RR、后台FCFS）
- 只有当前台队列空时，才执行后台队列

| 优点 | 缺点 |
|------|------|
| 兼顾分时及批量处理任务 | 未考虑紧迫性作业 |

---

### 3.2.8 多级反馈队列（MFQ）调度算法

**算法原理**：
1. 设置多个就绪队列，从高到低赋予**不同优先级**
2. 每个队列采用RR算法，时间片长度从高优先级到低优先级依次**增加**（一般加倍）：`S1 < S2 < ... < Sn`
3. **新进程**放到**最高优先级**就绪队列尾部
4. 进程在当前队列执行一次**未完成** → 插入**低一级**队列尾部
5. **较高优先级队列全部为空**时，才执行当前队列进程
6. 如有更高优先级进程到达，当前进程放回当前队列尾部，转而执行高优先级进程

**特性**：

| 用户类型 | 表现 |
|----------|------|
| **终端型用户**（交互性作业小） | 在最高优先级队列快速完成 |
| **短作业用户** | 执行时间短，很快降到低优先级队列前完成 |
| **长作业用户** | 在第n个队列中按RR算法轮转执行 |

| 优点 | 缺点 |
|------|------|
| 兼顾实时、分时及批量处理 | 调度算法复杂，开销较大 |

> **复习要点**：多级反馈队列是Unix等系统实际使用的算法，能自动适应各种进程类型。

---

### 3.2.9 各调度算法比较总结

| 算法 | 抢占性 | 适用调度 | 主要优点 | 主要缺点 |
|------|--------|----------|----------|----------|
| **FCFS** | 非抢占 | 作业+进程 | 简单公平 | 护航效应，平均等待时间长 |
| **SJF** | 均可 | 作业+进程 | 最小平均等待时间 | 长作业饥饿，估计运行时间 |
| **SRTF** | 抢占 | 进程 | 真正最优平均等待 | 上下文切换开销大 |
| **HRN** | 非抢占 | 作业+进程 | 兼顾长短作业 | 每次需计算响应比 |
| **HPF** | 均可 | 作业+进程 | 灵活，适应实时 | 可能饥饿低优先级 |
| **RR** | 抢占 | 进程 | 公平响应快 | 时间片选择困难 |
| **多级队列** | 不一定 | 进程 | 兼顾多种类型 | 不够灵活 |
| **多级反馈队列** | 抢占 | 进程 | 自动适应各类进程 | 实现复杂 |

## 3.3 实时调度

### 3.3.1 实现实时调度的基本条件

1. **提供必要的信息**：就绪时间、开始/完成截止时间、处理时间、资源要求、优先级
2. **系统处理能力强**：单处理机需满足 `Σ(Ci/Pi) ≤ 1`；N处理机需满足 `Σ(Ci/Pi) ≤ N`
3. **采用抢占式调度机制**
4. **具有快速切换机制**：对外部中断快速响应、快速任务分派

### 3.3.2 实时调度算法

| 算法 | 响应时间 | 适用场景 |
|------|----------|----------|
| **时间片轮转** | 数秒 | 一般实时系统 |
| **非抢占优先权** | 数百毫秒 | 不太严格的实时控制 |
| **基于时间中断的抢占优先权** | 几毫秒~几十毫秒 | 大多数实时控制系统 |
| **立即抢占的优先权** | 微秒~几毫秒 | 所有实时控制系统 |

#### 最早截止时间优先（EDF）算法

- 根据任务的**开始截止时间**确定优先级
- 开始截止时间越早，优先级越高
- 是**最优的动态优先级调度算法**

#### 最低松弛度优先（LLF）算法

- 根据任务的**紧急（松弛）程度**确定优先级

```
松弛度 = 完成截止时间 - 处理时间 - 当前时间
```

- 松弛度越小，优先级越高
- **松弛度为0时发生抢占**
- 处理时间指**剩余需要CPU时间**

> **复习要点**：LLF中当两个任务松弛度相等时，通常继续执行当前任务以减少切换开销。

---

## 3.4 死锁的基本概念

### 3.4.1 死锁的定义

**死锁**：指多个进程因竞争资源而造成的一种僵局，若无外力作用，这些进程都将永远不能再向前推进。

### 3.4.2 产生死锁的原因

1. **竞争资源**：两个或以上进程需要两个或以上资源
   - **可剥夺性资源**：可被其他进程剥夺（如CPU、RAM）
   - **非剥夺性资源**：不可剥夺，只能自行释放（如打印机）
   - **临时性资源**：由某进程产生，另一进程临时使用（如信号量）
2. **进程推进顺序非法**：请求和释放资源的顺序不当

### 3.4.3 产生死锁的必要条件（同时满足）

| 条件 | 说明 |
|------|------|
| **互斥条件** | 请求的资源为临界资源，一次仅能被一个进程使用 |
| **请求和保持条件** | 申请新资源的同时保持已占有的旧资源 |
| **不剥夺条件** | 已获得的资源在使用完之前不能被外力剥夺 |
| **环路等待条件** | 存在进程-资源的循环等待链 |

> **复习要点**：四个条件必须**同时满足**才会产生死锁。破坏任意一个即可预防死锁。

---

## 3.5 死锁预防

通过设置限制条件，**破坏**产生死锁的四个必要条件中的一个或几个。

| 被破坏条件 | 方法 | 优点 | 缺点 |
|------------|------|------|------|
| **请求和保持** | 一次将所有资源全部分配 | 简单安全 | 资源浪费严重，进程延迟 |
| **不剥夺** | 请求不满足时释放已分配资源 | 避免死锁 | 前期工作作废，代价高 |
| **环路等待** | 对资源申请必须按一定顺序 | 有效 | 限制用户编程，资源可能闲置 |
| **互斥** | 将独占资源改为共享（Spooling） | 彻底 | 不适用于所有资源 |

> **复习要点**：预防死锁是**静态**策略，通过限制资源分配来杜绝死锁可能，但可能导致资源利用率降低。

---

## 3.6 死锁避免——银行家算法

### 3.6.1 安全状态

- **安全状态**：系统能按某种进程顺序（安全序列）为每个进程分配所需资源，使每个进程都可顺利完成
- **不安全状态**：系统无法找到这样的安全序列
- 不安全状态**不一定**发生死锁，但死锁一定发生在不安全状态

> **复习要点**：避免死锁的核心是**确保系统始终处于安全状态**，只在安全状态下才分配资源。

### 3.6.2 银行家算法数据结构

| 数据结构 | 说明 | 维度 |
|----------|------|------|
| **Available** | 可利用资源向量 | m个元素 |
| **Max** | 最大需求矩阵 | n×m |
| **Allocation** | 分配矩阵 | n×m |
| **Need** | 需求矩阵 | n×m |

核心关系：
```
Need[i,j] = Max[i,j] - Allocation[i,j]
```

### 3.6.3 银行家算法步骤

设 `Requesti` 是进程 `Pi` 的请求向量：

1. 若 `Requesti[j] ≤ Need[i,j]`，转步骤2；否则出错（超过最大值）
2. 若 `Requesti[j] ≤ Available[j]`，转步骤3；否则 `Pi` 须等待
3. 系统**试探分配**，修改数据结构：
   ```
   Available[j]  = Available[j] - Requesti[j]
   Allocation[i,j] = Allocation[i,j] + Requesti[j]
   Need[i,j]  = Need[i,j] - Requesti[j]
   ```
4. 执行**安全性算法**检查是否安全
   - 若安全 → 正式分配
   - 若不安全 → 恢复原来状态，`Pi` 等待

### 3.6.4 安全性算法

1. 设置 `Work := Available`，`Finish[i] := false`
2. 找到满足 `Finish[i] = false` 且 `Need[i,j] ≤ Work[j]` 的进程 `Pi`
3. 若找到：
   ```
   Work[j] := Work[j] + Allocation[i,j]
   Finish[i] := true
   返回步骤2
   ```
4. 若所有 `Finish[i] = true`，则系统**安全**；否则**不安全**

> **复习要点**：银行家算法的关键是"**试探分配 + 安全性检查**"。安全性算法寻找安全序列的过程就是模拟资源回收。

### 3.6.5 银行家算法的缺点

1. 很少有进程能在运行前就知道所需资源的最大值
2. 进程数不固定（用户登录/退出）
3. 可用资源可能突然不可用（设备故障）
4. 算法开销较大，实时性不好

---

## 3.7 死锁检测与解除

### 3.7.1 死锁的检测

#### 资源分配图

| 元素 | 表示 |
|------|------|
| **进程** | 圆圈 ○ |
| **资源** | 方框 □ |
| **请求边** | 进程 → 资源（有向边） |
| **分配边** | 资源 → 进程（有向边） |

#### 死锁定理

> 系统处于死锁状态的**充分必要条件**：当且仅当系统的资源分配图是**不可完全简化**的。

**简化过程**：
1. 找到一个可获得所有资源的进程结点，使其变成孤立点（释放资源）
2. 重复找下一个可获得所有资源的进程结点
3. 如果**所有进程都变成孤立点** → **无死锁**
4. 否则 → **存在死锁**

**注意**：有环**不一定**导致死锁（资源有多个实例时）。

### 3.7.2 死锁的解除

| 方法 | 说明 |
|------|------|
| **剥夺资源** | 从其他进程剥夺资源给死锁进程 |
| **撤销进程** | 按某种策略逐个撤销死锁进程，直到解除死锁 |

**撤销进程的选择标准**：
- 程序的优先数（代价与优先级成正比）
- 作业的外部代价（学生 < 行政 < 系统程序）
- 运行代价（重新启动到撤销点的代价）

### 3.7.3 处理死锁的基本方法总结

| 方法 | 策略 | 特点 |
|------|------|------|
| **预防死锁** | 静态 | 破坏必要条件，资源利用率低 |
| **避免死锁** | 动态 | 银行家算法，保持安全状态 |
| **检测死锁** | 被动 | 定期检测，发现后解除 |
| **解除死锁** | 补救 | 剥夺资源或撤销进程 |
| **忽略死锁** | 鸵鸟策略 | UNIX/Linux/Windows实际采用 |

### 调度算法核心对比

```
FCFS：公平但慢，护航效应
SJF：最优平均等待，但会饥饿
HRN：FCFS + SJF 的折中
HPF：灵活，适合实时
RR：分时系统必备
多级反馈队列：综合最优，实际应用广泛
```

### 银行家算法核心步骤

```
1. Request ≤ Need?  否→出错
2. Request ≤ Available? 否→等待
3. 试探分配（修改Available/Allocation/Need）
4. 安全性检查 → 安全则分配，否则恢复等待
```

# 第4章 存储器管理

## 4.1 存储管理基础

### 4.1.1 存储层次结构

冯·诺依曼计算机系统采用**多级存储层次结构**：
- **高层**：CPU寄存器、高速缓存（Cache）——速度快、成本高、容量小
- **中层**：主存（RAM）——可执行的存储器
- **低层**：辅存（硬盘/SSD）——容量大、成本低、速度慢

### 4.1.2 存储管理的功能

| 功能 | 说明 |
|------|------|
| **内存分配** | 为每道程序分配内存空间，提高利用率 |
| **地址映射** | 将逻辑地址转换为物理地址 |
| **存储保护** | 确保每道程序在自己的内存空间运行，互不干扰 |
| **内存扩充** | 从逻辑上扩充内存容量（虚拟存储） |

---

## 4.2 程序的装入和链接

### 4.2.1 程序的装入方式

| 装入方式 | 重定位时机 | 特点 | 硬件支持 |
|----------|-----------|------|----------|
| **绝对装入** | 编译/汇编时确定地址 | 程序员需熟悉内存情况，程序不可移动 | 不需要 |
| **可重定位装入（静态重定位）** | 装入时修改地址 | 程序装入后不可移动，需连续空间 | 不需要 |
| **动态运行时装入（动态重定位）** | 执行时动态转换 | 程序可移动，可分散存放 | **重定位寄存器** |

**静态重定位示例**：
```
相对地址指令：LOAD 1, 2500  （将地址2500的内容装入累加器1）
装入起始地址：10000
修改后指令：  LOAD 1, 12500 （10000 + 2500 = 12500）
```

**动态重定位原理**：
```
物理地址 = 逻辑地址 + 重定位寄存器值
```

- 目标模块装入时无需修改地址
- 每次访问内存前由**MMU（存储管理部件）**自动完成地址变换
- 程序可在内存中移动（只需更新重定位寄存器）
- 程序可分散存放在不连续区域（每区域对应一个重定位寄存器）

### 4.2.2 程序的链接方式

| 链接方式 | 时机 | 特点 |
|----------|------|------|
| **静态链接** | 装入前 | 将目标模块和库函数链接成完整装入模块，不便于更新和共享 |
| **装入时动态链接** | 装入时 | 便于修改更新，便于共享目标模块 |
| **运行时动态链接** | 执行时 | 只装入需要的模块，加快装入速度，节省内存 |

> **复习要点**：运行时动态链接是Windows DLL的基础，用到时才装入内存。

---

## 4.3 连续分配方式

### 4.3.1 单一连续分配

- 一个用户程序独占连续的内存用户区
- 只能用于**单用户、单任务**OS
- 系统分为**系统区**和**用户区**

### 4.3.2 固定分区分配

将内存划分为多个区域，每个区域存放一个用户程序。

**划分分区方法**：

| 方法 | 优点 | 缺点 |
|------|------|------|
| **分区大小相等** | 简单 | 大程序装不下，小程序浪费严重 |
| **分区大小不等** | 适应性较强 | 特别大的程序可能仍装不下 |

**内存分配算法**：
- **首次适应算法(FF)**：按序选择第一个满足要求的内存区
- **最佳适应算法(BF)**：仅当与程序大小相当的分区空闲时才予分配

### 4.3.3 动态分区分配

按进程实际需要，**动态地**分配内存空间。

**数据结构**：
- **空闲分区表**：记录空闲分区
- **空闲分区双向链表**：将空闲分区链接起来

**分配算法**：

| 算法 | 查找方式 | 优点 | 缺点 |
|------|----------|------|------|
| **首次适应(FF)** | 按地址顺序，选第一个满足的 | 保留高地址大空闲区 | 低地址碎片多，查找时间长 |
| **循环首次适应(CF)** | 从上次找到位置继续查找 | 空闲分区均匀分布 | 缺乏大空闲区 |
| **最佳适应(BF)** | 选大小最接近的空闲区 | 提高内存使用效率 | 产生许多小碎片 |
| **最坏适应(WF)** | 选最大的空闲区 | 剩下空闲区仍较大 | 大分区迅速消耗 |
| **快速适应(分类搜索)** | 按大小分类组织 | 查找效率高 | 回收开销大，分配有浪费 |

**回收操作的四种情况**：
1. 回收区与**前**空闲区F1相邻接 → **合并到F1**（F1大小增加）
2. 回收区与**后**空闲区F2相邻接 → **与F2合并**（首址用回收区首址）
3. 回收区与**前后**都相邻接 → **三者合并**（用F1首址，大小为三区之和，取消F2）
4. 回收区**不与任何空闲区邻接** → 新建表项插入链表

> **50%规则**：N个已分配块，另有约0.5N块会因碎片而损失（即1/3内存不可用）。

### 4.3.4 可重定位分区分配

- **紧凑（拼接）**：对内存中正在使用的分区进行搬迁，使多个小碎片合并为一个大空闲区
- 紧凑需要**动态重定位**机制支持

### 4.3.5 对换(Swapping)

- 把内存中暂不能运行的进程或不用的数据**换出到外存**，腾出内存空间
- **整体对换**：以进程为单位（即挂起操作）
- **部分对换**：以页或段为单位（虚拟存储器）

**对换空间管理**：
- 外存分为**文件区**和**对换区**
- 对换区是连续的

**进程的换出**：优先选择阻塞进程，然后选择低优先级的就绪进程。
**进程的换入**：对就绪且已换出的进程，先申请内存，成功则换入。

## 4.4 基本分页存储管理方式

### 4.4.1 页面和页表

#### 基本概念

| 概念 | 说明 |
|------|------|
| **页面（页）** | 进程逻辑空间分成大小相等的片 |
| **物理块** | 内存空间分成大小相等的存储块 |
| **页号** | 进程页面的编号 |
| **块号** | 内存物理块的编号 |

**页面大小选择**：
- 通常在 **512字节 ~ 4KB** 之间
- 页面大小是**2的指数幂**（便于地址映射）
- 页面太小 → 页表过长
- 页面太大 → 页内碎片大

#### 进程地址结构

逻辑地址A分为两部分：**页号P** 与 **页内地址d**

```
页号 P = INT(A / L)      （L为页面大小）
页内地址 d = A MOD L
```

**示例**：A = 2170B，L = 1KB = 1024B
```
P = INT(2170 / 1024) = 2
d = 2170 MOD 1024 = 122
```

#### 页表

- 页表记录进程每一页在内存中存放的**块号**
- 页表存放在**内存**中
- 页表的始址和长度存放在**进程PCB**中，执行时送入**页表寄存器(PTR)**

### 4.4.2 地址变换机构

#### 基本地址变换

```
逻辑地址 [页号P | 页内地址d]
           ↓
     页号 ≥ 页表长度? → 越界中断
           ↓
     页表始址 + P × 页表项长度
           ↓
     查页表得物理块号
           ↓
物理地址 [物理块号 | 页内地址d]  （页内地址不变）
```

**缺点**：访问一次逻辑地址需**两次访问内存**（查页表+取数据）。

#### 具有快表（TLB）的地址变换

- **快表（TLB）**：具有**并行查寻能力**的特殊高速存储器，也称"联想存储器"
- 快表访问速度为内存的数倍（如20ns vs 100ns）

**地址变换过程**：
```
逻辑地址 [页号P | 页内地址d]
           ↓
     页号 ≥ 页表长度? → 越界中断
           ↓
     页号送入快表并行查找
        ↙         ↘
    命中(TLB hit)   未命中(TLB miss)
      ↓              ↓
  直接得块号      查内存页表得块号
      ↓              ↓
  物理地址      更新快表（淘汰策略）
      ↓              ↓
   访问内存      物理地址→访问内存
```

**有效访问时间**：
- 快表命中：20ns + 100ns = **120ns**
- 快表未命中：20ns + 100ns + 100ns = **220ns**（或200ns若并行访问）

#### 地址变换计算示例

**例1**：页长1KB，逻辑地址2500
```
(1) P = INT(2500 / 1024) = 2
    d = 2500 mod 1024 = 452
(2) 查页表得块号 = 5
(3) 物理地址 = 5 × 1024 + 452 = 5572
```

**例2**：页大小2KB，逻辑地址4832
```
P = INT(4832 / 2048) = 2
d = 4832 mod 2048 = 736
查页表得块号 → 物理地址 = 块号 × 2048 + 736
```

> **复习要点**：分页系统中，**页内地址在物理地址中保持不变**，只有页号需要转换为块号。

### 4.4.3 两级页表

#### 为什么需要两级页表

当逻辑地址空间非常大时（如2^32），页表本身会很大：
- 页面大小4KB → 需要2^20 = 1M页表项
- 每项4字节 → 每个进程需4MB连续页表空间

#### 两级页表原理

- **外层页表**：存放页表存放的位置（页表的页表）
- **页表**：以离散方式放入内存
- 地址结构变为三层：**外层页号 | 外层页内地址 | 页内地址**

**逻辑空间计算**：
- 内存块512B，每个块号4B
- 每块可放 512/4 = 128 个页表项
- 一级页表最大逻辑空间：128 × 512 = **64KB**
- 二级页表最大逻辑空间：128 × 128 × 512 = **8MB**

## 4.5 基本分段存储管理方式

### 4.5.1 分段的概念

- 将进程划分为多个具有**逻辑意义且相对独立**的部分
- 如主程序段、数据段、堆栈段、子程序段等
- **每个段都是从0开始的独立逻辑地址空间**

**引入分段的好处**：方便编程、信息共享、信息保护、动态增长、动态链接。

### 4.5.2 分段地址结构

- 作业（进程）由多个段组成
- 每个段独立编址（从0开始）
- 整个作业的地址是**二维结构**：**[段号S | 段内地址d]**

### 4.5.3 段表

- 段表记录每一段在内存中存放的**开始地址**及**段的长度**

### 4.5.4 地址变换

```
逻辑地址 [段号S | 段内地址d]
           ↓
     段号 ≥ 段表长度? → 越界中断
           ↓
     段表始址 + S × 段表项长度
           ↓
     得段首址和段长度
           ↓
     段内地址 > 段长度? → 越界中断
           ↓
     物理地址 = 段首址 + 段内地址
```

**注意**：分段有**两次越界检查**（段号越界 + 段内地址越界）。

### 4.5.5 具有快表的地址变换

- 将段表放入快表中
- 段的数目远比页少，快表命中率较高
- 采用快表的分段地址变换，一般仅比常规内存访问慢**10%~15%**

### 4.5.6 分页与分段的主要区别

| 对比项 | 分页 | 分段 |
|--------|------|------|
| **信息单位** | 页是**物理单位**（系统管理需要） | 段是**逻辑单位**（用户需要） |
| **大小** | 页大小**固定**（由系统决定） | 段大小**不固定**（由用户决定） |
| **地址空间** | **一维**（逻辑地址） | **二维**（段号+段内地址） |
| **碎片** | 存在**页内碎片** | 存在**外部碎片** |
| **共享** | 不易实现完整逻辑共享 | 容易实现逻辑信息共享 |

### 4.5.7 段页式存储管理方式

**基本原理**：
- 先将用户作业分成若干**段**，再将段分成大小固定的若干**页**
- 需要同时设置**段表**和**页表**
- 段表中存放页表始址及页表大小

**地址变换过程**：
```
逻辑地址 [段号S | 段内页号 | 页内地址d]
           ↓
     查段表 → 得页表首址
           ↓
     查页表 → 得物理块号
           ↓
     物理地址 = [物理块号 | 页内地址d]
```

## 4.6 虚拟存储器

### 4.6.1 实存管理的问题

连续内存分配、基本分页/分段的共性：
1. **一次性**：作业必须全部装入才能运行
2. **驻留性**：作业装入后一直驻留内存直到结束

**实存管理的缺点**：
- 作业逻辑空间不能超过实际内存大小
- 同时存在的进程数目受限

### 4.6.2 局部性原理

1968年P. Denning提出**局部性原理**：
- **时间局部性**：最近被访问的指令/数据，短期内很可能再次被访问
- **空间局部性**：程序访问的地址在空间上往往集中在一定范围内

### 4.6.3 虚拟存储器的定义

虚拟存储器是采用**请求调入**和**置换**功能，将内存和外存统一管理，达到把作业的一部分装入内存便可运行，给用户提供的比内存实际容量大的一维逻辑地址空间。

## 4.7 请求分页存储管理

### 4.7.1 页表机制

请求分页系统的页表项包含以下字段：

| 字段 | 说明 |
|------|------|
| **物理块号** | 该页在内存中的块号 |
| **状态位P** | 指示该页是否已调入内存 |
| **访问字段A** | 记录本页在一段时间内被访问的次数 |
| **修改位M** | 该页调入内存后是否被修改过 |
| **外存地址** | 该页在外存中的位置 |

> **复习要点**：状态位用于判断缺页；修改位用于决定是否写回外存（未修改则直接覆盖）。

### 4.7.2 缺页中断

- 当需要访问的页面不在内存时，产生**缺页中断**
- 缺页中断属于**内中断**
- 处理过程：
  1. 保留CPU现场
  2. 从外存中找到所需页面
  3. 若内存有空闲块 → 直接调入
  4. 若无空闲块 → 选择一页换出（页面置换算法），再调入
  5. 修改页表，恢复现场

### 4.7.3 地址变换机构

```
逻辑地址 [页号P | 页内地址d]
           ↓
     越界检查
           ↓
     快表查找
        ↙         ↘
     命中          未命中
      ↓              ↓
  得块号        查内存页表
      ↓              ↓
  访问内存      在内存? → 否→缺页中断
      ↓              ↓
               是→修改快表
                  ↓
               物理地址
```

### 4.7.4 页面分配策略

| 策略 | 说明 |
|------|------|
| **固定分配局部置换** | 为每个进程分配固定页数的内存，缺页时只能从自己的页面中换出 |
| **可变分配全局置换** | 系统维护空闲物理块池，缺页时从全局空闲块或任何进程中换出 |
| **可变分配局部置换** | 根据进程表现动态调整分配块数，但换出限于本进程 |

### 4.7.5 最小物理块数

- 保证进程正常运行所需的**最小物理块数**
- 与硬件结构有关：单地址直接寻址需**2块**（指令+数据）；允许间接寻址需**3块**

## 4.8 页面置换算法

### 4.8.1 最佳置换算法（OPT）

- 选择**永不使用**或**最长时间内不被使用**的页面淘汰
- **优点**：保证最低缺页率
- **缺点**：**无法实现**，只能作为其他算法的评价参考

**示例**：物理块3，访问序列 7,0,1,2,0,3,0,4,2,3,0,3,2,1,2,0,1,7,0,1
- OPT需要**6次**页面置换

### 4.8.2 先进先出置换算法（FIFO）

- 选择**最先进入内存**的页面淘汰（在内存驻留时间最长）
- **优点**：算法简单
- **缺点**：性能不佳，可能产生**Belady异常**

**示例**：同上序列，FIFO需要**12次**页面置换。

#### Belady异常

- **Belady现象**：采用FIFO算法时，分配的**页面数增多**，缺页率反而**提高**的异常现象
- **示例**：访问序列 `A B C D A B E A B C D E`
  - 物理块3块 → 缺页9次
  - 物理块4块 → 缺页10次
- **原因**：FIFO未考虑页面的访问频率和局部性

> **复习要点**：**FIFO是唯一会产生Belady异常的常见页面置换算法**。LRU、OPT等基于栈的算法不会出现Belady异常。

### 4.8.3 最近最久未使用置换算法（LRU）

- 选择**最近最久未被使用**的页面淘汰
- **优点**：性能接近OPT，不会产生Belady异常
- **缺点**：需要较多硬件支持

**示例**：同上序列，LRU需要**9次**页面置换。

#### LRU硬件支持

1. **移位寄存器**：
   - 每页一个寄存器 `R = Rn-1Rn-2...R1R0`
   - 页面被访问 → 最高位置1
   - 定时右移一位
   - **R值最小**的页面被淘汰

2. **栈**：
   - 栈顶 = 最新被访问的页
   - 栈底 = 最久未被访问的页（淘汰目标）

### 4.8.4 Clock置换算法（NRU - 最近未用）

- Clock是LRU的**近似算法**
- 每页设置**1位访问位A**（访问过为1，未访问为0）

**算法步骤**：
1. 将所有页面用指针链成**循环队列**
2. 沿链查找，访问位为**0** → 换出
3. 访问位为**1** → 置为0，继续查找
4. 直到找到访问位为0的页面

### 4.8.5 改进型Clock置换算法

除访问位A外，增加**修改位M**，将页面分为四类：

| 类别 | A | M | 说明 |
|------|---|---|------|
| **第1类** | 0 | 0 | **最佳淘汰页**（未访问未修改） |
| **第2类** | 0 | 1 | 未访问但已修改 |
| **第3类** | 1 | 0 | 已访问未修改 |
| **第4类** | 1 | 1 | 最不应淘汰（已访问已修改） |

**算法步骤**：
1. 查找第1类页面（A=0, M=0），找到则淘汰
2. 未找到 → 查找第2类（A=0, M=1），同时将所有访问过的页面A置0
3. 仍未找到 → 所有页面A已置0，重复步骤1

### 4.8.6 最少使用置换算法（LFU）

- 选择**使用次数最少**的页面淘汰
- 为每个页表项配置**计数器**
- 页面被访问 → 计数器+1
- 淘汰时选择**计数器值最小**的页面

### 4.8.7 各页面置换算法比较

| 算法 | 置换依据 | 是否会产生Belady异常 | 实现复杂度 | 缺页率 |
|------|----------|---------------------|-----------|--------|
| **OPT** | 未来最远使用 | 否 | 不可实现 | 最低（理论） |
| **FIFO** | 进入内存时间 | **是** | 简单 | 较高 |
| **LRU** | 最近最久未用 | 否 | 较复杂 | 接近OPT |
| **Clock** | 最近未用 | 否 | 简单 | 接近LRU |
| **改进Clock** | A+M位 | 否 | 中等 | 接近LRU |
| **LFU** | 使用频率 | 否 | 中等 | 中等 |

### 4.8.8 有效访问时间计算

```
有效访问时间 = (1 - p) × T1 + p × T2

其中：
  p = 缺页率
  T1 = 不缺页时访问时间 = 2 × Tr（查页表+取数据）
  T2 = 缺页时访问时间 ≈ Tt（缺页中断时间，20ms级）
  Tr = 内存访问时间（约100ns）
```

**示例**：p = 0.1%，Tr = 100ns，Tt = 20ms
```
有效访问时间 = (1 - 0.001) × 0.2μs + 0.001 × 20000μs
             ≈ 0.2 + 20 = 20.2μs
```

> **复习要点**：缺页率对性能影响巨大。0.1%的缺页率就会使访问时间从0.2μs增加到约20μs！

## 4.9 工作集模型与抖动

### 4.9.1 工作集

- **工作集**：在某段时间间隔内，进程实际要访问的页面集合
- 工作集大小反映了进程的**局部性特征**
- 如果系统为进程分配的物理块数 **≥ 工作集大小** → 进程可高效运行

### 4.9.2 抖动（Thrashing）

- **抖动现象**：刚换出的页面又立即被访问，需要重新调入，导致系统频繁进行页面置换
- **根本原因**：进程获得的物理块数**不足**（低于工作集大小）
- **后果**：CPU利用率急剧下降，系统吞吐量骤降

**防止抖动的方法**：
1. **局部置换策略**：限制缺页进程只能换出自己的页面
2. **工作集算法**：为每个进程分配不低于其工作集大小的物理块数
3. **L=S准则**：产生缺页的平均时间 = 处理缺页的平均时间
4. **挂起进程**：暂停某些进程，释放其物理块

## 4.10 请求分段存储管理

### 4.10.1 段表机制

请求分段系统的段表项包含：

| 字段 | 说明 |
|------|------|
| **段名/段号** | 段的标识 |
| **段长** | 该段的长度 |
| **内存始址** | 段在内存中的起始地址 |
| **存在位P** | 是否已调入内存 |
| **访问字段A** | 被访问次数 |
| **修改位M** | 是否被修改过 |
| **外存地址** | 段在外存中的位置 |
| **存取方式** | 执行/只读/读+写 |
| **增补位** | 是否进行过动态增长 |

### 4.10.2 缺段中断

- 需要访问的段不在内存时产生**缺段中断**
- 与缺页中断的**相同点**：可能发生在指令执行过程中，一条指令可能多次中断
- 与缺页中断的**不同点**：
  - 一条指令只可能存在于一个段中
  - 一个（组）数据只可能存在于一个段中

### 4.10.3 分段的共享

- 通过**共享段表**管理共享段
- **共享进程计数器COUNT**：记录共享该段的进程数
- 当COUNT=0时回收物理内存
- 不同进程可用**不同段号**共享同一段

## 第4章重点回顾

### 地址变换过程总结

```
【分页地址变换】
逻辑地址 → [页号 | 页内地址]
              ↓
           查页表(或TLB)
              ↓
物理地址 = [块号 | 页内地址]

【分段地址变换】
逻辑地址 → [段号 | 段内地址]
              ↓
           查段表 → 段基址 + 段长度检查
              ↓
物理地址 = 段基址 + 段内地址

【段页式地址变换】
逻辑地址 → [段号 | 页号 | 页内地址]
              ↓        ↓
           查段表 → 查页表
              ↓        ↓
物理地址 = [块号 | 页内地址]
```

### 页面置换算法核心对比

```
OPT：理论上最优，不可实现，用于评价基准
FIFO：简单但性能差，会产生Belady异常
LRU：性能接近OPT，硬件支持复杂
Clock：LRU近似，实现简单高效
改进Clock：考虑修改位，减少I/O开销
LFU：按使用频率，可能保留过时热门页
```

### 关键公式汇总

| 公式 | 说明 |
|------|------|
| `物理地址 = 块号 × 页大小 + 页内地址` | 分页地址变换 |
| `P = INT(A/L), d = A MOD L` | 页号和页内地址计算 |
| `有效访问时间 = (1-p)×T1 + p×T2` | 考虑缺页的有效时间 |
| `两级页表空间 = 每块项数^层数 × 块大小` | 多级页表逻辑空间 |

# 第一部分：第5章 设备管理

## 5.1 I/O系统

### 5.1.1 I/O设备的类型

**按传输速率分类：**

| 类型 | 速率范围 | 示例 |
|------|----------|------|
| **低速设备** | < 1 KB/s | 键盘、鼠标 |
| **中速设备** | 1~100 KB/s | 打印机 |
| **高速设备** | > 100 KB/s | 磁盘机 |

**按信息交换单位分类：**
- **块设备**：以数据块为单位存取，如磁盘。特点是传输速率高、可寻址
- **字符设备**：用于无结构数据I/O，如键盘、打印机。特点是传输速率低、不可寻址

**按共享属性分类：**
- **独占设备**：一段时间只允许一个用户使用（如打印机）
- **共享设备**：可同时允许多个用户使用（如磁盘）
- **虚拟设备**：通过虚拟技术将一台物理设备变换为多个逻辑设备

**按外部特征分类：**
- 存储设备（磁盘、磁带）、输入设备（键盘）、输出设备（打印机）

### 5.1.2 设备控制器

**设备控制器**是CPU与I/O设备之间的接口：
- 用地址（数据寄存器、状态寄存器、控制寄存器）进行识别
- 一个设备控制器可挂接多个设备

**设备控制器的基本功能：**
1. 接收和识别命令
2. 标识和报告设备状态
3. 数据交换
4. 地址识别
5. 数据缓冲
6. 差错控制

**设备与控制器之间的接口信号：**
- 数据信号线、控制信号线、状态信号线

### 5.1.3 I/O通道

**I/O通道**是一种特殊的处理机，具有执行I/O指令的能力，通过执行通道程序控制I/O操作。

与一般处理机的区别：
1. 指令类型单一（局限于I/O操作相关指令）
2. 没有自己的内存，通道程序存放在主机内存中（与CPU共享内存）

**通道类型：**

| 类型 | 特点 | 适用场景 |
|------|------|----------|
| **字节多路通道** | 分时为多个设备服务，每次传送一个字节 | 连接大量低速设备 |
| **数组选择通道** | 只有一个分配型子通道，一次控制一台设备 | 高速设备，但利用率低 |
| **数组多路通道** | 多个非分配型子通道，传输速率高且通道利用率高 | 多台高中速设备 |

**"瓶颈"问题：** 单通路I/O系统中通道数量少会成为系统瓶颈，可通过**多通路I/O系统**解决。

### 5.1.4 总线系统

- **ISA总线**：16位，最高传输速率16 MB/s
- **EISA总线**：32位，传输速率32 MB/s
- **PCI总线**：局部总线，高速外设连接

## 5.2 I/O控制方式

### 5.2.1 程序直接控制方式（程序I/O方式）

- CPU不断轮询I/O设备状态，处于忙等待循环
- 由于CPU高速性与I/O设备低速性矛盾，CPU绝大部分时间处于等待状态
- **缺点**：CPU利用率极低

### 5.2.2 中断驱动I/O控制方式

- I/O设备输入每个数据的过程中无须CPU干预，CPU与I/O设备并行工作
- 仅当输完一个数据时，才需CPU做中断处理
- **优点**：CPU和I/O设备都处于忙碌状态，提高了系统资源利用率
- 例如：终端输入一个字符约100ms，送入缓冲区小于0.1ms，CPU利用率提高约1000倍

### 5.2.3 DMA（直接存储器访问）控制方式

**特点：**
1. 数据传输基本单位是**数据块**（每次至少一个数据块）
2. 数据从设备直接送入内存（或相反），不经CPU
3. 仅在传送开始和结束时需CPU干预

**DMA控制器的四类寄存器：**

| 寄存器 | 缩写 | 功能 |
|--------|------|------|
| 命令/状态寄存器 | CR | 接收I/O命令、控制信息和设备状态 |
| 内存地址寄存器 | MAR | 存放内存起始地址（输入为目标地址，输出为源地址） |
| 数据寄存器 | DR | 暂存传输数据 |
| 数据计数器 | DC | 存放本次要读写的字（节）数 |

**DMA数据传送三阶段：**
1. **预处理阶段**：CPU将内存起始地址、传送字节数送入DMAC，启动设备
2. **数据传送阶段**：DMAC窃取CPU工作周期，执行数据传送（地址修改、计数）
3. **后处理阶段**：DMAC通过中断请求线发中断信号，CPU做后续处理

### 5.2.4 I/O通道控制方式

- DMA方式的发展，进一步减少CPU干预
- 把对一个数据块的读写干预，减少为对**一组数据块**的读写及控制管理干预
- 实现**CPU、通道和I/O设备三者并行**操作
- CPU只需发送一条I/O指令，给出通道程序首址和要访问的设备

**通道指令**：包含操作码、内存地址、计数、通道程序结束位P、记录结束标志R

**通道工作原理：**
1. CPU将通道程序首地址放入**通道地址字(CAW)**
2. 执行"启动I/O"指令
3. 通道从CAW取首地址，取出第一条指令放入**通道命令字(CCW)**
4. 通道执行通道程序，与CPU并行完成数据传输
5. 完成后向CPU发I/O中断，CPU从**通道状态字(CSW)**获取状态信息

> **复习要点**：四种I/O控制方式对比
> | 方式 | CPU干预频率 | 数据单位 | 并行性 |
> |------|------------|----------|--------|
> | 程序I/O | 每个字符 | 字/字节 | 无 |
> | 中断驱动 | 每个数据 | 字/字节 | CPU与设备 |
> | DMA | 每个数据块 | 数据块 | CPU与设备 |
> | I/O通道 | 一组数据块 | 数据块组 | CPU、通道、设备三者 |

## 5.3 缓冲管理

### 5.3.1 缓冲的引入

**引入缓冲的原因：**
1. 缓和CPU与I/O设备间速度不匹配的矛盾
2. 减少对CPU的中断频率，放宽中断响应时间限制
3. 提高CPU和I/O设备之间的并行性

### 5.3.2 单缓冲

- 每当用户进程发出I/O请求时，系统分配一个缓冲区
- 处理一个数据时间：`T + M + C`（T=输入时间，M=传送时间，C=计算时间）
- 连续处理时：`max(C, T) + M`

### 5.3.3 双缓冲

- 设置两个缓冲区A和B，交替使用
- 输入与传送/计算可并行
- 连续处理一个数据时间：`max(T, M + C)`
- 当输入和传送速度相匹配时效果最佳

### 5.3.4 循环缓冲

- 多个缓冲区组织成**循环队列**
- 输入进程不断向空缓冲区R输入数据
- 计算进程从满缓冲区G中提取数据
- 需要同步机制防止指针追赶冲突

### 5.3.5 缓冲池（Buffer Pool）

**三个队列：**
- **emq**：空缓冲区队列
- **inq**：输入缓冲区队列
- **outq**：输出缓冲区队列

**四种工作缓冲区：**
1. **hin**：收容输入数据的工作缓冲区
2. **sin**：提取输入数据的工作缓冲区
3. **hout**：收容输出数据的工作缓冲区
4. **sout**：提取输出数据的工作缓冲区

**四种工作方式：**
1. 收容输入：设备 → hin → inq
2. 提取输入：inq → sin → 用户区
3. 收容输出：用户区 → hout → outq
4. 提取输出：outq → sout → 设备

## 5.4 设备分配

### 5.4.1 设备分配中的数据结构

| 数据结构 | 缩写 | 作用 |
|----------|------|------|
| 设备控制表 | DCT | 每个设备一张，记录设备状态 |
| 控制器控制表 | COCT | 每个控制器一张 |
| 通道控制表 | CHCT | 每个通道一张 |
| 系统设备表 | SDT | 整个系统一张，记录所有设备 |

### 5.4.2 设备分配应考虑的因素

**1. 设备固有属性：**
- 独享设备：分配后独占使用
- 共享设备：可同时分配给多个进程
- 虚拟设备：通过SPOOLing技术实现

**2. 设备分配算法：**
- **先来先服务(FCFS)**
- **优先级高者优先**

**3. 安全性：**
- **安全分配方式**：进程发出I/O请求后进入阻塞，直到I/O完成才被唤醒（不会死锁，但进程缓慢）
- **不安全分配方式**：进程发出I/O请求后继续运行（可能死锁，但进程推进快）

### 5.4.3 设备独立性

**设备独立性**：应用程序独立于具体使用的物理设备。

- 用户采用**逻辑设备名**使用设备
- 系统实现逻辑设备到物理设备的映射

**好处：**
1. 设备分配灵活
2. 易于实现I/O重定向

**逻辑设备表(LUT)**：实现逻辑设备名到物理设备名的映射

### 5.4.4 独占设备的分配程序

**分配步骤：**
1. 分配设备（查SDT → DCT）
2. 分配控制器（查COCT）
3. 分配通道（查CHCT）

**改进：**
- 增加设备独立性（用逻辑设备名请求I/O）
- 考虑多通路情况（查找所有控制器和通道）

### 5.4.5 SPOOLing技术（假脱机操作）

**SPOOLing(Simultaneous Peripheral Operations On-Line)**：由系统中两个专门负责I/O的进程模拟I/O外围机功能，实现假脱机输入/输出。

**系统组成：**
1. **输入井和输出井**：磁盘上开辟的两个大存储空间
   - 输入井：收容I/O设备输入的数据
   - 输出井：收容用户进程的输出数据
2. **输入缓冲区和输出缓冲区**：内存中开辟的暂存区
3. **输入进程SPi和输出进程SPo**：分别模拟脱机输入/输出时的外围控制机

**系统特点：**
- 提高I/O速度
- 将**独占设备改造为共享设备**
- 实现**虚拟设备**功能

**SPOOLing使一台打印机虚拟成多台打印机的原理：**
- 用户打印请求将数据输出到输出井
- 打印请求登记后排到打印队列
- 打印机空闲时取队列首请求，从输出井传送到内存缓冲区进行打印

### 5.5.1 设备驱动程序的功能和特点

**功能：**
- 接收独立性软件的抽象要求（read/write），转换为具体要求

### 5.5.2 设备驱动程序的处理过程

1. 将抽象要求转换为具体要求
2. 检查I/O请求的合法性
3. 读出和检查设备状态
4. 传送必要的参数
5. 工作方式设置
6. 启动I/O设备

## 5.6 磁盘存储器管理

### 5.6.1 磁盘结构

**磁盘数据组织：**
- 磁盘由多个盘片组成，每盘片分两面
- 每盘面分为若干个**磁道（track）**，每磁道分为若干个**扇区（sector）**
- **柱面（cylinder）**：处于相同磁道位置但在不同盘面上的磁道集合
- 内磁道数据密度大于外磁道

**磁盘容量计算：**
```
硬盘容量 = 每扇区字节数(512B) x 扇区数/道 x 柱面数 x 磁头数
```

**磁盘类型：**
- **固定头磁盘**：每个磁道上都有一个磁头
- **移动头磁盘**：一个盘面仅有一个磁头

### 5.6.2 磁盘访问时间

**总访问时间：** `Ta = Ts + Tr + Tt`

| 时间类型 | 符号 | 说明 | 公式 |
|----------|------|------|------|
| **寻道时间** | Ts | 磁头移至指定磁道 | `Ts = m x n + s`（s:启动时间，m:每道时间，n:距离） |
| **旋转延迟** | Tr | 磁头移至指定扇区 | `Tr = 1/(2r)`（r:转速，约5.55ms@90r/s） |
| **传输时间** | Tt | 数据读出/写入 | `Tt = b/(rN)`（b:字节数，N:单磁道字节数） |

> **复习要点**：寻道时间是影响磁盘性能的主要因素。将同一文件的数据块集中存放（同一柱面）可降低访问时间。

### 5.6.3 磁盘调度算法

> **复习要点**：磁盘调度是期末重点！重点掌握各算法的原理、优缺点和计算。

**1. 先来先服务（FCFS）**
- 按请求访问磁盘的先后次序调度
- **优点**：简单公平，每个请求依次处理
- **缺点**：平均寻道距离大，性能差

**2. 最短寻道时间优先（SSTF）**
- 优先访问与当前磁头位置最近的磁道
- **优点**：平均寻道时间较短
- **缺点**：可能导致某些进程"饥饿"（磁臂粘着）

**3. 扫描算法（SCAN / 电梯算法）**
- 按磁头移动方向及距离最短两个原则
- 磁头来回扫描（如：向外→向内→向外→...）
- **优点**：不会出现饥饿，平均寻道时间较短
- **缺点**：与磁头近但在反方向的磁道等待时间长

**4. 循环扫描算法（C-SCAN）**
- 到达最外磁道后，返回最小磁道重新开始
- 只单向服务（如只向外扫描，到达末端后直接回起点）
- **优点**：不会出现饥饿，平均寻道时间较短，**最长等待时间为SCAN的一半**
- **缺点**：仍有磁臂粘着问题

**5. LOOK算法**
- SCAN的改进：只走到最远的请求磁道就折返
- C-SCAN对应的改进为C-LOOK

**6. N-Step-SCAN算法**
- 将请求分成若干个长度为N的子队列
- 子队列间按FCFS，子队列内按SCAN
- N=1时退化为FCFS，N=∞时退化为SCAN

**7. FSCAN算法**
- 将请求分成两个队列：当前申请队列和扫描期间申请队列
- 用SCAN处理当前队列，扫描期间的新请求放入第二个队列

### 5.6.4 提高磁盘I/O速度的方法

**1. 磁盘高速缓存**
- 利用内存空间暂存从磁盘读出的盘块信息
- 形式：单独存储空间（固定大小）或利用未用内存（动态变化）
- **置换算法**：LRU、NRU(Clock)、FIFO
- **周期性写回磁盘**（如每30秒），防止数据丢失

**2. 数据交付方式**
- **数据交付**：将数据传送到进程指定内存区
- **指针交付**：将数据位置交付给进程

**3. 其他方法**
- **提前读**：依据局部性原理，将邻近盘块提前读入内存
- **延迟写**：被修改的页面不立即写回，挂在空闲缓冲区链尾
- **优化物理块分布**：将相邻盘块分配给同一文件，以簇为单位分配
- **虚拟盘（RAM盘）**：利用内存仿真磁盘

### 5.6.5 RAID（廉价磁盘冗余阵列）

**并行交叉存取**：将一簇中不同盘块内容分别放在不同磁盘的相同位置。

| RAID级别 | 特点 | 效率 | 缺点 |
|----------|------|------|------|
| **RAID 0** | 仅并行交叉存取 | 100% | 无容错 |
| **RAID 1** | 并行+磁盘镜像 | 50% | 效率低 |
| **RAID 3** | 并行+专用校验盘 | (N-1)/N | 校验盘成瓶颈 |
| **RAID 5** | 并行+校验信息散布在所有盘 | (N-1)/N | 写操作复杂 |

**RAID 5优点**：可靠性高、磁盘I/O速度高（N-1倍）、性能/价格比高。

# 第二部分：第6章 文件管理

## 6.1 文件和文件系统

### 6.1.1 文件的概念

**文件**：具有文件名的一组相关信息的集合。

- 文件具有**可保存性**：便于信息长期保存
- 用户**按名存取**：不需知道文件在磁盘的具体位置

### 6.1.2 文件类型

**有结构文件（记录式文件）**：
- 由**记录**组成，记录是一组相关数据项的集合
- 数据项：描述对象某种属性的字符集

**无结构文件（流式文件）**：
- 直接由字符/字节组成
- Unix、DOS、Windows等都采用流式文件

### 6.1.3 文件系统的功能

1. 文件存储空间的分配与回收（磁盘管理）
2. 文件名到文件空间的映射
3. 提供文件共享、保护与保密措施
4. 实现用户要求的各种文件操作

**文件系统接口：**
- **命令接口**：dir/ls等
- **程序接口**：open, read, write, close等系统调用

## 6.2 文件的逻辑结构

### 6.2.1 文件逻辑结构的类型

| 类型 | 子类型 | 特点 |
|------|--------|------|
| **有结构文件** | 顺序文件 | 记录按顺序排列 |
| | 索引文件 | 为可变长记录建立索引表 |
| | 索引顺序文件 | 分组索引，兼顾效率和空间 |
| **无结构文件** | 流式文件 | 直接由字符/字节组成 |

### 6.2.2 顺序文件

**排序方式：**
- **串结构**：各记录顺序与关键字无关
- **顺序结构**：记录按关键字排列

**读写方式：**
- 定长记录：直接定位
- 变长记录：逐个顺序读写

**缺点：**
- 对单个记录读写困难（尤其变长记录）
- 增加或删除记录困难

### 6.2.3 索引文件

- 为可变长记录文件建立一张**索引表**
- 索引表按记录键排序，本身构成定长记录的顺序文件
- 支持高效直接存取

### 6.2.4 索引顺序文件

- 将记录分为若干组
- 为每组第一个记录建立索引项
- 可建立多级索引（二级、三级等）

**查找效率对比（100万条记录）：**
- 顺序文件：平均50万次
- 索引文件：需100万个索引空间
- 索引顺序文件（二级索引）：约150次

## 6.3 外存分配方式

### 6.3.1 连续分配

- 为文件分配一组相邻接的盘块
- **优点**：顺序访问容易，一次大批量访问速度快，可通过计算直接定位：`J = ADDR + I`
- **缺点**：要求连续外存空间（存在碎片），必须事先知道文件长度

### 6.3.2 链接分配

**隐式链接：**
- 链接指针放在盘块数据区中
- **优点**：无碎片，不需连续空间，不需事先知道文件长度
- **缺点**：只适合顺序访问（随机访问低效），可靠性差（指针出错则链断开）

**显式链接（FAT）：**
- 链接指针存放在**文件分配表(FAT)**中
- **优点**：FAT可调入内存，查找速度快，支持随机存取
- **缺点**：FAT占用外存和内存空间

### 6.3.3 索引分配

**单级索引：**
- 每个文件分配一个索引块，存放盘块号
- **优点**：方便直接存取
- **缺点**：索引块可能装不满，大文件效率仍低

**两级索引：**
- 索引块中存放下一级索引块的盘块号
- 最大文件空间：256 x 256 = 64K盘块 = 64MB（盘块1KB，盘块号4B）

**混合索引（UNIX方式）：**

| 索引类型 | 说明 | 最大文件大小（4KB块） |
|----------|------|----------------------|
| 直接地址（10项） | 直接存放盘块号 | 40 KB |
| 一次间接地址 | 单级索引 | 40KB + 4MB = 4.04MB |
| 二次间接地址 | 二级索引 | 4.04MB + 4GB |
| 三次间接地址 | 三级索引 | 4GB + 4TB |

## 6.4 文件目录

> **复习要点**：文件目录结构是期末重点！

### 6.4.1 文件控制块（FCB）

**FCB**是描述和控制文件的数据结构，与文件一一对应，是目录项的组成部分。

**FCB包含信息：**

| 类别 | 内容 |
|------|------|
| **基本信息** | 文件名、物理位置、逻辑结构、物理结构 |
| **存取控制信息** | 文件主权限、核准用户权限、一般用户权限 |
| **使用信息** | 建立时间、最后修改/访问时间、当前使用信息 |

### 6.4.2 索引结点（Inode）

**引入原因：**
- FCB太大（64B），单个盘块存储FCB数量少
- 查找文件时需要访问多个盘块

**改进方法：**
- 将文件名与FCB其他描述信息分开存放
- 文件名+索引结点指针组成目录项（16B：14B文件名+2B指针）
- FCB其他描述信息组合成**索引结点**数据结构

**效果：** 1KB盘块可存放64个目录项，查找效率提高4倍。

### 6.4.3 目录结构

**1. 单级目录**
- 整个系统一张目录表，每个文件一个FCB
- **优点**：简单
- **缺点**：查找慢、不允许重名、不便于共享

**2. 两级目录**
- 主文件目录(MFD) + 用户文件目录(UFD)
- **优点**：提高检索速度、允许不同用户目录同名、支持共享
- **缺点**：用户不能建立子目录

**3. 树型目录结构**
- 具有三级或三级以上的目录结构
- 允许在任何子目录中再创建新子目录
- **路径名**：从根目录到文件经过的所有目录及文件名用"/"连接
- **当前目录（工作目录）**：用户当前操作的目录

**4. 图形目录结构**
- 在树型基础上增加共享链接，形成有向无环图

### 6.4.4 目录查询技术

1. 按名查询文件FCB或索引结点
2. 依据物理地址（盘块号）换算物理位置
3. 启动磁盘驱动程序读取文件

## 6.5 文件存储空间的管理

### 6.5.1 空闲表法

- 为外存上所有空闲区建立一张空闲表（连续分配方式）
- **分配算法**：首次适应、循环首次适应、最佳适应、最坏适应
- **回收**：对邻接盘区进行拼接
- 主要用于对换区分配

### 6.5.2 空闲链表法

**空闲盘块链表：**
- 所有空闲盘块连成链表
- **优点**：分配回收操作简单
- **缺点**：链表可能很长，指针出错导致空闲盘块丢失

**空闲盘区链表：**
- 空闲盘区（多个连续盘块）连成链表
- **优点**：链表较短
- **缺点**：分配回收操作较复杂

### 6.5.3 位示图法

- 用二进制的一位表示一个盘块的使用情况（0=空闲，1=已分配）
- **盘块分配**：
  1. 扫描位示图找到值为0的位
  2. 转换为盘块号：`n = i x 8 + j`（第i字节，第j位）
  3. 修改位示图对应位为1
- **盘块回收**：
  1. 盘块号转位置：`i = n / 8, j = n mod 8`
  2. 修改对应位为0
- **优点**：占用空间小可调入内存，分配回收操作简单有效

### 6.5.4 成组链接法

- 将空闲盘块每N个（如100个）组成一组
- 空闲盘块组连成链表
- **空闲盘块号栈**：存放当前可用的一组空闲盘块号

**盘块分配：**
1. 若`s.free = 1`，将栈底盘块内容调入空闲盘块号栈，再分配该盘块
2. 否则，直接分配栈底盘块，`s.free`递减

**盘块回收：**
1. 若`s.free = N`，将栈内容写入新释放盘块，`s.free = 1`，该盘块作为栈底
2. 否则，`s.free`递增，该盘块入栈

## 6.6 文件共享与文件保护

### 6.6.1 基于索引结点的共享方式（硬链接）

- 目录项只包含文件名及指向索引结点的指针
- 共享同一文件的不同目录项，文件名不同但指向同一索引结点
- 设置**链接计数器**，指示共享目录项数目
- **优点**：任意用户对文件的修改其他共享者都可见
- **缺点**：文件主删除文件后若有其他用户共享，文件仍占用空间

### 6.6.2 基于符号链的共享方式（软链接）

- 符号链是指向文件所在目录项位置的指针（路径名或IP+路径）
- 文件主可完全删除文件，共享者访问时出错误
- **优点**：可共享网络上任何地点的文件
- **缺点**：访问速度较慢（沿链查找）

### 6.6.3 文件保护

**访问控制：**
- 文件主权限（只读、读写、执行）
- 核准用户权限
- 一般用户权限

**磁盘容错技术：**

| 级别 | 名称 | 技术 |
|------|------|------|
| SFT-I | 第一级容错 | 双份目录/FAT、热修复重定向、写后读校验 |
| SFT-II | 第二级容错 | 磁盘镜像、磁盘双工 |

### 6.6.4 数据一致性控制

**事务**：用于访问和修改数据项的程序单位，具有**原子性**。

**事务记录**：存放在稳定存储器，包含事务名、数据项名、旧值、新值。

**恢复算法**：
- **Undo(Ti)**：将事务Ti修改的数据恢复为旧值
- **Redo(Ti)**：将事务Ti修改的数据置为新值

**并发控制**：互斥锁、信号量机制实现顺序性。

## 6.7 磁盘调度算法练习题

**例题：** 假设活动头磁盘编号0-189，当前磁头在123道，刚完成113道。请求序列：76, 125, 81, 160, 83, 140, 92, 180, 120。

**SSTF算法**：每次选择距离当前磁头最近的请求
**SCAN算法**：沿当前方向（向内）移动到端点，再反向
**C-SCAN算法**：沿一个方向（向内）移动到端点，直接回到另一端再同向

## 2. 操作系统接口

### 2.1 进程与内存

- 进程由**用户空间内存**（指令、数据、堆栈）和**内核私有进程状态**组成
- xv6进程共享CPU，透明切换
- `fork()`：创建子进程，内存内容与父进程相同
- `exec()`：用新内存映像替换当前进程内存
- `sbrk(n)`：进程运行时增加内存

### 2.2 文件描述符

- 文件描述符是**小整数**，代表内核管理对象
- 每个进程有私有文件描述符空间（从0开始）
- **标准约定**：0=标准输入，1=标准输出，2=标准错误
- `read/write`操作有偏移量，顺序推进
- `fork`复制的文件描述符共享偏移量，`dup`也共享偏移量

**I/O重定向原理：**
```c
if (fork() == 0) {
    close(0);                    // 释放标准输入
    open("input.txt", O_RDONLY); // 获得fd=0
    exec("cat", argv);           // cat从input.txt读取
}
```

### 2.3 管道

- 管道是**小的内核缓冲区**，一对文件描述符（一读一写）
- 提供进程间通信方式
- 管道的四个优势：自动清理、可传任意长数据流、可分段并行执行、阻塞读写更高效

### 2.4 文件系统

- **数据文件**：拥有字节数组
- **目录**：拥有对数据文件和其他目录的命名引用
- 目录形成树，从**根目录**开始
- **inode**：底层文件，保存metadata（类型、长度、磁盘位置、链接数）
- `link()`创建硬链接（同一inode多个名称），`unlink()`删除文件名

## 3. 操作系统组织

### 3.1 宏内核 vs 微内核

| 特性 | 宏内核 | 微内核 |
|------|--------|--------|
| 设计 | 整个操作系统在内核中运行 | 尽量减少内核代码，大部分在用户级服务器 |
| 权限 | 全硬件权限 | 仅基本功能在内核 |
| 优点 | 方便，各部分易合作 | 内核简单，bug影响小 |
| 缺点 | bug致命（内核崩溃则全崩） | 进程间通信复杂 |
| 代表 | xv6、Linux | L4、Minix、QNX |

### 3.2 用户模式与监督者模式

- **RISC-V三种模式**：机器模式、监督者(supervisor)模式、用户模式
- **用户模式**：只能执行用户指令（如数字相加），运行用户空间程序
- **监督者模式**：可执行特权指令（开关中断、读写页表等），运行内核
- **ecall指令**：从用户模式切换到监督者模式，进入内核
- **sret指令**：从内核返回用户空间

### 3.3 进程结构

**进程抽象实现隔离的机制：**
- 用户/监督模式标志
- 地址空间
- 线程的时间分割

**进程内存布局（用户空间）：**
```
低地址 ┌──────────┐
       │  指令    │
       ├──────────┤
       │ 全局变量 │
       ├──────────┤
       │   栈     │ ← 向下增长
       ├──────────┤
       │   堆     │ ← 向上增长（sbrk扩展）
       │          │
       ├──────────┤
       │trapframe │ ← 用户页表映射
       ├──────────┤
高地址 │trampoline│ ← 用户/内核页表同地址映射
       └──────────┘
       MAXVA = 0x3fffffffff
```

**进程关键状态（proc结构体）：**
- `p->pagetable`：进程页表
- `p->kstack`：内核栈
- `p->state`：进程状态（CREATED/READY/RUNNING/WAITING/EXITED）

**进程有两个栈：**
- **用户栈**：执行用户指令时使用
- **内核栈**：进入内核时使用（系统调用或中断）

## 4. 页表

### 4.1 Sv39 RISC-V分页机制

- 使用64位虚拟地址的**低39位**，高25位未使用
- 页表逻辑上是`2^27`（约1.34亿）个**页表项(PTE)**的数组
- 每个PTE包含44位**物理页号(PPN)**和标志位
- **页大小**：4096字节(2^12)

**PTE标志位：**

| 标志位 | 含义 |
|--------|------|
| `PTE_V` | PTE是否有效 |
| `PTE_R` | 是否可读 |
| `PTE_W` | 是否可写 |
| `PTE_X` | 是否可执行 |
| `PTE_U` | 用户模式是否可访问 |

### 4.2 三级页表

页表以**三层树**形式存储：
1. **根页表页**（4096字节，512个PTE）
2. **二级页表页**（512个PTE）
3. **三级页表页**（512个PTE指向物理页）

虚拟地址39位划分：
- 高9位：选择根页表页中的PTE
- 中9位：选择二级页表页中的PTE
- 低9位：选择三级页表页中的PTE
- 低12位：页内偏移

### 4.3 地址空间

**内核地址空间布局：**
- **直接映射**：虚拟地址 = 物理地址（KERNBASE=0x80000000到PHYSTOP=0x86400000）
- **trampoline页**：映射在虚拟地址空间顶端，用户/内核页表共享
- **内核栈页**：映射在高地址处，下方有未映射的守护页（防止栈溢出）

**关键代码文件**：`kernel/vm.c`
- `walk()`：模拟分页硬件查找虚拟地址的PTE
- `mappages()`：将虚拟地址映射到物理地址
- `kvm*`函数：操作内核页表
- `uvm*`函数：操作用户页表

### 4.4 物理内存分配

- 使用**内核地址结束到PHYSTOP**之间的物理内存进行运行时分配
- 每次分配/释放**4096字节**的整页
- 通过**空闲页链表**记录空闲页

**关键代码**：`kernel/kalloc.c`
- `kalloc()`：从空闲链表头部pop一页
- `kfree()`：将页push到空闲链表
- 空闲页的`run`结构体存储在空闲页本身

## 5. Trap与系统调用

### 5.1 Trap类型

三种事件导致CPU转移控制权：
1. **系统调用**：用户程序执行`ecall`指令
2. **异常**：指令做了非法操作（除零、无效地址等）
3. **设备中断**：设备发出注意信号

### 5.2 RISC-V Trap机制

**关键控制寄存器：**

| 寄存器 | 作用 |
|--------|------|
| `stvec` | trap处理程序地址 |
| `sepc` | trap发生时保存的程序计数器 |
| `scause` | trap原因编号 |
| `sscratch` | 方便trap恢复/保存用户上下文的值 |
| `sstatus` | SIE位控制中断使能，SPP位表示来源模式 |

**Trap硬件执行步骤：**
1. 如果是设备中断且SIE=0，不处理
2. 清除SIE禁用中断
3. 复制PC到sepc
4. 保存当前模式到SPP位
5. 在scause设置原因
6. 转换为监督者模式
7. 将stvec复制到PC

### 5.3 用户空间Trap处理

**路径**：`uservec` → `usertrap` → 处理 → `usertrapret` → `userret`

**trampoline页机制：**
- 用户页表不映射内核，但trap时需要切换页表
- xv6在内核页表和每个用户页表的**相同虚拟地址**上映射trampoline页
- `uservec`（在trampoline页中）保存寄存器到trapframe，切换satp到内核页表

**trapframe**：保存所有用户寄存器，包含内核栈指针、hartid、usertrap地址、内核页表地址。

### 5.4 系统调用实现

1. 用户代码将参数放寄存器`a0/a1`，系统调用号放`a7`
2. `ecall`进入内核
3. `syscall()`从trapframe的`a7`获取系统调用号
4. 在`syscalls`数组中查找并执行对应函数
5. 返回值记录在`p->trapframe->a0`

## 6. 中断与设备驱动

### 6.1 驱动程序结构

驱动在两个**上下文**执行代码：
- **上半部分(top half)**：在进程内核线程中运行（通过系统调用如read/write）
- **下半部分(bottom half)**：在中断时执行（中断处理程序）

### 6.2 UART控制台驱动

- 通过**UART串行端口硬件**接受输入字符
- UART硬件是一组**内存映射控制寄存器**
- `consoleinit()`配置UART：接收输入时产生接收中断，完成发送时产生传输完成中断

**控制台输入流程：**
1. 用户按键 → UART硬件产生中断
2. trap处理 → `devintr()` → `uartintr()`
3. `uartintr()`读取字符 → `consoleintr()`累积到`cons.buf`
4. 新行到达 → `wakeup`唤醒`consoleread`
5. `consoleread`将输入复制到用户空间

**控制台输出流程：**
1. `uartputc`追加字符到输出缓冲区
2. `uartstart`启动设备发送
3. 发送完成产生中断 → `uartintr`调用`uartstart`发送下一个字符

### 6.3 并发问题

驱动中的三个并发风险：
1. 两个进程同时调用`consoleread`
2. 硬件在执行`consoleread`时抛出中断
3. 硬件在`consoleread`执行时向另一个CPU抛中断

**解决**：使用锁保护驱动数据结构。

### 6.4 定时器中断

- xv6使用定时器中断维护时钟、实现进程切换
- RISC-V要求在**机器模式**下处理定时器中断
- `timervec`：机器模式定时器中断向量
- 策略：处理程序要求RISC-V引发**软件中断**并立即返回，再用普通trap机制处理

## 7. 锁

### 7.1 竞争条件

**竞争条件**：同时访问一个内存位置，且至少有一次是写操作。

**丢失更新示例**（链表push）：
```c
l->next = list;   // 两个CPU同时执行
list = l;         // 第二次赋值覆盖第一次
```

### 7.2 自旋锁

**xv6自旋锁结构**：`spinlock`，核心字段`locked`（0=可获得，非0=被持有）

**原子操作**：RISC-V的`amoswap`指令，原子地交换寄存器和内存值。

```c
void acquire(struct spinlock *lk) {
    while(__sync_lock_test_and_set(&lk->locked, 1) != 0)
        ; // 自旋等待
    lk->cpu = mycpu(); // 记录获取锁的CPU
}

void release(struct spinlock *lk) {
    lk->cpu = 0;
    __sync_lock_release(&lk->locked);
}
```

**使用原则**：
1. 一个CPU写入时另一CPU读写，必须用锁
2. 涉及多个内存位置的invariant，用同一个锁保护
3. 不要过度使用锁（降低并行性）

### 7.3 死锁避免

**死锁条件**：多个代码路径以不同顺序获取多个锁。

**避免方法**：所有代码路径以**相同顺序**获取锁。

**xv6锁顺序示例**：
- `cons.lock`必须在任何进程锁之前获取
- 文件系统：目录锁 → 新文件inode锁 → 磁盘块缓冲区锁 → 磁盘驱动锁 → 进程锁

### 7.4 锁与中断

**危险场景**：线程持有锁L时被中断，中断处理程序也试图获取L → 死锁。

**xv6解决方案**：获取任何锁时，**禁用该CPU上的中断**。

### 7.5 睡眠锁

**自旋锁缺点**：
- 长时间等待浪费CPU
- 持有自旋锁时不能让出CPU

**睡眠锁**：
- `acquiresleep`等待时让出CPU
- 使用spinlock保护locked字段
- `sleep`调用原子性地让出CPU并释放spinlock
- **限制**：不能在中断处理程序中使用，不能在spinlock临界区内使用

**适用场景**：
- 自旋锁：短临界区
- 睡眠锁：长时间操作（如磁盘I/O）

## 8. 调度

### 8.1 复用实现

xv6在两种情况下切换CPU：
1. `sleep/wakeup`：进程等待设备/管道I/O、子进程退出
2. **周期强制切换**：应对长时间计算进程（定时器中断）

### 8.2 上下文切换

**切换步骤（用户进程A → 用户进程B）：**
1. 用户内核转换（系统调用或中断）→ A的内核线程
2. `swtch`保存A的上下文 → CPU调度器线程
3. `swtch`从调度器恢复B的上下文 → B的内核线程
4. trap返回到B的用户空间

**`swtch(struct context *old, struct context *new)`**：
- 保存callee-saved寄存器到old
- 从new恢复寄存器
- 返回时在新线程的栈上执行

**调度器**：每个CPU一个专门的调度器线程（保存寄存器和栈），运行`scheduler()`函数。

### 8.3 调度器实现

```c
void scheduler() {
    for(;;) {
        // 找可运行进程
        for(p = proc; p < &proc[NPROC]; p++) {
            if(p->state == RUNNABLE) {
                p->state = RUNNING;
                c->proc = p;
                swtch(&c->context, &p->context);
                c->proc = 0;
            }
        }
    }
}
```

- 简单**轮询调度(round robin)**
- 找到`RUNNABLE`进程 → 设为`RUNNING` → `swtch`运行它

### 8.4 Sleep与Wakeup

**丢失唤醒问题**：
```c
// 错误版本：P在检查条件和sleep之间，V可能执行wakeup
while(s->count == 0)
    sleep(chan);  // V在此间隙执行，wakeup丢失
```

**正确方案**：`sleep(chan, lock)` - 调用者传递条件锁
- sleep原子性地：标记SLEEPING → 释放条件锁 → 让出CPU
- wakeup在持有条件锁时调用，确保不会错过sleep的进程

### 8.5 管道实现

**管道结构**：
```c
struct pipe {
    struct spinlock lock;
    char buf[PIPESIZE];
    uint nread;   // 已读取字节数
    uint nwrite;  // 已写入字节数
};
```

- 缓冲区环形：`buf[PIPESIZE-1]`之后是`buf[0]`
- `nwrite == nread`：空缓冲区
- `nwrite == nread + PIPESIZE`：满缓冲区
- 使用**两个sleep chan**（`&pi->nread`和`&pi->nwrite`）提高效率

### 8.6 Wait、Exit与Kill

**Exit：**
- 记录退出状态，释放资源
- 子进程交给init进程
- 唤醒父进程，标记为ZOMBIE

**Wait：**
- 扫描进程表找ZOMBIE子进程
- 释放子进程资源，返回子进程ID
- 无子进程退出则sleep等待

**Kill：**
- 设置`p->killed`标志
- 如进程在sleep则wakeup它
- 进程最终进入/离开内核时检查标志并调用exit

## 9. 文件系统

### 9.1 文件系统层次结构

xv6文件系统分为**七层**：

| 层次 | 功能 | 关键文件 |
|------|------|----------|
| **文件描述符层** | 抽象Unix资源（文件、管道、设备） | `kernel/file.c` |
| **路径名层** | 层次化路径名解析 | `kernel/fs.c` |
| **目录层** | 目录项操作 | `kernel/fs.c` |
| **Inode层** | 文件表示为inode | `kernel/fs.c` |
| **日志层** | 崩溃恢复，事务原子性 | `kernel/log.c` |
| **Buffer Cache层** | 缓存磁盘块，同步访问 | `kernel/bio.c` |
| **磁盘层** | 在virtio磁盘上读写块 | `kernel/virtio_disk.c` |

**磁盘布局：**
```
块0: boot sector
块1: superblock（元数据）
块2~: 日志
日志后: inodes
inodes后: 位图块
其余: 数据块
```

### 9.2 Buffer Cache

**两个功能**：
1. **同步访问**：每个磁盘块在内存中只有一个buffer，一次只有一个内核线程使用
2. **缓存**：缓存常用块，避免从慢速磁盘重新读取

**核心接口**：
- `bread()`：返回磁盘块的内存副本（已锁定）
- `bwrite()`：将修改后的buffer写回磁盘
- `brelse()`：释放buffer

**LRU替换**：
- buffer组成双端链表
- `brelse`将buffer移到链表头部（最近使用）
- 需要新buffer时从尾部找（最近最少使用）

**两种锁**：
- `bcache.lock`：保护缓存块的信息（哪个块被缓存）
- `b->lock`（sleep-lock）：保护块内容的读写

### 9.3 日志层（Logging）

**目的**：解决崩溃恢复问题，确保文件系统操作原子性。

**工作流程**：
1. 系统调用将写入记录在磁盘日志中（`begin_op()` ~ `end_op()`）
2. 全部记录完成后写**提交记录**（commit）
3. 将日志中的写入复制到磁盘文件系统位置
4. 清除日志

**崩溃恢复**：
- 崩溃前提交 → 日志不完整 → 忽略（操作未发生）
- 崩溃后提交 → 日志完整 → 重新执行写入

**组提交（group commit）**：多个系统调用的写入累积到一个事务中一起提交。

**Commit四阶段**：
1. `write_log()`：修改的块从buffer复制到磁盘日志槽
2. `write_head()`：写header块（标记已提交）
3. `install_trans()`：日志块写到文件系统对应位置
4. 重置日志块计数为0

### 9.4 Inode层

**两种含义**：
1. **磁盘inode(dinode)**：磁盘上的数据结构，包含文件大小和数据块号列表
2. **内存inode(inode)**：磁盘inode的副本，加内核需要的信息

**磁盘inode结构**：
```c
struct dinode {
    short type;      // 文件类型（文件/目录/设备），0=空闲
    short nlink;     // 链接数
    uint size;       // 文件大小（字节）
    uint addrs[NDIRECT+1];  // 数据块号（直接+间接）
};
```

**数据块寻址**：
- `NDIRECT`个直接块号存放在`addrs`数组前部
- 第`NDIRECT+1`个元素指向**间接块**（含256个块号）
- 最大文件：12KB(直接) + 256KB(间接) = 268KB

**内存inode管理**：
- `iget()`：获取inode引用（增加ref计数）
- `ilock()`：锁定inode（使用sleep-lock）
- `iunlock()`：解锁
- `iput()`：释放引用（ref=0时释放缓存槽位）

**四种锁/类似锁机制**：
1. `icache.lock`：保证inode缓存唯一性
2. `inode->lock`（sleep-lock）：独占访问inode内容
3. `ref`引用计数：保持inode在缓存中
4. `nlink`链接计数：链接数>0时不释放inode

### 9.5 目录层

**目录结构**：
- inode类型为`T_DIR`
- 数据是一系列**目录项(dirent)**
- 每个目录项：名称(最多DIRSIZ=14字符) + inode号
- inode号为0表示空闲目录项

**核心函数**：
- `dirlookup()`：在目录中搜索给定名称的条目
- `dirlink()`：在目录中创建新目录项

### 9.6 路径名层

**路径解析**：`namex()`函数
1. 以`/`开头从根目录开始，否则从当前目录
2. 使用`skipelem`遍历路径每个元素
3. 对每个元素调用`dirlookup`查找
4. `namei()`返回对应inode，`nameiparent()`返回父目录inode

**并发安全**：
- 查找线程持有目录锁
- `dirlookup`返回`iget`获得的inode（增加引用计数）
- 收到inode后才释放目录锁
- 避免目录被删除后inode被重用

### 9.7 文件描述符层

**结构体file**：
```c
struct file {
    enum { FD_NONE, FD_PIPE, FD_INODE, FD_DEVICE } type;
    int ref;          // 引用计数
    char readable;
    char writable;
    struct pipe *pipe;
    struct inode *ip;
    uint off;         // I/O偏移量
};
```

- 每次`open`创建新的`file`（新偏移量）
- `dup/fork`共享偏移量
- `fileread/filewrite`使用偏移量并原子更新

## 10. 重点总结

### 10.1 磁盘调度算法对比

| 算法 | 原理 | 平均寻道 | 饥饿 | 最长等待 |
|------|------|----------|------|----------|
| FCFS | 按请求顺序 | 大 | 无 | 大 |
| SSTF | 最短寻道优先 | 较小 | 有 | 不确定 |
| SCAN | 来回扫描 | 较小 | 无 | 大（反方向） |
| C-SCAN | 单向扫描+回跳 | 较小 | 无 | SCAN的一半 |
| LOOK | SCAN改进（不到端点） | 小 | 无 | 小 |

### 10.2 文件目录结构对比

| 结构 | 特点 | 重名 | 共享 | 检索速度 |
|------|------|------|------|----------|
| 单级 | 一张目录表 | 不允许 | 不支持 | 慢 |
| 两级 | MFD+UFD | 用户间可 | 有限支持 | 较快 |
| 树型 | 多级子目录 | 允许 | 支持 | 快 |
| 图形 | 树型+共享链接 | 允许 | 强支持 | 快 |

### 10.3 xv6关键机制总结

| 机制 | 实现要点 |
|------|----------|
| **进程调度** | 轮询调度，swtch上下文切换，sleep/wakeup同步 |
| **内存管理** | Sv39三级页表，每进程独立页表，kalloc空闲链表 |
| **文件系统** | 七层结构，Buffer Cache+LRU，日志保证原子性，inode直接/间接索引 |
| **锁** | 自旋锁（amoswap，短临界区）+ 睡眠锁（长操作，可让出CPU） |
| **设备驱动** | 上半部/下半部分离，UART中断驱动，trampoline页处理trap |

> **复习要点**：xv6的核心设计理念是**简单明了**，同时展示操作系统的基本概念：进程隔离、虚拟内存、并发控制、持久化存储。
