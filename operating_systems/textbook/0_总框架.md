# 0. 先建立总框架

操作系统可以理解为：

> **管理硬件资源，并给应用程序提供统一服务的软件。**

电脑里真正的硬件有：

```text
CPU
内存
磁盘
键盘、鼠标、显示器、网卡等 I/O 设备
```

但程序员不可能每次都直接控制硬件，所以操作系统把硬件包装成更好用的抽象：

```text
CPU      → 进程 / 线程
内存     → 虚拟地址空间
磁盘     → 文件
设备     → read / write / open / close
```

所以整门操作系统课可以压缩成一句话：

> **操作系统通过进程管理 CPU，通过存储管理内存，通过文件系统管理磁盘，通过 I/O 系统管理设备，并通过系统调用给应用程序提供服务。**

# 1. 操作系统 OS

## 1.1 概念

**操作系统**是配置在计算机硬件上的第一层系统软件。

它负责：

```text
管理 CPU
管理内存
管理 I/O 设备
管理文件
为用户和程序提供接口
```

## 1.2 类比

你可以把计算机想成一家饭店：

```text
CPU      = 厨师
内存     = 厨房操作台
磁盘     = 仓库
I/O设备  = 服务员、收银机、外卖窗口
操作系统 = 店长
进程     = 一个个订单
```

如果没有店长，所有订单都抢厨师、抢操作台、抢仓库，饭店会乱套。

操作系统就是那个负责协调的人。

# 2. 并发与并行

这是第一章非常重要的概念。

## 2.1 并发

**并发**是指多个任务在同一时间段内都在推进。

单核 CPU 上，并发其实是快速切换：

```text
0~10ms    执行进程 A
10~20ms   执行进程 B
20~30ms   执行进程 C
30~40ms   又回到进程 A
```

看起来多个程序同时运行，其实 CPU 每一瞬间只运行一个。

## 2.2 并行

**并行**是真正同时执行。

需要多个 CPU 或多核 CPU。

```text
核心 1 执行 A
核心 2 执行 B
核心 3 执行 C
```

## 2.3 考试区分

```text
并发：宏观同时，微观交替
并行：宏观同时，微观也同时
```

# 3. 进程

## 3.1 概念

**进程 = 正在运行的程序。**

程序是静态的，进程是动态的。

例如：

```text
磁盘上的 QQ.exe        → 程序
正在运行的 QQ          → 进程
打开两个 QQ            → 两个进程
```

## 3.2 进程由什么组成？

一个进程通常包括：

```text
程序代码
数据
堆
栈
寄存器状态
打开的文件
PCB
```

其中最重要的是 **PCB**。

# 4. PCB：进程控制块

## 4.1 概念

**PCB 是操作系统用来记录和管理进程的数据结构。**

它是进程存在的唯一标志。

没有 PCB，操作系统就不知道这个进程是谁、运行到哪、用了什么资源。

## 4.2 PCB 里面有什么？

常见内容：

```text
进程 ID
进程状态
程序计数器 PC
CPU 寄存器
内存信息
打开文件信息
调度信息
I/O 状态信息
```

## 4.3 类比

PCB 就像学生档案：

```text
姓名
学号
班级
成绩
宿舍
课程表
奖惩记录
```

学校通过学生档案管理学生；操作系统通过 PCB 管理进程。

# 5. 进程状态

## 5.1 三种基本状态

```text
就绪 Ready
运行 Running
阻塞 Blocked
```

### 就绪

进程已经准备好，只差 CPU。

```text
我准备好了，就等老师点我回答问题。
```

### 运行

进程正在 CPU 上执行。

```text
老师正在让我回答问题。
```

### 阻塞

进程在等某件事，暂时不能运行。

例如：

```text
等待键盘输入
等待磁盘读写
等待网络数据
等待信号量
```

## 5.2 状态转换

```text
就绪 → 运行：获得 CPU
运行 → 就绪：时间片用完
运行 → 阻塞：等待 I/O 或资源
阻塞 → 就绪：等待事件完成
运行 → 终止：程序结束
```

这个图必须会背：

```text
Ready → Running → Blocked
  ↑        ↓          ↓
  └────────┴──────────┘
```

# 6. 线程

## 6.1 概念

**线程是进程中的一个执行流。**

一个进程可以包含多个线程。

例如浏览器：

```text
一个线程负责界面
一个线程负责下载
一个线程负责播放视频
一个线程负责执行 JavaScript
```

## 6.2 进程和线程区别

这是高频考点：

| 对比     | 进程         | 线程                         |
| -------- | ------------ | ---------------------------- |
| 本质     | 资源分配单位 | CPU 调度单位                 |
| 地址空间 | 进程之间独立 | 同一进程内线程共享           |
| 切换开销 | 大           | 小                           |
| 通信     | 较复杂       | 较方便                       |
| 安全性   | 隔离性强     | 一个线程出错可能影响整个进程 |

## 6.3 最重要一句话

> **进程是资源分配的基本单位，线程是 CPU 调度的基本单位。**

# 7. 临界资源与临界区

## 7.1 临界资源

**一次只允许一个进程使用的资源**叫临界资源。

例如：

```text
打印机
共享变量
共享缓冲区
某些文件表项
```

## 7.2 临界区

访问临界资源的代码段叫临界区。

例如：

```c
count++;
```

如果多个线程同时执行这句，就可能出错。

因为 `count++` 看起来是一句，实际上可能是三步：

```text
读取 count
count + 1
写回 count
```

两个线程同时操作，可能发生数据丢失。

# 8. 互斥与同步

## 8.1 互斥

互斥解决的是：

> **不能同时做。**

例如：

```text
同一时刻只能一个人使用打印机
同一时刻只能一个线程修改 count
```

## 8.2 同步

同步解决的是：

> **必须按顺序做。**

例如：

```text
必须先生产产品，再消费产品
必须先写入数据，再读取数据
```

## 8.3 区别

```text
互斥：强调不能同时
同步：强调先后顺序
```

# 9. 信号量

## 9.1 概念

信号量可以理解为一个整数，用来表示资源数量。

它有两个基本操作：

```text
P 操作 / wait：申请资源
V 操作 / signal：释放资源
```

## 9.2 P 操作

```text
S = S - 1
如果 S < 0，进程阻塞
```

意思是：

```text
我要申请一个资源
如果没有资源，就等着
```

## 9.3 V 操作

```text
S = S + 1
如果有进程等待，就唤醒一个
```

意思是：

```text
我释放一个资源
如果有人在等，就叫他来用
```

## 9.4 互斥写法

```c
semaphore mutex = 1;

P(mutex);
临界区;
V(mutex);
```

## 9.5 同步写法

假设必须先执行 A，再执行 B：

```c
semaphore S = 0;

进程 1:
A;
V(S);

进程 2:
P(S);
B;
```

# 10. 生产者—消费者问题

这是同步互斥经典题。

## 10.1 场景

```text
生产者：往缓冲区放产品
消费者：从缓冲区取产品
缓冲区：容量有限
```

要解决两个问题：

```text
不能同时操作缓冲区      → 互斥
空了不能消费，满了不能生产 → 同步
```

## 10.2 三个信号量

```c
semaphore mutex = 1;  // 互斥访问缓冲区
semaphore empty = n;  // 空缓冲区数量
semaphore full = 0;   // 满缓冲区数量
```

## 10.3 生产者

```c
P(empty);   // 申请空位
P(mutex);   // 进入缓冲区

放入产品;

V(mutex);   // 离开缓冲区
V(full);    // 满位 +1
```

## 10.4 消费者

```c
P(full);    // 申请产品
P(mutex);   // 进入缓冲区

取出产品;

V(mutex);   // 离开缓冲区
V(empty);   // 空位 +1
```

## 10.5 记忆方法

```text
生产者：先要空位 empty，再放产品，最后增加 full
消费者：先要产品 full，再取产品，最后增加 empty
```

# 11. 处理机调度

## 11.1 概念

处理机调度就是：

> **就绪队列里有很多进程，操作系统决定让谁先用 CPU。**

## 11.2 调度层次

```text
高级调度：作业调度
中级调度：内存调度
低级调度：进程调度
```

最常考的是低级调度，也就是进程调度。

# 12. 常见调度算法

## 12.1 FCFS：先来先服务

谁先到，谁先运行。

优点：

```text
简单
公平
```

缺点：

```text
短作业可能等很久
```

## 12.2 SJF：短作业优先

运行时间短的先执行。

优点：

```text
平均等待时间短
```

缺点：

```text
长作业可能饥饿
需要知道运行时间
```

## 12.3 优先级调度

优先级高的先执行。

问题：

```text
低优先级进程可能一直得不到 CPU
```

解决：

```text
老化 Aging
等待越久，优先级越高
```

## 12.4 时间片轮转 RR

每个进程运行一个时间片。

```text
A 运行 10ms
B 运行 10ms
C 运行 10ms
再回到 A
```

适合：

```text
分时系统
交互式系统
```

时间片太大：

```text
退化成 FCFS
```

时间片太小：

```text
上下文切换太频繁
```

# 13. 死锁

## 13.1 概念

死锁是指多个进程互相等待对方占有的资源，导致谁都无法继续。

例子：

```text
进程 A 拿着资源 R1，等待 R2
进程 B 拿着资源 R2，等待 R1
```

结果：

```text
A 等 B
B 等 A
谁都不释放
```

## 13.2 死锁四个必要条件

必须背：

```text
互斥条件
请求和保持条件
不可剥夺条件
循环等待条件
```

### 互斥

资源一次只能给一个进程用。

### 请求和保持

进程已经拿着资源，又去申请新资源。

### 不可剥夺

资源不能被强行抢走。

### 循环等待

形成等待环。

```text
P1 等 P2
P2 等 P3
P3 等 P1
```

## 13.3 处理死锁的方法

```text
死锁预防
死锁避免
死锁检测
死锁解除
```

### 死锁预防

破坏四个必要条件之一。

### 死锁避免

分配资源前判断是否安全。

典型算法：

```text
银行家算法
```

### 死锁检测

允许死锁发生，然后检查。

### 死锁解除

发现死锁后解除，比如：

```text
撤销进程
剥夺资源
进程回退
```

# 14. 银行家算法

## 14.1 核心思想

银行家算法的思想是：

> **系统在分配资源前，先判断分配后是否仍然安全。安全才分配，不安全就拒绝。**

## 14.2 安全状态

如果系统能找到一个进程完成顺序，让所有进程最终都能完成，则系统安全。

```text
安全状态：一定不会死锁
不安全状态：不一定死锁，但有死锁风险
死锁状态：一定不安全
```

## 14.3 常见数据结构

```text
Available：当前可用资源
Max：每个进程最大需求
Allocation：已经分配资源
Need：还需要资源
```

关系：

```text
Need = Max - Allocation
```

做题步骤：

```text
1. 看 Request <= Need 是否成立
2. 看 Request <= Available 是否成立
3. 假设分配
4. 检查是否存在安全序列
```

# 15. 存储器管理

## 15.1 目标

存储器管理就是管理内存。

主要任务：

```text
内存分配
内存回收
地址转换
内存保护
内存扩充
```

## 15.2 逻辑地址与物理地址

### 逻辑地址

程序看到的地址。

### 物理地址

真实内存地址。

例如程序以为自己从 0 地址开始：

```text
逻辑地址：0, 1, 2, 3 ...
```

但实际可能装入内存的某个位置：

```text
物理地址：4000, 4001, 4002 ...
```

## 15.3 地址重定位

把逻辑地址转换成物理地址。

```text
逻辑地址 → 地址变换机构 → 物理地址
```

# 16. 分页

## 16.1 概念

分页就是把程序地址空间和物理内存都切成固定大小的小块。

```text
程序地址空间 → 页 page
物理内存     → 页框 frame
```

页和页框大小相同。

## 16.2 页表

页表记录：

```text
页号 → 页框号
```

例如：

```text
第 0 页 → 第 5 块
第 1 页 → 第 9 块
第 2 页 → 第 3 块
```

## 16.3 地址结构

逻辑地址分成：

```text
页号 P + 页内位移 W
```

地址转换过程：

```text
逻辑地址
→ 分离出页号和页内位移
→ 用页号查页表
→ 得到物理块号
→ 物理块号 + 页内位移
→ 物理地址
```

## 16.4 分页的优缺点

优点：

```text
没有外部碎片
内存利用率较高
便于虚拟内存
```

缺点：

```text
有内部碎片
需要页表
地址转换有开销
```

# 17. 分段

例如：

```text

栈段
子程序段
共享段
```

## 17.2 地址结构

```text
段号 S + 段内位移 W
```

## 17.3 段表

段表记录：

```text
段基址
段长
```

| 项目         | 分页           | 分段             |
| ------------ | -------------- | ---------------- |
| 划分方式     | 固定大小       | 按逻辑功能       |

| 地址空间     | 一维           | 二维             |
| 主要目的     | 提高内存利用率 | 方便程序逻辑组织 |

# 18. 虚拟存储器

## 18.1 概念

虚拟存储器让程序感觉自己拥有很大的内存。

实际上：

```text
程序的一部分在内存
另一部分在磁盘
需要时再调入
```

## 18.2 核心思想

```text
部分装入
按需调入
页面置换
```

## 18.3 局部性原理

虚拟内存能工作，是因为程序具有局部性。

### 时间局部性

刚访问过的数据，近期可能再次访问。

例如：

```text
循环变量 i 会反复使用
```

### 空间局部性

访问某地址后，附近地址可能也会被访问。

例如：

```text
数组 a[0], a[1], a[2] 连续访问
```

# 19. 缺页中断

## 19.1 概念

程序访问某页时，如果该页不在内存，就发生缺页中断。

## 19.2 处理过程

```text
1. CPU 访问页面
2. 页表发现该页不在内存
3. 产生缺页中断
4. 操作系统找到该页在磁盘的位置
5. 找空闲页框
6. 如果没有空闲页框，执行页面置换
7. 把页面调入内存
8. 修改页表
9. 重新执行被中断的指令
```

## 19.3 类比

你学习时桌子上放不下所有书。

```text
桌面 = 内存
书架 = 磁盘
正在看的书 = 当前页面
```

需要某本书时，如果桌面没有，就去书架拿。这就是缺页。

# 20. 页面置换算法

## 20.1 为什么需要页面置换？

内存满了，但又要调入新页面，就必须淘汰一个旧页面。

## 20.2 OPT 最佳置换

淘汰未来最长时间不会用的页面。

优点：

```text
缺页次数最少
```

缺点：

```text
无法实现，因为不知道未来
```

## 20.3 FIFO 先进先出

谁最早进入内存，就淘汰谁。

优点：

```text
简单
```

缺点：

```text
可能淘汰常用页面
可能出现 Belady 异常
```

## 20.4 LRU 最近最久未使用

淘汰最近最长时间没用过的页面。

优点：

```text
符合局部性原理
效果较好
```

缺点：

```text
实现开销大
```

## 20.5 Clock 算法

LRU 的近似算法。

每个页面有访问位：

```text
访问位 = 1：最近访问过
访问位 = 0：最近没访问过
```

扫描时：

```text
遇到 0：淘汰
遇到 1：改成 0，继续找
```

# 21. 抖动

## 21.1 概念

抖动是指进程频繁发生缺页，大量时间用于页面换入换出，真正执行时间很少。

表现：

```text
CPU 利用率下降
磁盘 I/O 很忙
系统响应变慢
```

## 21.2 原因

```text
分配给进程的物理块太少
系统中进程太多
工作集装不下
```

## 21.3 解决方法

```text
增加物理块
减少并发进程数
使用工作集模型
改进页面置换算法
```

设备完成后主动通知 CPU。

```text
CPU 去做别的事
设备完成后发中断
CPU 再回来处理
```

优点：

```text
提高 CPU 利用率
```

## 22.5 DMA

DMA 是直接存储器访问。

数据可以直接在设备和内存之间传输，CPU 不必一个字节一个字节搬。

适合：

```text
磁盘
网卡
高速设备
```

# 23. 缓冲

## 23.1 概念

缓冲区是内存中临时存放数据的区域。

目的：

```text
缓和 CPU 和 I/O 设备速度不匹配
减少等待
提高并行性
```

## 23.2 类比

你去食堂打饭：

```text
厨师做菜很慢
学生取菜很快
```

如果有一个菜品暂存区，厨师可以提前做好，学生来了直接拿。

这个暂存区就是缓冲区。

# 24. SPOOLing

## 24.1 概念

SPOOLing 又叫假脱机技术。

它把独占设备改造成逻辑上的共享设备。

典型例子：

```text
打印机
```

## 24.2 打印机例子

```text
先把打印内容放到磁盘输出井
然后由打印进程一个个打印
```

用户感觉打印机好像可以共享。

# 26. 文件控制块 FCB 与 inode

## 26.1 FCB

FCB 是文件控制块，用来记录文件信息。

包括：

```text
文件名
文件大小
文件类型
物理位置
权限
时间信息
```

## 26.2 inode

UNIX/Linux 中常用 inode 管理文件元数据。

inode 保存：

```text
文件大小
权限
拥有者
时间
数据块地址
```

但通常不保存文件名。

## 26.3 目录项和 inode 的关系

```text
目录项：文件名 → inode 号
inode：文件属性 + 数据块位置
```

例如：

```text
a.txt → inode 100
inode 100 → 文件大小、权限、磁盘块位置
```

# 27. 打开文件表 / 文件头数组

你前面问过“文件头数组”，可以放在这里理解。

## 27.1 概念

当程序打开文件时，操作系统会在内存中建立一个记录，保存这个文件的打开状态。

这个表通常叫：

```text
打开文件表
文件表
文件头数组
```

## 27.2 里面记录什么？

```text
文件当前读写位置
打开方式
引用计数
指向 FCB/inode 的指针
文件状态
```

例如：

```c
fd = open("a.txt", O_RDONLY);
```

返回的 `fd` 可以理解为这个打开文件表中的编号。

## 27.3 和 inode 的区别

```text
inode / FCB：描述文件本身
打开文件表：描述文件被打开后的状态
```

同一个文件可以被打开多次，所以：

```text
一个 inode
可能对应多个打开文件表项
```

# 29. 文件物理结构

## 29.1 连续分配

文件占用连续磁盘块。

优点：

```text
顺序访问快
随机访问方便
```

缺点：

```text
外部碎片
文件扩展困难
```

## 29.2 链接分配

文件块之间用指针链接。

优点：

```text
没有外部碎片
文件容易扩展
```

缺点：

```text
随机访问慢
指针损坏会影响文件
```

## 29.3 索引分配

建立索引块，记录文件所有磁盘块号。

优点：

```text
支持随机访问
没有外部碎片
文件容易扩展
```

缺点：

```text
索引块占空间
```

# 30. 磁盘调度

## 30.1 为什么需要磁盘调度？

磁盘访问慢，主要时间花在：

```text
寻道时间
旋转延迟
传输时间
```

其中寻道时间最重要。

磁盘调度就是减少磁头移动距离。

------

## 30.2 FCFS

按请求到达顺序处理。

优点：

```text
公平
简单
```

缺点：

```text
磁头移动距离可能很大
```

------

## 30.3 SSTF

最短寻道时间优先。

每次选择距离当前磁头最近的请求。

优点：

```text
平均寻道距离短
```

缺点：

```text
远处请求可能饥饿
```

------

## 30.4 SCAN

电梯算法。

磁头像电梯一样：

```text
先朝一个方向服务
到头后再反向
```

优点：

```text
比较公平
寻道性能较好
```

------

## 30.5 C-SCAN

循环扫描。

只在一个方向服务，到头后直接回到另一端。

优点：

```text
等待时间更均匀
```

# 31. 系统调用

## 31.1 概念

系统调用是用户程序请求操作系统服务的接口。

应用程序不能直接操作硬件，必须通过系统调用。

例如：

```c
open()
read()
write()
close()
fork()
exec()
wait()
```

## 31.2 系统调用过程

```text
用户程序调用库函数
触发陷入 trap
CPU 从用户态进入核心态
操作系统执行服务
返回用户态
```

------

# 32. 用户态与核心态

## 32.1 用户态

普通程序运行的状态。

不能：

```text
直接访问硬件
执行特权指令
随意访问内核内存
```

## 32.2 核心态

操作系统内核运行的状态。

可以：

```text
访问硬件
处理中断
管理内存
执行特权指令
```

## 32.3 为什么要区分？

为了安全。

如果普通程序可以随便操作硬件和内核，系统会很容易崩溃。


# 36. 这门课的学习主线

你可以按这条线理解整门课：

```text
程序运行起来 → 形成进程
多个进程抢 CPU → 需要调度
多个进程共享资源 → 需要同步互斥
资源分配不当 → 可能死锁
进程需要内存 → 需要存储管理
内存不够 → 需要虚拟内存
进程要读写设备 → 需要 I/O 管理
数据要长期保存 → 需要文件系统
文件存在磁盘上 → 需要磁盘管理
程序请求 OS 服务 → 需要系统调用
系统要可靠运行 → 需要保护与安全
```

# 37. 最适合背的 10 个核心句子

1. **操作系统是管理计算机硬件和软件资源、为用户和程序提供接口的系统软件。**
2. **并发是多个程序在同一时间段内交替执行，并行是多个程序在同一时刻真正同时执行。**
3. **进程是程序的一次执行过程，是资源分配的基本单位。**
4. **线程是进程中的执行流，是 CPU 调度的基本单位。**
5. **PCB 是进程存在的唯一标志，保存进程管理所需的全部信息。**
6. **临界区是访问临界资源的代码段，临界资源一次只允许一个进程访问。**
7. **P 操作用于申请资源，V 操作用于释放资源。**
8. **死锁产生必须同时满足互斥、请求和保持、不可剥夺、循环等待四个条件。**
9. **虚拟存储器通过部分装入、请求调入和页面置换，在逻辑上扩充内存容量。**
10. **系统调用是用户程序获得操作系统服务的唯一途径。**

------

```text
2. P/V 操作与经典同步问题

4. 死锁与银行家算法
5. 分页、分段、虚拟内存
6. 文件系统与磁盘调度
```

这 6 块基本覆盖考试和面试的大部分核心题。
