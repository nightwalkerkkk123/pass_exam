# 第三章 处理机调度与死锁

## 3.1 处理机调度的层次和调度算法的目标
### 处理机调度的层次
1. **高级调度（作业调度/长程调度）**
   - 调度对象：作业，决定把外存上处于后备队列中的哪些作业调入内存，为它们创建进程、分配必要的资源，然后放入就绪队列
   - 主要功能：接纳多少作业（多道程序度）、接纳哪些作业（调度算法）
   - 运行频率：低，几分钟一次，只在批处理系统中存在，分时和实时系统中通常没有高级调度

2. **中级调度（内存调度/中程调度）**
   - 调度对象：进程，目的是提高内存利用率和系统吞吐量
   - 主要功能：将暂时不能运行的进程调至外存等待（挂起状态），当它们具备运行条件且内存有空闲时，再调回内存（就绪状态）
   - 运行频率：中等，几十秒一次

3. **低级调度（进程调度/短程调度）**
   - 调度对象：进程（或内核级线程），决定就绪队列中的哪个进程获得处理机
   - 主要功能：保存当前进程的处理机上下文、按某种算法选择就绪进程、把处理机上下文分配给选中的进程
   - 运行频率：高，几十毫秒一次，是操作系统中最基本的调度，所有类型的OS都必须配置低级调度

### 调度队列模型
- **仅有进程调度的调度队列模型**：分时系统常用，所有进程都在就绪队列等待调度，执行中因阻塞进入对应阻塞队列，唤醒后返回就绪队列
- **具有高级和低级调度的调度队列模型**：批处理系统常用，作业从后备队列经高级调度进入内存就绪队列，进程调度分配CPU
- **具有三级调度的调度队列模型**：引入中级调度，增加了挂起状态，进程可以在内存和外存之间对换

### 调度算法的目标
#### 共同目标
1. **资源利用率**：使CPU和其他资源都尽可能保持忙碌状态
2. **公平性**：使每个进程都获得合理的CPU时间，防止进程饥饿
3. **平衡性**：保持系统资源使用的平衡性，尽可能使CPU和IO设备都处于忙碌状态
4. **策略强制执行**：对所规定的策略（如高优先级进程优先）必须予以执行

#### 批处理系统的目标
1. **吞吐量高**：单位时间内完成的作业数尽可能多
2. **周转时间短**：从作业提交到作业完成的时间尽可能短（周转时间=完成时间-提交时间，带权周转时间=周转时间/服务时间）
3. **处理机利用率高**：使CPU尽可能处于忙碌状态

#### 分时系统的目标
1. **响应时间快**：从用户提交请求到系统首次产生响应的时间尽可能短
2. **均衡性**：系统对用户请求的响应时间应尽可能均衡，与用户请求服务的类型相适应

#### 实时系统的目标
1. **截止时间的保证**：实时任务的执行必须在规定的截止时间内完成，这是实时系统最关键的目标
2. **可预测性**：系统的行为应该是可预测的，能够预先知道任务的执行时间和资源需求

## 3.2 作业与作业调度
### 作业的基本概念
- **作业**：用户向系统提交的一个独立任务，由程序、数据和作业说明书三部分组成
- **作业控制块（JCB）**：系统为每个作业建立的管理数据结构，记录作业的相关信息，是作业存在的唯一标志
- **作业的状态**：提交状态、后备状态、运行状态、完成状态

### 作业调度算法
#### 1. 先来先服务（FCFS, First-Come First-Served）
- **原理**：按照作业到达的先后顺序进行调度，优先考虑在系统中等待时间最长的作业，而不管其要求运行时间的长短
- **优点**：实现简单，公平
- **缺点**：不利于短作业，短作业的带权周转时间可能很长；不利于IO繁忙型作业

#### 2. 短作业优先（SJF, Short Job First）
- **原理**：优先选择估计运行时间最短的作业投入运行
- **优点**：能有效降低作业的平均周转时间和平均带权周转时间，提高系统的吞吐量
- **缺点**：
  - 对长作业不利，可能出现饥饿现象（短作业源源不断到来时，长作业长期得不到调度）
  - 作业的估计运行时间由用户提供，不一定准确
  - 未考虑作业的紧迫程度，不能保证紧迫性作业得到及时处理

#### 3. 优先级调度算法（PSA, Priority Scheduling Algorithm）
- **原理**：根据作业的优先级高低进行调度，优先级高的作业优先调入内存
- **优先级的确定**：可以根据作业的紧迫程度、资源需求、用户类型等因素确定，分为静态优先级和动态优先级

#### 4. 高响应比优先调度算法（HRRN, Highest Response Ratio Next）
- **原理**：调度时计算每个作业的响应比，选择响应比最高的作业投入运行
  $$ 响应比R_p = \frac{等待时间 + 要求服务时间}{要求服务时间} = 1 + \frac{等待时间}{要求服务时间} $$
- **优点**：
  1. 兼顾了短作业和长作业，短作业响应比高，优先调度
  2. 长作业等待时间足够长时，响应比会升高，不会出现饥饿现象
- **缺点**：每次调度都需要计算所有作业的响应比，增加了系统开销

## 3.3 进程调度
### 进程调度的任务、机制和方式
#### 任务
1. 保存处理机的上下文
2. 按某种算法选择一个就绪进程
3. 把处理机分配给选中的进程

#### 机制
1. 排队器：将就绪进程按一定策略组织成队列
2. 分派器：从就绪队列中取出选中的进程，进行上下文切换，把CPU分配给它
3. 上下文切换机制：保存当前进程的上下文，恢复新进程的上下文

#### 调度方式
1. **非抢占方式**：一旦把处理机分配给某进程后，就让它一直运行下去，直到该进程完成或发生某事件而被阻塞时，才把处理机分配给其他进程
   - 优点：实现简单，系统开销小
   - 缺点：不适合分时系统和实时系统，难以满足紧急任务的需求
2. **抢占方式**：允许调度程序根据某种原则，去暂停某个正在执行的进程，将已分配给该进程的处理机重新分配给另一进程
   - 抢占原则：优先级原则、短进程优先原则、时间片原则
   - 优点：可以满足分时系统和实时系统的需求，防止一个进程长期占用CPU
   - 缺点：系统开销较大

### 进程调度算法
#### 1. 轮转调度算法（RR, Round Robin）
- **原理**：系统将所有就绪进程按先来先服务的原则排成一个队列，每次调度时把CPU分配给队首进程，令其执行一个时间片，时间片用完后，进程被送到就绪队列末尾，等待下一次调度
- **时间片大小的确定**：时间片太小会导致频繁的进程切换，增加系统开销；时间片太大则轮转算法退化为FCFS算法，响应时间变长。通常时间片大小略大于一次典型交互所需的时间（如10-100ms）
- **适用场景**：分时系统

#### 2. 优先级调度算法
- **原理**：把处理机分配给就绪队列中优先级最高的进程
- **优先级类型**：
  - **静态优先级**：进程的优先级在创建时确定，整个运行期间保持不变，实现简单，但公平性差，可能出现饥饿
  - **动态优先级**：进程的优先级随进程的推进或等待时间的增加而改变，避免饥饿，更公平，但实现复杂
- **抢占式优先级调度**：新到达的进程优先级高于当前运行进程时，抢占CPU，适合实时系统
- **非抢占式优先级调度**：直到当前进程运行完成或阻塞才调度，适合批处理系统

#### 3. 多级反馈队列调度算法（MFQ, Multilevel Feedback Queue）
- **原理**：
  1. 设置多个就绪队列，每个队列赋予不同的优先级，第一队列优先级最高，其余依次降低
  2. 每个队列的时间片大小不同，优先级越高的队列时间片越小
  3. 新进程进入内存后，首先放入第一队列末尾，按FCFS原则排队等待调度，若能在时间片内完成则撤离系统，否则放入第二队列末尾，以此类推，降到最后一个队列后采用轮转方式运行
  4. 只有当第一队列空闲时，调度程序才调度第二队列中的进程运行；仅当所有更高优先级队列都空闲时，才调度最低优先级队列中的进程运行
- **优点**：
  1. 兼顾了短作业和长作业，短作业可以在第一队列很快完成
  2. 兼顾了IO繁忙型作业和计算繁忙型作业
  3. 不必事先估计作业的运行时间
  4. 能满足分时系统和实时系统的需求，是目前公认的较好的进程调度算法

## 3.4 实时调度
### 实时调度的基本条件
1. 提供必要的调度信息：就绪时间、开始截止时间和完成截止时间、处理时间、资源要求、优先级
2. 系统处理能力强：要保证所有实时任务都能在截止时间内完成，系统处理能力必须足够，通常需要多处理机系统
3. 采用抢占式调度机制：硬实时系统必须采用抢占式调度，才能保证紧急任务的及时处理
4. 具有快速切换机制：减小进程切换的开销，提高响应速度

### 实时调度算法的分类
1. **非抢占式调度算法**：适合弱实时系统
   - 非抢占式轮转调度：响应时间在几秒量级
   - 非抢占式优先调度：响应时间在几百毫秒量级
2. **抢占式调度算法**：适合强实时系统
   - 基于时钟中断的抢占式优先级调度：响应时间在几十毫秒量级
   - 立即抢占的优先级调度：响应时间在几毫秒到几百微秒量级

### 常用的实时调度算法
#### 1. 最早截止时间优先算法（EDF, Earliest Deadline First）
- **原理**：根据任务的开始截止时间确定任务的优先级，截止时间越早，优先级越高
- **适用场景**：既可用于抢占式调度，也可用于非抢占式调度，在任务可调度的情况下，能保证所有任务的截止时间要求
- **可调度性判定**：对于n个独立的周期性任务，每个任务的周期为P_i，执行时间为C_i，当满足 $\sum_{i=1}^{n} \frac{C_i}{P_i} \leq 1$ 时，EDF算法可以保证所有任务都满足截止时间要求

#### 2. 最低松弛度优先算法（LLF, Least Laxity First）
- **原理**：根据任务的松弛度确定优先级，松弛度越低，优先级越高
  $$ 松弛度 = 必须完成时间 - 本身剩余运行时间 - 当前时间 $$
- **适用场景**：抢占式调度，主要用于调度周期性实时任务
- **特点**：任务的松弛度会随着时间推移而动态变化，当松弛度为0时，必须立即调度执行，否则会错过截止时间

## 3.5 死锁概述
### 死锁的定义
死锁是指多个进程在运行过程中因争夺资源而造成的一种僵局，当进程处于这种僵持状态时，若无外力作用，它们都将无法再向前推进。

### 产生死锁的原因
1. **竞争不可抢占性资源**：多个进程都需要访问不可抢占的资源，当资源不足时，可能导致死锁
2. **竞争可消耗资源**：进程之间通信时，对消息这种可消耗资源的竞争也可能导致死锁
3. **进程推进顺序不当**：进程运行过程中请求和释放资源的顺序不当，也可能导致死锁

### 产生死锁的四个必要条件
1. **互斥条件**：进程对所分配到的资源进行排他性使用，即在一段时间内某资源只由一个进程占用
2. **请求和保持条件**：进程已经保持了至少一个资源，但又提出了新的资源请求，而该资源已被其他进程占有，此时请求进程被阻塞，但又对自己已获得的其他资源保持不放
3. **不可抢占条件**：进程已获得的资源，在未使用完之前，不能被抢占，只能在进程使用完时由自己释放
4. **循环等待条件**：在发生死锁时，必然存在一个进程-资源的循环链，链中的每个进程已获得的资源同时被链中下一个进程所请求

### 处理死锁的基本方法
1. **预防死锁**：破坏死锁产生的四个必要条件中的一个或几个，从根本上避免死锁的发生
2. **避免死锁**：在资源的动态分配过程中，用某种方法防止系统进入不安全状态，从而避免死锁
3. **检测死锁**：允许系统在运行过程中发生死锁，通过系统所设置的检测机构，及时检测出死锁的发生
4. **解除死锁**：检测到死锁发生后，采取适当的措施将进程从死锁状态中解脱出来

## 3.6 预防死锁
预防死锁是通过破坏死锁产生的四个必要条件中的一个或几个来实现的，其中互斥条件是某些资源的固有特性，不能破坏，只能破坏其他三个条件。

### 破坏"请求和保持"条件
- **方案1**：进程在运行之前一次性申请它在整个运行过程中所需的全部资源，系统满足后才分配，运行期间不再提出新的资源请求
  - 优点：简单、安全、容易实现
  - 缺点：资源利用率低，很多资源长期被占用而得不到使用；作业延迟运行，需要等待所有资源都满足才能开始运行
- **方案2**：允许进程只获得运行初期所需的资源便开始运行，运行过程中逐步释放已用完的资源，再请求新的资源
  - 优点：提高了资源利用率，减少了进程饥饿的概率

### 破坏"不可抢占"条件
- **方案**：当一个已经保持了某些不可抢占资源的进程提出新的资源请求而不能得到满足时，必须释放它已经保持的所有资源，待以后需要时再重新申请
- **优点**：可以避免死锁
- **缺点**：实现复杂，代价大，可能导致前段工作失效、进程执行无限推迟，降低系统吞吐量

### 破坏"循环等待"条件
- **方案**：对系统所有资源类型进行线性排序，赋予不同的序号，所有进程对资源的请求必须严格按照资源序号递增的顺序提出
- **优点**：资源利用率和系统吞吐量比前两种方法好
- **缺点**：资源序号必须相对稳定，限制了新类型设备的增加；进程使用资源的顺序可能与系统规定的顺序不同，造成资源浪费；限制了用户编程的自由

## 3.7 避免死锁
### 安全状态与不安全状态
- **安全状态**：系统能按某种进程推进顺序（P1, P2, ..., Pn），为每个进程Pi分配其所需资源，直至满足每个进程对资源的最大需求，使每个进程都可顺利地完成，这个序列称为安全序列
- **不安全状态**：系统无法找到一个安全序列，使得所有进程都能顺利完成，不安全状态可能导致死锁
- **死锁与安全状态的关系**：死锁状态一定是不安全状态，但不安全状态不一定是死锁状态；安全状态一定不会死锁，只要系统处于安全状态，就可以避免发生死锁

### 银行家算法（Banker's Algorithm）
#### 数据结构
1. **可利用资源向量Available**：长度为m的向量，表示系统中各类可用资源的数量，`Available[j] = k`表示第j类资源有k个可用
2. **最大需求矩阵Max**：n×m矩阵，`Max[i][j] = k`表示进程i需要第j类资源的最大数目为k
3. **分配矩阵Allocation**：n×m矩阵，`Allocation[i][j] = k`表示进程i已经分得第j类资源的数目为k
4. **需求矩阵Need**：n×m矩阵，`Need[i][j] = k`表示进程i还需要第j类资源的数目为k，`Need[i][j] = Max[i][j] - Allocation[i][j]`

#### 算法流程
1. **进程Pi发出资源请求Request_i**，`Request_i[j]`表示进程Pi请求第j类资源的数量
2. **检查请求合法性**：
   - 如果`Request_i[j] <= Need[i][j]`，继续；否则请求非法，进程需要的资源超过了它申报的最大值
   - 如果`Request_i[j] <= Available[j]`，继续；否则请求的资源不足，进程必须等待
3. **试探分配资源**：
   - `Available[j] = Available[j] - Request_i[j]`
   - `Allocation[i][j] = Allocation[i][j] + Request_i[j]`
   - `Need[i][j] = Need[i][j] - Request_i[j]`
4. **调用安全性算法检查系统状态是否安全**：
   - 如果安全，则分配资源，完成请求
   - 如果不安全，则撤销试探分配，恢复原来的状态，进程Pi等待

#### 安全性算法
1. 设置两个向量：
   - **工作向量Work**：长度为m，初始值为`Work = Available`，表示系统可提供的各类资源数量
   - **完成标志Finish**：长度为n的布尔向量，初始值为`Finish[i] = false`，表示进程i是否已经完成
2. 寻找满足条件的进程Pi：
   - `Finish[i] == false`
   - `Need[i][j] <= Work[j]` 对所有j成立
   若找到，执行步骤3；否则执行步骤4
3. 进程Pi可以获得所需资源顺利完成，执行：
   - `Work[j] = Work[j] + Allocation[i][j]`（进程完成后释放所有资源）
   - `Finish[i] = true`
   转回步骤2继续寻找
4. 检查所有进程的Finish标志：
   - 如果所有`Finish[i] == true`，则系统处于安全状态，存在安全序列
   - 否则系统处于不安全状态

### 银行家算法的特点
- **优点**：比预防死锁的限制条件宽松，资源利用率更高，可以避免死锁发生
- **缺点**：
  1. 要求进程必须预先说明自己的最大资源需求，实际应用中难以做到
  2. 进程运行期间资源数量必须固定，要求系统资源数量不变，实际中资源可能损坏或增减
  3. 进程的数量必须固定，不能动态增加新进程
  4. 每次资源分配都需要执行安全性算法，系统开销较大

## 3.8 死锁的检测与解除
### 死锁的检测
#### 资源分配图
- **组成**：
  - 进程结点：用圆圈表示，代表一个进程
  - 资源结点：用方框表示，代表一类资源，方框中的圆点代表该类资源的一个实例
  - 有向边：请求边（进程→资源，表示进程请求该类资源一个实例）、分配边（资源→进程，表示该类资源的一个实例已分配给进程）
- **死锁定理**：如果资源分配图是不可完全简化的，那么系统处于死锁状态，不可简化是指不存在任何进程可以获得它所需的全部资源顺利完成，从而释放它占有的所有资源

#### 死锁检测算法
1. 类似于安全性算法，通过寻找可以运行完成的进程，释放其占有的资源，看是否能将所有进程都运行完成，如果不能则存在死锁
2. 死锁检测的时机：可以在每次资源分配时检测，也可以周期性检测，或者当CPU利用率下降、进程长时间阻塞时检测

### 死锁的解除
1. **抢占资源**：从一个或多个死锁进程中抢占足够数量的资源，分配给其他死锁进程，以解除死锁状态
   - 需要考虑：选择哪些进程抢占、抢占的代价、保证被抢占进程不会饥饿
2. **终止（撤销）进程**：终止系统中的一个或多个死锁进程，直至打破循环等待，解除死锁状态
   - 终止策略：可以终止所有死锁进程，代价最大；也可以逐个终止进程，直到死锁解除，每次选择代价最小的进程终止（如优先级低、已运行时间短、剩余时间长、占有的资源少等）
