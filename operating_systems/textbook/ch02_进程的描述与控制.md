# 第二章 进程的描述与控制

## 2.1 前趋图和程序执行
### 前趋图（Precedence Graph）
- **定义**：有向无循环图，用于描述进程之间执行的先后顺序，结点表示进程/程序段，有向边表示前趋关系（`→`表示前一个操作完成后，后一个操作才能开始）
- **特征**：不存在循环，所有结点都只有有限个前趋和后继
- **前趋关系的表示**：若存在`Pi → Pj`，则称`Pi`是`Pj`的直接前趋，`Pj`是`Pi`的直接后继，没有前趋的结点称为初始结点，没有后继的结点称为终止结点

### 程序的顺序执行
- **特征**：
  1. **顺序性**：处理机严格按照程序所规定的顺序执行
  2. **封闭性**：程序运行时独占全机资源，资源的状态只有本程序才能改变
  3. **可再现性**：只要程序执行的环境和初始条件相同，多次运行结果相同

### 程序的并发执行
- **特征**：
  1. **间断性**：程序执行时呈现出"执行-暂停-执行"的间断性活动规律
  2. **失去封闭性**：多个程序共享系统资源，资源状态由多个程序共同改变
  3. **不可再现性**：程序经过多次执行后，其执行结果可能不同（这是不允许的，需要通过同步机制保证可再现）

## 2.2 进程的描述
### 进程的定义和特征
- **定义**：进程是进程实体的运行过程，是系统进行资源分配和调度的一个独立单位。进程实体由程序段、相关数据段和PCB（进程控制块）三部分组成。
- **特征**：
  1. **动态性**：进程是程序的一次执行过程，具有生命周期，是进程最基本的特征
  2. **并发性**：多个进程实体同存于内存中，能在一段时间内同时运行
  3. **独立性**：进程实体是一个能独立运行、独立分配资源、独立接受调度的基本单位
  4. **异步性**：进程按各自独立的、不可预知的速度向前推进

### 进程的状态与转换
- **三种基本状态**：
  1. **就绪状态（Ready）**：进程已经获得除CPU之外的所有必要资源，等待分配CPU
  2. **执行状态（Running）**：进程已经获得CPU，正在运行
  3. **阻塞状态（Blocked）**：进程正在等待某一事件发生（如IO完成、等待信号量）而暂时停止运行
- **状态转换**：
  - 就绪 → 执行：调度程序选中就绪进程，分配CPU
  - 执行 → 就绪：进程时间片用完，或被更高优先级进程抢占
  - 执行 → 阻塞：进程请求资源失败、等待IO完成、等待某事件发生
  - 阻塞 → 就绪：进程等待的事件发生

### 进程控制块（PCB）
- **作用**：操作系统用于描述进程的基本情况和运行状态，进而控制和管理进程的核心数据结构，是进程存在的唯一标志
- **包含信息**：
  1. 进程标识符：内部标识符（系统唯一ID）、外部标识符（用户可见）
  2. 处理机状态：寄存器内容（通用寄存器、指令计数器、程序状态字PSW等）
  3. 进程调度信息：进程状态、优先级、调度所需其他信息、事件（阻塞原因）
  4. 进程控制信息：程序和数据地址、进程同步和通信机制、资源清单、链接指针

## 2.3 进程控制
### 进程的创建
- **引起创建进程的事件**：用户登录、作业调度、用户请求、提供服务（如打印服务）
- **创建过程**：
  1. 申请空白PCB
  2. 为新进程分配所需资源（内存、文件、IO设备等）
  3. 初始化PCB（标识符、处理机状态、处理机控制信息等）
  4. 将新进程插入就绪队列

### 进程的终止
- **引起终止的事件**：正常结束、异常结束（越界、非法指令、特权指令错、算术错、IO故障）、外界干预（操作员/OS干预、父进程请求、父进程终止）
- **终止过程**：
  1. 根据进程标识符找到PCB，读取进程状态
  2. 若进程处于执行状态，立即停止执行，置调度标志为真
  3. 终止其所有子孙进程，防止成为孤儿进程
  4. 将进程拥有的全部资源归还给父进程或系统
  5. 将PCB从所在队列中移出，等待其他程序收集信息

### 进程的阻塞与唤醒
- **引起阻塞的事件**：请求系统服务失败、启动某种IO操作、等待数据到达、等待新任务
- **阻塞过程**：
  1. 进程主动调用阻塞原语block()
  2. 立即停止执行，将PCB中的状态从执行改为阻塞
  3. 将PCB插入到对应事件的阻塞队列
  4. 转调度程序重新调度
- **唤醒过程**：
  1. 当阻塞进程等待的事件发生，由相关进程调用唤醒原语wakeup()
  2. 将被阻塞进程的PCB从阻塞队列移出
  3. 将PCB中的状态从阻塞改为就绪
  4. 将PCB插入到就绪队列

### 进程的挂起与激活
- **挂起**：将进程从内存换到外存，处于静止状态（静止就绪、静止阻塞），暂时不参与调度
- **激活**：将挂起的进程从外存换回内存，变为活动状态（活动就绪、活动阻塞）

## 2.4 进程同步
### 基本概念
- **进程同步的主要任务**：对多个相关进程在执行次序上进行协调，使并发执行的进程之间能按照一定的规则（或时序）共享系统资源，并且能很好地相互合作，从而使程序的执行具有可再现性。
- **两种形式的制约关系**：
  1. **间接相互制约关系**：资源共享导致（如多个进程竞争打印机）
  2. **直接相互制约关系**：进程合作导致（如生产者-消费者进程）
- **临界资源**：一段时间内只允许一个进程访问的资源（如打印机、共享变量）
- **临界区**：每个进程中访问临界资源的那段代码，必须互斥访问
- **同步机制应遵循的规则**：
  1. **空闲让进**：临界区空闲时，允许请求进入的进程进入
  2. **忙则等待**：临界区已被占用时，其他请求进入的进程必须等待
  3. **有限等待**：进程进入临界区的请求必须在有限时间内得到满足，避免死等
  4. **让权等待**：当进程不能进入临界区时，应立即释放CPU，避免忙等

### 硬件同步机制
- **关中断**：进入临界区前关中断，退出后开中断，简单高效，但不适用于多处理机系统
- **利用Test-and-Set指令（TS指令）**：硬件实现的原子操作，通过测试并设置标志位实现互斥
- **利用Swap指令（交换指令）**：硬件实现的原子操作，交换两个字的内容实现互斥

### 信号量机制
- **整型信号量**：一个表示资源数量的整型变量，仅能通过两个原子操作`wait(S)`（P操作）和`signal(S)`（V操作）访问
  ```
  wait(S): while S <= 0 do no-op; S = S - 1;
  signal(S): S = S + 1;
  ```
  缺点：不满足"让权等待"，会出现忙等
- **记录型信号量**：在整型信号量基础上增加进程链表指针，用于链接所有等待该资源的进程，解决了忙等问题
  ```
  typedef struct {
    int value;          // 资源数量
    struct PCB *list;   // 等待队列
  } semaphore;

  wait(semaphore S):
    S.value--;
    if S.value < 0:
      block(S.list);    // 进程阻塞，加入等待队列，放弃CPU

  signal(semaphore S):
    S.value++;
    if S.value <= 0:
      wakeup(S.list);   // 唤醒等待队列中的第一个进程
  ```
  - `S.value > 0`：表示可用资源的数量
  - `S.value = 0`：表示没有可用资源，也没有等待进程
  - `S.value < 0`：绝对值表示等待该资源的进程数量
- **AND型信号量**：一次性申请进程需要的所有资源，全部满足才分配，避免死锁
- **信号量集**：一次可以申请多种类型的多个资源，且可以设置资源下限

### 信号量的应用
- **实现进程互斥**：设置互斥信号量`mutex`，初值为1，临界区放在`wait(mutex)`和`signal(mutex)`之间
- **实现前趋关系**：对于前趋关系`Pi → Pj`，设置信号量`S`初值为0，`Pi`执行完后执行`signal(S)`，`Pj`执行前执行`wait(S)`

### 管程机制
- **定义**：管程是由一组数据以及定义在这组数据之上的对这组数据的操作组成的软件模块，这组操作能同步进程和改变管程中的数据。
- **特征**：
  1. 管程内部的数据只能被管程内部的过程访问
  2. 进程只能通过调用管程内的过程才能进入管程访问共享数据
  3. 每次仅允许一个进程在管程内执行某个内部过程
- **组成**：管程名称、共享数据结构说明、对数据结构操作的一组过程、对共享数据设置初始值的语句
- **条件变量**：管程中用于实现进程等待和唤醒的机制，每个条件变量对应一个等待队列，对条件变量的操作有`wait()`和`signal()`

## 2.5 经典进程的同步问题
### 生产者-消费者问题
- **问题描述**：多个生产者进程生产产品放入缓冲池，多个消费者进程从缓冲池中取走产品消费，缓冲池同一时间只能被一个进程访问
- **信号量实现**：
  ```c
  #define BUFFER_SIZE n  // 缓冲池大小
  semaphore mutex = 1;   // 互斥访问缓冲池的信号量
  semaphore empty = n;   // 空缓冲区数量
  semaphore full = 0;    // 满缓冲区数量
  int in = 0, out = 0;   // 缓冲池的输入和输出指针
  item buffer[BUFFER_SIZE];

  // 生产者进程
  void producer() {
    while(true) {
      item nextp = produce_item(); // 生产产品
      wait(empty);                 // 申请空缓冲区
      wait(mutex);                 // 进入临界区
      buffer[in] = nextp;          // 放入产品
      in = (in + 1) % BUFFER_SIZE;
      signal(mutex);               // 退出临界区
      signal(full);                // 增加满缓冲区数量
    }
  }

  // 消费者进程
  void consumer() {
    while(true) {
      wait(full);                  // 申请满缓冲区
      wait(mutex);                 // 进入临界区
      item nextc = buffer[out];    // 取出产品
      out = (out + 1) % BUFFER_SIZE;
      signal(mutex);               // 退出临界区
      signal(empty);               // 增加空缓冲区数量
      consume_item(nextc);         // 消费产品
    }
  }
  ```
  注意：`wait()`操作的顺序不能颠倒，必须先执行对资源信号量的`wait()`，再执行对互斥信号量的`wait()`，否则会引起死锁。

### 哲学家进餐问题
- **问题描述**：5个哲学家围坐在圆桌旁，相邻两个哲学家之间放一根筷子，哲学家交替进行思考和进餐，进餐需要同时获得左右两根筷子，吃完后放下筷子继续思考
- **信号量实现（避免死锁的方案）**：
  方案1：最多允许4个哲学家同时进餐，保证至少有一个哲学家能获得两根筷子
  方案2：仅当哲学家的左右两根筷子都可用时，才允许他拿起筷子
  方案3：规定奇数号哲学家先拿左边筷子，偶数号哲学家先拿右边筷子，避免循环等待
  ```c
  semaphore chopstick[5] = {1,1,1,1,1}; // 5根筷子，初始值1
  semaphore eating_count = 4;           // 最多4人同时进餐，方案1

  void philosopher(int i) {
    while(true) {
      think();
      wait(eating_count);               // 方案1：控制进餐人数
      wait(chopstick[i]);               // 拿左边筷子
      wait(chopstick[(i+1)%5]);         // 拿右边筷子
      eat();
      signal(chopstick[i]);             // 放下左边筷子
      signal(chopstick[(i+1)%5]);       // 放下右边筷子
      signal(eating_count);             // 方案1：释放名额
    }
  }
  ```

### 读者-写者问题
- **问题描述**：多个读者进程可以同时读共享文件，写者进程需要互斥访问共享文件（写时不允许其他读者或写者访问）
- **第一类读者-写者问题（读者优先）**：只要有读者在读，后续读者都可以进入读，写者必须等待所有读者读完才能写
  ```c
  semaphore rmutex = 1;  // 保护读者计数的互斥信号量
  semaphore wmutex = 1;  // 写操作的互斥信号量
  int readcount = 0;     // 正在读的读者数量

  // 读者进程
  void reader() {
    while(true) {
      wait(rmutex);
      if readcount == 0: wait(wmutex); // 第一个读者需要获取写互斥锁
      readcount++;
      signal(rmutex);

      read_file(); // 读文件

      wait(rmutex);
      readcount--;
      if readcount == 0: signal(wmutex); // 最后一个读者释放写互斥锁
      signal(rmutex);
    }
  }

  // 写者进程
  void writer() {
    while(true) {
      wait(wmutex);
      write_file(); // 写文件
      signal(wmutex);
    }
  }
  ```
- **第二类读者-写者问题（写者优先）**：一旦有写者等待，后续读者必须等待，优先满足写者的写请求

## 2.6 进程通信
- **进程通信的类型**：
  1. **共享存储器系统**：进程通过共享一个存储区进行通信，需要同步机制控制读写
     - 基于共享数据结构的通信（低级通信）
     - 基于共享存储区的通信（高级通信）
  2. **消息传递系统**：进程通过格式化的消息进行通信，是当前应用最广泛的进程通信机制
     - 直接通信：发送进程直接把消息发送给接收进程，挂在接收进程的消息缓冲队列上
     - 间接通信：发送进程把消息发送到中间实体（信箱），接收进程从信箱取消息
  3. **管道通信**：通过管道文件（pipe）实现进程之间的通信，管道是连接读进程和写进程的共享文件，以字符流方式传输数据
     - 管道需要提供三方面协调能力：互斥、同步、确定对方存在

### 消息缓冲队列通信机制
- **数据结构**：消息缓冲区、PCB中增加消息队列首指针、消息队列互斥信号量、消息队列资源信号量
- **原语**：`send(receiver, message)`发送消息原语、`receive(sender, message)`接收消息原语

### 信箱通信
- **信箱结构**：信箱头（信箱名称、大小、容量、拥有者等）、信箱体（存放消息的队列）
- **类型**：私用信箱、公用信箱、共享信箱
- **通信规则**：发送进程和接收进程都需要知道信箱名称，信箱满时发送阻塞，信箱空时接收阻塞

## 2.7 线程的基本概念
### 线程的引入
- **引入目的**：减少程序并发执行时的时空开销，使操作系统具有更好的并发性，进程是资源分配的单位，线程是调度和分派的基本单位
- **线程的属性**：
  1. 轻型实体：基本不拥有系统资源，只有一点必不可少的、能保证独立运行的资源（如TCB、程序计数器、寄存器、栈）
  2. 独立调度和分派的基本单位：线程切换的开销远小于进程切换
  3. 可并发执行：同一进程中的多个线程之间可以并发执行，不同进程中的线程也可以并发执行
  4. 共享进程资源：同一进程中的所有线程共享该进程所拥有的全部资源
- **线程与进程的比较**：
  | 比较维度 | 进程 | 线程 |
  |----------|------|------|
  | 调度单位 | 传统OS中进程是调度单位，引入线程后线程是调度单位 | 线程是独立调度和分派的基本单位 |
  | 资源拥有 | 进程是资源拥有的基本单位，拥有系统资源 | 线程基本不拥有资源，共享所属进程的资源 |
  | 并发性 | 进程之间可以并发执行 | 同一进程的线程之间、不同进程的线程之间都可以并发执行 |
  | 系统开销 | 进程切换开销大（需要切换上下文、资源映射等） | 线程切换开销小（仅需要切换少量寄存器） |

### 线程的状态
- 执行状态：线程正在CPU上运行
- 就绪状态：线程已经获得除CPU之外的所有资源，等待调度
- 阻塞状态：线程等待某事件发生而暂停运行
- 线程没有挂起状态，因为线程是进程内部的执行单元，进程被挂起时所有线程都被挂起，进程被激活时所有线程都被激活

## 2.8 线程的实现
### 内核支持线程（KST）
- **定义**：线程的创建、撤销、调度都由内核在内核空间实现，内核为每个内核线程维护上下文信息和状态
- **优点**：
  1. 内核可以同时调度同一进程中的多个线程到多个处理机上并行运行
  2. 如果一个进程中的某个线程被阻塞，内核可以调度该进程中的其他线程运行
  3. 内核支持线程本身只有很小的数据结构和栈，切换速度快，开销小
- **缺点**：用户线程的切换需要从用户态切换到内核态，开销较大

### 用户级线程（ULT）
- **定义**：线程的创建、撤销、调度和同步都在用户空间实现，内核完全感知不到用户级线程的存在，内核只对进程进行调度
- **优点**：
  1. 线程切换不需要内核支持，不需要切换到内核态，切换开销非常小
  2. 可以根据应用需要定制调度算法，不受OS调度的限制
  3. 与OS平台无关，用户级线程的实现可以在不支持线程的OS上运行
- **缺点**：
  1. 系统调用阻塞问题：如果一个线程执行系统调用被阻塞，那么整个进程都会被阻塞
  2. 多处理机系统中，内核一次只把一个进程分配给一个处理机，因此同一进程中的多个线程不能同时在多个处理机上并行运行

### 组合方式
- **轻量级进程（LWP）**：内核支持的用户线程，是用户级线程和内核线程之间的中间层，一个进程可以有多个LWP，每个LWP对应一个内核线程，用户级线程可以映射到LWP上运行
