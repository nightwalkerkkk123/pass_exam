# 第一部分：第5章 设备管理

## 5.1 I/O系统

### 5.1.1 I/O设备的类型

**按传输速率分类：**

| 类型 | 速率范围 | 示例 |
|------|----------|------|
| **低速设备** | < 1 KB/s | 键盘、鼠标 |
| **中速设备** | 1~100 KB/s | 打印机 |
| **高速设备** | > 100 KB/s | 磁盘机 |

**按信息交换单位分类：**
- **块设备**：以数据块为单位存取，如磁盘。特点是传输速率高、可寻址
- **字符设备**：用于无结构数据I/O，如键盘、打印机。特点是传输速率低、不可寻址

**按共享属性分类：**
- **独占设备**：一段时间只允许一个用户使用（如打印机）
- **共享设备**：可同时允许多个用户使用（如磁盘）
- **虚拟设备**：通过虚拟技术将一台物理设备变换为多个逻辑设备

**按外部特征分类：**
- 存储设备（磁盘、磁带）、输入设备（键盘）、输出设备（打印机）

### 5.1.2 设备控制器

**设备控制器**是CPU与I/O设备之间的接口：
- 用地址（数据寄存器、状态寄存器、控制寄存器）进行识别
- 一个设备控制器可挂接多个设备

**设备控制器的基本功能：**
1. 接收和识别命令
2. 标识和报告设备状态
3. 数据交换
4. 地址识别
5. 数据缓冲
6. 差错控制

**设备与控制器之间的接口信号：**
- 数据信号线、控制信号线、状态信号线

### 5.1.3 I/O通道

**I/O通道**是一种特殊的处理机，具有执行I/O指令的能力，通过执行通道程序控制I/O操作。

与一般处理机的区别：
1. 指令类型单一（局限于I/O操作相关指令）
2. 没有自己的内存，通道程序存放在主机内存中（与CPU共享内存）

**通道类型：**

| 类型 | 特点 | 适用场景 |
|------|------|----------|
| **字节多路通道** | 分时为多个设备服务，每次传送一个字节 | 连接大量低速设备 |
| **数组选择通道** | 只有一个分配型子通道，一次控制一台设备 | 高速设备，但利用率低 |
| **数组多路通道** | 多个非分配型子通道，传输速率高且通道利用率高 | 多台高中速设备 |

**"瓶颈"问题：** 单通路I/O系统中通道数量少会成为系统瓶颈，可通过**多通路I/O系统**解决。

### 5.1.4 总线系统

- **ISA总线**：16位，最高传输速率16 MB/s
- **EISA总线**：32位，传输速率32 MB/s
- **PCI总线**：局部总线，高速外设连接

## 5.2 I/O控制方式

### 5.2.1 程序直接控制方式（程序I/O方式）

- CPU不断轮询I/O设备状态，处于忙等待循环
- 由于CPU高速性与I/O设备低速性矛盾，CPU绝大部分时间处于等待状态
- **缺点**：CPU利用率极低

### 5.2.2 中断驱动I/O控制方式

- I/O设备输入每个数据的过程中无须CPU干预，CPU与I/O设备并行工作
- 仅当输完一个数据时，才需CPU做中断处理
- **优点**：CPU和I/O设备都处于忙碌状态，提高了系统资源利用率
- 例如：终端输入一个字符约100ms，送入缓冲区小于0.1ms，CPU利用率提高约1000倍

### 5.2.3 DMA（直接存储器访问）控制方式

**特点：**
1. 数据传输基本单位是**数据块**（每次至少一个数据块）
2. 数据从设备直接送入内存（或相反），不经CPU
3. 仅在传送开始和结束时需CPU干预

**DMA控制器的四类寄存器：**

| 寄存器 | 缩写 | 功能 |
|--------|------|------|
| 命令/状态寄存器 | CR | 接收I/O命令、控制信息和设备状态 |
| 内存地址寄存器 | MAR | 存放内存起始地址（输入为目标地址，输出为源地址） |
| 数据寄存器 | DR | 暂存传输数据 |
| 数据计数器 | DC | 存放本次要读写的字（节）数 |

**DMA数据传送三阶段：**
1. **预处理阶段**：CPU将内存起始地址、传送字节数送入DMAC，启动设备
2. **数据传送阶段**：DMAC窃取CPU工作周期，执行数据传送（地址修改、计数）
3. **后处理阶段**：DMAC通过中断请求线发中断信号，CPU做后续处理

### 5.2.4 I/O通道控制方式

- DMA方式的发展，进一步减少CPU干预
- 把对一个数据块的读写干预，减少为对**一组数据块**的读写及控制管理干预
- 实现**CPU、通道和I/O设备三者并行**操作
- CPU只需发送一条I/O指令，给出通道程序首址和要访问的设备

**通道指令**：包含操作码、内存地址、计数、通道程序结束位P、记录结束标志R

**通道工作原理：**
1. CPU将通道程序首地址放入**通道地址字(CAW)**
2. 执行"启动I/O"指令
3. 通道从CAW取首地址，取出第一条指令放入**通道命令字(CCW)**
4. 通道执行通道程序，与CPU并行完成数据传输
5. 完成后向CPU发I/O中断，CPU从**通道状态字(CSW)**获取状态信息

> **复习要点**：四种I/O控制方式对比
> | 方式 | CPU干预频率 | 数据单位 | 并行性 |
> |------|------------|----------|--------|
> | 程序I/O | 每个字符 | 字/字节 | 无 |
> | 中断驱动 | 每个数据 | 字/字节 | CPU与设备 |
> | DMA | 每个数据块 | 数据块 | CPU与设备 |
> | I/O通道 | 一组数据块 | 数据块组 | CPU、通道、设备三者 |

## 5.3 缓冲管理

### 5.3.1 缓冲的引入

**引入缓冲的原因：**
1. 缓和CPU与I/O设备间速度不匹配的矛盾
2. 减少对CPU的中断频率，放宽中断响应时间限制
3. 提高CPU和I/O设备之间的并行性

### 5.3.2 单缓冲

- 每当用户进程发出I/O请求时，系统分配一个缓冲区
- 处理一个数据时间：`T + M + C`（T=输入时间，M=传送时间，C=计算时间）
- 连续处理时：`max(C, T) + M`

### 5.3.3 双缓冲

- 设置两个缓冲区A和B，交替使用
- 输入与传送/计算可并行
- 连续处理一个数据时间：`max(T, M + C)`
- 当输入和传送速度相匹配时效果最佳

### 5.3.4 循环缓冲

- 多个缓冲区组织成**循环队列**
- 输入进程不断向空缓冲区R输入数据
- 计算进程从满缓冲区G中提取数据
- 需要同步机制防止指针追赶冲突

### 5.3.5 缓冲池（Buffer Pool）

**三个队列：**
- **emq**：空缓冲区队列
- **inq**：输入缓冲区队列
- **outq**：输出缓冲区队列

**四种工作缓冲区：**
1. **hin**：收容输入数据的工作缓冲区
2. **sin**：提取输入数据的工作缓冲区
3. **hout**：收容输出数据的工作缓冲区
4. **sout**：提取输出数据的工作缓冲区

**四种工作方式：**
1. 收容输入：设备 → hin → inq
2. 提取输入：inq → sin → 用户区
3. 收容输出：用户区 → hout → outq
4. 提取输出：outq → sout → 设备

## 5.4 设备分配

### 5.4.1 设备分配中的数据结构

| 数据结构 | 缩写 | 作用 |
|----------|------|------|
| 设备控制表 | DCT | 每个设备一张，记录设备状态 |
| 控制器控制表 | COCT | 每个控制器一张 |
| 通道控制表 | CHCT | 每个通道一张 |
| 系统设备表 | SDT | 整个系统一张，记录所有设备 |

### 5.4.2 设备分配应考虑的因素

**1. 设备固有属性：**
- 独享设备：分配后独占使用
- 共享设备：可同时分配给多个进程
- 虚拟设备：通过SPOOLing技术实现

**2. 设备分配算法：**
- **先来先服务(FCFS)**
- **优先级高者优先**

**3. 安全性：**
- **安全分配方式**：进程发出I/O请求后进入阻塞，直到I/O完成才被唤醒（不会死锁，但进程缓慢）
- **不安全分配方式**：进程发出I/O请求后继续运行（可能死锁，但进程推进快）

### 5.4.3 设备独立性

**设备独立性**：应用程序独立于具体使用的物理设备。

- 用户采用**逻辑设备名**使用设备
- 系统实现逻辑设备到物理设备的映射

**好处：**
1. 设备分配灵活
2. 易于实现I/O重定向

**逻辑设备表(LUT)**：实现逻辑设备名到物理设备名的映射

### 5.4.4 独占设备的分配程序

**分配步骤：**
1. 分配设备（查SDT → DCT）
2. 分配控制器（查COCT）
3. 分配通道（查CHCT）

**改进：**
- 增加设备独立性（用逻辑设备名请求I/O）
- 考虑多通路情况（查找所有控制器和通道）

### 5.4.5 SPOOLing技术（假脱机操作）

**SPOOLing(Simultaneous Peripheral Operations On-Line)**：由系统中两个专门负责I/O的进程模拟I/O外围机功能，实现假脱机输入/输出。

**系统组成：**
1. **输入井和输出井**：磁盘上开辟的两个大存储空间
   - 输入井：收容I/O设备输入的数据
   - 输出井：收容用户进程的输出数据
2. **输入缓冲区和输出缓冲区**：内存中开辟的暂存区
3. **输入进程SPi和输出进程SPo**：分别模拟脱机输入/输出时的外围控制机

**系统特点：**
- 提高I/O速度
- 将**独占设备改造为共享设备**
- 实现**虚拟设备**功能

**SPOOLing使一台打印机虚拟成多台打印机的原理：**
- 用户打印请求将数据输出到输出井
- 打印请求登记后排到打印队列
- 打印机空闲时取队列首请求，从输出井传送到内存缓冲区进行打印

## 5.5 设备处理（设备驱动程序）

### 5.5.1 设备驱动程序的功能和特点

**功能：**
- 接收独立性软件的抽象要求（read/write），转换为具体要求
- 检查用户I/O请求的合法性，检查设备状态
- 发出I/O命令，启动设备
- 响应中断请求并处理

**特点：**
- 介于用户与设备之间的通信程序
- 与I/O设备紧密相关
- 与I/O控制方式紧密相关
- 部分程序用汇编编写

### 5.5.2 设备驱动程序的处理过程

1. 将抽象要求转换为具体要求
2. 检查I/O请求的合法性
3. 读出和检查设备状态
4. 传送必要的参数
5. 工作方式设置
6. 启动I/O设备

## 5.6 磁盘存储器管理

### 5.6.1 磁盘结构

**磁盘数据组织：**
- 磁盘由多个盘片组成，每盘片分两面
- 每盘面分为若干个**磁道（track）**，每磁道分为若干个**扇区（sector）**
- **柱面（cylinder）**：处于相同磁道位置但在不同盘面上的磁道集合
- 内磁道数据密度大于外磁道

**磁盘容量计算：**
```
硬盘容量 = 每扇区字节数(512B) x 扇区数/道 x 柱面数 x 磁头数
```

**磁盘类型：**
- **固定头磁盘**：每个磁道上都有一个磁头
- **移动头磁盘**：一个盘面仅有一个磁头

### 5.6.2 磁盘访问时间

**总访问时间：** `Ta = Ts + Tr + Tt`

| 时间类型 | 符号 | 说明 | 公式 |
|----------|------|------|------|
| **寻道时间** | Ts | 磁头移至指定磁道 | `Ts = m x n + s`（s:启动时间，m:每道时间，n:距离） |
| **旋转延迟** | Tr | 磁头移至指定扇区 | `Tr = 1/(2r)`（r:转速，约5.55ms@90r/s） |
| **传输时间** | Tt | 数据读出/写入 | `Tt = b/(rN)`（b:字节数，N:单磁道字节数） |

> **复习要点**：寻道时间是影响磁盘性能的主要因素。将同一文件的数据块集中存放（同一柱面）可降低访问时间。

### 5.6.3 磁盘调度算法

> **复习要点**：磁盘调度是期末重点！重点掌握各算法的原理、优缺点和计算。

**1. 先来先服务（FCFS）**
- 按请求访问磁盘的先后次序调度
- **优点**：简单公平，每个请求依次处理
- **缺点**：平均寻道距离大，性能差

**2. 最短寻道时间优先（SSTF）**
- 优先访问与当前磁头位置最近的磁道
- **优点**：平均寻道时间较短
- **缺点**：可能导致某些进程"饥饿"（磁臂粘着）

**3. 扫描算法（SCAN / 电梯算法）**
- 按磁头移动方向及距离最短两个原则
- 磁头来回扫描（如：向外→向内→向外→...）
- **优点**：不会出现饥饿，平均寻道时间较短
- **缺点**：与磁头近但在反方向的磁道等待时间长

**4. 循环扫描算法（C-SCAN）**
- 到达最外磁道后，返回最小磁道重新开始
- 只单向服务（如只向外扫描，到达末端后直接回起点）
- **优点**：不会出现饥饿，平均寻道时间较短，**最长等待时间为SCAN的一半**
- **缺点**：仍有磁臂粘着问题

**5. LOOK算法**
- SCAN的改进：只走到最远的请求磁道就折返
- C-SCAN对应的改进为C-LOOK

**6. N-Step-SCAN算法**
- 将请求分成若干个长度为N的子队列
- 子队列间按FCFS，子队列内按SCAN
- N=1时退化为FCFS，N=∞时退化为SCAN

**7. FSCAN算法**
- 将请求分成两个队列：当前申请队列和扫描期间申请队列
- 用SCAN处理当前队列，扫描期间的新请求放入第二个队列

### 5.6.4 提高磁盘I/O速度的方法

**1. 磁盘高速缓存**
- 利用内存空间暂存从磁盘读出的盘块信息
- 形式：单独存储空间（固定大小）或利用未用内存（动态变化）
- **置换算法**：LRU、NRU(Clock)、FIFO
- **周期性写回磁盘**（如每30秒），防止数据丢失

**2. 数据交付方式**
- **数据交付**：将数据传送到进程指定内存区
- **指针交付**：将数据位置交付给进程

**3. 其他方法**
- **提前读**：依据局部性原理，将邻近盘块提前读入内存
- **延迟写**：被修改的页面不立即写回，挂在空闲缓冲区链尾
- **优化物理块分布**：将相邻盘块分配给同一文件，以簇为单位分配
- **虚拟盘（RAM盘）**：利用内存仿真磁盘

### 5.6.5 RAID（廉价磁盘冗余阵列）

**并行交叉存取**：将一簇中不同盘块内容分别放在不同磁盘的相同位置。

| RAID级别 | 特点 | 效率 | 缺点 |
|----------|------|------|------|
| **RAID 0** | 仅并行交叉存取 | 100% | 无容错 |
| **RAID 1** | 并行+磁盘镜像 | 50% | 效率低 |
| **RAID 3** | 并行+专用校验盘 | (N-1)/N | 校验盘成瓶颈 |
| **RAID 5** | 并行+校验信息散布在所有盘 | (N-1)/N | 写操作复杂 |

**RAID 5优点**：可靠性高、磁盘I/O速度高（N-1倍）、性能/价格比高。

# 第二部分：第6章 文件管理

## 6.1 文件和文件系统

### 6.1.1 文件的概念

**文件**：具有文件名的一组相关信息的集合。

- 文件具有**可保存性**：便于信息长期保存
- 用户**按名存取**：不需知道文件在磁盘的具体位置

### 6.1.2 文件类型

**有结构文件（记录式文件）**：
- 由**记录**组成，记录是一组相关数据项的集合
- 数据项：描述对象某种属性的字符集

**无结构文件（流式文件）**：
- 直接由字符/字节组成
- Unix、DOS、Windows等都采用流式文件

### 6.1.3 文件系统的功能

1. 文件存储空间的分配与回收（磁盘管理）
2. 文件名到文件空间的映射
3. 提供文件共享、保护与保密措施
4. 实现用户要求的各种文件操作

**文件系统接口：**
- **命令接口**：dir/ls等
- **程序接口**：open, read, write, close等系统调用

---

## 6.2 文件的逻辑结构

### 6.2.1 文件逻辑结构的类型

| 类型 | 子类型 | 特点 |
|------|--------|------|
| **有结构文件** | 顺序文件 | 记录按顺序排列 |
| | 索引文件 | 为可变长记录建立索引表 |
| | 索引顺序文件 | 分组索引，兼顾效率和空间 |
| **无结构文件** | 流式文件 | 直接由字符/字节组成 |

### 6.2.2 顺序文件

**排序方式：**
- **串结构**：各记录顺序与关键字无关
- **顺序结构**：记录按关键字排列

**读写方式：**
- 定长记录：直接定位
- 变长记录：逐个顺序读写

**缺点：**
- 对单个记录读写困难（尤其变长记录）
- 增加或删除记录困难

### 6.2.3 索引文件

- 为可变长记录文件建立一张**索引表**
- 索引表按记录键排序，本身构成定长记录的顺序文件
- 支持高效直接存取

### 6.2.4 索引顺序文件

- 将记录分为若干组
- 为每组第一个记录建立索引项
- 可建立多级索引（二级、三级等）

**查找效率对比（100万条记录）：**
- 顺序文件：平均50万次
- 索引文件：需100万个索引空间
- 索引顺序文件（二级索引）：约150次

---

## 6.3 外存分配方式

### 6.3.1 连续分配

- 为文件分配一组相邻接的盘块
- **优点**：顺序访问容易，一次大批量访问速度快，可通过计算直接定位：`J = ADDR + I`
- **缺点**：要求连续外存空间（存在碎片），必须事先知道文件长度

### 6.3.2 链接分配

**隐式链接：**
- 链接指针放在盘块数据区中
- **优点**：无碎片，不需连续空间，不需事先知道文件长度
- **缺点**：只适合顺序访问（随机访问低效），可靠性差（指针出错则链断开）

**显式链接（FAT）：**
- 链接指针存放在**文件分配表(FAT)**中
- **优点**：FAT可调入内存，查找速度快，支持随机存取
- **缺点**：FAT占用外存和内存空间

### 6.3.3 索引分配

**单级索引：**
- 每个文件分配一个索引块，存放盘块号
- **优点**：方便直接存取
- **缺点**：索引块可能装不满，大文件效率仍低

**两级索引：**
- 索引块中存放下一级索引块的盘块号
- 最大文件空间：256 x 256 = 64K盘块 = 64MB（盘块1KB，盘块号4B）

**混合索引（UNIX方式）：**

| 索引类型 | 说明 | 最大文件大小（4KB块） |
|----------|------|----------------------|
| 直接地址（10项） | 直接存放盘块号 | 40 KB |
| 一次间接地址 | 单级索引 | 40KB + 4MB = 4.04MB |
| 二次间接地址 | 二级索引 | 4.04MB + 4GB |
| 三次间接地址 | 三级索引 | 4GB + 4TB |

---

## 6.4 文件目录

> **复习要点**：文件目录结构是期末重点！

### 6.4.1 文件控制块（FCB）

**FCB**是描述和控制文件的数据结构，与文件一一对应，是目录项的组成部分。

**FCB包含信息：**

| 类别 | 内容 |
|------|------|
| **基本信息** | 文件名、物理位置、逻辑结构、物理结构 |
| **存取控制信息** | 文件主权限、核准用户权限、一般用户权限 |
| **使用信息** | 建立时间、最后修改/访问时间、当前使用信息 |

### 6.4.2 索引结点（Inode）

**引入原因：**
- FCB太大（64B），单个盘块存储FCB数量少
- 查找文件时需要访问多个盘块

**改进方法：**
- 将文件名与FCB其他描述信息分开存放
- 文件名+索引结点指针组成目录项（16B：14B文件名+2B指针）
- FCB其他描述信息组合成**索引结点**数据结构

**效果：** 1KB盘块可存放64个目录项，查找效率提高4倍。

### 6.4.3 目录结构

**1. 单级目录**
- 整个系统一张目录表，每个文件一个FCB
- **优点**：简单
- **缺点**：查找慢、不允许重名、不便于共享

**2. 两级目录**
- 主文件目录(MFD) + 用户文件目录(UFD)
- **优点**：提高检索速度、允许不同用户目录同名、支持共享
- **缺点**：用户不能建立子目录

**3. 树型目录结构**
- 具有三级或三级以上的目录结构
- 允许在任何子目录中再创建新子目录
- **路径名**：从根目录到文件经过的所有目录及文件名用"/"连接
- **当前目录（工作目录）**：用户当前操作的目录

**4. 图形目录结构**
- 在树型基础上增加共享链接，形成有向无环图

### 6.4.4 目录查询技术

1. 按名查询文件FCB或索引结点
2. 依据物理地址（盘块号）换算物理位置
3. 启动磁盘驱动程序读取文件

## 6.5 文件存储空间的管理

### 6.5.1 空闲表法

- 为外存上所有空闲区建立一张空闲表（连续分配方式）
- **分配算法**：首次适应、循环首次适应、最佳适应、最坏适应
- **回收**：对邻接盘区进行拼接
- 主要用于对换区分配

### 6.5.2 空闲链表法

**空闲盘块链表：**
- 所有空闲盘块连成链表
- **优点**：分配回收操作简单
- **缺点**：链表可能很长，指针出错导致空闲盘块丢失

**空闲盘区链表：**
- 空闲盘区（多个连续盘块）连成链表
- **优点**：链表较短
- **缺点**：分配回收操作较复杂

### 6.5.3 位示图法

- 用二进制的一位表示一个盘块的使用情况（0=空闲，1=已分配）
- **盘块分配**：
  1. 扫描位示图找到值为0的位
  2. 转换为盘块号：`n = i x 8 + j`（第i字节，第j位）
  3. 修改位示图对应位为1
- **盘块回收**：
  1. 盘块号转位置：`i = n / 8, j = n mod 8`
  2. 修改对应位为0
- **优点**：占用空间小可调入内存，分配回收操作简单有效

### 6.5.4 成组链接法

- 将空闲盘块每N个（如100个）组成一组
- 空闲盘块组连成链表
- **空闲盘块号栈**：存放当前可用的一组空闲盘块号

**盘块分配：**
1. 若`s.free = 1`，将栈底盘块内容调入空闲盘块号栈，再分配该盘块
2. 否则，直接分配栈底盘块，`s.free`递减

**盘块回收：**
1. 若`s.free = N`，将栈内容写入新释放盘块，`s.free = 1`，该盘块作为栈底
2. 否则，`s.free`递增，该盘块入栈

## 6.6 文件共享与文件保护

### 6.6.1 基于索引结点的共享方式（硬链接）

- 目录项只包含文件名及指向索引结点的指针
- 共享同一文件的不同目录项，文件名不同但指向同一索引结点
- 设置**链接计数器**，指示共享目录项数目
- **优点**：任意用户对文件的修改其他共享者都可见
- **缺点**：文件主删除文件后若有其他用户共享，文件仍占用空间

### 6.6.2 基于符号链的共享方式（软链接）

- 符号链是指向文件所在目录项位置的指针（路径名或IP+路径）
- 文件主可完全删除文件，共享者访问时出错误
- **优点**：可共享网络上任何地点的文件
- **缺点**：访问速度较慢（沿链查找）

### 6.6.3 文件保护

**访问控制：**
- 文件主权限（只读、读写、执行）
- 核准用户权限
- 一般用户权限

**磁盘容错技术：**

| 级别 | 名称 | 技术 |
|------|------|------|
| SFT-I | 第一级容错 | 双份目录/FAT、热修复重定向、写后读校验 |
| SFT-II | 第二级容错 | 磁盘镜像、磁盘双工 |

### 6.6.4 数据一致性控制

**事务**：用于访问和修改数据项的程序单位，具有**原子性**。

**事务记录**：存放在稳定存储器，包含事务名、数据项名、旧值、新值。

**恢复算法**：
- **Undo(Ti)**：将事务Ti修改的数据恢复为旧值
- **Redo(Ti)**：将事务Ti修改的数据置为新值

**并发控制**：互斥锁、信号量机制实现顺序性。

## 2. 操作系统接口

### 2.1 进程与内存

- 进程由**用户空间内存**（指令、数据、堆栈）和**内核私有进程状态**组成
- xv6进程共享CPU，透明切换
- `fork()`：创建子进程，内存内容与父进程相同
- `exec()`：用新内存映像替换当前进程内存
- `sbrk(n)`：进程运行时增加内存

### 2.2 文件描述符

- 文件描述符是**小整数**，代表内核管理对象
- 每个进程有私有文件描述符空间（从0开始）
- **标准约定**：0=标准输入，1=标准输出，2=标准错误
- `read/write`操作有偏移量，顺序推进
- `fork`复制的文件描述符共享偏移量，`dup`也共享偏移量

**I/O重定向原理：**
```c
if (fork() == 0) {
    close(0);                    // 释放标准输入
    open("input.txt", O_RDONLY); // 获得fd=0
    exec("cat", argv);           // cat从input.txt读取
}
```

### 2.3 管道

- 管道是**小的内核缓冲区**，一对文件描述符（一读一写）
- 提供进程间通信方式
- 管道的四个优势：自动清理、可传任意长数据流、可分段并行执行、阻塞读写更高效

### 2.4 文件系统

- **数据文件**：拥有字节数组
- **目录**：拥有对数据文件和其他目录的命名引用
- 目录形成树，从**根目录**开始
- **inode**：底层文件，保存metadata（类型、长度、磁盘位置、链接数）
- `link()`创建硬链接（同一inode多个名称），`unlink()`删除文件名

## 3. 操作系统组织

### 3.1 宏内核 vs 微内核

| 特性 | 宏内核 | 微内核 |
|------|--------|--------|
| 设计 | 整个操作系统在内核中运行 | 尽量减少内核代码，大部分在用户级服务器 |
| 权限 | 全硬件权限 | 仅基本功能在内核 |
| 优点 | 方便，各部分易合作 | 内核简单，bug影响小 |
| 缺点 | bug致命（内核崩溃则全崩） | 进程间通信复杂 |
| 代表 | xv6、Linux | L4、Minix、QNX |

### 3.2 用户模式与监督者模式

- **RISC-V三种模式**：机器模式、监督者(supervisor)模式、用户模式
- **用户模式**：只能执行用户指令（如数字相加），运行用户空间程序
- **监督者模式**：可执行特权指令（开关中断、读写页表等），运行内核
- **ecall指令**：从用户模式切换到监督者模式，进入内核
- **sret指令**：从内核返回用户空间

### 3.3 进程结构

**进程抽象实现隔离的机制：**
- 用户/监督模式标志
- 地址空间
- 线程的时间分割

**进程内存布局（用户空间）：**
```
低地址 ┌──────────┐
       │  指令    │
       ├──────────┤
       │ 全局变量 │
       ├──────────┤
       │   栈     │ ← 向下增长
       ├──────────┤
       │   堆     │ ← 向上增长（sbrk扩展）
       │          │
       ├──────────┤
       │trapframe │ ← 用户页表映射
       ├──────────┤
高地址 │trampoline│ ← 用户/内核页表同地址映射
       └──────────┘
       MAXVA = 0x3fffffffff
```

**进程关键状态（proc结构体）：**
- `p->pagetable`：进程页表
- `p->kstack`：内核栈
- `p->state`：进程状态（CREATED/READY/RUNNING/WAITING/EXITED）

**进程有两个栈：**
- **用户栈**：执行用户指令时使用
- **内核栈**：进入内核时使用（系统调用或中断）

## 4. 页表

### 4.1 Sv39 RISC-V分页机制

- 使用64位虚拟地址的**低39位**，高25位未使用
- 页表逻辑上是`2^27`（约1.34亿）个**页表项(PTE)**的数组
- 每个PTE包含44位**物理页号(PPN)**和标志位
- **页大小**：4096字节(2^12)

**PTE标志位：**

| 标志位 | 含义 |
|--------|------|
| `PTE_V` | PTE是否有效 |
| `PTE_R` | 是否可读 |
| `PTE_W` | 是否可写 |
| `PTE_X` | 是否可执行 |
| `PTE_U` | 用户模式是否可访问 |

### 4.2 三级页表

页表以**三层树**形式存储：
1. **根页表页**（4096字节，512个PTE）
2. **二级页表页**（512个PTE）
3. **三级页表页**（512个PTE指向物理页）

虚拟地址39位划分：
- 高9位：选择根页表页中的PTE
- 中9位：选择二级页表页中的PTE
- 低9位：选择三级页表页中的PTE
- 低12位：页内偏移

### 4.3 地址空间

**内核地址空间布局：**
- **直接映射**：虚拟地址 = 物理地址（KERNBASE=0x80000000到PHYSTOP=0x86400000）
- **trampoline页**：映射在虚拟地址空间顶端，用户/内核页表共享
- **内核栈页**：映射在高地址处，下方有未映射的守护页（防止栈溢出）

**关键代码文件**：`kernel/vm.c`
- `walk()`：模拟分页硬件查找虚拟地址的PTE
- `mappages()`：将虚拟地址映射到物理地址
- `kvm*`函数：操作内核页表
- `uvm*`函数：操作用户页表

### 4.4 物理内存分配

- 使用**内核地址结束到PHYSTOP**之间的物理内存进行运行时分配
- 每次分配/释放**4096字节**的整页
- 通过**空闲页链表**记录空闲页

**关键代码**：`kernel/kalloc.c`
- `kalloc()`：从空闲链表头部pop一页
- `kfree()`：将页push到空闲链表
- 空闲页的`run`结构体存储在空闲页本身

## 5. Trap与系统调用

### 5.1 Trap类型

三种事件导致CPU转移控制权：
1. **系统调用**：用户程序执行`ecall`指令
2. **异常**：指令做了非法操作（除零、无效地址等）
3. **设备中断**：设备发出注意信号

### 5.2 RISC-V Trap机制

**关键控制寄存器：**

| 寄存器 | 作用 |
|--------|------|
| `stvec` | trap处理程序地址 |
| `sepc` | trap发生时保存的程序计数器 |
| `scause` | trap原因编号 |
| `sscratch` | 方便trap恢复/保存用户上下文的值 |
| `sstatus` | SIE位控制中断使能，SPP位表示来源模式 |

**Trap硬件执行步骤：**
1. 如果是设备中断且SIE=0，不处理
2. 清除SIE禁用中断
3. 复制PC到sepc
4. 保存当前模式到SPP位
5. 在scause设置原因
6. 转换为监督者模式
7. 将stvec复制到PC

### 5.3 用户空间Trap处理

**路径**：`uservec` → `usertrap` → 处理 → `usertrapret` → `userret`

**trampoline页机制：**
- 用户页表不映射内核，但trap时需要切换页表
- xv6在内核页表和每个用户页表的**相同虚拟地址**上映射trampoline页
- `uservec`（在trampoline页中）保存寄存器到trapframe，切换satp到内核页表

**trapframe**：保存所有用户寄存器，包含内核栈指针、hartid、usertrap地址、内核页表地址。

### 5.4 系统调用实现

1. 用户代码将参数放寄存器`a0/a1`，系统调用号放`a7`
2. `ecall`进入内核
3. `syscall()`从trapframe的`a7`获取系统调用号
4. 在`syscalls`数组中查找并执行对应函数
5. 返回值记录在`p->trapframe->a0`

## 6. 中断与设备驱动

### 6.1 驱动程序结构

驱动在两个**上下文**执行代码：
- **上半部分(top half)**：在进程内核线程中运行（通过系统调用如read/write）
- **下半部分(bottom half)**：在中断时执行（中断处理程序）

### 6.2 UART控制台驱动

- 通过**UART串行端口硬件**接受输入字符
- UART硬件是一组**内存映射控制寄存器**
- `consoleinit()`配置UART：接收输入时产生接收中断，完成发送时产生传输完成中断

**控制台输入流程：**
1. 用户按键 → UART硬件产生中断
2. trap处理 → `devintr()` → `uartintr()`
3. `uartintr()`读取字符 → `consoleintr()`累积到`cons.buf`
4. 新行到达 → `wakeup`唤醒`consoleread`
5. `consoleread`将输入复制到用户空间

**控制台输出流程：**
1. `uartputc`追加字符到输出缓冲区
2. `uartstart`启动设备发送
3. 发送完成产生中断 → `uartintr`调用`uartstart`发送下一个字符

### 6.3 并发问题

驱动中的三个并发风险：
1. 两个进程同时调用`consoleread`
2. 硬件在执行`consoleread`时抛出中断
3. 硬件在`consoleread`执行时向另一个CPU抛中断

**解决**：使用锁保护驱动数据结构。

### 6.4 定时器中断

- xv6使用定时器中断维护时钟、实现进程切换
- RISC-V要求在**机器模式**下处理定时器中断
- `timervec`：机器模式定时器中断向量
- 策略：处理程序要求RISC-V引发**软件中断**并立即返回，再用普通trap机制处理

## 7. 锁

### 7.1 竞争条件

**竞争条件**：同时访问一个内存位置，且至少有一次是写操作。

**丢失更新示例**（链表push）：
```c
l->next = list;   // 两个CPU同时执行
list = l;         // 第二次赋值覆盖第一次
```

### 7.2 自旋锁

**xv6自旋锁结构**：`spinlock`，核心字段`locked`（0=可获得，非0=被持有）

**原子操作**：RISC-V的`amoswap`指令，原子地交换寄存器和内存值。

```c
void acquire(struct spinlock *lk) {
    while(__sync_lock_test_and_set(&lk->locked, 1) != 0)
        ; // 自旋等待
    lk->cpu = mycpu(); // 记录获取锁的CPU
}

void release(struct spinlock *lk) {
    lk->cpu = 0;
    __sync_lock_release(&lk->locked);
}
```

**使用原则**：
1. 一个CPU写入时另一CPU读写，必须用锁
2. 涉及多个内存位置的invariant，用同一个锁保护
3. 不要过度使用锁（降低并行性）

### 7.3 死锁避免

**死锁条件**：多个代码路径以不同顺序获取多个锁。

**避免方法**：所有代码路径以**相同顺序**获取锁。

**xv6锁顺序示例**：
- `cons.lock`必须在任何进程锁之前获取
- 文件系统：目录锁 → 新文件inode锁 → 磁盘块缓冲区锁 → 磁盘驱动锁 → 进程锁

### 7.4 锁与中断

**危险场景**：线程持有锁L时被中断，中断处理程序也试图获取L → 死锁。

**xv6解决方案**：获取任何锁时，**禁用该CPU上的中断**。

### 7.5 睡眠锁

**自旋锁缺点**：
- 长时间等待浪费CPU
- 持有自旋锁时不能让出CPU

**睡眠锁**：
- `acquiresleep`等待时让出CPU
- 使用spinlock保护locked字段
- `sleep`调用原子性地让出CPU并释放spinlock
- **限制**：不能在中断处理程序中使用，不能在spinlock临界区内使用

**适用场景**：
- 自旋锁：短临界区
- 睡眠锁：长时间操作（如磁盘I/O）

## 8. 调度

### 8.1 复用实现

xv6在两种情况下切换CPU：
1. `sleep/wakeup`：进程等待设备/管道I/O、子进程退出
2. **周期强制切换**：应对长时间计算进程（定时器中断）

### 8.2 上下文切换

**切换步骤（用户进程A → 用户进程B）：**
1. 用户内核转换（系统调用或中断）→ A的内核线程
2. `swtch`保存A的上下文 → CPU调度器线程
3. `swtch`从调度器恢复B的上下文 → B的内核线程
4. trap返回到B的用户空间

**`swtch(struct context *old, struct context *new)`**：
- 保存callee-saved寄存器到old
- 从new恢复寄存器
- 返回时在新线程的栈上执行

**调度器**：每个CPU一个专门的调度器线程（保存寄存器和栈），运行`scheduler()`函数。

### 8.3 调度器实现

```c
void scheduler() {
    for(;;) {
        // 找可运行进程
        for(p = proc; p < &proc[NPROC]; p++) {
            if(p->state == RUNNABLE) {
                p->state = RUNNING;
                c->proc = p;
                swtch(&c->context, &p->context);
                c->proc = 0;
            }
        }
    }
}
```

- 简单**轮询调度(round robin)**
- 找到`RUNNABLE`进程 → 设为`RUNNING` → `swtch`运行它

### 8.4 Sleep与Wakeup

**丢失唤醒问题**：
```c
// 错误版本：P在检查条件和sleep之间，V可能执行wakeup
while(s->count == 0)
    sleep(chan);  // V在此间隙执行，wakeup丢失
```

**正确方案**：`sleep(chan, lock)` - 调用者传递条件锁
- sleep原子性地：标记SLEEPING → 释放条件锁 → 让出CPU
- wakeup在持有条件锁时调用，确保不会错过sleep的进程

### 8.5 管道实现

**管道结构**：
```c
struct pipe {
    struct spinlock lock;
    char buf[PIPESIZE];
    uint nread;   // 已读取字节数
    uint nwrite;  // 已写入字节数
};
```

- 缓冲区环形：`buf[PIPESIZE-1]`之后是`buf[0]`
- `nwrite == nread`：空缓冲区
- `nwrite == nread + PIPESIZE`：满缓冲区
- 使用**两个sleep chan**（`&pi->nread`和`&pi->nwrite`）提高效率

### 8.6 Wait、Exit与Kill

**Exit：**
- 记录退出状态，释放资源
- 子进程交给init进程
- 唤醒父进程，标记为ZOMBIE

**Wait：**
- 扫描进程表找ZOMBIE子进程
- 释放子进程资源，返回子进程ID
- 无子进程退出则sleep等待

**Kill：**
- 设置`p->killed`标志
- 如进程在sleep则wakeup它
- 进程最终进入/离开内核时检查标志并调用exit

## 9. 文件系统

### 9.1 文件系统层次结构

xv6文件系统分为**七层**：

| 层次 | 功能 | 关键文件 |
|------|------|----------|
| **文件描述符层** | 抽象Unix资源（文件、管道、设备） | `kernel/file.c` |
| **路径名层** | 层次化路径名解析 | `kernel/fs.c` |
| **目录层** | 目录项操作 | `kernel/fs.c` |
| **Inode层** | 文件表示为inode | `kernel/fs.c` |
| **日志层** | 崩溃恢复，事务原子性 | `kernel/log.c` |
| **Buffer Cache层** | 缓存磁盘块，同步访问 | `kernel/bio.c` |
| **磁盘层** | 在virtio磁盘上读写块 | `kernel/virtio_disk.c` |

**磁盘布局：**
```
块0: boot sector
块1: superblock（元数据）
块2~: 日志
日志后: inodes
inodes后: 位图块
其余: 数据块
```

### 9.2 Buffer Cache

**两个功能**：
1. **同步访问**：每个磁盘块在内存中只有一个buffer，一次只有一个内核线程使用
2. **缓存**：缓存常用块，避免从慢速磁盘重新读取

**核心接口**：
- `bread()`：返回磁盘块的内存副本（已锁定）
- `bwrite()`：将修改后的buffer写回磁盘
- `brelse()`：释放buffer

**LRU替换**：
- buffer组成双端链表
- `brelse`将buffer移到链表头部（最近使用）
- 需要新buffer时从尾部找（最近最少使用）

**两种锁**：
- `bcache.lock`：保护缓存块的信息（哪个块被缓存）
- `b->lock`（sleep-lock）：保护块内容的读写

### 9.3 日志层（Logging）

**目的**：解决崩溃恢复问题，确保文件系统操作原子性。

**工作流程**：
1. 系统调用将写入记录在磁盘日志中（`begin_op()` ~ `end_op()`）
2. 全部记录完成后写**提交记录**（commit）
3. 将日志中的写入复制到磁盘文件系统位置
4. 清除日志

**崩溃恢复**：
- 崩溃前提交 → 日志不完整 → 忽略（操作未发生）
- 崩溃后提交 → 日志完整 → 重新执行写入

**组提交（group commit）**：多个系统调用的写入累积到一个事务中一起提交。

**Commit四阶段**：
1. `write_log()`：修改的块从buffer复制到磁盘日志槽
2. `write_head()`：写header块（标记已提交）
3. `install_trans()`：日志块写到文件系统对应位置
4. 重置日志块计数为0

### 9.4 Inode层

**两种含义**：
1. **磁盘inode(dinode)**：磁盘上的数据结构，包含文件大小和数据块号列表
2. **内存inode(inode)**：磁盘inode的副本，加内核需要的信息

**磁盘inode结构**：
```c
struct dinode {
    short type;      // 文件类型（文件/目录/设备），0=空闲
    short nlink;     // 链接数
    uint size;       // 文件大小（字节）
    uint addrs[NDIRECT+1];  // 数据块号（直接+间接）
};
```

**数据块寻址**：
- `NDIRECT`个直接块号存放在`addrs`数组前部
- 第`NDIRECT+1`个元素指向**间接块**（含256个块号）
- 最大文件：12KB(直接) + 256KB(间接) = 268KB

**内存inode管理**：
- `iget()`：获取inode引用（增加ref计数）
- `ilock()`：锁定inode（使用sleep-lock）
- `iunlock()`：解锁
- `iput()`：释放引用（ref=0时释放缓存槽位）

**四种锁/类似锁机制**：
1. `icache.lock`：保证inode缓存唯一性
2. `inode->lock`（sleep-lock）：独占访问inode内容
3. `ref`引用计数：保持inode在缓存中
4. `nlink`链接计数：链接数>0时不释放inode

### 9.5 目录层

**目录结构**：
- inode类型为`T_DIR`
- 数据是一系列**目录项(dirent)**
- 每个目录项：名称(最多DIRSIZ=14字符) + inode号
- inode号为0表示空闲目录项

**核心函数**：
- `dirlookup()`：在目录中搜索给定名称的条目
- `dirlink()`：在目录中创建新目录项

### 9.6 路径名层

**路径解析**：`namex()`函数
1. 以`/`开头从根目录开始，否则从当前目录
2. 使用`skipelem`遍历路径每个元素
3. 对每个元素调用`dirlookup`查找
4. `namei()`返回对应inode，`nameiparent()`返回父目录inode

**并发安全**：
- 查找线程持有目录锁
- `dirlookup`返回`iget`获得的inode（增加引用计数）
- 收到inode后才释放目录锁
- 避免目录被删除后inode被重用

### 9.7 文件描述符层

**结构体file**：
```c
struct file {
    enum { FD_NONE, FD_PIPE, FD_INODE, FD_DEVICE } type;
    int ref;          // 引用计数
    char readable;
    char writable;
    struct pipe *pipe;
    struct inode *ip;
    uint off;         // I/O偏移量
};
```

- 每次`open`创建新的`file`（新偏移量）
- `dup/fork`共享偏移量
- `fileread/filewrite`使用偏移量并原子更新

## 10. 重点总结

### 10.1 磁盘调度算法对比

| 算法 | 原理 | 平均寻道 | 饥饿 | 最长等待 |
|------|------|----------|------|----------|
| FCFS | 按请求顺序 | 大 | 无 | 大 |
| SSTF | 最短寻道优先 | 较小 | 有 | 不确定 |
| SCAN | 来回扫描 | 较小 | 无 | 大（反方向） |
| C-SCAN | 单向扫描+回跳 | 较小 | 无 | SCAN的一半 |
| LOOK | SCAN改进（不到端点） | 小 | 无 | 小 |

### 10.2 文件目录结构对比

| 结构 | 特点 | 重名 | 共享 | 检索速度 |
|------|------|------|------|----------|
| 单级 | 一张目录表 | 不允许 | 不支持 | 慢 |
| 两级 | MFD+UFD | 用户间可 | 有限支持 | 较快 |
| 树型 | 多级子目录 | 允许 | 支持 | 快 |
| 图形 | 树型+共享链接 | 允许 | 强支持 | 快 |

### 10.3 xv6关键机制总结

| 机制 | 实现要点 |
|------|----------|
| **进程调度** | 轮询调度，swtch上下文切换，sleep/wakeup同步 |
| **内存管理** | Sv39三级页表，每进程独立页表，kalloc空闲链表 |
| **文件系统** | 七层结构，Buffer Cache+LRU，日志保证原子性，inode直接/间接索引 |
| **锁** | 自旋锁（amoswap，短临界区）+ 睡眠锁（长操作，可让出CPU） |
| **设备驱动** | 上半部/下半部分离，UART中断驱动，trampoline页处理trap |

> **复习要点**：xv6的核心设计理念是**简单明了**，同时展示操作系统的基本概念：进程隔离、虚拟内存、并发控制、持久化存储。
