第 23 课 · 银行家算法（4 矩阵 + 安全性检查 + 资源请求流程）

20 分钟把「四张矩阵 + 安全性算法 + 资源请求算法」一次装完，能独立做计算大题。
一句话：银行家算法 = 借钱前先预演一遍还款路径——能还清就借，还不清就拒绝（让进程等）。
本质：属于死锁的「避免」（不是预防！），允许 4 个死锁条件存在，但运行时动态绕开可能死锁的局面。

一、死锁处理的 4 个层次（先认清"避免"在哪一层）

层次做法代表方法代价
预防设计阶段破坏 4 条件之一一次性申请全部资源 / 按序申请资源利用率低
避免 ✅运行时算账，绕开死锁银行家算法每次分配都要算，开销大
检测允许死锁发生，事后检测资源分配图化简需要检测机制
解除检测到死锁后处理撤销进程 / 资源抢占损失已做的工作
考试最爱问的选择："银行家算法是避免死锁，不是预防。"——避免和预防一字之差，避免不破坏任何条件，预防破坏条件。默写三遍。

三、安全性算法（找安全序列）

有了 4 张表，银行家算法第一步：判断当前状态是不是「安全」——能找到一条顺序让所有进程都顺利跑完。

  1. 设两个变量：Work = Available（手头钱），所有进程 Finish = false。
  2. 从所有 Finish = false 的进程里，找一个满足 Need ≤ Work（每一类资源都 ≤）。
  3. 假设它跑完了 → 它会还回占用的全部资源 → Work = Work + Allocation，Finish = true。回到第 2 步。
  4. 所有进程 Finish = true → 安全（记下完成顺序 = 安全序列）；中途找不到 → 不安全。
「Need ≤ Work」的向量比较：必须每个分量都 ≤，不能只看总和。
例：Need = (1,2,2) 与 Work = (3,3,2) → 1≤3, 2≤3, 2≤2 ✅；如果 Need = (1,2,2) 与 Work = (3,1,5) → B 类 2>1 ❌。

四、资源请求算法（重点！请求来了怎么判）

上面是"看现状安不安全"。这一节是"进程 Pi 真的来要资源了，银行家要不要给？"——这是考试大题的标准问法。

设进程 Pi 发出请求向量 Request_i（例：Request₁ = (1,0,2) 表示要 1 个 A、0 个 B、2 个 C）。银行家按下面 5 步判断：

第①步：预判能不能给
如果 Request_i ≤ Need_i（它要的不超过它声明的最大值）→ 继续；否则报错「超过最大需求」，拒绝。
↓
第②步：预判金库够不够
如果 Request_i ≤ Available（金库现有量 ≥ 它要的）→ 继续；否则「等」，进程阻塞（资源不够，现在分不了）。
↓
第③步：试探性分配（假装已经给了）
修改四个数据结构： Available = Available − Request_i； Allocation_i = Allocation_i + Request_i； Need_i = Need_i − Request_i。
↓
第④步：跑安全性算法
用第三节的方法，看试探分配后系统是否仍安全（能找到安全序列）。
↓
第⑤步 a：安全 → 真分配
把第③步的修改真正生效，进程 Pi 拿到资源，继续运行。
或
第⑤步 b：不安全 → 撤回，恢复原状
把第③步的修改回滚（Available、Allocation、Need 全部还原），Pi 继续等待。
核心逻辑（记住一句话）："先试着给 → 跑安全性算法 → 安全就真给，不安全就撤销、不给、让它等。"
试探性分配 + 安全性检查 = 银行家算法的灵魂，比单看「金库够不够」多一道闸。

对比：安全性算法 vs 资源请求算法

安全性算法资源请求算法

五、计算例题（P1 申请 Request₁ = (1,0,2)）

接着上面 P0–P4 的初始状态（Work/Available = (3,3,2)）。现在 P1 提出申请 Request₁ = (1,0,2)。问：能否分配？

第①步：Request₁ ≤ Need₁ 吗？

Need₁ = (1,2,2)，Request₁ = (1,0,2)。逐分量：1≤1 ✅, 0≤2 ✅, 2≤2 ✅ → 通过。

第②步：Request₁ ≤ Available 吗？

Available = (3,3,2)，Request₁ = (1,0,2)。逐分量：1≤3 ✅, 0≤3 ✅, 2≤2 ✅ → 通过，可以试探。

第④步：跑安全性算法

Work = Available = (2,3,0)。逐个检查：

步Work挑中谁（Need ≤ Work）跑完后 Work += Allocation
1(2,3,0)P1：Need (0,2,0) ≤ ✅(2+3, 3+0, 0+2) = (5,3,2)
2(5,3,2)P3：Need (0,1,1) ≤ ✅(5+2, 3+1, 2+1) = (7,4,3)
3(7,4,3)P4：Need (4,3,1) ≤ ✅(7+0, 4+0, 3+2) = (7,4,5)
4(7,4,5)P2：Need (6,0,0) ≤ ✅(7+3, 4+0, 5+2) = (10,4,7)
5(10,4,7)P0：Need (7,4,3) ≤ ✅(10+0, 4+1, 7+0) = (10,5,7)

全部完成 → 系统仍安全，安全序列 P1 → P3 → P4 → P2 → P0。

第⑤步：真正分配

结论：P1 的请求可以满足。把第③步试探的修改真正生效，P1 拿到 (1,0,2) 资源，继续运行；其他进程按安全序列等待。
反例（考试最爱挖坑）：如果 Request₁ = (1,0,3) 呢？第①步：Need₁ 的 C 分量只有 2，但请求 3 个 C，3>2 → 超过最大需求，第①步就直接拒绝，不用进第②步。
记住：Request > Need → 报错；Request ≤ Need 但 Request > Available → 让它等；只有两步都通过才进入第③步试探。

六、必背三句（默写 3 遍）

① 公式Need = Max − Allocation，按行按列逐个减。
② 安全性算法找 Need ≤ Work → 它跑完 Work += Allocation → 重复，全 Finish = true 则安全。
③ 资源请求 5 步① Request≤Need → ② Request≤Available → ③ 试探分配 → ④ 跑安全性算法 → ⑤ 安全真给/不安全撤销等。
④ 归属银行家算法 = 死锁避免，不是预防（避免不破坏条件，预防才破坏）。
