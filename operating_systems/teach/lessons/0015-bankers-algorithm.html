第 15 课:银行家算法

目标:看懂什么是安全状态,会用"安全性算法"判断系统是否安全、找出安全序列。约 18 分钟。
它解决什么: 死锁的"避免"——系统在每次分配资源前先预演一遍:这次分配后,还能不能保证所有进程最终都顺利跑完?能→分配;不能(进入"不安全状态")→暂不分配,让它等。从不踏进可能死锁的局面。

一、比喻:谨慎的银行家

银行家手里有固定现金。每个客户开户时预先声明"我最多会借多少"(=Max)。银行家放贷的铁律:

每次客户要借钱,银行家先在心里预演:如果借了,我手里剩的钱,够不够按某个顺序让客户们一个接一个借满、用完、还钱?
✅ 能找到这样的顺序 →「安全」→ 放款;　❌ 找不到 →「不安全」→ 拒绝,让他等。

进程 = 客户,资源 = 现金。这个"能让所有人陆续跑完的顺序",就是安全序列。

二、四个量(记牢)

名称含义类比
Max 最大需求进程最多要多少客户声明的最高额度
Allocation 已分配进程已经拿到多少客户已借走的
Need 还需要Need = Max − Allocation客户还能再借的
Available 可用系统当前剩多少银行金库现金
最常考的一句: Need = Max − Allocation(还差的 = 最多要的 − 已经拿的)。

三、安全性算法(心脏,就 4 步)

  1. 令 Work = Available(手头可用),所有进程标记 Finish = 假(没完成)。
  2. 找一个还没完成、且 Need ≤ Work 的进程(它要的,手头够给)。
  3. 假设它跑完了:它会把占用的全部资源还回来 → Work = Work + Allocation,标记它 Finish=真。回到第 2 步继续找。
  4. 如果所有进程都能 Finish=真 → 安全(记下完成顺序=安全序列);如果中途卡住(没有进程的 Need≤Work)→ 不安全。
直觉:先让"要得少、现在就能满足"的进程跑完,它一还钱,手头的钱就变多,于是能满足下一个原本不够的进程……像滚雪球一样,只要能一路滚到底,就安全。

四、例题(单一资源,最易懂)

系统共 12 个某资源。三个进程的情况如下(Need = Max − 已分配):

P11055

已分配合计 = 5+2+2 = 9,所以 Available = 12 − 9 = 3。问:系统安全吗?

跑安全性算法(Work 从 3 开始):

步Work(手头)找谁(Need≤Work)它跑完还钱后 Work
13P1 Need5? 否。P2 Need2≤3 ✅3 + P2的2 = 5
25P3 Need7? 否。P1 Need5≤5 ✅5 + P1的5 = 10
310P3 Need7≤10 ✅10 + P3的2 = 12
三个进程全部 Finish → 系统安全。安全序列:P2 → P1 → P3。
(注意第 1 步:P1 一开始要 5、手头只有 3,给不了;但先让 P2 跑完还钱,手头变 5,P1 就能满足了——这就是"滚雪球"。)

五、完整银行家算法 = 试探分配 + 安全性检查

当进程 Pi 请求 Request 时:
① 检查 Request ≤ Need(没超过它声明的最大)且 Request ≤ Available(手头够);
② 试探着分配:Available−Request、Allocation+Request、Need−Request;
③ 用安全性算法检查:安全→真的分配;不安全→撤销试探,让 Pi 等待。

六、必背(三句)

① Need = Max − Allocation。

② 安全性算法:找 Need≤Work 的进程 → 它跑完 Work+=Allocation → 重复,全完成则安全。

③ 银行家算法是死锁避免(不是预防):每次分配前预演,不进入不安全状态。

Work=3。P1 Need2≤3 ✅→跑完还3,Work=6;P2 Need5≤6 ✅→还2,Work=8;P3 Need4≤8 ✅→还2,Work=10。全完成 → 安全,序列 P1→P2→P3。
