0. 复盘：Day 2 流程图

5 步：find_empty_process → copy_process → copy_mem → copy_files → 返回

1. copy_process 完整源码（精简注释版）

打开 kernel/fork.c,看 copy_process() 主体。我把它切成 5 个 step,逐段读：

int copy_process(int nr, long ebp, long edi, long esi, long gs,
                 long none, long ebx, long ecx, long edx,
                 long fs, long es, long ds,
                 long eip, long cs, long eflags, long esp, long ss)
{
    struct task_struct *p;
    int i;
    struct file *f;

    p = (struct task_struct *) get_free_page();   // ← Step A: 分配一页物理内存给子 PCB
    if (!p) return -EAGAIN;

    *p = *current;                                  // ← Step B: 直接 memcpy 父 PCB 到子 PCB (80 字段全拷)
    p->state = TASK_RUNNING;                       // ← Step C: 初始化关键字段
    p->pid = last_pid;
    p->father = current->pid;
    p->counter = p->priority;
    p->signal = 0;
    p->alarm = 0;
    p->leader = 0;
    p->utime = p->stime = 0;
    p->cutime = p->cstime = 0;
    p->start_time = jiffies;
    p->tss.back_link = 0;
    p->tss.esp0 = PAGE_SIZE + (long) p;          // 内核栈顶
    p->tss.ss0 = 0x10;
    p->tss.eip = eip;                              // 子进程从这开始执行
    p->tss.eflags = eflags;
    p->tss.eax = 0;                                // ← Step D: 关键! 子进程 fork() 返回 0
    p->tss.esp = esp;
    p->tss.cs = cs;
    ...
    p->ldt = current->ldt;                         // 复制 LDT
    set_ldt_desc(gdt + (nr << 1) + FIRST_LDT_ENTRY, p->ldt, 512);
    set_tss_desc(gdt + (nr << 1) + FIRST_TSS_ENTRY, &(p->tss));

    if (copy_mem(nr, p)) return -EAGAIN;            // ← Step E1: 复制页表
    if (copy_files(p))   return -EAGAIN;            // ← Step E2: 复制文件表
    if (copy_fs(p, 1))   return -EAGAIN;            // ← Step E3: 复制 FS 信息

    SET_LINKS(p);                                   // ← Step F: 链入就绪队列
    return last_pid;
}

代码看着多,但结构非常清晰——5 个 step,每步 1 个职责：

Step职责代码量对应考点
A 分配内存给子 PCB 申请一页物理页2 行Ch4 内存分配
B 复制 PCBmemcpy 父 PCB（80 字段全拷）1 行Ch2 进程创建
C 初始化关键字段state, pid, counter, signal 等15 行PCB 字段
D 设置 tss (CPU 上下文)设 eax=0,设 esp,设 eip20 行进程切换
E 复制资源copy_mem + copy_files + copy_fs3 行调用页表/fd 表
F 链入就绪队列SET_LINKS 宏1 行调度队列

2. 5 个 step 详解：「为什么这样设计」

Step A: get_free_page() —— 为什么是 1 页？

在做什么：get_free_page() 是 0.11 的物理内存分配函数,返回 1 页（4KB）空闲页框的物理地址。

💡 为什么是 1 页 4KB？
① task_struct 大小：0.11 的 task_struct 只有 80 多个字段,加起来不到 1KB,但为了内核栈(kernel stack) 的考虑,1 页是合适的——内核栈也需要一段连续内存。
② 页对齐：4KB 对齐的物理页,内核栈、tss 都能放进同一页里,避免跨页的复杂性。
③ 浪费 3KB：实际 task_struct 不到 1KB,剩下 3KB 给内核栈——单进程内核栈很浅,3KB 足够。

Step B: *p = *current; —— 1 行复制 80 字段

在做什么：C 语言的结构体赋值——把 current 指向的整个 task_struct 内存(80 字段)直接 memcpy 到 p 指向的内存。

💡 为什么能 1 行复制整个结构体？
① C 语言的语义：*p = *current 是结构体赋值,编译器翻译成 memcpy(dest, src, sizeof(task_struct))。
② 0.11 时代很合理：80 字段 ≈ 几百字节,1 行 memcpy 比逐字段赋值好懂、好维护。
③ 现代 Linux 不用了：现代 task_struct 几千字节(各种 cgroup、namespace、rseq、信号表),整段 memcpy 会浪费——所以现代用 memcpy(dst, src, offsetof(struct task_struct, fields_to_init)),只拷需要的前 N 字段,然后单独初始化其余。

Step C: 初始化 9 个关键字段

复制完后,要改 9 个字段——子进程不能跟父进程共享这些值：

字段父进程值子进程改成的值为什么改
stateRUNNING（父正在跑）RUNNING（子也可被调度）子立即可调度,不是阻塞
pid父 PIDlast_pid（新分配）子必须有自己 PID
father父的父父 PID记录父子关系
counter父剩余时间片priority（计数初值）子进程从头开始数时间片
signal父的待处理信号0信号不继承
alarm父的 alarm 值0定时器不继承
utime/stime父已用的 CPU 时间0子的 CPU 时间从 0 计
start_time父启动时间jiffies（当前）子的启动时间 = fork 时刻
💡 设计的精妙：「继承 + 改 9 字段」 是个非常 Unix 的设计——默认继承父的全部状态,然后只改那些必须不同的。这避免了大量「要不要继承」的争论:默认继承,显式覆盖。比现代 Linux 那种「白名单继承」(明确列出哪些继承)更激进,也更省事。代价是有些东西本不该继承(比如 alarm),但 0.11 没意识到——这是 bug 也是 day 4 质疑点之一。

Step D: 设置 tss —— 进程切换的核心

tss（Task State Segment）是 x86 CPU 提供的「任务状态段」硬件机制。Linux 0.11 用它保存进程的 CPU 上下文（eax/ebx/ecx/.../eip/esp/eflags 等 30+ 寄存器）。

关键 4 个赋值：

p->tss.eax = 0;          // fork() 在子里返回 0
p->tss.esp = esp;        // 用户栈指针 (从 sys_fork 实参来)
p->tss.eip = eip;        // 从 sys_fork 退出后执行的指令
p->tss.eflags = eflags;  // 标志寄存器
💡 父的 eip/esp/eax 在「sys_fork 入口」时被保存
你看 copy_process 的函数签名那一长串参数 (ebp, edi, esi, ... ss)——这不是作者手写,这是 gcc 从 int $0x80 入口处把所有寄存器值当实参传过来了。当子进程第一次被调度时,CPU 一加载 tss,这些寄存器全部恢复,子进程就好像从 sys_fork 调用后接着往下执行——但 eax 是 0,所以它「以为」fork 返回了 0。

Step E + F: 复制资源 + 链入队列

这 2 步比较直观,详见 Day 2。补充 2 个细节：

  • set_ldt_desc / set_tss_desc：在 GDT（全局描述符表）里给子进程分配一个 LDT 槽位 + 一个 TSS 槽位。GDT 是 CPU 硬件用的「段描述符表」。
  • SET_LINKS(p) 宏：把 p 链入就绪队列的双向循环链表。关键:这一刻子进程才真正「活了」——它可以被 schedule() 选中。

3. 整体设计的「三个一以贯之」

🎓 给老师讲的时候,一定要把这 3 个「设计哲学」讲出来——这是把「翻译代码」变成「讲设计」的关键。

哲学 1：默认继承,显式覆盖

子进程默认继承父的所有状态(state、tss、mm、files...),只改 9 个必须不同的字段。Unix 哲学的「白板复制」——子从父那张「写好的纸」开始,只擦掉 9 处不一样的地方。

哲学 2：硬件语义就是设计语义

Linux 0.11 严重依赖 x86 硬件:tss 段是 x86 提供的,LDT 是 x86 提供的,GDT 是 x86 提供的。换到 ARM 上,task_struct 的 tss 字段就完全不需要了(ARM 用 software context switch)。这意味着:读 0.11 的代码 = 在读 x86 时代的设计取舍。

哲学 3：内核态和用户态严格分离

copy_process 在内核态跑完前,子进程不可见;SET_LINKS 之后,调度器才能挑中它。这一刻的「可见性切换」是个隐式 invariant——破坏了就乱套了。

4. 自测 3 题

Q1. copy_process 里的 *p = *current; 是在做什么？为什么 1 行能复制 80 字段？必答

参考答案：C 语言结构体赋值 = 整段 memcpy (dest, src, sizeof(task_struct))。0.11 时代 80 字段不到 1KB,1 行比逐字段好懂、好维护。现代 Linux 改用 offset+单独初始化,只拷需要的前 N 字段。
Q2. 子进程 state 字段设成 RUNNING,counter 字段设成 priority。为什么不能直接继承父的 counter？扩展

参考答案：counter = 剩余时间片,父的 counter 可能已经很小（比如 1）。子如果继承,第一次调度就只能跑 1 tick,不公平。设成 priority = 时间片初始值,子是「全新进程」,享受完整时间片。
Q3. 子进程从 fork() 返回 0 的「硬件实现」是什么？必答

参考答案：通过 tss.eax = 0。子被调度时,CPU 加载 tss,eax 寄存器被恢复成 0,如同 fork() 在子进程眼里返回了 0。父进程走的是 return last_pid; 路径,eax 是子 PID。
✅ Day 3 通过标准：Q1 + Q3 答对（2/3）。Q2 是 Day 4 质疑训练的前置。
