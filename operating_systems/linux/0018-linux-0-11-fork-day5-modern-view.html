0. 观点的 4 个维度（cgroup / namespace / eBPF / rseq）

现代 Linux 看待「进程」这个概念,有了 4 个新维度。这些维度都不是 0.11 时代存在的——它们是 2007-2020 年间逐步加进内核的。你能从这 4 个维度看 0.11 的 fork,老师听了会觉得你看过现代内核。

视角加入年份解决什么问题对 fork 的影响
cgroup2008 (2.6.24)资源隔离（CPU/内存/IO 配额）fork 后子进程的 cgroup 归属问题
namespace2002-2014 (逐步)资源虚拟化（容器的基础）fork 后的 PID namespace / mount namespace
eBPF2014 (3.18)内核可编程（不用改内核就能 hook 关键路径）fork 时的 eBPF 探针 (tracepoint) 观测
rseq2018 (4.18)per-CPU 原子操作,加速用户态fork 后子进程的 rseq 注册

1. 视角 1: cgroup 视角 —— 资源配额继承问题

1.1 0.11 看不到的问题

0.11 的 fork 默认所有进程共享 16MB 物理内存和 CPU 时间——没有「这个进程只能用 1MB 内存」的概念。

为什么 0.11 不需要 cgroup？因为只有 64 个进程,且都是 root 创建的——大家都平等,没有「普通用户进程要限速,系统进程不限速」的需求。

1.2 现代场景

想象一个云服务器：

  • 你在跑一个 Python 脚本（cgroup memory.max=100MB）
  • 同一台机器还有 9 个租户的进程（memory.max=100MB 各自）
  • 如果你的 Python fork() 出 100 个子进程,每个都试图分配 1GB 内存——会发生什么？

现代 Linux 行为：

  1. 父进程在 cgroup A（限额 100MB）
  2. fork() 默认让子进程继承 cgroup A
  3. 子进程也受 100MB 限额
  4. 如果父想「隔离」,用 cgroup.clone_children=0 让子进 root cgroup

1.3 0.11 时代「等价的概念」

0.11 的 nice() 系统调用 + 优先级 (priority 字段) 是一种极简版的 cgroup 思想——给进程分配一个「权重」,调度时优先跑高权重的。但 nice 是「软限制」(高 nice 进程还是会被调度,只是不优先),cgroup 是「硬限制」(到上限直接 OOM kill)。

💡 我的观点（直接对老师讲）：
"如果 0.11 时代有 cgroup,Linus 可能会把 task_struct 里的 priority 字段换成 cgroup pointer——而不是只填一个 long。这样 1991 年的 'nice' 思路可以平滑进化到 2008 年的 cgroup 资源隔离。当然,cgroup 的复杂度是 nice 的 100 倍——这是 17 年才补完的。
🎤 讲稿（背这一段就够）：

"老师,从 cgroup 视角看 0.11 有一个有意思的观察：task_struct 里的 priority 字段是 long,只支持 nice 软优先级。如果 1991 年 Linus 用 cgroup pointer 替代它,nice 可以平滑进化到 2008 年的 cgroup 资源隔离——软优先级 → 硬限额。但 cgroup 的复杂度是 nice 的 100 倍,所以 17 年才补完是可以理解的。"

2. 视角 2: namespace 视角 —— 容器与 PID 虚拟化

2.1 0.11 看不到的问题

0.11 全系统所有进程共享一个 PID 命名空间——PID 1 永远只能是 init。想象一个场景：

  • 你在容器里跑一个 Python 脚本,以为自己是 PID 1
  • 但宿主机上 PID 1 是 systemd
  • 你的 kill -1 杀的是 systemd,不是 Python

2.2 现代 Linux 的 namespace

PID namespace (2002 年加入) 让你看到的世界是「隔离的」：

  • 容器内 PID 1 = 你的进程（看着像 init）
  • 宿主机 PID 32768 = 容器内 PID 1（真实 ID）
  • kill -1 在容器内只杀容器内的 PID 1

对 fork 的影响：

  • clone(CLONE_NEWPID) 创建新的 PID namespace
  • 子进程在 namespace 内有自己的 PID 1,父进程看不到
  • 这是 Docker / Kubernetes 的基础

2.3 0.11 时代「等价的概念」

0.11 的 getpid() 直接返回 task->pid,没考虑「隔离」概念。Unix 早期也是这种「全局 PID」模型——直到 Solaris 10 引入 zones 才改变。

💡 我的观点（直接对老师讲）：
"如果 0.11 有 PID namespace,find_empty_process() 就不应该返回全局 task[] 数组下标,而应该返回 namespace 内的局部 PID。容器场景下,这个改动让 fork 从 '单租户系统调用' 变成 '多租户系统调用'——这是云计算时代的基础能力。但加 namespace 会让 task_struct 多个指针,内存压力下又得改——这跟 64 进程上限的设计冲突,所以 Linus 没加是合理的。
🎤 讲稿（背这一段就够）：

"老师,从 namespace 视角看 0.11 有一个朴素的观察：find_empty_process 返回的是 task[64] 全局数组下标,不是 namespace 内的局部 PID。如果 1991 年 Linus 让 PID 在 namespace 内重映射,今天的 Docker 早 20 年诞生——但 namespace 增加 task_struct 指针,跟 64 进程上限的设计直接冲突,所以他没加是合理的——这个设计冲突,直到 2002 年才用动态 task_struct 解决。"

3. 视角 3: eBPF 视角 —— fork 的可观测性

3.1 0.11 看不到的问题

0.11 想知道「fork 了几次」或「哪个进程 fork 最多」,只能：

  1. 改内核源码,加 printk,重新编译内核（30 分钟）
  2. 用 strace -f 跟踪（但 0.11 没 strace）

3.2 现代 Linux 的 eBPF

eBPF 让你在内核里跑「沙盒小程序」,不需要改内核源码：

# bpftrace -e 'tracepoint:sched:sched_process_fork { @[comm] = count(); }'
# 输出:python 12, bash 8, sshd 3

这是个「bpftrace 一行命令」就能完成的 fork 监控——0.11 时代要写 100 行内核代码。

3.3 0.11 时代「等价的概念」

0.11 的 printk() 是 eBPF 的祖先——都是「在关键路径打日志」。但 printk 改不了控制流,eBPF 能改（基于条件跳过往下走）。这是「被动观测 → 主动控制」的进化。

💡 我的观点（直接对老师讲）：
"如果 0.11 有 eBPF,Linux 0.11 时代我们就能在 fork 路径上插 hook,做 '限制某进程 1 秒最多 fork 10 次' 这种安全策略——这是今天 Kubernetes 的 'pid limiter' 功能。eBPF 把 '内核可观测性' 从 '改内核源码' 变成 '写一段小程序',这是 30 年来内核最大的可扩展性进步。
🎤 讲稿（背这一段就够）：

"老师,eBPF 是我看 0.11 时最想要的东西。如果 0.11 有 eBPF,我能在 fork 路径上插一行 hook: '限制某进程 1 秒最多 fork 10 次'——这是今天 K8s 的 'pid limiter'。eBPF 把 '内核观测' 从 '改内核源码 30 分钟' 变成 '写一段小程序 30 秒'——这是 2014 年以来内核最大的进步。"

4. 视角 4: rseq 视角 —— 用户态加速

4.1 0.11 看不到的问题

0.11 的 fork 后,子进程第一次被调度时,eax 寄存器被设成 0——这是「硬件的恩赐」(x86 提供 tss)。但 ARM、RISC-V 没有 tss,这个技巧用不了。

4.2 现代 Linux 的 rseq

rseq (Restartable Sequences) 2018 年加,解决的是另一个问题：

  • 用户态程序想做「per-CPU 计数器」(不用锁,直接读写 per-cpu 变量)
  • 问题：如果这个线程被 fork 出去,rseq 注册被破坏,下次调度回来计数器读错
  • rseq 解决方案：clone(CLONE_RSEQ) 让子进程继承 rseq 注册,内核负责同步

4.3 0.11 时代「等价的概念」

0.11 没有 per-CPU 概念(单核),也没有 rseq(没需求)。这个视角在 0.11 上没什么发挥空间——如果老师问到这个视角,直接说「这个不适用于 0.11」,反而显得有判断力。

💡 观点训练的诀窍：敢说「这个不适用」
不是每个现代视角都能套到 0.11 上。rseq 就不适用——单核系统没 per-CPU 概念。敢说「这个不适用,理由是 X」,比硬套一个不合适的视角要好 10 倍。老师会觉得你有判断力,不是背诵机器。

5. 整体策略：观点的 4 种强度

讲观点时,有 4 种强度,从弱到强：

强度句式适用场景老师反应
L1 描述"现代 Linux 用 COW 替代了整页复制"事实陈述听完点头
L2 对比"0.11 拷 1MB,现代拷 0 字节,差 100 倍"量化对比记笔记
L3 观点"这是 Linus 自己承认的设计缺陷"有立场看着你
L4 方案"如果是我,我会用 cgroup pointer 替代 priority"给替代方案眼睛亮
🎯 目标：8 分钟讲稿里至少 3 处 L3-L4 强度。光描述 0.11 是什么,没观点——背书机器。光批判,没方案——键盘侠。「有立场 + 有替代」= 工业界 P6+。

6. 自测 3 题

Q1. 0.11 时代为什么不需要 cgroup？现代什么场景必须有？必答

参考答案：0.11 只有 64 进程,都是 root 创建,平等共享资源——没「限额」需求。现代多租户云服务器场景,10 个用户分 1 台机器,每用户限额——必须 cgroup。priority (nice) 是软限制,cgroup 是硬限额。
Q2. 如果 0.11 时代有 namespace,Docker 是不是早 20 年诞生？为什么？必答

参考答案：是的。PID namespace 让进程看到「局部 PID 1」——Docker 容器每个进程都「以为自己是 init」,基于此构建镜像、分层、端口隔离。但 namespace 跟 64 进程上限的设计冲突,直到 2002 年动态 task_struct 才解决。
Q3. 4 个现代视角里,哪个最不适用于 0.11？为什么？扩展

参考答案：rseq 最不适用。0.11 是单核系统,没有 per-CPU 概念,rseq 解决的问题不存在。敢说「不适用」=有判断力,比硬套一个不合适的好。
