1. fork 的全貌（一张图讲完整个调用）

在画流程图之前,先理解 fork 在「用户态 - 内核态 - 子进程」三个空间里发生的事：

【用户态】 用户调用 fork() —— C 标准库函数,gcc 会把它替换成 int $0x80 软中断
↓
【内核态 入口】 system_call.s 的 sys_fork 处理 —— 保存寄存器,调用 C 函数
↓
【内核态 step 1】 find_empty_process() —— 在 task[64] 数组里找空 PCB
↓
【内核态 step 2】 copy_process(nr, ...) —— 复制父进程 PCB 到 nr 号 PCB,设 state=TASK_RUNNING
↓
【内核态 step 3】 copy_mem(nr, p) —— 复制父进程的页表 (0.11 直接整页拷,慢)
↓
【内核态 step 4】 copy_files() / copy_fs() —— 复制文件描述符表 + FS 信息
↓
【内核态 step 5】 父进程返回子 PID;子进程被设 state=RUNNING,等调度器挑中
↓
【用户态 分叉】 父进程: fork() 返回 > 0 的子 PID;子进程: fork() 返回 0。同一段代码,两个返回值。
⚠ 这张图是 Day 2 的核心产物。把它画 5 遍以上,直到闭眼能复现。老师一定会让你画这张图。

2. 5 个 step 详解

Step 1: find_empty_process() —— 找空 PCB

在做什么：在 task[64] 数组里从 0 扫到 63,找第一个 state == 0（空）的槽位。

为什么这样设计：0.11 内核最多支持 64 个进程（这是它最大的限制之一,Day 4 会详细质疑）。

// 简化版
int find_empty_process(void) {
    int i;
    for (i = 0; i < 64; i++)
        if (!task[i])
            return i;
    return -EAGAIN;  // 没空位
}
🎤 讲稿钩子
"这里要特别说一下,Linux 0.11 用的是静态数组 task[64] —— 最多 64 进程,这是 1991 年的内存限制(16MB 物理内存装不下更多 PCB)。现代 Linux 用的是动态分配的 task_struct,理论上能跑到 PID 上限 4 百万。"

Step 2: copy_process() —— 复制 PCB

在做什么：把父进程 task[father_pid] 整个 memcpy 到 task[nr]（new pid）,然后初始化子进程的 state、tss、ldt。

关键点：

  • p->pid = last_pid（分配新 PID）
  • p->state = TASK_RUNNING（注意:这里设成 RUNNING 而不是 UNINTERRUPTIBLE,所以子进程立即可被调度）
  • p->father = current->pid（记录父进程,后面 getppid() 用）
  • SET_LINKS(p)（把子 PCB 链入就绪队列双向链表）

Step 3: copy_mem() —— 复制内存页表

在做什么：

  1. 给子进程分配一页 PCB 用的物理页(给 task_struct 本身用)
  2. 把父进程的页表项全拷过来（这里是性能瓶颈！）
  3. 设置子进程的页目录项,指向新拷的页表

为什么 0.11 没有 COW：COW (Copy-on-Write) 是 1994 年 Linux 1.x 才加的特性。0.11 直接整页复制内存,父进程有多少页,子进程就复制多少页。所以 0.11 的 fork 慢得离谱。

🎤 讲稿钩子（这是 Day 4 质疑点的前置铺垫）：

"老师,这一步是 0.11 性能最差的一步——它直接把父进程的所有页都复制一遍。我们后面会看到,这一步如果用 COW 可以快几十倍。"

Step 4: copy_files() / copy_fs() —— 复制文件表

在做什么：把父进程的 files_struct（fd 表）复制一份给子进程。

关键点：

  • 子进程和父进程 共享同一个 file 结构体（file->f_count++ 增加引用计数）
  • 这就是为什么父子进程写同一个文件,会「覆盖」——他们写的是同一个 file offset（除非用 O_APPEND 或各自 lseek）

Step 5: 父进程返回,子进程等调度

关键技巧：父进程从 sys_fork 返回时,返回值是子进程的 PID（> 0）；子进程被调度器选中后,从 copy_process 末尾返回时,返回值是 0。

这是怎么做到的？看 copy_process 末尾：

if (last_pid != pid)  // 父进程走的分支
    return last_pid;
else  // 子进程走的分支（通过 goto 或 set regs.eax = 0）
    return 0;

子进程返回 0 是通过修改 tss.eax 实现的——子进程的 eax 寄存器被预先设成 0,等调度器切换到子进程时,CPU 一加载 tss,看到 eax=0,以为 fork() 返回了 0。

✅ 设计精妙之处：父子进程共用同一段 fork() 调用代码,但通过 「eax 寄存器的不同初值」 实现了「同代码,不同返回值」。这正是 C 标准库 fork() 神奇之处的实现细节。

3. 完整流程图（闭眼可复现版）

用户态: pid = fork();
↓ libc 把它换成 int $0x80 + eax=2 (sys_fork 编号)
内核入口: system_call (system_call.s) → sys_fork
↓
Step 1: find_empty_process() → 返回 nr (空槽位号 0-63)
↓ 没空位返回 -EAGAIN (用户态 errno=EAGAIN)
Step 2: copy_process(nr, eip, ebp, ...)
↓
Step 3: copy_mem(nr, p) 复制页表 (慢!没有 COW)
↓
Step 4: copy_files() + copy_fs() 复制 fd 表
↓
Step 5: 设子 p->state=RUNNING,链入就绪队列
↓ 父进程: return last_pid; (子 PID > 0)
用户态 分叉:
→ 父进程继续 if(pid > 0) 分支
→ 调度器挑中子进程,tss.eax=0 被加载,子进程从 fork() 返回 0

4. 自测 4 题

Q1. fork 流程有 5 个 step,你能完整说出吗？必答

参考答案：① find_empty_process 找空 PCB;② copy_process 复制 PCB;③ copy_mem 复制页表;④ copy_files 复制文件表;⑤ 父返回子 PID,子返回 0。
Q2. 父进程从 fork 返回的是？子进程从 fork 返回的是？必答

参考答案：父进程返回子进程 PID（> 0 的整数）;子进程返回 0。通过子进程 tss.eax 预设为 0 实现。
Q3. 子进程被创建后,state 字段被设成什么值？为什么这么设？扩展

参考答案：TASK_RUNNING（值为 0）。意味着子进程一被创建就立即可被调度,而不是等父进程显式唤醒（vfork 才需要等）。
Q4. 父子进程为什么能区分返回值（一个返回子 PID,一个返回 0）？底层机制是什么？必答

参考答案：通过 tss.eax 预设值。子进程的 tss.eax 被预设为 0,等调度器切到子进程时,CPU 一加载 tss,eax=0,如同 fork() 返回 0。父进程走的是 return last_pid; 路径。
✅ Day 2 通过标准：Q1 + Q2 + Q4 答对（3/4）。Q3 是 Day 3 copy_process 详解的前置。
