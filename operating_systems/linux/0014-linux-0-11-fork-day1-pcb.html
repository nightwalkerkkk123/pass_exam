0. 为什么是 fork？

老师让你「精讲源码」,但源码可以选。挑哪个子系统,决定了你的 1 周好不好过。我的选择标准：

候选代码量跟考试挂钩能发挥质疑吗总评
进程切换（switch_to）~200 行汇编★★★汇编太多,讲不动
内存管理（page fault）~400 行★★★★★硬件绑太紧,易讲错
文件系统（sys_open）~600 行★★★★★代码最长,1 周讲不完
fork 系统调用~100 行核心★★★★★★★★★✅ 选这个

为什么 fork 是最优解：

  1. 跟汤小丹 Ch2 直接对应：Ch2 高频题「画出进程创建流程」就是 fork 流程图,精讲 fork = 期末 Ch2 简答必满分。
  2. 代码短：核心 copy_process() 只有 80 行,5 个步骤,容易讲清楚。
  3. 有大量设计选择可以质疑（Day 4 详细讲）：为什么没有 COW？为什么固定 64 进程？为什么用全局 runqueue？这些都有 1991 年 vs 2026 年的不同答案。
  4. 考试+实战双杀：期末 + 老师加分,一石二鸟。
📦 资料准备
本课使用 Linux 0.11（1991 年 Linus 写的最早版本）,不是现代 Linux。版本 0.11 是因为：
① 整个内核 ~1 万行,现代 Linux 是 ~3000 万行,零基础 1 周讲得动 0.11,讲不动现代版
② 0.11 的 fork 是「教科书式」实现,每个字段、每行代码都对应到考试概念
③ 老师大概率也是讲 0.11（汤小丹教材配套的就是 0.11）

获取方式：git clone https://github.com/zhaojiewen/Linux-0.11
只看 3 个文件：include/linux/sched.h（task_struct 定义）、kernel/fork.c（copy_process）、kernel/system_call.s（sys_fork 入口）

1. PCB：进程控制块（Task Struct）

在讲 fork 之前,必须先懂 PCB。原因：fork() 本质上就是「复制一个 PCB + 复制它的资源」。如果 PCB 字段你都不认识,fork 的每行代码你都看不懂。

汤小丹 Ch2 怎么定义 PCB：

"PCB 是进程实体的一部分,是操作系统为每个进程维护的数据结构,用于描述进程状态、管理进程资源、实现进程切换。PCB 是进程存在的唯一标志。"

把 PCB 想象成 「学生档案」:

  • 每个学生入学 = 创建一个 PCB
  • 档案里写:学号(state)、姓名(pid)、在哪个班(runqueue)、家庭住址(mm)、借了几本书(files)、有没有处分(signal)……
  • 学生转学 = PCB 状态切换(state)
  • 学生退学 = 撤销 PCB

1.1 task_struct 字段速查（Linux 0.11）

打开 include/linux/sched.h,找 struct task_struct 定义。字段多,但 1 周精讲只需要记 9 个核心字段：

字段类型含义考试挂钩讲的时候要说什么
statelong进程状态5 状态转换"这个字段就是 Ch2 必考的 5 状态转换表"
pidint进程标识符PID 唯一性"进程的唯一 ID,内核用 64 个槽位管理"（限制！）
counterlong时间片剩余RR 调度"这个就是 Ch3 时间片轮转里那个'剩余时间片'"
prioritylong优先级（计数初值）优先级调度"counter 每 tick 减 1,降到 0 就重新调度"
tssstruct tss_struct任务状态段（CPU 上下文）进程切换"最关键的字段,CPU 寄存器值全在这里。切换进程 = 换 tss"
ldt[3]struct desc_struct局部描述符表段式存储"这个进程的内存视图"（Ch4 段式管理）
mmstruct mm_struct*内存管理信息（页表）分页"fork 时要复制这个指针链,这就是'为什么 fork 慢'"
filesstruct files_struct*打开的文件表Ch7 文件"父进程 open 过的 fd,子进程也能用"
p_signal / sigsigset_t信号位图Ch2 信号"kill -9 那个 9,对应的是 bit 9"（位图）
📌 讲稿钩子
你对着老师讲到这里时,可以停顿一下说：
「老师,你看这 9 个字段,其实就对应了汤小丹第 2 章那一整章——5 状态转换在 state、时间片在 counter、内存视图在 mm、文件表在 files。task_struct 不是孤立的,它就是 Ch2 概念在 C 语言里的一个具体化。」
这句话老师听了会眼前一亮。

1.2 0.11 vs 现代 Linux：3 个明显差异

字段Linux 0.11 (1991)现代 Linux 6.x为什么改
状态字段一个 state long多个位域 + 等待队列0.11 只能表示「就绪/运行/阻塞」,现代要区分「可中断睡眠/不可中断睡眠/暂停」等 6+ 种
文件表单链 files 指针struct fdtable + RCU0.11 fork 时直接复制整个数组,O(N),现代用引用计数 + RCU 锁
内存单一 mm 指针mm + vma 链表 + rcu 保护0.11 写时复制(COW)是后续 2.x 才加的,0.11 直接整页复制——很慢（Day 3 详讲）

3 个必读文件（按精讲顺序）：

  1. include/linux/sched.h —— 找 struct task_struct（约 60 行）
  2. kernel/fork.c —— 读 copy_process()（约 80 行）
  3. kernel/system_call.s —— 看 sys_fork 入口（10 行汇编）

不要试图读 sched.c 里的 schedule(),那 200 行函数涉及就绪队列遍历、counter 比较、tss 切换,1 周讲不动。Day 2 给你看简化版流程图就够了。

3. 读源码的 3 个姿势（Day 1 必掌握）

零基础读源码最容易犯的错是「从第一行开始逐行读」。这是错的——会被汇编、宏、跳转淹没,3 小时就放弃。

正确姿势：先骨架后细节,先流程后代码,先注释后实现。

姿势 1：先骨架后细节

拿到一个 C 函数,先问 3 件事:

  1. 它输入什么参数？
  2. 它输出什么返回值？
  3. 它修改什么全局状态？

例：copy_process(nr, ...) —— 输入 nr（PCB 数组下标）,返回 last_pid（子进程 PID）,修改 task[] 数组。

姿势 2：先流程后代码

把函数体用注释切成 5-7 个步骤,先理解每步「在干什么」,再回去看代码怎么实现。

姿势 3：先注释后实现

Linux 0.11 是 1991 年的代码,作者 Linus 的注释比现代代码多得多。注释里通常会写「为什么这么写」,这正是老师最爱问的。

🎤 讲稿片段（对着老师讲）：

"老师,我读 fork 源码用的是三步法:第一,先看函数签名,确定输入输出;第二,把函数体切成 5 个 step 画流程图;第三,再回到每行代码看具体实现。

这种读法的好处是——您问任何一行,我都能回答它属于哪个 step,以及那个 step 在整个 fork 流程里的位置。"

4. 自测 5 题（Day 1 通过标准：5/5）

Q1. PCB 在汤小丹 Ch2 里被定义为什么？必答

参考答案：进程实体的一部分,是 OS 为每个进程维护的数据结构,用于描述进程状态、管理进程资源、实现进程切换。PCB 是进程存在的唯一标志。
Q2. task_struct 里 state 字段对应汤小丹 Ch2 的哪个知识点？必答

参考答案：5 状态转换图（运行/就绪/阻塞/创建/终止）。state 字段的值（如 0=运行、1=就绪、2=阻塞）就是那张图的实现。
Q3. task_struct 里最关键的字段是哪个？为什么？必答

参考答案：tss（任务状态段）。原因：CPU 寄存器值全部存在 tss 里,进程切换的本质就是切换 tss。其他字段都是「数据」,tss 是「CPU 上下文」。这一点对应汤小丹 Ch2「进程的切换」一节。
Q4. Linux 0.11 跟现代 Linux 6.x 在 PCB 上有 3 个明显差异,你能说出 1 个吗？扩展

参考答案：① state 一个 long → 现代多 bit 域 + waitqueue；② files 单链 → 现代 fdtable + RCU；③ 0.11 fork 直接整页复制 → 现代 COW（写时复制）。任答一个即可。
Q5. 「读源码三步法」是什么？必答

参考答案：① 先骨架后细节（输入/输出/全局状态）;② 先流程后代码（切成 5-7 step）;③ 先注释后实现（注释里有「为什么这么写」,老师最爱问的）。
✅ Day 1 通过标准：Q1-Q3 + Q5 全对（4/5 通过,Q4 是 Day 4 质疑训练的前置）。
如果 5 题有 2 题以上答不上来,说明 task_struct 字段没记牢,回上面表格再读一遍,再来自测。
