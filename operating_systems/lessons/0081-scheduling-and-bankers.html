🧠 1. 做题通用套路（背下来就能套）

调度题三公式

调度题三公式（必背）：
指标公式含义
周转时间 TT = 完成时刻 − 到达时刻作业从「入队」到「干完」的总时长
带权周转 WW = T ÷ 服务时间「等多久才算真正在干活」的比例，≥1
平均指标= 所有作业之和 ÷ 作业数整批作业的平均表现

银行家四表

银行家四表（必背）：
表公式含义
Allocation已分得的资源（题目给）当前各进程手里有什么
Max最大需求（题目给）进程声明「最多需要多少」
NeedNeed = Max − Allocation「还需要多少」才能完成
AvailableAvailable = 总资源 − ΣAllocation当前空闲资源池
⚠️ 易错三连：
  1. 周转公式记反 → T = 完成 − 到达（不是到达 − 完成）
  2. 带权公式忘除 → 带权 = T ÷ 服务时间（必须除，少除就丢分）
  3. 银行家忘了 Need = Max − Allocation → 后面安全序列全崩

Q26-1FCFS 先来先服务

题面

5 个作业 A/B/C/D/E，到达时间 0/1/2/3/4，服务时间 4/3/4/2/4。求平均周转时间和平均带权周转时间。

FCFS 口诀

「先来的先跑，跑到自然结束才换下一个」。每个作业的开始时刻 = 上一个作业的完成时刻，与到达时间无关（除 A 之外）。

Q26-2SJF 短作业优先（非抢占）

SJF 口诀

「队列里服务时间最短的先跑，跑完才考虑下一个」。开始新作业前挑队里最短的；相同 → 按到达顺序或字母序。

🧠 关键观察：D 是 SJF 的真正受益者——FCFS 下 D 周转 = 10，SJF 下 D 周转 = 3（D 服务时间 2，等 1）。这就是「SJF 偏爱短作业」的具体体现。
⚠️ 易错：SJF「非抢占」意味着 A 在 t=0 开始跑后中途不会被打断——即使后面来了更短的 D 也不能换。如果题面写「抢占 SJF / SRTF」才允许中断。本题明确「非抢占」。
🔍 拓展：本题如果是 SJF 抢占 (SRTF) 会怎样？
SJF 抢占（又称 SRTF）允许新到作业打断当前作业——只要新作业的服务时间 < 当前作业的剩余时间。

本例中 A 在 t=0 开始（服 4），到 t=3 时 A 剩 1，D 刚到（服 2）。2 > 1，不抢占。所以本例 SRTF 和非抢占 SJF 结果相同，都是完成时刻 4/6/9/13/17。

但如果 D 的服务时间是 1 而不是 2，t=3 时 D(1) < A(剩 1)？仍然相等不抢占。所以本例无论抢占非抢占都一样。

Q26-3RR(q=2) 时间片轮转

题面（同 Q26-1，q=2）

RR 口诀

「先来先服务 + 时间片到就放队尾」。每个作业最多跑 q 个时间单位；新到的作业直接插队尾；时间片到了还没干完的也插队尾。

Q27-1银行家算法·安全性判断

题面

5 个进程 P0~P4，3 类资源 A/B/C 共 10/5/7。T0 时刻：

进程Allocation (A,B,C)Max (A,B,C)
P0(0,1,0)(7,5,3)
P1(2,0,0)(3,2,2)
P2(3,0,2)(9,0,2)
P3(2,1,1)(2,2,2)
P4(0,0,2)(4,3,3)

求：① Need 矩阵；② Available；③ 判断 T0 是否安全？

步骤 1·算 Need = Max − Allocation

进程MaxAllocationNeed = Max − Allocation
P0(7,5,3)(0,1,0)(7,4,3)

P3(2,2,2)(2,1,1)(0,1,1)
P4(4,3,3)(0,0,2)(4,3,1)

ΣAllocation = (0+2+3+2+0, 1+0+0+1+0, 0+0+2+1+2) = (7, 2, 5) Available = (10−7, 5−2, 7−5) = (3, 3, 2)

  • 第一轮「只剩 P3」很容易看错——P2 的 Need(6,0,0) 虽然 A 类少，但加上 B/C 后整体看，逐分量比较不是「总和」
  • P4 的 Need(4,3,1) vs Avail(5,4,3)：是 ≤ 的，但前面轮次 Avail 还没到 5，别看错时机
  • Need 比较是逐分量独立：Need(i) ≤ Available 当且仅当每个分量都 ≤（不是总和比）

Q27-2银行家算法·请求试探

题面

P1 提出请求 Request(1, 0, 2)，能否立即分配？

三道闸门

📐 银行家请求试探三道闸门：
  1. 闸 1：Request ≤ Need（不能超过声明需求）
  2. 闸 2：Request ≤ Available（不能超过空闲资源）
  3. 闸 3：试探分配后系统仍安全（找得到安全序列）
任一不通过 → 等待。

闸 1：Request(1,0,2) ≤ Need[P1](1,2,2)？

1 ≤ 1 ✓ 0 ≤ 2 ✓ 2 ≤ 2 ✓ → 通过闸 1 ✓

闸 2：Request(1,0,2) ≤ Available(3,3,2)？

1 ≤ 3 ✓ 0 ≤ 3 ✓ 2 ≤ 2 ✓ → 通过闸 2 ✓

闸 3：试探分配后找安全序列

3.1 假装分配后，更新三个表

项目旧新
Available(3,3,2)(3−1, 3−0, 2−2) = (2,3,0)
Allocation[P1](2,0,0)(2+1, 0+0, 0+2) = (3,0,2)
Need[P1](1,2,2)(1−1, 2−0, 2−2) = (0,2,0)

3.2 用新 Available 找安全序列

轮次Avail候选 (Need ≤ Avail)选中释放 → 新 Avail
1(2,3,0)P0(7,4,3) ✗
P1(0,2,0) ✓
P2(6,0,0) ✗ (6>2)
P3(0,1,1) ✗ (1>0)
P4(4,3,1) ✗		P1(2+3, 3+0, 0+2) = (5,3,2)
2(5,3,2)P0 ✗
P2(6,0,0) ✗ (6>5)
P3(0,1,1) ✓
P4(4,3,1) ✓		P3(5+2, 3+1, 2+1) = (7,4,3)
3(7,4,3)P0(7,4,3) ✓
P2(6,0,0) ✓
P4(4,3,1) ✓		P2（或者 P0、P4 都行）—
4—剩余任选——
✅ 试探后安全序列：P1 → P3 → P2 → P4 → P0（或 P1 → P3 → P0 → P2 → P4 等多种）
→ 通过闸 3 ✓

结论

🎯 三道闸门全部通过 → 可以立即分配。

具体动作： 
// 把 P1 的请求当真分配
Available = (2,3,0)
Allocation[P1] = (3,0,2)
Need[P1] = (0,2,0)
// 然后让系统按 P1→P3→P2→P4→P0 的顺序推进即可
⚠️ 易错：
  • 只验闸 1+2 就说「能分配」→ 大错！必须三道闸门都通过才能分配
  • 「试探」是临时假设——如果闸 3 不通过要回滚（Available、Allocation、Need 恢复原值，让 P1 等）
  • 第一轮只能选 P1 是因为 P1 的 Need 退化为 (0,2,0) 极小——这是「试探」带 P1 占便宜的副作用

✅ 8. 自测 4 题（动手写，再看答案）

自测 1 · Q26 调度. FCFS 下 D 的带权 5.0 怎么来的？如果改成 RR(q=4)，D 的带权会变大还是变小？
答案
FCFS：D 周转 = 完成 13 − 到达 3 = 10，D 服务 = 2，所以带权 = 10 ÷ 2 = 5.0。

改成 RR(q=4)：D 在 t=3 到达，t=8 轮上只需 1 个时间片就完成（t=8 → t=9），完成时刻变成 9，周转 = 9 − 3 = 6，带权 = 6 ÷ 2 = 3.0，比 5.0 小。

结论：时间片越大，RR 越像 FCFS；时间片越小，RR 越公平但周转越长。
自测 2 · Q26 调度. SJF 「非抢占」下，如果某个作业到达时 CPU 空闲，谁先占用？为什么？
答案
CPU 空闲 + 队列里有等它的作业 → 队里最短的先跑。本题 t=4 时 CPU 空，队列是 A(4)/B(3)/C(4)/D(2)/E(4)，最短的 D 先跑（A 已经完成）——这正是 SJF 强调「选最短」的核心。

容易错的点：如果多个最短相同（比如 C 和 E 都 4），就退化为 FCFS（按到达顺序）或按字母序——题目默认前者。
自测 3 · Q27 银行家. 如果 P4 提出 Request(0, 0, 1)，能分配吗？（提示：原题是 P1 申请 (1,0,2)）
答案
能。
闸 1：Request(0,0,1) ≤ Need[P4](4,3,1)? 0≤4✓ 0≤3✓ 1≤1✓ 通过。
闸 2：Request(0,0,1) ≤ Available(3,3,2)? 0≤3✓ 0≤3✓ 1≤2✓ 通过。
闸 3：试探后 Available = (3,3,1)，找安全序列——尝试 P3(0,1,1) ≤ (3,3,1) ✓ → 释放 → Avail = (5,4,2)... 然后 P1/P2/P4/P0 都能找到，三道闸门全过，能分配。
自测 4 · Q27 银行家. 如果 P1 改成请求 Request(2, 0, 2)，闸门哪道会卡住？
答案
闸 1 卡住。Request(2,0,2) ≤ Need[P1](1,2,2)? 第 1 分量 2 ≤ 1? 不成立。
结论：P1 申请 2 个 A 类资源，但 Need 里 A 只剩 1——超过声明需求，必须等待。这种情况 P1 应该修改自己的 Max 声明（如果需要），但在当前 Max 下不能批准。
