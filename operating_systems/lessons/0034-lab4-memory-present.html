1先亮宏：虚拟地址是怎么被切开的

数据结构kernel/riscv.h
// Sv39：64 位虚拟地址的低 39 位 = 9 + 9 + 9 + 12
//   30..38 → L2 索引(9 位)
//   21..29 → L1 索引(9 位)
//   12..20 → L0 索引(9 位)
//    0..11 → 页内偏移(12 位)
#define PGSHIFT 12                  // 页内偏移占的位数
#define PXMASK  0x1FF                 // 9 个 1，取出一段索引
#define PXSHIFT(level)  (PGSHIFT+(9*(level)))
#define PX(level, va) ((((uint64)(va)) >> PXSHIFT(level)) & PXMASK)

#define PTE_V (1L << 0)  // 有效位：这一项是不是有效
#define PTE2PA(pte) (((pte) >> 10) << 12)  // 从页表项里抠出物理页地址
🎤 怎么说

"先看 CPU 怎么拆地址。PX(level, va) 就是个取索引器:右移把那一段挪到最低位，再 & 0x1FF 砍出 9 位。level=2 取最高那 9 位、level=1 取中间、level=0 取最低那 9 位索引。最低 12 位是页内偏移，它不参与查表。还有 PTE_V 这个有效位，记住它——下面每查一级都要看它。"

2核心：walk 逐级查三级页表（重点）

地址变换kernel/vm.c · walk()
pte_t *
walk(pagetable_t pagetable, uint64 va, int alloc)
{
  if(va >= MAXVA)
    panic("walk");

  for(int level = 2; level > 0; level--) {  // ★走 L2、L1 两级
    pte_t *pte = &pagetable[PX(level, va)];   // 用该级 9 位索引取一行
        if(*pte & PTE_V) {                  // 这一项有效?
      pagetable = (pagetable_t)PTE2PA(*pte); // 顺着走到下一级表
    } else {
            if(!alloc || (pagetable = (pde_t*)kalloc()) == 0)
        return 0;                       // 没建表&不让分配 → 失败
      memset(pagetable, 0, PGSIZE);      // kalloc 出一张新表，清零
      *pte = PA2PTE(pagetable) | PTE_V;   // 挂上去，置有效位
    }
  }
  return &pagetable[PX(0, va)];          // L0：返回最后那个 PTE 的地址
}
🎤 怎么说（这段最容易被追问，慢讲）

"看这个 for(level=2; level>0; level--)——它就是'逐级查表'本身。第一圈 level=2:用 PX(2,va) 在根页表取一行，如果有效位 PTE_V=1，就 PTE2PA 把它指向的下一级表地址取出来、换过去。第二圈 level=1 同样操作，走到 L0 表。"

👉 指着 else 那支说:"如果这一格无效（PTE_V=0），说明这片地址还没建过表。alloc 参数就在这儿起作用:alloc=0 就直接失败返回;alloc≠0 就现场 kalloc 一张新表、清零、挂上去并置有效位。"

👉 一句收口:"循环只走两级，最后一级 PX(0,va) 在循环外，直接返回那个 PTE 的地址——因为调用者要的就是它。三级查完，物理页号配上页内偏移，地址变换完成。"

3另一半：kalloc / kfree —— 物理页的分配与回收

分配回收kernel/kalloc.c
struct run {
  struct run *next;   // ★指针借空闲页自己开头存
};
struct {
  struct spinlock lock;
    struct run *freelist;  // ★所有空闲页串成一条链
} kmem;

void kfree(void *pa) {              // 回收一页
  struct run *r = (struct run*)pa;
  acquire(&kmem.lock);
    r->next = kmem.freelist;       // 让这页指向原表头
    kmem.freelist = r;             // 这页成为新表头 → 挂头 O(1)
  release(&kmem.lock);
}

void *kalloc(void) {              // 分配一页
  acquire(&kmem.lock);
  struct run *r = kmem.freelist;
    if(r) kmem.freelist = r->next; // 摘下表头 → 摘头 O(1)
  release(&kmem.lock);
  return (void*)r;
}
🎤 怎么说

"刚才 walk 要新表时调的 kalloc，就是这个。xv6 管空闲内存的办法极简:把所有 4KB 空闲页串成一条单链表 freelist。kalloc 就是摘表头、kfree 就是挂表头，都只动一两个指针，不用遍历，所以是 O(1)。"

👉 指着 struct run 说:"妙在链表的 next 指针存哪——就借空闲页自己开头那 8 个字节。反正它空着，不另花内存。这就是课本'空闲块链表法'最干净的实现。"

  • 类比:"省 → 市 → 区，逐级查名册。L2 索引查到'L1 名册在哪'，L1 查到'L0 名册在哪'，L0 才查到'物理页大门'，最后按页内偏移进具体那一户。没用到的省份，连下级名册都不印——这就是多级页表省内存。"

5答辩 Q&A（老师最可能问的 5 个）

Q1为什么用多级页表，不用一张单级大表?
答法: "省内存。单级表要为每个虚拟页都留一个表项，Sv39 有 512×512×512≈1.3 亿项，每个进程都得占着，绝大多数还用不到。多级把表拆成树，没用到的整片区域连下一级表都不建（有效位为 0），真正占内存的只有实际用到的那几张小表——这就是'多级页表节省页表空间'。"
Q2walk 里的 alloc 参数是干嘛的?
答法: "它控制查到某级页表不存在时，要不要现场建一张。alloc=0（如 walkaddr 纯查地址）就直接返回 0;alloc≠0（如 mappages 要建映射）就 kalloc 一页当新页表、清零、挂上去并置 PTE_V。一句话:查路 vs 修路。"
Q3PTE_V 是什么? 为什么每级都要判它?
答法: "PTE_V 是页表项的有效位。*pte & PTE_V 为真，说明这一项指向的下级表/物理页是有效的、能顺着走;为 0 说明这片地址还没建立映射，要么报错，要么去分配。它是逐级查表能不能继续往下走的开关。"
Q4kalloc 为什么用链表管理空闲页? 为什么说是 O(1)?
答法: "所有空闲页串成单链表 freelist。分配=摘表头、回收=挂表头，都只改一两个指针、不遍历，所以是 O(1)。而且链表的 next 指针借空闲页自己开头 8 字节存（struct run），管理几乎不额外占内存——对应课本'空闲块链表法'。"
Q5这段代码对应教材哪个知识点?
答法: "对应 Ch4 存储器管理:walk 是分页里的地址变换机构 + 多级页表（虚拟地址→物理地址）;kalloc/kfree 是内存的分配与回收（空闲块链表法）。这两块正是本学期实验④ 内存分配与回收的核心。"
