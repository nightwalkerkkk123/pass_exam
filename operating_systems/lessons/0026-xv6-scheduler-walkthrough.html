0. 先记这句主线(讲偏了就拉回来)

“每个 CPU 跑一个 scheduler() 死循环:不停扫描进程表,找一个 RUNNABLE(就绪)的进程,把它设成 RUNNING,用 swtch 切进去运行;等这个进程主动让出(yield)或阻塞(sleep),再用 swtch 切回 scheduler,继续找下一个。如此往复——这就是课本说的低级调度(进程调度),而切来切去那一下,就是进程切换 / 上下文切换。”
ELI5 比喻(开场可用):调度器像一个不睡觉的值班调度员,手里只有一支麦克风(CPU)。他绕着会议桌一圈圈巡查,谁举手准备好了(RUNNABLE),就把麦克风递给谁(swtch 切入);那人说完一段或被叫停(yield/时钟中断),就把麦克风还回来(swtch 切回),调度员接着巡查下一个。“切进程”本质就是换一张说话的桌签——把当前寄存器存好,把下一个人的寄存器恢复出来。

1. 背景:上下文(context)与就绪状态(进程切换保存了什么)

讲调度前,先给老师看“切换到底切什么”。来自 kernel/proc.h 的 struct context——这就是课本说的“进程上下文 / 现场”在 xv6 里的真身:

1// Saved registers for kernel context switches.
2struct context {
3  uint64 ra;   // ★ 返回地址:切回来时从哪条指令继续执行
4  uint64 sp;   // ★ 栈指针:指向这个内核线程自己的栈(切 sp = 切进程)
5
6  // callee-saved 被调用者保存寄存器
7  uint64 s0;
8  uint64 s1;
..  ...
18  uint64 s11;  // s0~s11 共 12 个,加 ra、sp,一共 14 个寄存器
19};
讲这页时只强调两行:“ra 是返回地址——切回来时从哪儿接着跑;sp 是栈指针——指向这个进程自己的内核栈。只要把 sp 一换,就等于换了整条栈、换了整个执行现场,这就是‘进程切换’的核心。 至于 s0~s11,是 RISC-V 规定‘被调用者保存’的寄存器,必须存下来,切回去才能原样恢复。”

进程的几种状态(来自 proc.h 的枚举,对应课本五状态):UNUSED / USED / SLEEPING(阻塞)/ RUNNABLE(就绪)/ RUNNING(运行)/ ZOMBIE(终止)。调度器只挑 RUNNABLE——也就是课本的“就绪队列”里的进程。 xv6 没有显式的就绪队列链表,而是用一个全局数组 proc[NPROC],靠遍历找 RUNNABLE 来代替“从就绪队列取队首”。

2. 逐行讲 scheduler()(来自 kernel/proc.c)

背景注释先念:“每个 CPU 启动后都调用 scheduler(),它永不返回,只是循环:挑一个进程 → swtch 切进去运行 → 那个进程最终再 swtch 切回来。”

444void
445scheduler(void)
446{
447  struct proc *p;
448  struct cpu *c = mycpu();   // 取当前这个 CPU 的结构体
449
450  c->proc = 0;
451  for(;;){                       // ★ 外层死循环:调度器永不退出
452    // Avoid deadlock by ensuring that devices can interrupt.
453    intr_on();                  // 开中断:保证时钟中断能进来(否则没法被抢占)
454
455    for(p = proc; p < &proc[NPROC]; p++) { // ★ 遍历整张进程表 proc[]
456      acquire(&p->lock);        // 上锁,防止别的 CPU 同时动这个进程
457      if(p->state == RUNNABLE) { // ★ 找到一个“就绪”的进程
461        p->state = RUNNING;     // ★ 状态:就绪 → 运行
462        c->proc = p;            // 记下当前 CPU 正在跑谁
463        swtch(&c->context, &p->context); // ★★ 进程切换:存调度器现场,载入 p 的现场,跳进 p 运行
464
465        // 执行到这里,说明 p 又把 CPU 让回来了
467        c->proc = 0;            // 当前 CPU 暂时没在跑用户进程
468      }
469      release(&p->lock);        // 解锁,继续看下一个进程
470    }
471  }
472}

把代码切成“三件事”讲(这是你讲解的主结构)

代码块行一句话讲法对应课本
① 死循环 + 开中断451–453“调度器是个永不返回的死循环。先开中断,这样时钟中断才能进来,才有‘抢占’和‘时间片’可言。”抢占式调度 / 时钟
② 遍历找 RUNNABLE455–457“扫描整张进程表 proc[],找一个状态是 RUNNABLE(就绪)的进程。这就是‘从就绪队列里挑一个’的最朴素实现——顺序扫描。”低级调度(进程调度)/ 就绪队列
③ 设运行 + swtch 切入461–463“把它设成 RUNNING,然后 swtch:存下调度器自己的现场,载入这个进程的现场,CPU 就跳进它运行了。这一行就是‘进程切换’。”进程切换 / 上下文切换
④ 切回后继续465–470“swtch 之后那行很关键:能执行到这里,说明刚才那个进程又把 CPU 让回来了(它自己改了 state 再切回)。然后解锁,继续扫下一个。”调度往复
最容易被追问、也最妙的一点: 第 463 行 swtch 是一次“去”、第 464 行之后是一次“回”,中间隔着的不是几纳秒,而是整个进程的一段运行时间。CPU 从 463 行跳走去跑别的进程,可能过了好几毫秒、跑完一个时间片,才从 swtch 里“回来”落到 464 行。同一个 swtch 调用,进去和出来跨越了一次完整的进程运行。

3. 逐行讲 yield() / sched()(进程怎么“让出 CPU”)

上面调度器切进了进程。那进程什么时候、怎么把 CPU 还回来?答案是 yield()——它在时钟中断里被调用(每个时间片到点)。来自 kernel/proc.c:

501// Give up the CPU for one scheduling round.(让出 CPU 一轮)
502void
503yield(void)
504{
505  struct proc *p = myproc();   // 当前正在跑的进程
506  acquire(&p->lock);
507  p->state = RUNNABLE;          // ★ 运行 → 就绪:我还能跑,但先让出,重新排队
508  sched();                       // ★ 切回调度器
509  release(&p->lock);
510}

yield 只做一件事:把自己从 RUNNING 改回 RUNNABLE(注意不是阻塞,而是“还就绪、只是排到后面”),然后调用 sched() 真正切走。sched() 是“切回调度器”的统一出口:

481void
482sched(void)
483{
484  int intena;
485  struct proc *p = myproc();
486
487  if(!holding(&p->lock))      // 安全检查:必须持有 p->lock
488    panic("sched p->lock");
491  if(p->state == RUNNING)       // 安全检查:此时不能还是 RUNNING(必须已改成别的态)
492    panic("sched running");
493  if(intr_get())              // 安全检查:切换期间必须关中断
494    panic("sched interruptible");
495
496  intena = mycpu()->intena;
497  swtch(&p->context, &mycpu()->context); // ★★ 与 scheduler 里那次方向相反:存进程现场,载入调度器现场
498  mycpu()->intena = intena;
499}
对称之美(讲这点显水平): 看清这两行就懂了整个机制——
· 进入进程(scheduler 第 463 行):swtch(&c->context, &p->context) — 存调度器、载进程。
· 离开进程(sched 第 497 行):swtch(&p->context, &mycpu()->context) — 存进程、载调度器。
两次 swtch 一进一出,参数刚好对调。进程从 463 行被切进来运行,跑到时钟中断调 yield→sched,在 497 行把自己存好、切回调度器,于是调度器就从它的 463 行“回来”了,落到 464 行继续扫表。调度器和进程,就这样靠两次反向 swtch 互相接力。
三种“让出”的区别(老师爱问):
· 时间片到 → 时钟中断 → yield():state 改成 RUNNABLE(还就绪,排队后面)→ 这就是时间片轮转 RR的抢占。
· 等资源/等事件 → sleep():state 改成 SLEEPING(阻塞)→ 被 wakeup 唤醒后才变回 RUNNABLE。
· 进程结束 → exit():state 改成 ZOMBIE(终止)。
三者都殊途同归:改完 state,都调用 sched(),都用同一行 swtch 切回调度器。

4. 逐行讲 swtch 汇编:上下文切换到底切了什么(来自 kernel/swtch.S)

这是全课最硬核、也最容易拿分的一段。swtch(old, new) 只做两件事:把当前寄存器存进 old,再把 new 里的寄存器加载出来。 RISC-V 调用约定里,参数 old 在寄存器 a0、new 在 a1。只摘关键几行:

8.globl swtch
9swtch:
10  sd ra, 0(a0)      // ★ 把返回地址 ra 存到 old->context(偏移 0)
11  sd sp, 8(a0)      // ★ 把栈指针 sp 存到 old(偏移 8)
12  sd s0, 16(a0)     // 把 s0~s11 依次存进 old
..  ...               // (s1, s2, ... 共 12 个)
23  sd s11, 104(a0)
24
25  ld ra, 0(a1)      // ★ 从 new->context 加载 ra:决定“ret 之后跳到哪”
26  ld sp, 8(a1)      // ★ 从 new 加载 sp:换成新进程的栈 = 换了整个执行现场
27  ld s0, 16(a1)     // 把 new 的 s0~s11 恢复出来
..  ...
38  ld s11, 104(a1)
39
40  ret               // ★ 跳到刚加载的 ra —— 此刻已经在“另一个进程”里执行了
逐句拆解(答辩金句):
① 第 10–23 行 sd(store):把当前的 ra、sp、s0~s11 一个个存进 old 指向的 context。这是“保存现场”。
② 第 25–38 行 ld(load):把 new 里的 ra、sp、s0~s11 加载回 CPU 寄存器。这是“恢复现场”。
③ 第 40 行 ret:跳到刚加载进来的 ra。关键就在这:ra 已经是新进程上次被切走时存的返回地址,所以 ret 一执行,CPU 就回到了新进程当初离开的地方继续跑。
“为什么切 sp 就等于切进程”——这是必考题,务必讲清: 每个进程有自己独立的内核栈,局部变量、函数调用链、返回地址都压在各自的栈上。sp 指向哪条栈,CPU 就“站在”哪个进程的执行现场里。第 26 行把 sp 一换,后面所有 push/pop、所有 ret 找返回地址,都到新进程的栈上去做了——执行流就此整个搬到了另一个进程。 配合换掉的 ra(决定下一跳指令),一存一载之间,进程就被切换了。swtch 没碰页表/通用寄存器 a0–a7、t0–t6,因为切换发生在两个内核线程之间的函数调用边界,调用者已按约定处理好那些,只剩 callee-saved 这 14 个需要 swtch 亲自存。

6. xv6 的调度算法 vs 课本调度算法(对照,显深度)

老师一定会问“这对应课本哪种算法”。把握一个核心:xv6 是“顺序扫描进程表 + 时钟中断强制 yield”,效果近似时间片轮转(RR),但实现极简。

维度xv6 实际做法课本对应
就绪队列没有真链表,用全局数组 proc[NPROC] 顺序遍历就绪队列(逻辑上等价)
选谁从当前位置往后,遇到的第一个 RUNNABLE近似 RR 的“轮流”;也带点 FCFS 的扫描顺序味道
抢占时钟中断到点 → yield() 把当前进程踢回 RUNNABLE时间片轮转 RR(抢占式)
时间片固定:每个时钟滴答(tick)就触发一次 yield等长时间片
优先级没有。所有就绪进程一视同仁课本的优先级 / 短作业优先 / 多级反馈队列 xv6 都没实现
一句话定性(背下来):“xv6 用的是最朴素的时间片轮转:时钟中断每滴答一次就让运行进程 yield,调度器再顺序扫到下一个就绪进程切入。它不做优先级、不做短作业优先,所以是 RR 的极简版——课本的 FCFS/SJF/优先级/多级反馈队列都是在‘选谁’这一步上做更聪明的策略,而 xv6 把‘选谁’简化成了‘顺序找下一个就绪的’。”

8. 答辩:老师可能逐行问的 10 题

逐行讲源码,老师最爱指着某一行问“这行干嘛”。下面是高危行的标准答法。

Q1. 指着 swtch.S 问:swtch 到底切了哪些寄存器?为什么是这些?必答
答:切 14 个:ra、sp、外加 s0~s11 共 12 个。这些是 RISC-V 调用约定里的“被调用者保存(callee-saved)寄存器”。因为 swtch 是被当作一次普通函数调用进入的,调用者保存(caller-saved)的寄存器编译器已经按约定在调用前处理好了,swtch 只需负责 callee-saved 这批。不切页表、不切 a0–a7/t 寄存器。
Q2. 为什么“切 sp 就等于切进程”?高危
答:因为每个进程有独立的内核栈,它的局部变量、函数调用链、压栈的返回地址全在自己的栈上。sp 指向哪条栈,CPU 就站在哪个进程的现场。swtch 第 26 行把 sp 一换,之后所有压栈出栈、所有 ret 找返回地址都到新进程的栈上做了,执行流就整个搬到了新进程。配合换掉的 ra(下一跳指令),进程就被切换了。
Q3. ra 在切换里起什么作用?为什么必须存它?高危
答:ra 是返回地址。swtch 结尾的 ret 会跳到 ra。加载 new 的 ra,意味着 ret 后 CPU 跳到新进程上次被切走时记下的返回点,从那儿继续执行。不存 ra,就不知道切回来后从哪条指令接着跑。
Q4. scheduler 怎么选下一个进程?对应课本哪种算法?必答
答:顺序遍历全局进程表 proc[NPROC],遇到的第一个 state==RUNNABLE 的进程就选它。没有优先级、没有短作业优先。配合时钟中断每滴答强制当前进程 yield,整体是最朴素的时间片轮转(RR)。课本其他算法(FCFS/SJF/优先级/多级反馈)是在“选谁”这一步做更复杂的策略。
Q5. 进程什么时候会被切走?(调度时机)必答
答:三种:① 时间片用完——时钟中断里调 yield(),state 改回 RUNNABLE(抢占);② 主动阻塞——等 I/O、等锁、等事件时调 sleep(),state 改成 SLEEPING;③ 进程结束——exit(),state 改成 ZOMBIE。三者都改完 state 后调用 sched(),统一用 swtch 切回调度器。对应课本“低级调度的时机:运行完成/自动阻塞/发生抢占事件”。
Q6. 上下文(进程现场)保存在哪里?必答
答:保存在每个进程 PCB 里的 struct context(proc.h 第 103 行 p->context),以及每个 CPU 自己的 c->context(第 24 行)。swtch 就是在“某个进程的 context”和“调度器的 context”之间存取。注意它只存内核态切换需要的 14 个寄存器;用户态的完整现场(全部寄存器、程序计数器等)是在进入内核时由 trap 机制存到 trapframe 里的,两者分工不同。
Q7. 第 463 行 swtch 之后那行(464)什么时候才会执行?高危
答:要等被切入的那个进程再次让出 CPU(yield/sleep/exit→sched→swtch 切回)时才执行。所以 463 行“进去”和 464 行“回来”之间,隔着那个进程一整段的运行时间(可能好几毫秒)。同一次 swtch 调用,进入和返回跨越了一次完整的进程运行——这是 swtch 最反直觉、也最精彩的地方。
Q8. 第 453 行 intr_on() 为什么要在循环里开中断?进阶
答:为了保证时钟中断和设备中断能进来。没有时钟中断,就没法在时间片到点时触发 yield,抢占就失效了,RR 也就无从谈起。注释还提到“避免死锁”:如果所有进程都在等中断、而中断又被关着,系统就卡死。而真正执行 swtch 切换的那一小段(在 sched 里)是要求关中断的——见 Q9。
Q9. sched() 开头那几个 panic 检查是干嘛的?进阶
答:切换前的三道安全闸:① 必须持有 p->lock(保护切换过程);② 此刻 state 不能还是 RUNNING(说明调用者已经把状态改成 RUNNABLE/SLEEPING/ZOMBIE 了,否则切走后没人知道它该是什么态);③ 中断必须是关闭的(切换过程不能被打断)。任一不满足就 panic,把潜在的并发 bug 立刻暴露出来。
Q10. yield 把 state 设成 RUNNABLE 而不是 SLEEPING,区别在哪?对比
答:RUNNABLE = 就绪:进程没在等任何东西,只是时间片到了被踢回队列,下一轮调度器扫到它还能继续跑。SLEEPING = 阻塞:进程在等某个事件(I/O 完成、锁释放),必须被 wakeup 唤醒变回 RUNNABLE 后,调度器才会再选它。这正是课本“就绪态”和“阻塞态”的区别——yield 是时间片轮转的让出,sleep 是因事件而阻塞。
