死锁检测与解除:资源分配图,一步步“化简”出真相

这一课只回答一件事:给你一张资源分配图,怎么用“化简法”判断系统到底死没死锁,以及死了怎么救。配合 速查地图 使用。

0·全局线 1·化简器 2·ELI5 3·对照表 4·是什么 5·为什么 6·自测
一句话先记住: 反复找一个“它要的资源现在都能给”的进程,让它跑完、把资源全还回去变成孤立点;全部能变孤立 = 图可完全化简 = 无死锁;化不动、剩下一圈 = 不可完全化简 = 那些进程死锁。

🗺 0. 全局故事线:处理死锁的四条路,本课站在最后一条

别把“检测”当孤立知识点——它是处理死锁四种策略里最后、也最“事后”的一条。

① 预防破坏四个必要条件之一(如一次分配全部、可剥夺、按序申请),最严
→
② 避免银行家算法,分配前先算安全序列,不安全就不给
→
③ 检测(本课)先让它跑,事后周期体检:画资源分配图→化简→判死锁
→
④ 解除(本课)确诊后动手:剥夺资源 / 撤销进程 / 回退,挑代价最小的

口径:预防&避免是“事前不让它发生”,检测&解除是“先发生、再确诊、再救治”。[textbook/ch03_处理机调度与死锁.md · 3.8;3_处理机调度/ppt文本.md · 幻灯片 346]

🖼 1. 先看画面:亲手把一张图“化简”到底

🧸 一句话: 圆圈是进程、方框是资源;箭头从资源指向进程 = “已经分给你”(分配边),从进程指向资源 = “我还想要”(请求边)。化简 = 找一个“想要的现在货架上都有”的进程,让它拿到→跑完→把手里的全还回货架→它就没边了,变成孤立点。
👇 点“化简一步”:系统会高亮它这一步为什么选它,然后消去它的所有边使其孤立;一直点到走不动,底部给出死锁判定。可切换两张示例图。

要点: 化简的唯一判据是——“该进程当前请求的资源都能被满足”(它请求的每类资源数 ≤ 当前空闲数)。满足就让它跑完释放,不满足就先跳过。[3_处理机调度/ppt文本.md · 幻灯片 360]

🧸 2. 掰开揉碎:一个日常例子 + 一个反例

🧸 例子: 车间里几个工友共用一批工具。资源分配图就是墙上那张“谁手里攥着哪把工具、谁伸手要哪把工具”的图。化简 = 你挨个看:“张三还差的工具,货架上现在正好都有” → 让张三干完活,把他手里的工具全部还回货架 → 张三下班走人(变孤立点)。他一还,货架变多,可能李四本来卡着的活现在也能干了。所有人都能这样陆续下班 = 没死锁。
⚠️ 反例(如果没有检测会怎样): 假如四个工友围成一圈,每人都攥着下一个人要的工具、又都伸手要上一个人手里的——谁也不肯先还,谁也拿不到。没有“化简/检测”这套体检手段,系统只会看到 CPU 利用率悄悄掉到 0、一堆进程永久阻塞,却不知道这是死锁。检测就是给系统做“事后确诊”的手段。[textbook/ch03_处理机调度与死锁.md · 死锁检测算法]

上面交互件的示例B就是这个反例:P1→R2、P2→R3、P3→R1 首尾相接,三人抢成一个环,空闲资源全是 0,谁也化简不动 → 死锁。

⚖️ 3. 对照表:检测 vs 银行家,别再混

🧸 一句话: 银行家是“打疫苗”(分配前先算,宁可不给也不让你进危险),检测是“看急诊”(先让你病,病了再查再治)。

⚖️ 检测化简 vs 银行家避免(照教材口径)

比什么银行家算法(避免)资源分配图化简(检测)
时机事前:每次分配前做安全性检查事后:分配照给,周期性 / 阻塞时才体检
核心动作找一条安全序列,找不到就拒绝这次分配反复约简可满足的进程,看图能否完全化简
判据Need ≤ Work 时假设它能跑完并释放Request ≤ Work(空闲)时消其边使之孤立
结论含义“安全 / 不安全”(不安全≠已死锁)“可完全化简 = 无死锁 / 否则有死锁”
代价每次分配都算,平时开销大,但不会死锁可能真死锁;换来分配灵活,但要额外解除
结论: 两者算法骨架很像(都在“找一个能跑完的进程→释放→推下去”),差别只在“什么时候做、做完拿去干嘛”:银行家拿去决定“这次给不给”,化简拿去判定“已经死没死”。[textbook/ch03_处理机调度与死锁.md · 3.7 银行家 / 3.8 检测]
⚠️ 易错陷阱: “图里有环 = 死锁” 是错的! 当每类资源只有 1 个实例时,有环才 ⇔ 死锁;若资源有多个实例,有环也可能不死锁(别的实例还能满足别人)。判死锁的正解永远是“能否完全化简”,不是“有没有环”。[3_处理机调度/ppt文本.md · 幻灯片 350 “有环不一定导致死锁”]

📖 4. 是什么:精确定义 + 解除手段

🧸 一句话: 把上面的直觉,换成能写进卷子的标准话。
资源分配图(Resource Allocation Graph): 用圆圈表示进程结点、方框表示一类资源(框内圆点 = 该类资源的一个实例);请求边由进程指向资源(进程请求该类资源一个实例)、分配边由资源指向进程(已分配一个实例给它)。[3_处理机调度/ppt文本.md · 幻灯片 345;textbook/ch03 · 资源分配图组成]
化简法(约简步骤): ⑴在图中找一个“可获得它所需全部资源”的进程结点 Pi(即它的请求当前都能满足),让它拿到资源跑完、释放全部资源,消去它的请求边与分配边,使之成为孤立点;⑵再找下一个这样的 Pj 重复;⑶直到没有可约简的进程。[3_处理机调度/ppt文本.md · 幻灯片 347]
死锁定理: 系统处于死锁状态的充分必要条件是——当且仅当其资源分配图是不可完全化简的。全部结点都能约简为孤立点 = 可完全化简 = 无死锁;化到最后残留的那些进程,就是处于死锁的进程。[3_处理机调度/ppt文本.md · 幻灯片 353;textbook/ch03 · 死锁定理]

🚑 死锁的解除(确诊之后怎么救)

📐 三类解除手段 + 一条总原则:
  1. 剥夺(抢占)资源:从一个或多个死锁进程手里强行抢走足够资源,分给别人打破僵局。要考虑抢谁、抢的代价、别让被抢者饥饿。
  2. 撤销进程:终止一个或多个死锁进程,直到循环等待被打破;可全撤(代价最大),更常用的是逐个撤、每次挑代价最小的。
  3. 进程回退(rollback):让进程回退到之前的某个安全检查点、交出那之后占的资源,再重新推进(需要系统保存历史状态)。
总原则——代价最小:撤销那些代价最小的进程,或使被撤销的进程数量最少。衡量代价常看:优先级低、已运行时间短、剩余时间长、占有资源少的,优先被撤。[3_处理机调度/ppt文本.md · 幻灯片 364-365;textbook/ch03 · 死锁的解除]

口径提醒:检测与解除是“相互合作”的关系——检测负责确诊,解除负责收回资源。[3_处理机调度/ppt文本.md · 幻灯片行 909]

❓ 5. 为什么这样判:层层追问

🧸 一句话: 我们不是随便约简,每一步背后都有硬道理。

第 1 层:为什么用“请求能否被满足”当判据?

因为一个进程只要能拿到它还差的资源,就一定能跑完;跑完必然归还它占的一切。所以“Request ≤ 空闲”这一步是一次乐观推演:假设它拿到就一定结束。能推到底,说明存在一个让所有进程都跑完的顺序,自然无死锁。

第 2 层:为什么化简顺序不影响最终结论?

因为“释放资源”只会让别的进程更容易被满足,绝不会更难。所以无论你先约简谁,可约简的机会只增不减——最终“能否完全化简”是唯一确定的。这就是为什么交互件里随便挑一个可满足的进程都行,结论一致。

第 3 层:为什么这是“事后”而不是“事前”?边界在哪?

银行家在分配前就问“分了会不会进入不安全状态”,宁可让进程等——代价是每次分配都要算、系统开销大。检测法反过来:先允许系统陷进死锁,再靠周期性(或 CPU 利用率骤降、进程久阻塞时)体检发现它,然后解除。换来的是分配灵活,代价是解除时一定有损失(剥夺/回退/撤销都要作废一部分工作)。这就是“打疫苗 vs 看急诊”的取舍。[textbook/ch03 · 死锁检测的时机 / 死锁的解除]

选择题 1

在资源分配图化简法中,一个进程结点可以被约简(消去其边、变成孤立点)的条件是?

A. 它当前没有持有任何资源
B. 它请求的资源当前都能被满足
C. 它是优先级最高的进程
D. 它已经进入阻塞队列

点我看答案

答案:B。 化简的唯一判据就是“它请求的每类资源数 ≤ 当前空闲数”,满足才能让它跑完并释放。A 错:持有资源恰恰意味着它非孤立、正是可约简的候选,不持有不是条件。C 错:化简与优先级无关,顺序不影响结论。D 错:进入阻塞队列说明它的请求得不到满足,反而不能约简。

选择题 2

关于“资源分配图可完全化简”,下列说法正确的是?

A. 只要图中有环就一定不可完全化简
B. 化简顺序不同,可能得出不同的死锁结论
C. 图可完全化简 ⇔ 系统无死锁
D. 不可完全化简说明系统总资源不足

点我看答案

答案:C。 这正是死锁定理:可完全化简当且仅当无死锁。A 错:资源多实例时“有环不一定死锁”,判据是能否完全化简而非有没有环。B 错:释放资源只会让别人更易满足,最终能否化简是唯一确定的,与顺序无关。D 错:不可化简是“互相等待成环”的结构问题,不等于总量不足——加不加资源都可能救不了当前僵局。

选择题 3

死锁检测(化简法)与银行家算法的根本区别是?

A. 检测是事后诊断,银行家是事前避免
B. 检测通过破坏四个必要条件之一来工作
C. 银行家允许系统先进入死锁、再解除
D. 两者都要求进程一次性申请全部资源

点我看答案

答案:A。 银行家在分配前算安全序列(避免/事前);化简在系统运行后周期体检判死锁(检测/事后)。B 错:破坏必要条件是“预防”策略,不是检测。C 错:说反了——是检测法允许先死锁再解除,银行家从不让系统进入不安全状态。D 错:“一次性申请全部”是预防里破坏“请求和保持”的手段,与这两者无关。

选择题 4

解除死锁时,按“代价最小原则”撤销进程,通常优先撤销哪种进程?

A. 持有资源最多的进程
B. 已运行时间很长、快要完成的进程
C. 正在等待 I/O 的进程
D. 优先级低、占有资源少、已运行时间短的进程

点我看答案

答案:D。 代价最小 = 撤了不太可惜:优先级低、已运行时间短(重跑损失小)、占资源少的先撤。A 错:持有资源多的撤了确实能一次收回很多,但它往往也是投入大、代价高的,不是“代价最小”的首选。B 错:快完成的进程撤掉最可惜,前功尽弃。C 错:是否等 I/O 与撤销代价无直接关系。
