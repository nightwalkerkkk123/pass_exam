连续分配 & 动态分区:FF / NF / BF / WF

这一课只回答一件事:内存怎么整块整块地分给作业?先动手点一个模拟器,看四种“挑法”结果不一样,再上术语。配合 速查地图 使用。

一句话先记住: 连续分配就是“给一个程序一整块连着的内存”;动态分区按作业大小现切一块给它,切来切去,分区之间会留下一些“太小、谁也用不上的小空隙”——这就是外部碎片。而 FF / NF / BF / WF 只是“从一堆空闲块里挑哪一块给它”的四种挑法而已。

🖼 1. 直觉图:签名交互件「动态分区分配模拟器」

👇 先在上排选一种算法,再反复点【分配下一个作业】。看同样的内存、同样的作业队列,四种算法挑中的分区完全不同——尤其对比 FF 和 BF。

挑法:
💾 空闲分区(按地址从左到右排列，宽度 ≈ 容量)
📋 作业队列(依次请求内存)
← 已选【首次适应 FF】。点【分配下一个作业】开始。
看懂这个模拟器,本节就懂一半了: 每次分配都是“从空闲块里挑一块 → 切一刀 → 剩下的还是空闲块”。挑法不同,剩下的碎片形状就不同;切到最后,可能空闲总量还够、却没有一整块装得下——那就是外部碎片,只能靠“紧凑”来救。

🧸 2. 具体例子(ELI5):停车场找车位

🧸 5 岁也懂的比喻: 内存像一排大小不一的空车位,作业就是要停进来的车。四种“挑法”就是四种停车习惯:
• FF 首次适应——从入口开始,遇到第一个塞得下的车位就停进去(不挑,图快)。
• NF 循环首次——从上次停车的地方接着往后找,找到第一个够大的就停(不总从头挤在入口)。
• BF 最佳适应——专挑刚好塞得下的最小车位,把大车位省给大车(看着最“省”)。
• WF 最坏适应——专挑最大的车位停,想让剩下的空间还够再停别的车。
🧸 那“外部碎片”是什么? 车停久了、进进出出,车位之间会剩下一些“太窄、任何车都停不进去”的缝。把这些缝加起来其实能停好几辆车,可惜它们东一条西一条、连不成一整块——这些没法用的空隙,就叫外部碎片。想用它们,只能把已停的车全部往一头挪、挪拢,腾出一大块——这就是后面要讲的紧凑。

⚖️ 3. 对照表(照教材口径)

对照一:三种「连续分配」方式

方式怎么分碎片情况数据结构 / 特点
单一连续分配 内存分成系统区 + 用户区,用户区只放一道作业 有内部碎片 单道、只用于单用户单任务;利用率极低
固定分区分配 预先把内存划成若干固定大小的分区,每区装一道作业 有内部碎片、无外部碎片 用分区说明表登记每个分区的大小/起址/状态
动态分区分配 不预先划,按作业大小现切一块正好够用的 无内部碎片、有外部碎片 用空闲分区表 / 空闲分区链管理空闲区;要靠 FF/NF/BF/WF 挑块

💡 记忆钩子:固定分区=装不满一格 → 内碎片;动态分区=格与格之间的缝 → 外碎片。这是最爱考的一对辨析。

对照二:四种「动态分区分配算法」

算法空闲链怎么排怎么挑优点缺点
FF 首次适应 按地址递增 从低地址找第一个够用的 简单快;优先用低地址、保留高地址大块 低地址越切越碎、生小碎片
NF 循环首次 按地址递增(成环) 从上次结束的位置继续往后找第一个够用的 空闲区分布更均匀 会分割高地址大块、容易缺乏大的连续空闲区
BF 最佳适应 按容量递增 选能装下里最小的那块 尽量保留大块给大作业 产生大量难用的小碎片;每次几乎要遍历整表
WF 最坏适应 按容量递减 选最大的那块 剩下的碎片较大、容易再被利用 很快耗尽大块,大作业进不来

💡 排序口诀:FF/NF 按地址排,BF 按大小从小到大,WF 按大小从大到小。BF 和 WF 一“小”一“大”正好相反。

📖 4. 是什么:精确定义(对齐教材)

内部碎片 vs 外部碎片(必考辨析):
• 内部碎片 = 分配给作业的分区“内部”没用上的空间(分区比作业大,多出来的塞在里面浪费掉)。固定分区、分页会有。
• 外部碎片 = 分区与分区“之间”那些太小、无法被利用的空闲块。动态分区、连续分配会有。

动态分区的数据结构(怎么记住哪些是空闲的):

结构说明
空闲分区表用一张表登记每个空闲分区的起始地址、大小、状态。
空闲分区链用链表把所有空闲分区串起来,分配 / 回收时增删链上的结点。FF/NF/BF/WF 决定这条链按什么顺序排。
紧凑(拼接)技术: 当外部碎片攒多了、单块凑不出足够空间时,把内存里所有正在运行的进程都搬动、使它们相邻,从而把分散的多个小空闲分区合并成一个大的连续空闲区。
• 硬件前提:必须有动态重定位支持,也就是重定位寄存器——因为进程被搬走了、物理地址变了,程序里的地址要能跟着自动改,否则一搬就跑飞。
• 代价:移动进程要耗大量 CPU、还要暂停相关进程,很贵。
• 时机:分配失败时做一次,或定期做。

❓ 5. 为什么:逐层往下挖 3 层

第 1 层:为什么会有外部碎片?

因为动态分区是反复地分配、回收:每来一个作业就从某块空闲区切一刀,走了又还回来。切来切去,原本大而完整的空闲区被剁得越来越零碎。到后来,空闲总量明明还够(比如上面模拟器里剩了 959K),却拼不出一整块 426K 连续的地方——这些拼不起来的零头,就是外部碎片。

第 2 层:为什么要搞出四种算法?它们在权衡什么?

核心矛盾只有一句话:“是把大块留住,还是把碎块用掉?”
• BF 想把大块留给大作业,于是每次挑最小的够用块——结果切出一堆比谁都小、谁也用不上的碎渣。
• WF 反过来,每次挑最大的、让剩余仍然可用——但大块很快被用光,后面来的大作业没地方去。
• FF 简单最快,但总从低地址挤,低地址一带碎片扎堆。
• NF 让分配点在内存里“转圈”,分布更均匀,但代价是把高地址的大块也切开了。
所以没有一种绝对最优——每种都在“留大块 / 用碎块 / 查得快 / 分布匀”之间做取舍,选哪种看场景。

第 3 层:为什么最后要靠紧凑?它贵在哪、凭什么能做?

再聪明的挑法也消灭不了外部碎片,只能推迟它。碎片攒够了,唯一的救法就是把进程搬到一起、把空隙挤到一头合并成大块——这就是紧凑。
再问一句“凭什么能搬?” → 进程一搬,它的物理地址就变了,程序里写死的地址会全部失效。所以必须有动态重定位(重定位寄存器):CPU 每次访问时用“重定位寄存器里的新基址 + 逻辑地址”临时算出真实物理地址,进程搬到哪、基址改到哪,程序照跑不误。
再问一句“所以呢?” → 紧凑这么贵(搬进程、停 CPU),让人忍不住想:与其费劲搬,不如干脆别要求“连续”——把作业拆成小块离散地塞进内存,从根上消灭外部碎片。这正是下一步分页 / 分段的动机。

✅ 6. 自测:先自己想,再点开答案

选择题 1

最佳适应算法(BF)的空闲分区链是按什么排列的、分配时又选哪一块?

A. 按地址递增排列,选第一个够用的
B. 按容量递减排列,选最大的一块
C. 按地址递减排列,选最后一个够用的
D. 按容量(大小)递增排列,选能满足要求的最小分区

点我看答案

答案:D。 BF 的空闲链按容量从小到大排,这样从头找到第一个够用的就自然是“最小的够用块”。
陷阱:A 说的是 FF(按地址排);B 描述的是 WF(按容量递减、选最大)。BF 是按“大小”排,不是按“地址”排——这一点最容易记反。

选择题 2

动态分区分配方式产生的,是内部碎片还是外部碎片?

A. 内部碎片——分配给作业的分区内部没用上的那块空间
B. 外部碎片——分区与分区之间那些太小、装不下作业的空闲块
C. 两种都没有,动态分区不产生任何碎片
D. 只有固定分区才有碎片,动态分区完全没有碎片

点我看答案

答案:B。 动态分区按需现切,不会在分区内部留浪费(所以没有内部碎片),但反复切割会在分区之间留下零碎空隙 → 外部碎片。
陷阱:A 正好是内部碎片的定义,专门用来骗把两者记反的人——那是固定分区 / 分页的毛病。C、D 都错:动态分区确实有(外部)碎片。

选择题 3

“紧凑(拼接)”技术能够顺利进行,依赖的硬件前提是什么?

A. 动态重定位(重定位寄存器)
B. 更大的内存容量
C. 更快的 CPU 主频
D. 空闲分区表必须按大小排序

点我看答案

答案:A。 紧凑会搬动进程,进程的物理地址随之改变,必须靠动态重定位(重定位寄存器)在运行时用“新基址 + 逻辑地址”重新算出物理地址,程序才能照常跑。
陷阱:B、C 是硬件性能,和“能不能搬”无关;D 是软件层面的空闲链组织方式,不是紧凑的硬件前提。

选择题 4

“从上次查找结束的位置继续往下找第一个够用的分区”,说的是哪种算法?

A. 首次适应 FF
B. 最佳适应 BF
C. 循环首次适应 NF
D. 最坏适应 WF

点我看答案

答案:C。 NF(Next Fit)= 循环首次适应,每次接着上次的位置往后找(到头就绕回开头),好处是分布更均匀。
陷阱:A(FF)是每次都从头(低地址)开始找,这正是 NF 想避免的“低地址扎堆”;B/D 是按容量挑,不是“接着上次位置找”。

口述题(别打字,张嘴说)

  1. 不看屏幕,用一句话各说清 FF / NF / BF / WF 各自怎么挑块,以及 BF、WF 的空闲链分别按什么排序。
  2. 说清内部碎片和外部碎片的区别,并各举一个会产生它的分配方式(内部→固定分区;外部→动态分区)。
  3. 说出紧凑是干嘛的、为什么非要动态重定位不可,以及它为什么会引出“干脆用离散分配(分页)”的想法。

三题都能顺口说出来 → 你是真懂了。哪题卡壳,回到对应小节,再去上面的模拟器点几下 FF 和 BF 对比着看。
