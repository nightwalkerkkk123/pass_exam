0. 先记这句主线(讲偏了就拉回来)

“用户程序把要调用的系统调用号放进寄存器 a7,执行 ecall 触发陷入(trap)进入内核;内核的 syscall() 拿这个号去函数指针表 syscalls[] 里查表,找到对应的内核函数 sys_xxx 并调用,把返回值写回 a0 交给用户。”
🍽️ ELI5 比喻(开场可以用):“这就像餐厅点餐——你不会冲进后厨自己炒菜(用户态不能直接碰硬件),你只把菜单号报给服务员(把系统调用号放进 a7),按一下铃(ecall 陷入)。后厨(内核)按号查菜谱(syscalls[] 表)做菜,做好把菜端回你桌上(返回值放进 a0)。报的号不存在,后厨就回你‘没这道菜’(返回 -1)。”

1. 背景:用户态 / 核心态与“陷入”(引出 a7、a0)

课本第 1 章讲:CPU 有两种运行状态——用户态(目态)和核心态(管态)。这是理解系统调用的前提:

状态谁在跑能干什么
用户态(目态)普通应用程序不能执行特权指令、不能直接访问硬件、不能访问内核空间
核心态(管态)操作系统内核执行特权指令、访问硬件、管理内存、处理中断

用户程序想读文件、想创建进程,都得借内核之手。它怎么“喊”内核? 不能直接 call 一个内核函数(那是跨特权级,硬件不允许),只能用课本说的「陷入指令 trap」。在 RISC-V 上这条指令叫 ecall。

用户侧入口:库函数其实只有 3 行汇编

课本说“高级语言提供与系统调用一一对应的库函数”。在 xv6 里,这些库函数(stub)由 user/usys.pl 脚本自动生成,每个就 3 行:

9sub entry {                 // usys.pl:为每个系统调用生成一段汇编 stub
13    print " li a7, SYS_${name}\n"; // ① 把“系统调用号”装进 a7 寄存器
14    print " ecall\n";            // ② ecall = 陷入指令 trap,切到核心态
15    print " ret\n";              // ③ 内核返回后,带着 a0 里的结果回到用户程序
}
讲这页时说:“你在 C 里写的 getpid(),展开后就是这 3 行:把号填进 a7 → ecall 陷入 → 回来。li a7, SYS_getpid 这一句,就是课本说的‘库函数与系统调用一一对应’——库函数干的活,核心就是替你把系统调用号填好、按下陷入指令。”

系统调用号:一个 #define 数字

所谓“系统调用号”,就是 kernel/syscall.h 里一串 #define——每个系统调用配一个整数编号:

1// System call numbers — 每个系统调用一个号
2#define SYS_fork    1
3#define SYS_exit    2
11#define SYS_getpid 11      // getpid 的号是 11
16#define SYS_open   15
22#define SYS_close  21

记住这个号:用户侧 li a7, SYS_getpid 就是把 11 装进 a7。内核侧 syscall() 待会儿就拿这个 11 去查表。a7 是“点菜单号”,a0 是“上菜口”。

2. 逐行讲 syscall()(来自 kernel/syscall.c)

背景注释先念:“陷入进内核后,trap 处理流程最终会调到 syscall()。它的任务:看 a7 里是哪个号,把活派给对应的内核函数。” 这是本课的核心代码,只有 16 行:

131void
132syscall(void)
133{
134  int num;
135  struct proc *p = myproc();          // 取当前进程
136
137  num = p->trapframe->a7;               // ★① 从 a7 取出“系统调用号”
138  if(num > 0 && num < NELEM(syscalls) && syscalls[num]) { // ★② 号合法 且 表里有这个函数
139    // Use num to lookup the system call function for num, call it,
140    // and store its return value in p->trapframe->a0
141    p->trapframe->a0 = syscalls[num]();  // ★③ 查表分发 + 调用,返回值写回 a0
142  } else {
143    printf("%d %s: unknown sys call %d\n",
144            p->pid, p->name, num);
145    p->trapframe->a0 = -1;             // ★④ 非法号 → 返回 -1 给用户
146  }
147}

把 16 行切成“四步命运”讲(这是你讲解的主结构)

步骤行一句话讲法对应课本
① 取号137“p->trapframe 是陷入时保存下来的用户寄存器快照。a7 就是用户刚才填的菜单号,这里把它读出来。”系统调用号
② 校验138“检查号 大于 0、小于表长(NELEM 算数组元素个数)、并且表里那一格非空。三道关,防止用户乱报一个号让内核跑飞。”系统调用的安全检查
③ 查表分发141“syscalls[num] 取出函数指针,后面 () 就是调用它;返回值直接赋给 a0。一个数字 → 一个内核函数 → 返回值回写,全在这一行。”程序接口/服务分发
④ 出错143–145“号非法就打印警告,并把 a0 设成 -1。用户程序回去一看返回值是 -1,就知道调用失败了。”系统调用返回值约定

3. 核心:syscalls[] 怎么把“一个数字”变成“一个内核函数”

第 141 行 syscalls[num]() 是全课的灵魂。它能成立,靠的是顶部这张函数指针数组(来自 kernel/syscall.c):

105// An array mapping syscall numbers from syscall.h
106// to the function that handles the system call.
107static uint64 (*syscalls[])(void) = {  // ★ 一个“函数指针数组”:下标=调用号,元素=内核函数
108[SYS_fork]    sys_fork,                // 下标 1  → sys_fork
109[SYS_exit]    sys_exit,                // 下标 2  → sys_exit
118[SYS_getpid]  sys_getpid,              // 下标 11 → sys_getpid
122[SYS_open]    sys_open,                // 下标 15 → sys_open
128[SYS_close]   sys_close,               // 下标 21 → sys_close
129};
讲法(答辩金句):“uint64 (*syscalls[])(void) 读作‘一个数组,每个元素是指向「无参、返回 uint64」的函数的指针’。[SYS_getpid] sys_getpid 是 C 的指定初始化器,意思是‘把数组第 SYS_getpid(=11)格,填上 sys_getpid 这个函数的地址’。于是 syscalls[11] 就是 sys_getpid。这张表,就是课本说的‘系统调用号’到‘服务程序’的映射——用 C 的语言写出来,就是一个数组下标查找。”
为什么用函数指针数组,而不是一长串 if/switch? 21 个系统调用,若写成 if(num==1) sys_fork(); else if(num==2)... 要 21 个分支,慢且难维护。用数组,查表是 O(1) 一步到位,加一个系统调用只要在表里加一行——这就是“用数据结构代替控制流”的经典做法。

派到的内核函数长什么样:以 sys_getpid 为例

表里指向的 sys_xxx 函数,才是真正干活的。最简单的一个(来自 kernel/sysproc.c):

18uint64
19sys_getpid(void)
20{
21  return myproc()->pid;        // 返回当前进程的 pid,这个返回值会被写进 a0
22}

带参数的系统调用(如 sys_exit(int n))又怎么拿到参数?用 syscall.c 里的辅助函数 argint / argaddr,它们去 a0~a5 寄存器里取(用户陷入前把参数放在那):

56void
57argint(int n, int *ip)        // 取第 n 个整型参数
58{
59  *ip = argraw(n);              // argraw(n) 按 n 返回 a0/a1/.../a5
60}
一句话串起来:“调用号走 a7,参数走 a0~a5,返回值走 a0。这套‘谁放哪个寄存器’的约定,就是系统调用的调用约定(ABI)——用户和内核约好的暗号。”

6. 答辩:老师可能逐行问的 10 题

逐行讲源码,老师最爱指着某一行问“这行干嘛”,也爱问概念对比。下面是高危题的标准答法。

Q1. a7 是什么?为什么调用号放在 a7?必答
答:a7 是 RISC-V 的一个通用寄存器。系统调用的调用约定规定:用 a7 传系统调用号、a0~a5 传参数、a0 传返回值。用户陷入前把号写进 a7,内核 syscall() 第 137 行就从 p->trapframe->a7 把它读出来。这是用户和内核约好的“暗号位置”。
Q2. 指着 137 行 num = p->trapframe->a7 问:trapframe 是什么?高危
答:trapframe 是陷入(trap)发生时,内核保存的用户寄存器快照。因为陷入后内核要用这些寄存器干自己的活,必须先把用户的 a0~a7 等存起来,返回时再恢复。所以这里读 trapframe->a7,读的就是用户陷入那一刻 a7 里的号。
Q3. 指着 138 行的三个条件,为什么要这么严格地校验?高危
答:因为号是用户给的,不可信。num>0 防负数,num<NELEM(syscalls) 防越界(NELEM 是数组元素个数),syscalls[num] 非空防“号在范围内但那格没填函数”。少一道关,用户随便报个号就可能让内核跳到非法地址——这是安全底线。
Q4. 指着 141 行 p->trapframe->a0 = syscalls[num]() 问:这一行到底发生了什么?必答
答:三件事压在一行:① syscalls[num] 用调用号当下标,从函数指针表取出对应内核函数的地址;② 后面的 () 调用它;③ 把它的返回值赋给 trapframe->a0。这样返回用户态后,a0 里就是结果。“查表、分发、回写”全在这。
Q5. syscalls[] 为什么用函数指针数组,而不是一堆 if/switch?高危
答:① 快:数组下标访问是 O(1),不像 if/else 要逐个比;② 清晰可扩展:加系统调用只需在表里加一行,不改 syscall() 主逻辑;③ 它直接对应课本“系统调用号 → 服务程序”的映射表概念。本质是“用数据结构代替控制流”。
Q6. 系统调用和普通函数调用、库函数有什么区别?必答 · 高频考点
答:三层:普通函数调用是同一特权级里的 call,不切状态、开销小;系统调用必须经陷入指令(ecall)从用户态切到核心态,要保存/恢复现场、做安全检查,开销大但能安全地用特权资源;库函数是对系统调用的封装(xv6 里就是 li a7;ecall;ret 那 3 行),它本身仍在用户态,真正进内核的是其中的 ecall。课本原话:库函数与系统调用一一对应。
Q7. 系统调用的返回值是怎么传回用户程序的?必答
答:走 a0。第 141 行把内核函数返回值写进 trapframe->a0;返回用户态时,内核会用 trapframe 恢复寄存器,a0 就被还原成这个值。用户的库函数 ret 回去后,a0 里就是结果。出错时第 145 行把 a0 设成 -1,用户据此判断失败。
Q8. 用户程序怎么进到 syscall() 的?谁调用它?进阶
答:不是被“调用”,是被“陷入”引来的。用户执行 ecall 触发陷入,CPU 跳到内核预设的 trap 入口,trap 处理流程判断这是一次系统调用(而非时钟中断、缺页等),于是调用 syscall()。所以 syscall() 是 trap 处理的一个分支。
Q9. 带参数的系统调用,内核怎么拿到参数?进阶
答:用 syscall.c 里的 argint / argaddr / argstr。它们调 argraw(n),按 n 返回 a0~a5 寄存器里的值——参数也是陷入前由用户放进这些寄存器的。比如 sys_exit 里 argint(0,&n) 就是取第 0 个参数(放在 a0)。
Q10. 报一个不存在的系统调用号会怎样?易问
答:走 else 分支(143–145):内核 printf 打印 "unknown sys call" 警告,并把 a0 设为 -1。注意它不会让内核崩溃——这正是第 138 行三道校验的价值:把非法输入挡在分发之前,只温和地返回错误。
