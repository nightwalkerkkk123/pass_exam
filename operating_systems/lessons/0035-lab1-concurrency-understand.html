实验① 并发程序设计 · 理解版

两个进程同时改一个数，为什么会"丢"?

这一页不教你上台讲，只让你自己真正看懂一件事:并发时多个进程抢着改同一个变量，结果会出错（竞态）；"锁"用一个不可打断的原子操作保证一次只有一个进去改。看懂了，再去看讲解版（第 36 课）。

对应教材 Ch2 进程同步 对应 xv6 kernel/spinlock.c 的 acquire() 关键词:临界区 · 互斥 · 原子操作

1先建立一个画面

🧸 ELI5

把那个共享变量想成一间只有一把钥匙的公共卫生间。一次只能进一个人，钥匙挂在门口。

正常情况:你拿钥匙进去、锁上、出来、把钥匙挂回——下一个人才能进。这就是"互斥"。

出问题的情况:钥匙没了，A 和 B 同时推门进去。两人都以为里面没人，各干各的，最后互相把对方做的事抹掉了。在程序里，这就是两个进程同时改一个变量、结果对不上的"竞态条件"。

2核心矛盾:一句 count++ 其实是三步

你以为 count++ 是一步完成的，其实 CPU 要分三步:

步骤动作说明
① 读 (read)把内存里的 count 读进寄存器"现在是几?"
② 改 (modify)寄存器里 +1"加一"
③ 写 (write)把结果写回内存 count"存回去"

麻烦就在:这三步中间随时可能被切换走，让另一个进程插队。一插队，两边读到的是同一个旧值，各自 +1 再写回，两次加一只生效了一次。

做法结果安全吗
不加锁，读-改-写交错两次 ++ 只涨了 1
→ 丢失更新、数据错乱		❌
加锁，一次只放一个进临界区两次 ++ 老老实实涨 2
→ 串行、安全		✅

3动手:亲眼看一次"丢失更新"

下面 A、B 两个进程都要执行一次 count++（即"读→改→写"三步）。不加锁时，你每点一次"单步"，就让下一个进程走它的下一步——故意让它们交错，看 count 怎么算错。再切到加锁模式，看它怎么被强制串行、算对。

共享变量 count = 0
🅰 进程 A  寄存器=?
① read：r ← count
② modify：r ← r + 1
③ write：count ← r
🅱 进程 B  寄存器=?
① read：r ← count
② modify：r ← r + 1
③ write：count ← r
← 选模式，然后点"单步交错执行"。目标:两个进程各 ++ 一次，count 应该 = 2。
锁:关（点我切换）
🔑 你应该看出来的事

不加锁:如果 A 读了 0、还没写回，就被切到 B；B 也读到 0、加成 1 写回；A 再回来把自己手里的 1 写回——两次 ++ 最后 count 还是 1，丢了一次。这就是竞态条件:结果取决于"谁先谁后"这种偶然的执行顺序。

加锁:A 先拿到锁，B 想进只能原地等（忙等待 / spin），直到 A 把"读改写"三步整段做完、放锁。B 才进去，读到的是 1，加成 2。"读改写"被锁保护成了不可分割的一整块——这块就叫临界区。

4那"锁"本身靠什么保证不被插队?

这里有个套娃问题:锁是用来防止 count 被同时改的，可"locked 这个变量"本身也是个共享变量啊——两个进程会不会同时把它从 0 抢成 1，都以为自己拿到了锁?

💡 解决方案:原子操作（atomic）

硬件提供一条"测试并设置"指令:把 locked 读出来、同时写成 1，这两件事在一条 CPU 指令里一气呵成，中间绝不可能被打断。在 xv6 里就是 __sync_lock_test_and_set(&lk->locked, 1)(底层是 RISC-V 的 amoswap 原子交换)。

它返回 locked 的旧值:旧值是 0 → 说明刚才没人占，你抢到了；旧值是 1 → 别人占着，你继续转圈等。因为"读+写"原子化，绝不会出现两个进程都拿到旧值 0 的情况。原子操作就是整座锁的地基。

⚠ 还有一步:acquire 里为什么要先关中断?

xv6 的 acquire() 第一行是 push_off()——关本 CPU 的中断。原因:如果我拿着锁，这时来个中断、中断处理程序又想拿同一把锁，它会原地 spin 等我放锁，可我被它打断了根本没机会放——自己把自己锁死（死锁）。所以持锁期间先关中断，放锁时(pop_off())再开回来。

5延伸:等不到资源，与其干转，不如睡一觉

自旋锁让等待者原地转圈（忙等待）,适合"马上就能拿到"的短临界区。可如果要等的是"磁盘数据还没来"这种很久的事，干转就是白白烧 CPU。

💡 sleep / wakeup:让出 CPU，别人备好了再叫醒你

xv6 的 sleep(chan, lk):进程把自己标成 SLEEPING、挂在某个"等待频道 chan"上，然后主动让出 CPU给别人跑。等资源准备好了，另一个进程调 wakeup(chan) 把睡在这个频道上的进程改回 RUNNABLE，它就能被重新调度。这对应教材信号量 P/V 操作里的"睡眠等待"。

⚠ 关键陷阱:sleep 为什么必须"带着锁"进去?

设想没有锁保护:进程 A 检查到"资源还没好"，正准备睡。就在这个空档，进程 B 把资源备好了、还喊了一声 wakeup——可 A 这会儿还没真正睡下，这一声叫醒落空了。等 A 反应过来真睡下去，再没人来叫它，它就永远睡死了。这叫 lost wakeup（丢失唤醒）。
解法:A 必须握着保护资源的那把锁去 sleep，sleep 内部会在确实睡好之后才原子地放掉这把锁——这样 B 想喊 wakeup 就得先拿到同一把锁，必然等到 A 睡稳了才喊得出，那一声就不会丢。

6复述检查（看懂了就能答）

Q1. 一句 count++ 在并发下为什么会"丢更新"? 它到底分几步、错在哪一步之间?
count++ 是读→改→写三步。错在:A 读完旧值还没写回时被切走，B 也读到同一个旧值，两人各 +1 写回，两次加一只生效一次。结果取决于执行顺序，就是竞态条件。把这三步用锁圈成临界区就安全了。
Q2. 锁保护 count，可 locked 自己也是共享变量，凭什么不会被两个进程同时抢到?
靠原子操作__sync_lock_test_and_set:把 locked"读出旧值 + 写成 1"合成一条不可打断的指令。返回旧值 0 才算抢到。因为读写一气呵成，不可能两个进程都拿到旧值 0。原子性是锁的地基。
Q3. 自旋锁 acquire 进来第一件事就是关中断，为什么? 不关会怎样?
防死锁。若持锁期间来中断、中断处理又要拿同一把锁，它会 spin 等我放锁，可我被它打断了没法放——自锁死。所以 push_off() 关中断、pop_off() 放锁后再开回来。
