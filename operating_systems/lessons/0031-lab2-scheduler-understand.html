CPU 是怎么从一个进程"切"到另一个进程的?

这一页不教你上台讲,只让你自己真正看懂一件事:一颗 CPU 同一时刻只能跑一个进程,要换人,就得把当前进程的"现场"存起来、把下一个进程的"现场"装回去。看懂了,再去看讲解版(第 32 课)。

对应教材 Ch3 处理机调度 / 进程切换 对应 xv6 scheduler() + swtch.S 核心:上下文的保存与恢复

1先建立一个画面

🧸 ELI5

把 CPU 想成一条接力跑道,同一时刻只有一个人能拿着接力棒在跑。要换人,不能直接把棒一扔——你得先记下现在跑到哪了(第几圈、脚下姿势),把这张"现场快照"塞进口袋,下一个选手再从他自己的口袋掏出他上次的快照、原样恢复,接着跑。

这里有个关键角色:跑道边上站着一位裁判(scheduler 调度器)。选手累了不会直接把棒递给下一个,而是先把棒交回裁判,由裁判看名单、挑出下一个"准备好了(就绪态)"的选手,再把棒递过去。每条跑道(每个 CPU)都有自己专属的一位裁判。

2核心矛盾:一颗 CPU,多个进程都想跑

CPU 里只有一套寄存器(放着当前在算什么、栈在哪)。多个进程要轮流用这套寄存器,就产生了矛盾:

如果换人时……会发生什么结果
不存旧进程的现场旧进程的寄存器/栈被新进程覆盖旧进程再也回不到原地
→ 算到一半的全丢
存了旧进程的现场旧进程的寄存器/栈被原样收进它的 context下次轮到它
→ 装回去,接着算
💡 解决方案:保存/恢复"上下文(context)"

所谓进程切换,本质就是两步:① 把当前进程在 CPU 里的一组关键寄存器存进它自己的 context;② 把下一个要跑的进程的 context 装回 CPU 寄存器。装回去的一刻,栈指针 sp 和返回地址 ra 都变了,CPU 一执行 ret,就"穿越"到了另一个进程上次离开的地方。这就是教材说的"保存现场、恢复现场"。

🔑 你应该看出来的事

① 切换 = 换栈 + 换寄存器,不是换地址。swtch 只动 ra/sp/s0–s11 这一小撮寄存器,谁的内核栈、回到哪行代码,全藏在这里面。
② A 不会直接切到 B。A 先 swtch 回 scheduler(存 A、装 scheduler),scheduler 挑中 B,再 swtch 到 B(存 scheduler、装 B)。一次"换人"其实是两次 swtch。
③ 每个 CPU 有一个独立的 scheduler 线程(它有自己的 context、自己的栈)。所以多核能同时各跑各的,互不踩。

4为什么 context 只存 ra/sp + s0–s11?

CPU 寄存器有几十个,可 xv6 的 struct context 只存了 14 个。不是偷懒,是 C 语言的"调用约定"替我们省了:

寄存器类别例子要不要 swtch 存?
被调用者保存
(callee-saved)		s0–s11、sp、ra要存
函数返回后调用方还指望它们没变
调用者保存
(caller-saved)		a0–a7、t0–t6不用
调 swtch 之前编译器已自动压栈了
⚠ 别和"进程地址空间切换"搞混

swtch 不切换页表/地址空间!它只换那 14 个寄存器(栈 + 返回点)。xv6 里换不换用户页表是另一段代码的事。考试常考的辨析:进程切换(换 CPU 现场) ≠ 地址空间切换(换页表)。这一节只讲前者——也就是"上下文切换"最核心的那一步。

5复述检查(看懂了就能答)

Q1. 进程 A 想让出 CPU 给进程 B,中间为什么必须经过 scheduler? 一共 swtch 几次?
因为 xv6 规定进程之间不直接切换,都先切回本 CPU 的 scheduler 线程,由它统一挑下一个 RUNNABLE 进程。所以一次"换人"是两次 swtch:A→scheduler,scheduler→B。这样调度逻辑只写一处、不会乱。
Q2. swtch 到底切换了什么? 切的是地址空间(页表)吗?
切的是栈指针 sp + 返回地址 ra + 被调用者保存寄存器 s0–s11,一共 14 个。不是切地址空间。换了 sp 就等于换了内核栈,换了 ra 就等于换了"ret 之后回到哪行",于是 CPU 一返回就身处另一个进程的执行流里。
Q3. RUNNABLE(就绪态)和 RUNNING(运行态)差在哪? 进程 A 让出 CPU 后变成哪个?
RUNNING=此刻正占着某个 CPU 在跑;RUNNABLE=万事俱备、只差一个 CPU,在队列里等调度。A 让出 CPU 时把自己改成 RUNNABLE(还想跑,只是先排队),scheduler 挑中谁就把谁改成 RUNNING。这正是教材"运行态↔就绪态"的转换。
