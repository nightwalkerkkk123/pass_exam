进程同步 · 信号量、P/V 操作、生产者-消费者

目标：20 分钟装下"信号量是啥"+"P/V 怎么写"+"生产者-消费者问题怎么解"——能写出 PV 代码就过线

1. 一句话 + 图书馆讨论间类比

多个进程/线程要抢共享资源——用 信号量 计数，用 P/V 操作 排队。 信号量 = "还剩几把钥匙"；P = 借钥匙；V = 还钥匙。
📚 图书馆讨论间（= 共享资源）
🗝️🗝️🗝️ 5 间讨论间，门上挂着 5 把钥匙
📋 借钥匙本 写"还剩几把钥匙" + "谁在等"
🧑‍🎓 借钥匙 看了本上的数 > 0，借一把进去
🧑‍🎓 没钥匙 在门口排队等
🧑‍🎓 还钥匙 把钥匙挂回门口 + 唤醒队首
💻 进程 / 线程（= 借钥匙的学生）
资源数 = 讨论间数（= 信号量初值 S）
信号量 S = 借钥匙本上"还剩几把"
P(S) = 借钥匙：S--，不够就阻塞
V(S) = 还钥匙：S++，唤醒队首
原子操作 = 借/还一气呵成，不可打断

2. 信号量 S 的 3 种状态（必背）

S 的值含义图书馆翻译
S > 0 还有 S 个资源可用（没人等） 还剩 3 把钥匙，没人排队
S = 0 刚好用完，无人等待 钥匙刚好借完，没人排队
S < 0 |S| 个人在等（资源用光） 钥匙借完，且有 3 人在排队
易记 1 句话：S 正数 = 还剩几个资源；S 负数 = 有几个人在等。

3. P / V 操作（伪代码）

/* 记录型信号量：含 value + 等待队列 */
typedef struct {
    int value;           // 资源计数
    struct PCB *list;    // 等待队列
} semaphore;

P(S)  // 借钥匙 = 申请资源
    S.value--;
    if (S.value < 0)
        block(S.list);  // 排队阻塞

V(S)  // 还钥匙 = 释放资源
    S.value++;
    if (S.value <= 0)
        wakeup(S.list); // 唤醒队首
P 和 V 的方向：
• P = 申请资源（Proberen 试）：我想用，扣 1 个资源
• V = 释放资源（Verhogen 加）：我用完了，还 1 个资源

4. 互斥信号量（最常用模式）

保护"临界区"——一段代码同一时刻只能有 1 个进程/线程跑。

semaphore mutex = 1;     // 初值 1 = 1 把钥匙

process() {
    while(true) {
        P(mutex);          // 借钥匙进门
        // 临界区代码（只能 1 人同时跑）
        critical_section();
        V(mutex);          // 还钥匙出门
        // 剩下的非临界区代码
    }
}
P 和 V 必须成对出现！少 P = 不互斥（会乱）；少 V = 永远借不出（死锁）。

5. 生产者-消费者问题（必考 10 分大题）

问题：

  • 多个生产者生产产品，放进一个有 N 格子的缓冲池
  • 多个消费者从缓冲池取走产品消费
  • 同一时刻只能 1 人操作缓冲池（互斥）
  • 缓冲池满时生产者阻塞；空时消费者阻塞

5.1 3 个信号量（必背）

信号量初值作用
mutex1互斥访问缓冲池（一次 1 人）
emptyN空格子数（还能放几份）
full0满格子数（还能取几份）

5.2 标准答案（伪代码）

semaphore mutex = 1, empty = N, full = 0;

producer() {                          consumer() {
    while(true) {                       while(true) {
        item x = produce();                     P(full);             // ① 等"有货"
        P(empty);           // ① 等"有空位"      P(mutex);            // ② 进门
        P(mutex);           // ② 进门            item x = buffer[out];
        buffer[in] = x;                          out = (out+1) % N;
        in = (in+1) % N;                        V(mutex);            // ③ 出门
        V(mutex);           // ③ 出门            V(empty);            // ④ 通知"多一个空位"
        V(full);            // ④ 通知"多一份货"   consume(x);
    }                                       }
}                                           }
必背陷阱：P 顺序不能颠倒！
• 生产者：先 P(empty)，再 P(mutex)
• 消费者：先 P(full)，再 P(mutex)
如果先 P(mutex)，会导致 2 个进程都拿到 mutex 后再等 empty/full，互相等 → 死锁。
口诀：先资源（empty/full），后互斥（mutex）。

6. 必背 4 句话（考试原话）

① 信号量 S = 资源计数器：S>0 还剩几个资源，S<0 绝对值是等待进程数。
② P(S) = 申请资源（S--，不够就阻塞）；V(S) = 释放资源（S++，唤醒队首）。
③ 互斥：mutex 初值 1，P/V 夹住临界区；同步：资源类信号量（S 初值=资源数）。
④ 生产者-消费者：先 P(资源) 后 P(mutex)，顺序反了会死锁。

7. 自检（先做再翻答案）

Q1（口述） 信号量 S=3、S=0、S=-2 各自代表什么意思？
Q2（口述） P 和 V 操作分别做什么？为什么说"必须成对出现"？
Q3（默写） 写出"互斥访问临界区"的标准 PV 代码框架（互斥信号量 mutex 初值 1）。
Q4（选择） 生产者-消费者问题中，P 操作的正确顺序是？
A. 先 P(mutex)，再 P(empty)
B. 先 P(empty)，再 P(mutex)
C. P(mutex) 和 P(empty) 顺序无关
D. 只 P(mutex) 不 P(empty)
Q5（改错） 下面代码哪里错了？
semaphore mutex = 1, empty = N;
producer() {
    P(mutex);
    P(empty);
    buffer[in] = x;
    V(mutex);
    V(empty);
}
答案对照
A1：
• S=3：还剩 3 个资源可用，没人等
• S=0：资源用完，没人等
• S=-2：资源用完 + 2 个人在等
A2：P = 申请资源（S--，不够就阻塞）；V = 释放资源（S++，有等待者就唤醒）。少 P = 不互斥；少 V = 永远借不出（死锁）。
A3：
semaphore mutex = 1;
P(mutex);
// 临界区
V(mutex);
A4：B。先 P(资源) 再 P(mutex)，反了会死锁。
A5：错在 P 顺序——先 P(mutex) 再 P(empty)，应该是先 P(empty) 再 P(mutex)。题目里 V(mutex) 和 V(empty) 顺序无所谓（都在临界区外），但 P 顺序不能错。

8. 完整流程速记（手算/手写代码时按这 4 步走）

已知：进程 P1 / P2，共享资源 X，初值 N

① 找资源 → 设信号量 S，初值 = 资源数 N
② 申请顺序：先资源信号量 S（P），再互斥信号量 mutex（P）
③ 释放顺序：先 mutex（V），再 S（V）—— V 顺序没 P 那么严格，但保持一致
④ 验证：P/V 配对（每个 P 配一个 V），P 顺序对
