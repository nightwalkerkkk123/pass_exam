进程状态转换:五态 / 七态 + PCB + 四个控制原语

这一课只回答一件事:一个进程在它的一生里,会在哪几种"状态"之间跳来跳去,什么事件能让它跳、哪两条路根本走不通——顺手讲清 PCB 和"原语"为什么必须是原子的。

0·全局线 1·状态机(交互) 2·ELI5 类比 3·对照表 4·是什么(定义) 5·为什么(追问) 6·自测
一句话先记住: 就绪 = 只差 CPU(给它 CPU 立刻能跑);阻塞 = 在等别的事件(给它 CPU 也没用)。 全部状态转换的对错,都由这一句判定。

🗺 0. 全局故事线:这些点怎么串起来

别把下面这些当成孤立知识点——它们是"OS 怎么管好一堆并发进程"这一个问题一路逼出来的。

进程 ≠ 程序程序是静态代码,进程是"程序的一次执行",有生命期
→
要管它就得记它OS 用一张表 PCB 记录每个进程,PCB 是进程存在的唯一标志
→
它会变状态就绪/运行/阻塞…状态字段就存在 PCB 里
→
改状态要原子创建/撤销/block/wakeup 用原语=不可中断,否则 PCB 会被改乱

🧸 2. 用课堂类比讲清五个状态

🧸 一句话(照总框架的口径): 把 CPU 想成"老师点名回答问题",一个学生(进程)就在下面几种状态里转:
  • 新建 New:刚转学进来,学籍还没办好(PCB 还在初始化),还没进班排队。
  • 就绪 Ready:"我准备好了,就等老师点我"——万事俱备,只差 CPU。
  • 运行 Running:"老师正在让我回答问题"——真正占着 CPU 在执行。
  • 阻塞 Blocked:"我在等课代表把资料发给我,没资料我答不了"——在等某个事件(等 I/O、等信号量、等数据),这时点我也没用。
  • 终止 Terminated:答完下课离场,学籍注销(PCB 回收)。

反例:如果没有"阻塞"这个状态会怎样?那么一个在等磁盘数据的进程只能一直占着 CPU 空转("忙等"),CPU 被白白浪费——正是为了把"在等事件"的进程踢下 CPU,才需要阻塞态。

⚖️ 3. 对照表:一辈子都别再混的两组概念

🧸 一句话: 考试最爱在"就绪 vs 阻塞"和"五态 vs 七态"上设坑,先把这两张表刻进脑子。

⚖️ 就绪 Ready vs 阻塞 Blocked(核心判据)

比什么就绪 Ready阻塞 Blocked
它在等什么只等 CPU(其它资源都拿到了)在等某个事件(I/O完成、数据到达、信号量…)
现在给它 CPU 会怎样立刻就能跑照样跑不了——事件没来,拿到 CPU 也没用
排在哪个队列就绪队列 RL(通常一条)阻塞队列 WL(按等待事件常分成多条)
谁把它弄成这样被 wakeup/调度让出/创建接纳运行中自己调用 block() 主动进入
结论: 判断一个进程该是就绪还是阻塞,就问一句——"现在把 CPU 给它,它跑得动吗?"跑得动=就绪,跑不动=阻塞。

⚖️ 五态 vs 七态(七态=五态 + 两个"挂起")

模型状态解决什么问题
五态新建 / 就绪 / 运行 / 阻塞 / 终止描述一个进程在内存里的完整生命期
七态再加 静止就绪(就绪挂起) + 静止阻塞(阻塞挂起)内存不够时把进程换到外存暂存;"活动"在内存、"静止"在外存

口径(照 PPT 2.1.4):挂起前叫"活动就绪/活动阻塞",挂起后叫"静止就绪/静止阻塞"。运行态或活动就绪被挂起→静止就绪;活动阻塞被挂起→静止阻塞。

📖 4. 是什么:三个精确定义(对齐教材原文)

定义①·三种基本状态: 运行态(Running)=进程已获得 CPU 正在运行;就绪态(Ready)=已获得除 CPU 外的一切所需资源,一旦得到 CPU 即可运行;阻塞态(Blocked)=正在等待某事件发生(如请求 I/O 而等待 I/O 完成)而暂停,此时即使把 CPU 分给它也无法运行。
定义②·PCB(进程控制块): PCB 是 OS 用来描述进程的基本情况和运行状态、进而控制和管理进程的核心数据结构,是进程存在的唯一标志。它让一个本来不能独立运行的程序,成为能与其它进程并发执行的进程——OS 就是靠 PCB 来控制和管理进程的。

PCB 里存了什么(四大类字段)

类别典型字段
进程标识符内部标识符(唯一序号)、外部标识符、父/子进程标识、用户标识
处理机状态通用寄存器、指令计数器 PC、程序状态字 PSW、用户栈指针(切换时保存/恢复现场)
进程调度信息进程状态、优先级、已等待/已执行时间、事件(阻塞原因)
进程控制信息程序和数据的地址、同步与通信机制(信号量/消息队列指针)、资源清单、链接指针

注意:"进程状态"这个字段就存在 PCB 里——所以状态转换本质上就是改 PCB 里的 state 字段 + 把 PCB 挪到对应队列。

定义③·四个进程控制原语: 创建原语 create(申请空白 PCB→分配资源→初始化 PCB→插入就绪队列)、撤销原语 destroy(找到 PCB→若在运行则停止→撤销其子孙→回收资源→PCB 移出队列)、阻塞原语 block()(运行进程主动调用:停止执行→state 改"阻塞"→插入阻塞队列→转调度)、唤醒原语 wakeup()(事件发生时由别的进程调用:从阻塞队列移出→state 改"就绪"→插入就绪队列)。

❓ 5. 为什么:把动机挖到底

第 1 层:为什么"阻塞→运行"和"就绪→阻塞"根本不可能?

这是本课最高频的送分/送命题。用那句判据一击致命:

  • 阻塞 → 运行 ✘:阻塞进程在等事件,就算把 CPU 硬塞给它也没法用。它必须先等到事件("I/O 完成")→被 wakeup 变就绪→再"被调度"才能运行。中间必须经过就绪做跳板。
  • 就绪 → 阻塞 ✘:只有正在运行的进程才会发起 I/O 请求(要有 CPU 在执行才谈得上"执行到一条 I/O 指令")。就绪进程手里没有 CPU、根本没在跑,谈不上"请求 I/O",自然进不了阻塞。它必须先"被调度"变运行,运行中请求 I/O 才掉进阻塞。
⚠️ 红线两条(点交互件里的这两步会报"此路不通"):
1) 阻塞 → 运行 不存在,要经就绪中转;
2) 就绪 → 阻塞 不存在,要经运行中转。
记忆:进出 CPU 的门只有"就绪"和"运行"这一对;阻塞永远只能和"就绪"打交道,不直接碰"运行"。

第 2 层:为什么进程控制非得用"原语(原子操作)"?

"原语"= primitive,执行期间不可被中断、要么全做完要么全不做。为什么必须这样?因为 block/wakeup/create/destroy 都要同时改好几处:改 PCB 的 state 字段 + 把 PCB 从这条队列摘下 + 挂到那条队列。

假如中途被打断:比如 wakeup 刚把 state 改成"就绪",还没来得及把 PCB 插进就绪队列,就被切走——此时这个进程状态说它就绪,却不在任何队列里,调度器永远选不到它,它就"人间蒸发"了。用原语保证这一串操作原子完成,PCB 和队列才始终一致。

第 3 层:为什么七态要引入"挂起"?边界在哪?

动机(照 PPT):内存紧张、或管理员要暂停进程排故障、或用户想看中间结果时,把进程换到外存腾内存。于是活动态(在内存)分裂出静止态(在外存)。

边界细节要记牢:静止阻塞的进程,即使人在外存,它等的事件也可能完成——这时它转成静止就绪(注意仍在外存!要再"激活"换回内存才变活动就绪)。这就是交互件里 静止阻塞 --I/O完成--> 静止就绪 那条边。

✅ 6. 自测:先自己想,再点开答案

选择题 1

下列哪一种状态转换是不可能直接发生的?

A. 运行 → 就绪
B. 阻塞 → 运行
C. 运行 → 阻塞
D. 就绪 → 运行

点我看答案

答案:B。 阻塞进程在等事件,给它 CPU 也跑不了,必须先经"就绪"中转(阻塞→就绪→运行)。 A 时间片到,合法;C 运行中请求 I/O,合法;D 被调度选中,合法。

选择题 2

"就绪"和"阻塞"最根本的区别是?

A. 就绪在内存、阻塞在外存
B. 就绪在等 I/O、阻塞在等 CPU
C. 就绪只差 CPU(给 CPU 立刻能跑),阻塞在等其它事件(给 CPU 也没用)
D. 两者没有本质区别,只是叫法不同

点我看答案

答案:C。 判据就是"给它 CPU 跑得动吗"。 A 混淆的是"活动/静止(挂起)";B 把两者说反了(就绪等 CPU、阻塞等事件);D 显然错。

选择题 3

为什么进程控制(创建/撤销/block/wakeup)要用"原语"实现?

A. 因为原语运行速度更快
B. 因为原语是原子操作、执行中不可被中断,避免 PCB 和队列被并发改乱、出现状态不一致
C. 因为原语只能由用户程序调用
D. 因为原语执行时不占用 CPU

点我看答案

答案:B。 原语=原子、不可分割,保证"改 state + 挪队列"这一串要么全做完要么全不做。 A 快慢不是重点;C 恰恰相反,原语多由 OS 内核执行;D 执行任何指令都要占 CPU。

选择题 4

关于 PCB 的地位,下列正确的是?

A. PCB 是进程存在的唯一标志,OS 靠它来控制和管理进程
B. PCB 里保存的是进程的全部程序代码本身
C. 进程终止后,它的 PCB 会被永久保留
D. 一个进程可以没有 PCB 也照样被调度运行

点我看答案

答案:A。 PCB 是进程存在的唯一标志。 B 错:PCB 存的是程序/数据的地址和管理信息,不是代码本身;C 错:撤销原语会回收 PCB;D 错:没 PCB 就不是"进程",无从调度。

选择题 5(七态)

一个处于静止阻塞(阻塞挂起)的进程,它等待的 I/O 完成后,会转到哪个状态?

A. 活动就绪
B. 运行
C. 静止就绪(仍在外存,要"激活"才回内存)
D. 活动阻塞

点我看答案

答案:C。 人在外存也能等到事件完成:静止阻塞→静止就绪,但仍在外存,须再"激活"才变活动就绪。 A/D 要先"激活"换回内存才谈得上;B 任何状态都不能直接跳到运行,必须经(活动)就绪+被调度。

口述题(别打字,张嘴说)——通过条件

  1. 用一句话说出"就绪 vs 阻塞"的判据。
  2. 不看屏幕,画/说出五态转换图,并说清每条边由什么事件触发。
  3. 说出为什么"阻塞→运行"和"就绪→阻塞"不可能,各要经哪个态中转。
  4. 说清 PCB 是什么、为什么是"进程存在的唯一标志",以及原语为什么必须原子。

通过条件:四条都能顺口说出、且能在交互件上一步不错地走完一条"新建→就绪→运行→阻塞→就绪→运行→终止"的完整路径 → 你是真懂了。哪条卡壳,回到对应小节再点一遍交互件。
