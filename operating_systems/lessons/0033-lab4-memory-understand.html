实验④ 内存分配与回收 · 理解版

一个虚拟地址，是怎么"逐级查表"变成物理地址的?

这一页不教你上台讲，只让你自己真正看懂两件事:① 一个虚拟地址要走三级页表、逐级查表，才落到物理页;② "分配/回收一页内存"就是从一条空闲链表上摘一个、还回去。看懂了，再去看讲解版（第 34 课）。

对应教材 Ch4 存储器管理 · 分页 · 地址变换 对应 xv6 kernel/vm.c 的 walk() + kernel/kalloc.c 的 kalloc/kfree

1先建立一个画面

🧸 ELI5：省 → 市 → 区，三级查地址

你要寄一个快递，地址写的是"2 省 1 市 5 区 偏移 23 号"。快递员不是一下子就知道门牌在哪，而是一级一级查表:

先翻全国省级名册，按"2"找到那一省的市级名册在哪 → 再翻这本市级名册，按"1"找到区级名册在哪 → 再翻区级名册，按"5"找到这个小区的大门 → 最后进了门，按"偏移 23 号"找到具体那一户。

CPU 把虚拟地址翻成物理地址，干的就是一模一样的事:虚拟地址被切成"省 / 市 / 区 / 户号"四段，前三段各查一级页表，最后一段是页内偏移，直接落到页里的某个字节。

2核心矛盾:为什么不用一张大表，偏要分三级?

地址变换本来一张表就够:虚拟页号当下标，查出物理页号。问题是这张表会大到离谱:

做法表有多大问题
单级页表
(一张大表全列出)		512×512×512 个表项
≈ 1.3 亿项 · 每进程		大部分虚拟地址根本没用到，
表项却必须先占着 → 极度浪费
三级页表
(按需分层建)		只建用到的那几张
小表（每张 512 项）		没用到的整片区域，
连下一级表都不用建 → 省
💡 解决方案:多级页表（xv6 是三级，叫 Sv39）

把一张巨表拆成树状的多级。上一级的表项不直接存数据，而是存"下一级表在物理内存哪一页"。哪片虚拟地址没被用到，对应的下级表就干脆不建——表项缺省为空（有效位是 0）。于是真正占内存的，只有你实际用到的那几张小表。这就是教材"多级页表节省页表空间"那段话的本质。

3动手:选一个虚拟地址，看它逐级点亮三级页表

🔑 你应该看出来的事

每一级页表只用对应那 9 位索引去查一行:L2 用 L2 索引、L1 用 L1 索引、L0 用 L0 索引。查出来的不是数据，而是"下一级表在物理内存的哪一页"（直到最后一级才是真正的物理页号）。

查每一行时都要看有效位 PTE_V:如果是 1，说明这一项有效、可以顺着走下去;如果是 0，说明这片地址还没建过表——walk() 里要么报错返回，要么（带 alloc 时）现场 kalloc() 一页新表挂上去。这就是为什么源码里反复出现 if(*pte & PTE_V)。

三级查完，把最后查到的物理页号和虚拟地址的页内偏移一拼，就是最终物理地址。偏移那 12 位从头到尾原样不动——它不参与查表。

4另一半:"分配/回收一页"就是链表摘挂

翻译要用到的那些表页、还有进程要用的内存页，都是一页一页（4096 字节）发出去的。谁来管"哪些页还空着"? xv6 的答案简单到不可思议:把所有空闲页串成一条单链表，叫 freelist。

kalloc() 分配一页 = 从链表头摘下第一个，返回给你:

页A→ 页B→ 页C→ null

kfree(p) 回收一页 = 把这一页重新挂回链表头:

页B→ 页C→ null
空闲页就是一条链。摘头、挂头，都不用遍历——所以分配和回收都是 O(1)。
💡 一个巧思:链表的"指针"藏在空闲页自己身上

你可能会问:这条链表的 next 指针存哪? 答案是——就存在空闲页的开头。反正这页现在空着没人用，xv6 就把它强转成 struct run { struct run *next; }，借这一页的前 8 个字节存"下一个空闲页的地址"。所以管理空闲页几乎不额外花内存。这正是教材"空闲分区/空闲块链表法"在真实代码里的样子。

⚠ 别把"分页"和"页表查找"两件事搅在一起

分页解决的是"内存按固定大小切块、按页分配回收"（第 4 节这条链表）;地址变换 / 多级页表解决的是"程序里的虚拟地址怎么映射到物理页"（第 3 节那棵树）。考试里它们常一起出现，但回答时要分清:链表管的是物理页够不够用，页表管的是这个虚拟地址该落到哪一页。

5复述检查（看懂了就能答）

Q1. 一个虚拟地址被切成几段? 每段干什么用? 哪一段不参与查表?
切成 9+9+9+12 四段。前三段（各 9 位）分别是 L2、L1、L0 的索引，用来逐级在三级页表里查行;最后 12 位是页内偏移，不参与查表，最后直接拼到物理页号后面定位到具体字节。
Q2. walk() 那个 for(level=2; level>0; level--) 循环在干嘛? 为什么是从 2 到 1、只循环两次?
循环负责从上一级表走到下一级表:用 PX(level,va) 取出该级的 9 位索引、查出下一级表的物理页，再换页继续。循环只走 level=2 和 level=1 两次（走完 L2、L1），最后一级 L0 在循环外用 &pagetable[PX(0,va)] 直接返回那个 PTE 的地址——因为最后要的是 PTE 本身，不再往下跳了。
Q3. 为什么 kalloc/kfree 用链表就能做到 O(1)? "省内存"体现在哪?
分配只是摘链表头、回收只是挂回链表头，都不用遍历，所以是 O(1)。省内存体现在:链表的 next 指针借空闲页自己的开头 8 字节存（struct run），不需要另开一张大数组去记每页空闲与否。
