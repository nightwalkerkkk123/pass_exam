三级调度 + 多级反馈队列 MFQ:一个"不用估时间"的万能调度器

这一课只回答一件事:MFQ 凭什么能同时让短作业跑得快、又让长作业不饿死? 配合 速查地图、调度四件套 使用。

0·全局线 1·先看它跑 2·ELI5类比 3·对照表 4·是什么 5·为什么 6·自测
一句话先记住: 三级调度管"谁能进内存、谁被换出去、谁上 CPU"三个不同频率的问题;MFQ 则是低级(进程)调度里的明星——多条优先级递减的队列、时间片逐级翻倍、新进程进最高队、一个时间片没跑完就降一级、随时可被更高队抢占,于是短作业在高队秒完、长作业沉到底队按 RR 慢慢跑,谁也不饿死。它就是 SJF + RR 的工程折中。

🗺 0. 全局故事线:这些点怎么串起来

别把三级调度、SJF、RR、MFQ 当孤立知识点——它们是"怎么公平又高效地分 CPU"这一个问题一路逼出来的。

三级调度先分层:作业调度(进不进内存)/内存调度(换不换出)/进程调度(谁上 CPU)
→
SJF 短作业优先响应最快,但要预估运行时间,还饿死长作业
→
RR 时间片轮转公平不饿死,但对短作业不够快,长短一视同仁
→
MFQ 多级反馈队列两者折中:不用估时间,又快又不饿死(老师重点★)

🧸 2. ELI5:医院分诊台 + 一个反例

🧸 想象一个分诊台(MFQ): 病人(进程)来了先进1 号快速通道 Q1(优先级最高,但每人只给 2 分钟)。2 分钟看完就走人(短作业秒完);没看完的转到 2 号通道 Q2(每人给 4 分钟);还没完再转 3 号通道 Q3(每人 8 分钟,反复排队慢慢看)。只要 1 号还有人,医生绝不去叫 2 号、3 号。这样小病人来去如风、大病人也总有 3 号通道兜底,不会被永远晾着。
⚠️ 反例(如果没有 MFQ,只用 SJF 会怎样): 分诊台改成"永远先看预计最快的那个"。结果只要不断有小病人进来,一个需要长时间的大病人就永远排在最后,饿死(starvation);而且你还得预先精确估计每人要看多久——现实里根本估不准。MFQ 用"跑一段没完就降级"替代了"预估时间",这正是它的精髓。(SJF 缺点见 3_处理机调度/ppt文本.md 幻灯片 对 SF 的"对长作业不利 / 估计值"论述)

⚖️ 3. 对照表:MFQ 到底是 RR 和 SJF 的什么折中

🧸 一句话: MFQ 不是单一队列的 RR!它是"很多条 RR 队列摞起来 + 会降级",从而模拟出 SJF 的效果却不用估时间。

⚖️ 三算法并排(照教材口径)

比什么SJF 短作业优先RR 时间片轮转MFQ 多级反馈队列 ★
队列数1 条(按预估时间排)1 条(FCFS 排队轮转)多条,优先级递减
要预估运行时间吗要(致命弱点)不要不要(用降级代替预估)
短作业响应最快一般很快(在 Q1 就跑完)
长作业会饿死吗会不会不会(沉到底队 RR 兜底)
时间片无(跑到完 / 被抢)固定一个逐队翻倍 S1<S2<…<Sn
抢占可抢占 / 非抢占抢占(时间片到)抢占(更高队新进程到达)
适用批处理作业分时系统通用,兼顾实时 / 分时 / 批处理
结论: MFQ = 「多条 RR 队列」+「跑不完就降级」。高队时间片小 → 短作业几下就完,行为像 SJF;长作业逐级下沉到底队 → 像 RR 一样轮转不饿死。所以它是 SJF 与 RR 的工程折中。(来源:textbook/算法速查表.md 第 11 行:"兼顾长短作业,交互性好,无需估计运行时间")

📖 4. 是什么:先三级调度,再 MFQ 精确定义

🧸 一句话: "调度"分三层楼,楼层越低跑得越勤;MFQ 只住在最低那层(进程调度)里。

4.1 三级调度:三个不同频率的问题

层次(别名)调度对象管什么频率哪种系统有
高级调度
作业调度 / 长程调度		作业把外存后备队列里的哪些作业调入内存、建进程、给资源(决定多道程序度)低,几分钟一次只在批处理系统
中级调度
内存调度 / 对换		进程把暂不能运行的进程换到外存(挂起),有条件再换回内存(就绪),提高内存利用率中,几十秒一次引入对换的系统(常见于分时)
低级调度
进程调度 / 短程调度		进程 / 内核级线程决定就绪队列里哪个进程上 CPU(MFQ 就在这一层)高,几十毫秒一次所有 OS 都必备
关键定义(照教材): 中级调度(内存调度)= 实现进程的"挂起"与"激活",把进程在内存与外存之间对换(swap),通常用于分时系统。(来源:3_处理机调度/ppt文本.md 中级调度幻灯片;textbook/ch03 §3.1 处理机调度的层次)

4.2 MFQ 的四条精确规则

多级反馈队列(MFQ, Multilevel Feedback Queue)——照 textbook/ch03 §3.3 与 ppt 幻灯片 155/160 原文:
  1. 设置多个就绪队列,从高到低赋予不同优先级(Q1 最高),队内用 RR。
  2. 各队时间片不同,优先级越高时间片越小,一般逐级翻倍(S1<S2<…<Sn)。
  3. 新进程进第一队(最高队)队尾;在本队一个时间片内完成就撤离,否则降到下一队队尾;降到最后一队后按 RR 反复轮转。
  4. 只有更高优先级队列全空,才调度低一级队列;若正跑低队进程时更高队来了新进程,则把当前进程放回本队队尾,转去跑高队进程(抢占)。

❓ 5. 为什么:一层层追问 MFQ 的设计动机

第 1 层:为什么要"多条队列 + 降级",不用一条队列?

因为要同时满足两个矛盾的目标:短作业要响应快(像 SJF),长作业要不饿死(像 RR)。一条队列做不到两全:RR 对短作业不够快,SJF 会饿死长作业还得预估时间。多条队列让"跑得少的天然待在高队(快)、跑得多的自动沉到低队(不饿死)"——用观察到的行为区分长短,而不是预估。

第 2 层:为什么时间片要"逐级翻倍",而不是都一样?

高队时间片小,是为了让真·短作业几下就在高队跑完、快速离场,同时高频切换保证交互响应;越往下的进程已经证明自己是"长家伙",给它大时间片可以减少切换开销(一次多跑一点),又因为它优先级低,不会挤占短作业。这就是"短的图快、长的图省"的分层折中。

第 3 层:边界与陷阱——它和 RR/SJF 的界线在哪?

  • MFQ ≠ 单队列 RR。 有多条队列 + 降级机制,别在卷子上把它画成一条队列。
  • 降级 vs 抢占回队要分清: "时间片用完没跑完"→降一级(去下一队队尾);"被更高队新进程抢占"→退回本队队尾(不降级)。
  • 底队(最后一队)不再降级,只在本队 RR;所以长作业最终稳定在底队轮转,不会饿死。
  • 无需预估运行时间——这是它相对 SJF/HRRN 的最大工程优势。
  • 代价: 实现较复杂、调度开销较大。(来源:3_处理机调度/ppt文本.md 幻灯片 189 "缺点:调度开销较大")
📐 手算 MFQ 甘特图(4 步):
  1. 建队:新进程按到达时间进 Q1 队尾;每一刻先"接纳"到达者。
  2. 选谁跑:永远挑最高的非空队列的队首。
  3. 跑多久:跑 min(本队时间片, 剩余时间, 到下一个更高队到达前);跑完或时间片到就结算。
  4. 放哪去:跑完→离场;时间片到没完→降一级队尾(底队则回本队尾);被高队抢占→回本队尾。回到第 2 步。

✅ 6. 自测:先自己想,再点开答案

选择题 1

MFQ 中,一个新到达的进程首先被放到哪里?

A. 最低优先级队列的末尾
B. 第一队列(最高优先级)的末尾
C. 当前正在运行进程所在的那条队列
D. 由系统预估其运行时间后决定的队列

点我看答案

答案:B。 规则明写"新进程放到最高优先级就绪队列尾部"。A 错——那是长作业反复降级后的归宿,不是起点;C 错——新进程不看谁在跑;D 错——MFQ 的卖点正是不需要预估运行时间,用降级代替预估。

选择题 2

关于 MFQ 各队列的时间片,下列正确的是?

A. 所有队列时间片完全相同
B. 优先级越高的队列时间片越大
C. 优先级越低的队列时间片越大(一般逐级翻倍)
D. 时间片随进程的等待时间动态增长

点我看答案

答案:C。 教材:"时间片长度从高优先级到低优先级依次增加(一般加倍)S1<S2<…<Sn"。A 错(那退化成分层 FCFS);B 恰好说反;D 是把 MFQ 和"动态优先级 / HRRN 随等待变化"搞混了——MFQ 的时间片是按队列固定的。

选择题 3

一个长进程(长作业)在 MFQ 中的最终命运是?

A. 逐级下降到最后一个队列,按 RR 轮转执行,不会饿死
B. 永远停留在第一队列反复运行
C. 因始终抢不到 CPU 而饿死(starvation)
D. 被系统识别为长作业后强制终止

点我看答案

答案:A。 长进程一次次用完时间片就一路降级,最后沉到底队按 RR 轮转,总能分到 CPU。B 错——它跑一个时间片没完就会被降走;C 恰恰是 MFQ 要避免的(那是 SJF 的毛病);D 纯属虚构,OS 不会因为作业长就杀它。

选择题 4

三级调度中,运行频率最高、且每种 OS 都必须配置的是?

A. 中级调度(内存调度 / 对换)
B. 低级调度(进程调度)
C. 高级调度(作业调度)
D. 三者频率一样高

点我看答案

答案:B。 低级(进程)调度几十毫秒一次,是最基本、所有 OS 都必备的调度。C 高级调度几分钟一次且只在批处理系统有;A 中级调度几十秒一次,只在有对换的系统;D 错,三层频率差着数量级。

选择题 5

关于 MFQ 与 RR/SJF 的关系,哪句最准确?

A. MFQ 就是单一队列的 RR,只是换了个名字
B. MFQ 必须预先精确估计每个进程的运行时间
C. MFQ 是 SJF 与 RR 的工程折中:多队列 + 降级让短作业先跑完、长作业不饿死,且无需预估运行时间
D. MFQ 只适用于硬实时系统

点我看答案

答案:C。 A 错——MFQ 有多条队列且会降级,不是单队列 RR;B 恰好把 MFQ 最大的优点说反了(它不需要预估);D 错——它是通用调度算法,兼顾实时 / 分时 / 批处理。C 正是本课主线。
