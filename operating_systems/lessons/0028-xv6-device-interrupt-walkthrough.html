0. 先记这句主线(讲偏了就拉回来)

「键盘/串口有数据时,会向 CPU 发一个中断。内核的 devintr() 先判断是哪个设备的中断,是串口就交给驱动 uartintr() 去读字符,再交给 consoleintr() 把字符放进输入缓冲;一整行到齐了就 wakeup 唤醒正在 read 上睡觉的进程。」 这就是课本说的中断驱动 I/O——CPU 不用一直轮询设备,数据来了设备主动「敲门」。

1. 背景:三种 I/O 控制方式,为什么这里是「中断驱动」

课本第 5 章讲了四种 I/O 控制方式,先用一张表对齐老师口径,再引出我们这条代码线属于哪种:

控制方式CPU 怎么知道设备好了缺点 / 特点xv6 对应
程序查询(轮询)CPU 反复读状态寄存器「好了没?好了没?」CPU 空转忙等,浪费uartputc_sync() 里的 while(...)==0; 自旋等待(uart.c:122)
★ 中断驱动设备好了主动发中断,CPU 收到才去处理CPU 可去干别的,效率高 (本课主角)devintr→uartintr→consoleintr
DMA整块数据由 DMA 控制器搬,搬完才中断一次适合磁盘等大块传输virtio_disk(磁盘,本课不展开)

讲这页的金句:「串口/键盘是字符设备,一次来一个字节,用中断驱动最合适——来一个中断一次,CPU 平时该干嘛干嘛。如果用程序查询,CPU 就得一直盯着串口空转。所以 xv6 输入走中断,这正是课本『中断驱动 I/O』的活教材。」(注意:输出有一份程序查询版 uartputc_sync 专给 panic/printf 用,正好和中断版形成对照。)

2. 逐行讲 devintr() ——「这是哪个设备的中断?」(来自 kernel/trap.c)

背景:CPU 收到中断后,硬件跳到内核的陷入处理(usertrap/kerneltrap),它们都调用 devintr() 去辨认中断来源。注释先念:「检查是外部中断还是软件(时钟)中断,并处理它。返回 2=时钟中断,1=其它设备,0=不认识。」

177int
178devintr()
179{
180  uint64 scause = r_scause();        // ① 读「中断原因寄存器」,问硬件:为什么陷进来?
181
182  if((scause & 0x8000000000000000L) &&  // 最高位=1 → 是中断(不是异常/系统调用)
183     (scause & 0xff) == 9){            // 低位=9 → 是「外部中断」(来自外设)
184    // this is a supervisor external interrupt, via PLIC.
185
186    // irq indicates which device interrupted.
187    int irq = plic_claim();          // ② 问中断控制器 PLIC:到底是几号设备?(取中断号)
188
189    if(irq == UART0_IRQ){          // ③ 是串口(UART)的中断号 →
190      uartintr();                  // ★ 派给串口驱动去处理(读字符)
191    } else if(irq == VIRTIO0_IRQ){  // 是磁盘的中断号 →
192      virtio_disk_intr();          // 派给磁盘驱动
193    } else if(irq){                // 认不出的设备号
194      printf("unexpected interrupt irq=%d\n", irq);
195    }
196
200    if(irq)
201      plic_complete(irq);          // ④ 告诉 PLIC:这个设备处理完了,可以再中断了
202
203    return 1;                       // 返回 1:是某个外设的中断
204  } else if(scause == 0x8000000000000001L){ // 否则:是软件(时钟)中断
208    if(cpuid() == 0)
209      clockintr();                  // 处理时钟滴答
214    w_sip(r_sip() & ~2);            // 清中断标志位,应答
216    return 2;                       // 返回 2:时钟中断(上层据此 yield 切换进程)
217  } else {
218    return 0;                       // 都不是:不认识
219  }
220}

把 devintr 拆成「判断来源 + 分发」两件事(这是你讲解的主结构)

代码行一句话讲法对应课本
读中断原因180「scause 是中断原因寄存器,硬件把『为什么陷进来』写在这里。最高位=1 说明是中断,低 8 位=9 说明是外部设备中断。」中断响应 / 中断向量
取中断号187「plic_claim() 向中断控制器 PLIC 要『中断号 irq』,这就是回答『是哪个设备』的关键一步。」中断控制器 / 设备识别
★ 分发到驱动189–195「按 irq 分发:是 UART0_IRQ 就调串口驱动 uartintr,是磁盘就调磁盘驱动。这就是设备驱动程序分层的分发点。」设备驱动程序
中断应答200–201「plic_complete 告诉 PLIC『我处理完了』,设备才能发下一次中断。」中断处理结束

3. 逐行讲 uartintr() → consoleintr() ——读字符 + 唤醒进程

3.1 uartintr() ——串口驱动:把硬件里的字符一个个读出来(来自 kernel/uart.c)

注释先念:「处理一次 uart 中断:可能是有输入到了,或 uart 准备好继续输出,或两者都有。由 devintr() 调用。」

175void
176uartintr(void)
177{
178  // read and process incoming characters.
179  while(1){
180    int c = uartgetc();           // ① 从串口硬件寄存器读一个字符(读不到返回 -1)
181    if(c == -1)
182      break;                      // 读空了就退出
183    consoleintr(c);              // ★ 把字符交给上层控制台处理
184  }
185
186  // send buffered characters.(顺便处理输出:把发送缓冲里攒的字节送出去)
187  acquire(&uart_tx_lock);
188  uartstart();                    // ② 输出走「缓冲队列」:uart 空闲就从缓冲里取一个字节发
189  release(&uart_tx_lock);
190}
讲清「输入来中断、输出走缓冲」(本课重点):
• 输入:键盘按一个键 → 串口收到 → 发中断 → uartintr 用 uartgetc() 把字符读出来(180 行),交给 consoleintr。
• 输出:进程 write 时,字符先进发送缓冲 uart_tx_buf[](uart.c:100, uartputc),并不直接死等硬件;等串口「我能收下一个字节了」再发中断,uartstart()(188 行)从缓冲取一个字节写给硬件。这就是课本说的缓冲技术——用缓冲区把「快的 CPU」和「慢的设备」解耦。

配套看 uartgetc()(uart.c:161)——它怎么「读一个字符」:

161int uartgetc(void){
164  if(ReadReg(LSR) & 0x01){    // 查「线路状态寄存器」:输入就绪位=1?
166    return ReadReg(RHR);       // 从「接收保持寄存器」读出那个字节
167  } else {
168    return -1;                 // 没有数据
169  }
170}

3.2 consoleintr() ——控制台:字符入缓冲,一行到齐就 wakeup(来自 kernel/console.c)

注释先念:「控制台输入中断处理。uartintr() 对每个输入字符调用它。做删除/退格处理,追加到 cons.buf,若一整行到了就唤醒 consoleread()。」(只看默认分支这条主线,特殊控制键省略)

135void
136consoleintr(int c)
137{
138  acquire(&cons.lock);            // 加锁:中断处理和 read 进程会同时碰 cons 缓冲
139
140  switch(c){
141  // case C('P')/C('U')/C('H') ... 特殊键:打印进程表、删行、退格(此处省略)
158  default:
159    if(c != 0 && cons.e-cons.r < INPUT_BUF_SIZE){ // 普通字符 且 缓冲没满
160      c = (c == '\r') ? '\n' : c;   // 回车统一成换行
161
162      // echo back to the user.
163      consputc(c);                 // ① 回显:把你敲的字符显示到屏幕
164
165      // store for consumption by consoleread().
166      cons.buf[cons.e++ % INPUT_BUF_SIZE] = c; // ② ★ 字符存入控制台输入缓冲
167
168      if(c == '\n' || c == C('D') || cons.e-cons.r == INPUT_BUF_SIZE){
169        // 一整行到了(换行)、文件结束、或缓冲满 →
171        cons.w = cons.e;            // 把「可读边界 w」推到当前位置
172        wakeup(&cons.r);            // ③ ★★ 唤醒在 consoleread 上睡觉等输入的进程
173      }
174    }
175    break;
176  }
177
178  release(&cons.lock);
179}
为什么要 wakeup?(高频考点) 进程调用 read() 读键盘时,如果还没输入,它不会忙等,而是在 consoleread() 里 sleep(&cons.r, ...) 睡着让出 CPU(console.c:96)。等中断来了、一整行到齐,consoleintr 第 172 行 wakeup(&cons.r) 把它叫醒,它才继续把字符拷给用户。这就是「中断驱动 I/O」的精髓:进程睡着等,设备用中断把它唤醒,全程不浪费 CPU。注意 sleep 和 wakeup 用同一个地址 &cons.r 作「等待事件」,这就是它俩能对上的暗号。

对照看 consoleread() 的睡眠点(console.c:91-97):

91    while(cons.r == cons.w){          // 没有新数据可读
96      sleep(&cons.r, &cons.lock);     // ★ 睡在 &cons.r 上,等中断 wakeup
97    }
99    c = cons.buf[cons.r++ % INPUT_BUF_SIZE]; // 醒来后从缓冲取字符

6. 答辩:老师可能问的 10 题(每题给标准答法)

逐行讲源码,老师最爱指着某一行问「这行干嘛」,或问「这属于课本哪个概念」。下面是高危题的标准答法。

Q1. 中断驱动比程序查询(轮询)好在哪?必答
答:程序查询时 CPU 要反复读状态寄存器忙等,设备慢的时候 CPU 全在空转,浪费。中断驱动是设备就绪后主动发中断,CPU 平时可以去执行别的进程,中断来了才花一点时间处理。所以中断驱动大幅提高了 CPU 利用率和并发度,这也是它取代纯轮询的根本原因。
Q2. devintr 怎么知道是哪个设备的中断?高危
答:两步。第一步 r_scause() 读中断原因寄存器,确认这是外部设备中断(最高位=1 且低 8 位=9)。第二步 plic_claim() 向中断控制器 PLIC 要一个中断号 irq,每个设备有固定的 irq(串口是 UART0_IRQ、磁盘是 VIRTIO0_IRQ)。代码用 if(irq==UART0_IRQ) 一比对就知道是谁,再分发给对应驱动。
Q3. 读到字符后为什么要 wakeup?高危
答:因为进程 read 键盘但还没输入时,它在 consoleread() 里 sleep 睡着让出了 CPU(不忙等)。中断处理把字符入缓冲、一整行到齐后,必须用 wakeup(&cons.r) 把那个睡着的进程叫醒,它才能继续读走数据。sleep 和 wakeup 用同一个地址 &cons.r 配对。这正是中断驱动 I/O 里「进程阻塞等待、设备中断唤醒」的标准配合。
Q4. 这套流程属于课本四种 I/O 控制方式中的哪一种?必答
答:属于中断驱动(中断控制)方式。判据:设备就绪靠中断通知 CPU(不是程序查询的忙等),且数据是一个字符一个字符由 CPU 在中断里搬运(不是 DMA 整块搬)。串口/键盘这类字符设备正适合中断驱动。
Q5. 设备驱动程序在这套流程的哪一层?分了几层?高危
答:从下到上四层:① 陷入层 usertrap/kerneltrap(硬件中断进内核的统一入口)→ ② 中断分发层 devintr(辨认是哪个设备、分发)→ ③ 设备驱动层 uartintr/uartgetc(直接读写 UART 硬件寄存器,这才是狭义的「设备驱动程序」)→ ④ 设备无关的控制台逻辑层 consoleintr/consoleread(缓冲、行编辑、与进程交互)。分层好处:上层不用关心硬件细节,换个设备只改驱动层。
Q6. 指着 trap.c:180 r_scause() 问:这行干嘛?高危
答:scause 是 RISC-V 的中断原因寄存器。CPU 陷入内核时硬件会把「为什么陷进来」写进它。devintr 读它来区分:最高位=1 表示是中断(否则是异常/系统调用),低 8 位=9 表示是外部设备中断,=1(配合那个常量)表示时钟中断。它是辨认中断来源的第一手依据。
Q7. 「输入来中断,输出走缓冲」是什么意思?用了课本哪个技术?易问
答:输入:键盘来字符靠中断触发 uartintr 去读。输出:进程 write 时,uartputc 把字符先放进发送缓冲队列 uart_tx_buf 就返回,不死等硬件;等串口就绪发中断,uartstart 再从缓冲取字节发出。这用的是课本的缓冲技术,目的是缓和「CPU 快、I/O 设备慢」的速度矛盾,让进程不被慢设备拖住。
Q8. 处理完中断为什么要调 plic_complete(irq)?进阶
答:PLIC 规定一个设备同一时刻最多挂起一个中断。plic_claim 取走中断号后,该设备的中断被「占用」;只有调 plic_complete(irq) 告诉 PLIC 这次处理完了,这个设备才被允许再次发中断。漏掉它,该设备就再也中断不了了。这是中断的应答/结束环节。
Q9. consoleintr 第 138 行为什么要 acquire(&cons.lock) 加锁?进阶
答:因为中断处理程序和普通进程的 consoleread 会同时访问同一个 cons.buf 及读写指针。不加锁,中断在 read 改指针的半途插进来,缓冲就会被写坏(竞态)。加自旋锁保证对缓冲的访问是互斥的,这是课本「临界资源 / 进程同步」在驱动里的体现。
Q10. 为什么 consoleintr 不是每来一个字符就唤醒,而是等换行?对比
答:因为 xv6 控制台是行缓冲(line at a time):一行没结束前,用户还可能退格修改。所以只在遇到换行 \n、文件结束符或缓冲满时才把可读边界 cons.w=cons.e 推进并 wakeup,让 read 一次拿到一整行。这减少了无谓的唤醒,也符合终端「按回车才提交一行」的使用习惯。
