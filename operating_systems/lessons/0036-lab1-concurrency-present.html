1先亮锁的"本体"：一个 locked 标志

数据结构kernel/spinlock.h
// Mutual exclusion lock.
struct spinlock {
    uint locked;       // Is the lock held? 0=空闲, 1=被占

  // For debugging:
  char *name;        // Name of lock.
  struct cpu *cpu;   // The cpu holding the lock.
};
🎤 怎么说

"自旋锁本体就这一个字段——locked:0 表示没人占，1 表示被占。整个互斥，归根到底就是抢这一个标志位。后面 name 和 cpu 只是调试用的。问题是:两个进程会不会同时把 0 抢成 1、都以为拿到了锁? 看下一段。"

2acquire：用原子操作把锁抢到手（重点）

拿锁kernel/spinlock.c · acquire()
void
acquire(struct spinlock *lk)
{
    push_off(); // disable interrupts to avoid deadlock.
  if(holding(lk))
    panic("acquire");

  // On RISC-V, 编译成原子交换 amoswap.w.aq
    while(__sync_lock_test_and_set(&lk->locked, 1) != 0)
    ;                          // 旧值!=0 说明别人占着 → 原地自旋等

    __sync_synchronize();        // 内存屏障:临界区的访存不许越过这里

  lk->cpu = mycpu();            // 记下持锁者(调试用)
}
🎤 怎么说（这段最容易被追问，慢讲三个高亮行）

"三行最关键。第一，push_off() 先关中断——为了防死锁，等会儿答辩细说。第二，__sync_lock_test_and_set(&locked, 1):它把 locked 读出旧值、同时写成 1，这是一条不可打断的原子指令(RISC-V 上是 amoswap)。返回旧值 0 → 我抢到了，跳出循环；返回 1 → 别人占着，while 让我原地转圈等，这就是'忙等待 / 自旋'。因为读写一气呵成，绝不可能两个进程都拿到旧值 0。"

👉 指着 __sync_synchronize() 说:"这是内存屏障，挡住编译器/CPU 把临界区的读写'乱序'挪到拿锁之前——保证'先拿到锁，才碰共享数据'。"

3release：原子放锁，再开回中断

放锁kernel/spinlock.c · release()
void
release(struct spinlock *lk)
{
  if(!holding(lk))
    panic("release");
  lk->cpu = 0;

    __sync_synchronize();           // 屏障:临界区的写要先让别的CPU看到
    __sync_lock_release(&lk->locked); // 原子地把 locked 置回 0

    pop_off();                      // 与 acquire 的 push_off 配对,恢复中断
}
🎤 怎么说

"放锁是拿锁的镜像。先 __sync_synchronize() 屏障，保证我在临界区里写的东西，别的 CPU 先看见，再放锁。__sync_lock_release 把 locked 原子置 0——注意它不用普通的 locked=0 赋值，因为 C 标准下普通赋值可能被拆成多条存储指令，必须用原子的。最后 pop_off() 把当初关掉的中断配对恢复。"

4sleep / wakeup：长等待就让出 CPU（重点）

睡眠kernel/proc.c · sleep()
// Atomically release lock and sleep on chan.
void
sleep(void *chan, struct spinlock *lk)
{
  struct proc *p = myproc();

    acquire(&p->lock);  // 先拿到进程自己的锁
    release(lk);        // 才放掉外面那把锁(此后不会丢唤醒)

  p->chan = chan;       // 挂在这个"等待频道"上
    p->state = SLEEPING;  // 标记为睡眠
  sched();              // 主动让出 CPU,切给别的进程跑

  p->chan = 0;          // 被唤醒后:收尾
  release(&p->lock);
  acquire(lk);          // 重新拿回外面那把锁,接着干
}
🎤 怎么说

"自旋是干等、烧 CPU；等很久的事(比如等磁盘)就该睡觉让出 CPU。关键是前两行的顺序:先 acquire(&p->lock)，再 release(lk)。这样在'我标记 SLEEPING 并睡下去'这整个过程里，进程锁一直握着——别人想 wakeup 我，必须先拿到这把锁，就一定得等我睡稳了才喊得出，那一声唤醒就不会丢。"

唤醒kernel/proc.c · wakeup()
void
wakeup(void *chan)
{
  struct proc *p;
  for(p = proc; p < &proc[NPROC]; p++) {
    if(p != myproc()){
      acquire(&p->lock);
            if(p->state == SLEEPING && p->chan == chan)
                p->state = RUNNABLE;  // 睡在同一频道的,改成可运行
      release(&p->lock);
    }
  }
}
🎤 怎么说

"wakeup 扫一遍所有进程，把睡在同一个频道 chan 上的从 SLEEPING 改回 RUNNABLE，它们就能被重新调度。注意它也要 acquire(&p->lock)——和 sleep 抢同一把锁，这正是'不丢唤醒'的保证。这套 sleep/wakeup，对应课本信号量 P/V 操作的'睡眠等待'实现。"

6答辩 Q&A（老师最可能问的 5 个）

Q1什么是竞态条件? 什么是临界区?
答法: "竞态条件:多个进程并发访问临界资源，最终结果取决于它们执行的相对先后顺序，导致结果不确定、出错。比如两个进程对同一变量各做一次 count++(读→改→写三步)，交错执行会丢更新。临界区:程序中那段访问临界资源、必须互斥执行的代码——把'读改写'圈进临界区、加锁保护，就安全了。"
Q2自旋锁和睡眠锁(sleep/wakeup)有什么区别? 各用在哪?
答法: "自旋锁:拿不到锁就原地循环等(忙等待)，不让出 CPU。优点是没有切换开销，适合临界区很短、马上能拿到的场景；缺点是等久了白烧 CPU，且持锁期间不能睡。睡眠锁/sleep-wakeup:拿不到就让出 CPU 去睡，别人备好资源再 wakeup 叫醒，适合要等很久的场景(如等 I/O)。对应课本就是忙等待 vs 睡眠等待。"
Q3acquire 一进来为什么要 push_off() 关中断?
答法: "防死锁。如果持锁期间被中断打断，而中断处理程序又想拿同一把锁，它会 spin 等我放锁，可我正被它打断、没机会放——就自己把自己锁死了。所以拿锁时 push_off() 关本 CPU 中断，放锁时 pop_off() 配对恢复。"
Q4sleep 为什么必须"带着锁"进去? 不带会怎样?
答法: "防 lost wakeup(丢失唤醒)。若不带锁:进程检查到资源没好、正准备睡，就在这空档别人备好资源并 wakeup 了，可我还没真睡下，这声唤醒落空；等我真睡下就再没人叫，永久睡死。解法:握着保护资源的锁去 sleep，sleep 内部先 acquire(&p->lock) 再 release(lk)，让 wakeup 必须等我睡稳(拿到 p->lock)才喊得出，唤醒就不会丢。"
Q5这段代码对应教材哪个知识点?
答法: "对应 Ch2 进程同步与互斥:共享变量是临界资源，访问它的代码是临界区，并发出错叫竞态条件。xv6 的自旋锁对应课本的互斥/忙等待，test_and_set 就是课本讲的硬件原子指令(TS 指令)；sleep/wakeup 对应信号量 P/V 操作的睡眠等待——P 相当于'拿不到就睡(sleep)'，V 相当于'放资源并唤醒(wakeup)'。这也是本学期实验① 并发程序设计的核心。"
