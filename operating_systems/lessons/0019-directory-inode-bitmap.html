目录与 inode(inode 不含文件名)· 位示图

这一课只钉牢一个高频陷阱:文件名不在 inode 里。先动手点一张图看它,再上术语。配合 速查地图 使用。

一句话先记住: 文件名住在目录项里(目录每一行 = <文件名, inode号>);文件的其余全部属性——类型、权限、所有者、大小、时间、链接计数、物理块指针——全住在 inode 里。inode 里没有文件名(高频陷阱!)。正因为“名”与“身份”分了家,改名、硬链接、共享才都那么轻。另外:空闲磁盘块用位示图管理,约定 0 = 空闲、1 = 已分配。

❌ 全课最该记住的一句: inode 里没有文件名。文件名只存在目录项里。inode 只管“这个文件是谁、多大、在磁盘哪儿、有几个名字指着它(链接计数)”——它不知道自己叫什么。

🧸 2. 具体例子:身份证档案 vs 外号

🧸 5 岁也懂的比喻: inode 像一个人的身份证档案——记着出生日期、住址、指纹、身高……但档案上偏偏不写“外号”。
文件名像这个人的外号,写在通讯录(目录)里,通讯录每一行是“外号 → 档案编号”。
• 给这个人起个新外号(= 硬链接):只是通讯录多加一行,指向同一份档案,档案本身一个字都不用复制,只把档案上“有几个人叫我”的计数 +1。
• 改外号(= 重命名):只改通讯录那一行,档案原封不动。
一句话:名字写在通讯录,身份写在档案;档案里从不写名字——这就是“inode 不含文件名”。

⚖️ 3. 对照表:两张表钉牢重点

对照一:目录项 vs inode(谁装什么)

比什么目录项(在目录里)inode(索引结点)
装不装文件名装文件名不装文件名 ❌(重点)
核心内容<文件名, inode号>,就这两样类型 / 权限 / 所有者 / 大小 / 时间戳 / 链接计数 / 物理块指针
大小很小(只放名 + 号)较大(装满属性)
作用把“名字”映射到“inode 号”描述文件本体:是谁、多大、在磁盘哪儿

查找一个文件的完整路线:①在目录里逐行比文件名 → 命中后拿到 inode 号 → ②按号读出 inode → 从 inode 拿到属性和物理块位置 → ③再去读数据块。

对照二:传统 FCB vs 拆分后(目录项 + inode)

比什么传统 FCB(文件名 + 属性全塞一起)拆分后(目录项 + inode)
目录项里放什么文件名 + 一大堆属性,全在一起只留 <文件名, inode号>
单个目录项大小大(教材例:约 46 字节)小(教材例:约 16 字节)
一个 4KB 块能放几条约 89 个约 256 个
查找时的磁盘 IO多(每读一屏目录项,顺带把没用的属性也读进来)少(一次 IO 读到更多目录项,比名更快)
共享方不方便难(属性绑死在名旁边)易(多目录项指同一 inode,只加链接计数)

📖 4. 是什么:精确定义(对齐教材口径)

FCB(文件控制块 / 目录项): 描述文件属性、位置、控制信息的数据结构。每个文件对应一个 FCB;FCB 的集合就是文件目录;而目录本身也被组织成一个文件,叫目录文件。

FCB 里通常有哪些信息:

类别内容
文件名用户给文件起的名字
物理位置文件存在磁盘哪些块
逻辑 / 物理结构顺序 / 索引…,连续 / 链接 / 索引分配…
属性大小、建立 / 修改时间、权限、所有者
其他控制信息共享计数 / 链接计数 等
inode(索引结点): 把 FCB 拆成两半——文件名放到目录项里,其余全部描述信息(属性 / 物理地址 / 时间 / 权限 / 链接计数 / 物理块指针)放到 inode 里。于是目录项 = <文件名, inode号>,每个 inode 有唯一编号。
⚠️ inode 不含文件名,文件名只在目录项里(重点考点)。
好处:① 目录项变小 → 查找 IO 少;② 便于共享(多个目录项指同一 inode,只需增加链接计数);③ 改名 / 移动只改目录项,inode 不动。

目录结构:三种(会问区别)

结构特点解决 / 遗留
单级目录全系统一张目录表简单,但不能重名、查找慢 → 已淘汰
两级目录主目录 MFD + 每个用户一张 UFD解决不同用户之间的重名
多级(树形)目录现代系统。绝对路径从根 / 开始;相对路径从当前目录开始彻底解决重名 + 便于分类管理

目录查询方式

方式说明
线性检索在目录里逐个比对文件名,命中拿 inode 号(最常用)
Hash 检索用文件名算哈希直接定位,更快
目录缓存把常用目录项缓存在内存,加速查找
位示图法(管理空闲磁盘块): 用一张二进制位表,表里每个物理块对应一个 bit,记录该块用没用。约定:0 = 空闲、1 = 已分配(教材 / 老师口径,别记反)。
• 分配:扫描找到一个值为 0 的位 → 块号 = 行号 × 每行位数 + 列号 → 把该位置 1。
• 回收:由块号反推 → 行号 = 块号 ÷ 每行位数、列号 = 块号 mod 每行位数 → 把该位置 0。
优点:占空间极小(1 块只占 1 bit),可整张缓存进内存;分配 / 回收快;也容易找连续块。 (位运算级别的代码实现——如 balloc 怎么用位操作找 0 位——见 第 24 课;本课只记考试口径。)

❓ 5. 为什么:一层层往下挖

第 1 层:为什么要目录?

为了按名存取。你只想记住 report.txt 这个名字,不想记它在磁盘第几块。目录把“文件名 ↔ 物理位置”对应起来:你说名字,系统替你查到位置。顺带它还允许重名(不同目录下)、支持共享(多个名字指同一份数据)。

第 2 层:为什么把 FCB 拆成“目录项 + inode”?

因为传统 FCB 把文件名和一大堆属性塞在一起。可是查找文件时只需要比文件名——那堆属性此刻白占磁盘 IO。拆开后,目录项只留 <名, inode号> 变得很小 → 一次 IO 能读到更多目录项 → 比名比得更快。顺带还解决了共享:多个名字指同一个 inode 即可。

第 3 层:为什么 inode 一定不能含文件名?

这是本课的核心“为什么”。如果 inode 里含文件名,那么:
• 硬链接就废了——两个名字想共享同一份数据、同一个 inode,可 inode 里却写着某一个名字,矛盾;
• 改名也得动 inode——本该只改目录一行,却要去改文件本体的描述块。
把“名”抽到目录项、把“身份”留在 inode,名与身份彻底解耦——这才是硬链接、改名、共享都如此轻量的根本原因。

(补)位示图的两个“为什么”

• 为什么 0 表示空闲? 这是教材 / 老师口径的约定,做题一律按“0=空闲、1=已分配”,别自己记反。
• 为什么位示图省空间? 因为1 块只占 1 bit。哪怕 1TB 的盘、4KB 的块,位示图也只要约 32MB,整张都能缓存在内存里,分配 / 回收才快。

✅ 6. 自测:先自己想,再点开答案

选择题 1

下面哪一项,inode 里不包含?

A. 文件名
B. 文件大小
C. 文件的物理块指针
D. 访问权限

点我看答案

答案:A。 这是全课最核心的陷阱——inode 不含文件名,文件名只在目录项里。B(大小)、C(物理块指针)、D(权限)都实打实住在 inode 里,是最常见的干扰项:很多人会想当然地以为“文件名当然在 inode 里”,恰恰错。

选择题 2

采用 inode 后,目录项的结构是?

A. 文件名 + 全部属性
B. 只有 inode 号,没有文件名
C. <文件名, inode号>
D. 只有文件名,没有 inode 号

点我看答案

答案:C。 拆分后目录项只留 <文件名, inode号> 两样。A 是传统 FCB的样子(没拆);B 少了文件名——那样就没法“按名查找”了;D 少了 inode 号——那样查到名也找不到文件本体。

选择题 3

给一个文件改名,系统真正改动的是?

A. 复制一份新的 inode
B. 文件的所有数据块
C. 位示图里对应的那一位
D. 目录项(inode 原封不动)

点我看答案

答案:D。 文件名只在目录项里,改名只改目录项那一行,inode 一个字节都不用动——所以改名极快。A/B 都是“大动干戈”式的错误想象;C 是空闲块管理的事,和改名无关。

选择题 4

位示图中,0 表示什么?(注意老师口径)

A. 该块已分配
B. 该块空闲
C. 该块损坏
D. 该块被锁定

点我看答案

答案:B。 本教材 / 老师口径:0 = 空闲、1 = 已分配,做题一律按此。A 是最毒的陷阱项——正好记反(有些资料确实反着定义,但本课考试按 0=空闲)。C、D 都不是位示图这一个 bit 能表达的含义。

选择题 5

位示图里某位于第 i 行、第 j 列,它对应的块号 = ?

A. i × 每行位数 + j
B. i + j
C. j × 每行位数 + i
D. i × j

点我看答案

答案:A。 块号 = 行号 × 每行位数 + 列号 = i × 每行位数 + j。反推(回收)时:行号 = 块号 ÷ 每行位数、列号 = 块号 mod 每行位数。B(直接相加)、C(行列写反)、D(相乘)都算不出正确块号。
