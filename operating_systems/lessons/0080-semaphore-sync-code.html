🗺 0. 这 6 题在考什么

🧸 一句话： 6 道题全是「用信号量把多个进程安排成对的样子」。不管题面是生产-消费、前趋图、还是司机-售票员，本质都是——
  • 谁先谁后 → 同步信号量（初值 0，先做的人 V）
  • 同一时刻只许一个 → 互斥信号量（初值 1）
  • 同类资源 N 个 → 资源信号量（初值 N）

🧠 1. 做题 5 步法（背下来就能套）

📐 5 步法：
  1. 画进程/箭头：把所有进程和"先后依赖"画出来（箭头 = 谁等谁）
  2. 列信号量：每条"先后依赖"配 1 个同步信号量；每个临界资源配 1 个互斥信号量
  3. 定初值：同步 = 0；互斥 = 1；同类资源数 = N
  4. 写代码：先做者 V 末尾；后做者 P 开头；互斥夹在临界区前后
  5. 查死锁：P(mutex) 顺序不能颠倒；多 P 时必对应同序 V
初值速查表（必背）：
类型含义初值典型名字
同步「某事件已发生」0S, S1, S2, S12…
互斥「临界资源空闲」1mutex, wmutex, rmutex
资源计数「资源还剩 N 个」Nempty, full, chopstick[i]
⚠️ 易错三连：
  1. 同步/互斥搞混 → 初值写错（同步 = 0，互斥 = 1）
  2. P 顺序颠倒 → 死锁（V 顺序无所谓，P 顺序不能换）
  3. 忘 V → 全部阻塞（一次 P 必须配一次 V）

Q33P、Q、R 三进程 + BufI/BufO 双缓冲（最大并行）

题面

P 读入到 BufI，Q 把 BufI 变换后送 BufO，R 从 BufO 输出。要求最大并行性。

分析

  • P→Q：BufI 有数据 → 同步信号量 S1（初值 0）
  • Q→R：BufO 有数据 → 同步信号量 S2（初值 0）
  • BufI 是临界资源 → 互斥信号量 mutex1（初值 1）
  • BufO 是临界资源 → 互斥信号量 mutex2（初值 1）

信号量表

信号量含义初值类型
S1BufI 里有数据（可取）0同步
S2BufO 里有数据（可取）0同步
mutex1BufI 临界资源空闲1互斥
mutex2BufO 临界资源空闲1互斥

代码

// 进程 P：读入到 BufI
P: P(mutex1);
   Add to BufI;
   V(mutex1);
   V(S1);           // 通知 Q：BufI 有数据了

// 进程 Q：变换 BufI → BufO
Q: P(S1);           // 等 BufI 有数据
   P(mutex1);
   Remove from BufI;
   V(mutex1);
   P(mutex2);
   Add to BufO;
   V(mutex2);
   V(S2);           // 通知 R：BufO 有数据了

// 进程 R：从 BufO 输出
R: P(S2);           // 等 BufO 有数据
   P(mutex2);
   Remove from BufO;
   V(mutex2);
   Output;
⚠️ 易错：只写 S1/S2 不写 mutex1/mutex2 = 漏一半分；忘 V(S2) = R 永远等。

Q346 进程 P1…P6 前趋图（7 条边 = 7 个信号量）

题面

前趋关系：P1→P2, P1→P3, P1→P4；P2→P6；P3→P5；P4→P5；P5→P6。用 7 个初值 = 0 的信号量 s12,s13,s14,s26,s35,s45,s56 写出同步代码。

分析

口诀：「边 = 信号量」，每条边配一个同步信号量（初值 0）；边起点 P 在动作完成后 V，终点 P 在动作开始前 P。

代码

// 入度 = 出度统计：P1=3出, P2=1出1入, P3=1出1入, P4=1出1入, P5=1出2入, P6=0出2入

P1: ...; V(s12); V(s13); V(s14);   // P1 是三条边的起点

P2: P(s12); ...; V(s26);           // P2 收 P1 的信号，发给 P6

P3: P(s13); ...; V(s35);           // P3 收 P1 的信号，发给 P5

P4: P(s14); ...; V(s45);           // P4 收 P1 的信号，发给 P5

P5: P(s35); P(s45); ...; V(s56);   // P5 收 P3+P4 的信号，发给 P6

P6: P(s26); P(s56); ...;           // P6 收 P2+P5 的信号，终点
🎯 速算技巧：数每个进程的入边和出边
进程入边（前趋）出边（后继）代码动作
P103开头直接做事，末尾 3 个 V
P2111 个 P，1 个 V
P3111 个 P，1 个 V
P4111 个 P，1 个 V
P5212 个 P，1 个 V
P6202 个 P，0 个 V
结论：起点的 V 数 = 出边数；终点的 P 数 = 入边数。
⚠️ 易错：把"边方向"记反（P 写 V 写错位置）= 进程永远等不到。判断方法：箭头指向谁，谁先 P；箭头离开谁，谁后 V。

Q35爸爸放苹果/桔子，儿子吃桔子，女儿吃苹果

题面

桌上 1 个空盘（容量 1），爸爸放苹果/桔子，儿子只吃桔子，女儿只吃苹果。3 个信号量 S, So, Sa 初值 1, 0, 0。

分析

  • S 初值 1：盘子空才能放（资源信号量）
  • So 初值 0：盘里是桔子才能吃（同步信号量）
  • Sa 初值 0：盘里是苹果才能吃（同步信号量）

代码

Father: P(S);            // 等盘子空
        if (桔子) V(So);
        else       V(Sa);

Son:    P(So);           // 等盘里是桔子
        取桔子;
        V(S);            // 释放盘子
        吃;

Daughter: P(Sa);         // 等盘里是苹果
          取苹果;
          V(S);          // 释放盘子
          吃;
为何 S 初值 = 1？ 盘子容量为 1，初始时空着可以放，所以 S 资源数 = 1。吃完 V(S) 释放盘子，下一个水果才能进。
⚠️ 易错：把 So/Sa 当成资源信号量写初值 1（应该是 0——没人通知你盘里有桔子前不能吃）。

Q36哲学家进餐（奇偶分拿防死锁）

题面

5 个哲学家，5 根筷子 chopstick[5]（初值 1）。用奇偶分拿写出 5 个哲学家进程的伪代码。

分析

死锁根因：5 个人同时「先左后右」→ 每人左手各占一根 → 没人能凑齐两根 → 死锁。
奇偶分拿：奇数号先拿左、偶数号先拿右 → 相邻哲学家拿筷顺序相反，不会同时持同一根。
  • 奇数号 (1,3)：chopstick[i] 是左，chopstick[(i+4)%5] 是右
  • 偶数号 (0,2,4)：chopstick[i] 是左，chopstick[(i+1)%5] 是右

代码

Pi: do {
        P(chopstick[i]);              // 先拿左边
        if (i % 2 == 0) {
            P(chopstick[(i+1)%5]);    // 偶数：再拿右边
        } else {
            P(chopstick[(i+4)%5]);    // 奇数：再拿右边 (即左邻的左)
        }
        吃;
        V(chopstick[i]);              // 放回两根
        V(chopstick[(i+1)%5]);
        if (i % 2 == 0) {
            V(chopstick[(i+4)%5]);
        } else {
            V(chopstick[(i+1)%5]);
        }
    } while (1);
⚠️ 易错：
  • (i+1)%5 是右边，(i+4)%5 = (i-1)%5 是左边 —— 别算反
  • 奇偶的 if/else 必须写成「不一样」的样子（如果一样 = 退化成先左后右，仍会死锁）
  • 放回时 V 的顺序不影响（V 不阻塞，顺序无所谓）

Q37读者-写者问题（读者优先）

题面

用 rmutex、wmutex、readcount 三个信号量/变量，写出完整伪代码。

分析

  • readcount：当前读者数（普通 int 变量，初值 0）
  • rmutex：保护 readcount 的互斥（初值 1）
  • wmutex：写者与第一个/最后一个读者互斥（初值 1）
关键三句话：
  1. 第一个读者进来时 P(wmutex) 锁住写者
  2. 最后一个读者出去时 V(wmutex) 释放写者
  3. 所有读者进出 readcount 时用 rmutex 互斥

代码

// 共享变量
readcount = 0;
rmutex = 1;
wmutex = 1;

Reader:
    P(rmutex);
    if (readcount == 0)    // 第一个读者：锁写者
        P(wmutex);
    readcount++;
    V(rmutex);
    读;
    P(rmutex);
    readcount--;
    if (readcount == 0)    // 最后一个读者：放写者
        V(wmutex);
    V(rmutex);

Writer:
    P(wmutex);              // 写者单刀直入
    写;
    V(wmutex);
⚠️ 易错：
  • if (readcount == 1) —— 错！应该是 == 0（"从 0 变 1"才是第一个读者进入）
  • 忘 P(rmutex) 保护 readcount++ —— 多读者并发改 readcount 会乱
  • 「读者优先」= 写者饥饿；「写者优先」= 读者饥饿 —— 题面说读者优先就用这个模板

Q38司机-售票员问题（同步初值判别）

题面

司机 P(S1) 启动车辆，售票员 V(S1) 关门。S1 初值 = ?，S2 初值 = ?

分析

判断同步还是互斥？看「谁等谁」。
  • 司机启动前要等"门已关"（关门是司机的先决条件）→ S1 是同步
  • 售票员开门前要等"车已停"（停车是开门员的先决条件）→ S2 是同步

答案

  • S1 初值 = 0（S1 表示"车已停好可以启动"——初始时车没停，所以 = 0）
  • S2 初值 = 0（S2 表示"门已关好可以开门"——初始时门没关，所以 = 0）

完整代码

Driver:    // 司机
  P(S1);          // 等"门已关"
  启动车辆;
  正常运行;
  到站停车;
  V(S2);          // 通知"车已停"

Conductor:  // 售票员
  关门;
  V(S1);          // 通知"门已关"
  售票;
  P(S2);          // 等"车已停"
  开门;
⚠️ 易错：看到"两个进程"就以为是互斥写 1。本题司机和售票员是协作关系（"司机等门关、售票员等车停"），都是同步信号量，初值都是 0。

✅ 9. 自测 3 题（动手写，再看答案）

自测 1. 若 Q33 的"最大并行性"不要求（允许 Q 等 P 全部完成），S1 初值应改为多少？
答案
仍可保持 S1=0（同步信号量初值永远是 0，初值 = 0 不影响并行度）。所谓"最大并行"指 P 还没结束 Q 就可以开始取走 BufI 处理。要真正"串行等全部完成"应改用 join/wait() 同步，不是改 S1 初值。
自测 2. Q36 哲学家奇偶分拿，如果写成了"奇数先右后左、偶数先左后右"，死锁吗？
答案
不死锁。"奇偶分拿"只要保证相邻哲学家拿筷顺序相反即可，先左后右还是先右后左均可。题目给的"奇数先左偶数先右"是惯例写法之一，不是唯一解。判断口诀：「邻座拿序相反」即可。
自测 3. Q37 读者写者，如果把 if (readcount == 0) P(wmutex) 改成 if (readcount == 1) P(wmutex)，会出什么问题？
答案
出错。语义反了：第 1 个读者进入时，readcount 从 0 → 1，应该在 == 0 时锁写者；第 N 个读者进来时 readcount = 1, 2, 3... 不应再锁写者（写者已被第一个读者锁了）。如果用 == 1，每个读者都会重复 P(wmutex)，死锁；或者写者永远进不来（取决于初值细节，但一定错）。
自测 4. Q38 司机-售票员，售票员代码写成"P(S1); 关门; V(S1); 售票"对吗？
答案
不对。S1 是"门已关"的同步信号量，应该由关门动作完成后 V，不是先 P 再 V。正确顺序：关门; V(S1); 售票; P(S2); 开门;。P(S1) 应该是司机做的事，不是售票员。
