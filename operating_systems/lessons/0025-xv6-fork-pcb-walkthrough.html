0. 先记这句主线(讲偏了就拉回来)

"进程在内核里就是一个 struct proc——这就是课本说的 PCB,装着进程的一切(状态、PID、内存、寄存器、打开的文件)。fork() 干的事就一句话:复制父进程的 PCB 和内存,造出一个子进程。一次调用、返回两次——父进程拿到子进程的 pid,子进程拿到 0。"

1. 数据结构:struct proc 就是课本的 PCB

先给老师看进程在内核里的"身份证"。来自 kernel/proc.h,第 85 行。课本说"PCB 保存进程标识、处理机状态、调度信息、控制信息,是进程存在的唯一标志"——这个结构体就是它的真身:

82enum procstate { UNUSED, USED, SLEEPING, RUNNABLE, RUNNING, ZOMBIE }; // ★进程状态枚举
84// Per-process state —— 每个进程的全部状态
85struct proc {
86  struct spinlock lock;        // 保护本 PCB 的自旋锁(多核要加锁)
89  enum procstate state;        // ★进程状态 → 对应课本"就绪/运行/阻塞"
90  void *chan;                  // 阻塞时"睡在"哪个事件上(非0即在等待)
91  int killed;                  // 是否被要求杀死
92  int xstate;                  // 退出码,留给父进程 wait() 取走
93  int pid;                     // ★进程标识符 PID → 课本"内部标识符"
96  struct proc *parent;         // ★父进程指针 → 课本"父进程标识/家族关系"
99  uint64 kstack;               // 内核栈的虚拟地址
100  uint64 sz;                   // ★进程内存大小(字节)
101  pagetable_t pagetable;       // ★用户页表 → 进程的地址空间(程序段+数据段)
102  struct trapframe *trapframe; // ★陷入帧:保存用户态寄存器 → 课本"处理机状态"
103  struct context context;      // ★内核上下文:切换进程时存/取的寄存器
104  struct file *ofile[NOFILE];  // ★打开文件表 → 课本"资源清单"
105  struct inode *cwd;           // 当前工作目录
106  char name[16];               // 进程名(调试用)
107};

state 字段对照课本"进程状态"(高频考点)

xv6 的 enum procstate 有 6 个值,正好覆盖课本的进程状态模型:

xv6 状态含义对应课本状态
RUNNING正在 CPU 上执行运行态(执行态)
RUNNABLE万事俱备,只差 CPU,随时可跑就绪态
SLEEPING在等某个事件(如 I/O),给 CPU 也跑不了阻塞态(等待态)
UNUSED这个 PCB 槽还空着,没分配给任何进程(进程尚未创建)
USEDPCB 刚被 allocproc 占用,还没初始化完创建态(过渡)
ZOMBIE已退出但父进程还没 wait() 回收终止态(过渡)
答辩金句:"课本最常讲三态(运行/就绪/阻塞),xv6 的 RUNNING / RUNNABLE / SLEEPING 就是这三态。课本如果讲五态,会再加上创建态和终止态——对应 xv6 的 USED(刚分配、正在初始化)和 ZOMBIE(已退出、等回收)。UNUSED 则表示这个 PCB 槽位还空着,连进程都还没有。"

2. 逐行讲 fork()(来自 kernel/proc.c,第 280 行)

背景注释先念:"创建一个新进程,复制父进程。设置子进程的内核栈,让它像刚从 fork() 系统调用返回一样。"

2.1 先看 allocproc():申请并初始化一个空白 PCB

fork 第一步就调它。这正是课本进程创建的前两步:申请空白 PCB、分配资源。来自 proc.c 第 110 行:

109static struct proc*
110allocproc(void)
111{
112  struct proc *p;
114  for(p = proc; p < &proc[NPROC]; p++) { // ① 扫描进程表,找一个空槽
115    acquire(&p->lock);
116    if(p->state == UNUSED) {              // ★ 找到 UNUSED 的就是空白 PCB
117      goto found;
118    } else { release(&p->lock); }
121  }
122  return 0;                              // 进程表满了,创建失败
124found:
125  p->pid = allocpid();                   // ★ 分配一个全新 PID
126  p->state = USED;                       // ★ 标记"创建中"(对应创建态)
129  if((p->trapframe = (...)kalloc()) == 0){ // ② 分配 trapframe 页(处理机状态区)
130    freeproc(p); release(&p->lock); return 0;
133  }
136  p->pagetable = proc_pagetable(p);     // ② 建一张空用户页表(地址空间)
137  if(p->pagetable == 0){ ... return 0; }
145  memset(&p->context, 0, sizeof(p->context)); // ③ 初始化内核上下文
146  p->context.ra = (uint64)forkret;       // 让它被调度时从 forkret 开始跑
147  p->context.sp = p->kstack + PGSIZE;
149  return p;                              // 返回这个初始化好的空白 PCB(锁还握着)
150}
对上课本:"课本说进程创建第一步是'申请空白 PCB',就是第 116 行找一个 state==UNUSED 的槽;第二步'为新进程分配资源',就是第 129、136 行分配 trapframe 和页表;第三步'初始化 PCB',就是第 125–147 行填 pid、置状态、设上下文。"

2.2 fork() 本体:复制父进程,造出子进程

279int
280fork(void)
281{
282  int i, pid;
283  struct proc *np;                 // np = new proc,子进程
284  struct proc *p = myproc();        // p = 当前进程,父进程
287  if((np = allocproc()) == 0){    // ① 申请一个空白 PCB 给子进程
288    return -1;                       // 没有空槽 → fork 失败返回 -1
289  }
291  if(uvmcopy(p->pagetable, np->pagetable, p->sz) < 0){ // ② ★复制父进程整片内存
293    freeproc(np); release(&np->lock); return -1;
295  }
297  np->sz = p->sz;                    // 子进程内存大小 = 父进程
300  *(np->trapframe) = *(p->trapframe); // ③ ★复制处理机状态(全部用户寄存器)
302  // Cause fork to return 0 in the child. —— 让 fork 在子进程里返回 0
303  np->trapframe->a0 = 0;             // ★★★ 这一行就是"子进程返回 0"的实现!
306  for(i = 0; i < NOFILE; i++)       // ④ ★复制打开文件表(共享文件,引用计数+1)
307    if(p->ofile[i])
308      np->ofile[i] = filedup(p->ofile[i]);
309  np->cwd = idup(p->cwd);            // 复制当前工作目录
311  safestrcpy(np->name, p->name, ...); // 复制进程名
313  pid = np->pid;                     // ★记下子进程的 pid(待会儿返回给父进程)
315  release(&np->lock);
317  acquire(&wait_lock);
318  np->parent = p;                    // ⑤ ★认父亲:建立家族关系
319  release(&wait_lock);
321  acquire(&np->lock);
322  np->state = RUNNABLE;              // ⑥ ★置就绪态 → 课本"插入就绪队列"
323  release(&np->lock);
325  return pid;                       // ★★ 父进程在这里返回:拿到的是子进程的 pid
326}

步骤行一句话讲法对应课本
① 申请空白 PCB287"allocproc 给子进程要一个空 PCB,没有就返回 -1。"创建·申请空白 PCB
② 复制内存291–297"uvmcopy 把父进程整片内存逐页拷给子进程——子进程拿到一份一模一样的地址空间。"创建·分配资源

3. 核心:为什么"一次调用,返回两次"?(必考)

ELI5 比喻:fork 就像把自己整个人复印了一份。复印件(子进程)从复印机里站起来时,它和你长得一模一样、记忆一样、手里拿的东西也一样(内存、寄存器、打开的文件全是复制的)。唯一的区别:你手里的纸条上写着复印件的编号(子 pid),复印件手里的纸条上写着 0。两个人看着各自的纸条,就知道"我是本体还是复印件",于是走向不同的人生分支。

从代码讲清楚"返回两次":

  • 第 300 行把父进程的 trapframe 整个复制给子进程,包括返回地址——所以父子进程都会"从 fork 系统调用返回"这同一个点继续执行。这就是"返回两次"的物理原因:有两个进程,各自都要从这里返回一次。
  • 父进程这条路:执行到第 325 行 return pid,返回值是子进程的 pid(一个 >0 的数)。
  • 子进程这条路:它不走 325 行(它是被调度器唤醒、从 trapframe 恢复现场的)。它的返回值来自 trapframe->a0,而第 303 行已经把 a0 设成了 0。所以子进程"看到"的 fork 返回值是 0。
为什么改 a0?(指到第 303 行答这题)RISC-V 调用约定里,函数返回值放在寄存器 a0。系统调用返回用户态时,用户程序就是从 a0 读返回值的。父进程的 a0 会被设成 pid(由内核 syscall 返回机制写入),而子进程的 a0 被这行代码手动设成 0。同一个 fork,两个进程读各自的 a0,就得到了不同的返回值——这就是教科书那句"fork 对父进程返回子 pid、对子进程返回 0"在代码里的落地。

6. 答辩:老师可能问的 10 题(每题给标准答法)

逐行讲源码,老师最爱指着某一行问"这行干嘛"。下面是高频题的标准答法。

Q1. PCB 里都装了什么?在代码里指给我看。必答
答:PCB 就是 struct proc(proc.h 第85行)。课本分四类信息,代码里都有:进程标识=pid/parent;处理机状态=trapframe(用户寄存器)和 context(内核切换现场);调度信息=state;控制信息=pagetable(地址空间)、ofile[](资源清单)、cwd。
Q2. 为什么说"PCB 是进程存在的唯一标志"?必答
答:因为进程的一切都挂在 PCB 上——没有 PCB,内核就无从知道这个进程是谁、跑到哪、有什么资源。代码里,一个 struct proc 的 state 是 UNUSED 就表示"这里没进程";allocproc 占用它、freeproc 把它清回 UNUSED,进程就从创建到消亡。PCB 在,进程在;PCB 没了,进程就不存在了。
Q3. state 字段对应课本哪几个状态?高危
答:RUNNING=运行态、RUNNABLE=就绪态、SLEEPING=阻塞态,这是课本的三态。若讲五态,再加 USED=创建态(刚分配、初始化中)、ZOMBIE=终止态(已退出、等父进程回收)。UNUSED 是空槽,还没有进程。
Q4. fork() 为什么会"返回两次"?哪行实现的?高危必答
答:因为 fork 之后有两个进程,各自都要从 fork 系统调用返回一次。物理原因在第 300 行 *(np->trapframe) = *(p->trapframe)——把父进程的全部寄存器(含返回地址)复制给子进程,所以子进程也会"从 fork 返回点"继续跑。父进程走第 325 行 return pid;子进程被调度后从它的 trapframe 恢复,返回值取自 a0,而第 303 行已把它设成 0。
Q5. 子进程的 a0 为什么是 0?这行(303)什么意思?高危
答:RISC-V 约定函数返回值放在寄存器 a0。子进程的 a0 被第 303 行手动设成 0,所以子进程看到的 fork 返回值是 0;父进程的 a0 由内核 syscall 返回机制写成子 pid。这一个寄存器的差异,就是"父返回 pid、子返回 0"的全部秘密。
Q6. fork 和 exec 有什么区别?易问
答:fork 只复制、不换程序:子进程是父进程的副本,跑的还是同一个程序。exec 是换程序:把当前进程的内存映像替换成一个新可执行文件,pid 不变但代码全换了。Unix 创建新程序的经典套路是 fork + exec:先 fork 出一个副本,子进程再 exec 换成想运行的程序。这正是课本"进程创建"的工程实现。
Q7. 指着第 291 行 uvmcopy 问:它在干嘛?高危
答:把父进程用户内存的每一页逐页复制到子进程的页表里(从地址 0 复制到 p->sz)。复制完,子进程拥有一份和父进程内容相同、但物理上独立的地址空间——父进程改自己的变量,不会影响子进程。这就是"复印一份自己"里的"内存"部分。
Q8. 指着第 322 行 np->state = RUNNABLE 问:为什么是 RUNNABLE 不是 RUNNING?进阶
答:因为子进程还没轮到 CPU。RUNNABLE=就绪态,意思是"万事俱备、等调度器挑我"。只有当调度器真正把 CPU 给它、切换上去时,它才变成 RUNNING(运行态)。这对应课本进程创建最后一步:"将新进程插入就绪队列"。
Q9. allocproc 怎么找到一个能用的 PCB?易问
答:第 114–116 行,遍历全局进程表 proc[NPROC],找第一个 state==UNUSED 的槽。xv6 用固定大小的静态数组管理所有 PCB(最多 NPROC 个进程)。找到就 goto found 初始化它,找不到就返回 0(进程表满,创建失败)。这对应课本"申请空白 PCB"。
Q10. fork 失败会怎样?返回什么?进阶
答:两种失败:进程表满(allocproc 返回 0,见第287–288行)或复制内存失败(uvmcopy <0,见第291–295行,会先 freeproc 回收已分配的资源)。两种情况 fork 都返回 -1 给父进程,且不创建子进程。所以用户态写代码要判断 fork 的三种返回:<0 失败、==0 子进程、>0 父进程(值为子 pid)。
