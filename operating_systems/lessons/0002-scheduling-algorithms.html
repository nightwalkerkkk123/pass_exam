调度算法 4 件套 + RR 实战

目标：20 分钟内把 FCFS / SJF / HRRN / RR 的走法 + 周转时间计算 + 关键 trade-off 装进脑子

1. 一句话：4 个调度算法

OS 的"处理机调度"要回答 1 个问题：就绪队列里这么多进程，让谁先上 CPU？常见 4 个答案：

算法英文核心思想类比
先来先服务FCFS按到达顺序，谁先到谁先跑银行排号，先来先办
短作业优先SJF谁的服务时间短，谁先跑医院看病，小病先看
高响应比优先HRRN响应比最高的先跑等得越久越优先（兼顾长短）
轮转调度RR每人轮流跑 1 个时间片 q老师让每人发言 1 分钟

2. 评价指标：两个公式（必背）

周转时间 T = 完成时间 − 到达时间
带权周转时间 W = T / 服务时间

W 越大 = 这个进程"等得久 / 跑得短" = 越不划算。SJF 的目标就是让 W 最小。

SJF 的胜利：D 不用等，A 也几乎不卡。平均 T 从 12 降到 10，平均 W 从 3.45 降到 2.21。这就是教材金句：「SJF 是使系统平均周转时间最短的算法」。

6. RR 走法（重点：4 步流程 + 甘特图）

假设时间片 q = 2（每个进程最多跑 2 分钟，时间片到了就送回队尾）。

6.1 四步走法（背下来）

  1. 就绪队列按 FCFS 排：[A, B, C, D, E]
  2. 调度队首进程上 CPU
  3. 让它跑 min(剩余, q) 分钟：剩余 ≤ q 就一次跑完；否则跑满 q
  4. 没跑完 → 送回队尾；跑完 → 撤出

7. 三个算法对比：响应时间 vs 周转时间

算法平均 T平均 W响应时间谁赢？
FCFS12.03.45差（D 等了 13 分钟）短作业亏
SJF10.02.21中周转时间最短
RR (q=2)14.63.72最好响应最快
必背 trade-off（金句）：
「RR 的平均周转时间不是最短，但响应时间最快」——分时系统（如 Windows / Linux 桌面）就要响应快，让每个用户感觉"自己独占 CPU"。
时间片 q 太大 → 退化为 FCFS；q 太小 → 切换开销爆炸。通常 q ≈ 10-100 ms。

8. HRRN（一句话）

响应比 = (等待时间 + 服务时间) / 服务时间 = 1 + 等待/服务。每次调度选响应比最高的。

  • 短作业：服务小 → 响应比大 → 优先 ✓
  • 长作业：等久了等待变大 → 响应比会升高 → 不会饥饿 ✓
  • 缺点：每次调度都要算响应比，开销大
一句话总结：HRRN = SJF + FCFS 的折中，兼顾短作业和不饥饿。

9. 4 个算法的"何时用"

算法适用场景关键弱点
FCFS批处理、辅助调度短作业等长作业
SJF批处理（要求预估服务时间）长作业可能饥饿
HRRN批处理（折中方案）每次都要算响应比
RR分时系统（Windows / Linux 桌面）q 不当会退化

Q4（口述）RR 的时间片 q 太大或太小各会出什么问题？
答案对照（自检后再看）
A1：T = 完成 − 到达；W = T / 服务；W 越大说明这个进程"等得久 / 跑得短"，越不划算。
A2：SJF 把短作业（D=2）放在最前面，D 不用等就跑完；FCFS 把它压在最后，D 苦等 13 分钟（W=7.5）。所以 SJF 的 W 整体更小。
A3：B（SJF 才是 A；RR 的优势是响应快）
A4：q 太大 → 退化为 FCFS（响应慢）；q 太小 → 切换开销爆炸。
