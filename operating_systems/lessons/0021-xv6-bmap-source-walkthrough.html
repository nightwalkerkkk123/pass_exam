0. 先记这句主线(讲偏了就拉回来)

“一个文件由很多数据块组成。bmap(ip, bn) 干的事就一句话:给我文件的第 bn 块,我告诉你它在磁盘的第几号块。小块号走直接地址、大块号走一次间接——这就是课本说的‘混合索引’,只不过 xv6 简化到一级间接。”

1. 数据结构:inode 就是“混合索引”本体

先给老师看文件在磁盘上的“身份证”——索引结点 inode。来自 kernel/fs.h:

27#define NDIRECT 12                       // 直接地址项:12 个
28#define NINDIRECT (BSIZE / sizeof(uint)) // 一个间接块能装多少块号 = 1024/4 = 256
29#define MAXFILE (NDIRECT + NINDIRECT)      // 单文件最大块数 = 12 + 256 = 268 块
32struct dinode {
33  short type;            // 文件类型(目录/文件/设备)
37  uint  size;            // 文件大小(字节)
38  uint  addrs[NDIRECT+1]; // ★ 13 个地址:前 12 个=直接块,第 13 个=一次间接块
39};
讲这页时指着 addrs[NDIRECT+1] 说:“这一行就是‘混合索引’。数组 13 格——前 12 格直接装数据块号(小文件够用),第 13 格装一个‘间接块’的块号,那个间接块里再装 256 个数据块号(留给大文件)。课本画的‘直接地址 + 一次间接’,在 C 里就是这一个数组。”

2. 逐行讲 bmap()(来自 kernel/fs.c)

背景注释先念:“返回 inode 第 bn 块的磁盘块地址;若该块还不存在,就分配一个。”

382static uint
383bmap(struct inode *ip, uint bn)
384{
385  uint addr, *a;
386  struct buf *bp;
387
388  if(bn < NDIRECT){              // ① 块号 < 12 → 直接块
389    if((addr = ip->addrs[bn]) == 0){ // 这一格还没分配?
390      addr = balloc(ip->dev);    // 找一个空闲磁盘块(★空闲块管理)
391      if(addr == 0)
392        return 0;               // 磁盘满了
393      ip->addrs[bn] = addr;       // 记进 inode 第 bn 格
394    }
395    return addr;                  // 直接返回块号——一步到位
396  }
397  bn -= NDIRECT;                  // 把块号“减掉 12”,换算成间接区里的下标
398
399  if(bn < NINDIRECT){            // ② 还在 256 以内 → 一次间接块
401    if((addr = ip->addrs[NDIRECT]) == 0) // 间接块本身还没分配?
402      ip->addrs[NDIRECT] = addr = balloc(ip->dev); // 分配间接块
407    bp = bread(ip->dev, addr);   // 把间接块从磁盘读进内存(★缓冲)
408    a = (uint*)bp->data;          // 间接块的内容 = 一个块号数组 a[256]
409    if((addr = a[bn]) == 0){      // 间接块里第 bn 项还没填?
410      addr = balloc(ip->dev);    // 再分配一个数据块
411      if(addr){
412        a[bn] = addr;             // 填进间接块
413        log_write(bp);            // 改动经“日志”落盘(★事务/一致性)
414      }
415    }
416    brelse(bp);                  // 释放缓冲块
417    return addr;
418  }
420  panic("bmap: out of range"); // 超过 268 块?xv6 装不下
421}

把代码切成“两段命运”讲(这是你讲解的主结构)

代码块行一句话讲法对应课本
① 直接块分支388–396“块号小于 12,直接在 inode 里第 bn 格拿块号;没有就 balloc 一个填进去。读一次就到,这就是小文件为什么快。”直接地址
换算397“块号 ≥ 12,减掉 12,剩下的是‘间接区里的第几项’。”—
② 一次间接分支399–418“先拿到间接块(没有就分配)→ 读进内存当数组 a[256] → 取 a[bn] 才是真正的数据块号。多读一层,这就是大文件的代价。”一次间接地址
越界420“超过 268 块直接 panic——xv6 只做到一级间接。”(课本还有二/三级)

3. balloc() —— 这就是课本“位示图法”的真身

上面 bmap 多次调用 balloc “要一个空闲块”。它怎么找空闲块?来自 kernel/fs.c,这正是课本第六章“文件存储空间管理 · 位示图法”:

66balloc(uint dev){
72  for(b = 0; b < sb.size; b += BPB){      // 一块块位图扫过去
73    bp = bread(dev, BBLOCK(b, sb));       // 读入这一段位图(bitmap)
74    for(bi = 0; bi < BPB && b+bi < sb.size; bi++){
75      m = 1 << (bi % 8);                  // ★ 第 bi 位的“掩码”:1 左移 (bi%8)
76      if((bp->data[bi/8] & m) == 0){       // ★ 该位 = 0 ?→ 这个块空闲
77        bp->data[bi/8] |= m;              // ★ 置 1 = 标记“已分配”
78        log_write(bp);                   // 改动写日志(一致性)
80        bzero(dev, b + bi);              // 把这个块清零再交出去
81        return b + bi;                   // ★ 块号 = 字节下标×8 + 位内偏移
82      }
83    }
84    brelse(bp);
85  }
86  printf("balloc: out of blocks\n");    // 整张位图都满了
87  return 0;
88}
把课本对上代码(答辩金句):“课本说位示图‘0 空闲、1 已分配,盘块号 n = i×8 + j’。看代码:bp->data[bi/8] 就是课本的第 i 个字节,bi % 8 就是第 j 位,m = 1<<(bi%8) 是取那一位的掩码,return b + bi 就是 n = i×8 + j。课本的公式,在这里一行行兑现。”

配套:bfree() —— 回收就是把那一位清 0

92bfree(int dev, uint b){
97  bp = bread(dev, BBLOCK(b, sb));
98  bi = b % BPB;
99  m = 1 << (bi % 8);
100  if((bp->data[bi/8] & m) == 0)
101    panic("freeing free block");    // 想回收一个本就空闲的块?说明出错了
102  bp->data[bi/8] &= ~m;                 // ★ 把该位清 0 = 标记“空闲”
103  log_write(bp);
104}

讲法:“分配是 |= m 把位置 1,回收是 &= ~m 把位清 0,完全对称。第 100 行还做了个检查:回收一个本来就空闲的块就 panic——这正是课本‘数据一致性’里说的空闲块重复错误。”

4. itrunc() —— 删文件:把 bmap 反着走一遍(回收)

删除文件时要把它占的所有块还给磁盘。itrunc 的结构和 bmap 一一对应:先直接块、再间接块。来自 kernel/fs.c:

426itrunc(struct inode *ip){
432  for(i = 0; i < NDIRECT; i++){       // ① 回收 12 个直接块
433    if(ip->addrs[i]){
434      bfree(ip->dev, ip->addrs[i]);   // 每个块还给位图
435      ip->addrs[i] = 0;
436    }
437  }
439  if(ip->addrs[NDIRECT]){            // ② 有间接块?
440    bp = bread(ip->dev, ip->addrs[NDIRECT]);
441    a = (uint*)bp->data;
442    for(j = 0; j < NINDIRECT; j++)     // 先回收间接块里 256 个数据块
443      if(a[j]) bfree(ip->dev, a[j]);
446    brelse(bp);
447    bfree(ip->dev, ip->addrs[NDIRECT]); // 再回收间接块本身
448    ip->addrs[NDIRECT] = 0;
449  }
451  ip->size = 0;                       // 文件大小归零
452  iupdate(ip);                       // inode 改动落盘
453}
对称之美(讲这点显水平):“bmap 是分配 + 寻址:先直接、再间接。itrunc 是回收:同样先直接、再间接,只是把 balloc 换成 bfree。注意第 442 行必须先回收间接块里的 256 个数据块,最后才回收间接块本身——顺序反了就会丢块(找不到那 256 个块号了)。这正是课本说的‘丢失盘块’。”

5. 一张图:bmap 怎么走(PPT 核心页)

bmap(ip, bn) ——“文件第 bn 块在磁盘第几号?” │ bn < 12 ? ──是──► ip->addrs[bn] ← 直接块,1 步到位(小文件) │否 bn -= 12 │ bn < 256 ? ─是─► ip->addrs[12] ─读─► 间接块 a[256] ─► a[bn] ← 2 步(大文件) │否 panic（超过 268 块,xv6 装不下） 容量: 12 直接块 × 1KB = 12KB + 256 间接 × 1KB = 256KB = 268KB / 文件

6. 计算:这个文件系统单文件最大多大?(算给老师看)

盘块 BSIZE = 1024B,块号 sizeof(uint) = 4B。
• 一个间接块能装 1024 ÷ 4 = 256 个块号 → NINDIRECT = 256
• 直接块:12 × 1KB = 12KB
• 一次间接:256 × 1KB = 256KB
• 合计 MAXFILE = (12 + 256) × 1KB = 268 块 = 268KB
对比课本(4KB 块、三级间接能到 4TB):xv6 砍到一级是教学简化,把同一个机制讲清楚,不被三层间接绕晕。

8. 答辩:老师可能逐行问的 12 题

逐行讲源码,老师最爱指着某一行问“这行干嘛”。下面是高危行的标准答法。

Q1. 指着 388 行 if(bn < NDIRECT) 问:为什么用 12 这个数?高危
答:NDIRECT=12 是直接地址的个数。意思是文件前 12 块用直接地址、一步到位。选 12 是工程权衡——让 inode 大小凑整、又能让多数小文件(≤12KB)完全走直接块。改大它,小文件更快但 inode 更大。
Q2. 指着 397 行 bn -= NDIRECT 问:为什么要减?高危
答:前 12 个块号被直接地址用掉了。到这里说明 bn≥12,减掉 12 后,得到的是它在间接块那 256 项里的下标。比如第 12 块,减完是 0,就是间接块里的第 0 项。
Q3. 指着 408 行 a = (uint*)bp->data 问:这行什么意思?高危
答:间接块从磁盘读进内存后,它的内容 bp->data 本质是一串字节。这行把它当成一个块号数组 uint a[] 来看,于是 a[bn] 就能取出第 bn 个数据块号。一次间接块里装的全是块号,不是数据。
Q4. balloc 是干嘛的?跟我们学的哪节有关?必答
答:balloc = 分配一个空闲磁盘块,返回块号,没空闲就返回 0。它正是课本“文件存储空间管理”那一节——位示图法/成组链接法解决的就是“balloc 怎么快速找到空闲块”。
Q5. 413 行 log_write(bp) 为什么不直接写磁盘?进阶
答:因为要保证崩溃一致性。xv6 用日志(事务):改动先记进日志,提交后才真正落盘。万一中途断电,重启能靠日志 redo/撤销,文件系统不会半新半旧。这就是课本“数据一致性控制 / 事务 / Undo-Redo”。
Q6. 访问一个大文件的某一块,要读几次磁盘?对比
答:走间接分支:先读间接块(1 次),再读数据块(1 次)= 2 次盘访问。走直接分支只要 1 次。这量化了“大文件比小文件慢一层”。
Q7. 为什么 xv6 只做一级间接,课本却讲到三级?易问
答:教学取舍。一级间接已能讲清“直接+间接”的核心机制,代码短、不绕。课本三级是为支持超大文件(到 TB 级),原理完全一样——每多一级,就是把‘块号数组’再套一层。
Q8. 这个文件系统单文件最大多大?必答
答:(12 + 256) × 1KB = 268KB。其中 12 直接块、256 来自一个间接块(1024/4)。能在黑板上推给老师看。
Q9. 指着 balloc 第 75 行 m = 1 << (bi % 8) 问:这行干嘛?高危
答:位示图里一个字节管 8 个块。bi%8 是这个块在字节里的第几位,1<<(bi%8) 造出“只有那一位是 1”的掩码 m。后面用它查这一位(&m)或改这一位(|=m 置 1、&=~m 清 0)。
Q10. 位示图里 0 和 1 分别代表什么?这跟课本一致吗?必答
答:一致。0=空闲,1=已分配。代码第 76 行 (data & m)==0 就是“找值为 0 的位 = 找空闲块”,找到后第 77 行 |= m 置 1 标记已用。和课本位示图法定义完全相同。
Q11. itrunc 第 442 行,为什么要先回收间接块里的 256 个块,最后才回收间接块本身?高危
答:因为那 256 个数据块的块号就存在间接块里。要是先把间接块回收了,这 256 个块号就读不到了,它们会变成谁也找不到的“丢失盘块”(课本一致性检查里的错误)。所以必须“先掏空、再扔盒子”。
Q12. bfree 第 100–101 行,为什么回收前要检查那一位是不是已经是 0?进阶
答:位为 0 表示这个块本来就空闲。若此时还要回收它,说明出现了“空闲块重复”(同一块被释放两次),是文件系统不一致的信号,所以直接 panic 报错,而不是默默放过。
