第 3 课 · 期末冲刺

银行家算法 · 死锁避免

目标：20 分钟内把 4 个矩阵 + 安全性检查 + 资源请求流程装进脑子，能独立做计算大题

1. 一句话：银行家算法在干什么

借钱之前先算一笔账——算出"这条路径还得起吗"→还得起就借，还得不起就拒绝（让进程等），避免死锁。

类比：银行贷款给你之前，先模拟一遍还款路径：你借了 → 能还吗 → 你又借了 → 还能还吗 → …… → 都能还得起？→ 借钱。→ 中间有一步还不起？→ 拒绝。

银行家算法属于"避免死锁"，它和"预防死锁"不一样（破坏 4 条件）。避免=运行时算账，允许条件存在但动态避免。

2. 死锁处理的 4 个层次（必背）

层次做法代表方法代价
预防破坏 4 条件之一（设计阶段）一次性申请全部资源资源利用率低
避免运行时算账，绕开死锁银行家算法每次都要算，开销大
检测允许死锁发生，事后检测资源分配图化简需要检测机制
解除检测到死锁后处理撤销进程 / 资源抢占损失已做的工作

3. 4 个核心数据结构（必背矩阵）

名字符号含义求法
最大需求Max进程总共最多需要多少资源题目给
已分配Allocation进程已经拿走的资源题目给
还需要的Need进程还需要多少资源才能完成Need = Max − Allocation
系统空闲Available系统还剩多少可用资源题目给
必背公式（默写 3 遍）：
Need[i][j] = Max[i][j] − Allocation[i][j]
考试 90% 的第一步就是这个！

4. 两个关键流程

4.1 安全性检查（找"安全序列"）

目的：判断当前状态下所有进程能不能都跑完。如果能 → 安全；不能 → 不安全（可能死锁）。

1

初始化 Work = Available

Work 是个"模拟可用资源"，从系统的 Available 开始。

2

找一个 Need ≤ Work 的进程

挨个看：哪个进程的 Need 全部 ≤ 当前 Work？

3

假设它跑完 → Work += Allocation

跑完会归还它占的资源，所以 Work 加上它的 Allocation。

4

把该进程标 Finish = true，循环 2-3

继续找下一个能跑完的进程。

5

所有进程都 Finish？

是 → 安全，安全序列就是跑完顺序；否 → 不安全，存在死锁风险。

4.2 资源分配请求（"借钱"流程）

进程 Pi 提出请求 Request_i，要走 4 步：

1

Request ≤ Need？

否 → 出错（要的比声明的还多）；是 → 继续。

2

Request ≤ Available？

否 → 等（系统没货了）；是 → 继续。

3

试探性分配

假装真的分给它：Available −= Request；Allocation += Request；Need −= Request。

4

跑安全性检查

安全 → 真正分配；不安全 → 撤销（回滚到试探前状态），让进程等。

金句（默写）：「试探分配后跑安全性检查——安全才真分，不安全就回滚」。

5.3 第二步：跑安全性检查（找安全序列）

初始 Work = Available = (10, 5, 7)（用完所有进程后 Work 会涨到 10,5,7 + 全部 Allocation = 17,7,9）

轮次选谁？判断（Need ≤ Work？）Work += Allocation新 Work
1 P1 (1,2,2) ≤ (10,5,7) ✓ + (2,0,0) (10,5,7)→(12,5,7)
2 P3 (0,1,1) ≤ (12,5,7) ✓ + (2,1,1) (12,5,7)→(14,6,8)
3 P4 (4,3,1) ≤ (14,6,8) ✓ + (0,0,2) (14,6,8)→(14,6,10)
4 P2 (6,0,0) ≤ (14,6,10) ✓ + (3,0,2) (14,6,10)→(17,6,12)
5 P0 (7,4,3) ≤ (17,6,12) ✓ + (0,1,0) (17,6,12)→(17,7,12)
安全序列：P1 → P3 → P4 → P2 → P0。所有进程都能跑完 → 当前状态 安全。

5.4 第三步：处理资源请求（典型变种）

题目：P1 提出新请求 Request = (1, 0, 2)，能分吗？

1

Request ≤ Need(P1)？

(1,0,2) ≤ (1,2,2) ✓

2

Request ≤ Available？

(1,0,2) ≤ (10,5,7) ✓

3

试探性分配

Available: (10,5,7) − (1,0,2) = (9, 5, 5)
Allocation(P1): (2,0,0) + (1,0,2) = (3, 0, 2)
Need(P1): (1,2,2) − (1,0,2) = (0, 2, 0)

4

跑安全性检查（Work = (9,5,5)）

找 Need ≤ (9,5,5) 的进程：
• P1 Need (0,2,0) ≤ (9,5,5) ✓ → Work = (9,5,5)+(3,0,2) = (12,5,7)
• P3 Need (0,1,1) ≤ (12,5,7) ✓ → Work = (14,6,8)
• P4 Need (4,3,1) ≤ (14,6,8) ✓ → Work = (14,6,10)
• P2 Need (6,0,0) ≤ (14,6,10) ✓ → Work = (17,6,12)
• P0 Need (7,4,3) ≤ (17,6,12) ✓ → Work = (17,7,12)
安全序列：P1 → P3 → P4 → P2 → P0 ✓

结论：试探后仍然安全 → 真正把资源分给 P1。

6. 关键判断点（考试最爱问）

情况判断动作
Request > Need进程出错（要超量）报错
Request > Available系统没货进程等
试探后能找到安全序列安全真正分配
试探后找不到安全序列不安全回滚，进程等

Q4（选择）试探分配后跑安全性检查，结果是不安全，OS 应该怎么办？
A. 强行分配，反正只是可能死锁
B. 回滚到试探前，让进程等
C. 把进程杀掉
D. 重启系统
答案对照

A2：属于"避免"层；预防是设计阶段破坏 4 条件，避免是运行时算账
A3：① Request ≤ Need？② Request ≤ Available？③ 试探分配 ④ 安全性检查
A4：B（不安全必须回滚，否则会真死锁）
