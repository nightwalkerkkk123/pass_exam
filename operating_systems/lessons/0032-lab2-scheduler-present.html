1先亮"现场"长什么样:状态 + context

数据结构kernel/proc.h
// 进程的 6 种状态
enum procstate { UNUSED, USED, SLEEPING, RUNNABLE, RUNNING, ZOMBIE };

// 切换时要保存的寄存器(就这 14 个!)
struct context {
    uint64 ra;            // 返回地址:ret 之后跳到哪
    uint64 sp;            // 栈指针:用的是哪个内核栈
  // callee-saved
  uint64 s0;
  uint64 s1;
  ...
  uint64 s11;           // s0–s11 共 12 个被调用者保存寄存器
};
🎤 怎么说

"先看两样东西。上面是进程的状态,今天只关心两个:RUNNING(正占着 CPU 跑)和 RUNNABLE(就绪,排队等 CPU)。下面这个 struct context 就是一个进程的'现场快照'——注意它只存 14 个寄存器:ra(回到哪)、sp(用哪个栈)、加 s0–s11。切换,本质就是存好旧的这 14 个、装回新的这 14 个。"

2scheduler 主循环:挑一个、切过去

主循环kernel/proc.c · scheduler()
void
scheduler(void)
{
  struct proc *p;
  struct cpu *c = mycpu();   // 本 CPU 自己的结构
  c->proc = 0;
  for(;;){                    // 永远循环,scheduler 不返回
    intr_on();               // 开中断,避免死锁
    for(p = proc; p < &proc[NPROC]; p++) {  // 扫一遍进程表
      acquire(&p->lock);
            if(p->state == RUNNABLE) {        // 找到就绪进程
        p->state = RUNNING;             // 标记为运行态
        c->proc = p;
                swtch(&c->context, &p->context); // ★切过去:存scheduler,装p
        c->proc = 0;                    // 回来了:p 让出了 CPU
      }
      release(&p->lock);
    }
  }
}
🎤 怎么说

"这是每个 CPU 自己跑的调度循环,for(;;) 永不退出。它扫进程表,一看到 state == RUNNABLE 的进程,就把它改成 RUNNING、然后 swtch 过去。"

👉 指着那行 swtch(&c->context, &p->context) 说:"注意参数——第一个是 scheduler 自己的 context(往里存),第二个是进程 p 的 context(从里面装)。这一行执行完,CPU 就不在 scheduler 里了,而是在 p 里跑。等 p 让出 CPU,控制流又从这一行的下一句回来,继续找下一个。"

3swtch 切的到底是什么?(重点)

汇编kernel/swtch.S
# void swtch(struct context *old, struct context *new);
# a0 = old(往里存),  a1 = new(从里面装)
swtch:
        sd ra, 0(a0)      # 存:把当前 ra 写进 old->ra
        sd sp, 8(a0)      # 存:把当前 sp 写进 old->sp(记住旧栈)
        sd s0, 16(a0)
        ...               # s1..s11 一路存进 old
        sd s11, 104(a0)

        ld ra, 0(a1)      # 装:从 new->ra 取出新的返回地址
        ld sp, 8(a1)      # 装:从 new->sp 换上新栈(换人关键!)
        ld s0, 16(a1)
        ...               # s1..s11 一路装回
        ld s11, 104(a1)
        ret               # 跳到新的 ra —— 已经在另一个线程里了
🎤 怎么说(这段最容易被追问,慢讲)

"swtch 没什么魔法,就两段:上半段 sd(store)把当前这 14 个寄存器存进 old;下半段 ld(load)从 new 里把另一套 14 个装回 CPU。最关键是这两行:ld sp 换了栈指针——等于换了内核栈;ld ra 换了返回地址。所以最后那个 ret 一执行,CPU '返回'到的根本不是调它的地方,而是 new 这个线程上次离开的位置。换人就这么发生了。"

👉 一句收口:"它切的是栈 + 寄存器,不是地址空间——页表没动。这是辨析题最爱考的点。"

3.5进程怎么"让出":yield → sched

让出kernel/proc.c · yield() / sched()
void yield(void) {
  struct proc *p = myproc();
  acquire(&p->lock);
    p->state = RUNNABLE;    // 我还想跑,先改回就绪、去排队
  sched();
  release(&p->lock);
}

void sched(void) {
  ...
    if(!holding(&p->lock)) panic("sched p->lock");   // 必须持锁
    if(intr_get())        panic("sched interruptible");// 必须关中断
    swtch(&p->context, &mycpu()->context);  // 切回本CPU的scheduler
}
🎤 怎么说

"进程时间片到了,时钟中断里会调 yield。它做两件事:把自己的状态从 RUNNING 改回 RUNNABLE(我还想跑,只是先排队),然后调 sched()。sched 里那行 swtch(&p->context, &mycpu()->context)——正好和 scheduler 里的方向相反:存进程 p、装回 scheduler。于是控制流回到了刚才 scheduler 那行 swtch 的下一句。"

👉 点一下两个 panic:"注意上面这两行检查——调 sched 之前必须持着 p->lock、必须关中断。为什么,留到答辩 Q4 说。"

5答辩 Q&A(老师最可能问的 5 个)

Q1为什么每个 CPU 要有一个独立的 scheduler 线程?
答法: "多核要同时调度,各调各的。如果共用一个 scheduler,多核就得抢同一套调度寄存器/栈,反而要加锁串行。每个 CPU 一个 scheduler(有自己的 c->context、自己的栈),互不干扰,真正并行。而且它是个独立的内核线程,进程切回它、它再切给下一个进程,所有切换路径统一。"
Q2swtch 切换了什么?切地址空间(页表)吗?
答法: "切的是 14 个寄存器:ra、sp、s0–s11。换 sp 等于换内核栈,换 ra 等于换返回点,ret 后就在另一个线程里了。它不切页表/地址空间——那是另一回事。所以'进程切换(换 CPU 现场)'和'地址空间切换(换页表)'要分清。"
Q3RUNNABLE 和 RUNNING 区别?进程让出 CPU 后变哪个?
答法: "RUNNING = 此刻正占着某个 CPU 在跑;RUNNABLE = 就绪,万事俱备只差 CPU,在队列里等。进程 yield 让出时把自己改成 RUNNABLE(还想跑,先排队),scheduler 挑中谁就把谁改成 RUNNING。对应教材的'运行态 ↔ 就绪态'转换。"
Q4为什么 sched() 之前要持锁 + 关中断?
答法: "① 持 p->lock:切换过程里进程状态(RUNNABLE)和现场是'半截'的,锁防止另一个 CPU 同时把这个进程也调走跑两份。要等切到 scheduler、对方拿到锁才安全放手。② 关中断:切换中途若来中断、又触发调度,会嵌套切换、栈乱套。所以 sched 里用 panic 强制检查这两个前提。"
Q5这段代码对应教材哪个知识点?
答法: "对应 Ch3 处理机调度里的进程切换 / 上下文(现场)的保存与恢复,以及进程的运行态↔就绪态状态转换。scheduler 体现'调度程序按策略选进程',swtch 体现'保存现场、恢复现场'。这也是本学期实验② 处理机调度的核心。"
