这一节只要记住两件事:
① rmutex 保护计数器 readcount(读者们抢着改同一个数字,要让"改数字"这步互斥);
② wmutex 是真正"挡写者"的门——只有第一个读者按门铃抢锁,也只有最后一个读者按门铃放锁,中间读者都不碰它(否则读者之间就不能并发读了)。

🗺 0. 全局故事线:这张图在考什么

学校自命题里,伪代码 10 分必出一道,读者写者是 PV 信号量最常考的第二个问题(第一个是生产者-消费者)。两个锁、两个角色、一个计数器,三个变量要默写对。

🧸 一句话: 想象一个自习室:读者= 来翻书的学生(可以好几个人同时翻);写者= 来改书内容的人(必须独占自习室)。这道题考的是——怎么让学生之间不打架、学生和改书的人不冲突、还让尽量多的学生同时在。

你已经在 0050 里栽过一次:把 rmutex 和 wmutex 搞混、把"if(readcount==0)"写成"if(readcount==1)"、把中间读者也加了 P(wmutex)。这一节我们只做一件事:把每行代码为什么在那里,讲明白。

🎬 1. 画面:自习室门口的两个物件

两个物件 = 两个信号量 + 一个计数器

物件对应代码真实类比初值
🔢 借书登记本(readcount) int readcount = 0; 自习室门口的计数器:现在里面有几个学生在翻书 0(一开始没人)
🧾 改登记本的笔(rmutex) semaphore rmutex = 1; 改这个计数器只有一支笔——你拿起来改的时候别人不许改,否则数字会乱 1(只有一支)
🚪 自习室大门锁(wmutex) semaphore wmutex = 1; 真正"挡住改书的人"的锁:第一个学生进门要锁上(不让改书的人进),最后一个学生走要打开(让改书的人能进) 1(一开始门是开的)
关键区分(记住这一条,后面所有坑都解开):
· rmutex = 保护计数器用的"小锁"——每个读者进出都要借/还,但只用来改 readcount 这一个动作;
· wmutex = 挡写者用的"大门锁"——只有第一个和最后一个读者碰它,中间读者碰都不碰(碰了就错)。

🔑 2. 两个锁,各管一摊

🧾 rmutex(小笔锁)

  • 作用对象:共享变量 readcount
  • 谁碰:每个读者(进门出门都碰)
  • 碰几次:每个读者 P 一次 V 一次
  • 互斥粒度:只锁住"读+改 readcount"这两步,读文件本身不被它锁

🚪 wmutex(大门锁)

  • 作用对象:整个"读者群体"作为一个整体,挡住写者
  • 谁碰:只有第一个读者进门和最后一个读者出门碰
  • 碰几次:整个读者群体只 P 一次 V 一次
  • 互斥粒度:锁住的是"写者不能进"这件事
⚠️ 90% 的错都栽在这: 你看到代码里有 P(rmutex) 又 P(wmutex),就以为"两个锁挨着用,干的事差不多"——错!rmutex 是给计数器上锁,wmutex 是给写者上锁,完全两码事。下面 3、4 节会一行一行拆给你看。

👀 3. 读者的"四步走"(逐行动画)

下面演示3 个读者依次进门、1 个写者在门外等的完整流程。点"▶ 走一遍"看顺序,也可以单步点 ⏵。

readcount0
rmutex (笔锁)1 空闲
wmutex (门锁)1 开着
屋里读者无
写者门外等
每步会高亮当前读者 + 更新状态灯 + 在日志里说明发生了什么
读完动画后,记住这四步(任何读者进门都做这四步,顺序死死不能换):
  1. 借笔改数:P(rmutex) → 改 readcount → V(rmutex)(就改一个数字,小锁护一下)
  2. (第一个读者额外)按门铃锁门:if(readcount==0) P(wmutex)
  3. 读文件:read_file()(这一步不在任何锁里,可以和别的读者并发)
  4. 借笔改数:P(rmutex) → 改 readcount → V(rmutex)
  5. (最后一个读者额外)按门铃开门:if(readcount==0) V(wmutex)

✍️ 4. 写者:三行搞定

// 写者最简单——只要抢到大门锁,屋里就一定没有读者 writer() { P(wmutex); // 门外等,直到屋里没人 write_file(); // 改书(此时屋里没人,你也独占) V(wmutex); // 改完放人进来 }
🧸 一句话: 写者不碰 rmutex(它不关心屋里几个人),只抢大门锁 wmutex。抢到 = 屋里没读者,改完就放人。

📖 5. 完整代码 + 逐行解释(贴出来就能默写)

// ====== 信号量与计数器 ====== semaphore rmutex = 1; // 保护 readcount 的小笔锁 semaphore wmutex = 1; // 挡写者的大门锁 int readcount = 0; // 当前屋里有多少读者 // ====== 读者 ====== reader() { // === 进门段:借笔改数 + (首读者)锁门 === P(rmutex); ① 借笔 if (readcount == 0) P(wmutex); ② 第一个读者:锁门挡写者 readcount++; ③ 计数器 +1 V(rmutex); ④ 还笔 // === 真正干活 === read_file(); ⑤ 读文件(此刻 rmutex 已放,可与别人并发读) // === 出门段:借笔改数 + (末读者)开门 === P(rmutex); ⑥ 借笔 readcount--; ⑦ 计数器 -1 if (readcount == 0) V(wmutex); ⑧ 最后一个读者:开门放写者 V(rmutex); ⑨ 还笔 } // ====== 写者 ====== writer() { P(wmutex); ① 抢门锁——必须屋里没读者才能进 write_file(); ② 改文件(独占) V(wmutex); ③ 放人进来 }
行号代码为什么在这里
①P(rmutex)借笔。准备改计数器 readcount,要独占改的动作
②if(readcount==0) P(wmutex)只有屋里空无一人时(我是第一个),才锁大门。如果屋里已经有人,门已经被锁过了,我进门不需要再锁
③readcount++计数器 +1,告诉世界"屋里多了一个我"
④V(rmutex)还笔。改完数赶紧放,后面真正读文件不要占着笔
⑤read_file()真正读文件。这一步没有 rmutex 也没有 wmutex,可以和别人并发读
⑥P(rmutex)出门前再借一次笔,准备改计数器
⑦readcount--计数器 -1,告诉世界"我走了"
⑧if(readcount==0) V(wmutex)只有屋里最后一个人走的时候才开门。屋里还有别人,我开门会让别人冲出去
⑨V(rmutex)还笔,完成
一句话浓缩: 进门借笔改数,出门借笔改数——两次借笔都是为了改 readcount(所以两次都 P/V rmutex);锁门开门各一次,且只在我是第一个/最后一个时才做。

❓ 6. 为什么这样设计(三层追问)

第 1 层 · 为什么需要 rmutex?

因为 readcount 是所有读者共享的整数。多个读者同时执行 readcount++ 会发生什么?

举例:readcount 当前是 1,两个读者 A、B 同时读到 readcount=1,同时 +1 写入,结果 readcount=2——但屋里应该 3 个人!少算了一个。

rmutex 就是让"读 readcount → 改 readcount"这两步绑成一个原子动作,中间不会被打断。

第 2 层 · 为什么需要 wmutex?

如果只有 rmutex 没有 wmutex,会出现什么?

读者 A 在屋里读着;写者进来改文件;A 读到一半,数据已经变了——A 读到旧数据和新数据的拼接,数据损坏。

所以读者和写者必须互斥。但读者之间不能互斥(否则就一个人看,效率崩了)——所以单独拿一个 wmutex,让读者群体作为整体和写者互斥。

第 3 层 · 为什么"第一个"和"最后一个"才碰 wmutex?

关键:读者之间要并发。如果每个读者都 P(wmutex) → 读 → V(wmutex),那同一时刻屋里只能有 1 个读者——这就是普通互斥,不是读者写者。

技巧:把 wmutex 当成读者的"集体占座"。第一个读者代表全体占座(P(wmutex)),最后一个读者代表全体让座(V(wmutex))。中间读者不算"占座"也不"让座",所以不碰 wmutex。

这也回答了"为什么是 if(readcount==0) 不是 if(readcount==1)":进入时屋里 0 个人意味着我是第一个;离开后屋里 0 个人意味着我是最后一个——0 表示"无人",不是"1 个人"。

⚠️ 7. 考场上常踩的 5 个坑

坑 1 · 把 rmutex 和 wmutex 搞混:
· rmutex = 改计数器用的小锁,所有读者都碰;
· wmutex = 挡写者用的大门锁,只有首末读者碰。
· 记反了整段代码全是错的。
坑 2 · "if(readcount==0)"写成"if(readcount==1)":
· readcount==0 表示"屋里现在没人,我来当第一个"或"我走了,屋里没人了";
· 写成 1,逻辑就反了——第一个人进门反而不锁门,最后一个人走反而不开门。
坑 3 · 中间读者也 P(wmutex):
· 这样读者之间就互斥了,屋里同时只能 1 人——退化成普通互斥,失去读者写者的意义。
· 记住:中间读者只改 readcount,不碰 wmutex。
坑 4 · readcount++ 没放在 P(rmutex) 和 V(rmutex) 之间:
· 这步必须被 rmutex 保护,否则多个读者同时改计数会丢更新。
· 改 readcount 这步必须"借笔—改—还笔"三步走齐。
坑 5 · 初值填错:
· rmutex、wmutex 都初值 1(互斥信号量);
· readcount 初值 0(一开始没人);
· 没有第三个信号量(不要画蛇添足加 mutex)。

✅ 8. 自测:每题先自己想,再点开答案

Q1 · 两个信号量各管什么?

说出 rmutex 和 wmutex 各自保护什么、谁会碰它。
点我看答案

· rmutex:保护计数器 readcount。每个读者进门出门都要 P/V 它,目的是让"改 readcount"这一步互斥。

· wmutex:把写者挡在门外(让读者群体整体和写者互斥)。只有第一个读者进门时 P,最后一个读者出门时 V,中间读者不碰。

Q2 · if(readcount==0) 为什么是 0 不是 1?

能不能把"if(readcount==0)"写成"if(readcount==1)"?为什么?
点我看答案

不能。readcount==0 在进门段表示"屋里当前是空的,我是第一个进来的人"——所以要锁门;在出门段表示"我走之后屋里空了"——所以要开门。

写成 readcount==1,逻辑就反了:第一个人进门时 readcount 已经是 0,根本不等于 1,不会锁门;最后一个人走之前 readcount 是 1 倒是能匹配——但这时候开门会让屋里还在读书的人冲出去。0 = "无人" = "我是第一个/最后一个人",不是"1 个人"。

Q3 · 中间读者为什么不碰 wmutex?

如果中间读者也 P(wmutex) 再 V(wmutex),会发生什么?
点我看答案

屋里就只能同时有 1 个读者——读者之间互斥了。这等于退化成普通互斥,失去读者写者问题"多读单写"的意义。

读者写者问题的核心就是多读(多个读者可以并发读)。中间读者不碰 wmutex,就能让所有读者共享"集体占座"的那把锁,实现"读者群体内部不互斥"。

Q4 · 写者饥饿(读者优先的代价)

这个方案叫"读者优先"。如果有源源不断的读者进来,写者会怎样?
点我看答案

写者可能饿死(starvation)。

原因:只要 readcount > 0,门就被第一个读者锁着,写者一直在 P(wmutex) 上阻塞。每来一个新读者,readcount 又 +1,门继续锁着,写者永远进不去。

这是"读者优先策略的代价"——公平换吞吐量。想避免饿死,要用写者优先或公平队列方案,但那是另一道题。

Q5 · 默写挑战(在纸上闭眼写)

不看屏幕,把整段读者优先的代码默写出来,包括:
  1. 三个变量声明(rmutex, wmutex, readcount)
  2. reader() 函数的完整 P/V 序列
  3. writer() 函数
点我对答案
semaphore rmutex = 1; semaphore wmutex = 1; int readcount = 0; reader() { P(rmutex); if (readcount == 0) P(wmutex); readcount++; V(rmutex); read_file(); P(rmutex); readcount--; if (readcount == 0) V(wmutex); V(rmutex); } writer() { P(wmutex); write_file(); V(wmutex); }

能默写对 = 这道 10 分你拿稳。卡壳在哪一步,就回到第 5 节对照表再读一遍。

Q6 · 打乱顺序判断

下面这段代码是正确的读者优先实现吗?有错的话指出哪行错了,会出什么后果?
reader() { P(wmutex); // (1) readcount++; V(wmutex); // (2) read_file(); P(wmutex); // (3) readcount--; V(wmutex); // (4) }
点我看答案

错在三处:

  1. (1)(2) 用 wmutex 保护 readcount++ 是错的——应该用 rmutex。这样会让读者之间也互斥,而且写者会被无意挡住。
  2. (1) 进门就 P(wmutex) 而非"第一个读者才 P(wmutex)"——每个读者都抢大门锁,屋里只能同时有 1 个读者,退化成普通互斥。
  3. 没有 if(readcount==0) 判断——就算修对锁,也必须首读者才锁、末读者才开,否则门会乱。

正确版本见 Q5 答案。
